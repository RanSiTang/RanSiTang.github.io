数据库迁移

发表于 2026-06-08 分类于 数据库与持久化 ， Java数据访问与持久化

一、为什么需要数据库迁移

1. 核心问题：Schema 的演化与管理

1.1 问题的起源

在软件开发的早期，数据库变更通常是最”原始”的操作方式——开发人员直接在数据库服务器上执行 SQL 脚本来修改表结构。这种方式在单人开发或极小团队中勉强可行，但随着团队规模和系统复杂度的增长，它暴露出一系列致命问题。

想象一个典型的场景：

开发人员 A 在本地数据库添加了一个 phone 列，开发人员 B 删除了 legacy_status 列并重命名了 user_type 表。两周后，当代码部署到测试环境时，没人记得究竟执行过哪些 SQL，更没人知道应该如何把生产环境的数据库升级到与代码匹配的状态。

这种混乱就是”数据库变更地狱 (Database Change Hell)”。

1.2 没有版本控制的数据库变更会引发什么？

问题 具体表现
状态漂移 (Drift) 开发、测试、生产三个环境的数据库 Schema 各不相同，Bug 只在特定环境复现
变更不可追溯 “谁在什么时候改了什么？为什么？” — 无法回答这些问题
部署噩梦 每次发布都是 DBA 手工执行一堆 SQL 脚本，顺序、依赖全凭记忆
回滚困难 出问题时无法准确知道应该执行哪些反向 SQL 来恢复
多分支冲突 Git 可以自动合并代码，但数据库变更的合并没有任何自动化支持
新人上手成本 新开发者拿到代码后，不知道如何搭建一个与团队一致的本地数据库

1.3 数据库迁移的核心哲学

数据库迁移 (Database Migration) 的核心思想是：像管理源代码一样，用版本控制的方式管理数据库 Schema 的变更。

这个类比可以帮助理解：

概念 源代码 数据库 Schema
版本控制工具 Git Flyway / Liquibase
变更单元 Commit Migration / Changeset
变更历史 git log flyway_schema_history 表 或 DATABASECHANGELOG 表
当前状态 HEAD 指向的快照 数据库所有已应用迁移的集合
协作方式 拉取 → 合并 → 推送 编写迁移脚本 → 提交到代码库 → 部署时自动执行
回滚 git revert 编写反向迁移 (Undo Migration) 或手动回滚

核心原则：数据库的每一次结构变更（DDL）和数据变更（DML）都应该被记录为一个有序的、不可变的迁移脚本，这些脚本与应用程序源代码一同存储在版本控制系统中。应用启动时，迁移工具会自动检测并执行尚未被应用过的迁移脚本，从而保证数据库 Schema 始终与代码期望的状态一致。

2. 问题域界定：ORM 能替代迁移工具吗？

2.1 Hibernate/JPA 的 ddl-auto 策略

许多开发者第一次接触到 Schema 自动化管理是通过 Hibernate/JPA 的 ddl-auto 配置：

策略 行为
none 什么都不做
validate 仅校验实体与表结构是否匹配，不匹配则启动失败
update 根据实体定义自动修改表结构（仅新增，不会删除列或表）
create 每次启动时删除旧表并重建
create-drop 启动时创建，关闭时删除（常用于集成测试）

2.2 为什么 ddl-auto 不够？

维度 ddl-auto=update 专用迁移工具 (Flyway/Liquibase)
变更粒度的控制 全自动，无法精确控制每一步变更 精确到每一步，开发者定义每次变更
生产环境安全性 极不安全，可能误操作 安全的，变更经过审核和测试
数据迁移 不支持（只改结构，不改数据） 支持，迁移脚本可以同时包含 DDL 和 DML
回滚能力 不支持 支持（Liquibase 原生支持，Flyway 企业版支持）
变更历史追溯 无记录 完整的变更历史表，记录每次迁移的详细信息
多环境一致性 依赖”魔法的”自动推断 确定的，所有环境执行相同的迁移脚本
复杂变更 无法处理（如：列重命名、数据清洗后改类型） 完全可控
团队协作 无法审查变更 SQL 脚本随代码提交，走 Code Review 流程

2.3 最佳实践的组合：ORM + 迁移工具

结论：ORM 负责”表达”对象与表之间的映射关系，迁移工具负责”执行” Schema 的变更。二者不是替代关系，而是互补关系。生产环境中，ddl-auto 应始终设置为 validate 或 none，Schema 的变更完全交由迁移工具管理。

二、Flyway

1. Flyway 概述

1.1 定义与定位

Flyway 是 Boxfuse 公司开发的开源数据库迁移工具，是 Java 生态中最流行的 Schema 版本管理方案之一。它的设计哲学极其简单：

“版本化的 SQL 脚本，按顺序执行。”

Flyway 不做任何复杂的事情——它只是追踪哪些 SQL 脚本已经执行过，然后按版本号顺序执行尚未被应用的脚本。这种极简主义是其最大的优势。

1.2 核心设计原则

  1. 约定优于配置 (Convention over Configuration)：遵循文件命名规范即可，几乎不需要额外配置。
  2. SQL 优先 (SQL-First)：迁移脚本就是纯 SQL 文件，无学习成本，DBA 可以直接审查和修改。
  3. 不可变性 (Immutability)：一旦某个版本的迁移被应用到数据库，它就永远不能被修改。
  4. 幂等性追求：虽然 Flyway 不强制幂等，但通过追踪表保证同一个迁移只被执行一次。

1.3 Flyway 的版本管理模式

Flyway 在目标数据库中维护一张元数据表（默认表名为 flyway_schema_history），用于记录所有已应用的迁移：

这张表是 Flyway 的”大脑”——每次应用启动时，Flyway 会扫描 classpath 中的迁移文件，与这张表中已记录的迁移进行比对，只执行那些未曾被应用过的迁移。

2.4 版本化迁移 (V) vs 可重复迁移 (R)

特性 版本化迁移 (V) 可重复迁移 (R)
执行次数 仅一次 每次内容变更后重新执行

重要行为细节：可重复迁移在所有版本化迁移执行完毕后才按描述字母顺序执行。如果可重复迁移的内容发生了变更（即 checksum 变化），Flyway 会删除旧的定义并重新执行该脚本。如果内容未变，则跳过。

2.5 命名避坑指南

  1. 版本号一旦发布就不可修改：V1.0.0__xxx.sql 已提交到 Git 并被应用到了数据库后，就不能再改它的内容或版本号。
  2. 分隔符是双下划线 __：单下划线会破坏版本号与描述的解析。
  3. 描述部分使用下划线分隔单词：V1__Create_user_table.sql 而不是 V1__Create user table.sql（空格在命令行中可能导致问题）。
  4. 不要在一个版本号上”追加”变更：如果要修改 V1 中创建的表，应该写一个新的 V2 迁移，而不是修改 V1 本身。

3. 迁移状态机

3.1 迁移的生命周期

每个迁移在 Flyway 的管理下会经历以下状态：

3.2 状态详解

状态 含义 数据库表现
Pending 迁移文件存在，但尚未被应用到数据库 flyway_schema_history 表中没有该版本记录
Success 迁移已成功执行 flyway_schema_history 表中该版本 success = true
Failed 迁移执行过程中发生错误（如 SQL 语法错误） flyway_schema_history 表中该版本 success = false
Above Target 版本号高于当前目标版本 不执行，通常出现在设置 target 参数时
Below Baseline 版本号低于基线版本 不执行，用于在已有数据库上引入 Flyway
Missing 迁移文件已被删除，但 history 表中有该版本记录 启动报错
Future history 表中有比当前迁移文件更新版本的记录 启动报错（除非设置 ignoreMigrationPatterns）
Out of Order 文件版本号早于已执行的某版本但未按顺序 启动报错（除非设置 outOfOrder=true）
Outdated 可重复迁移的 checksum 已变化 将重新执行
Superseded 可重复迁移的最新执行记录与新文件比对 仅用于信息展示
Ignored 被配置为忽略 不执行也不报错

3.3 关键状态问题的处理

Failed 状态：这是最需要处理的状态。当迁移执行失败时（例如 SQL 语法错误），Flyway 会在数据库中记录一条 success = false 的记录。在旧版 Flyway 中，修复后续的迁移脚本后，你需要手动删除这条失败记录才能重新运行。从 Flyway 9.x 开始，可以使用 flyway repair 命令自动清理失败的迁移记录。

Missing 状态：这意味着有人删除了一个已被应用的迁移文件。Flyway 会拒绝启动。你需要恢复该文件，或者使用 flyway repair 将其从 history 表中移除（谨慎！这样做可能会丢失关键的 DDL 变更历史）。

4. 核心命令详解

4.1 migrate (最核心的命令)

作用：扫描并执行所有处于 Pending 状态的迁移。

执行流程：

  1. 扫描 classpath 中 db/migration 目录下的所有迁移文件
  2. 查询 flyway_schema_history 表获取已应用的迁移列表
  3. 计算增量：Pending = 文件列表 - 已应用列表
  4. 按版本号升序执行所有 Pending 的版本化迁移
  5. 检查可重复迁移是否有变更，如有则重新执行
  6. 将成功/失败记录写入 flyway_schema_history 表

在 Spring Boot 中：只要引入了 flyway-core 依赖，migrate 会在应用启动时自动执行，无需任何手动调用。

使用场景：你的数据库已经运行了几年，有一百多张表，现在想引入 Flyway 来管理后续的变更。你不能从 V1__Create_all_tables.sql 开始，因为那些表已经存在了。此时你使用 baseline 命令，设置基线版本为 1.0。之后 Flyway 只会执行版本号大于 1.0 的迁移。

4.6 repair

作用：修复 Flyway 的 history 表。

具体会做什么：

  • 删除所有 success = false 的记录（即失败迁移的记录）
  • 重新计算已应用迁移的 checksum（适用场景：迁移文件意外被微小修改但不影响实际 DDL 结果）
  • 删除版本号在当前迁移文件中找不到的 history 记录

使用场景：迁移执行失败后，你修复了 SQL 错误，在执行 repair 之后重新 migrate。

4.7 undo (企业版特性)

作用：回滚最近执行的版本化迁移。

需要编写对应的 U 前缀迁移文件（如 U1.0.1__Drop_email_column.sql）。Flyway 会按版本号降序执行撤销迁移。

三、Liquibase

1. Liquibase 概述

1.1 定义与定位

Liquibase 是另一款重量级的开源数据库迁移工具，由 Datical 公司维护。与 Flyway 的极简 SQL 哲学不同，Liquibase 提供了一种基于抽象描述（Changelog）的、多格式支持的数据库变更管理方案。

如果用一个比喻：

  • Flyway 像是 Git，简单直接：一个版本对应一个文件，按顺序执行。
  • Liquibase 更像是声明式的 Kubernetes YAML：你描述”数据库应该长什么样”的变更步骤，Liquibase 负责翻译和执行。

1.2 核心设计理念

  1. 抽象 DSL：不直接写 SQL，而是通过 XML / YAML / JSON 描述变更意图，Liquibase 自动翻译为目标数据库的 SQL 方言。
  2. 跨数据库兼容：同一套 ChangeLog 可以（理论上）在 MySQL、PostgreSQL、Oracle 等多种数据库上执行。
  3. 回滚内建：每个 Changeset 都可以声明对应的 rollback 语句，Liquibase 原生支持回滚操作。
  4. 上下文 (Context) 与标签 (Labels)：精细控制哪些迁移应在哪些环境执行。

2. 核心概念

2.1 ChangeLog（变更日志）

ChangeLog 是 Liquibase 的根配置文件，可以理解为迁移的”目录”。它引用一个或多个 Changeset 文件。

2.2 Changeset（变更集）

Changeset 是 Liquibase 的最小迁移单元。每个 Changeset 由 id + author + 文件路径 三者联合唯一标识。

2.3 Changeset 的标识与不可变性

每个 Changeset 的唯一标识由以下三者组合：

  1. id — Changeset 的 ID
  2. author — 作者
  3. filePath — Changeset 所在的文件路径（自动确定）

三元组确定后，一旦 Changeset 被执行，它的 id、author 和文件路径都不应该再改变，内容也不能被修改。Liquibase 通过 checksum（MD5）来检测 Changeset 是否被篡改。

如果你需要修改一个已应用的迁移，正确做法是：

  • 编写一个新的 Changeset 来补救
  • 或者使用 clearCheckSums 命令重置校验和（谨慎！）

2.4 DATABASECHANGELOG 表

与 Flyway 的 flyway_schema_history 表类似，Liquibase 使用 DATABASECHANGELOG 表追踪迁移历史：

列名 说明
ID Changeset 的 id
AUTHOR Changeset 的 author
FILENAME Changeset 所在文件的路径
DATEEXECUTED 执行时间
ORDEREXECUTED 执行顺序
EXECTYPE 执行类型：EXECUTED、MARK_RAN、RERAN 等
MD5SUM Changeset 内容的校验和
DESCRIPTION 变更摘要
COMMENTS 注释
TAG 标签（用于回滚目标定位）
DEPLOYMENT_ID 部署 ID

3. 四大格式对比

Liquibase 支持四种格式来编写 Changeset定义：

3.1 XML 格式（最成熟、最常用）

优点：IDE 支持最佳（有 Schema 约束），功能最完整，文档最丰富。

缺点：啰嗦，写起来比较繁琐。

3.2 YAML 格式（简洁，可读性好）

优点：比 XML 简洁很多，可读性好。

缺点：IDE 支持不如 XML。

3.3 JSON 格式

优点：适合 JSON 工具链（如前端项目、Node.js 项目）。

缺点：冗长，很少在 Java 生态中使用。

3.4 格式化 SQL（最接近 Flyway 风格）

优点：SQL 开发者最熟悉，简单场景最直接。

值 行为

3. 实际场景选型建议

3.1 选 Flyway 的情况

  • 团队高度以 SQL 为中心，DBA 需要直接审查和编写迁移
  • 追求极简，不想引入复杂的概念和配置
  • 大多数时候不需要回滚迁移（或者通过”正向迁移修复错误”的方式来补救）
  • 使用 Spring Boot 生态，享受开箱即用的集成体验
  • 数据库类型单一（只使用 MySQL 或 PostgreSQL），不需要跨数据库兼容

3.2 选 Liquibase 的情况

  • 需要跨多种数据库类型部署（一套 ChangeLog 适配 MySQL + Oracle + PostgreSQL）
  • 需要社区版就能用的回滚功能
  • 需要 Preconditions 来做执行前的安全检查
  • 需要在不同环境执行不同的迁移（Context / Labels 精细控制）
  • 已经有一定规模的团队，愿意承受学习成本换取灵活性
  • 需要 Diff 功能来比较两个数据库的差异

3.3 一句话总结

Flyway 用 SQL 的简单换取了很多高级功能（需要企业版）；Liquibase 用学习的复杂度换取了开源版的功能完整性。

  • Flyway 在 Spring Boot 生态中占据主导地位（Spring Boot 默认集成方案）
  • Liquibase 在大型企业和需要多数据库兼容的场景中仍有稳固的市场份额
  • 如果从零开始选型且没有特殊需求，Flyway 是更安全的选择

1.1 回滚的三种哲学

哲学 A：”正向修复”（Flyway 社区版的默认立场）

不做回滚。如果迁移有问题，编写一个新的迁移来修复：

优点：最简单，不需要额外的回滚工具。所有迁移都是”正向的”，变更历史线性清晰。

缺点：如果 V1.0.0 还未广泛部署，在其他环境可能增加不必要的”修正”步骤。

哲学 B：”声明式回滚”（Liquibase 的方式）

每个迁移先编写正向变更，同时声明回滚逻辑。出问题时一键回滚。

优点：操作简便，自动化程度高。

缺点：回滚脚本本身也需要测试，增加了开发负担。有些变更在物理上无法安全回滚（如删除了某列中的数据）。

哲学 C：”混合策略”（最常见的企业实践）

大多数错误通过正向修复解决。只有少数可预测的高风险变更（如大表重建、核心业务逻辑相关的数据变更）才准备回滚方案。

一、大语言模型（LLM）基本概念

1. 什么是 LLM

大语言模型（Large Language Model，简称 LLM）是一种基于 Transformer 架构（一种利用自注意力机制处理序列数据的深度学习架构），在海量文本数据上训练得到的神经网络模型。它的核心能力是根据给定的上文，预测下一个最可能出现的 Token，这个看似简单的”下一个词预测”任务，在大规模数据和参数的加持下，涌现出了理解、推理、生成等复杂能力。

LLM 的发展经历了几个关键阶段：

  • 2017 年：Google 发布论文《Attention Is All You Need》，提出 Transformer 架构，奠定了后续所有大模型的技术基础
  • 2018-2019 年：BERT（Google）、GPT-1/GPT-2（OpenAI）相继发布，验证了预训练 + 微调（Pre-training + Fine-tuning）范式的有效性
  • 2020-2022 年：GPT-3 展示了惊人的 Few-shot 学习能力，模型参数规模突破千亿
  • 2022 年底：ChatGPT 发布，将 LLM 从实验室带入了大众视野，也开启了 AI 应用开发的浪潮
  • 2023-2025 年：开源模型（Llama、Qwen、DeepSeek 等）快速追赶闭源模型，多模态能力、Agent 能力、长上下文成为竞争焦点

对于后端工程师而言，不需要深入理解 Transformer 的数学细节，但需要理解 LLM 的几个核心概念，因为它们直接影响 API 调用方式、成本控制和系统设计。

2. 主流模型简介

当前市场上的主流 LLM 可以分为闭源和开源两大阵营：

闭源模型（通过 API 调用）：

模型 提供商 特点
GPT-4o / GPT-4.1 OpenAI 综合能力最强，多模态原生支持，生态最完善
Claude Opus 4 / Sonnet 4 Anthropic 长上下文能力突出（200K），安全性设计领先，代码与推理能力强
Gemini 2.5 Pro Google 原生多模态，上下文窗口可达 2M Token，与 Google 生态深度整合

开源模型（可自部署或通过 API 调用）：

模型 提供商 特点
DeepSeek-V3 / R1 深度求索 性价比极高，R1 的推理链（Chain of Thought）能力在开源模型中领先
Qwen 3 / Qwen-Max 阿里云（通义千问） 中英文能力强，生态完善，提供多种尺寸满足不同场景
GLM-4 系列 智谱 AI 国内最早的 LLM 团队之一，中文理解能力扎实
Llama 4 Meta 开源社区的基石，生态最丰富，适合二次开发和微调

选型建议：生产环境中通常不会只用一个模型。一个务实的策略是——主力模型选一个能力强且稳定的（如 GPT-4o 或 Claude Sonnet 4），同时接入一个高性价比模型（如 DeepSeek-V3）处理简单任务，再备一个开源模型（如 Qwen）用于敏感数据场景的私有化部署。

3. Token

Token（词元）是 LLM 处理文本的最小语义单位。它不是字符也不是单词，而是介于两者之间的”子词”（Subword）：

  • 英文中，一个常见单词如 “hello” 通常是 1 个 Token，而 “indescribable” 可能是 3-4 个 Token（in + describ + able）
  • 中文中，一个汉字通常是 1-2 个 Token。例如”你好”约 2 个 Token，”人工智能”约 3-4 个 Token
  • 标点符号、空格、特殊字符也各占 Token

Token 之所以重要，是因为大模型的计费、性能限制和上下文容量全部以 Token 为单位：

  • 计费：API 按 Token 数量收费，分为 Input Token（输入）和 Output Token（输出），通常 Output Token 价格是 Input Token 的 2-5 倍
  • 上下文限制：每个模型有最大 Context Window，即单次能处理的 Token 总数上限
  • 延迟：Token 生成速度（Tokens Per Second, TPS）决定了用户等待时间

实际项目中，预估 Token 消耗的简单公式：英文文本字符数 ÷ 4 ≈ Token 数，中文文本字符数 ÷ 1.5 ≈ Token 数。更准确的计数可以使用模型提供商的 Tokenizer 工具（如 OpenAI 的 tiktoken 库）或在线工具。

4. Context Window

Context Window（上下文窗口）是指模型单次推理能处理的 Token 总数上限——包括输入 Prompt 和模型生成的 Output 之和。

主流模型的 Context Window 大致范围：

  • 128K Token：GPT-4o、Claude Sonnet 4、DeepSeek-V3、Qwen 3 的标准窗口。足够容纳一本约 300 页的英文小说
  • 200K Token：Claude Opus 4 等高端模型
  • 1M-2M Token：Gemini 2.5 Pro（2M）、Qwen 3 某些配置（1M）。理论上可以一次处理整部《三体》三部曲

Context Window 对应用设计的直接影响：

  • 超窗口内容会被截断：输入超过上下文窗口时，模型提供商通常会截断最早的部分（或拒绝请求）。你需要在前端截断，而不是等模型报错
  • 长对话会累积成本：多轮对话中，每次请求都要将历史 Messages 全部带上。对话越长，每次调用的 Input Token 越多，成本越高。这就需要对话摘要（Summarization）或滑动窗口等策略来管理上下文
  • “大海捞针”是真实挑战：窗口大不代表模型能有效利用全部内容。大窗口模型在长文本中间位置的信息提取准确率（Needle in a Haystack 测试）各有差异，选型时需关注

5. 采样参数

LLM 在生成每个 Token 时，实际上是计算词汇表中每个可能的 Token 的概率分布，然后从中采样。以下三个参数控制了这个采样过程：

Temperature（温度，范围 0-2）：

控制输出的随机性和创造性。

  • Temperature = 0 或接近 0：模型几乎总是选择概率最高的 Token，输出高度确定、一致。适用于代码生成、数据提取、分类等需要稳定性的任务
  • Temperature ≈ 0.7-1.0：适度的随机性，适用于通用对话和内容生成
  • Temperature > 1.0：输出更多样化但也更不可控，容易出现离题和幻觉。适用于创意写作、头脑风暴

需要注意的是，Temperature = 0 并不意味着输出完全确定——模型提供商的后端差异、浮点运算精度等因素仍可能导致微小变化。

Top-P（核采样，Nucleus Sampling，范围 0-1）：

从累积概率超过 Top-P 的最小 Token 集合中采样。例如 Top-P = 0.9 表示模型只从概率最高的、累积概率达 90% 的 Token 中采样，忽略长尾的低概率 Token。

实践中的建议：同时调整 Temperature 和 Top-P 通常没有必要且容易混乱，选择其一即可。Temperature 更直观常用；Top-P 在需要动态调整输出多样性时更灵活（如根据输入复杂度自动适配）。

Top-K：

只从概率最高的 K 个 Token 中采样。Top-K 在早期模型中常用，现在的模型多使用 Top-P 替代，因为它能根据概率分布自动调整候选集大小。

LLM 的 Chat API 采用结构化的消息列表（Messages）来组织对话。每条消息包含两个字段：Role（角色）和 Content（内容）。

三种核心角色：

System Message（系统消息）：

System Message 的设计原则：越简单越好，够用就行。冗长的 System Message 不仅消耗 Token，还可能让模型”过拟合”规则（Overfitting Rules），反而降低灵活性。实践中，一个高质量的 System Message 通常在 100-500 Token 之间。

User Message（用户消息）：

用户输入的实际问题和上下文。在多轮对话中，通过交替排列 User 和 Assistant 消息来构建对话历史。

Assistant Message（助手消息）：

模型的历史回复。在构造 API 请求时，你需要把之前模型生成的回复以 Assistant Message 的形式放入 Messages 列表，这样模型才能”记住”自己说过什么。

消息构造的关键规则：

  1. Messages 必须是 Role 交替排列，以 User 消息结束（不能以 Assistant 消息结尾）
  2. 历史消息越长，每次调用的成本越高——所有历史 Messages 都计入 Input Token
  3. Function Call 结果以特殊的 role: "tool" 消息形式插入（详见后续 Function Calling 章节）
  4. 不同模型对 System Message 的重视程度不同——开源模型（尤其微调版）对 System Message 的遵循度不如 GPT-4/Claude 等闭源模型高

四、流式调用（SSE）深度解析

1. SSE 协议原理

SSE（Server-Sent Events，服务端推送事件）是一种基于 HTTP 的服务端向客户端推送数据的技术。与 WebSocket 的全双工不同，SSE 是单向的——服务端推送，客户端只能接收。

SSE 之所以成为 LLM 流式输出的标准协议，原因有二：

  • 实现简单：服务端只需设置 Content-Type: text/event-stream，然后逐行输出 data: <json>\n\n 即可。不需要 WebSocket 的握手和帧协议
  • 浏览器原生支持：前端可以直接使用 EventSource API 消费 SSE，不需要额外库

SSE 数据格式：

每条消息以 data: 开头，以双换行 \n\n 结束。[DONE] 是 OpenAI 定义的特殊结束标记。

3. 流式调用对用户体验的影响

流式调用的价值需要从两个指标来理解：

  • TTFT（Time To First Token，首 Token 延迟）：从用户发送请求到收到第一个 Token 的时间。流式方式下这个值通常明显低于非流式的总耗时，用户感觉系统”立刻开始响应”
  • 总耗时（Total Latency）：所有 Token 生成完毕的时间。流式和非流式的总耗时基本相同

流式输出的心理效应：用户感知的延迟主要是 TTFT，而非总耗时。一个 200ms 首 Token + 10 秒总耗时的流式体验，远好于 10 秒后一次性返回的非流式体验。

非流式调用的适用场景：并非所有场景都需要流式。以下场景适合非流式：

  • 后端异步任务（如离线生成报告、批量处理）
  • 需要完整结果后再处理（如 JSON 解析、数据提取）
  • 输出很短（几十 Token），流式的边际收益不大

4. 流式场景下的错误处理

流式调用的错误处理比非流式更复杂，因为错误可能发生在流的中间（模型已经生成了部分内容）：

  • 连接中断：使用 retry() 操作符实现自动重连。注意：流式调用通常在生成内容后不支持断点续传，重连意味着重新发起完整请求
  • 超时处理：流式调用中，两次 Token 推送之间的间隔如果超过阈值（Idle Timeout），应触发超时。可以设置读超时（Read Timeout）和空闲超时（Idle Timeout）
  • 部分内容处理：流中断时，前端应展示已生成的部分内容，并提示用户”生成中断，请重试”
  • 背压（Backpressure）：当下游消费速度慢于模型生成速度时，框架应能缓存或丢弃中间结果。WebFlux 默认支持背压

五、多模型管理与容错策略

1. 多模型切换的典型场景

生产环境中使用多个模型不是过度设计，而是应对以下现实需求的必然选择：

成本优化：

简单任务（文本分类、关键词提取、格式转换）用小模型，复杂任务（多步推理、代码生成、长文档分析）用大模型。例如：

  • DeepSeek-V3 处理 80% 的日常调用（便宜，每百万 Token 约 2 元 RMB）
  • GPT-4o 处理 15% 的复杂推理（贵，每百万 Token 约 15 元 RMB）
  • Claude Opus 4 处理 5% 的高难度长文档分析

这种分层策略可以将总成本降低 60% 以上。

能力互补：

不同模型有各自的强项。例如：

  • GPT-4o 综合能力强，JSON 结构化输出最稳定
  • Claude 长文本理解和代码生成极其出色
  • DeepSeek-R1 推理链详细，适合需要展示推导过程的场景
  • Qwen 中文能力扎实，适合中文知识库问答

地域合规与数据安全：

敏感数据不能出境时，使用国内模型的 API（如通过阿里云百炼调用 Qwen）或私有化部署开源模型。非敏感数据可以调用海外模型利用其综合能力优势。

AI应用开发全景概论

一、AI 在后端开发中的角色定位

以 ChatGPT 在 2022 年底发布为起点，大语言模型（Large Language Model, LLM）在过去几年间以前所未有的速度渗透到软件工程的各个层面。LLM 是一种基于海量文本数据训练得到的深度学习模型，能够理解自然语言并生成类人文本。对于后端工程师而言，最重要的问题不是”AI 会不会取代我”，而是”我应该把 AI 放在系统的什么位置”。

1. AI 不是替代者，而是增强者

AI 并不替代后端。后端系统本质上解决的是数据的存储、流转、计算和权限控制问题，这些职责不会因为 AI 的出现而消失。真正发生变化的是后端能力的边界被大幅拓宽了——很多以前后端做不到或者做不好的事情，现在可以做到了。

举一个具体的例子：一个传统的工单系统，后端能做的是 CRUD（创建、查询、更新、删除工单），以及基于规则的状态流转。接入 AI 之后，后端可以额外提供的能力包括但不限于：

  • 智能分类：根据工单内容自动判断所属类别和优先级，背后是 LLM 的文本分类能力
  • 相似工单检索：从历史工单中找到与当前问题最相似的处理记录，背后是 Embedding 向量化与语义搜索
  • 自动摘要：将冗长的工单描述压缩为关键信息摘要，方便客服快速定位问题
  • 知识库问答：基于产品文档自动回答常见问题，背后是 RAG（Retrieval-Augmented Generation，检索增强生成）

这些能力不是替代后端逻辑，而是嵌入到后端业务流程中，成为新的业务能力层。

2. 从”业务实现者”到”AI 能力编排者”

传统 Java 后端工程师的核心工作是：理解需求 → 设计数据模型 → 编写 Service/DAO 层代码 → 暴露 REST API。这是一条确定的、可预测的工作链路。

AI 时代，后端工程师的角色发生了微妙但重要的迁移：你不再只是自己实现业务逻辑，而是要学会编排 AI 来完成逻辑。这意味着你需要同时理解：

  • 业务本身（和以前一样重要）
  • AI 能做什么、不能做什么（能力边界判断）
  • 如何把业务问题分解为”AI 适合做的部分”和”传统代码适合做的部分”（任务分解与编排）
  • 如何在不可靠的 AI 输出之上构建可靠的系统（工程化保障）

这种角色的变化，可以类比为从”手工艺人”到”工厂产线设计师”的转变——你不是不干活了，而是把精力从亲自完成每一道工序，转向设计工序之间的流转规则与质量检验标准。

3. 后端工程师的独特优势

在 AI 应用开发领域，后端工程师拥有一些其他人不具备的优势：

  • 系统设计能力：你知道如何设计高可用、可扩展的系统，而 AI 应用的工程化落地最终依赖的正是这些能力
  • 数据理解：你清楚数据从哪里来、流经哪些系统、最终去哪里，而 AI 应用的核心就是对数据的加工和增强
  • 工程化经验：你熟悉 CI/CD、监控告警、灰度发布、容错降级——这些是 AI 应用从 Demo 走向生产的关键
  • 业务边界意识：你知道哪些操作需要权限校验、哪些数据不能外泄、哪些行为需要审计——这是 AI 安全的基础

换句话说，训练模型可能是算法工程师的强项，但把模型变成可靠的生产服务，是后端工程师的主场。

二、后端系统中 AI 的三种集成深度

AI 在后端系统中的集成不是”全有或全无”的二元选择，而是一个渐进深入的连续谱。根据模型在决策链路中的自主程度，可以划分为三个层级。理解这三个层级的关系，有助于你在实际项目中做出合适的架构决策。

1. Level 1：API 调用层

这是最低门槛的集成方式。后端像调用任何第三方 API 一样调用 LLM 的接口，把模型的返回结果当作一个”高级函数”的输出来使用。

典型架构：客户端 → 后端 Service → LLM API → 后端处理结果 → 返回客户端

适用场景：文本分类、情感分析、内容摘要、翻译、代码注释生成等输入确定、输出可直接消费的单次调用任务。

关键特征：模型在这里只是一个被动的计算单元，它不做任何决策，只是根据输入生成输出。后端代码完全掌控调用时机、参数和结果处理。

这种模式的技术门槛最低，但它存在两个明显局限：一是模型无法访问企业内部数据（它只能基于训练数据回答），二是模型无法执行任何实际操作（它只能说，不能做）。

2. Level 2：能力编排层

在这个层级，模型不再孤立工作，而是与外部知识库和业务系统连接起来。这主要通过两种机制实现：

  • RAG（检索增强生成，Retrieval-Augmented Generation）：在调用模型之前，先从向量数据库或搜索引擎中检索出与用户问题相关的文档片段，将这些片段作为上下文注入 Prompt，让模型基于”外部知识”而非仅凭”训练记忆”来生成回答
  • Function Calling（函数调用，也称 Tool Calling）：让模型输出结构化的函数调用请求，后端据此执行真实的业务操作（查数据库、调接口、发消息等），将执行结果反馈给模型继续对话

典型架构：客户端 → 后端 Service → 知识检索（向量数据库/搜索引擎） → 构造增强 Prompt → LLM API → 结果处理（可能触发 Function Call → 执行业务逻辑 → 结果回传 → 模型继续推理）→ 返回客户端

适用场景：智能客服、企业知识库问答、业务查询助手（”帮我查一下这个订单的状态”）、文档审查等需要外部知识 + 业务操作的场景。

关键特征：模型开始有了”眼睛”（能看外部知识）和”手”（能调用业务接口），但它仍然按照预定义的流程执行——先检索还是先调工具、检索哪些数据源、工具调用的顺序，这些都由后端代码预先编排好。模型在这个层级是”被赋能”的，但不是”自主”的。

Level 2 是目前企业级 AI 应用的主流深度，它兼顾了能力增强和可控制性，覆盖了约 80% 的实际业务场景。

3. Level 3：Agent 自主决策层

Agent（智能体）是当前 AI 应用开发的最高集成深度。在这个层级，模型不再按照预先编排的固定流程执行，而是自主规划任务步骤、自主选择调用哪些工具、自主判断任务是否完成。

典型架构：用户给定目标 → Agent 自主分解为子任务 → 自主选择工具执行每个子任务 → 观察执行结果 → 根据结果调整计划 → 循环直到目标达成 → 返回最终结果

关键特征：模型从”执行者”变为”决策者”。后端工程师的角色从编排每一步做什么，变为定义模型可用的工具集（Tools）、设定行为约束（System Prompt）、设置安全边界（Guardrails），然后让模型在这些约束内自主行动。

Agent 的典型循环模式是 ReAct（Reasoning + Acting，推理与行动交替）：模型先思考下一步应该做什么（Reasoning），然后调用工具执行（Acting），观察工具返回结果，再基于结果思考下一步……如此往复直至任务完成。

适用场景：自动化运维（”检查所有服务健康状态，对有问题的服务查看日志并尝试重启”）、复杂数据分析（”分析这个季度各区域的销售情况，找出异常点并给出建议”）、多步骤业务流程自动化。

Agent 的自主性既是它的优势也是风险来源：模型可能做出错误的工具选择、陷入循环、或被恶意 Prompt 诱导执行危险操作。因此 Level 3 对安全设计、可观测性和人工干预机制的要求远高于前两个层级。

4. 三种深度的关系与选型原则

三种深度不是互相替代的关系，而是层层叠加的关系。Level 2 包含 Level 1 的所有能力，Level 3 又包含 Level 2 的所有能力。

选型原则很简单：不是越深越好，够用就好。如果 Level 1 能解决问题就不要上 Level 2，Level 2 能覆盖就不要上 Agent。每上升一个层级，系统的复杂度、不可控性和调试难度都会显著增加。实践中，大多数有价值的 AI 应用都落在 Level 2。

三、AI 给后端架构带来的五个核心变化

AI 的集成不是简单地”加一个 API 调用”0，而是从底层改变了后端系统的若干基本假设。理解这些变化，是做好 AI 后端工程化的前提。

1. 从确定性到概率性

传统后端的核心逻辑是确定性的：给定相同的输入和相同的数据库状态，输出一定是相同的。单元测试、集成测试、回归测试都建立在这个前提之上。

AI 改变了这个假设。LLM 的输出本质上是概率性的——同一个 Prompt 调用两次，模型可能给出措辞不同甚至结论不同的回答。Temperature（温度参数，控制输出的随机性，值越低输出越确定、越高越有创造性）可以调节随机程度，但不能消除它。

这对后端工程意味着：

  • 测试策略需要改变：无法再用”期望输出等于某字符串”的方式写测试，需要引入语义相似度评判、结构化输出 Schema 校验、LLM-as-Judge（用模型评判模型输出）等新的测试方法
  • 容错设计需要加强：必须假设模型可能输出不合法 JSON、遗漏关键字段、或给出不合逻辑的结论，在代码层做校验和兜底
  • 监控需要新维度：除了传统 QPS 和延迟，还需要监控幻觉率、输出格式合规率、内容安全违规率等

2. 从纯代码到 Prompt + Code

传统后端业务逻辑全部用代码（Java/Go/Python 等）编写。AI 引入后，业务逻辑扩展为三层结构：

  • Prompt 层：用自然语言描述任务、设定规则、提供示例，本质是在”编程”模型的行为
  • 工具函数层：传统 Java 代码，实现模型无法完成的确定性操作（数据库查询、权限校验、数值计算等）
  • 编排代码层：Java 代码负责串联 Prompt 和工具函数，控制调用顺序、处理异常、管理状态

Prompt 从”一段随手写的文本”升级为需要工程化管理的一等公民：版本管理、A/B 测试、效果评测、回滚机制——这些传统代码享有的工程实践，Prompt 同样需要。

3. 从 REST 到多协议通信

传统后端系统的对外通信几乎全是 REST（Representational State Transfer，表现层状态转移）风格的 HTTP 请求/响应。AI 集成引入了两种重要的新通信模式：

  • SSE（Server-Sent Events，服务端推送事件）：用于 LLM 的流式输出。模型边生成边推送 Token，用户可以看到逐字输出的效果，大幅改善体验。Spring Boot 中通过 SseEmitter 或 WebFlux 的 Flux<ServerSentEvent> 实现
  • MCP（Model Context Protocol，模型上下文协议）：Anthropic 提出的标准化协议，用于模型与外部工具/数据源之间的通信。MCP 定义了统一的 Server-Client 架构，使得不同的 AI 客户端可以复用同一套工具实现

理解 REST 的工程师需要适应 SSE 的长连接语义和背压处理（Backpressure，当下游消费速度慢于上游生产速度时进行的流量控制），以及 MCP 的松耦合工具注册模型。

4. 从固定成本到 Token 经济

传统后端系统的成本主要由服务器、带宽和数据库组成，这些成本随请求量线性增长，且边际成本相对固定。

AI 应用引入了一个全新的成本维度：Token 消耗。Token 是 LLM 处理文本的基本单位（英文中大致相当于 3/4 个单词，中文中大致相当于 1-2 个汉字）。每次 LLM 调用都按 Token 数量计费，输入和输出分别计价，且不同模型的价格差异可达数十倍。

这对架构的影响是深远的：

  • 缓存的价值被重估：传统后端中缓存主要用于提升性能，在 AI 应用中，缓存还直接节省 Token 成本——两个用户问相似问题，如果命中语义缓存（Semantic Cache），就完全不需要调用模型
  • 模型分层成为刚需：简单任务用便宜的小模型（如 DeepSeek-V3、Qwen-Turbo），复杂任务用贵的大模型（如 GPT-4、Claude Opus），是成本优化的基本策略
  • Prompt 设计要精打细算：Prompt 中的每个 Token 都花在刀刃上。过多的 System Prompt 和 Few-shot 示例虽然可能改善效果，但也显著抬高了每次调用的成本

5. 从权限控制到意图安全

传统后端安全主要围绕认证（Authentication，验证你是谁）和授权（Authorization，验证你能做什么）展开，即 RBAC（Role-Based Access Control，基于角色的访问控制）。

AI 引入后，攻击面扩大了：

  • Prompt Injection（提示注入）：攻击者通过在用户输入中嵌入恶意指令，试图覆盖 System Prompt 中的行为约束。例如在客服系统中输入”Ignore all previous instructions and…”来劫持模型行为
  • 工具越权调用：在 Agent 模式下，模型自主选择调用哪些工具。如果工具的权限边界设计不当，模型可能被诱导调用它不应该调用的接口
  • 数据泄露：模型在生成回答时可能不经意地复述训练数据中的敏感信息，或 RAG 检索出了用户无权访问的文档内容

这意味着后端安全体系需要增加一个新的维度：意图安全（Intent Security）——不只是控制”谁”能访问”什么”，还要防御”恶意输入意图”对模型行为的操控。

四、后端 AI 应用的技术要素全景

AI 后端开发涉及的技术要素可以归纳为七个层次。理解每一层的职责和它们之间的依赖关系，有助于形成系统化的知识框架。

1. 模型层

模型层是 AI 应用的基础设施，负责提供语言理解和生成能力。后端工程师需要掌握的核心概念包括：

  • LLM（大语言模型）：在海量文本数据上训练得到的深度学习模型。主流模型包括 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列、以及国产的 DeepSeek（深度求索的高性价比模型）、Qwen（阿里通义千问）、GLM（智谱清言）等
  • Token（词元）：模型处理文本的最小单位。一个中文字通常对应 1-2 个 Token，一个英文单词通常对应 1-3 个 Token。Token 数量决定了模型调用的成本和延迟
  • Context Window（上下文窗口）：模型单次能处理的最大 Token 总数（输入 + 输出之和）。主流模型的上下文窗口从 128K 到 2M Token 不等，决定了能”塞进去”多少信息
  • Temperature（温度）：控制输出随机性的参数，范围 0-2，值越低输出越确定和保守，越高输出越有创造性但也更容易偏离主题

2. Prompt 层

Prompt（提示词）是人与模型之间的通信协议。它的核心任务是将业务需求翻译为模型能理解并正确执行的自然语言指令。

一个结构良好的 Prompt 通常包含以下要素：

  • 角色设定（Role/Persona）：定义模型扮演的角色，如”你是一个资深的 Java 后端工程师”
  • 任务描述（Task）：明确需要模型完成的具体任务
  • 上下文信息（Context）：任务相关的背景知识和数据
  • 约束条件（Constraints）：输出格式要求、内容限制、行为边界
  • 示例（Few-shot Examples）：通过一至多个输入-输出示例，让模型理解期望的行为模式

Prompt 的核心方法论包括 Zero-shot（不提供示例，直接给指令）、Few-shot（提供少量示例）、Chain of Thought（思维链，要求模型展示逐步推理过程）和 ReAct（推理与行动交替）。Prompt 工程的目标不是写出最复杂的 Prompt，而是找到用最小的 Token 开销达到稳定效果的 Prompt。

3. 知识层

LLM 的知识截止于其训练数据的时间点，且无法访问企业内部的私有数据。知识层通过引入外部知识来解决这两个问题：

  • Embedding（向量嵌入）：将文本映射为高维空间中的稠密向量。语义相近的文本，其向量在空间中的距离也相近。Embedding 是实现语义搜索的基础
  • 向量数据库（Vector Database）：专门存储和检索向量的数据库。核心能力是基于向量相似度（如余弦相似度、欧氏距离）快速找到 Top-K 个最相似的向量。常见方案包括 Milvus、Qdrant、Weaviate、Pinecone，以及传统数据库的向量扩展如 pgvector
  • RAG（检索增强生成）：在调用 LLM 之前，先检索出相关的外部知识片段，注入到 Prompt 的上下文中，让模型基于这些知识生成回答。RAG 的核心流程是：文档加载 → 文本切分（Chunking）→ 向量化 → 存储 → 检索 → 增强生成

4. 行动层

模型能”说”但不能”做”。行动层通过 Function Calling 机制让模型能够触发真实的业务操作：

  • Function Calling（函数调用）：在调用 LLM 时，不仅传 Prompt，还传一组可用函数的定义（名称、描述、参数 Schema）。模型在生成回答时，如果判断需要调用某个函数来获取信息或执行操作，它不直接执行函数，而是返回一个结构化的函数调用请求（包含函数名和参数）。后端代码解析这个请求、执行实际的业务函数、将结果回传给模型，模型再基于结果继续推理
  • Tool Calling（工具调用）：Function Calling 的广义延伸，不限于后端函数，也包括调用外部 API、操作数据库、读写文件等

Function Calling 是连接”AI 推理”和”业务执行”的桥梁，也是 Level 2 和 Level 3 集成的关键技术。

5. 编排层

当单个 LLM 调用不够用时（多步推理、多数据源、多工具协作），编排层介入来管理执行流程：

  • Workflow（工作流）：预先定义好的、确定性的执行流程。每一步做什么（调 LLM、查数据库、调工具、等待人工审批）由代码决定。Dify、Coze、LangGraph 等平台提供了可视化的流程编排能力
  • Agent（智能体）：模型自主决定执行流程。Agent 根据目标自行分解任务、选择工具、判断完成条件

两者的本质区别在于决策权在谁手上：Workflow 的决策权在工程师（确定性），Agent 的决策权在模型（自主性）。

6. 框架层

框架层提供统一的抽象来减少重复代码。Java 生态中两个主要的 AI 集成框架：

  • Spring AI：Spring 生态的官方 AI 扩展。提供 ChatClient（统一模型调用）、EmbeddingClient（统一向量化）、VectorStore（统一向量数据库访问）、@Tool 注解（声明工具函数）等开箱即用的能力，与 Spring Boot 深度整合
  • LangChain4j：LangChain 的 Java 实现，设计上更接近 LangChain/Python 的编程模型，提供 ChatLanguageModel、EmbeddingModel、AiServices、@Tool 等 API

7. 工程层

工程层解决 AI 应用从”能跑”到”能上线”的问题，包括安全防护、缓存策略、成本追踪、可观测性（Observability，通过 Log/Trace/Metrics 了解系统内部运行状态的能力）和评测体系。这一层是目前大多数 AI 应用从 Demo 阶段迈向生产环境的主要瓶颈。

五、典型的 AI 后端架构模式

将上述技术要素组合起来，实践中形成了几种可复用的架构模式。

1. 简单代理模式（Proxy Pattern）

这是最简单的集成模式。后端作为 LLM API 的代理层，封装 API Key、格式化请求、处理响应。后端在这里的核心价值是安全和控制：不暴露 API Key 给前端、记录调用日志、实施访问控制、对输入输出做内容审核。

适用场景：文本润色、翻译、摘要、代码注释生成等无需外部知识的单次任务。

2. RAG 知识库模式（RAG Pattern）

核心流程是一个完整的 ETL（Extract-Transform-Load，数据抽取-转换-加载）Pipeline：

  1. 离线阶段（索引构建）：文档上传 → 解析（PDF/Word/Markdown → 纯文本）→ 文本切分（按语义边界切为适当大小的 Chunk）→ Embedding 向量化 → 存入向量数据库
  2. 在线阶段（检索生成）：用户提问 → 问题向量化 → 向量数据库检索 Top-K 相关 Chunk → 将检索结果与用户问题一起构造 Prompt → LLM 生成回答

RAG 模式的关键设计决策包括：Chunk Size（文本块大小，决定检索粒度）、Overlap（相邻块之间的重叠量，防止语义断裂）、检索策略（纯向量 vs 混合检索 vs 多路召回重排序）。

适用场景：企业知识库问答、产品文档助手、合同/法规检索回答、智能客服。

3. 工具增强模式（Tool-Augmented Pattern）

与 RAG 模式提供”知识”不同，工具增强模式提供的是”行动能力”。模型不只是能回答问题，还能帮用户查订单、看库存、创建工单、发送通知。

关键设计要点：

  • 工具定义：每个工具需要清晰的名称、自然语言描述、参数 JSON Schema。描述的清晰度直接影响模型选择工具的准确率
  • 幂等与安全：查询类工具应幂等（多次调用结果一致）；写入类工具需要额外的权限校验和安全确认
  • 超时与降级：工具调用可能超时或失败，需要合理的超时设置和降级策略

适用场景：业务数据查询助手、智能运维助手、企业内部效率工具。

4. Agent 编排模式（Agent Orchestration Pattern）

Agent 模式将流程控制权从代码转移给模型。Agent 的典型执行循环如下：

  1. 思考（Thought）：分析当前状态，决定下一步应该做什么
  2. 行动（Action）：选择并调用合适的工具
  3. 观察（Observation）：获取工具执行结果
  4. 反思（Reflection）：评估是否达成目标，决定继续或结束

关键设计要点：

  • 工具集设计：Agent 可用的工具需要职责清晰、边界明确、描述准确。工具数量不宜过多（通常建议 5-15 个），过多会降低模型选择的准确率
  • 终止条件：Agent 可能陷入无效循环，需要设置最大步数、超时时间和成本上限
  • 人工介入点：对于高风险操作（如删除数据、发送通知），设置人工确认环节（Human-in-the-Loop）

适用场景：复杂数据分析、自动化运维、多步骤业务流程。

5. 混合模式（Hybrid Pattern）

实际的企业级 AI 应用中，很少只使用单一模式。混合模式将上述模式组合起来：

  • 简单问题走 Proxy 模式直接回答
  • 知识类问题走 RAG 模式检索文档
  • 操作类问题走 Tool-Augmented 模式调用后端服务
  • 复杂的多步骤任务走 Agent 模式自主编排

混合模式需要一个**路由层（Router）**来判断请求应该走哪条路径——这本身可以是一个轻量级 LLM 调用，也可以是基于规则的分类器。混合模式是构建企业 AI 中台（AI Middle Platform）的典型架构。

六、Java 在 AI 生态中的定位

1. 正视 Java 在 AI 领域的真实位置

AI 模型的训练、微调和研究的首选语言是 Python，这一点短期内不会改变。PyTorch、Transformers、vLLM 等核心库都是 Python 生态的产物。

但 AI 应用不等于 AI 模型训练。AI 工程化落地与企业级服务化的主力语言，Java 仍然占据重要位置。绝大多数企业的后端技术栈是 Java/Spring Boot，它们需要的不是用 Java 重新训练模型，而是把模型能力可靠地集成到现有系统中。

因此 Java 在 AI 生态的定位是清晰的：Python 负责造发动机（模型训练与推理），Java 负责造整车（将发动机封装为可驾驶的汽车，配上方向盘、刹车、仪表盘）。

2. Java 生态的核心优势

在 AI 应用开发场景下，Java 生态有几项关键优势：

  • Spring Boot 的成熟度：依赖注入、自动配置、Starter 机制、Actuator 监控等能力，使得 AI 集成框架（如 Spring AI）可以以与 Spring Data、Spring Security 一致的方式融入现有体系
  • 微服务治理：服务发现、负载均衡、熔断降级、分布式追踪等能力，对于需要调用多个模型和外部服务的复杂 AI 应用至关重要
  • 数据基础设施集成：Java 在关系型数据库（MySQL/PostgreSQL → MyBatis/JPA）、缓存（Redis → Spring Cache）、消息队列（Kafka/RabbitMQ → Spring Messaging）等方面的集成生态极为成熟——而 AI 应用恰好需要频繁地与这些基础设施交互
  • 运维与监控：Micrometer + Prometheus + Grafana 的技术栈可以平滑延伸覆盖 AI 应用的 Token 消耗、模型延迟等新指标

3. Java 调用 AI 的典型模式

Java 后端调用 AI 能力的典型模式不是”用 Java 跑模型”，而是 Java 作为 HTTP 客户端 + 编排层：

  • 通过 HTTP 调用 OpenAI 兼容 API（几乎所有国产模型都兼容 OpenAI 的 API 格式）
  • 通过 SSE 流式消费模型的输出
  • 通过 MCP 协议注册和管理工具
  • 与 Python 服务协作：Python 运行模型推理或 GraphRAG 这类特定服务，Java 通过 HTTP/gRPC 调用

Spring AI 和 LangChain4j 的目标正是为上述模式提供统一的、符合 Java 习惯的抽象层，让 Java 工程师不需要关心底层 HTTP 细节，用熟悉的 @Service、@Tool、ChatClient 等 API 即可集成 AI 能力。

4. 务实的技术选型态度

对于 Java 后端团队，务实的态度是：

  • 模型层面：不造轮子，直接使用成熟的模型 API 服务和托管平台
  • 框架层面：优先评估 Spring AI 或 LangChain4j，避免裸写 HTTP + JSON 解析
  • 特殊需求：需要 GraphRAG、需要模型微调等场景，部署 Python Sidecar 服务，不要用 Java 硬扛
  • 核心竞争力：把精力放在业务理解、架构设计、工程化保障上——这些才是后端工程师不可替代的价值

七、后端工程师需要建立的 AI 思维模型

人适应技术，而不是技术适应人。以下五个思维转变，是 Java 后端工程师在 AI 时代保持竞争力的关键。

1. Prompt 即代码

传统认知中，Prompt 是”写给 AI 看的一段话”，是临时的、一次性的。但在工程实践中，Prompt 应当被视为一种新的编程形式——用自然语言描述逻辑，由 LLM 执行。

这意味着 Prompt 需要享有与代码同等的工程待遇：

  • 版本管理：Prompt 修改应该走 Git，有 commit message 记录变更原因
  • A/B 测试：新旧 Prompt 并行运行，对比效果数据后决定是否切换
  • 效果评测：每次 Prompt 变更应该有对应的评测 Pipeline 自动跑分
  • 回滚机制：效果变差时应能快速回退到上一个 Prompt 版本
  • 代码审查：Prompt 变更也应该经过 Code Review，检查逻辑完整性、边界情况覆盖、Token 开销

2. 模型是第三方依赖

把模型看作一种特殊的第三方依赖，会帮助你建立正确的工程习惯：

  • 模型有版本号（如 gpt-4-0613、claude-sonnet-4-6），版本之间行为可能不同
  • 模型会 deprecate（废弃） 和 sunset（下线），需要关注模型提供商的版本生命周期通知
  • 模型有性能差异，两个同样号称”支持 128K 上下文”的模型，在长文本检索精度上可能差距显著
  • 模型需要 Fallback（故障转移）策略：主模型不可用时自动切换到备用模型，或降级为缓存响应

用管理 Maven 依赖的方式来管理模型：锁定版本、测试升级、监控变化、准备回退方案。

3. 非确定性是常态

在传统后端开发中，非确定性行为（Non-deterministic Behavior）是需要追查和修复的 Bug。在 AI 应用中，非确定性是系统设计的基线假设，不是异常。

如何在一个非确定性的组件之上构建可靠的系统？核心策略是在合适的层面引入确定性：

  • 结构化输出（Structured Output/JSON Mode）：约束模型输出合法 JSON，用 JSON Schema 校验字段完整性，将”非确定性的自然语言”转化为”确定性的结构化数据”
  • 校验与兜底：对模型输出进行规则校验（字段必填、格式合法、值域正确），校验失败时走降级逻辑（返回默认值 / 提示用户重试 / 切换备用模型）
  • 多层防御：不依赖单一 Prompt 调用来做关键决策。重要场景加入二次确认、人工审核、或并行调用多个模型交叉验证

4. Token 即成本

传统后端中的成本优化通常聚焦于服务器资源（CPU、内存、带宽）和数据库查询。AI 应用增加了一个必须持续关注的成本维度：Token 消耗。

在实践中，有效的 Token 成本控制策略包括：

  • 模型分层路由：简单任务用小模型（便宜），复杂任务用大模型（贵但能力强）。路由决策可以由规则驱动（关键词匹配）或由一个低成本的小模型来做
  • 语义缓存（Semantic Cache）：将历史问答记录 Embedding 后缓存。新问题先计算与缓存中问题的相似度，相似度超过阈值则直接返回缓存答案，完全不调用模型
  • 精确匹配缓存：对于固定的 System Prompt + 相同/高度相似的 User Input，直接返回缓存的模型响应
  • Prompt 瘦身：定期审查 Prompt 模板，删除无效的冗余描述、过长的 Few-shot 示例
  • 可观测性驱动优化：必须有 Token 消耗的实时看板，知道钱花在了哪里，才能做有针对性的优化

5. 安全边界外移

传统后端安全模型中，WAF（Web Application Firewall，Web应用防火墙）和 API Gateway 是第一道防线，负责拦截 SQL 注入、XSS 等攻击。但在 AI 应用中，用户输入直接进入 Prompt 上下文并影响模型行为，传统的攻击签名匹配无法识别 Prompt Injection。

安全防线需要从网关层外移到模型交互层：

  • 输入过滤：在构造 Prompt 之前，对用户输入做安全扫描和敏感词过滤
  • Prompt 沙箱：将用户输入放在 Prompt 的特定标记区域内（如 <user_input>...</user_input>），明确告诉模型”这部分是用户输入，不包含系统指令”
  • 工具白名单与权限最小化：Agent 可调用的工具应遵循最小权限原则（Least Privilege），工具实现中应包含独立的权限校验，不依赖模型判断
  • 输出审核：模型输出在返回给用户之前，经过内容安全审核和敏感信息脱敏
  • 全链路审计：记录每次 LLM 调用的完整输入输出，便于事后追溯和分析

一、Embedding 的基本概念

1. 什么是 Embedding

Embedding（向量嵌入）是将文本、图像、音频等非结构化数据映射为高维空间中的稠密向量（Dense Vector）的技术。所谓”稠密”，是指向量的每一维都是有意义的浮点数，共同编码了原始数据的语义信息。

用形象的方式来理解：想象语义空间是一张巨大的地图，Embedding 就是给每个文本在这张地图上分配一个坐标。语义相近的文本——如”苹果很好吃”和”这个水果味道不错”——会被分配到地图上相邻的位置；而语义无关的文本——如”苹果很好吃”和”如何配置 Nginx”——则在地图上相距很远。

Embedding 之所以是 AI 应用开发的核心基础设施，原因在于：计算机无法直接”理解”自然语言的语义，但它可以高效地计算向量之间的距离。Embedding 将”语义理解”这个人类擅长但计算机不擅长的问题，转化为了”向量相似度计算”这个计算机极其擅长的问题。

从向量到距离再到相似度的转换，是 Embedding 工程实践的起点，后续所有关于检索、推荐和聚类的讨论都建立在这个基础之上。

2. 为什么需要 Embedding

在没有 Embedding 的时代，文本搜索主要依赖关键词匹配（如 MySQL 的 LIKE、Elasticsearch 的倒排索引）。关键词匹配存在几个难以克服的问题：

  • 同义词问题：用户搜”笔记本”，无法匹配文档中的”笔记本电脑”
  • 语义理解缺失：用户问”怎么退款”，无法匹配标题为”退货流程说明”的文档
  • 多语言断裂：中文搜索”苹果”，无法匹配英文文档中的”Apple”

Embedding 解决了这些问题：它不是匹配关键词，而是匹配语义。向量空间中距离相近的文本，语义就是相近的——不管用了什么词、什么语言。

Embedding 与 LLM 的关系：LLM 是”生成答案”的大脑，Embedding 是”找到相关信息”的眼睛。在 RAG 架构中，Embedding 负责从海量文档中检索出最相关的片段，LLM 负责基于这些片段生成答案。两者各司其职。

二、文本向量化的原理与实践

1. Embedding 模型的工作原理

Embedding 模型是一种专门训练的神经网络，输入一段文本，输出一个固定长度的浮点数数组（向量）。这个向量就是文本在高维语义空间中的坐标。

主流 Embedding 模型：

模型 维度 特点
OpenAI text-embedding-3-small 512/1536 性价比高，支持动态维度缩减
OpenAI text-embedding-3-large 256/1024/3072 精度最高，支持维度缩减
Cohere Embed v3 1024 多语言效果好，支持分类和聚类场景

GTE-Qwen2（阿里） 1536/4096 基于 Qwen 架构，中文表现最佳之一

Embedding 模型与 LLM 的区别：

Embedding 模型 LLM（Chat 模型）
输入 文本 文本（Messages）
输出 固定长度向量 可变长度文本
用途 语义搜索、聚类、分类 对话、生成、推理
调用方式 单次请求/响应 单次或流式
成本 极低（约为 LLM 调用的 1/100） 较高

Embedding 模型的核心价值是低成本地将语义转化为可计算的向量——它是 RAG 检索和语义搜索的基础引擎，而不是对话能力本身。

2. 向量维度

Embedding 向量的维度（Dimensions）是关键的工程决策参数。维度越高，语义编码越精细，但存储和计算成本也越高。

维度的直接影响：

因素 低维度（256-512） 高维度（1536-3072）
检索精度 略低，但足够应对大多数场景 最高
存储空间 每向量 1-2KB 每向量 6-12KB
检索速度 更快（索引小，计算量小） 更慢
适用场景 简单语义匹配、大规模语料 高精度检索、专业领域
  • 存储规模估算：以 1536 维为例，一个向量占用 1536 × 4 字节（float32）= 6KB。100 万条文档的向量索引约 6GB。如果切分后每个文档产生 3-5 个 Chunk，内存占用还会成倍增长
  • 动态维度缩减：OpenAI 的 text-embedding-3 系列支持在 API 参数中指定 dimensions，例如将 3072 维缩减为 256 维，精度损失约 5%-10%，但存储和计算成本降低 90% 以上
  • 精度损失的非线性特征：维度从 3072 降到 1536 时，精度损失通常不到 2%；但降到 256 以下时，损失开始明显加速。1024-1536 维是大多数场景的”甜点区”

选型建议：

  • 小型知识库（< 10 万条）：直接使用 1536 维，无需优化
  • 中型知识库（10 万-100 万条）：使用 1024 维，或使用 1536 维 + 向量索引压缩
  • 大型知识库（> 100 万条）：使用 256-768 维 + 多阶段检索（粗筛+精排）

实际项目中，初始化知识库时需要向量化大量文档（可能上万条）。逐条调用 Embedding API 会导致请求数爆炸和延迟过高。正确的做法是使用批量 API：

  • 分组之间使用并发线程池并行发起多个批量请求，进一步加速

三、文本切分策略

在 RAG 应用中，文档通常很长（几页到几百页），而 Embedding 模型有输入长度限制（通常 512-8192 Token），且检索粒度越细结果越精准。因此需要将长文档切分为适当大小的文本块（Chunk）。

文本切分，或称 Chunking，是 RAG 系统设计和运维中最重要的预处理步骤之一。切分方式直接决定了检索召回的质量——切得太粗（Chunk 太大），检索精度下降且浪费 Token；切得太细（Chunk 太小），语义断裂、上下文丢失。

1. 文本切分的核心参数

Chunk Size（文本块大小）：

单个 Chunk 的 Token 数量上限。选择 Chunk Size 时需要在语义完整性和检索精度之间找到平衡：

  • 过大（> 2000 Token）：一个 Chunk 包含太多信息，检索时不够精准。而且大 Chunk 占用更多 Context Window 空间，减少了可注入的 Chunk 数量
  • 过小（< 100 Token）：语义不够完整。一个完整的段落可能被切成两半，导致下半部分失去上下文
  • 推荐范围：对于大多数中文文档，256-1024 Token 是合理的区间。问答类应用推荐 256-512 Token；长文档分析类推荐 512-1024 Token

Overlap（重叠量）：

相邻两个 Chunk 之间重叠的 Token 数量。Overlap 的目的是防止关键信息恰好出现在 Chunk 的边界处被切断。

  • 过小：边界信息丢失风险增加。一个句子被切在两段之间，两段都无法完整表达句意
  • 过大：存储冗余增加，且检索时可能返回内容高度重复的多个 Chunk
  • 推荐范围：通常设为 Chunk Size 的 10%-20%。如 Chunk Size = 512 Token，Overlap 设为 50-100 Token

Metadata（元数据）：

每个 Chunk 携带的结构化附加信息，用于检索时的过滤和排序。Metadata 本身不参与向量化，但存储在向量数据库中，用于检索后的过滤。常见的 Metadata 包括：

  • 来源文档 ID 和标题
  • Chunk 在原文中的页码或段落号
  • 文档类型（FAQ / 手册 / 合同 / 研报）
  • 创建时间和更新时间
  • 权限标签（哪些用户组可以检索到此 Chunk）

Metadata 的设计原则：在入库时多存，在检索时按需过滤。Metadata 不参与 Embedding 计算，存储成本极低，但能极大增强检索的灵活性和安全性。

2. 文本切分策略

文本切分有许多种策略，选择取决于文档类型、语义粒度和检索需求。以下是工程实践中最常用的几种策略及其适用条件。

2.1 基于字符的切分（Character Splitter）

最简单的切分方式，按字符数或 Token 数将文本切分为等长的块。

适用场景：格式规整的文档、初步验证时的快速切分、对检索精度要求不高的场景。

缺点：可能在句子中间切断，导致语义不完整。

2.2 基于分隔符的递归切分（Recursive Character Splitter）

这是目前最推荐、使用最广泛的切分策略。它会按照预设的分隔符优先级递归地切分文本：先尝试按段落（\n\n）切分，如果段落仍然太长，再按句子（。、！、？）切分，最后按从句或短语分隔符（，、；）切分。

适用场景：几乎所有文本类型，是 RAG 应用的默认首选策略。

优势：最大程度保留语义边界，尊重文档的自然结构。

2.3 基于语义的切分（Semantic Splitter）

使用 Embedding 模型计算相邻句子之间的语义相似度。当相似度显著下降时（即”话题切换”），在此处切分。语义切分的核心思路是让每个 Chunk 对应一个相对独立的话题或子主题。

适用场景：长文档、叙事性文本、会议记录等话题边界明显的文档。

代价：需要额外的 Embedding 计算，构建索引的时间是前两种方法的 2-3 倍。

2.4 基于文档结构的切分（Document Structure Splitter）

根据文档自身的结构标记（如 Markdown 的 # ## ### 标题层级、HTML 标签、PDF 的章节标记）进行切分。这种策略以文档的结构为切分边界，确保每个 Chunk 是文档中的一个逻辑单元。

适用场景：结构化良好的技术文档、API 文档、Markdown 文件。

优势：切分结果与文档的层级结构一致，检索结果更易于定位和追溯原文。

3. 多粒度索引策略

单一粒度的 Chunk 难以同时满足”精准检索”和”完整上下文”两个需求。多粒度索引是解决这个问题的工程方案：

  • 小 Chunk（128-256 Token）：用于向量检索，提供精准的匹配
  • 父文档（Parent Document）：小 Chunk 的来源文档或大段落，检索命中小 Chunk 后，将其父文档一起注入 LLM。也可以选择检索命中小 Chunk 后，向前后各扩展 N 个 Chunk（Context Window Expansion），作为上下文一起注入
  • 摘要 Chunk：对整个文档或章节生成摘要并向量化。全局性问题（”这份文档讲了什么？”）命中摘要 Chunk 而非碎片化的小 Chunk

这种策略在 LangChain 中称为 Parent Document Retriever，在 LlamaIndex 中称为 Hierarchical Node Parser。但策略本身与框架无关——核心思想是用不同粒度的索引处理不同粒度的查询。

四、向量相似度计算

向量化的目的不是为了”存储向量”，而是为了计算向量之间的相似度（距离）。相似度计算是语义搜索的数学基础。

1. 三种主要的相似度度量

余弦相似度（Cosine Similarity）：

衡量两个向量之间夹角的余弦值。这是 Embedding 场景中最常用的相似度度量。取值范围 [-1, 1]，1 表示方向完全一致（语义最接近），0 表示正交（无关），-1 表示方向完全相反。

余弦相似度之所以最常用于 Embedding 比较，是因为它忽略向量的长度（模），只关注方向。在语义空间中，向量的方向编码了语义内容，长度编码了置信度或信息量。只比较方向意味着不管文本长短，只要语义相近就能被检索到。

欧氏距离（Euclidean Distance）：

两个向量在多维空间中的直线距离。距离越小，语义越相近。欧氏距离受向量长度影响，推荐用于已做归一化处理的向量。

点积（Dot Product）：

两个向量对应维度乘积之和。当向量已归一化（模为 1）时，点积等价于余弦相似度。部分向量数据库（如 Pinecone、Qdrant）默认使用点积作为相似度度量以提升计算效率。

2. 相似度度量选型

度量方式 适用场景 说明
余弦相似度 通用语义搜索（默认首选） 不受文本长度影响，只关注语义方向
欧氏距离 向量已归一化、需要严格的距离度量 对向量长度敏感
点积 向量已归一化、追求极致性能 计算效率最高

实践建议：绝大多数场景下使用余弦相似度，它的语义区分能力最稳定。如果向量数据库只支持点积，确保 Embedding 向量已归一化即可等价替代。不要在这个环节过度优化——相似度度量之间的差异，远小于 Chunk 策略和 Embedding 模型选择带来的差异。

一、RAG 的基本概念与定位

1. 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与文本生成结合的 AI 应用架构模式。它的核心思想是：在 LLM 生成答案之前，先从外部知识库中检索出与用户问题最相关的文档片段，然后将这些片段作为”参考资料”注入 Prompt，让 LLM 基于这些资料生成答案。

RAG 的本质是给 LLM 装上”搜索引擎”——LLM 不再仅依赖训练时记忆的知识，而是可以实时查询外部知识库，获取最新的、领域特定的、可信的信息。

RAG 的运行流程可以用一句话概括：用户提问 → 向量化查询 → 检索相关文档 → 组装增强 Prompt → LLM 基于资料生成答案。

RAG 不是某个特定的库或产品，而是一套架构范式。只要是”检索外部知识 → 注入 Prompt → 交给 LLM 生成”的模式，无论用什么工具实现，都属于 RAG 的范畴。

2. RAG 解决的核心问题：LLM 的三大局限

RAG 之所以成为 AI 应用开发中最基础的架构模式之一，是因为它直接解决了 LLM 的三个固有局限：

局限一：知识截止日期（Knowledge Cutoff）

LLM 的知识来自训练数据，训练完成后其知识停留在训练截止日期。对于截止日期之后的信息（如新产品发布、政策变更、最新技术动态），LLM 完全无知。RAG 通过检索外部实时更新的知识库，让 LLM 可以回答关于最新信息的问题。

局限二：幻觉（Hallucination）

当 LLM 被问到它不知道的信息时，它倾向于”编造”听起来合理但实际错误的答案——这就是幻觉。幻觉之所以危险，不是因为它明显错误（那样用户会察觉），而是因为它编造的答案往往听起来非常可信。RAG 通过将答案限定在检索到的真实文档范围内（in-context learning），大幅降低了幻觉的发生概率——LLM 被要求”基于资料回答”，而不是”凭记忆回答”。

局限三：领域知识缺失

通用 LLM 的训练语料以互联网公开内容为主，对于企业内部的私有知识（代码仓库、设计文档、运维手册、业务规范）完全不了解。RAG 通过检索企业内部知识库，让 LLM 能够回答需要私有领域知识的问题。

LLM 局限 表现 RAG 如何解决
知识截止日期 不知道最新信息 检索实时更新的外部知识库
幻觉 编造看似合理但错误的答案 基于检索到的真实文档回答，限定答案边界
领域知识缺失 不了解企业内部信息 检索企业内部私有知识库

RAG 的附加价值：

  • 可溯源（Attribution）：RAG 生成的答案可以附带引用来源（”这个答案来自《API 网关运维手册》第 3 章”），用户可以自行验证
  • 知识热更新：修改外部知识库后，RAG 的回答立即生效——不需要重新训练模型
  • 权限控制：不同用户检索不同权限的知识库，可以实现知识隔离

3. RAG vs 模型微调：两种知识注入范式

RAG 和模型微调（Fine-tuning）是给 LLM 注入外部知识的两种主要方式，它们不是互斥的，而是互补的。

对比维度 RAG 模型微调

知识容量 几乎无限（受限于向量数据库容量） 有限（受限于模型容量）

决策框架：

在绝大多数企业 AI 应用中，RAG 是首选的知识注入方式，微调是进阶优化手段。原因很简单：企业在绝大多数情况下面临的是”怎么让模型知道我的最新文档”，而不是”怎么让模型学会一个全新的推理范式”。

二、RAG 的核心流程

RAG 系统包含两个阶段的处理管线：离线索引阶段（Offline Ingestion Pipeline）和在线检索生成阶段（Online Retrieval & Generation Pipeline）。

1. 离线阶段：文档索引管线

离线阶段的任务是将原始文档（PDF、Word、Markdown、网页等）转化为可供向量检索的索引数据。这个过程通常只在文档入库时执行一次，或在文档更新时增量执行。

1.1 文档加载（Document Loading）

文档加载是将各种格式的原始文件读入系统，转换为统一的文本表示。不同的文件格式需要不同的加载器（Reader）。

文档加载的工程实践：

  • 统一文档模型：定义统一的 Document 对象，包含 content（文本内容）和 metadata（元数据，如来源文件路径、页码、作者、修改时间）
  • 格式白名单：只加载支持的格式，对不支持的格式给出明确错误提示
  • 文件大小限制：设置单文件大小上限（如 50MB），避免过大的文件（如扫描件、图片 PDF）阻塞处理管线

1.2 文档解析（Document Parsing）

文档解析是将加载后的文本进一步清洗和结构化——提取有意义的文本内容，去除噪声。

解析的核心挑战不是”读文件”，而是”提取出真正有用的文本”。

  • PDF 解析的难点：多栏布局导致文本顺序错乱；表格数据的结构化提取；扫描件需要 OCR；页眉页脚和页码的去除；嵌入图片中文字的提取
  • HTML 解析的难点：导航栏、广告、推荐列表等非正文内容的去除（Boilerplate Removal）；<table>、<code> 等语义标签的正确处理
  • 代码文件的解析：保留包路径、类名、方法名等结构化信息作为元数据；Javadoc / 注释的提取

1.3 文本切分（Chunking）

文本切分已在 04-Embedding与向量化.md 中详细讨论。对于 RAG 来说，文本切分是决定检索质量的关键预处理步骤。

RAG 场景下文本切分的特殊考量：

  • Chunk Size 与 LLM Context Window 的关系：Chunk Size 不能只看检索精度，还要考虑返回的 Chunk 在 LLM Context Window 中的占比。如果每个 Chunk 太大（如 4096 Token），返回 5 个就占用了 20000+ Token，留给 LLM 生成回答的空间很少
  • 语义完整性优先于大小均匀：宁可大小不均匀，也要保证每个 Chunk 的语义完整。一个好的切分器会在”自然断点”（段落边界、章节边界）处切分，而不是机械地按字符数切分
  • Overlap 在 RAG 中的实际价值：Overlap 不仅防止”边界截断”，更重要的是保证当一个答案需要的信息恰好分布在两个 Chunk 之间时，至少有一个 Chunk 能完整包含该信息。这在技术文档中特别重要——一个配置步骤可能跨越两个段落，Overlap 确保有一个 Chunk 同时包含两个段落的完整内容

长文章 / 研究报告 1024-2048 Token 100-200 Token

向量化是将切分后的文本 Chunk 转换为向量的过程。详见 04-Embedding与向量化.md。

在 RAG 的离线索引阶段，向量化的工程要点是：

  • 批量处理：逐条调用 Embedding API 会导致大量 HTTP 往返。使用批量 API，每次发送 100-500 条文本
  • 失败重试与幂等性：Embedding API 调用可能因网络抖动、限流等原因失败。每条文本使用唯一 ID，支持重试而不产生重复向量
  • 成本估算：每条 Chunk 的 Embedding 成本约为 LLM 调用的 1/100 - 1/200，但大规模索引时总量不容忽视。100 万条 Chunk × $0.02/1M tokens × 500 tokens/Chunk ≈ $10 USD
  • 模型一致性：索引阶段和检索阶段必须使用相同的 Embedding 模型。索引用 OpenAI text-embedding-3-small 而检索用 BGE-M3，会导致语义空间不一致，检索效果极差

1.5 向量存储（Vector Store）

向量化后的数据写入向量数据库。详见 05-向量数据库.md。

在 RAG 的索引阶段，向量存储的要点是：

  • 元数据保留：每个 Chunk 必须保留充分的元数据——来源文档名、页码/章节、文档分类、创建时间、Chunk 编号。这些元数据在检索时用于过滤，在生成答案时用于溯源引用
  • 索引类型选择：RAG 场景推荐 HNSW 索引，检索速度快、召回率高
  • Collection 隔离策略：不同知识域的数据放入不同 Collection（如 product_docs、internal_wiki、troubleshooting），检索时可以按需指定 Collection

2. 在线阶段：检索与生成

在线阶段是 RAG 系统在运行时的执行流程——用户发起提问，系统完成检索和生成。

2.1 查询向量化

用户的自然语言问题需要转换为向量，才能与向量数据库中的文档向量进行相似度匹配。

直接向量化 vs 查询预处理：

最简单的做法是将用户原始问题直接向量化。但在实际应用中，用户的问题往往不适合直接做检索查询——可能包含口语化表达、指代词、信息冗余或不足。因此，在向量化之前对查询进行预处理（Query Rewrite、查询扩展）是提升 RAG 检索质量的关键优化，详见第三章。

查询向量化的工程要点：

  • 延迟敏感：查询向量化是同步操作，直接影响用户体验。使用低延迟的 Embedding 模型（如 text-embedding-3-small 通常在 50-200ms 内完成）
  • 缓存：对于高频查询问题（如常见 FAQ），缓存查询向量和检索结果，避免重复计算和检索
  • 用户问题长度：过长的用户问题可能包含噪声信息，影响向量化质量。在向量化前做轻量清洗——截断过长问题、去除礼貌用语等

2.2 相似性检索

查询向量传入向量数据库，执行 ANN 检索，返回 TopK 个语义最相似的文档 Chunk。

检索参数设计：

参数 推荐值 理由
TopK 3-10 太少可能遗漏关键信息，太多引入噪声
相似度阈值 0.65-0.75 过滤掉语义不相关的结果
标量过滤 按需 按分类、时间、权限等维度缩小检索范围

检索结果的后处理：

  • 去重：高度相似的 Chunk 可能返回多条，去重后保留最相关的一条
  • 排序调整：按相似度降序排列，将最相关的结果放在 Prompt 最前面（LLM 对 Prompt 中靠前的内容关注度更高）
  • 上下文拼接：如果返回的 Chunk 在原文中是连续的（如 Chunk 3 和 Chunk 4 来自同一文档且相邻），将它们拼接为更大的上下文片段

2.3 Prompt 组装

检索结果需要被组织成一个结构化的 Prompt，才能注入 LLM。这是 RAG 中连接”检索”和”生成”的桥梁，Prompt 的组装质量直接影响最终答案的质量。

增强 Prompt 的典型结构：

Prompt 组装的关键原则：

  • 明确信息来源：每个 Chunk 附带来源标签（文档名、章节），让 LLM 在回答中能够引用
  • 边界约束：显式告诉 LLM “如果资料中没有相关信息，请说明而不是编造”。这降低了 LLM 在检索结果不相关时强行作答的概率
  • 相关性排序：将相似度最高的 Chunk 放在最前面——LLM 对 Prompt 中位置靠前的内容赋予更高的注意力权重
  • 不要过度注入：Chunk 数量不是越多越好，注入过多文档会稀释最相关信息的注意力。通常在 3-10 个 Chunk 之间

2.4 LLM 生成

LLM 基于增强后的 Prompt 生成最终答案。这个阶段的核心考量是引用溯源和拒答机制。

引用溯源（Citation）：要求 LLM 在答案中引用具体的信息来源。可以在 Prompt 中明确要求：”请在回答的每个关键事实后标注来源，格式为 [来源: 文档名, 章节]”。

拒答机制（Refusal）：当检索结果与问题相关性很低时，LLM 应该拒绝回答而不是”硬编”。实现方式是设置相似度阈值，当最佳匹配的相似度低于阈值（如 < 0.65）时，在 Prompt 中注入”参考资料：无相关信息”，并引导 LLM 生成拒答回复。

三、RAG 的检索优化策略

基础的 RAG（用户提问 → 直接向量化 → 检索 → 生成）在很多情况下表现已经不错，但在面对模糊查询、复杂查询和需要精确匹配的场景时，检索召回率会显著下降。RAG 检索优化是提升 RAG 系统质量的核心投入方向。

1. Query Rewrite（查询重写）

用户在使用 RAG 系统时，经常提出不完整、口语化或模糊的问题。这些问题直接向量化后，与知识库中正式的技术文档之间的语义距离很大，导致检索效果不佳。

问题示例：

用户原始问题 问题 重写后的检索查询
“那个限流怎么配啊？” 指代不明，口语化 “API 网关限流配置方法”
“上次说的那个数据库连接池的问题” 依赖对话上下文 “数据库连接池配置问题和解决方案”
“挂了” 极其模糊 “服务挂掉的原因和恢复方法”（需结合上下文推断）

Query Rewrite 的实现方式：

  • LLM 重写：使用 LLM 本身（通常是轻量模型以降低成本）将用户问题改写为更适合检索的表达。在 Prompt 中要求 LLM 消除指代、补全上下文、使用专业术语替代口语表达
  • 多轮对话上下文融合：在多轮对话场景中，用户的后续问题可能依赖对话历史（如”那第二个呢？”）。Query Rewrite 需要融合历史对话上下文，生成独立完整的检索查询

Query Rewrite 的关键原则：重写后的查询不是为了给用户看，而是为了最大化检索召回率。重写时应该使用知识库中可能出现的术语和表达方式，而不是用户的原话。

2. Multi-Query Retrieval（多查询检索）

对于复杂问题，一个检索查询可能无法覆盖所有相关方面。Multi-Query Retrieval 的策略是：从不同角度生成多个查询，分别检索后合并去重。

例如，用户问”如何提高 Spring Boot 应用的启动速度？”，可以生成：

  • 查询 1：”Spring Boot 启动优化方法”（直接角度）
  • 查询 2：”延迟初始化、懒加载配置”（技术角度）
  • 查询 3：”Spring Boot 启动慢的常见原因”（诊断角度）

这种策略特别适合需要综合多方面信息的复杂问题，检索的多样性显著优于单查询。

3. Hybrid Search（混合搜索）

混合搜索在 05-向量数据库.md 中有详细介绍。在 RAG 场景中，混合搜索的典型实践是向量检索（语义） + BM25 检索（关键词），通过 RRF（Reciprocal Rank Fusion，倒数排名融合）合并结果。

在 RAG 中启用混合搜索的信号：

  • 知识库包含大量专有名词、产品型号、错误码（这些是关键词检索的优势领域）
  • 用户经常搜索精确的代码片段、配置项名称
  • 知识库中的文档使用了多样化的术语表达同一概念

混合搜索的融合参数调优：

场景 α（语义权重） 说明
通用知识 QA 0.7 偏重语义理解
技术文档搜索（含代码/配置） 0.5 均衡语义和精确匹配
错误码/产品型号搜索 0.3 偏重关键词精确匹配

4. Rerank（重排序）

向量检索的 TopK 结果是基于向量相似度排序的，但向量相似度并不总是等于实际相关性——两个文本向量相似度高，可能是因为它们在主题上相近，但不一定意味着它们在内容上直接回答了用户的问题。

Rerank 的原理：使用一个更精确但更慢的相关性模型（通常是 Cross-Encoder），对向量检索返回的 TopK 结果进行精确的相关性打分，然后按新分数重新排序。Cross-Encoder 同时接收查询和文档作为输入，通过注意力机制计算精确的相关性分数，比向量相似度准确得多，但计算成本也高得多（不能用于全量检索）。

Rerank 模型选择：

模型 特点
Cohere Rerank v3 目前效果最佳的商业 Rerank 模型
BGE-Reranker-v2（BAAI） 开源首选，中英文表现好
Cross-Encoder（Sentence-Transformers） 通用方案，可自定义训练

Rerank 的成本权衡：Rerank 的单位成本远高于向量检索（Cross-Encoder 对每条查询-文档对都需要完整推理）。典型策略是先用向量检索召回 50-100 条候选，再用 Rerank 精选 5-10 条。

四、高级 RAG 模式

1. Self-Query Retrieval（自查询检索）

Self-Query Retrieval 是让 LLM 从用户的自然语言问题中自动抽取出结构化过滤条件，然后将过滤条件应用于向量检索。

例如，用户问”2024 年发布的关于微服务的架构文档有哪些？”，Self-Query 能将问题拆解为：

  • 语义查询：”微服务架构”
  • 结构化过滤：year = 2024 AND category = '架构文档'

Self-Query 的核心价值是让用户用自然语言表达过滤意图，而不需要在 UI 上设计复杂的筛选条件表单。在企业知识库搜索场景中，用户经常需要表达”某个时间段内的””某类文档中的””某个人写的”等条件，Self-Query 让这些条件能自然融入对话。

2. Parent-Child Chunk（父子文本块）

Parent-Child Chunk 是一种解决”检索粒度”与”上下文完整性”矛盾的策略。

问题：小 Chunk 检索精准，但缺乏上下文；大 Chunk 上下文完整，但检索精度下降。

解决方案：将文档切分为两种粒度的 Chunk：

  • Child Chunk（子块）：小块（如 256 Token），用于向量检索。小块语义聚焦，检索精度高
  • Parent Chunk（父块）：大块（如 1024-2048 Token），包含子块的完整上下文。在检索时，先通过子块匹配到最相关的小块，然后返回包含该子块的完整父块

Parent-Child Chunk 在技术文档、学术论文等结构化强的文档类型中效果特别好——检索精准定位到相关段落（Child），但让 LLM 看到该段落所属完整章节的上下文（Parent）。

3. 多跳检索（Multi-hop Retrieval）

多跳检索（Multi-hop Retrieval）处理的是需要综合多个文档的信息才能回答的问题。

例如，用户问：”张三参与的那个使用 Spring Cloud Gateway 的项目，部署在哪个 K8s 集群上？”这个问题需要：

  • 跳 1：检索”张三 参与 Spring Cloud Gateway 项目”→ 得到项目名称”电商中台”
  • 跳 2：检索”电商中台 K8s 集群 部署”→ 得到集群名称”prod-cluster-east-1”

多跳检索需要 RAG 系统具备”检索→获取信息→生成新查询→再检索”的迭代能力。实现方式通常是 Agent 模式（后续 09-Agent 章节会详细讨论）：Agent 在执行过程中自主判断是否需要进一步检索，并生成后续的检索查询。

4. 上下文压缩（Context Compression）

当检索返回的文档数量多、长度大时，直接全部注入 LLM 会占用大量 Context Window。上下文压缩（Context Compression）的策略是：对检索返回的文档内容进行摘要压缩，只保留与用户问题相关的部分。

实现方式是让 LLM 逐条处理检索结果：”以下文档片段中，哪些内容与用户问题’{用户问题}’相关？请提取出相关的句子，丢弃不相关的部分。”经过压缩的上下文更精简，可以注入更多来源的文档，或者节省 Context Window 留给 LLM 的回答。

上下文压缩适用于检索源文档很长（如整篇论文、长报告）的场景，但需要注意压缩本身会增加延迟和 LLM 调用成本。

七、RAG 的工程化问题

RAG 系统从 Demo 走到生产环境，需要解决一系列工程化问题。这些问题在原型阶段通常被忽略，但在真实业务场景中每一个都可能导致系统不可用。

1. 索引更新与增量同步

知识库不是静态的——文档会新增、修改、删除。RAG 系统需要感知这些变化并同步更新向量索引。

全量重建（Full Re-index）：定期（如每周）清空 Collection 并重新处理所有文档。这是最简单可靠的方案，但成本高（重新 Embedding 所有文档）、周期长（大规模知识库可能需要数小时）。

增量同步（Incremental Sync）：只处理变化的文档。通过监听文档变更事件（文件系统监听、数据库 binlog、消息队列），触发增量索引更新。

增量同步的工程挑战：

  • Chunk ID 的稳定性：文档内容变更后，切分边界可能移动，导致所有 Chunk 的 ID 都发生变化。需要设计稳定的 Chunk ID 生成策略（如基于内容哈希），使未变化的 Chunk 能复用已有的向量
  • 删除的可靠性：删除文档意味着需要删除与之关联的所有 Chunk。需要在元数据中记录”文档 ID → 所有 Chunk ID”的映射关系

2. 数据一致性

向量数据库和关系数据库之间的数据一致性是 RAG 工程化的核心挑战。

常见的不一致场景：

  • 向量已写入但事务回滚：关系数据库中的文档写入事务提交失败，但向量已经写入向量数据库（大多数向量数据库不支持事务回滚）
  • 向量未写入但文档已可见：用户在界面上看到了新上传的文档，但向量索引还未完成，导致该文档暂时搜索不到
  • 删除不彻底：关系数据库中的文档已标记为删除，但向量数据库中的对应向量未被清理

推荐的工程方案：

  • 消息队列解耦：文档写入 PostgreSQL 后，发送一条消息通知索引服务异步处理。核心数据库操作不受向量索引速度的影响
  • 状态标记：为每个文档维护一个 index_status 字段（pending / indexing / indexed / failed），搜索时只检索状态为 indexed 的文档
  • 定期对账：定时对比关系数据库中的文档列表和向量数据库中的向量列表，发现差异后补录或清理

4. RAG 质量评估

RAG 系统的质量不能仅靠”感觉还行”来评判，需要建立量化评估体系。

核心评估指标：

指标 定义 测量方法
召回率（Recall） 相关文档被检索到的比例 标注测试集中”应检索到的文档”与实际检索结果的交集
精确率（Precision） 检索结果中真正相关的比例 人工标注每个检索结果是否相关
上下文相关性 检索结果与问题的语义相关性 可以用 LLM 自动评分（如 Ragas 的 Context Relevancy）
答案忠实度（Faithfulness） 答案是否基于检索结果而非编造 逐句对照答案与检索结果，检测有无幻觉（Ragas Faithfulness）
答案相关性 答案是否直接回答了问题 LLM 评估答案与问题的相关性（Ragas Answer Relevancy）

评估工具：

  • Ragas：开源的 RAG 评估框架，提供 Context Precision、Context Recall、Faithfulness、Answer Relevancy 等指标
  • LangSmith：LangChain 的商业评估平台

一、Prompt 的基本概念与结构

1. 什么是 Prompt

Prompt（提示词）是人与大语言模型之间的通信协议。如果 LLM 是一个能力强大但需要精确指令的执行者，那么 Prompt 就是你向它下达指令的语言。同样的模型，Prompt 写得好与不好，输出质量的差距可能大到像是两个完全不同的模型。

从工程角度来看，Prompt 的实质是：用自然语言描述业务需求，由 LLM 执行并返回结果。这与传统编程有本质区别——传统编程中，你用 Java 精确地告诉计算机每一步做什么；Prompt 编程中，你描述目标、设定约束、提供示例，模型自己决定如何达成目标。

Prompt 的组成从简单到复杂可以分为三个层次：

  • 基础层：一条简单的自然语言指令，如”将以下文本翻译成英文”
  • 结构层：包含角色设定、任务描述、输出格式约束的结构化 Prompt
  • 增强层：在结构层的基础上，加入 Few-shot 示例、思维链引导、外部知识上下文（RAG 检索结果）等增强要素

Prompt 工程（Prompt Engineering）就是系统化地设计、测试和优化 Prompt，以达到稳定、高质量输出的实践。它不是一次性的”写一段话”，而是需要版本管理、A/B 测试和持续迭代的工程活动。

二、Prompt 的组成要素

每一个有效的 Prompt 要素都有明确的职责。理解每个要素的作用，才能在设计 Prompt 时做出正确的取舍——加什么、不加什么，每个要素占多少 Token。

1. 角色设定（Role / Persona）

角色设定告诉模型”你是谁”，它决定了模型的回答视角和知识领域。例如：

  • “你是一名资深的 Java 后端工程师，精通 Spring Boot 和微服务架构”
  • “你是一位严谨的代码审查者，你的职责是发现代码中的潜在 Bug 和安全漏洞”

角色设定的效果机制：LLM 在训练过程中学习了大量不同角色的语料。指定角色相当于把模型的生成空间约束到该角色对应的语料分布区域，使其输出更符合该领域的表达习惯和知识深度。

角色设定的实践要点：

  • 越具体越好：”你是一个工程师”远不如”你是一个有 10 年经验的 Java 后端工程师，专注于高并发系统设计”有效
  • 与任务匹配：角色设定要和实际任务一致。让”安全审查者”去写代码，和让”后端开发”去做安全审查，效果都会打折扣
  • 不要过度依赖：角色设定有助于引导输出风格，但不能替代清晰的指令和约束。它只是辅助，不是核心

2. 任务描述（Task）

任务描述是 Prompt 的核心——明确告诉模型要做什么。一个好的任务描述有三个特征：

  • 明确（Explicit）：不说”分析这段代码”，而是”分析这段代码中可能导致死锁的并发问题”
  • 可执行（Actionable）：模型能基于给定的信息完成任务。如果任务需要模型没有的数据，要么提供数据，要么告诉它如何获取
  • 可验证（Verifiable）：任务的完成标准是清晰的。理想情况下，输出是否满足要求应该是可自动判断的

不好的任务描述 vs 好的任务描述：

不好 好
“优化这段代码” “优化以下 Java 方法的执行效率，重点关注循环中的数据库查询和对象创建”
“写个单元测试” “为以下 Service 方法编写 JUnit 5 单元测试，使用 Mockito Mock 所有外部依赖，覆盖正常路径、空参数和异常三种情况”
“分析日志” “从以下应用日志中找出所有 ERROR 级别日志，按发生频率排序，并给出每个错误的原因分析和修复建议”

3. 上下文信息（Context）

上下文是任务相关的背景知识和数据。模型无法访问外部信息，你必须显式地告诉它所有它需要知道的东西。

上下文的类型：

  • 领域知识：术语定义、业务规则、行业惯例——模型训练数据中可能有也可能没有的信息
  • 当前状态：用户的身份、系统的当前状态、历史对话摘要——这些模型绝不可能知道
  • 参考数据：文档片段（RAG 检索结果）、数据库查询结果、API 返回数据——模型的”外部记忆”

上下文的设计原则：

  • 先给定义，后给任务：模型对靠前的信息更关注。重要的定义和规则放在 Prompt 的靠前位置
  • 压缩而非堆砌：上下文不是越多越好。一段 500 Token 的精炼摘要，效果往往优于 5000 Token 的原始文档。LLM 在长文本的”中间部分”提取信息能力相对较弱（Lost in the Middle 现象）
  • 标注上下文边界：用 XML 标签或 Markdown 分隔符明确标记上下文的起止，如 <context>...</context>，帮助模型区分”这是参考资料”和”这是指令”

4. 约束条件（Constraints）

约束条件定义了输出的边界和规则——不是告诉模型做什么，而是告诉它不要做什么或在什么范围内做。约束的重要性不亚于任务描述本身，因为 LLM 在缺乏约束时可能产生意外但有破坏性的输出。

常见的约束类型：

  • 内容约束：”不要对你不确定的事实进行推断，如果信息不足请直接说明”
  • 格式约束：”输出必须是合法的 JSON，不要包含 Markdown 代码块标记”
  • 长度约束：”回答不超过 200 字，只需给出结论不需要解释过程”
  • 安全约束：”不要输出任何可执行的代码，不要提供可能被用于攻击的信息”
  • 语言约束：”始终使用中文回答，除非用户使用英文提问”

约束的设计要点：如果说任务描述是”开车去北京”，约束就是”限速 120、不要走国道、中途不要加油”。好的约束让模型在正确轨道上运行，过多的约束则会让模型束手束脚。

5. 输出格式（Output Format）

输出格式是约束条件中最重要的子类型。明确的格式要求能将非确定性的自然语言输出转化为可解析的结构化数据，这是 AI 模型与后端业务系统对接的关键。

6. Few-shot 示例（Examples）

Few-shot 示例是指在 Prompt 中提供一至多个输入-输出的范例，让模型理解期望的行为模式。

示例的三大作用：

  • 澄清格式：当输出格式较复杂时，一个例子比 500 字的格式描述更直观
  • 展示标准：示例展现了”什么是好的输出”的标准，比文字描述”输出质量要高”有效得多
  • 处理边界情况：通过精心挑选的边角示例，教会模型如何处理模糊输入和特殊情况

示例设计的最佳实践：

  • 从简单到复杂：如果给多个示例，难度应递增。第一个示例展示基础模式，后续示例逐步引入边界情况
  • 标注示例的意图：用注释说明”这个示例展示了 XX 情况”，帮助模型理解示例的用意
  • 示例要真实：用真实业务数据构造示例，而不是凭空编造。模型对虚构的示例适应性较差
  • 不要过量：2-3 个精心挑选的示例通常足够。5 个以上的示例收益递减，而且大量消耗 Token

三、Prompt 工程核心方法论

Prompt 工程不是玄学，有一套系统的方法可以让 Prompt 从”碰运气”变成”可预测”。以下是四种核心方法论，按复杂度从低到高排列。

1. Zero-shot：直接下指令，不给示例

Zero-shot（零样本）是最简单的 Prompt 方式——直接告诉模型做什么，不给任何示例。

Zero-shot 适用于：

  • 任务足够简单明确，模型已有充分的理解
  • 输出格式简单（如分类、二元判断）
  • Token 预算紧张的场景

Zero-shot 的局限：对于复杂任务（特定格式输出、多步骤推理、领域专有术语理解），纯 Zero-shot 的成功率较低。这时需要升级到 Few-shot。

2. Few-shot：给出范例，让模型举一反三

Few-shot（少样本）通过在 Prompt 中提供示例来引导模型的行为。LLM 具有强大的 In-Context Learning（上下文学习）能力——即不需要更新模型参数，仅通过在 Prompt 中展示几个例子，模型就能学会并复制这种模式。

Few-shot 的关键技巧：

  • 示例的多样性：示例应覆盖不同类型的输入，这样才能教会模型处理变化
  • 关键模式的重复：如果想让模型学会某个输出模式（如输出 JSON Schema 的某字段），在多个示例中反复展示该模式
  • 负面示例的价值：展示”什么是错的”有时比展示”什么是对的”更有效。例如：展示一个错误的 JSON 结构并标注”不要这样”

3. Chain of Thought（CoT）：让模型展示推理过程

Chain of Thought（思维链，简称 CoT）是一种引导模型逐步推理而不是直接给出答案的技术。它的核心思想来自对人类解决问题方式的观察——面对复杂问题时，大多数人不直接得出结论，而是通过一系列推理步骤逐步逼近答案。

CoT 最核心的技巧是在 Prompt 中加上一句：“Let’s think step by step”（让我们一步一步思考）。这看似简单的一句话，能将复杂推理任务的准确率提高 20%-50%。

  • 数学计算和逻辑推理
  • 多条件判断（需综合多条规则作出决策）
  • 代码分析（Bug 诊断、安全审查）
  • 需要展示可解释性的场景（让用户看到 AI 为什么得出这个结论）

  • 创意生成（思维链会限制创造力）
  • 只需要结论不需要过程的场景

4. ReAct：推理与行动交替

ReAct（Reasoning + Acting）是 CoT 的升级版——模型不仅推理，还在推理过程中穿插调用外部工具。每次工具调用都是一次”行动-观察”循环：模型推理出需要什么信息，调用工具获取信息，观察结果后继续推理。

ReAct 是 Agent 智能体的核心推理范式（将在后续章节详述）。在 Prompt 工程层面，理解 ReAct 的关键是：对需要多步信息和多工具协作的复杂任务，设计 Prompt 时要预留”工具调用占位符”，让模型知道何时、如何请求外部帮助。

5. 方法论选型指南

实际项目中，这些方法通常是组合使用的：一个 Prompt 可能同时包含 Few-shot 示例、CoT 引导和 ReAct 工具调用。关键是根据任务复杂度选择恰当的层级，而非盲目追求高级方法。

四、Prompt 安全：注入攻击与防护

1. Prompt Injection 的原理

Prompt Injection（提示注入）是目前 AI 应用面临的最重要且独特的安全威胁。它的攻击原理是：通过在用户输入中嵌入恶意的指令文本，试图覆盖或绕过 System Prompt 中设定的行为约束。

Prompt Injection 的分类：

  • 直接注入（Direct Injection）：攻击者在用户输入中直接写入越狱指令，试图覆盖 System Prompt
  • 间接注入（Indirect Injection）：攻击者将恶意指令隐藏在 RAG 检索的文档中。当模型检索到该文档并作为上下文使用时，隐藏的指令就会被执行
  • 多模态注入：在图片、PDF 等非纯文本载体中嵌入隐藏指令

2. 防护策略

输入层防护：

  • 输入安全检查：在构造 Prompt 之前，对用户输入做安全扫描。检测常见的注入模式（如 “ignore previous instructions”、”you are now DAN” 等已知的越狱提示）
  • 输入长度限制：过长的用户输入更可能包含攻击内容，适当限制单次输入长度
  • 敏感词过滤：过滤掉用户输入中的 System Prompt 关键词（如 “system”、”instruction”、”guideline”）

  • 重复强调约束：在 Prompt 的末尾再次强调核心安全规则，因为模型对 Prompt 末尾的信息关注度更高
  • 最小权限原则：Prompt 中不给模型超出任务需要的权限描述。不要写”你能访问所有系统功能”，只写”你能查询产品信息和订单状态”

输出层防护：

  • 输出内容审核：模型生成的回答在返回给用户之前，经过一道内容安全审核——检测是否泄露了内部信息、是否执行了注入指令
  • 结构化输出校验：如果期望 JSON 输出，校验其 Schema 是否合规。注入攻击常导致输出偏离预期格式，可被检测到

  • 全链路日志：记录每次 LLM 调用的完整 Prompt 和输出，便于事后审计和分析攻击模式
  • 异常检测：监控输出内容的异常模式（如模型突然输出了 System Prompt 片段），触发告警

一、Function Calling 的基本概念与原理

1. 什么是 Function Calling

Function Calling（函数调用，也称 Tool Calling / 工具调用）是 LLM 的一项关键能力：模型不再只能输出文本，它可以”决定”调用一个预定义的外部函数，并生成符合函数签名的参数。

准确地说，Function Calling 本身并不是模型在”执行”函数，而是模型在”请求”执行——模型输出一个结构化的函数调用请求（包含函数名和参数），由外部程序（如你的 Java 后端）真正执行函数，然后将执行结果返回给模型，模型再基于结果生成最终响应。

将 Function Calling 放在整个 AI 应用架构的坐标系里观察，它是连接 LLM 与外部世界的桥梁。没有它，LLM 只是一个”会说话的脑子”——能理解问题、能生成文本，但无法查询实时数据、无法执行操作。有了 Function Calling，LLM 变成了一个”能办事的智能体”——它可以查询数据库、调用 API、触发流程，真正参与到业务系统的运作中。

2. Function Calling 解决的核心问题

问题一：LLM 无法访问实时数据

LLM 的知识来自静态的训练数据。对于实时信息——当前股价、库存数量、用户订单状态——模型要么不知道，要么用训练数据中的过时信息”编造”。Function Calling 让模型能调用后端 API 查询实时数据，再基于真实数据生成回答。

问题二：LLM 无法执行操作

LLM 天然只能输出文本，不能执行任何实际操作——不能创建工单、不能发送邮件、不能修改数据库。Function Calling 让模型能通过调用后端函数来”触发”这些操作。

问题三：LLM 需要结构化的外部交互

用户用自然语言表达意图，但后端 API 需要结构化的参数。Function Calling 承担了”自然语言 → 结构化参数”的转换——用户说”帮我查一下上周的订单”，模型提取出 { "dateRange": {"start": "2026-05-26", "end": "2026-06-01"} }。

能力 没有 Function Calling 的 LLM 有 Function Calling 的 LLM
获取实时数据 ❌ 只能编造或拒绝 ✅ 调用 API 查询真实数据
执行操作 ❌ 只能输出文本建议 ✅ 触发业务流程执行
结构化交互 ❌ 只能输出自然语言 ✅ 生成结构化的函数调用参数
多系统协作 ❌ 完全隔离 ✅ 作为协调者调用多个系统

3. 工作原理：模型端的工具选择与参数推理

Function Calling 的核心机制可以分为两个阶段：

阶段一：工具选择（Tool Selection）

模型在收到用户消息后，基于 Prompt 中注册的工具定义（Tool Definitions），判断是否需要调用某个工具。如果需要，选择最合适的工具；如果不需要，直接生成文本回复。

模型做这个判断的依据是工具定义中的 description——工具描述写得好坏，直接决定了模型能否在正确的时机选择正确的工具。描述模糊的工具可能永远不会被调用，描述过于宽泛的工具可能在不恰当的时机被调用。

阶段二：参数推理（Argument Inference）

选定工具后，模型基于工具定义中的 parameters（JSON Schema）和用户消息，推理出调用该工具所需的参数值。这个过程本质上是将自然语言中的信息抽取并映射为结构化的 JSON 参数。

2. 工具描述的最佳实践

工具定义的 description 是模型判断何时调用该工具的唯一依据。写好工具描述是 Function Calling 工程中最重要也最容易被忽视的环节。

好的工具描述应该包含：

  • 做什么（核心功能）：一句话概括工具的核心功能
  • 什么时候用（适用场景）：明确告诉模型在什么情况下应该调用这个工具
  • 什么时候不用（排除场景）：明确告诉模型什么情况下不应该调用这个工具
  • 参数说明：每个参数的含义、格式和约束

描述中毒与防护：工具描述可以被视为一种特殊的”System Prompt”。如果工具描述中包含敏感信息（如内部系统的接口地址、数据库表名），这些信息会泄露给模型提供商。工具描述应只包含模型选择工具和生成参数所需的信息，避免暴露内部实现细节。

3. 参数设计原则

一、明确性：每个参数的目的必须在模型的认知范围内可理解

参数 description 不能太简略。"type": "string", "description": "ID" 是不够的——什么 ID？格式是什么？模型需要这些信息来正确推理。

二、约束性：尽量使用 enum 而非自由文本

如果某个字段只有有限的几个合法值，使用 enum 约束。这不仅提高参数生成准确率，也是安全加固的起点。

三、简洁性：参数数量控制在合理范围

研究表明，参数数量超过 5 个时，模型生成参数的质量开始下降。如果确实需要很多参数，考虑将部分参数设为非必填 required 之外，让模型只在必要时填充。

四、区分度：不同工具的职责要明确分离

两个功能相近的工具（如 searchOrders 和 listOrders）会让模型选择困难，导致调用错误的工具。合并功能相近的工具，或为每个工具写清适用场景的差异。

三、工具调用执行机制

1. 同步执行与异步执行

同步执行是最简单直接的模型——收到 Function Calling 请求后，阻塞当前线程等待执行结果，将结果发回模型后继续对话。适合绝大多数实时问答场景，用户等待时间 = 模型推理时间 + 工具执行时间 + 模型再次推理时间。

异步执行适用于耗时较长的操作（如批量处理、数据导出）。基本模式是：收到调用请求后立即返回一个”任务已提交”的确认，后端异步执行，执行完成后通过回调或轮询将结果注入对话。

同步执行 异步执行
适用场景 查询、简单操作（< 5s） 批量处理、外部调用（> 5s）
用户体验 等待完整结果 先收到确认，后续获取结果
实现复杂度 低 中（需要任务状态管理和回调）
超时处理 直接返回超时错误 轮询状态或主动推送

2. 多轮工具调用（Multi-turn）

单轮对话中可以发生多次工具调用。典型场景是链式查询——先查 A 得到结果，再基于 A 的结果查 B。

多轮调用中的上下文管理：每一轮的模型输入包含前面所有轮的消息（用户消息 + 工具调用 + 工具返回）。这个上下文需要被妥善管理——部分模型的 Context Window 有限，过长的工具调用链可能导致上下文溢出。

终止条件：模型在任一回合都可以选择不调用工具（finish_reason: "stop"），直接生成最终回复。这是自然的终止条件。但在代码实现中需要设置一个最大轮次上限（如 10 轮），防止模型陷入无限循环调用。

3. 并行工具调用（Parallel Tool Calling）

当模型判断用户的问题需要调用多个相互独立的工具时，可以在单轮中同时发起多个工具调用。

判断工具是否可以并行的标准：两个工具调用之间没有数据依赖。如果工具 B 的输入参数依赖于工具 A 的输出，则不能并行，必须先执行 A 再执行 B。

性能收益：对于 I/O 密集型工具（如调用外部 API、查询数据库），并行执行可以将多轮串行调用的延迟从”T1 + T2 + …”降低为”max(T1, T2, …)”。

4. 强制调用工具与工具选择策略

在某些场景下，业务逻辑要求必须调用特定工具，而不是让模型”自己决定”。例如，用户进入”订单查询”菜单时，系统强制调用 queryOrder 工具，模型不能选择直接聊天。

控制方式：

策略 说明 适用场景
tool_choice: "auto" 模型自行决定是否调用工具（默认） 大多数场景
tool_choice: "required" 必须调用至少一个工具 结构化输入表单、菜单导航
tool_choice: "none" 禁止调用工具 纯聊天场景
tool_choice: {"type": "function", "function": {"name": "xxx"}} 强制调用指定工具 明确的业务流程节点

实践建议：大多数场景保持 auto，让模型自行判断。在以下情况使用强制模式：

  • 业务流程明确需要特定数据时（如工单创建页面强制调用 createTicket）
  • 可用工具有限且都需要调用时（required）
  • 需要防止模型绕过工具直接回复的场景

四、工具调用的工程安全

Function Calling 将后端 API 暴露给了 LLM——这带来了一个全新的安全边界。传统 API 安全的假设是”调用方是可信任的内部系统”，但 LLM 作为调用方会引入新的风险：模型可能生成不合法的参数、可能被 Prompt Injection 诱导调用不该调用的工具、可能在循环中过度调用。

1. 参数白名单与输入校验

LLM 生成的参数不可信——模型可能因为幻觉生成不合法的参数值，也可能因为 Prompt Injection 被注入恶意的参数。

2. 权限控制

不是所有用户都有权调用所有工具。一个普通员工不应能调用 deleteAllOrders，即使 LLM “决定”调用它。

权限控制的核心问题：LLM 的”身份”是谁？答案是——LLM 的调用权限等于当前对话用户的权限。工具调用必须在当前用户的权限上下文中执行。

工具定义中的权限声明：在工具注册时标记所需的权限级别，注册阶段就过滤掉当前用户无权访问的工具——不将无权工具的定义发送给模型，从源头上消除模型调用无权限工具的可能性。

3. 幂等性设计

LLM 可能在同一个对话中多次调用同一个工具——例如，模型”忘记”刚才已经查过订单状态，又调用了一次。写操作工具必须设计为幂等的，保证多次调用不会产生副作用累积。

实践中可以显式暴露 idempotencyKey 参数：让模型在调用时生成一个唯一的幂等键（可以要求模型使用 UUID），后端基于此键去重。

4. 超时与重试

工具执行可能因下游系统故障而长时间阻塞。需要为每个工具调用设置超时和重试策略。

5. 调用频率限制

防止 LLM 在循环中过度调用工具——例如，模型陷入”调用-失败-重试”的死循环，或 Prompt Injection 触发高频调用。

一、Agent 的基本概念与定位

1. 什么是 Agent

Agent（智能体）是一种以 LLM 为推理核心、具备自主决策能力的 AI 应用形态。与被动等待用户指令的 Chatbot 不同，Agent 能够感知环境、制定计划、选择工具、执行动作、反思结果，在一个闭环中自主完成复杂任务。

用工程语言来定义：Agent = LLM（推理引擎）+ Planning（规划能力）+ Memory（记忆系统）+ Tools（工具集合）+ Reflection（反思机制）。这五个组件以 LLM 为中心协同工作，使 Agent 具备了超越单次问答的自主任务执行能力。

2. Agent 与 Chatbot、RAG、Workflow 的区别

这是 AI 应用架构中最基础也是最容易混淆的四个概念。它们的差异不在于”用了什么技术”，而在于谁掌握决策权。

维度 Chatbot RAG AI Workflow Agent
决策主体 用户 系统（固定流程） 流程引擎（预定义图） LLM（自主决策）
执行路径 单轮 Q&A 检索 → 生成（固定两阶段） 预定义的有向图 LLM 动态规划
灵活性 最低（一问一答） 低（固定模式） 中（分支预定义） 最高（自主选路）
可预测性 最高 高 高 低（模型可能走非预期路径）
适用任务 简单问答 知识库问答 标准化业务流程 开放式复杂任务

形态演进关系：

Agent 不是对前三种形态的替代，而是对它们的能力升级。一个 Agent 在执行任务时，内部可能调用 RAG 检索知识、执行预定义的 Workflow 子流程、进行多轮对话——但它自己决定何时调用什么、以什么顺序调用、什么时候任务算完成。

3. Agent 的自主层级

并非所有自称 Agent 的系统都有相同的自主程度。根据 LLM 掌握的控制权大小，可以划分出三个层级：

层级 控制权分布 典型表现 风险
L1：工具增强型 用户控制流程，Agent 只提供工具 用户说”查订单”，Agent 调用查询工具返回结果 低
L2：任务委派型 用户定义目标，Agent 规划并执行 用户说”帮我处理这个退款”，Agent 自主规划：查订单→判断可退款→创建退款单→通知用户 中
L3：完全自主型 Agent 持续运行，主动感知和响应 运维 Agent 持续监控系统指标，发现异常后自主排查、告警、甚至自动修复 高

目前企业级应用的最佳实践是L2 任务委派型——Agent 在明确的边界和约束下自主完成任务。L3 完全自主型在风险控制上仍有较大挑战，仅在少数高度受控的场景中可行。

二、Agent 的核心组成

1. Planning（规划能力）

Planning 是 Agent 区别于简单 Chatbot 的核心能力——面对一个复杂目标，Agent 不是一步到位，而是先制定执行计划，再逐步执行。

规划的两个层次：

  • 任务分解（Task Decomposition）：将复杂目标拆解为多个可执行的子任务。例如，”帮我准备 Q3 技术分享”被分解为”① 检索近半年的技术文档 ② 分析高频话题 ③ 选择 3 个候选主题 ④ 为每个主题生成大纲”
  • 执行排序（Task Ordering）：确定子任务的执行顺序和依赖关系。Agent 需要判断哪些任务可以并行（如同时检索多个数据源），哪些任务有依赖（必须先检索才能分析）

规划的两种生成方式：

方式 描述 优势 劣势
一次性规划 在开始执行前生成完整计划 全局视角，步骤之间逻辑一致 无法应对中途的意外发现
增量规划 执行一步，基于结果规划下一步 灵活，能基于新的信息调整 可能偏离全局最优路径

实践中，大多数 Agent 实现采用增量规划——执行每步后观察结果，再决定下一步。这更接近人类解决复杂问题的方式，也更好地利用了 LLM 的推理能力。

2. Memory（记忆系统）

Memory 是 Agent 实现上下文连续性的基础。没有记忆的 Agent 每一轮都像”失忆”一样重新开始。

三种记忆类型：

短期记忆的管理挑战：LLM Context Window 有限，对话历史增长到一定程度必须截断。简单的截断策略（如只保留最后 N 轮）会丢失早期的关键信息。更高级的做法是记忆摘要——用 LLM 将历史对话压缩为摘要，保留关键信息的同时控制 Token 消耗。

长期记忆的实现：长期记忆本质上是一个RAG 系统——将历史任务的经验、用户偏好存储为向量，当 Agent 处理新任务时检索相关记忆。例如，Agent 之前帮用户处理过”退款”任务，当下次用户再说”和上次一样退了吧”，Agent 能检索到上次的操作参数并复用。

3. Tool Use（工具使用）

Tool Use 是 Agent 与外部世界交互的接口。第 07 章 Function Calling 已详细讨论了工具定义和调用机制，在 Agent 语境下需要补充的是工具的使用策略。

Agent 对工具的使用比 Workflow 中预定的工具调用更灵活：

  • 自主选择：Agent 自己决定何时使用哪个工具——不需要流程图中预先指定
  • 组合使用：Agent 可以将多个工具组合使用——先用搜索工具找到相关文档，再用计算工具分析数据，最后用邮件工具发送报告
  • 错误恢复：工具调用失败时，Agent 可以分析失败原因并尝试替代方案——工具超时了就换一个查询策略，而不是简单报错

4. Reflection（反思与自我修正）

Reflection 是 Agent 在执行过程中评估自己的进展并调整策略的能力。它是 Agent 区别于”一次性输出”的关键特征——Agent 不是说一次就完，而是反复检查、修正、优化。

Reflection 的三个层次：

  • 结果评估：”工具返回的结果是否满足了我的需求？如果不满足，我该如何调整查询？”
  • 进展评估：”在完成总体目标的路径上，我进展到哪一步了？方向是否正确？”
  • 策略反思：”我目前的方法是最优的吗？有没有更好的方式？前几步的策略选择是否有改进空间？”

Reflection 的触发时机：

  • 每一步后：执行工具调用后，评估结果质量，决定是否需要调整
  • 阶段性检查：完成一组子任务后，检查进展是否符合预期
  • 遇到障碍时：工具报错、结果异常、信息不足时，分析原因并调整策略

5. Skills（技能体系）

Skills 是比 Tools 更高层次的抽象——Skill 是一组相关 Tool 的封装，加上使用这些 Tool 的领域知识和操作规范。

如果说 Tool 是”函数”（单一操作，输入→输出），Skill 就是”模块”（封装了领域知识、操作流程和多个 Tool 的协作方式）。例如：

  • Tool：queryDatabase(sql) ——执行一条 SQL 查询
  • Skill：DatabaseDiagnosis ——包含了查询慢查询日志、分析索引使用情况、检查连接池状态、生成优化建议等一系列操作，以及”先看整体指标再定位具体问题”的诊断方法论

Skills 的设计直接影响了 Agent 的实际效能——精心设计的 Skills 封装了领域专家的判断逻辑，让 Agent 的决策更接近专家行为；设计粗糙的 Skills 只是一堆函数的堆砌，反而增加了 Agent 选择的困难。

三、Agent 的三种核心循环模式

Agent 的行为驱动核心是一个循环（Loop）：感知 → 思考 → 行动 → 观察 → 反思 → 再思考…。不同的循环模式决定了 Agent 的决策方式。

1. ReAct Loop（推理-行动循环）

ReAct（Reasoning + Acting）是最经典也是使用最广泛的 Agent 循环模式。其核心思想是推理和行动交替进行：每一步 Agent 先输出推理（分析当前状态、思考下一步做什么），然后执行一个动作（调用工具或输出结果），观察动作结果，再进入下一轮推理。

ReAct 的每一步都强制 Agent 输出 “思考→行动→观察” 的完整链条，这使得 Agent 的决策过程完全透明——你不仅看到它做了什么，还看到它为什么这样做。这种透明性在调试和审计时至关重要。

2. Plan-Execute（规划-执行模式）

Plan-Execute 模式将 Agent 的决策分为两个阶段：先制定完整计划，再逐步执行。它比 ReAct 更适合目标明确、步骤可预见的任务。

Plan-Execute vs ReAct 的选择：

ReAct Plan-Execute
适用任务 探索性强，路径不确定 目标明确，步骤可预见
效率 每步都推理，Token 消耗大 规划一次，执行多步，效率更高
灵活性 高，能随时调整 中，需要回到规划阶段调整
用户可见性 逐步看到进展 可以先审查计划再执行

实践中，Plan-Execute 和 ReAct 并不互斥。可以在宏观层面使用 Plan-Execute（先规划整体策略），在微观层面使用 ReAct（执行每个步骤时灵活调整）。

3. Reflection Loop（反思-改进循环）

Reflection Loop 在 ReAct 或 Plan-Execute 的基础上增加了一个后置的反思阶段——任务执行完成后，Agent 反思自己的表现，总结经验教训，将改进点存入长期记忆。

Reflection Loop 的价值在于让 Agent 从经验中学习，随着使用次数增长变得越来越”懂”如何高效完成任务。这是 Agent 从”能干活”进化到”会干活”的关键机制。

四、单 Agent、多 Agent 与工程化约束

1. 单 Agent 架构

单 Agent 架构是最基础的 Agent 形态——一个 LLM 实例负责所有的推理、规划和工具调用。它简单直接，适合中等复杂度的任务。

单 Agent 的适用边界：

  • 任务边界清晰，不涉及多个专业领域
  • 可用的工具数量在 10-20 个以内
  • 任务的决策链不会过长（< 15 轮 ReAct 循环）
  • 用户期望的完成时间在分钟级（非小时级）

当任务跨多个不相关的专业领域，或者工具数量膨胀到 LLM 难以有效选择时，单 Agent 的效率和准确性都会下降——这时需要考虑多 Agent 协作。

2. 多 Agent 协作架构

多 Agent 架构将任务分配给多个专业化的 Agent，每个 Agent 负责一个子领域，通过协作完成复杂任务。

常见协作模式：

模式 描述 适用场景
主从模式（Orchestrator-Worker） 一个主 Agent 负责任务分解和调度，多个 Worker Agent 负责执行具体子任务 有明确主流程的多步骤任务
对等协作（Peer-to-Peer） 多个专业 Agent 平等对话，各自贡献专业视角 需要多领域交叉分析的任务
辩论模式（Debate） 多个 Agent 分别生成答案，互相质疑和评审，最终融合 需要高质量、高可靠性输出的任务
分层模式（Hierarchical） 多层 Agent 结构，上级负责策略，下级负责执行，层层分解 超复杂任务（如大型系统重构）

多 Agent 的工程挑战：

  • 通信开销：Agent 之间的消息传递增加了 Token 消耗和延迟
  • 一致性：多个 Agent 可能给出矛盾的信息或建议，需要融合机制
  • 错误传播：一个 Agent 的错误决策可能影响下游 Agent 的判断
  • 调试困难：多 Agent 系统的决策链更长，定位问题根因更难

3. Agent 的工程化约束

Agent 的自主性是一把双刃剑——灵活性的代价是失控风险。工程化约束是让 Agent 在安全边界内发挥能力的必要手段。

约束一：行动边界（Action Boundary）

不是所有工具都应该暴露给 Agent。区分”安全工具”（只读查询、数据分析）和”危险工具”（删除数据、发送外部消息、修改配置）。危险工具需要额外的确认机制或完全禁止 Agent 自主调用。

约束二：步数限制（Step Limit）

Agent 可能在复杂任务中陷入无限循环——反复搜索、反复调整但无法收敛。设置最大 ReAct 步数（如 15 步），超过限制时强制终止并给出当前状态摘要。

约束三：Token 预算（Token Budget）

Agent 的每次推理、每次工具调用都消耗 Token。复杂任务可能消耗数万甚至数十万 Token。为每个 Agent 任务设置 Token 预算上限，超出预算时优雅降级。

约束四：用户确认（Human Confirmation）

对于高风险操作，Agent 执行前必须获得用户确认。不是停止 Agent，而是”Agent 规划好方案 → 展示给用户审核 → 用户确认后执行”——Agent 仍然自主规划，但执行权受控。

约束五：沙箱化（Sandboxing）

Agent 的工具调用应该在受控的环境中执行——数据库只读副本、API 调用的参数校验层、文件操作的限制目录。Agent 永远不能直接访问生产系统的关键资源。

一、向量数据库的核心问题与定位

1. 什么是向量数据库

向量数据库（Vector Database）是专门用于存储、索引和检索高维向量数据的数据库系统。与传统数据库操作结构化字段（如 WHERE age > 18）不同，向量数据库的核心操作是相似性检索（Similarity Search）——给定一个查询向量，在海量向量中找到最相似的 TopK 个结果。

用更直观的方式理解：传统数据库回答的是”精确匹配”问题——“找出所有年龄等于 25 的用户”；向量数据库回答的是”语义相似”问题——“找出与这段话含义最接近的 10 篇文档”。传统数据库的查询结果是确定性的（要么满足条件，要么不满足）；向量数据库的查询结果是近似性的（每个结果有一个相似度分数，分数越高越相关）。

向量数据库与 Embedding 模型的关系是上下游协作：Embedding 模型负责将非结构化数据（文本、图像、音频）转换为向量；向量数据库负责存储这些向量，并在需要时快速找到最相似的向量。两者组合起来，构成了 AI 应用中语义检索的基础设施。

对比维度 传统数据库（MySQL/PostgreSQL） 向量数据库（Milvus/Qdrant）
核心查询 精确匹配、范围查询 相似性检索（ANN）
索引结构 B-Tree、Hash、倒排索引 HNSW、IVF、PQ 等向量索引
查询结果 确定性（满足条件/不满足） 近似性（相似度分数排序）
典型操作 WHERE, JOIN, GROUP BY search(vector, topK)
适用数据 结构化数据 非结构化数据的向量表示

2. 为什么需要专用的向量数据库

一个自然的疑问是：传统数据库不能存向量吗？PostgreSQL 可以用 float[] 或 ARRAY 类型存储向量，然后手动计算相似度。然而这种”手工”方案在数据量超过十万级时会遇到无法克服的性能瓶颈。

传统数据库存储向量的局限：

当数据量达到 10 万条时，用 PostgreSQL 做一次暴力全表扫描的相似度计算需要遍历所有向量，单次查询耗时可能达到数百毫秒甚至秒级。当数据量增长到百万级，暴力扫描变得不可接受——它是 O(n) 复杂度，且每增加一个维度，计算成本线性增长。

向量数据库的解决思路：使用 ANN（Approximate Nearest Neighbor，近似最近邻）算法。ANN 的核心思想是：不需要扫描所有向量，而是通过索引结构快速定位到向量空间中”可能包含最近邻”的局部区域，只在该区域内计算精确相似度。ANN 将检索复杂度从 O(n) 降低到 O(log n) 甚至更低，同时将检索精度损失控制在 1%-5% 以内。

传统数据库暴力扫描 向量数据库 ANN 索引
检索复杂度 O(n) O(log n)
10 万向量检索 ~100ms ~5ms
100 万向量检索 ~1s ~10ms
1 亿向量检索 ~100s（不可用） ~50ms
精度 100%（精确） 95%-99%（近似）

专用向量数据库的额外能力：

  • 向量索引的优化存储：向量数据以紧凑的二进制格式存储，支持内存映射（mmap），最大化 I/O 吞吐
  • 标量过滤与向量检索的融合：在向量检索的同时过滤元数据条件（如”只检索 2024 年之后的文档”），这比”先向量检索再应用过滤”高效得多
  • CRUD 与索引动态更新：支持增删改向量而不需要重建整个索引
  • 水平扩展：原生支持分片（Sharding）和副本（Replication），数据量增长时可以横向扩展

3. 向量数据库在 AI 架构中的位置

在典型的 RAG（Retrieval-Augmented Generation，检索增强生成）架构中，向量数据库处于检索层的核心位置：

向量数据库不是一个”存了就完”的组件。它的检索质量直接决定了 RAG 系统的召回率（Recall）——即相关文档中有多少比例被成功检索到。如果向量数据库的检索召回率低，后面的 LLM 再强也无法给出正确答案，因为”巧妇难为无米之炊”。

二、向量存储与数据模型

1. 向量数据库的基本数据模型

向量数据库的数据模型通常围绕以下几个核心概念构建：

Collection（集合）：向量数据库中最高层级的组织单位，类似于关系数据库中的”表”（Table）。一个 Collection 包含一组具有相同维度和相同 Schema 的向量数据。例如，一个 RAG 应用可能创建 documents 和 images 两个 Collection，分别存储文档 Embedding 和图片 Embedding。

Vector（向量）：Collection 的核心字段，是固定维度的浮点数数组。每个向量代表一条数据在语义空间中的位置。向量的维度由 Embedding 模型决定，一旦 Collection 创建并指定了维度，就不能再改变。

Payload / Metadata（负载/元数据）：除向量外，每条数据可以附带结构化字段（标量数据），如标题、作者、创建时间、分类标签等。这些元数据有两个关键作用：一是在检索时支持标量过滤（”只返回分类为’技术文档’的结果”），二是在检索结果返回后提供上下文信息（显示给用户看的文档标题和来源）。

ID（标识符）：每条向量数据的唯一标识，通常使用 UUID 或业务自定义 ID。ID 用于向量的增删改操作。

2. 向量存储格式

向量数据库在底层需要高效地存储高维浮点数数组。存储格式的选择直接影响 I/O 性能和内存占用。

Float32（FP32）：每个维度使用 4 字节 IEEE 754 单精度浮点数。这是最常见也是默认的存储格式。1536 维向量占用 6KB。精度无损，适用于大多数场景。

Float16（FP16 / 半精度）：每个维度使用 2 字节。存储空间减半，但精度有所损失。现代 GPU 原生支持 FP16 运算，在某些 GPU 加速向量数据库中可以获得更好的吞吐。

Binary（二值化向量）：将每个维度的浮点数量化为 0 或 1（根据符号或阈值），存储空间大幅降低（每维度 1 bit），检索使用汉明距离（Hamming Distance）。适用于对精度要求不高但数据量极大的场景（如数十亿级去重）。

Scalar Quantization（标量量化）：将 Float32 映射为 Int8（1 字节/维度），存储空间降为原来的 1/4。大多数向量数据库支持此选项，精度损失可控（通常 1%-3%）。

存储格式 每维度字节数 1536维向量大小 精度 适用场景
Float32 4 6 KB 无损 默认选择
Float16 2 3 KB 微量损失 GPU 加速场景
Int8（标量量化） 1 1.5 KB 1%-3% 损失 大规模低成本存储
Binary 1 bit 192 B 损失较大 数十亿级去重/粗筛

存储容量评估公式：

例如，100 万条 1536 维 Float32 向量，元数据平均 500 字节：

  • 向量数据：100万 × 1536 × 4B = 5.76 GB
  • 元数据：100万 × 500B = 0.5 GB
  • 索引开销：约 2-3 GB
  • 总计：约 8-10 GB

3. 标量存储与元数据管理

向量数据库不仅仅存储向量，还需要高效管理元数据（标量字段）。元数据管理的两个核心能力是标量索引和过滤下推。

标量索引：向量数据库通常为元数据字段建立倒排索引或 B-Tree 索引，以加速标量过滤条件。例如，WHERE category = '技术文档' 会使用倒排索引快速定位到满足条件的向量。

过滤下推（Predicate Push-Down）：在向量检索之前先应用标量过滤，缩小检索范围。高效的向量数据库会同时利用标量索引和向量索引，在索引遍历过程中就排除不满足标量条件的向量，而不是”先搜出 1000 条再过滤掉 900 条”。

元数据 Schema 设计建议：

  • 为高频过滤字段（如 category、status）建立标量索引
  • 避免将大段文本放入元数据，元数据应仅包含检索过滤和结果展示所需的最小字段集合
  • 枚举值优于自由文本（category: "technical" 比 description: "这是一篇技术文档..." 更适合做过滤条件）

三、向量索引：ANN 近似最近邻搜索

1. 精确检索 vs 近似检索

精确最近邻检索（KNN, K-Nearest Neighbors） 是对数据库中所有向量逐一计算与查询向量的距离，然后排序返回 TopK。精确检索的精度是 100%，但复杂度是 O(n × d)，其中 n 是向量总数，d 是维度。当 n 达到百万级时，KNN 已经不可接受。

近似最近邻检索（ANN, Approximate Nearest Neighbors） 的核心思想是：牺牲少量精度，换取数量级的性能提升。ANN 不扫描所有向量，而是通过精巧的索引结构快速定位到高概率包含最近邻的局部区域，只在该区域内做精确距离计算。

ANN 算法的核心权衡是精度 vs 速度 vs 内存的三角矛盾：

任何一个 ANN 算法在这三个维度上都有取舍。例如：

  • HNSW：精度高、速度快，但内存开销大（需要存储图结构）
  • IVF-PQ：内存省，但精度和速度相对较低
  • DiskANN：内存极省（依赖磁盘），但速度较慢

2. 主流向量索引算法

2.1 基于图的索引：HNSW

HNSW（Hierarchical Navigable Small World，分层可导航小世界图）是当前最广泛使用的向量索引算法，被 Milvus、Qdrant、Weaviate 等几乎所有向量数据库支持。

原理：HNSW 构建一个多层图结构。每一层是一个近邻图（每个向量连接其最近的 M 个邻居），层级越高节点越稀疏。检索时，从最高层（最稀疏的层）开始，沿着图结构快速下降到最底层（最稠密的层），在最底层进行精细搜索。

HNSW 的核心参数：

参数 含义 建议值 影响
M 每个节点的最大连接数 16-64 M 越大，精度越高，但内存和构建时间增加
efConstruction 构建时的搜索宽度 100-500 越大精度越高，构建越慢
efSearch 检索时的搜索宽度 64-512 越大精度越高，检索越慢

HNSW 的特点：

  • 增量构建：支持在线插入新向量，不需要重建索引
  • 检索速度极快：在内存中的检索延迟通常在毫秒级
  • 高召回率：通常可以达到 95%-99% 的召回率
  • 内存开销大：图结构本身占用额外内存，约为向量数据的 100%-200%

2.2 基于量化的索引：IVF-PQ

IVF-PQ（Inverted File with Product Quantization，倒排文件+乘积量化）是将聚类和量化结合的两阶段索引。它是大规模向量检索场景中性价比最高的方案之一。

原理：

  • IVF（Inverted File）：使用 K-Means 将所有向量聚类为 N 个簇（Cluster），每个簇有一个质心（Centroid）。检索时，只搜索与查询向量最近的几个簇（nprobe 参数控制搜索的簇数量），而不是全部簇
  • PQ（Product Quantization）：将高维向量切分为多个子段，每个子段独立做聚类和量化。向量不存储原始浮点数，而是存储每个子段所属的聚类 ID（一个很小的整数），极大压缩了存储
参数 含义 影响
nlist（簇数） K-Means 聚类数量 越大检索越快，但精度越低；推荐 sqrt(n) ~ 4 × sqrt(n)
nprobe 检索时搜索的簇数 越大精度越高，速度越慢
M（PQ 子段数） 向量切分为多少子段 必须是维度的因子；越大精度越高，压缩率越低

IVF-PQ 的特点：

  • 存储压缩率高（可以将 1536 维 Float32 向量从 6KB 压缩到几十字节）
  • 适合离线批量构建（K-Means 聚类需要全量数据），不支持高效的在线增量插入
  • 精度低于 HNSW，但对于超大规模场景（十亿级），往往是唯一可行的方案

2.3 基于树的索引：Annoy

Annoy（Approximate Nearest Neighbors Oh Yeah）是 Spotify 开源的基于树的 ANN 算法。原理是递归地使用随机超平面将向量空间二分，构建一棵二叉树。每棵树的构建是随机的，通过构建多棵树来提高检索精度。

Annoy 的特点是实现简单、内存友好，但精度和速度都不如 HNSW，现代向量数据库已较少将其作为默认索引。

2.4 基于哈希的索引：LSH

LSH（Locality-Sensitive Hashing，局部敏感哈希）是最早的 ANN 算法之一。原理是通过哈希函数将向量映射到不同的哈希桶中，使得相似的向量有更高的概率被映射到同一个桶。检索时只计算同一桶内的向量距离。

LSH 的优点是理论优雅、支持动态增删，但精度明显低于 HNSW 和 IVF-PQ，在现代向量数据库中已很少作为主力索引。

四、相似度度量与检索策略

1. 相似度度量

向量数据库支持多种相似度（距离）度量方式。度量方式的选择取决于 Embedding 模型的训练方式——大多数现代 Embedding 模型使用余弦相似度（Cosine Similarity）进行训练，因此将其作为默认推荐。

余弦相似度（Cosine Similarity）：衡量两个向量在方向上的相似程度，值域 [-1, 1]，1 表示完全相同方向，0 表示正交（无关），-1 表示完全相反。余弦相似度消除了向量长度的影响，只关注方向——这对于文本 Embedding 特别重要，因为文本长度不应影响语义相似度判断。

欧氏距离（Euclidean Distance / L2）：衡量两个向量在空间中的直线距离，值域 [0, ∞)，0 表示完全相同。欧氏距离受向量长度影响大，适用于向量长度本身有意义的场景（如图像特征向量）。

内积（Inner Product / Dot Product）：向量点积，值域 (-∞, ∞)。内积同时受向量方向和长度影响。某些经过归一化训练的 Embedding 模型使用内积作为相似度度量。对于归一化向量（长度为 1），内积等于余弦相似度。

度量方式 值域 受向量长度影响 适用场景
余弦相似度 [-1, 1] 否 文本 Embedding（推荐）
欧氏距离 [0, ∞) 是 图像特征、位置坐标
内积 (-∞, ∞) 是 归一化向量、推荐系统

在向量数据库中选择度量方式的实践原则：与 Embedding 模型的训练目标保持一致。OpenAI、BGE、Cohere 等主流文本 Embedding 模型都使用余弦相似度，因此 RAG 应用几乎统一使用余弦相似度。

2. TopK 检索

TopK 检索是向量数据库最基本的查询操作：给定一个查询向量，返回相似度最高的 K 个结果。

TopK 的参数设计：

  • K 值太小（如 K=1 或 K=2）：可能遗漏相关信息，特别是在知识库中有多篇内容相关的文档时
  • K 值太大（如 K=50 或 K=100）：虽然不容易遗漏信息，但会引入大量噪声，增加 LLM 的推理成本和干扰。且超大的 K 值意味着更多的 Context Window 消耗
  • K 值的”双重作用”：K 不只控制返回数量，它还是检索深度和搜索结果多样性的控制参数

对于 RAG 场景，推荐的 K 值范围是 3-10。具体选择取决于：

  • Chunk 大小：Chunk 越小，K 可以稍微增大
  • 知识库规模：知识库越大，信息来源越分散，K 需适当增大
  • LLM Context Window：窗口越大，可以接受更多的检索结果

相似度阈值（Score Threshold）：除了限制数量，还可以限制质量——只返回相似度高于某个阈值的结果。例如设置阈值为 0.7，排除相似度低于 0.7 的结果。这样即使设定 K=10，如果只有 3 条结果超过阈值，就只返回 3 条。阈值过滤可以防止 LLM 基于不相关内容”强行作答”。

3. Metadata Filter（标量过滤）

纯语义检索在某些场景下不足——例如用户只想搜索”2024 年之后的文档”或”某个特定作者的文章”。这些条件无法通过语义向量表达，需要标量过滤与语义检索结合。

Metadata Filter（标量过滤）的核心能力：

过滤策略的两种实现方式：

  • Pre-filtering（前置过滤）：先用标量索引筛选出满足条件的候选集，再在候选集中做向量检索。优点是保证结果一定满足过滤条件；缺点是如果过滤条件筛选出的候选集非常小，可能影响语义检索的多样性
  • Post-filtering（后置过滤）：先做向量检索获取 TopK 结果，再从结果中过滤掉不满足标量条件的。缺点是可能过滤后结果不足 K 个——极端情况向量检索返回的 K 个结果全都不满足过滤条件

现代向量数据库（Milvus、Qdrant 等）已经实现了融合过滤——在向量索引遍历的过程中同步应用标量过滤，兼顾效率和结果质量。

4. Hybrid Search（混合搜索）

Hybrid Search（混合搜索）是同时使用语义检索（向量相似度）和关键词检索（倒排索引/BM25）的搜索策略，融合两者的优势。

为什么需要混合搜索：

  • 纯向量检索在长尾查询、精确名称匹配（如型号”ThinkPad T480”）上表现欠佳——这些查询包含独特的专有名词，向量模型可能没有充分学习
  • 纯关键词检索无法理解语义——用户搜”笔记本散热问题”，关键词匹配找不到”笔记本电脑温度过高”的文档
  • 混合搜索结合两者：关键词检索负责精确匹配（专有名词、代码、编号），向量检索负责语义覆盖（同义词、语义相近的表达）

5. 多向量检索与重排序

多向量检索（Multi-Vector Retrieval）：对于某些复杂场景，单一的查询向量不足以表达用户的检索意图。可以生成多个不同角度的查询向量，分别检索后合并结果。例如，用户的原始问题是”如何优化 Spring Boot 应用的启动速度”，可以生成两个变体查询——“Spring Boot 启动优化方法”和”Spring Boot 慢启动原因分析”——分别检索后合并去重。

重排序（Re-ranking）：向量检索的 TopK 结果是基于向量相似度的初步排序。在某些高精度场景中，可以加一个重排序步骤——用一个更强大的模型（如 Cross-Encoder）对 TopK 结果重新打分排序。Cross-Encoder 同时接收查询和文档作为输入，输出精确的相关性分数，比向量相似度更准确，但计算成本也更高。

这种”粗筛+精排”的两阶段策略在 Google、Bing 等搜索引擎中广泛使用，也适用于对检索质量要求极高的企业 RAG 系统。

五、主流向量数据库方案对比

1. 专用向量数据库

1.1 Milvus

Milvus 是 LF AI & Data 基金会旗下的开源向量数据库（由 Zilliz 主导开发），是目前功能最全面、社区最活跃的向量数据库之一。采用云原生架构，支持存算分离、多租户和数据持久化。

核心特性：

  • 支持多种索引类型：HNSW、IVF-Flat、IVF-PQ、DiskANN 等 10+ 种索引
  • 支持标量过滤、混合搜索（向量 + BM25）、范围检索、多向量字段
  • 原生支持分片（Sharding）和副本（Replication），可水平扩展
  • SDK 覆盖 Python、Java、Go、Node.js 等主流语言
  • 提供 Milvus Lite（嵌入式版本）和 Zilliz Cloud（全托管版本）

架构层次：

Milvus 适合需要存算分离、大规模扩展的生产环境。部署复杂度较高（需要 ETCD、MinIO、Pulsar 等中间件），但 Milvus Lite 版本降低了入门门槛。

1.2 Qdrant

Qdrant 是用 Rust 语言编写的开源向量数据库，以单二进制部署、API 设计优雅、性能出色著称。

核心特性：

  • 单二进制文件部署，无需外部依赖，运维成本极低
  • 默认使用 HNSW 索引，检索性能优秀
  • 原生支持稀疏向量（Sparse Vector）和稠密向量（Dense Vector），可以原生实现混合搜索
  • 支持 Payload 标量过滤、全文检索、地理位置检索
  • 内置 Snapshots 快照备份和恢复

与 Milvus 的部署复杂度对比：

Milvus（分布式版） Qdrant
部署依赖 ETCD + MinIO + Pulsar 无（单二进制）
最小内存 8 GB+ 256 MB
水平扩展 原生支持 Qdrant Cloud 支持
适合场景 大规模生产环境 中小规模 / 快速启动

Qdrant 特别适合中小规模（千万级以下）的向量检索场景，以及在 Docker Compose 或 K8s 中快速搭建的私有化部署需求。

1.3 Weaviate

Weaviate 是一个同时支持向量检索和 GraphQL 查询的开源向量数据库，特色是支持原生混合搜索和多模态数据。

核心特性：

  • Hybrid Search 原生内置：不需要额外组件，单次查询即可融合 hybrid 搜索（向量 + BM25），通过 alpha 参数调节权重
  • 多模态支持：内置多模态模块，可以直接存储和检索图片、文本、音频
  • GraphQL API：查询语言基于 GraphQL，表达能力强
  • 模块化设计：通过插件模块集成外部 Embedding 模型（如 OpenAI、Cohere、HuggingFace），能在数据写入时自动向量化

Weaviate 的设计哲学是”一站式 AI 数据基础平台”，适合需要密集使用混合搜索和多模态数据的场景。其 GraphQL 查询方式和模块化 Embedding 集成对于快速开发非常友好。

1.4 Pinecone

Pinecone 是一个全托管的云向量数据库服务，不开源，不可私有化部署。

核心特性：

  • 零运维负担，创建 Index 后即可使用
  • 自动索引优化和扩缩容
  • 集成主流 Embedding 提供商和 LLM 框架
  • 支持 Serverless 计费模式

Pinecone 适合希望完全免运维、愿意使用云托管服务的团队。缺点在于数据无法本地化部署（合规性受限），且大规模使用成本可能显著高于自建方案。

2. 传统数据库的向量扩展

2.1 pgvector

pgvector 是 PostgreSQL 的向量检索扩展插件，让你可以在现有的 PostgreSQL 中直接存储和检索向量，无需引入新的数据库。

核心特性：

  • 在 PostgreSQL 中新增 vector 数据类型和 ivfflat、hnsw 索引
  • 支持余弦相似度、欧氏距离、内积三种距离度量
  • 向量检索可以与 SQL 的 WHERE、JOIN、ORDER BY 等标准语法无缝结合
  • 利用 PostgreSQL 的事务、备份、权限管理等成熟基础设施

pgvector 的优势与局限：

优势 局限
零额外部署成本（已有 PG 即可） 索引构建速度不如专用向量数据库
SQL + 向量检索天然融合 十亿级规模不如 Milvus 等专业方案
事务保证（ACID） HNSW 索引内存占用较高
运维成熟（备份、监控、HA） 缺少多向量字段、混合搜索等高级功能

pgvector 对于已经在使用 PostgreSQL 且向量检索规模在千万级以下的 Java 后端项目，是性价比最高的选择。

2.2 Elasticsearch Vector

Elasticsearch 从 8.0 版本开始原生支持向量检索（dense_vector 字段类型和 knn 查询）。

核心特性：

  • 在 Elasticsearch 中存储 dense_vector 字段，支持 HNSW 索引
  • 向量检索 + ES 原生查询（全文检索、聚合、过滤）可以在一条查询中融合
  • ES 的分布式架构提供天然的水平扩展能力
  • 将向量检索、全文检索、日志分析、指标聚合统一到一个平台上

Elasticsearch 向量检索适合已经使用 ELK 技术栈、需要在同一个平台上同时做传统搜索和语义搜索的团队。但 ES 的向量检索性能不如专用向量数据库，且内存开销较大。

2.3 Redis Vector Search

Redis 从 Redis Stack（Redis 7.2+）开始提供向量检索能力，通过 RediSearch 模块实现。

核心特性：

  • 使用 FLAT（暴力）或 HNSW 索引
  • 低延迟（Redis 是内存数据库，检索延迟极低）
  • 适合实时性要求极高的场景（如推荐系统的在线检索召回）
  • 与 Redis 的 Hash、JSON 等数据结构结合使用

Redis Vector Search 适合对延迟极其敏感的场景（毫秒级甚至亚毫秒级），但由于所有数据在内存中，不适合海量低成本存储。

七、向量数据库的工程实践要点

1. 索引构建的性能调优

向量索引的构建是一个计算密集型任务，涉及大量的距离计算和图结构构建。对于大规模数据入库（百万级以上），索引构建可能成为瓶颈。

批量入库的建议策略：

  • 分批写入：每批 1000-5000 条向量，批次之间给索引构建留出时间
  • 先写入后建索引：大批量写入时，先关闭索引构建，等数据全部写入后再触发全量索引构建（大部分向量数据库支持此模式）
  • 索引参数的分级设置：初始入库时降低 efConstruction 等参数以加快构建速度；入库完成后在线调整参数提高检索精度

2. 数据一致性考量

向量数据库在数据一致性模型上各有不同：

  • pgvector：享有 PostgreSQL 的 ACID 事务保证，向量和元数据写入是事务性的
  • Milvus：默认是最终一致性，flush 后数据可查
  • Qdrant：写入后立即可查（单节点模式）
  • Elasticsearch Vector：近实时（默认每秒 refresh）

对于需要强一致性的场景（如文档删除后必须立即在检索结果中不可见），pgvector 是唯一保证 ACID 的选择。
