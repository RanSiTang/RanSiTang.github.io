    一、核心容器

    1. 设计理念与实现

    1.1 控制反转

    1.1.1 核心概念

    控制反转 (Inversion of Control,IoC) 并不是 Spring 特有的，而是一种面向对象编程的设计原则。

    • 定义：IoC 是指将对象的创建、初始化、销毁以及对象间依赖关系的管理权，从程序代码移交给外部容器来接管。

    • 别名：好莱坞原则 (The Hollywood Principle) —— “Don’t call us, we’ll call you.”（别找我们要角色，有合适的我们会给你打电话）。在软件中意味着：别自己 new 对象，容器会把对象送到你手里。

    • 传统的程序设计：当对象 A 需要依赖对象 B 时，A 通常会在内部主动创建 B（例如 B b = new B()）。此时，A 既是使用者，也是创建者，A 控制了 B 的生命周期。

    • IoC 的程序设计：对象 A 不再负责创建对象 B，而是由一个外部的“容器”来负责创建 B，并把 B 注入给 A。

    • 反转了什么？

      • 反转了资源的获取方式：从“主动索取”变成了“被动接受”。
      • 反转了控制权：对象的创建、初始化、销毁的控制权，从“应用程序代码”转移到了“外部容器/框架”。

    1.1.2 核心价值

    • 解耦：
      • 痛点：若 A 类直接 new B()，当 B 的实现类变成 C 时，必须修改 A 的源代码并重新编译。
      • 解决：A 只依赖 B 的接口，具体注入 B 还是 C 由配置决定，A 的代码无需改动。
    • 简化对象管理：
      • 痛点：在大型系统中，对象之间依赖关系错综复杂（如单例、多例、懒加载、销毁顺序），手动管理极易出错。
      • 解决：容器统一进行生命周期管理。
    • 提升可测试性：
      • 痛点：单元测试时，如果对象内部硬编码了依赖（如数据库连接），很难进行 Mock 测试。
      • 解决：通过 IoC，测试时可以轻松注入 Mock 对象。

    1.2 实现方式

    1.2.1 依赖查找(Dependency Lookup,DL)

    • 定义：容器创建好对象后，应用程序代码主动调用容器提供的 API 接口，通过名称或类型去查找需要的资源。
    • **典型场景：**Spring 的 BeanFactory.getBean("name")
    • 查找方式：
      • 依赖拖拽：从容器中把需要的 Bean 拉出来（最常见，如 BeanFactory.getBean("name")）。
      • 上下文依赖查找：在特定上下文中查找（如查找当前线程绑定的资源）。
    • 优点：在某些底层架构或无法使用注入的场景下（如静态工具类中需要获取 Bean），是唯一的解决方案。
    • 缺点：
      • 强侵入性：业务代码必须依赖 Spring 的 API（如需要引入 ApplicationContext 接口）。
      • 代码耦合：业务逻辑与容器机制绑定，难以脱离 Spring 框架独立运行或测试。

    1.2.2 依赖注入(Dependency Injection,DI)

    • 定义：组件不做任何主动查找，容器在运行期间，自动将依赖的对象“塞”到组件中。

    • 典型表现：

      • 构造器注入：public User(Service s) { this.s = s; }

      • Setter 注入：public void setService(Service s) { ... }

      • 注解注入：@Autowired, @Resource

    • 为什么 DI 是 Spring 的主流？：

      • 零侵入性：代码中可以是纯粹的 POJO，不需要 import org.springframework...。

      • 符合单一职责原则：对象只负责业务逻辑，配置和组装交给专门的配置类。

    • 优点：

      • 非侵入式：业务代码中几乎不需要出现 Spring 的 API 类（是一个纯净的 POJO），代码整洁。
      • 易于测试：可以直接 new 对象并通过构造器传入 Mock 对象，无需启动容器。

    • 缺点：在极端复杂的依赖链中，静态排查流程相对困难（因为对象是运行时动态组装的）。

    1.2.3 DI与DL的对比总结

    维度 依赖查找 (DL) 依赖注入 (DI)
    主动性 主动：Bean 主动调用 getBean() 被动：Bean 被动等待容器赋值
    侵入性 高：业务代码依赖 Spring 核心接口 低：业务代码通常仅依赖 Java 语言特性或标准注解
    代码依赖 依赖容器 API 依赖具体对象或接口
    可读性 流程明确，但代码冗余 简洁，但依赖关系隐藏在元数据/注解中
    使用场景 框架底层、遗留系统集成、工具类 绝大多数业务开发场景 (99%)

    2. Spring IoC 容器架构与启动流程

    2.1 Spring IoC 容器概述

    • 定位：Spring IoC 容器是 Spring Framework 的核心引擎，是一个具体的软件程序。
    • 本质：它是一个管理 Java 对象（Bean）生命周期的环境。它将 IoC 的理论变成了代码层面的现实。
    • 核心责任：
      • 对象的实例化 (Instantiation)
        • 容器读取配置元数据，利用反射（Reflection）机制，替开发者 new 出对象。
      • 对象的装配/注入 (Wiring/DI)
        • 这是容器最核心的工作。它负责解析对象之间的依赖关系（A 依赖 B），并将 B 准确地赋值给 A。
        • 这就是“将分散的组件组装成协同工作的系统”的过程。
      • 生命周期管理 (Lifecycle Management)
        • 容器就像保姆，负责调用对象的初始化回调（如 @PostConstruct）和销毁回调（如 @PreDestroy）。
        • 它确保对象在被使用前是“就绪状态”，在容器关闭前能“优雅退场”。
      • 对象配置 (Configuration)
        • 容器不仅管理依赖，还管理对象的属性值（如数据库连接 URL、超时时间等配置参数）。

    2.2 核心接口体系

    2.2.1 概览

    Spring IoC 容器并非只有一个具体的实现，而是通过一系列接口来定义的。分为两条主线：

    1. 引擎：以 BeanFactory 为核心，纯粹关注 Bean 的定义、实例化和依赖注入。
    2. 门面：以 ApplicationContext 为核心，面向现代开发者，在引擎基础上增加了企业级功能（环境配置、事件流、国际化）。

    2.2.2 根接口：BeanFactory

    • 定位：Spring 容器的根接口，定义了访问容器最基本的方法。
    • 核心能力：
      • 查找：getBean(String name), getBean(Class<T> type)。
      • 判断：containsBean(String name), isSingleton(String name)。
    • 设计特点：
      • 懒加载 (Lazy Initialization)：默认情况下，只有当调用 getBean 时，才会实例化对象。
      • 低级容器：主要供框架内部使用，或者资源极受限的场景。

    2.2.3 BeanFactory 的三大子接口

    为了避免 BeanFactory 接口过于臃肿，Spring 将功能拆分到了三个子接口中：

    1. ListableBeanFactory (可列表化)
      • 能力：可以枚举容器中所有的 Bean。
      • 场景：当需要获取某种类型的所有实现类时（例如：getBeansOfType(plugin.class)），或者查看容器里到底定义了多少 Bean (getBeanDefinitionCount)。
      • 注：它只能看当前容器，看不到父容器的 Bean。
    2. HierarchicalBeanFactory (有层级)
      • 能力：支持父子容器架构。隔离规则：子容器可见父容器的 Bean，父容器不可见子容器的 Bean。
      • [老版本 SSM 时代]：绝对的核心机制。以前 Spring 管理 Service/DAO 构成父容器（ContextLoaderListener），Spring MVC 管理 Controller 构成子容器（DispatcherServlet）。
      • [新版本 Spring Boot 时代]：已经被弱化。Spring Boot 内嵌 Web 容器后，默认采用单一统一容器，不再搞父子容器隔离。
      • [现代残存场景]：目前主要在 Spring Cloud 中使用（如 OpenFeign、LoadBalancer 会为每个微服务客户端创建一个独立的子容器以隔离配置）。
    3. **AutowireCapableBeanFactory (具备装配能力) **
      • 能力：提供创建 Bean、自动注入 Bean、应用 BeanPostProcessor 的能力。
      • 场景：非常底层。通常用于让 非 Spring 管理的对象（如第三方框架 Struts/Quartz 创建的对象）也能享受到 Spring 的依赖注入（autowireBean(existingBean)）。
      • 警告：应用代码很少直接使用此接口。

    2.2.4 核心实现：DefaultListableBeanFactory

    • 定位：它是 BeanFactory 接口体系的默认实现类，也是 Spring 容器的底层底座。
    • 核心地位： 几乎所有的ApplicationContext具体实现类，内部都持有一个 DefaultListableBeanFactory 实例 来存储 Bean 定义和单例对象。
    • 它实现了所有接口：既是 Listable 的，也是 AutowireCapable 的，还实现了 BeanDefinitionRegistry（注册表接口）。

    2.2.5 高级接口：ApplicationContext

    • 定位：这它是一个聚合接口。它不干活，它只是把各种能力组合起来提供给用户。
    • 扩展能力：
      1. EnvironmentCapable：获取环境配置（Profile, Properties）。
      2. MessageSource：国际化 (i18n) 支持。
      3. ApplicationEventPublisher：事件发布机制（观察者模式）。
      4. ResourcePatternResolver：统一的资源加载机制（如 classpath:, file:）。
    • 与 BeanFactory 的核心区别：
      • 预加载（Pre-loading）：默认情况下，ApplicationContext 在启动时就会实例化所有 Singleton Bean（空间换时间，尽早发现配置错误）。

    2.2.6 生命周期接口：ConfigurableApplicationContext

    • 定位：ApplicationContext 的子接口，增加了容器的配置和生命周期控制能力。
    • 核心方法：
      • refresh()：Spring 启动的核心方法。加载配置、初始化 BeanFactory、注册 BeanPostProcessor、实例化单例 Bean 等全套流程都在这里。
      • close()：关闭容器，触发销毁回调。
      • registerShutdownHook()：注册 JVM 关闭钩子，确保 Web 应用停止时优雅销毁 Bean。

    2.2.7 BeanFactory vs ApplicationContext

    特性 BeanFactory ApplicationContext
    加载机制 懒加载 (Lazy)：调用 getBean 时才实例化。 预加载 (Eager)：容器启动(refresh)时实例化所有单例 Bean。
    功能丰富度 仅维护 Bean 之间的依赖关系 (DI)。 包含 BeanFactory 所有功能，且增加了 AOP、国际化、事件机制、环境抽象等。
    性能体验 启动快，但第一次请求慢。 启动慢（耗内存），但运行时响应快（推荐）。
    注册方式 手动注册 BeanPostProcessor。 自动检测并注册 BeanPostProcessor 和 BeanFactoryPostProcessor。
    适用场景 资源受限的移动设备或 Applet（已过时）。 绝大多数企业级应用、Web 应用。

    2.4 ApplicationContext 核心实现类

    2.4.1 AbstractApplicationContext (抽象基类)

    • 定位：Spring 容器的“骨架”，几乎所有具体的 ApplicationContext 实现类都继承自它。
    • 核心作用：
      • 它实现了 ApplicationContext 接口的大部分公共逻辑。
      • 设计模式：使用了 模板方法模式。定义了容器启动的标准流程——即 refresh() 方法。无论子类是基于 XML 还是注解，最终都会调用这个父类的 refresh() 来完成容器的初始化。
      • 承上启下：它实现了 ConfigurableApplicationContext 接口的 refresh() 方法，并内部持有了 DefaultListableBeanFactory 来干活。这个持有是通过 getBeanFactory() 抽象方法获取DLBF；而真正 new 并在成员变量中持有 DLBF 的，是它的核心子类（如 GenericApplicationContext）。

    2.4.2 AnnotationConfigApplicationContext (现代标准)

    1. 核心定位
    • 定位：Spring Boot 和现代纯 Java Config 项目的基石。
    • 核心特点：不依赖 XML，完全通过 注解 (@Configuration, @Component) 扫描。
    • 使用场景：非 Web 环境的 Spring Boot 应用。
    2. 核心架构与成员

    AnnotationConfigApplicationContext 内部通过组合两个核心组件来完成工作，而非继承。

    3. 构造函数与初始化原理

    核心深度解析：new AnnotatedBeanDefinitionReader(this) 做了什么？

    这一步非常关键。它会调用 AnnotationConfigUtils.registerAnnotationConfigProcessors()，向容器中注册 6 个系统级的 BeanPostProcessor (BPP)。

    其中最重要的两个：

    1. ConfigurationClassPostProcessor (CCPP)：绝对核心。负责解析 @Configuration, @Bean, @Import, @ComponentScan。没有它，Spring 就看不懂配置类。
    2. AutowiredAnnotationBeanPostProcessor：负责解析 @Autowired, @Value。

    结论：调用无参构造后，容器虽然是空的，但处理注解的能力已经就绪了。

    4. 核心方法：注册与扫描

    • register(Class<?>... componentClasses)

      • 作用：将手动传入的类（通常是启动配置类，如 AppConfig.class）注册为 BeanDefinition。

        • 流程：
          1. 包装：将 Class 包装为 AnnotatedGenericBeanDefinition。
          2. 解析通用注解：解析 @Scope, @Lazy, @Primary 等元数据。
          3. 注册：存入 BeanFactory 的 beanDefinitionMap。

      • 注意：此时，只注册了配置类本身，配置类里定义的 @Bean 或者 @ComponentScan 扫到的包，还没开始解析。

    • scan(String... basePackages)

      • 作用：扫描指定包路径。

      • 底层：利用 ASM 字节码技术 (MetadataReader) 读取 class 文件，不加载类（为了性能）。

      • 过滤：只匹配包含 @Component (及其衍生注解 @Service 等) 的类。

    2.4.3 AnnotationConfigServletWebServerApplicationContext (Spring Boot Web)

    1. 核心定位
    • 定位：Spring Boot Web 应用的默认容器。
    • 地位：它是 Spring Boot “内嵌服务器” 魔法的核心实现者。
    • 区别：
      • 普通 Spring 容器 (AnnotationConfigApplicationContext) 只负责管理 Bean。
      • 这个容器 额外负责创建并启动 Web 服务器 (Tomcat / Jetty / Undertow)。
    2. 继承体系
    • GenericWebApplicationContext：具备了 Web 环境的能力（如注册 ServletContext, ServletConfig）。
    • ServletWebServerApplicationContext：关键父类。它定义了 createWebServer() 和 onRefresh() 模板方法，是内嵌服务器启动的逻辑所在。
    3. 核心原理：如何启动 Tomcat？

    这个容器最核心的逻辑在于它重写了 AbstractApplicationContext 的 onRefresh() 方法。

    回顾：在 refresh() 的第 9 步 onRefresh() 是个空方法，留给子类扩展。Spring Boot 正是利用这个钩子启动了 Tomcat。

    启动流程源码解析：

    1. refresh() (容器启动)
      -> 调用 super.refresh()
      -> 执行到 onRefresh() 阶段。
    2. createWebServer() (核心方法)
      • 检查：判断是否已经存在 WebServer。
      • 获取工厂：通过 getWebServerFactory() 获取容器中的 ServletWebServerFactory Bean (通常是 TomcatServletWebServerFactory)。
        • 这也是为什么我们在配置文件里写 server.port=8080 能生效的原因：Spring Boot 自动配置把参数注入到了这个工厂 Bean 里。
      • 获取服务器：调用 factory.getWebServer(getSelfInitializer())。
        • Tomcat：new Tomcat() -> 设置连接器 -> 启动引擎。
        • Jetty/Undertow：类似逻辑。
      • 注册 ServletContextInitializer：将 Spring 的 DispatcherServlet 注册到 Tomcat 的 Servlet 容器中。
    3. startWebServer() (最后一步)
      • 在 finishRefresh() 阶段，容器会调用 WebServer.start()，此时 Tomcat 真正开始监听端口，接收请求。

    组件 普通容器 (AnnotationConfigApplicationContext) Spring Boot Web 容器
    环境 StandardEnvironment StandardServletEnvironment (多了 ServletContext 参数)
    Scope Singleton, Prototype 多了 Request, Session, Application
    启动项 仅刷新 BeanFactory 刷新 BeanFactory + 启动 Tomcat
    Bean 只有业务 Bean 自动注册 DispatcherServlet, CharacterEncodingFilter 等 Web 组件

    2.4.4 传统XML时代

    1. ClassPathXmlApplicationContext: 从类路径加载XML

    • 加载路径：从 Java 的 类路径 (Classpath) 加载配置文件。

    • 类路径：在 Java 中，类路径是一个“资源搜索目录列表”。

      • 机制：当请求资源时，JVM 会按顺序在这些目录下查找。
      • 在典型的Maven/Gradle项目中：
        • 源码 (src/main/java) 编译后的 .class 文件会被自动放入类路径。
        • 资源目录 (src/main/resources) 下的所有文件（如 application.xml）都会被构建工具（Maven/Gradle）自动复制到最终的类路径根目录下。

    • 特点：

      • 这是 XML 时代最标准的启动方式。
      • 路径通常写成 classpath:applicationContext.xml。
      • 只要配置文件在 src/main/resources 下或打进 jar 包里，它都能找到。
      • 可移植性强：项目移动位置不影响启动。
    2. FileSystemXmlApplicationContext: 从文件系统加载XML
    • 加载路径：从操作系统的 文件系统 (File System) 加载。
    • 特点：
      • 需要指定绝对路径，例如 D:/project/config/beans.xml 或 /etc/opt/app/beans.xml。
      • 耦合度高：如果换台电脑或服务器，路径变了，程序就起不来了。
      • 用途：通常用于加载外部独立于 Jar 包之外的配置文件。

    2.5 容器启动核心流程：refresh() 源码解析

    2.5.1 核心流程

    2.5.2 四大阶段

    阶段一：准备工作与容器预热
    1. prepareRefresh()：环境准备（校验必填属性，记录启动时间）。
    2. obtainFreshBeanFactory()：获取底层的引擎 DefaultListableBeanFactory (DLBF)。
    3. prepareBeanFactory()：配置 DLBF 的标准特征（类加载器、忽略某些自动装配接口等）。
    4. postProcessBeanFactory()：提供给子类的扩展点（Web 容器会在这注册 Request/Session 作用域）。
    阶段二：核心扫描与图纸解析
    1. invokeBeanFactoryPostProcessors()（超级重点）：执行工厂后置处理器。Spring Boot 的 @ComponentScan 扫描和 @Configuration 解析全在这里完成，生成所有的 BeanDefinition 图纸。
    阶段三：注册拦截器与基础设施
    1. registerBeanPostProcessors()：实例化并注册所有的 BeanPostProcessor（处理 @Autowired 和 AOP 的核心组件），此时不执行，只注册。
    2. initMessageSource()：初始化国际化组件。
    3. initApplicationEventMulticaster()：初始化事件广播器（事件总线）。
    4. onRefresh()（超级重点）：模板方法。Spring Boot 在这里启动内嵌的 Tomcat / Undertow。
    5. registerListeners()：将监听器注册到广播器中，并发布早期事件。
    阶段四：大批量制造 Bean 与收尾
    1. finishBeanFactoryInitialization()（超级重点）：大批量制造工厂。实例化所有非懒加载的单例 Bean，完成依赖注入（DI）和 AOP 代理。
    2. finishRefresh()：收尾工作。清理缓存，发布 ContextRefreshedEvent（容器刷新完毕事件）。

    2.5.3 核心详解

    重点关注步骤 5、9、11。这三个步骤是日常 Bug 排查、性能优化和框架扩展的核心。

    1. obtainFreshBeanFactory()：获取底层工厂
    • 动作：返回我们在上一节讲过的 DefaultListableBeanFactory。
    • 【新老版本核心差异】：
      • 老版（XML时代）：这里会销毁旧容器，创建新容器，并极其沉重地执行 XML 文件的解析，把BeanDefinition加载进来。
      • 现代 Spring Boot：基于 GenericApplicationContext，这一步极其轻量。工厂在 new 的时候已经创建好了，这里仅仅是设置一个序列化 ID 并把它直接 return 出来。现代的图纸解析被后移到了第 5 步。
    2. invokeBeanFactoryPostProcessors()：解析配置与组件扫描

    • 定位：这是 Spring 启动过程中第一个耗时大户。此时，所有的 Bean 尚未创建，容器里只有“图纸”。

    • 核心逻辑：调用所有的 BeanFactoryPostProcessor。

    • 现代开发实战真相：
      Spring Boot 的核心魔法类 ConfigurationClassPostProcessor 就在这里执行！
      它会去解析你的启动类上的 @SpringBootApplication，顺藤摸瓜执行 @ComponentScan（扫包），解析所有的 @Bean、@Import、@Component，并将它们全部转换为 BeanDefinition 注册到容器中。

    3. registerBeanPostProcessors()：注册 Bean 的增强器
    • 定位：为后续的 Bean 实例化准备“加工机器”。
    • 核心逻辑：去工厂里把所有实现了 BeanPostProcessor 接口的类找出来，实例化它们，并放进一个 List 里。
    • 重要组件举例：
      • AutowiredAnnotationBeanPostProcessor：用来处理 @Autowired 依赖注入的。
      • AnnotationAwareAspectJAutoProxyCreator：用来织入切面，生成 AOP 动态代理的。
    • 注意：这里只是把机器“造”出来放好，真正处理对象是在第 11 步。
    4. onRefresh()：启动 Web 服务器

    • 定位：这是一个留给子类实现的空模板方法（钩子方法）。

    • 现代开发实战真相：
      “Spring Boot 的 Tomcat 是什么时候启动的？” 答案就是这里。
      在 Spring Boot 的 ServletWebServerApplicationContext 中，重写了这个方法。它会在这里调用 createWebServer()，真正的去启动内嵌的 Tomcat、Jetty 或 Undertow。

      为什么放在这里启动？ 因为此时核心配置已解析完毕，但用户的 Bean 还没开始大批量创建。提前把 Web 容器准备好，可以确保后续如果 Bean 初始化需要依赖 Web 环境时不会报错。

    5. finishBeanFactoryInitialization()：实例化非懒加载单例
    • 定位：refresh() 中最核心、最复杂、耗时最长的一步。
    • 核心动作：容器遍历所有的 BeanDefinition，只要它是单例（Singleton）且不是懒加载（Lazy），就会调用 beanFactory.getBean(beanName)。
    • Bean 的完整生命周期就在这里面展开（即著名的 doCreateBean 流程）：
      1. 实例化（Instantiation）：通过反射或 CGLIB 调用构造方法 new 出原始对象。
      2. 属性填充（PopulateBean）：处理 @Autowired、@Value，完成依赖注入。
      3. 初始化（Initialization）：调用 @PostConstruct 注解的方法、执行 InitializingBean 接口的 afterPropertiesSet()。
      4. 代理创建（AOP）：在这个阶段的最后（由第 6 步注册的 BeanPostProcessor 介入），如果发现类被 @Transactional 或自定义切面拦截，就会创建一个动态代理对象，把原始对象替换掉。
    6. 异常处理机制（The Catch Block）
    • 动作：如果在这个漫长的刷新过程中发生了任何异常（比如 @Autowired 找不到对象、Bean 循环依赖无法解决），进入 Catch 块。
    • 设计原则（Fail-fast 与原子性）：Spring 保证容器的启动是原子的。要么全部成功，要么全部失败。
    • destroyBeans()：销毁之前已成功创建的所有单例 Bean，释放资源，防止内存泄漏。

    二、Bean 的定义与注册

    1. Spring Bean 的概述

    1.1 基本定义

    • 官方定义：Spring Bean 是那些由 Spring IoC 容器实例化、组装（注入依赖）、管理的对象。
    • 本质：Bean 就是一个 Java 对象。
    • 核心区别：
      • 我们自己 new 出来的对象，叫原生对象（Raw Object）。
      • 由 Spring 容器（ApplicationContext）管理的对象，叫 Spring Bean。

    1.2 Spring Bean 与 POJO 的区别

    1.2.1 核心机制：从 POJO 到 Bean 的蜕变

    什么是 POJO：

    • 指简单的 Java 对象。它不继承任何 Spring 特定的类，也不实现任何 Spring 特定的接口。
    • 好处：代码纯净，易于测试，解耦。

    POJO 要转化为 Spring Bean，经历了一套严谨的定义与构建流程。

    1. 配置元数据
      容器需要依据“图纸”来创建 Bean。开发者通过以下三种方式提供元数据：
      • 基于 XML 的配置：使用 <bean> 标签定义。
      • 基于注解的配置：使用 @Component, @Service, @Repository, @Controller 等构造型注解。
      • 基于 Java 类的配置：在 @Configuration 类中使用 @Bean 注解的方法。
    2. BeanDefinition（Bean 定义信息）
      Spring 容器解析上述元数据后，会将其转化为内部的 BeanDefinition 对象。
    3. 实例化与管理
      容器根据 BeanDefinition，利用Java 反射机制创建实例，注入依赖，并执行初始化回调，最终将一个完全可用的 Bean 放入容器缓存池（对于单例而言）供应用使用。

    1.2.2 Bean 的关键特性

    被 Spring 托管的 Bean 具备以下核心特征，这些特征将其与普通对象区分开来：

    1. 容器托管
      • Bean 的全生命周期（创建、初始化、使用、销毁）由容器统一调度，开发者只需关注业务逻辑，无需处理底层资源管理。
    2. 唯一标识
      • 每个 Bean 在容器中拥有唯一的 ID（或 Name）。这是容器索引和检索 Bean 的句柄。
      • 注：使用注解配置时，默认 ID 为类名首字母小写（如 UserService -> userService）。
    3. 依赖注入能力
      • Bean 不需要自行查找或创建它所依赖的对象。通过声明依赖（如字段注入、构造器注入），容器会在创建 Bean 时自动完成“装配”。
    4. 作用域策略
      • Bean 的存在形式由 Scope 定义。最常见的是 Singleton（单例，容器内唯一）和 Prototype（原型，每次获取创建新实例），此外还支持 Web 环境下的 request、session 等作用域。
    5. 生命周期回调
      • Bean 可以在初始化前后（@PostConstruct）或销毁前（@PreDestroy）执行特定的逻辑，这为资源加载和释放提供了标准钩子。

    2. BeanDefinition 对象

    2.1 核心概念

    • 定义：它是 Spring Framework 中定义 Bean 的配置元数据接口。
    • 类比：
      • Class 是 Java 对象的蓝图。
      • BeanDefinition 是 Spring Bean 的蓝图。
    • 作用：Spring 容器启动时，不管是 XML、注解还是 Java Config，第一步都是将它们解析成统一的 BeanDefinition 对象，存入 BeanFactory 中。后续的实例化、属性注入、初始化，完全依赖这个对象。

    2.2 核心属性

    一个 BeanDefinition 包含了创建 Bean 所需的所有信息。以下是开发中经常接触的关键属性：

    1. BeanClassName：
      • Bean 的全限定类名（如 com.example.UserService）。
      • 注：对于 @Bean 方法定义的 Bean，这里可能是 null，实际由 factoryBeanName + factoryMethodName 决定。
    2. Scope (作用域)：
      • 决定 Bean 是单例 (singleton) 还是多例 (prototype)。
      • 对应注解：@Scope("prototype")。
    3. LazyInit (懒加载)：
      • 决定 Bean 是在容器启动时创建，还是在首次使用时创建。
      • 对应注解：@Lazy。
    4. DependsOn (依赖关系)：
      • 强制指定 Bean 的初始化顺序（当前 Bean 初始化前，必须先初始化依赖的 Bean）。
      • 对应注解：@DependsOn。
    5. ConstructorArgumentValues (构造参数)：
      • 记录了构造函数的参数值或引用。
      • 这是 Spring 推断构造函数注入的关键依据。
    6. PropertyValues (属性值)：
      • 记录了 Setter 注入的属性值（键值对）。
      • 在 XML 时代常用，注解时代较少直接操作。
    7. InitMethodName / DestroyMethodName：
      • 初始化和销毁方法名。
      • 对应注解：@Bean(initMethod="init") 或 XML <bean init-method="...">。
      • 注意：@PostConstruct 不在这里记录，它是通过 BeanPostProcessor 处理的。

    2.3 核心实现类体系

    了解实现类有助于阅读源码，开发中很少直接使用。

    • AbstractBeanDefinition：基类，定义了大部分通用属性。
    • RootBeanDefinition：
      • 历史地位：Spring 2.0 之前的主要实现，用于 XML <bean>。
      • 现代作用：合并后的 BeanDefinition（Merged BeanDefinition）。无论原始定义是哪个子类，Spring 在运行时都会将其合并为 RootBeanDefinition 来创建 Bean。
    • GenericBeanDefinition：
      • 通用实现：XML 配置和大部分动态注册场景使用。
      • 特点：灵活，可以设置 parent（继承父 Bean 配置）。
    • AnnotatedGenericBeanDefinition：
      • 注解驱动的核心：当使用 AnnotationConfigApplicationContext.register(AppConfig.class) 时，配置类会被包装成这个类型。
      • 特点：它持有一个 AnnotationMetadata，能读取类上的注解信息（如 @Conditional）。
    • ScannedGenericBeanDefinition：
      • 扫描驱动的核心：@ComponentScan 扫描到的 @Component 类，会被解析成这个类型。
      • 特点：同样持有 AnnotationMetadata。

    2.4 BeanDefinition 的注册流程

    这是 Spring 启动中最关键的“前半场”：从配置到 BeanDefinition。

    1. Resource 定位：
      • 找到配置文件（XML）或扫描路径（Package）。
    2. 加载 (Loading)：
      • XML 模式：XmlBeanDefinitionReader 解析 XML 标签。
      • 注解模式：
        • AnnotatedBeanDefinitionReader：解析显式注册的配置类。
        • ClassPathBeanDefinitionScanner：扫描包路径，过滤出 @Component 类。
    3. 解析 (Parsing)：
      • 将读取到的信息提取为 BeanDefinition 对象。
      • 例如：读取 @Scope("prototype")，设置 bd.setScope("prototype")。
    4. 注册 (Registration)：
      • 调用 DefaultListableBeanFactory.registerBeanDefinition(beanName, bd)。
      • 核心动作：将 beanName -> BeanDefinition 存入一个 ConcurrentHashMap (beanDefinitionMap) 中。

    面试高频点：配置类 (@Configuration) 是什么时候解析的？

    • 初次注册：主配置类（如 AppConfig）在容器启动初期通过 reader 注册。
    • 深度解析：Spring 启动 refresh() -> invokeBeanFactoryPostProcessors 阶段 -> 调用 ConfigurationClassPostProcessor。
    • CCPP 会解析配置类上的 @ComponentScan, @Import, @Bean，生成新的 BeanDefinition 并注册到容器。

    2.5 动态注册 BeanDefinition (实战技巧)

    在开发中间件或复杂框架整合时，我们需要手动向容器注册 Bean。

    场景：MyBatis 的 @MapperScan 原理。

    实现手段：实现 ImportBeanDefinitionRegistrar 接口。

    3. 配置方式与原理

    3.1 基于注解配置

    3.1.1 组件扫描

    • 机制：我们首先需要在一个配置类（XML或Java Config）中启用组件扫描，并指定一个或多个需要扫描的基础包（base package）。

    • 过程：Spring容器在启动时，会像雷达一样扫描这些基础包以及其所有子包下的所有类。如果发现某个类被特定的注解标记，Spring就会将这个类自动注册为一个Bean。

    • 启用扫描：

      • Spring Boot：@SpringBootApplication 默认包含 @ComponentScan（扫描启动类所在包及其子包）。

      • Java Config：@ComponentScan("com.example")

      • XML：<context:component-scan base-package="com.example"/>

    3.1.2 核心注解体系

    • @Component：通用组件注解（父注解）。任何被它标记的类都会被 Spring 扫描并注册为 Bean。
      • Bean ID：默认情况下，被@Component标记的类的Bean ID是其类名的首字母小写形式。
      • 自定义ID：也可以手动指定Bean ID，例如 @Component("coolService")。
    • @Controller：标识表现层组件（Controller）。
    • @Service：标识业务层组件（Service）。
    • @Repository：标识持久层组件（DAO）。
      • 特权：它具有将数据库异常（SQLException）转换为 Spring 统一数据访问异常（DataAccessException）的能力。
    • 最佳实践：虽然 @Component 能替代所有，但为了语义清晰和分层架构，务必使用具体层的注解。

    3.2 基于 Java Config 的配置

    3.2.1 核心注解

    • @Configuration：声明一个类为配置类。Spring 容器会实例化这个类，并解析其中的 @Bean 方法。
      • 核心细节：proxyBeanMethods 属性（Spring 5.2 引入）
        • proxyBeanMethods = true（默认 Full 模式）：Spring 会使用 CGLIB 代理该配置类。在配置类内部调用其他 @Bean 方法时，Spring 会拦截调用，确保返回的是容器中的单例 Bean，而不是重新创建一个新对象。
        • proxyBeanMethods = false（Lite 模式）：不生成代理类。每次调用 @Bean 方法都会执行方法体并创建一个新对象。适用于配置类内部没有 Bean 依赖的情况，能提升启动速度，降低内存消耗。
    • @Bean：标识一个方法，其返回值将被注册为一个 Bean。
      • 常用属性：
        • name / value：指定 Bean 的名称。默认是方法名。可以指定多个名称（别名），如 @Bean({"dataSource", "db1"})。
        • initMethod：指定 Bean 实例化后调用的初始化方法名。
        • destroyMethod：指定 Bean 销毁前调用的方法名（对于实现了 Closeable 或 AutoCloseable 的对象，Spring 默认会推断并调用 close 或 shutdown 方法，除非设置 destroyMethod=""）。
    • @Import：将其他的配置类或普通的组件导入到当前容器中。它是实现模块化配置的核心。
      • 支持导入的三种类型：
        1. 普通类 / @Configuration 类：直接将指定类注册为 Bean。
        2. ImportSelector 接口实现类：动态返回需要导入的类的全限定名数组（Spring Boot 自动装配 @EnableAutoConfiguration 的核心机制）。
        3. ImportBeanDefinitionRegistrar 接口实现类：提供 BeanDefinitionRegistry，允许你手动、动态地向容器中注册复杂的 Bean（如 MyBatis 的 @MapperScan 原理）。

    3.2.2 使用场景

    • 第三方库整合：当引用的类（如 DataSource, RedisTemplate, Retrofit）来自第三方 jar 包，无法修改源码添加 @Component 时，必须使用 Java Config。

    4. Bean 的作用域

    4.1 什么是作用域？

    从 Spring 容器的底层实现来看，Bean Definition（Bean定义） 和 Bean Instance（Bean实例） 是两个完全不同的概念。

    • Bean Definition：是类的“蓝图”或“模具”（存储在 BeanDefinition 对象中）。它描述了怎么创建这个对象（构造函数、属性值等）。
    • Bean Instance：是根据蓝图生产出来的具体的 Java 对象（存储在堆内存中）。

    作用域（Scope）本质上是一个策略接口，它决定了容器如何使用这个“模具”来生产对象：

    1. 控制生产频率：是用同一个模具只造一个对象（Singleton），还是每次都造新对象（Prototype）？
    2. 控制存活时间：这个造出来的对象能活多久？是伴随应用生老病死（Singleton），还是请求结束就销毁（Request）？
    3. 控制可见范围：谁能看到这个对象？是全应用共享，还是只有当前线程可见？

    4.2 配置作用域

    4.2.1 基于注解配置 —— 最常用

    在类上直接使用 @Scope 注解。

    • 常量使用：建议使用 Spring 提供的常量，防止拼写错误。

      • ConfigurableBeanFactory.SCOPE_SINGLETON
      • ConfigurableBeanFactory.SCOPE_PROTOTYPE
      • WebApplicationContext.SCOPE_REQUEST
      • WebApplicationContext.SCOPE_SESSION

    4.2.2 基于 Java Config 配置 (@Bean)

    在配置类中，配合 @Bean 使用。

    4.3 标准作用域

    4.3.1 singleton (单例) - 默认作用域

    • 定义：在整个Spring IoC容器中，一个Bean定义只对应一个实例。

    • 底层实现：

      • 由 DefaultSingletonBeanRegistry 类管理。
      • 核心是一个 ConcurrentHashMap<String, Object> singletonObjects（即所谓的“一级缓存”或“单例池”）。

    • 行为机制：

      • 创建时机：容器启动时，refresh() 方法中的 finishBeanFactoryInitialization() 阶段，会预先实例化所有非懒加载的单例 Bean。
      • 获取逻辑：getBean() -> 检查 singletonObjects Map -> 有则直接返回；无则创建并放入 Map。

    • 线程安全警告：

      • 因为全应用共享同一个实例，Singleton Bean 必须是无状态的（Stateless）。
      • 严禁在单例 Bean 的成员变量中存储特定用户的数据（如 UserContext），否则会导致严重的线程安全问题（数据交叉）。

    • 适用场景：绝大多数无状态的Bean。例如 Service 层对象、Repository 层对象、Controller、配置类等。这些对象不保存与特定请求或用户相关的状态，只提供方法和功能。

    • 代码示例：

      无论你在多少个地方注入MySingletonService，它的构造函数在整个应用生命周期中只会被调用一次。

    4.3.2 prototype (原型/多例)

    • 定义：每次对该Bean进行请求（注入或通过getBean()方法获取）时，都会创建一个全新的实例。

    • 底层实现：

      • Spring 不会缓存 Prototype Bean。
      • 对应源码逻辑：在 doGetBean 方法中，如果是 prototype，直接调用 createBean 创建新对象，不走单例池逻辑。

    • 关键差异：

      • 生命周期截断：Spring 容器只负责 Bean 的 实例化、属性填充、初始化。一旦初始化完成并交给调用者，Spring 彻底放手，不再追踪其生命周期。
      • 意味着：@PreDestroy 或 DisposableBean.destroy() 方法永远不会被调用。开发者必须手动管理资源释放（例如手动关闭该 Bean 持有的数据库连接）。

    • 适用场景：需要保存状态的Bean。例如，一个记录多步骤操作向导的当前状态的对象，或者一个用于构建复杂对象的Builder。每个用户或每个线程都需要一个独立的状态对象，不能共享。

    • 代码示例：

      每次注入UserActionTracker，你都会得到一个全新的、拥有自己actions列表的对象。

    4.3.3 Web应用中的额外作用域

    这些作用域只有在 WebApplicationContext 环境下才有效。

    底层原理：ThreadLocal

    1. request
    • 定义：每个 HTTP 请求创建一个新实例。
    • 底层存储：Bean 被存储在 ServletRequest 的属性（Attribute）中。请求结束，Request 对象销毁，Bean 随之销毁。
    • 场景：API 请求的上下文对象（如记录当前请求的 TraceId）。
    2. session
    • 定义：每个 HTTP Session 创建一个新实例。
    • 底层存储：Bean 被存储在 HttpSession 的属性中。
    • 场景：用户登录信息 (UserSession)、购物车。
    3. application
    • 定义：整个 Web 应用共享一个实例。
    • 底层存储：Bean 被存储在 ServletContext 的属性中。
    • 与 Singleton 的区别：
      • singleton 作用域是针对 Spring 容器（ApplicationContext） 的，一个 Web 应用可以有多个 Spring 容器（父子容器）。
      • application 作用域是针对 ServletContext 的，通常对应 Web 容器中的一个 Web App。
    4. websocket
    • 定义：对于每一个WebSocket会话，创建一个Bean实例。

    4.3.4 作用域依赖问题与代理

    问题场景：

    假设一个 Singleton 的 UserService，它需要注入一个 Session 作用域的 UserPreferences。

    解决方案：Scoped Proxy (作用域代理)

    Spring 提供了一种机制，不直接注入原始的 Bean，而是注入一个智能代理对象。

    • 配置方式：

      或者在 Java Config 中：

      • ScopedProxyMode.TARGET_CLASS：使用 CGLIB 基于类生成代理（推荐）。
      • ScopedProxyMode.INTERFACES：使用 JDK 动态代理（要求 Bean 必须实现接口）。

    • 底层执行流程：

      1. Spring 启动时，为 UserPreferences 生成一个 CGLIB 代理对象。
      2. 将这个代理对象注入给 UserService。
      3. 当 UserService 调用 userPreferences.getData() 时，实际上调用的是代理对象。
      4. 代理对象内部逻辑：
        • 获取当前请求的 Session。
        • 从 Session 中取出属于当前用户的真实 UserPreferences 对象（如果没有就新建）。
        • 调用真实对象的方法。

    三、依赖注入与循环依赖

    1. 注入方式与原理

    1.1 三种注入方式

    1.1.1 构造器注入【官方推荐】

    • 写法：

    • 优点：

      • 强契约：保证 Bean 在实例化后依赖不为空（否则无法实例化），符合“完全初始化”原则。
      • 不可变性：依赖字段可声明为 final，线程安全。
      • 易于测试：在单元测试中可以直接 new UserServiceImpl(mockRepo)，无需启动 Spring 容器。

    • 缺点：

      • 循环依赖无法解决：A 构造器需 B，B 构造器需 A，直接抛出 BeanCurrentlyInCreationException。
      • 参数过多时构造器臃肿。

    1.1.2 Setter 注入【可选】

    • 写法：

    • 优点：

      • 灵活性：依赖是可选的，可以在对象创建后随时修改。
      • 解决循环依赖：这是 Spring 解决循环依赖的基础（通过三级缓存）。

    • 缺点：

      • 依赖可能为 null（非强契约）。
      • 无法注入 final 字段。

    1.1.3 字段注入【不推荐】

    • 写法：

    • 优点：代码极其简洁，开发效率高。

    • 缺点：

      • 隐藏依赖：外界（如测试类）无法通过构造器或 Setter 显式知道该类依赖什么。
      • 测试困难：必须通过反射或启动 Spring 容器才能注入 Mock 对象。
      • 破坏封装：依赖于 DI 框架的魔法（反射），脱离框架无法使用。
      • 不可变性缺失：无法声明 final。

    1.2 自动装配原理

    1.2.1 @Autowired（Spring 原生）

    原理：

    • 由 AutowiredAnnotationBeanPostProcessor 处理。
    • 匹配策略：
      1. 先按类型 (ByType)：查找容器中所有类型为 UserRepository 的 Bean。
        • 若找到 1 个：直接注入。
        • 若没找到：抛出异常（除非 @Autowired(required=false)）。
        • 若找到多个：
          1. 再按名称 (ByName)：尝试匹配字段名（如 userRepository）与 Bean Name。
          2. 配合 @Qualifier：若指定了 @Qualifier("myRepo")，强制按名称匹配。
          3. 配合 @Primary：若某个 Bean 标记了 @Primary，优先注入它。
          4. 都匹配不上：抛出 NoUniqueBeanDefinitionException。

    1.2.2 @Resource（JSR-250 标准）

    • 原理：

      • 由 CommonAnnotationBeanPostProcessor 处理。

      • 匹配策略：

        1. 先按名称 (ByName)：默认取字段名作为 Bean Name 查找。
          • 若找到：直接注入。
          • 若没找到：
            1. 回退按类型 (ByType)：尝试按类型查找（类似于 @Autowired）。
        • 注：如果指定了 name 属性（如 @Resource(name="myRepo")），则只按名称查找，找不到直接报错。

    1.2.3 对比

    特性 @Autowired @Resource
    来源 Spring 框架原生 Java 标准 (JSR-250)
    默认策略 ByType (类型优先) ByName (名称优先)
    指定名称 @Qualifier("name") @Resource(name="name")
    属性支持 required=false 不支持 (必须存在)
    推荐场景 强依赖 Spring 的项目 希望代码解耦、兼容其他容器

    1.3 注入时机

    • 构造器注入：发生在 实例化 阶段。对象创建时就需要依赖。
    • Setter/字段注入：发生在 属性赋值 阶段。
      • 具体是在 AbstractAutowireCapableBeanFactory.populateBean() 方法中。
      • 通过 AutowiredAnnotationBeanPostProcessor 介入，利用反射 (field.set()) 完成注入。

    2. 解决循环依赖

    2.1 什么是循环依赖

    循环依赖是指两个或多个 Bean 互相持有对方的引用，形成闭环。例如：AService 依赖 BService，而 BService 又依赖 AService。

    2.2 Spring 的解决机制：三级缓存

    Spring 只能解决 单例 (Singleton) 作用域下的 Setter/字段注入 产生的循环依赖。对于 构造器注入 或 多例 (Prototype)，Spring 无法解决，会直接抛出 BeanCurrentlyInCreationException。

    核心思想：

    提前暴露半成品。在 Bean 实例化之后、属性赋值之前，将这个“半成品”（早期对象）暴露出来，供其他 Bean 引用。

    三级缓存结构 (DefaultSingletonBeanRegistry)：

    级别 缓存名称 类型 作用
    一级 singletonObjects Map<String, Object> 成品池。存放完全初始化好的 Bean（可以直接使用）。
    二级 earlySingletonObjects Map<String, Object> 半成品/代理池。存放原始对象或提前生成的 AOP 代理对象。
    三级 singletonFactories Map<String, ObjectFactory<?>> 工厂池。存放能生成 Bean 的 lambda 表达式。这是打破循环的关键。

    解决流程 (A -> B -> A)：

    1. A 实例化：createBeanInstance(A)。此时 A 是个半成品。
    2. A 放入三级缓存：addSingletonFactory(A, () -> getEarlyBeanReference(A))。
      • 注意：此时放入的是一个 ObjectFactory，不是对象本身。
    3. A 属性填充：发现依赖 B，调用 getBean(B)。
    4. B 实例化：createBeanInstance(B)。
    5. B 放入三级缓存：同上。
    6. B 属性填充：发现依赖 A，调用 getBean(A)。
    7. B 获取 A：
      • 一级缓存？无。
      • 二级缓存？无。
      • 三级缓存？有！调用 objectFactory.getObject()。
        • 关键动作：如果 A 需要 AOP，这里会提前创建 A 的代理对象；否则返回 A 原对象。
      • 放入二级缓存：将 A 的（代理）对象放入 earlySingletonObjects，移除三级缓存。
      • 返回 A。
    8. B 初始化完成：B 拿到 A 的引用，属性填充完毕，执行初始化，放入一级缓存。
    9. A 获取 B：A 拿到 B 的引用（成品），属性填充完毕，执行初始化，放入一级缓存。
    10. 闭环解开。

    2.3 为什么需要三级缓存？

    这是面试中最刁钻的问题：“二级缓存能不能解决循环依赖？”

    答案：

    如果不考虑 AOP，二级缓存足够了。

    但如果 Bean A 被 AOP 代理了（如加了 @Transactional），必须使用三级缓存。

    深度解析：

    • AOP 的代理时机：正常情况下，代理对象是在 Bean 初始化后（postProcessAfterInitialization）创建的。
    • 冲突点：在循环依赖中，B 需要注入 A。如果注入的是 A 的原始对象，而 A 最后变成了代理对象，那么 B持有的就是错误的引用（原始对象）。
    • 三级缓存的作用：
      • singletonFactories 存的是一个 Lambda (回调)。
      • 当 B 需要 A 时，执行回调。回调内部会判断：A 是否需要代理？
      • 如果要：提前创建代理对象，并放入二级缓存。
      • 如果不要：直接返回原始对象，并放入二级缓存。
    • 结论：三级缓存是为了延迟代理对象的创建，只有在出现循环依赖时，才打破生命周期，提前创建代理对象。

    2.4 无法解决的场景与解决方案

    1. 构造器注入

    • 现象：

      启动直接报错：BeanCurrentlyInCreationException。

    • 原因：构造器执行时对象还没实例化，无法放入三级缓存。

    • 解决方案：

      • 改为 Setter/字段注入（推荐，最简单）。

      • 使用 @Lazy：

        原理：Spring 会注入一个 B 的代理对象（TargetSource），只有当真正调用 B 的方法时，才去容器找真实的 B。

    2. 多例 (Prototype) 作用域
    • 现象：@Scope("prototype") 的 Bean 互相依赖。
    • 原因：Spring 不缓存多例 Bean，无法通过缓存解决。
    • 解决方案：修改业务设计，避免循环依赖，或改为单例。
    3. @Async 的特殊坑
    • 现象：AService 有循环依赖，且使用了 @Async。
    • 原因：@Async 的代理是在 postProcessAfterInitialization 后期生成的，它比标准 AOP 代理更晚。这会导致 B 注入了早期的 AOP 代理（或原始对象），而 A 最后变成了 @Async 代理，导致 Bean 引用不一致 报错 (BeanCurrentlyInCreationException)。
    • 解决方案：
      • 将 @Async 方法移到单独的类中（推荐）。
      • 使用 @Lazy 注入。

    2.5 最佳实践

    1. 避免循环依赖：这是设计异味。通常意味着 A 和 B 职责耦合过重。尝试抽取公共代码到 CService，让 A 和 B 都依赖 C。
    2. 构造器注入优先：虽然它无法解决循环依赖，但这反而能强迫你发现并解决设计问题。
    3. 善用 @Lazy：如果重构成本太大，@Lazy 是解决构造器循环依赖的神器。

    1.2 启动触发

    Bean 的生命周期并不是凭空开始的，它是在 Spring 容器启动的核心方法 refresh() 的第 11 步被触发的。

    • 入口方法：AbstractApplicationContext.refresh()
    • 触发步骤：finishBeanFactoryInitialization(beanFactory)
    • 核心动作：
      1. 遍历所有 BeanDefinition。
      2. 判断：如果是 Singleton 且 非 Lazy。
      3. 调用 getBean(beanName)。
      4. 调用 createBean(beanName, ...) —— 由此正式进入 Bean 的生命周期。

    1.3 阶段详解

    阶段零：元数据加载与合并

    在实例化之前，Spring 必须先”看懂”你要造什么对象。这发生在容器启动早期的 invokeBeanFactoryPostProcessors 阶段。

    1. BeanDefinition 扫描与注册
      • Spring 扫描 @ComponentScan 指定的包路径，或解析 @Configuration 类中的 @Bean 方法。
      • 每个候选组件被解析为一个 GenericBeanDefinition（原始定义），注册到 BeanDefinitionRegistry 中。
    2. BeanDefinition 合并（Merging）
      • 进入创建流程前，Spring 调用 getMergedBeanDefinition()，将原始定义转换为 RootBeanDefinition。
      • 作用：处理父子 BeanDefinition 的继承关系（XML 时代遗留机制），并将 scope、lazy、init-method 等所有配置整合为一份最终定义。后续流程统一操作 RootBeanDefinition。
    3. Class 加载
      • 确保 BeanDefinition 中声明的 beanClassName 已被 ClassLoader 加载为 Class 对象，为后续的构造器推断和反射实例化做准备。

    阶段一：实例化

    对应源码：AbstractAutowireCapableBeanFactory#createBeanInstance()。相当于执行 Java 的 new，但比 new 多了很多扩展点。

    1. 实例化前置回调（短路点）
    • 调用所有 InstantiationAwareBeanPostProcessor#postProcessBeforeInstantiation。
    • 短路机制：如果任一处理器返回了非 null 对象（通常是代理），Spring 将跳过后续的实例化、属性填充、Aware 回调、初始化方法，仅执行 BeanPostProcessor#postProcessAfterInitialization，然后直接将该对象作为最终 Bean。
    • 典型场景：AbstractAutoProxyCreator 在某些特殊条件下会在此处提前返回代理。
    2. 推断构造方法
    • SmartInstantiationAwareBeanPostProcessor#determineCandidateConstructors 介入（核心实现：AutowiredAnnotationBeanPostProcessor）。
    • 选择策略（优先级从高到低）：
      1. 带 @Autowired 标记的构造器（可标注多个 required=false，但 required=true 最多一个）
      2. 唯一的有参构造器（Spring 6.x：单构造器时 @Autowired 可省略）
      3. 无参构造器（兜底）
    3. 反射创建原始对象
    • 通过反射（或 CGLIB / JDK）调用选定的构造器，在堆上创建对象，生成 原始对象包装器 (BeanWrapper)。
    • 此时状态：Bean 仅仅是一个空壳，所有字段为默认值（null / 0 / false）。
    4.【关键】提前暴露 ObjectFactory —— 为循环依赖埋伏笔
    • 触发条件：单例 && allowCircularReferences=true && 当前 Bean 正在创建中（singletonsCurrentlyInCreation 集合中存在）。
    • 动作：Spring 立即将一个 ObjectFactory（Lambda：() -> getEarlyBeanReference(beanName, mbd, bean)）放入 三级缓存（singletonFactories）。
    • 闭环原理：当后续属性填充时发生循环依赖（A → B → A），B 在 getBean(A) 时会依次查询一级 → 二级 → 三级缓存，命中后调用 ObjectFactory#getObject()，触发 SmartInstantiationAwareBeanPostProcessor#getEarlyBeanReference。如果 A 需要 AOP 代理，此时就提前生成代理对象（而非等到阶段三第4步），将结果移入二级缓存，删除三级缓存条目，从而打破循环。

    阶段二：属性填充

    对应源码：AbstractAutowireCapableBeanFactory#populateBean()。

    1. 实例化后置回调（中断点）
    • 调用所有 InstantiationAwareBeanPostProcessor#postProcessAfterInstantiation。
    • 如果任一处理器返回 false，则跳过后续的属性注入，直接进入阶段三。
    • 实际项目中极少使用此中断点。
    2. 属性注入

    • 调用所有 InstantiationAwareBeanPostProcessor#postProcessProperties。

    • 核心处理器：

      处理器 职责
      AutowiredAnnotationBeanPostProcessor 处理 @Autowired、@Value、@Inject
      CommonAnnotationBeanPostProcessor 处理 @Resource

    • 注入方式：通过反射直接写入字段 或调用 Setter 方法。构造器注入在阶段一第3步已经完成。

    注意：属性填充完成后，Bean 的所有依赖已就位。从下一阶段开始，可以安全地使用注入的依赖。

    CommonAnnotationBeanPostProcessor 解析并执行 @PostConstruct 标记的方法

    面试常问：为什么 Aware 分两组？

    第一组是 BeanFactory 级别的基础能力，直接硬编码调用即可；第二组是 ApplicationContext 级别的增强能力，通过 BPP 机制实现，解耦了 BeanFactory 和 ApplicationContext 的职责。

    3. 执行初始化方法（invokeInitMethods()）

    按以下固定顺序执行：

    1. InitializingBean#afterPropertiesSet() —— 检查 Bean 是否实现了该接口
    2. 自定义 init-method —— @Bean(initMethod = "xxx") 或 XML 中指定的方法
    4. 初始化后置处理（BeanPostProcessor#postProcessAfterInitialization）

    再次遍历所有 BPP，逐一调用。这是 AOP 代理生成的标准时机：

    • AbstractAutoProxyCreator（AnnotationAwareAspectJAutoProxyCreator 的父类）在此介入。
    • 如果 Bean 匹配了任何切面（@Transactional、@Async、自定义 @Aspect），则将原始对象包装为 Proxy（JDK 动态代理或 CGLIB），并返回代理对象替换原始对象。
    • 例外：如果该 Bean 在阶段一第4步已经因循环依赖而提前生成了代理（通过 getEarlyBeanReference），此处会检测到并跳过重复代理，直接返回二级缓存中已有的代理对象。

    阶段四：注册完成

    1. 放入一级缓存
    • Bean 创建完毕（可能已是代理对象）。Spring 调用 addSingleton()，将最终对象放入 一级缓存（singletonObjects）。
    • 同时清除该 Bean 在二级缓存（earlySingletonObjects）和三级缓存（singletonFactories）中的条目。
    2. SmartInitializingSingleton#afterSingletonsInstantiated()
    • 触发时机：当所有非懒加载的单例 Bean 全部完成上述流程后，容器在 preInstantiateSingletons() 末尾统一回调。
    • 与 @PostConstruct 的区别：@PostConstruct 在单个 Bean 初始化时执行，此时其他 Bean 可能尚未创建；afterSingletonsInstantiated 保证所有单例都已就绪。
    • 典型场景：EventListenerMethodProcessor 在此处扫描 @EventListener 方法并注册为监听器。。

    阶段五：销毁

    当容器关闭时触发（ctx.close() 或 JVM ShutdownHook）。

    Spring 遍历所有注册了销毁回调的单例 Bean（DisposableBeanAdapter），按以下固定顺序执行：

    1. @PreDestroy —— 由 CommonAnnotationBeanPostProcessor（实现了 DestructionAwareBeanPostProcessor）在销毁前置阶段执行
    2. DisposableBean#destroy() —— 接口回调
    3. 自定义 destroy-method —— @Bean(destroyMethod = "xxx") 指定的方法

    注意：prototype 作用域的 Bean，Spring 在创建后即不再管理，不会执行任何销毁回调。资源释放需调用方自行负责。

    1. AOP 核心

    1.1 定义

    AOP（Aspect-Oriented Programming）是一种编程范式，核心思想是将横切关注点（如日志、事务、权限校验）从业务逻辑中分离出来，避免代码重复侵入。Spring AOP 默认基于动态代理实现。

    1.2 AOP 术语速查

    术语 含义
    Aspect（切面） 横切关注点的模块化（一个类，包含 Advice + Pointcut）
    Join Point（连接点） 程序执行的某个点，Spring AOP 中只支持方法级别
    Pointcut（切入点） 匹配 Join Point 的表达式，决定”在哪切”
    Advice（通知） 切面在特定 Join Point 执行的动作，决定”做什么 + 什么时候做”
    Weaving（织入） 将切面应用到目标对象的过程，Spring AOP 在运行时织入
    Target Object 被代理的原始对象
    AOP Proxy Spring 创建的代理对象（JDK 动态代理 / CGLIB）

    代理方式 核心机制 工作原理 限制
    JDK 动态代理 基于接口 创建代理类实现与目标类相同的接口，通过 InvocationHandler 会先执行切面逻辑，再通过反射调用真实对象 目标类必须实现接口，只能代理接口方法

    3.3 通知顺序

    3.3.1 核心理念

    在Spring AOP中，当多个通知（Advice）应用到同一个连接点（JoinPoint，通常是方法）时，它们的执行顺序遵循一套明确的规则。理解这套规则对于编写健壮、可预测的AOP代码至关重要。

    核心思想是 “洋葱模型” 或 “同心圆模型”。优先级高的切面在最外层，它最先开始执行，也最后结束执行。

    sequenceDiagram
    participant Client as 调用方
    participant Aspect_A as "Aspect A (@Order(1))"
    participant Aspect_B as "Aspect B (@Order(2))"
    participant Target as 目标方法

    Client->>Aspect_A: 调用方法
    activate Aspect_A
    Note right of Aspect_A: 1. Aspect A 前置逻辑执行
    Aspect_A->>Aspect_B: chain.proceed()
    activate Aspect_B
    Note right of Aspect_B: 2. Aspect B 前置逻辑执行
    Aspect_B->>Target: chain.proceed()
    activate Target
    Note over Target: 3. 目标方法核心逻辑执行
    Target-->>Aspect_B: 方法返回
    deactivate Target
    Note right of Aspect_B: 4. Aspect B 后置逻辑执行
    Aspect_B-->>Aspect_A: 返回
    deactivate Aspect_B
    Note right of Aspect_A: 5. Aspect A 后置逻辑执行
    Aspect_A-->>Client: 最终结果返回
    deactivate Aspect_A

    3.3.2 具体情况

    1. 单一切面内的执行顺序

    当只有一个切面类，并且它的多个不同类型的通知都匹配了同一个方法时，执行顺序是固定的。

    目标方法正常执行（无异常）

    执行顺序如下：

    1. @Around 环绕通知（proceed() 方法之前的部分）
    2. @Before 前置通知
    3. 目标方法执行
    4. @Around 环绕通知（proceed() 方法之后的部分）
    5. @AfterReturning 返回通知
    6. @After 后置通知

    目标方法抛出异常

    执行顺序如下：

    1. @Around 环绕通知（proceed() 方法之前的部分）
    2. @Before 前置通知
    3. 目标方法执行，并抛出异常
    4. @AfterThrowing 异常通知
    5. @After 后置通知
    6. @Around 环绕通知会捕获到异常（如果在try-catch块中），然后决定是处理还是继续向外抛出。
    2. 多个不同切面间的执行顺序

    这是最常见的场景。当多个切面类中的通知都匹配了同一个方法时，切面类的顺序决定了通知的执行顺序。

    规则：使用 @Order(数字) 或实现 Ordered 接口

    1. @Order(数字) 注解：加在切面类上。
    2. Ordered 接口：让切面类实现 org.springframework.core.Ordered 接口，并重写 getOrder() 方法。
      • 数字越小，优先级越高。@Order(1) 的优先级高于 @Order(10)。
      • 如果没有指定顺序，则执行顺序是 未定义的（Undefined）。绝对不要依赖默认的类名字母顺序，因为它不可靠。
    3. 同一切面内多个同类型通知的执行顺序

    当同一个切面类中，有多个同类型的通知（例如，两个@Before）都匹配了同一个方法。

    规则：未定义（Undefined）

    Spring AOP规范没有定义在同一个切面内同类型通知的执行顺序。你不能依赖它们在代码文件中的书写顺序。

    最佳实践

    如果多个逻辑之间确实存在顺序依赖，请遵循以下两种方式之一：

    1. 合并为一个通知方法：将所有逻辑按顺序写在同一个@Before方法体内，这是最简单的方式。
    2. 拆分到不同的切面类中：将每个逻辑块放入一个独立的切面类中，然后使用@Order来精确控制这些切面类的执行顺序。这种方式更符合“单一职责原则”，代码结构更清晰。

    4. Pointcut表达式

    4.1 概述

    • 本质：AOP的查询语言。
    • 作用：告诉 Spring AOP 具体的连接点在哪里（筛选出需要被增强的方法）。
    • 组成：由一个或多个**切点指示符（Pointcut Designators, PCD）**组成，可以通过逻辑运算符 && , || , ! 进行组合。

    4.2 @Pointcut 注解详解

    如果说切点表达式是复杂的SQL查询语句，那么 @Pointcut 注解就是给这条SQL语句起了一个变量名。

    4.2.1 基本语法

    它由两部分组成：

    1. 表达式：定义在 @Pointcut("...") 中，即具体的筛选规则。
    2. 签名：一个返回值为void且方法体为空的普通Java方法。方法名就是这个切点的ID。

    4.2.2 作用

    1. 复用性 (DRY原则)：
      如果不使用 @Pointcut，你可能需要在五个不同的通知（Advice）里复制粘贴同一个长长的 execution(...) 字符串。一旦规则变化，你需要修改五处。使用 @Pointcut，只需修改一处。
    2. 可读性：
      businessLayer() 显然比 execution(* com.example.service..*.*(..)) 更能让人一眼看懂代码的意图。

    4.2.3 可见性原则

    切点方法的访问修饰符决定了它能被谁引用：

    • private：仅在当前切面类中可见。
    • public：在任何地方都可见。
      • 引用方式：其他类引用时需要使用全限定名，如 com.example.aspects.SystemArchitecture.businessLayer()。

    4.2 execution

    这是功能最强大、使用频率最高（90%）的指示符，用于精确匹配方法签名。

    4.2.1 语法解构

    • [] 表示该部分是可选的。

    4.2.2 核心通配符

    • * (星号)：匹配任意字符（一个元素）。
      • 用于返回类型：匹配任意返回值。
      • 用于包/类/方法名：匹配名称的全部或部分。
    • .. (双点)：匹配多个元素。
      • 用于包名：表示当前包及其所有子包。
      • 用于参数：表示0个或多个任意类型的参数。

    4.2.3 语法分解与示例

    • 修饰符 (可选): 如public, protected, private。通常省略，表示匹配所有修饰符。
      • execution(public * *(..)) - 匹配所有public方法。
    • 返回类型 (必选):
      • * - 匹配任意返回类型。
      • void - 匹配返回类型为void的方法。
      • java.lang.String - 匹配返回类型为String的方法。
    • 包名.类名.方法名 (包名和类名可选):
      • 完整匹配: com.example.service.UserService.createUser
      • 类内所有方法: com.example.service.UserService.*
      • 包内所有类的所有方法: com.example.service.*.*
      • 包及其子包下所有类的所有方法: com.example.service..*.* ( .. 在这里至关重要！)
      • 方法名通配符: * (所有方法), get* (所有以get开头的方法), *ById (所有以ById结尾的方法)。
    • 参数类型 (必选):
      • () - 匹配无参方法。
      • (..) - 匹配任意数量、任意类型的参数 (最常用)。
      • (*) - 匹配只有一个参数的方法，类型不限。
      • (String) - 匹配只有一个String类型参数的方法。
      • (String, ..) - 匹配第一个参数是String，后面有任意数量、任意类型参数的方法。
    • 异常 (可选): throws子句，匹配抛出特定异常的方法。很少使用。

    4.2.4 常用写法速查

    匹配范围 表达式写法 解释
    指定方法 execution(* com.demo.Svc.add(..)) 任意返回值，Svc类下的add方法，参数不限
    包下所有类 execution(* com.demo.svc.*.*(..)) svc包（不含子包）下所有类的所有方法
    包及子包 execution(* com.demo.svc..*.*(..)) svc包及其所有子包下所有类的所有方法
    特定前缀 execution(* *..get*(..)) 任意包下，所有以get开头的方法
    特定参数 execution(* *..find(String, ..)) 第一个参数必须是String，后续参数不限

    4.3 within

    用于匹配指定类型（类/接口）或包内的所有方法。

    它的粒度比 execution 粗，不关心方法的具体名称或参数，只关心它属于哪个类。

    • 语法：
      • within(com.example.service.UserServiceImpl)：仅匹配该特定类。
      • within(com.example.service.*)：匹配包下所有类（不含子包）。
      • within(com.example.service..*)：匹配包及其子包下所有类。

    4.4 @annotation

    这是现代Spring开发中最优雅的方式。它不匹配类名，而是匹配方法上的注解。

    场景：我不管你在哪个包，只要你的方法头上顶着 @Loggable 注解，我就拦截你。

    4.5 @within

    • @annotation：匹配方法级别的注解（Target是方法）。
    • @within：匹配类级别的注解（Target是类）。

    场景：只要一个类被标记了 @Secured，那么这个类里面的所有方法都会被拦截。

    4.6 bean

    Spring 特有的指示符，直接根据 Bean ID (名称) 进行筛选。

    • bean("userService")：匹配 ID 为 userService 的 Bean。
    • bean("*Service")：匹配所有 ID 以 Service 结尾的 Bean。

    5. 连接点

    当我们定义一个通知（Advice）时，可以声明一个org.aspectj.lang.JoinPoint类型的参数，Spring容器会自动将与该切点匹配的连接点信息注入进来。通过这个对象，我们可以在运行时获取到大量关于方法执行的上下文信息。

    5.1 JoinPoint

    5.1.1 获取方法签名信息 getSignature()

    这是最核心和最常用的方法，它返回一个Signature对象，包含了被拦截方法的详细信息。

    Signature对象本身有很多有用的方法：

    • String getName(): 获取被拦截的方法名。

    • Class getDeclaringType(): 获取方法所属的类。

    • String getDeclaringTypeName(): 获取方法所属类的全限定名。

    • int getModifiers(): 获取方法的修饰符，返回的是一个代表修饰符的整数（例如 public, static）。可以使用 java.lang.reflect.Modifier.toString() 来转换。

    • String toShortString(): 返回一个简短的方法签名描述。

    • String toLongString(): 返回一个详细的方法签名描述。

    5.1.2 获取方法参数 getArgs()

    获取传递给被拦截方法的参数数组。

    5.1.3 获取目标和代理对象

    • Object getTarget(): 获取被代理的 目标对象（原始的Bean对象）。

    • Object getThis(): 获取 代理对象 本身。通常是CGLIB或JDK动态代理生成的对象。

      注意：getTarget() 和 getThis() 返回的对象通常是不同的。getTarget() 是原始的业务逻辑类，而 getThis() 是Spring为了实现AOP而创建的代理类实例。

    5.1.4 其他静态信息

    • StaticPart getStaticPart(): 提供对连接点的静态部分的访问，功能与 getSignature() 类似，但提供了更通用的Aspect静态信息。
    • String getKind(): 获取连接点的类型。在Spring AOP中，这个值通常是 method-execution。

    5.2 ProceedingJoinPoint (仅适用于@Around环绕通知)

    ProceedingJoinPoint 继承自 JoinPoint，因此它拥有上述所有方法。除此之外，它增加了一个至关重要的核心方法，用于控制目标方法的执行。

    5.2.1 Object proceed() 和 Object proceed(Object[] args)

    这是ProceedingJoinPoint的专属方法，也是@Around通知如此强大的原因。

    • Object proceed(): 执行目标方法。调用此方法会触发目标方法的执行。其返回值就是目标方法的返回值。
    • Object proceed(Object[] args): 执行目标方法，并使用新的参数数组替换原有的参数。这允许你在执行目标方法前修改传入的参数。

    核心作用:

    1. 执行控制: 你可以决定是否调用proceed()方法。如果不调用，目标方法将永远不会被执行。
    2. 参数修改: 在调用proceed(newArgs)之前，可以修改参数。
    3. 返回值修改: proceed()方法调用结束后，可以获取到原始返回值，并对其进行修改或替换，然后返回一个新的值。
    4. 异常处理: 可以使用 try-catch 块来捕获目标方法抛出的异常，并进行处理或包装后重新抛出。
