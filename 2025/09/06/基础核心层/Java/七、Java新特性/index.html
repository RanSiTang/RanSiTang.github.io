七、Java新特性

1. Stream流

1.1 概述

Stream 是 Java 8 引入的接口（java.util.stream.Stream<T>），用于描述数据集合的序列化处理管道。

它不是数据结构，而是对数据源（如 List、Set、数组）的视图，支持函数式编程风格。

核心特性：

  • 惰性求值：中间操作不会立即执行，只有遇到终端操作时才触发计算。
  • 不可变：Stream 操作不修改原数据源。
  • 一次性使用：Stream 消费后即关闭，不能重用（会抛 IllegalStateException）。

优势：

  • 代码简洁：链式调用取代传统 for 循环。
  • 并行友好：易于并行处理，提高性能。
  • 函数式：支持 Lambda 表达式和方法引用。

1.2 Stream vs 传统集合操作

方面 传统方式 (for 循环) Stream API
可读性 较低，代码冗长 高，链式表达意图清晰
并行 需要手动线程管理 一键 parallel()
惰性 无 支持短路优化
适用场景 简单循环 复杂过滤/转换/聚合

1.3 Stream的生命周期与核心概念

  1. 数据源 (Source): 创建一个 Stream。
  2. 中间操作 (Intermediate Operations): 对 Stream 进行转换和处理。可以有零个或多个中间操作。这些操作是惰性求值 (Lazy Evaluation) 的，它们不会立即执行，而是返回一个新的 Stream。
  3. 终端操作 (Terminal Operation): 触发整个 Stream pipeline 的计算，并产生一个最终结果或副作用。一个 Stream 只能有一个终端操作，执行后该 Stream 就被消费掉了，不能再使用。

由静态方法创建 Stream.of(T... values) Stream<String> stream = Stream.of("a", "b", "c");

方法签名 描述

方法签名 描述

1.6 常用终端操作

这些操作会触发计算并关闭 Stream。

1.6.1 匹配与查找

这些是短路操作 (Short-circuiting)，一旦找到满足条件的结果，就会立即停止计算。

方法签名 描述 返回值
anyMatch(Predicate<? super T> predicate) 是否有至少一个元素匹配。 boolean
allMatch(Predicate<? super T> predicate) 是否所有元素都匹配。 boolean
noneMatch(Predicate<? super T> predicate) 是否没有元素匹配。 boolean
findFirst() 返回第一个元素。 Optional<T>

1.6.2 归约 (Reduction)

方法签名 描述
count() 返回流中元素的总数。
max(Comparator<? super T> comparator) 返回最大元素。
min(Comparator<? super T> comparator) 返回最小元素。
reduce(T identity, BinaryOperator<T> accumulator) 将流中的元素反复结合起来，得到一个单一的值。

示例：计算数字列表的和。

1.6.3 收集 (Collect)

collect 是最强大、最灵活的终端操作，它能将 Stream 中的元素转换成各种形式的结果，如 List, Set, Map 等。

它接收一个 Collector 作为参数，而 java.util.stream.Collectors 工具类提供了大量静态工厂方法来创建常用的 Collector。

Collector 描述 示例
toList(), toSet() 收集到 List 或 Set。 ...collect(Collectors.toList());
toMap(keyMapper, valueMapper) 收集到 Map。注意：key 不能重复，否则会抛异常。 ...collect(Collectors.toMap(Person::getId, person -> person));
toMap(k, v, mergeFunction) toMap 的重载版本，用于处理 key 冲突。 ...toMap(Person::getCity, Person::getName, (name1, name2) -> name1 + ", " + name2));
joining(delimiter) 将流中的 String 元素用分隔符连接起来。 ...collect(Collectors.joining(", "));
groupingBy(classifier) 分组。根据一个分类函数对元素进行分组，返回一个 Map。 Map<City, List<Person>> peopleByCity = ...collect(Collectors.groupingBy(Person::getCity));
partitioningBy(predicate) 分区。根据一个断言将元素分为 true 和 false 两组。 Map<Boolean, List<Person>> passingFailing = ...collect(Collectors.partitioningBy(s -> s.getScore() > 60));

1.8 并行流

  • 创建: collection.parallelStream() 或 stream.parallel()。
  • 原理: 底层使用 Fork/Join 框架，将大数据块分解（fork）成小块，交给不同的线程处理，最后将结果合并（join）。
  • 适用场景:
    • 数据量巨大（成千上万个元素以上）。
    • 每个元素的操作相对耗时（CPU密集型）。
    • 操作之间是独立的，没有顺序要求。
  • 警告与陷阱:
    • 线程安全: 传递给并行流的 Lambda 表达式必须是无状态的，且不能修改共享变量，否则会产生线程安全问题。
    • 装箱/拆箱开销: 对基本数据类型流（IntStream, LongStream, DoubleStream）进行操作可以避免昂贵的自动装箱/拆箱开oversized，性能更好。
    • 并非总是更快: 对于小数据量或简单的操作，线程切换和管理的开销可能超过并行带来的好处。使用前请务必进行性能测试。

1.9 最佳实践

  1. 无状态原则: 传递给中间操作的 Lambda 表达式应该是无状态的（Stateless），即其结果不应依赖于任何在 Stream pipeline 执行期间可能改变的状态。
  2. 副作用最小化: 尽量避免在 forEach 之外的 Lambda 中产生副作用（如修改外部变量）。让 Stream 操作专注于数据转换。
  3. 拥抱 Optional: findFirst(), max(), min() 等返回 Optional 的方法，强制你处理可能不存在的情况，能有效避免 NullPointerException。
  4. 选择正确的 Collector: groupingBy 是处理复杂分组聚合的利器，要熟练掌握。
  5. 注意可读性: 虽然可以写出很长的链式调用，但如果逻辑过于复杂，应考虑将其拆分成多个步骤或封装成私有方法，以保持代码的可读性。

2. Optional

2.1 概述

2.1.1 作用

在 Java 8 之前，null 引用是导致 NullPointerException (NPE) 的主要元凶。我们通常需要编写大量的防御性代码来避免它：

这种 if (obj != null) 的检查随处可见，不仅繁琐，而且容易遗忘，导致潜在的 NPE 风险。更糟糕的是，一个方法的返回类型是 String，但它有时可能会返回 null，这在方法的签名中完全看不出来，调用者必须去阅读文档或源码才能知道需要进行 null 检查。

Optional 的出现就是为了解决这个问题。它是一个容器类，可以代表一个值的存在（present）或不存在（absent）。通过将可能为 null 的对象包装在 Optional 中，方法的返回类型 Optional<T> 就能清晰地告诉调用者：“这个结果可能不存在，你必须显式地处理这种情况。”

2.1.2 核心设计哲学

  • 它不是 null 的替代品：Optional 的目的不是要消灭代码中所有的 null。它的主要应用场景是作为方法的返回值，明确表示一个值可能缺失。
  • 它是一种提醒和契约：使用 Optional 作为返回类型，就像是与调用者签订了一个契约，强制他们思考“如果值不存在该怎么办”。
  • 鼓励函数式编程：Optional 提供了丰富的函数式API（如 map, flatMap, filter），使得处理缺失值的逻辑可以优雅地链式调用，而不是嵌套在 if-else 块中。

2.2 创建Oprional对象

Optional 的构造函数是私有的，我们必须使用其提供的三个静态工厂方法来创建实例。

静态方法 描述 注意事项
Optional.of(T value) 创建一个包含给定值的 Optional。 参数 value 绝不能为 null，否则会立即抛出 NullPointerException。
Optional.ofNullable(T value) 创建一个 Optional，如果 value 不为 null，则行为与 of() 相同；如果 value 为 null，则创建一个空的 Optional。 最常用、最安全的创建方式。
Optional.empty() 创建一个空的 Optional 实例。

2.3 核心方法

一旦我们有了一个 Optional 对象，我们就可以使用它的实例方法来安全地处理可能存在或缺失的值。

2.3.1 判断与消费 (Checking and Consuming)

方法签名 描述
boolean isPresent() 如果值存在，返回 true，否则返回 false。（Java 11 后推荐使用 isEmpty()）
boolean isEmpty() （自 Java 11 起）如果值不存在，返回 true，否则返回 false。语义上更清晰。
void ifPresent(Consumer<? super T> consumer) 如果值存在，就对该值执行给定的操作（Consumer），否则什么也不做。

2.3.2 获取值 (Unwrapping the Value) - 谨慎使用 get()

方法签名 描述
T get() 如果值存在，返回值；否则抛出 NoSuchElementException。通常不推荐直接使用。
T orElse(T other) 如果值存在，返回值；否则返回指定的默认值 other。非常常用。
T orElseGet(Supplier<? extends T> other) 如果值存在，返回值；否则执行 Supplier 函数并返回其结果。
T orElseThrow() (自 Java 10 起) 如果值存在，返回值；否则抛出 NoSuchElementException。
<X extends Throwable> T orElseThrow(Supplier<? extends X> exceptionSupplier) 如果值存在，返回值；否则抛出由 Supplier 创建的异常。自定义异常的最佳方式。

orElse() vs orElseGet() 的区别:

  • orElse(defaultValue): 无论 Optional 中是否有值，defaultValue 这个参数总会被求值。
  • orElseGet(supplier): 只有当 Optional 为空时，supplier 这个 Lambda 表达式才会被执行。

如果默认值的创建开销很大（例如，需要调用一个方法或 new 一个新对象），orElseGet() 的懒加载特性可以带来性能优势。

2.3.3 函数式转换 (Functional Transformations)

这是 Optional 最强大的部分，允许我们以声明式的方式进行链式操作。

方法签名 描述
Optional<U> map(Function<? super T, ? extends U> mapper) 如果值存在，则对其应用 mapper 函数，并将结果包装在新的 Optional 中返回；如果为空，则返回一个空的 Optional。
Optional<U> flatMap(Function<? super T, ? extends Optional<? extends U>> mapper) 与 map 类似，但要求 mapper 函数的返回值本身就是一个 Optional。flatMap 会将这个结果“压平”，避免出现 Optional<Optional<T>> 这样的嵌套结构。
Optional<T> filter(Predicate<? super T> predicate) 如果值存在且满足 predicate 条件，则返回当前 Optional；否则返回一个空的 Optional。

2.5 常见误区与注意事项

  1. 不要用 Optional 作为类的字段：Optional 没有实现 Serializable 接口，这会导致序列化问题。字段为空可以直接用 null 表示。
  2. 不要用 Optional 作为方法的参数：这会使方法调用变得笨拙。如果一个参数是可选的，应该使用方法重载。
  3. 不要用 Optional 包装集合：一个返回集合的方法，如果没有结果，应该返回一个空的集合（如 Collections.emptyList()），而不是 Optional<List<T>>。这更符合直觉且调用方无需额外处理 Optional。

3. Lambda表达式

3.1 概述

Lambda 表达式是 Java 8 引入的一个核心特性，它允许我们将函数作为方法的参数传递，或者把代码看作数据来处理。本质上，Lambda 表达式是一个匿名函数，它没有名称，但有参数列表、函数主体、返回类型，并且可能抛出异常。

在 Java 8 之前，如果你想传递一段代码逻辑（比如一个比较器或一个任务），你必须使用匿名内部类。

示例：使用匿名内部类排序

这种写法的缺点显而易见：

  • 语法冗余（Boilerplate Code）: 为了实现一个简单的方法，需要写很多模板化的代码（new Comparator<String>(), @Override, 方法签名等）。
  • 可读性差: 核心逻辑 a.compareTo(b) 被包裹在大量的语法噪音中。

Lambda 表达式正是为了解决这些问题而生的。

3.2 Lambda 表达式的核心语法

Lambda 表达式的基本语法结构是：

(parameters) -> expression 或 (parameters) -> { statements; }

它由三部分组成：

  1. 参数列表 (Parameter list): (String a, String b)
  2. 箭头符号 (Arrow token): ->
  3. 主体 (Body): 可以是一个表达式，也可以是一个代码块。

语法简化规则

Java 编译器可以根据上下文推断出很多信息，因此 Lambda 表达式可以写得非常简洁。

  1. 类型推断: 编译器可以推断参数类型，所以可以省略。

  2. 单参数时省略括号: 如果参数列表只有一个参数，可以省略括号。

  3. 单行表达式时省略大括号和 return: 如果主体只有一行表达式，可以省略大括号 {} 和 return 关键字。该表达式的结果就是返回值。

  4. 无参数时使用空括号: 如果没有参数，需要使用一对空括号 ()。

3.3.2 function

Java 8 在 java.util.function 包中预定义了大量常用的函数式接口，以满足各种场景的需求。熟悉它们至关重要。

接口名称 抽象方法 描述 示例
Predicate<T> boolean test(T t) 断言/判断。接收一个参数，返回一个布尔值。 s -> s.length() > 5
Consumer<T> void accept(T t) 消费。接收一个参数，没有返回值。 s -> System.out.println(s)
Function<T, R> R apply(T t) 转换/映射。接收一个参数，返回一个结果。 s -> s.toUpperCase()
Supplier<T> T get() 供给。不接收参数，返回一个结果。 () -> "hello"
UnaryOperator<T> T apply(T t) 一元操作。Function的特例，输入输出类型相同。 n -> n * 2
BinaryOperator<T> T apply(T t1, T t2) 二元操作。接收两个同类型参数，返回同类型结果。 (a, b) -> a + b

此外，还有针对基本数据类型（int, long, double）的特化版本，如 IntPredicate, LongConsumer, DoubleFunction 等，以避免自动装箱/拆箱带来的性能开销。

3.4 方法引用

方法引用是 Lambda 表达式的一种特殊、更简洁的写法。当你需要实现的 Lambda 表达式的主体恰好是调用一个已存在的方法时，就可以使用方法引用。

四种类型的方法引用

  1. 静态方法引用 (Static Method Reference)
    • 语法: ClassName::staticMethodName
    • 示例: s -> Integer.parseInt(s) 可以写成 Integer::parseInt
  2. 实例方法引用 (Instance Method Reference on a Specific Object)
    • 语法: instance::instanceMethodName
    • 示例: () -> System.out.println("hello")。如果 out 是 System 的一个实例，可以写成 System.out::println。
  3. 特定类型的任意对象的实例方法引用
    • 语法: ClassName::instanceMethodName
    • 描述: Lambda 的第一个参数会成为调用该实例方法的对象。
    • 示例: (s1, s2) -> s1.compareToIgnoreCase(s2) 可以写成 String::compareToIgnoreCase。
  4. 构造方法引用 (Constructor Reference)
    • 语法: ClassName::new
    • 示例: () -> new ArrayList<>() 可以写成 ArrayList::new。

使用方法引用的好处：代码更短，意图更清晰，可读性更强。

3.5 Lambda表达式的变量捕获

Lambda 表达式可以访问其外部作用域的变量，这个过程称为“捕获”。

  • 可以捕获什么: 局部变量、实例变量、静态变量。
  • 捕获规则: Lambda 表达式只能捕获事实上的最终变量 (effectively final)。
    • final 变量: 明确声明为 final 的变量。
    • 事实上的 final 变量: 初始化后，其值再也没有被改变过的局部变量。

为什么有这个限制？

这是为了避免在多线程环境下（Lambda 表达式常用于并发）出现线程安全问题。如果 Lambda 可以修改外部变量，那么当 Lambda 在另一个线程执行时，就可能出现数据不一致的问题。Java 设计者通过强制变量为事实 final 来简化并发模型，避免了这类错误。

3.8 最佳实践

  1. 优先使用 Lambda: 在任何需要函数式接口的地方，优先使用 Lambda 表达式或方法引用，而不是匿名内部类。
  2. 保持简洁: Lambda 的威力在于其简洁性。如果一个 Lambda 的主体变得很长（超过3行），考虑将其重构为一个独立的私有方法，然后使用方法引用。
  3. 明确类型 vs 类型推断: 在大多数情况下，让编译器进行类型推断。但如果 Lambda 逻辑复杂，显式声明参数类型可以提高可读性。
  4. 熟悉 java.util.function: 这是使用 Lambda 和 Stream API 的基础，必须熟练掌握。
  5. 理解闭包和变量捕获: 清楚 Lambda 能够访问哪些变量以及相关的限制，是编写正确代码的关键。
