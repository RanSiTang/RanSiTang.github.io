一、Spring Boot 核心理念与入门

1. Spring 的发展痛点：从 XML 地狱到繁琐配置

  • SSM 时代的噩梦：
    • 配置繁琐：传统的 Spring 项目需要大量的 XML 配置（applicationContext.xml, dispatcher-servlet.xml, web.xml），或者复杂的 JavaConfig 类。
    • 依赖冲突（Dependency Hell）：手动管理依赖版本，经常出现 ClassNotFoundException 或 NoSuchMethodError，比如 Spring 4.x 需要配合 Jackson 哪个版本？Hibernate 哪个版本兼容？
    • 部署麻烦：必须依赖外部的 Servlet 容器（Tomcat/Jetty），打成 war 包部署，调试和运维流程割裂。
  • Boot 的定位：
    • Spring Boot 不是对 Spring 功能的增强，而是提供了一种快速使用 Spring 的方式。它是 Spring 生态的整合者和“脚手架”。

2. 核心理念：约定优于配置

  • 含义：
    • 框架为环境配置提供默认值（约定）。
    • 如果开发者遵循这些约定，就不需要编写任何配置。
    • 只有当需要偏离默认值时，才需要显式配置。
  • 体现：
    • Maven 目录结构约定：src/main/java，src/main/resources。
    • 默认配置约定：引入 spring-boot-starter-web，Boot 就“约定”你要开发 Web 应用，自动配置 Tomcat 端口为 8080，自动配置 Spring MVC 的 DispatcherServlet。
  • 优势：极大降低了项目的启动门槛和初期配置成本。

3. 核心功能

3.1 起步依赖（Starters）—— 解决依赖管理

  • 本质：
    • 一组依赖关系的集合（Maven BOM + 依赖聚合）。
    • 它告诉 Maven/Gradle 需要什么，而不需要知道具体的版本号。
  • 作用：
    • 开箱即用：想用 Redis？引入 spring-boot-starter-data-redis 即可，不需要单独引入 jedis, commons-pool, spring-data 等。
    • 版本仲裁：通过 spring-boot-dependencies (父工程) 统一管理版本，彻底解决 JAR 包冲突问题。

3.2 自动配置（Auto-Configuration）—— 解决配置繁琐

  • 本质：
    • 基于 Classpath（类路径）下的 jar 包依赖，动态注册 Bean。
  • 逻辑举例：
    • Boot 检测到类路径下有 H2 的驱动包 → 自动配置内存数据库连接。
    • Boot 检测到类路径下有 Spring MVC → 自动配置 ViewResolver、MessageConverter。
  • 核心差异：
    • 传统 Spring：显式配置（我知道我要用什么，所以我配什么）。
    • Spring Boot：推断配置（我猜你可能要用什么，所以我先帮你配好，不喜欢你再改）。

3.3 内嵌容器（Embedded Containers）—— 解决部署麻烦

  • 改变：
    • Old：服务器（Tomcat） → 部署 → 应用（War）。
    • New：应用（Jar） → 包含 → 服务器（Tomcat）。
  • 实现：
    • 直接将 Tomcat/Jetty/Undertow 作为库依赖引入应用。
    • 通过 public static void main 入口启动 Web 服务，像运行普通 Java 程序一样运行 Web 应用。

4. 项目结构最佳实践

  • 主启动类位置：

    • 原则：推荐将主启动类（Main Class）放在**根包（Root Package）**下，即所有其他类的顶层父包中。
    • 原因：@SpringBootApplication 注解包含了 @ComponentScan，默认只扫描当前包及其子包。放在根包可以免去手动配置 @ComponentScan(basePackages=...) 的麻烦。

  • 错误的结构：

二、配置文件与外部化配置

1. 配置文件格式与优先级

1.1 Properties vs YAML

  • Properties (.properties)：
    • Spring 的传统格式，扁平结构。
    • 缺点：冗余度高，层级深时可读性差。
  • YAML (.yml / .yaml)：
    • 推荐使用。树状结构，层次分明，简洁。
    • 【重难点】YAML 语法陷阱：
      1. 缩进：严禁使用 Tab 键，必须使用空格（通常 2 个或 4 个）。
      2. 冒号后空格：键值对的冒号后必须有一个空格（key: value），否则无法解析。
      3. 大小写敏感：属性名区分大小写。
      4. 类型推断：
        • true/false 会被识别为布尔值。如果需要存字符串 “true”，必须加引号。
        • 数字会被识别为 Int/Double。版本号如 1.10 如果不加引号，可能被解析器截断或误解。

1.2 配置文件加载优先级

Spring Boot 启动时会扫描以下位置的 application.properties/yml，优先级从高到低（高优先级的配置会覆盖低优先级的，互补配置）：

  1. 命令行参数 (最高优先级)：
    • java -jar app.jar --server.port=9090
    • 场景：运维在启动脚本中动态指定端口或环境。
  2. JAR 包外部的 config/ 子目录。
  3. JAR 包外部的根目录。
  4. JAR 包内部的 config/ 包。
  5. JAR 包内部的根目录 (classpath root)。

最佳实践：开发时配置写在项目内部（4/5）；生产部署时，如果有敏感配置（密码）或差异配置，在 Jar 包同级目录下创建 config/application.yml 进行覆盖，无需重新打包。

三、自动配置原理与核心注解

1. @SpringBootApplication 组合注解剖析

这是一个典型的 Spring 复合注解。它由其他三个核心注解组合而成。目的是为了简化配置，让开发者只需一个注解就能开启 Spring Boot 的所有核心功能。

1.1 @SpringBootConfiguration

  • 作用：声明当前类是一个 Spring Boot 的配置类。

  • 深层剖析：我们再次点进 @SpringBootConfiguration 的源码会发现：

    它本质上就是 Spring 框架的 @Configuration 注解。

  • 意义：

    1. 语义化：它在语义上表明这是一个Spring Boot应用的主配置类，而不仅仅是一个普通的Spring配置类。
    2. Bean定义：意味着我们可以在这个类中使用 @Bean 注解来手动定义和注册Bean到IoC容器中。
    3. 单例保证：Spring 会为 @Configuration 类创建一个CGLIB代理，以确保通过方法调用获取Bean时能遵循其作用域（如单例）。

1.2 @ComponentScan

  • 作用：开启 组件扫描，自动发现并注册被特定注解（如@Component, @Service, @Repository, @Controller）标记的类为Bean。
  • 深层剖析：
    • 默认扫描范围：@SpringBootApplication 中的 @ComponentScan 没有指定 basePackages 属性。在这种情况下，它的默认扫描范围是当前启动类所在的包及其所有子包。
    • 这就是为什么第一章强调主启动类要放在根包下的原因。

1.3 @EnableAutoConfiguration

  • 作用：这是 开启自动配置功能的主开关。
  • 深层剖析：它通过 @Import(AutoConfigurationImportSelector.class) 来实现其核心功能。
    • AutoConfigurationImportSelector 这个类会去扫描所有引入的JAR包中 META-INF/spring.factories 文件。
    • 它会找到 org.springframework.boot.autoconfigure.EnableAutoConfiguration 这个键所对应的所有自动配置类（如 DataSourceAutoConfiguration, WebMvcAutoConfiguration 等）。
    • 然后，它会将这个庞大的配置类列表加载到Spring IoC容器中。
    • 最后，每个自动配置类会根据自身的 @ConditionalOn... 条件来判断自己是否应该生效。

2. 自动配置运作机制

这是 Spring Boot 启动时的幕后工作流程，分为 加载 和 过滤 两个阶段。

阶段一：加载候选配置类

Spring Boot 需要知道有哪些自动配置类（比如 RedisAutoConfiguration, WebMvcAutoConfiguration）是可用的。这些类不是通过包扫描出来的，而是通过读取特定文件加载的。

  • 版本差异 (面试高频考点)：
    • Spring Boot 2.7 之前：
      • 使用 SpringFactoriesLoader 加载器。
      • 读取路径：META-INF/spring.factories。
      • 查找 Key：org.springframework.boot.autoconfigure.EnableAutoConfiguration 下的所有类。
    • Spring Boot 2.7 及 3.x 之后：
      • 引入了新的加载机制 ImportCandidates。
      • 读取路径：META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports。
      • 格式：纯文本文件，每行一个全限定类名，不再是 Key-Value 结构。
      • 优势：格式更简洁，解析效率更高。

阶段二：筛选与过滤

加载了上百个自动配置类，并不是全都会生效，需要经过两层筛选：

  1. 排除 (Exclusions)：
    • 检查 @SpringBootApplication(exclude=...) 或 spring.autoconfigure.exclude 配置，剔除用户明确禁用的自动配置类。
  2. 按需加载 (Conditions)：
    • 利用 @Conditional 及其派生注解，判断当前环境是否满足条件。如果不满足，该配置类直接跳过，不注入 Bean。
    • 例如：只有当 Classpath 下存在 redis.clients.jedis.Jedis 类时，RedisAutoConfiguration 才会生效。

核心组件：AutoConfigurationImportSelector

  • 它是 @EnableAutoConfiguration 导入的类。
  • 实现了 DeferredImportSelector 接口，确保自动配置是在用户自定义 Bean 注册之后才执行。这保证了用户的配置优先于默认配置（用户定义的 Bean 会覆盖自动配置的 Bean，或者导致自动配置不生效）。

3. 条件注解 (@Conditional)

这是实现“智能配置”的底层技术。掌握这些注解，你就能理解为什么“加个依赖就能用”以及“为什么我配了自己的 Bean，默认的就失效了”。

3.1 Class 条件 (判断依赖是否存在)

  • @ConditionalOnClass：当类路径下有指定的类时生效。
    • 场景：Redis 自动配置类上标注了 @ConditionalOnClass(RedisOperations.class)，只有你引入了 spring-data-redis 依赖，这个配置类才工作。
  • @ConditionalOnMissingClass：当类路径下没有指定的类时生效。

3.2 Bean 条件 (判断容器中是否有 Bean)

  • @ConditionalOnBean：当容器中存在指定 Bean 时生效。
  • @ConditionalOnMissingBean 【最重要】：当容器中不存在指定 Bean 时生效。
    • 场景：默认数据源配置。Boot 会检查你有没有自己定义 DataSource。如果你定义了，条件不满足，Boot 的默认数据源配置就会“退缩”，把控制权交给你。这是实现“约定优于配置”的核心手段。

3.3 Property 条件 (判断配置文件)

  • @ConditionalOnProperty：检查 application.yml 中是否有特定属性，或属性值是否匹配。
    • 示例：@ConditionalOnProperty(prefix = "app.email", name = "enabled", havingValue = "true")。只有配置了 app.email.enabled=true 时，邮件服务 Bean 才加载。

3.4 Web 条件

  • @ConditionalOnWebApplication：当前是 Web 应用时生效。
  • @ConditionalOnNotWebApplication：当前不是 Web 应用时生效。

3.5 资源条件

  • @ConditionalOnResource：类路径下存在指定文件（如 mybatis-config.xml）时生效。

四、生产准备

1. 嵌入式容器配置

1.1 容器切换

Spring Boot 默认使用 Tomcat，但在高并发非阻塞场景下，Undertow 表现更佳（内存占用更低，吞吐量更高）。

  • 切换步骤：

    1. 排除默认的 spring-boot-starter-tomcat。
    2. 引入 spring-boot-starter-undertow。

1.2 参数调优

无论使用哪个容器，生产环境都必须调整线程池配置，避免默认配置导致的性能瓶颈。

2. Spring Boot Actuator

2.1 Actuator 是什么？

2.1.1 核心定义

它的核心功能是为你的应用程序添加一系列生产就绪的特性，主要用于监控和管理正在运行中的应用程序。 能让你在不停止应用的情况下，通过HTTP端点或JMX（Java Management Extensions），轻松地窥探应用程序的内部状态、运行状况和各种度量指标。

2.1.2 解决应用的“黑盒”困境

想象一下，一个没有监控的应用被部署到生产服务器上后，它就像一个**“黑盒”**：

  • 它还活着吗？ 应用进程还在，但它是否能正常处理业务？是不是已经僵死或陷入了死循环？
  • 它健康吗？ 数据库连接是否正常？消息队列是否通畅？磁盘空间是否充足？
  • 它内部发生了什么？ 加载了哪些Bean？当前的环境变量和配置是什么？最近的HTTP请求是成功了还是失败了？JVM内存使用情况如何？
  • 如何优雅地关闭它？ 直接 kill 吗？这可能会导致正在处理的数据丢失。

在没有Actuator的时代，开发者和运维人员需要自己编写大量的代码、集成各种第三方库（如Metrics, JMX MBeans）来回答这些问题。这个过程不仅繁琐，而且缺乏统一的标准。

Actuator 的出现，就是为了打开这个“黑盒”，为所有Spring Boot应用提供一套标准化的、开箱即用的监控和管理解决方案。

2.1.3 工作原理与核心组件

Actuator 的工作原理非常简单，它通过 Spring Boot 的自动配置机制，向你的应用中自动注册一系列特殊的基础设施 Bean。其中最核心的组件是端点和健康检查体系。

1. 端点
  • 定义：一个端点就是 Actuator 暴露出的一个特定监控或管理功能的入口。每个端点都专注于一个方面的信息，例如 /health 端点用于检查应用健康状况，/metrics 端点用于提供详细的度量指标。
  • 暴露方式：默认情况下，Actuator 主要通过 HTTP 和 JMX 两种方式暴露这些端点。
    • HTTP：你可以通过访问特定的 URL（如 http://localhost:8081/actuator/health）来获取 JSON 格式的监控数据。这是最常用、最直观的方式，也便于与各种监控系统（如 Prometheus、Zabbix、Nagios）集成。
    • JMX：可以通过 JConsole、VisualVM 等 JMX 客户端连接到应用进程，以树状结构查看和管理应用。
2. 健康检查体系

/health 端点是 Actuator 中最重要、使用最频繁的端点。它背后有一套完整的体系，由健康指示器负责采集数据，由健康状态模型负责组织和表达数据。

1. 定时巡检 + 告警

五、原理深入与自定义 Starter

1. Spring Boot启动流程分析

我们所有Spring Boot应用的入口都是一个简单的 main 方法：

这行看似简单的 SpringApplication.run() 代码，背后隐藏了一个精心设计的、高度可扩展的启动流程。整个流程可以分为两大阶段：

  1. SpringApplication 实例的初始化阶段
  2. run() 方法的执行阶段

1.1 构造阶段 (new SpringApplication())

  1. 确定应用类型:
    • Spring Boot 会检查类路径中是否存在特定的类来判断当前应用的类型。
    • 如果 classpath 中存在 DispatcherHandler 并且不存在 DispatcherServlet，则应用类型为 REACTIVE (响应式Web应用)。
    • 如果两者都不存在，则应用类型为 NONE (非Web应用)。
    • 否则，默认类型为 SERVLET (基于Servlet的传统Web应用)。
    • 这个判断结果会决定后续是启动内嵌的Tomcat/Jetty还是Netty，或者不启动任何Web服务器。
  2. 加载 ApplicationContextInitializer 和 ApplicationListener
    • 主要来源：依然支持从 META-INF/spring.factories 加载（这是遗留的 SPI 机制）。
    • 新趋势：Spring Boot 3.0 开始支持并在核心包中逐步迁移到 META-INF/spring/*.imports 文件中。
    • ApplicationContextInitializer 允许我们在 ApplicationContext 刷新之前，对其进行一些编程方式的初始化操作，是一个非常早期的扩展点。例如，注册额外的属性源。
    • ApplicationListener 用于监听 SpringApplication 在启动过程中发布的各种事件（如 ApplicationStartingEvent, ApplicationReadyEvent 等），从而在特定阶段执行自定义逻辑。
    • 这些加载到的 Initializer 和 Listener 会被缓存到 SpringApplication 实例的成员变量中，等待后续在 run 阶段被调用。
  3. 确定主启动类:
    • 通过分析当前的线程调用栈（new RuntimeException().getStackTrace()），推断出哪个类包含了 main 方法，并将其记录下来。

1.2 运行阶段 (run() 方法)

Step 1: 创建并启动 StopWatch

  • Spring Boot 使用 StopWatch 来记录整个启动过程各个阶段的耗时，方便开发者进行性能分析。

Step 2: 准备 SpringApplicationRunListeners

  • 这是一个关键的事件发布机制，它允许我们在启动过程的各个生命周期节点发布事件。
  • 首先从spring.factories，加载所有 SpringApplicationRunListener 的实现类（默认是 EventPublishingRunListener）。
  • EventPublishingRunListener 的作用就是将 Spring Boot 内部事件转化为标准的 Spring ApplicationEvent，并广播给所有在第一阶段注册的 ApplicationListener。
  • 调用所有 RunListener 的 starting() 方法，标志着启动流程正式开始，发布 ApplicationStartingEvent 事件（但环境还未准备好）。

Step 3: 准备环境 (Environment)

  • 根据之前推断的应用类型，创建一个合适的 Environment 对象（例如，StandardServletEnvironment）。
  • 配置加载：Environment 负责管理应用的配置信息。它会加载和合并来自各种属性源（PropertySource）的配置。
  • 后处理：调用从 spring.factories 中加载的 EnvironmentPostProcessor，允许对 Environment 进行进一步的编程化修改。
  • 将准备好的 Environment 绑定到 SpringApplication 实例。
  • 调用 RunListener 的 environmentPrepared() 方法，并发布 ApplicationEnvironmentPreparedEvent 事件，通知监听者环境已经准备就绪，可以进行一些基于环境的调整了。

Step 4: 打印 Banner

  • 读取 classpath 下的 banner.txt 文件（如果存在），并将 Spring Boot 的横幅打印到控制台。可以通过 spring.main.banner-mode 配置来控制其行为（console, log, off）。

Step 5: 创建 ApplicationContext (应用上下文)

  • 根据推断出的应用类型（SERVLET, REACTIVE, NONE），创建对应的 ApplicationContext 实例。
    • Web应用 (SERVLET): 创建 AnnotationConfigServletWebServerApplicationContext。
    • 响应式Web应用 (REACTIVE): 创建 AnnotationConfigReactiveWebServerApplicationContext。
    • 非Web应用 (NONE): 创建 AnnotationConfigApplicationContext。

Step 6: prepareContext - 上下文预处理

在刷新上下文之前，进行一系列准备工作：

  • 将前面创建好的 Environment 设置到 ApplicationContext 中。
  • 调用所有在第一阶段加载的 ApplicationContextInitializer 的 initialize 方法，对上下文进行定制化处理。
  • 调用 RunListener 的 contextPrepared() 方法，并发布 ApplicationContextInitializedEvent 事件。
  • 将主启动类（如 MyApplication.class）加载到 ApplicationContext 中，这是配置的入口点。
  • 调用 RunListener 的 contextLoaded() 方法，并发布 ApplicationPreparedEvent 事件。

Step 7: refreshContext - 刷新上下文

  • 这是整个启动流程最核心、最耗时的一步。它会调用 ApplicationContext.refresh() 方法。
  • prepareRefresh(): 准备刷新，设置上下文状态。
  • obtainFreshBeanFactory(): 创建 DefaultListableBeanFactory，也就是 Bean 工厂，用于管理所有的 Bean 定义和实例。
  • prepareBeanFactory(): 对 Bean 工厂进行配置，比如设置类加载器、添加默认的 BeanPostProcessor 等。
  • postProcessBeanFactory(): 子类扩展点。对于 Web 应用，这里会添加与 Web 相关的 BeanPostProcessor。
  • invokeBeanFactoryPostProcessors(): 【关键】 调用所有注册的 BeanFactoryPostProcessor。这是自动配置的核心，@Configuration 注解的解析、@ComponentScan 的扫描、@Bean 方法的定义、@EnableAutoConfiguration 的触发都在这一步完成。Spring Boot 的自动配置类就是通过这个机制生效的。
  • registerBeanPostProcessors(): 【关键】 注册所有 BeanPostProcessor。这些处理器会在 Bean 初始化前、后对 Bean 实例进行干预，例如处理 @Autowired 依赖注入、AOP 代理的创建等。
  • initMessageSource(): 初始化国际化消息源。
  • initApplicationEventMulticaster(): 初始化事件广播器。
  • onRefresh(): 【关键】 子类扩展点。对于 Web 应用 (ServletWebServerApplicationContext)，这一步会 创建并启动内嵌的 Web 服务器 (如 Tomcat, Jetty)。
  • registerListeners(): 注册 ApplicationListener 监听器。
  • finishBeanFactoryInitialization(): 【关键】 实例化所有剩余的非懒加载单例 Bean。此时，IoC 容器中定义的绝大部分 Bean 都会被创建和初始化，依赖注入也在这里完成。
  • finishRefresh(): 完成刷新过程，发布 ContextRefreshedEvent 事件，标志着 IoC 容器已经准备就绪。

Step 8: afterRefresh 回调

  • 提供一个空的模板方法，留给子类扩展。

Step 9: 发布 ApplicationStartedEvent 事件

  • 调用 RunListener 的 started() 方法，并发布 ApplicationStartedEvent 事件。

Step 10: 调用 ApplicationRunner 和 CommandLineRunner

  • 从 ApplicationContext 中获取所有 ApplicationRunner 和 CommandLineRunner 类型的 Bean。
  • 对它们进行排序（@Order）。
  • 依次调用它们的 run 方法。这通常用于执行应用启动后需要立即运行的一些自定义任务（如加载初始数据、启动定时任务等）。

Step 11: 发布 ApplicationReadyEvent 事件

  • 调用 RunListener 的 ready() 方法 (在 Spring Boot 2.5 及以后版本中，原 running 方法被重命名为 ready)，并发布 ApplicationReadyEvent 事件。这标志着应用已完全启动，并准备好接收请求。
  • 如果启动过程中出现异常，则会发布 ApplicationFailedEvent 事件。

Step 12: 返回 ApplicationContext

  • run() 方法执行结束，返回创建并刷新好的 ApplicationContext 实例。

1.4.7 与其他初始化方式的对比

方式 执行时机 能否注入其他 Bean 适用场景
@PostConstruct 当前 Bean 初始化时 不保证其他 Bean 就绪 当前 Bean 自身的初始化
InitializingBean 当前 Bean 属性注入后 不保证其他 Bean 就绪 同上
ApplicationRunner 所有 Bean 就绪后 保证 应用级初始化任务
CommandLineRunner 所有 Bean 就绪后 保证 同上（参数处理简单时）
@EventListener(ApplicationReadyEvent) 应用完全就绪后 保证 同上（事件驱动风格）

核心区别：@PostConstruct 执行时，其他 Bean 可能还没初始化完；而 ApplicationRunner 执行时，整个容器已经完全就绪。

1.4.8 注意事项

1. Runner 中抛异常会阻止应用启动

这是实现 Fail-Fast 的基础：检查不通过就抛异常，阻止带病运行。

2. 长耗时任务应异步执行

3. 必须注册为 Spring Bean

2.1 命名规范 (千万别写反)

2.2 自定义

2.2.1 核心概念与组成

一个标准的自定义Starter通常由两个模块组成，这是最佳实践：

  1. xxx-spring-boot-autoconfigure 模块:
    • 核心功能模块。它包含了所有自动配置的代码、@ConfigurationProperties类以及需要被注册为Bean的业务类。
    • 这是Starter的“大脑”。它负责判断在何种条件下（@ConditionalOn...），应该创建哪些Bean，以及如何配置它们。
  2. xxx-spring-boot-starter 模块:
    • 依赖聚合模块。这个模块本身几乎没有任何代码。
    • 它的唯一作用就是在 pom.xml 中聚合依赖。它会依赖上面的 autoconfigure 模块，以及这个功能所需要的第三方库。
    • 这是用户唯一需要关心的模块。用户在他的项目中，只需要引入这一个Starter依赖，就能获得所有功能。

为什么要分成两个模块？

这是为了关注点分离。autoconfigure模块负责实现所有逻辑，而starter模块只负责提供一个简单的依赖入口。用户只需要依赖starter，而不需要关心具体的自动配置实现，使得依赖关系更清晰。
