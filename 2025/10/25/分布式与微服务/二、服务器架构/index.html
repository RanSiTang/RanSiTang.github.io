二、服务器架构

1. 服务治理 (Service Governance)

作用：管理微服务网络的生命周期，确保服务间可以互相发现、配置和通信。

  • 服务注册与发现：Nacos, Eureka, Consul
    • 核心：服务提供者启动时将自身信息注册到中心，消费者从中获取地址列表进行调用。
  • 配置中心：Nacos, Apollo, Spring Cloud Config
    • 核心：实现配置的集中管理、动态刷新，避免修改配置需要重启服务。

2. 服务通信 (Service Communication)

作用：定义服务之间如何交换信息，并保障通信的稳定性和效率。

  • 服务调用 (RPC/REST)
    • RESTful API: 基于HTTP协议，简单通用。
    • RPC (远程过程调用): 效率更高，协议私有。常用框架如 gRPC, Dubbo。
    • 声明式客户端: OpenFeign (将HTTP请求伪装成本地方法调用)。
  • 负载均衡 (Load Balancing)
    • 客户端负载均衡: Ribbon (已停更) / Spring Cloud LoadBalancer (官方推荐)。
    • 服务端负载均衡: Nginx, F5 等硬件/软件，以及API网关内置的负载均衡。
  • 服务容错 (Fault Tolerance)
    • 流量控制: Sentinel (QPS控制、线程数控制)。
    • 熔断降级: Hystrix (已停更) / Sentinel / Resilience4j (当依赖的服务异常时，快速失败或返回降级逻辑，防止雪崩)。
    • 线程/信号量隔离: Hystrix, Sentinel (限制单个服务可用资源，避免某个服务的故障耗尽整个系统的资源)。

3. API 网关 (API Gateway)

作用：作为系统的唯一入口，封装内部系统架构，为客户端提供统一服务。是流量的第一道关口。

  • 核心职责:
    • 请求路由: 将外部请求准确转发到内部对应的服务。
    • 安全认证: 统一处理登录、权限校验（JWT, OAuth2）。
    • 流量控制: 限流、熔断，保护后端服务。
    • 日志监控: 记录请求日志，实现全链路监控的入口。
    • 协议转换: 如 HTTP 转 RPC。
    • 响应聚合: 将多个内部服务的结果聚合后返回给客户端。
  • 主流产品: Spring Cloud Gateway (推荐), Zuul (旧版), Kong, APISIX。

4. 服务部署与容器化 (Service Deployment & Containerization)

作用：提供标准化的服务运行环境，简化部署流程，实现弹性伸缩和故障自愈。

  • 容器化技术: Docker
    • 核心：将服务及其依赖打包成一个轻量、可移植的容器镜像，实现“一次构建，到处运行”。
  • 容器编排: Kubernetes (K8s), Docker Swarm
    • 核心：自动化容器的部署、扩展和管理。K8s 是目前的事实标准，与云原生生态（Istio, Prometheus）深度集成。

5. 分布式安全 (Distributed Security)

作用：在分布式的环境下，解决认证（你是谁）和授权（你能做什么）的问题。

  • 认证方案:
    • 单点登录 (SSO): 用户只需登录一次，就可以访问所有相互信任的应用系统。
    • 认证协议: OAuth2, OpenID Connect (OIDC) 是目前主流的授权和认证标准。
  • 凭证技术: JWT (JSON Web Token)
    • 核心：在用户和服务器之间传递的一个紧凑且自包含的JSON对象，常用于承载用户身份信息和权限，实现无状态认证。
  • 实现框架: Spring Security, Sa-Token。
