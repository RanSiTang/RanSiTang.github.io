一、基础概念与架构

1. Spring Security 概述

Spring Security 是 Spring 生态中处理 认证（Authentication） 和 授权（Authorization） 的安全框架。Spring Boot 3.x 中引入 spring-boot-starter-security 即可开箱即用。

核心能力概括：

  • 认证：你是谁？（用户名密码 / Token / OAuth2 …）
  • 授权：你能干什么？（URL 级别 / 方法级别的权限控制）
  • 攻击防护：CSRF、CORS、Session 固定攻击、点击劫持等

2.1.1 DelegatingFilterProxy

  • **定位：**Servlet 容器管理的标准 Filter，由 Spring Boot 自动注册
  • **作用：**把请求 委托 给 Spring 容器中名为 springSecurityFilterChain 的 Bean
  • **原因：**Servlet 容器不认识 Spring Bean；它作为桥梁让 Spring 管理的安全逻辑能参与 Servlet FilterChain

你不需要手动配置它，Spring Boot 自动完成。

2.1.2 FilterChainProxy

  • 定位：Spring Security 的真正入口，本身也是一个 Filter, 也是 springSecurityFilterChain Bean
  • **作用：**内部维护 多个 SecurityFilterChain，根据请求 URL 匹配其中一条链执行
  • 特性：只匹配第一个命中的链，顺序由 @Order 控制（→ 参见五.2）

2.1.3 SecurityFilterChain

  • **定位：**一组有序的 Security Filter + 一个请求匹配规则
  • 一个项目可以有几个：多个，不同 URL 模式走不同安全策略
  • 由谁构建：HttpSecurity.build()

3.3 重点 Filter 详解

3.3.1 DisableEncodeUrlFilter

前后端分离不关注原因： 前后端分离使用 Token（JWT）而非 Session，不存在将 SessionId 编码到 URL 的场景，该 Filter 默认启用但无实际影响，无需额外配置关闭。

3.3.2 SecurityContextHolderFilter

职责： 管理当前线程的 SecurityContext 生命周期

面试考点：为什么请求结束要 clearContext()？

Tomcat 使用线程池，不清理会导致下一个请求拿到上一个用户的身份。

前后端分离要点： 虽然不用 Session，但 JWT Filter 解析 Token 后仍需要将认证信息存入 SecurityContextHolder，该 Filter 负责请求结束后清理，必须保留。

3.3.3 HeaderWriterFilter

职责： 写入安全响应头（X-Frame-Options、X-Content-Type-Options、X-XSS-Protection 等）

前后端分离中这些安全头仍有一定意义，但通常由 Nginx/网关统一处理。默认启用保留即可，一般不需要特别配置。

3.3.4 CorsFilter

职责： 处理跨域请求

前后端分离必须关注！ 前端（如 localhost:5173）与后端（如 localhost:8080）不同源，必须正确配置 CORS。

3.3.5 CsrfFilter

前后端分离不关注原因： CSRF 攻击利用浏览器自动携带 Cookie 的特性。前后端分离使用 Token（通过 Authorization Header 传递），浏览器不会自动附带，天然免疫 CSRF 攻击。

关闭方式：

3.3.6 LogoutFilter

前后端分离不关注原因： 传统登出需要清除服务端 Session 并重定向页面。前后端分离中登出由前端删除本地存储的 Token 完成（如需服务端 Token 黑名单则自行实现接口），不需要 Spring Security 内置的登出流程。

关闭方式：

3.3.7 UsernamePasswordAuthenticationFilter

前后端分离不关注原因： 该 Filter 只处理 表单提交（Content-Type: application/x-www-form-urlencoded），前后端分离发送的是 JSON（Content-Type: application/json），它无法解析。登录认证改为自定义 Controller 接收 JSON 请求。

关闭方式：

3.3.8 BasicAuthenticationFilter

前后端分离不关注原因： HTTP Basic 认证每次请求都需传输明文（Base64 编码的）用户名密码，安全性差且不适合前后端分离的 Token 认证模式。

关闭方式：

3.3.9 RequestCacheAwareFilter

前后端分离不关注原因： 该 Filter 用于”登录后跳转回原页面”的场景（缓存登录前的请求）。前后端分离中路由跳转由前端控制，不需要服务端记住原始请求。

关闭方式：

3.3.10 SecurityContextHolderAwareRequestFilter

职责： 包装 HttpServletRequest，使其支持 Servlet API 的安全方法（request.isUserInRole()、request.getRemoteUser() 等）

前后端分离中很少直接使用 Servlet API 的安全方法（通常通过 SecurityContextHolder 获取），但该 Filter 开销极小且默认启用，一般不需要关闭。

3.3.11 AnonymousAuthenticationFilter

职责： 未认证时填充匿名身份

前后端分离仍需关注： 该 Filter 确保 SecurityContextHolder 中始终有 Authentication 对象（即使是匿名的），使后续 AuthorizationFilter 能统一判断，避免空指针。建议保留默认行为。

3.3.12 ExceptionTranslationFilter

职责： 捕获 AuthorizationFilter 抛出的异常，转换为合适的响应

重要： 它只捕获后续 Filter 的异常

在传统的 Spring Security 架构中，登录请求是由内置的 Filter（如 UsernamePasswordAuthenticationFilter）拦截处理的。认证成功或失败后，会回调 AuthenticationSuccessHandler 和 AuthenticationFailureHandler。

但在前后端分离项目中，面对这套机制，我们面临一个重要的架构选择：用 Controller 接管，还是坚持用 Filter？

5.1 现代主流方案：Controller 登录（强烈推荐，无需 Handler）

在前后端分离项目中，前端发送的是包含 JSON 的 POST 请求。由于 Spring Security 自带的 formLogin() 只支持表单格式（x-www-form-urlencoded），不认 JSON，最简单且最主流的做法是完全抛弃 formLogin 和 Handler，直接在 Controller 中写登录接口。

优点：完全掌控请求与响应，支持 JSON，无需编写各种冗长的 Handler。

结论：如果你采用上述方案，你可以直接跳过本章后续的 Handler 自定义配置。

  1. 老旧项目改造：前端依然用表单提交，但要求返回 JSON 而非重定向。

  3. 没有 securityMatcher 的链匹配所有请求，必须放在最后

3. CORS 配置【详】

CORS（Cross-Origin Resource Sharing）是浏览器的同源策略限制下，允许前端跨域访问后端 API 的标准机制。

3.1 跨域问题的本质

简单请求 vs 非简单请求

类型 条件 示例

1. 接口幂等/防重放 与 Security 的结合【略】

接口幂等和防重放本质上是业务安全问题，不属于 Spring Security 的核心职责，但可以借助 Security 的 Filter 机制和用户上下文来实现。

常见方案

方案 实现方式 适用场景
Token 令牌机制 请求前获取一次性 Token，提交时携带，服务端校验后删除 表单重复提交
请求签名 + 时间戳 + Nonce 对请求参数签名，服务端校验时间窗口和 nonce 唯一性 开放 API 防重放

业务层幂等键 客户端生成唯一请求 ID，服务端用 Redis SETNX 去重 通用方案

  • 防重放 Filter 可通过 addFilterAfter(replayFilter, JwtAuthenticationFilter.class) 插入 Security 链
  • 从 SecurityContextHolder 获取当前用户信息，结合用户维度做限流/去重
