Function Calling与Tool Calling

发表于 2026-05-29 更新于 2026-06-09 分类于 AI

一、Function Calling 的基本概念与原理

1. 什么是 Function Calling

Function Calling（函数调用，也称 Tool Calling / 工具调用）是 LLM 的一项关键能力：模型不再只能输出文本，它可以”决定”调用一个预定义的外部函数，并生成符合函数签名的参数。

准确地说，Function Calling 本身并不是模型在”执行”函数，而是模型在”请求”执行——模型输出一个结构化的函数调用请求（包含函数名和参数），由外部程序（如你的 Java 后端）真正执行函数，然后将执行结果返回给模型，模型再基于结果生成最终响应。

将 Function Calling 放在整个 AI 应用架构的坐标系里观察，它是连接 LLM 与外部世界的桥梁。没有它，LLM 只是一个”会说话的脑子”——能理解问题、能生成文本，但无法查询实时数据、无法执行操作。有了 Function Calling，LLM 变成了一个”能办事的智能体”——它可以查询数据库、调用 API、触发流程，真正参与到业务系统的运作中。

2. Function Calling 解决的核心问题

问题一：LLM 无法访问实时数据

LLM 的知识来自静态的训练数据。对于实时信息——当前股价、库存数量、用户订单状态——模型要么不知道，要么用训练数据中的过时信息”编造”。Function Calling 让模型能调用后端 API 查询实时数据，再基于真实数据生成回答。

问题二：LLM 无法执行操作

LLM 天然只能输出文本，不能执行任何实际操作——不能创建工单、不能发送邮件、不能修改数据库。Function Calling 让模型能通过调用后端函数来”触发”这些操作。

问题三：LLM 需要结构化的外部交互

用户用自然语言表达意图，但后端 API 需要结构化的参数。Function Calling 承担了”自然语言 → 结构化参数”的转换——用户说”帮我查一下上周的订单”，模型提取出 { "dateRange": {"start": "2026-05-26", "end": "2026-06-01"} }。

能力 没有 Function Calling 的 LLM 有 Function Calling 的 LLM
获取实时数据 ❌ 只能编造或拒绝 ✅ 调用 API 查询真实数据
执行操作 ❌ 只能输出文本建议 ✅ 触发业务流程执行
结构化交互 ❌ 只能输出自然语言 ✅ 生成结构化的函数调用参数
多系统协作 ❌ 完全隔离 ✅ 作为协调者调用多个系统

3. 工作原理：模型端的工具选择与参数推理

Function Calling 的核心机制可以分为两个阶段：

阶段一：工具选择（Tool Selection）

模型在收到用户消息后，基于 Prompt 中注册的工具定义（Tool Definitions），判断是否需要调用某个工具。如果需要，选择最合适的工具；如果不需要，直接生成文本回复。

模型做这个判断的依据是工具定义中的 description——工具描述写得好坏，直接决定了模型能否在正确的时机选择正确的工具。描述模糊的工具可能永远不会被调用，描述过于宽泛的工具可能在不恰当的时机被调用。

阶段二：参数推理（Argument Inference）

选定工具后，模型基于工具定义中的 parameters（JSON Schema）和用户消息，推理出调用该工具所需的参数值。这个过程本质上是将自然语言中的信息抽取并映射为结构化的 JSON 参数。

模型并不总是选择工具：模型可以判断当前对话是否超出了所有工具的能力范围，从而选择不调用任何工具直接回复。这也是 Function Calling 设计的精妙之处——工具是”可用选项”，不是”强制执行”。

2. 工具描述的最佳实践

工具定义的 description 是模型判断何时调用该工具的唯一依据。写好工具描述是 Function Calling 工程中最重要也最容易被忽视的环节。

好的工具描述应该包含：

  • 做什么（核心功能）：一句话概括工具的核心功能
  • 什么时候用（适用场景）：明确告诉模型在什么情况下应该调用这个工具
  • 什么时候不用（排除场景）：明确告诉模型什么情况下不应该调用这个工具
  • 参数说明：每个参数的含义、格式和约束

描述中毒与防护：工具描述可以被视为一种特殊的”System Prompt”。如果工具描述中包含敏感信息（如内部系统的接口地址、数据库表名），这些信息会泄露给模型提供商。工具描述应只包含模型选择工具和生成参数所需的信息，避免暴露内部实现细节。

3. 参数设计原则

一、明确性：每个参数的目的必须在模型的认知范围内可理解

参数 description 不能太简略。"type": "string", "description": "ID" 是不够的——什么 ID？格式是什么？模型需要这些信息来正确推理。

二、约束性：尽量使用 enum 而非自由文本

如果某个字段只有有限的几个合法值，使用 enum 约束。这不仅提高参数生成准确率，也是安全加固的起点。

三、简洁性：参数数量控制在合理范围

研究表明，参数数量超过 5 个时，模型生成参数的质量开始下降。如果确实需要很多参数，考虑将部分参数设为非必填 required 之外，让模型只在必要时填充。

四、区分度：不同工具的职责要明确分离

两个功能相近的工具（如 searchOrders 和 listOrders）会让模型选择困难，导致调用错误的工具。合并功能相近的工具，或为每个工具写清适用场景的差异。

三、工具调用执行机制

1. 同步执行与异步执行

同步执行是最简单直接的模型——收到 Function Calling 请求后，阻塞当前线程等待执行结果，将结果发回模型后继续对话。适合绝大多数实时问答场景，用户等待时间 = 模型推理时间 + 工具执行时间 + 模型再次推理时间。

异步执行适用于耗时较长的操作（如批量处理、数据导出）。基本模式是：收到调用请求后立即返回一个”任务已提交”的确认，后端异步执行，执行完成后通过回调或轮询将结果注入对话。

同步执行 异步执行
适用场景 查询、简单操作（< 5s） 批量处理、外部调用（> 5s）
用户体验 等待完整结果 先收到确认，后续获取结果
实现复杂度 低 中（需要任务状态管理和回调）
超时处理 直接返回超时错误 轮询状态或主动推送

2. 多轮工具调用（Multi-turn）

单轮对话中可以发生多次工具调用。典型场景是链式查询——先查 A 得到结果，再基于 A 的结果查 B。

多轮调用中的上下文管理：每一轮的模型输入包含前面所有轮的消息（用户消息 + 工具调用 + 工具返回）。这个上下文需要被妥善管理——部分模型的 Context Window 有限，过长的工具调用链可能导致上下文溢出。

终止条件：模型在任一回合都可以选择不调用工具（finish_reason: "stop"），直接生成最终回复。这是自然的终止条件。但在代码实现中需要设置一个最大轮次上限（如 10 轮），防止模型陷入无限循环调用。

3. 并行工具调用（Parallel Tool Calling）

当模型判断用户的问题需要调用多个相互独立的工具时，可以在单轮中同时发起多个工具调用。

判断工具是否可以并行的标准：两个工具调用之间没有数据依赖。如果工具 B 的输入参数依赖于工具 A 的输出，则不能并行，必须先执行 A 再执行 B。

性能收益：对于 I/O 密集型工具（如调用外部 API、查询数据库），并行执行可以将多轮串行调用的延迟从”T1 + T2 + …”降低为”max(T1, T2, …)”。

4. 强制调用工具与工具选择策略

在某些场景下，业务逻辑要求必须调用特定工具，而不是让模型”自己决定”。例如，用户进入”订单查询”菜单时，系统强制调用 queryOrder 工具，模型不能选择直接聊天。

控制方式：

策略 说明 适用场景
tool_choice: "auto" 模型自行决定是否调用工具（默认） 大多数场景
tool_choice: "required" 必须调用至少一个工具 结构化输入表单、菜单导航
tool_choice: "none" 禁止调用工具 纯聊天场景
tool_choice: {"type": "function", "function": {"name": "xxx"}} 强制调用指定工具 明确的业务流程节点

实践建议：大多数场景保持 auto，让模型自行判断。在以下情况使用强制模式：

  • 业务流程明确需要特定数据时（如工单创建页面强制调用 createTicket）
  • 可用工具有限且都需要调用时（required）
  • 需要防止模型绕过工具直接回复的场景

四、工具调用的工程安全

Function Calling 将后端 API 暴露给了 LLM——这带来了一个全新的安全边界。传统 API 安全的假设是”调用方是可信任的内部系统”，但 LLM 作为调用方会引入新的风险：模型可能生成不合法的参数、可能被 Prompt Injection 诱导调用不该调用的工具、可能在循环中过度调用。

1. 参数白名单与输入校验

LLM 生成的参数不可信——模型可能因为幻觉生成不合法的参数值，也可能因为 Prompt Injection 被注入恶意的参数。

2. 权限控制

不是所有用户都有权调用所有工具。一个普通员工不应能调用 deleteAllOrders，即使 LLM “决定”调用它。

权限控制的核心问题：LLM 的”身份”是谁？答案是——LLM 的调用权限等于当前对话用户的权限。工具调用必须在当前用户的权限上下文中执行。

工具定义中的权限声明：在工具注册时标记所需的权限级别，注册阶段就过滤掉当前用户无权访问的工具——不将无权工具的定义发送给模型，从源头上消除模型调用无权限工具的可能性。

3. 幂等性设计

LLM 可能在同一个对话中多次调用同一个工具——例如，模型”忘记”刚才已经查过订单状态，又调用了一次。写操作工具必须设计为幂等的，保证多次调用不会产生副作用累积。

实践中可以显式暴露 idempotencyKey 参数：让模型在调用时生成一个唯一的幂等键（可以要求模型使用 UUID），后端基于此键去重。

4. 超时与重试

工具执行可能因下游系统故障而长时间阻塞。需要为每个工具调用设置超时和重试策略。

5. 调用频率限制

防止 LLM 在循环中过度调用工具——例如，模型陷入”调用-失败-重试”的死循环，或 Prompt Injection 触发高频调用。

3. 调用审计与监控

Function Calling 引入了与传统 API 调用不同的审计维度——需要记录的不只是”谁调用了什么”，还要记录”LLM 决定了调用什么”。

安全校验层不是 Fuction Calling 的可选附加项，而是必备组件。在将后端 API 暴露给 LLM 的架构中，安全校验是唯一的安全屏障——LLM 可能犯错、可能被注入、可能被滥用，但安全校验层必须确保这些错误不会转化为实际的业务损害。校验层的设计原则是：宁可拒绝合法的调用，也不能放过非法的调用（fail-safe）。
