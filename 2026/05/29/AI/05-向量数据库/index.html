向量数据库

发表于 2026-05-29 更新于 2026-06-10 分类于 AI

一、向量数据库的核心问题与定位

1. 什么是向量数据库

向量数据库（Vector Database）是专门用于存储、索引和检索高维向量数据的数据库系统。与传统数据库操作结构化字段（如 WHERE age > 18）不同，向量数据库的核心操作是相似性检索（Similarity Search）——给定一个查询向量，在海量向量中找到最相似的 TopK 个结果。

用更直观的方式理解：传统数据库回答的是”精确匹配”问题——“找出所有年龄等于 25 的用户”；向量数据库回答的是”语义相似”问题——“找出与这段话含义最接近的 10 篇文档”。传统数据库的查询结果是确定性的（要么满足条件，要么不满足）；向量数据库的查询结果是近似性的（每个结果有一个相似度分数，分数越高越相关）。

向量数据库与 Embedding 模型的关系是上下游协作：Embedding 模型负责将非结构化数据（文本、图像、音频）转换为向量；向量数据库负责存储这些向量，并在需要时快速找到最相似的向量。两者组合起来，构成了 AI 应用中语义检索的基础设施。

对比维度 传统数据库（MySQL/PostgreSQL） 向量数据库（Milvus/Qdrant）
核心查询 精确匹配、范围查询 相似性检索（ANN）
索引结构 B-Tree、Hash、倒排索引 HNSW、IVF、PQ 等向量索引
查询结果 确定性（满足条件/不满足） 近似性（相似度分数排序）
典型操作 WHERE, JOIN, GROUP BY search(vector, topK)
适用数据 结构化数据 非结构化数据的向量表示

2. 为什么需要专用的向量数据库

一个自然的疑问是：传统数据库不能存向量吗？PostgreSQL 可以用 float[] 或 ARRAY 类型存储向量，然后手动计算相似度。然而这种”手工”方案在数据量超过十万级时会遇到无法克服的性能瓶颈。

传统数据库存储向量的局限：

当数据量达到 10 万条时，用 PostgreSQL 做一次暴力全表扫描的相似度计算需要遍历所有向量，单次查询耗时可能达到数百毫秒甚至秒级。当数据量增长到百万级，暴力扫描变得不可接受——它是 O(n) 复杂度，且每增加一个维度，计算成本线性增长。

向量数据库的解决思路：使用 ANN（Approximate Nearest Neighbor，近似最近邻）算法。ANN 的核心思想是：不需要扫描所有向量，而是通过索引结构快速定位到向量空间中”可能包含最近邻”的局部区域，只在该区域内计算精确相似度。ANN 将检索复杂度从 O(n) 降低到 O(log n) 甚至更低，同时将检索精度损失控制在 1%-5% 以内。

传统数据库暴力扫描 向量数据库 ANN 索引
检索复杂度 O(n) O(log n)
10 万向量检索 ~100ms ~5ms
100 万向量检索 ~1s ~10ms
1 亿向量检索 ~100s（不可用） ~50ms
精度 100%（精确） 95%-99%（近似）

专用向量数据库的额外能力：

  • 向量索引的优化存储：向量数据以紧凑的二进制格式存储，支持内存映射（mmap），最大化 I/O 吞吐
  • 标量过滤与向量检索的融合：在向量检索的同时过滤元数据条件（如”只检索 2024 年之后的文档”），这比”先向量检索再应用过滤”高效得多
  • CRUD 与索引动态更新：支持增删改向量而不需要重建整个索引
  • 水平扩展：原生支持分片（Sharding）和副本（Replication），数据量增长时可以横向扩展

3. 向量数据库在 AI 架构中的位置

在典型的 RAG（Retrieval-Augmented Generation，检索增强生成）架构中，向量数据库处于检索层的核心位置：

向量数据库不是一个”存了就完”的组件。它的检索质量直接决定了 RAG 系统的召回率（Recall）——即相关文档中有多少比例被成功检索到。如果向量数据库的检索召回率低，后面的 LLM 再强也无法给出正确答案，因为”巧妇难为无米之炊”。

二、向量存储与数据模型

1. 向量数据库的基本数据模型

向量数据库的数据模型通常围绕以下几个核心概念构建：

Collection（集合）：向量数据库中最高层级的组织单位，类似于关系数据库中的”表”（Table）。一个 Collection 包含一组具有相同维度和相同 Schema 的向量数据。例如，一个 RAG 应用可能创建 documents 和 images 两个 Collection，分别存储文档 Embedding 和图片 Embedding。

Vector（向量）：Collection 的核心字段，是固定维度的浮点数数组。每个向量代表一条数据在语义空间中的位置。向量的维度由 Embedding 模型决定，一旦 Collection 创建并指定了维度，就不能再改变。

Payload / Metadata（负载/元数据）：除向量外，每条数据可以附带结构化字段（标量数据），如标题、作者、创建时间、分类标签等。这些元数据有两个关键作用：一是在检索时支持标量过滤（”只返回分类为’技术文档’的结果”），二是在检索结果返回后提供上下文信息（显示给用户看的文档标题和来源）。

ID（标识符）：每条向量数据的唯一标识，通常使用 UUID 或业务自定义 ID。ID 用于向量的增删改操作。

2. 向量存储格式

向量数据库在底层需要高效地存储高维浮点数数组。存储格式的选择直接影响 I/O 性能和内存占用。

Float32（FP32）：每个维度使用 4 字节 IEEE 754 单精度浮点数。这是最常见也是默认的存储格式。1536 维向量占用 6KB。精度无损，适用于大多数场景。

Float16（FP16 / 半精度）：每个维度使用 2 字节。存储空间减半，但精度有所损失。现代 GPU 原生支持 FP16 运算，在某些 GPU 加速向量数据库中可以获得更好的吞吐。

Binary（二值化向量）：将每个维度的浮点数量化为 0 或 1（根据符号或阈值），存储空间大幅降低（每维度 1 bit），检索使用汉明距离（Hamming Distance）。适用于对精度要求不高但数据量极大的场景（如数十亿级去重）。

Scalar Quantization（标量量化）：将 Float32 映射为 Int8（1 字节/维度），存储空间降为原来的 1/4。大多数向量数据库支持此选项，精度损失可控（通常 1%-3%）。

存储格式 每维度字节数 1536维向量大小 精度 适用场景
Float32 4 6 KB 无损 默认选择
Float16 2 3 KB 微量损失 GPU 加速场景
Int8（标量量化） 1 1.5 KB 1%-3% 损失 大规模低成本存储
Binary 1 bit 192 B 损失较大 数十亿级去重/粗筛

存储容量评估公式：

例如，100 万条 1536 维 Float32 向量，元数据平均 500 字节：

  • 向量数据：100万 × 1536 × 4B = 5.76 GB
  • 元数据：100万 × 500B = 0.5 GB
  • 索引开销：约 2-3 GB
  • 总计：约 8-10 GB

3. 标量存储与元数据管理

向量数据库不仅仅存储向量，还需要高效管理元数据（标量字段）。元数据管理的两个核心能力是标量索引和过滤下推。

标量索引：向量数据库通常为元数据字段建立倒排索引或 B-Tree 索引，以加速标量过滤条件。例如，WHERE category = '技术文档' 会使用倒排索引快速定位到满足条件的向量。

过滤下推（Predicate Push-Down）：在向量检索之前先应用标量过滤，缩小检索范围。高效的向量数据库会同时利用标量索引和向量索引，在索引遍历过程中就排除不满足标量条件的向量，而不是”先搜出 1000 条再过滤掉 900 条”。

元数据 Schema 设计建议：

  • 为高频过滤字段（如 category、status）建立标量索引
  • 避免将大段文本放入元数据，元数据应仅包含检索过滤和结果展示所需的最小字段集合
  • 枚举值优于自由文本（category: "technical" 比 description: "这是一篇技术文档..." 更适合做过滤条件）

三、向量索引：ANN 近似最近邻搜索

1. 精确检索 vs 近似检索

精确最近邻检索（KNN, K-Nearest Neighbors） 是对数据库中所有向量逐一计算与查询向量的距离，然后排序返回 TopK。精确检索的精度是 100%，但复杂度是 O(n × d)，其中 n 是向量总数，d 是维度。当 n 达到百万级时，KNN 已经不可接受。

近似最近邻检索（ANN, Approximate Nearest Neighbors） 的核心思想是：牺牲少量精度，换取数量级的性能提升。ANN 不扫描所有向量，而是通过精巧的索引结构快速定位到高概率包含最近邻的局部区域，只在该区域内做精确距离计算。

ANN 算法的核心权衡是精度 vs 速度 vs 内存的三角矛盾：

任何一个 ANN 算法在这三个维度上都有取舍。例如：

  • HNSW：精度高、速度快，但内存开销大（需要存储图结构）
  • IVF-PQ：内存省，但精度和速度相对较低
  • DiskANN：内存极省（依赖磁盘），但速度较慢

2. 主流向量索引算法

2.1 基于图的索引：HNSW

HNSW（Hierarchical Navigable Small World，分层可导航小世界图）是当前最广泛使用的向量索引算法，被 Milvus、Qdrant、Weaviate 等几乎所有向量数据库支持。

原理：HNSW 构建一个多层图结构。每一层是一个近邻图（每个向量连接其最近的 M 个邻居），层级越高节点越稀疏。检索时，从最高层（最稀疏的层）开始，沿着图结构快速下降到最底层（最稠密的层），在最底层进行精细搜索。

HNSW 的核心参数：

参数 含义 建议值 影响
M 每个节点的最大连接数 16-64 M 越大，精度越高，但内存和构建时间增加
efConstruction 构建时的搜索宽度 100-500 越大精度越高，构建越慢
efSearch 检索时的搜索宽度 64-512 越大精度越高，检索越慢

HNSW 的特点：

  • 增量构建：支持在线插入新向量，不需要重建索引
  • 检索速度极快：在内存中的检索延迟通常在毫秒级
  • 高召回率：通常可以达到 95%-99% 的召回率
  • 内存开销大：图结构本身占用额外内存，约为向量数据的 100%-200%

2.2 基于量化的索引：IVF-PQ

IVF-PQ（Inverted File with Product Quantization，倒排文件+乘积量化）是将聚类和量化结合的两阶段索引。它是大规模向量检索场景中性价比最高的方案之一。

原理：

  • IVF（Inverted File）：使用 K-Means 将所有向量聚类为 N 个簇（Cluster），每个簇有一个质心（Centroid）。检索时，只搜索与查询向量最近的几个簇（nprobe 参数控制搜索的簇数量），而不是全部簇
  • PQ（Product Quantization）：将高维向量切分为多个子段，每个子段独立做聚类和量化。向量不存储原始浮点数，而是存储每个子段所属的聚类 ID（一个很小的整数），极大压缩了存储
参数 含义 影响
nlist（簇数） K-Means 聚类数量 越大检索越快，但精度越低；推荐 sqrt(n) ~ 4 × sqrt(n)
nprobe 检索时搜索的簇数 越大精度越高，速度越慢
M（PQ 子段数） 向量切分为多少子段 必须是维度的因子；越大精度越高，压缩率越低

IVF-PQ 的特点：

  • 存储压缩率高（可以将 1536 维 Float32 向量从 6KB 压缩到几十字节）
  • 适合离线批量构建（K-Means 聚类需要全量数据），不支持高效的在线增量插入
  • 精度低于 HNSW，但对于超大规模场景（十亿级），往往是唯一可行的方案

2.3 基于树的索引：Annoy

Annoy（Approximate Nearest Neighbors Oh Yeah）是 Spotify 开源的基于树的 ANN 算法。原理是递归地使用随机超平面将向量空间二分，构建一棵二叉树。每棵树的构建是随机的，通过构建多棵树来提高检索精度。

Annoy 的特点是实现简单、内存友好，但精度和速度都不如 HNSW，现代向量数据库已较少将其作为默认索引。

2.4 基于哈希的索引：LSH

LSH（Locality-Sensitive Hashing，局部敏感哈希）是最早的 ANN 算法之一。原理是通过哈希函数将向量映射到不同的哈希桶中，使得相似的向量有更高的概率被映射到同一个桶。检索时只计算同一桶内的向量距离。

LSH 的优点是理论优雅、支持动态增删，但精度明显低于 HNSW 和 IVF-PQ，在现代向量数据库中已很少作为主力索引。

IVF-PQ ★★★☆☆ ★★★★☆ ★★★★★ ❌ 差 亿~十亿级

选型决策路径：

在 Java 后端实践中，大多数企业级 RAG 应用的数据量在十万到千万级，HNSW 是默认也是最优选择。只有在处理数十亿级数据（如大规模的网页索引、电商商品向量检索）时，才需要评估 IVF-PQ 等压缩索引方案。

四、相似度度量与检索策略

1. 相似度度量

向量数据库支持多种相似度（距离）度量方式。度量方式的选择取决于 Embedding 模型的训练方式——大多数现代 Embedding 模型使用余弦相似度（Cosine Similarity）进行训练，因此将其作为默认推荐。

余弦相似度（Cosine Similarity）：衡量两个向量在方向上的相似程度，值域 [-1, 1]，1 表示完全相同方向，0 表示正交（无关），-1 表示完全相反。余弦相似度消除了向量长度的影响，只关注方向——这对于文本 Embedding 特别重要，因为文本长度不应影响语义相似度判断。

欧氏距离（Euclidean Distance / L2）：衡量两个向量在空间中的直线距离，值域 [0, ∞)，0 表示完全相同。欧氏距离受向量长度影响大，适用于向量长度本身有意义的场景（如图像特征向量）。

内积（Inner Product / Dot Product）：向量点积，值域 (-∞, ∞)。内积同时受向量方向和长度影响。某些经过归一化训练的 Embedding 模型使用内积作为相似度度量。对于归一化向量（长度为 1），内积等于余弦相似度。

度量方式 值域 受向量长度影响 适用场景
余弦相似度 [-1, 1] 否 文本 Embedding（推荐）
欧氏距离 [0, ∞) 是 图像特征、位置坐标
内积 (-∞, ∞) 是 归一化向量、推荐系统

在向量数据库中选择度量方式的实践原则：与 Embedding 模型的训练目标保持一致。OpenAI、BGE、Cohere 等主流文本 Embedding 模型都使用余弦相似度，因此 RAG 应用几乎统一使用余弦相似度。

2. TopK 检索

TopK 检索是向量数据库最基本的查询操作：给定一个查询向量，返回相似度最高的 K 个结果。

TopK 的参数设计：

  • K 值太小（如 K=1 或 K=2）：可能遗漏相关信息，特别是在知识库中有多篇内容相关的文档时
  • K 值太大（如 K=50 或 K=100）：虽然不容易遗漏信息，但会引入大量噪声，增加 LLM 的推理成本和干扰。且超大的 K 值意味着更多的 Context Window 消耗
  • K 值的”双重作用”：K 不只控制返回数量，它还是检索深度和搜索结果多样性的控制参数

对于 RAG 场景，推荐的 K 值范围是 3-10。具体选择取决于：

  • Chunk 大小：Chunk 越小，K 可以稍微增大
  • 知识库规模：知识库越大，信息来源越分散，K 需适当增大
  • LLM Context Window：窗口越大，可以接受更多的检索结果

相似度阈值（Score Threshold）：除了限制数量，还可以限制质量——只返回相似度高于某个阈值的结果。例如设置阈值为 0.7，排除相似度低于 0.7 的结果。这样即使设定 K=10，如果只有 3 条结果超过阈值，就只返回 3 条。阈值过滤可以防止 LLM 基于不相关内容”强行作答”。

3. Metadata Filter（标量过滤）

纯语义检索在某些场景下不足——例如用户只想搜索”2024 年之后的文档”或”某个特定作者的文章”。这些条件无法通过语义向量表达，需要标量过滤与语义检索结合。

Metadata Filter（标量过滤）的核心能力：

过滤策略的两种实现方式：

  • Pre-filtering（前置过滤）：先用标量索引筛选出满足条件的候选集，再在候选集中做向量检索。优点是保证结果一定满足过滤条件；缺点是如果过滤条件筛选出的候选集非常小，可能影响语义检索的多样性
  • Post-filtering（后置过滤）：先做向量检索获取 TopK 结果，再从结果中过滤掉不满足标量条件的。缺点是可能过滤后结果不足 K 个——极端情况向量检索返回的 K 个结果全都不满足过滤条件

现代向量数据库（Milvus、Qdrant 等）已经实现了融合过滤——在向量索引遍历的过程中同步应用标量过滤，兼顾效率和结果质量。

4. Hybrid Search（混合搜索）

Hybrid Search（混合搜索）是同时使用语义检索（向量相似度）和关键词检索（倒排索引/BM25）的搜索策略，融合两者的优势。

为什么需要混合搜索：

  • 纯向量检索在长尾查询、精确名称匹配（如型号”ThinkPad T480”）上表现欠佳——这些查询包含独特的专有名词，向量模型可能没有充分学习
  • 纯关键词检索无法理解语义——用户搜”笔记本散热问题”，关键词匹配找不到”笔记本电脑温度过高”的文档
  • 混合搜索结合两者：关键词检索负责精确匹配（专有名词、代码、编号），向量检索负责语义覆盖（同义词、语义相近的表达）

5. 多向量检索与重排序

多向量检索（Multi-Vector Retrieval）：对于某些复杂场景，单一的查询向量不足以表达用户的检索意图。可以生成多个不同角度的查询向量，分别检索后合并结果。例如，用户的原始问题是”如何优化 Spring Boot 应用的启动速度”，可以生成两个变体查询——“Spring Boot 启动优化方法”和”Spring Boot 慢启动原因分析”——分别检索后合并去重。

重排序（Re-ranking）：向量检索的 TopK 结果是基于向量相似度的初步排序。在某些高精度场景中，可以加一个重排序步骤——用一个更强大的模型（如 Cross-Encoder）对 TopK 结果重新打分排序。Cross-Encoder 同时接收查询和文档作为输入，输出精确的相关性分数，比向量相似度更准确，但计算成本也更高。

这种”粗筛+精排”的两阶段策略在 Google、Bing 等搜索引擎中广泛使用，也适用于对检索质量要求极高的企业 RAG 系统。

五、主流向量数据库方案对比

1. 专用向量数据库

1.1 Milvus

Milvus 是 LF AI & Data 基金会旗下的开源向量数据库（由 Zilliz 主导开发），是目前功能最全面、社区最活跃的向量数据库之一。采用云原生架构，支持存算分离、多租户和数据持久化。

核心特性：

  • 支持多种索引类型：HNSW、IVF-Flat、IVF-PQ、DiskANN 等 10+ 种索引
  • 支持标量过滤、混合搜索（向量 + BM25）、范围检索、多向量字段
  • 原生支持分片（Sharding）和副本（Replication），可水平扩展
  • SDK 覆盖 Python、Java、Go、Node.js 等主流语言
  • 提供 Milvus Lite（嵌入式版本）和 Zilliz Cloud（全托管版本）

架构层次：

Milvus 适合需要存算分离、大规模扩展的生产环境。部署复杂度较高（需要 ETCD、MinIO、Pulsar 等中间件），但 Milvus Lite 版本降低了入门门槛。

1.2 Qdrant

Qdrant 是用 Rust 语言编写的开源向量数据库，以单二进制部署、API 设计优雅、性能出色著称。

核心特性：

  • 单二进制文件部署，无需外部依赖，运维成本极低
  • 默认使用 HNSW 索引，检索性能优秀
  • 原生支持稀疏向量（Sparse Vector）和稠密向量（Dense Vector），可以原生实现混合搜索
  • 支持 Payload 标量过滤、全文检索、地理位置检索
  • 内置 Snapshots 快照备份和恢复

与 Milvus 的部署复杂度对比：

Milvus（分布式版） Qdrant
部署依赖 ETCD + MinIO + Pulsar 无（单二进制）
最小内存 8 GB+ 256 MB
水平扩展 原生支持 Qdrant Cloud 支持
适合场景 大规模生产环境 中小规模 / 快速启动

Qdrant 特别适合中小规模（千万级以下）的向量检索场景，以及在 Docker Compose 或 K8s 中快速搭建的私有化部署需求。

1.3 Weaviate

Weaviate 是一个同时支持向量检索和 GraphQL 查询的开源向量数据库，特色是支持原生混合搜索和多模态数据。

核心特性：

  • Hybrid Search 原生内置：不需要额外组件，单次查询即可融合 hybrid 搜索（向量 + BM25），通过 alpha 参数调节权重
  • 多模态支持：内置多模态模块，可以直接存储和检索图片、文本、音频
  • GraphQL API：查询语言基于 GraphQL，表达能力强
  • 模块化设计：通过插件模块集成外部 Embedding 模型（如 OpenAI、Cohere、HuggingFace），能在数据写入时自动向量化

Weaviate 的设计哲学是”一站式 AI 数据基础平台”，适合需要密集使用混合搜索和多模态数据的场景。其 GraphQL 查询方式和模块化 Embedding 集成对于快速开发非常友好。

1.4 Pinecone

Pinecone 是一个全托管的云向量数据库服务，不开源，不可私有化部署。

核心特性：

  • 零运维负担，创建 Index 后即可使用
  • 自动索引优化和扩缩容
  • 集成主流 Embedding 提供商和 LLM 框架
  • 支持 Serverless 计费模式

Pinecone 适合希望完全免运维、愿意使用云托管服务的团队。缺点在于数据无法本地化部署（合规性受限），且大规模使用成本可能显著高于自建方案。

2. 传统数据库的向量扩展

2.1 pgvector

pgvector 是 PostgreSQL 的向量检索扩展插件，让你可以在现有的 PostgreSQL 中直接存储和检索向量，无需引入新的数据库。

核心特性：

  • 在 PostgreSQL 中新增 vector 数据类型和 ivfflat、hnsw 索引
  • 支持余弦相似度、欧氏距离、内积三种距离度量
  • 向量检索可以与 SQL 的 WHERE、JOIN、ORDER BY 等标准语法无缝结合
  • 利用 PostgreSQL 的事务、备份、权限管理等成熟基础设施

pgvector 的优势与局限：

优势 局限
零额外部署成本（已有 PG 即可） 索引构建速度不如专用向量数据库
SQL + 向量检索天然融合 十亿级规模不如 Milvus 等专业方案
事务保证（ACID） HNSW 索引内存占用较高
运维成熟（备份、监控、HA） 缺少多向量字段、混合搜索等高级功能

pgvector 对于已经在使用 PostgreSQL 且向量检索规模在千万级以下的 Java 后端项目，是性价比最高的选择。

2.2 Elasticsearch Vector

Elasticsearch 从 8.0 版本开始原生支持向量检索（dense_vector 字段类型和 knn 查询）。

核心特性：

  • 在 Elasticsearch 中存储 dense_vector 字段，支持 HNSW 索引
  • 向量检索 + ES 原生查询（全文检索、聚合、过滤）可以在一条查询中融合
  • ES 的分布式架构提供天然的水平扩展能力
  • 将向量检索、全文检索、日志分析、指标聚合统一到一个平台上

Elasticsearch 向量检索适合已经使用 ELK 技术栈、需要在同一个平台上同时做传统搜索和语义搜索的团队。但 ES 的向量检索性能不如专用向量数据库，且内存开销较大。

2.3 Redis Vector Search

Redis 从 Redis Stack（Redis 7.2+）开始提供向量检索能力，通过 RediSearch 模块实现。

核心特性：

  • 使用 FLAT（暴力）或 HNSW 索引
  • 低延迟（Redis 是内存数据库，检索延迟极低）
  • 适合实时性要求极高的场景（如推荐系统的在线检索召回）
  • 与 Redis 的 Hash、JSON 等数据结构结合使用

Redis Vector Search 适合对延迟极其敏感的场景（毫秒级甚至亚毫秒级），但由于所有数据在内存中，不适合海量低成本存储。

3. 向量数据库选型决策框架

Java 后端项目的推荐方向：

场景 推荐方案 理由
快速原型 / 小项目 pgvector 复用现有 PG，零额外运维
中型 RAG 应用（< 1000 万） Qdrant 或 Milvus Lite 部署简单，性能好
大型生产系统（> 1000 万） Milvus（分布式） 存算分离、水平扩展
已有 ES 技术栈 Elasticsearch Vector 技术栈统一，减少维护成本
不可私有化部署 Pinecone 免运维，按需付费

七、向量数据库的工程实践要点

1. 索引构建的性能调优

向量索引的构建是一个计算密集型任务，涉及大量的距离计算和图结构构建。对于大规模数据入库（百万级以上），索引构建可能成为瓶颈。

批量入库的建议策略：

  • 分批写入：每批 1000-5000 条向量，批次之间给索引构建留出时间
  • 先写入后建索引：大批量写入时，先关闭索引构建，等数据全部写入后再触发全量索引构建（大部分向量数据库支持此模式）
  • 索引参数的分级设置：初始入库时降低 efConstruction 等参数以加快构建速度；入库完成后在线调整参数提高检索精度

2. 数据一致性考量

向量数据库在数据一致性模型上各有不同：

  • pgvector：享有 PostgreSQL 的 ACID 事务保证，向量和元数据写入是事务性的
  • Milvus：默认是最终一致性，flush 后数据可查
  • Qdrant：写入后立即可查（单节点模式）
  • Elasticsearch Vector：近实时（默认每秒 refresh）

对于需要强一致性的场景（如文档删除后必须立即在检索结果中不可见），pgvector 是唯一保证 ACID 的选择。

3. 监控与可观测性

向量数据库的核心监控指标：

指标 含义 关注阈值
检索延迟（P99） 99% 的检索请求延迟 < 50ms 优秀，< 200ms 可接受
检索 QPS 每秒检索请求数 根据业务需求设定
索引内存占用 索引结构占用的内存 不超过可用内存的 70%
召回率 检索结果与暴力全扫描的重叠率 > 95%
Collection 向量数 当前存储的向量总数 监控增长趋势，预判扩容时机
