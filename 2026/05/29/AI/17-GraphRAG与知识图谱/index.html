GraphRAG与知识图谱

发表于 2026-05-29 更新于 2026-06-06 分类于 AI

GraphRAG与知识图谱

本文主要介绍

传统 RAG 的局限性

  • 碎片化检索：Vector Search 只能找到语义相近的片段，丢失全局结构和实体关系
  • 缺乏上下文关联：多个 Chunk 之间的关系链被切断
  • 复杂问答场景的短板：总结性问题、多跳推理、实体关系查询
  • 传统 RAG 在长文档、多文档场景下的不足

GraphRAG 的核心思想

  • 从向量检索到图谱检索的演进逻辑
  • Microsoft GraphRAG 的核心流程：实体提取 → 关系构建 → 社区检测 → 社区摘要 → 多级检索
  • 图结构在 LLM 应用中的四种使用方式：Graph as Index、Graph as Context、Graph as Memory、Graph as Tool
  • GraphRAG 如何处理全局性问题（如”这个数据集讲了什么主题？”）

知识图谱基础

  • 知识图谱的基本概念：实体、关系、属性、三元组
  • 知识图谱的构建方式：规则抽取、模型抽取（LLM抽取）、混合方式
  • 图数据库简介：Neo4j、NebulaGraph、Amazon Neptune、JanusGraph
  • 图谱 Schema 设计原则

GraphRAG 的工程实现

  • 文档 → 实体关系抽取 → 图谱存储 → 社区检测 → 摘要生成 → 检索图谱 的完整 pipeline
  • 实体与关系的 LLM 抽取（Prompt 设计、批量处理、去重合并）
  • 图社区检测算法（Leiden 算法）
  • 检索方式：Local Search（局部搜索）vs Global Search（全局搜索）

GraphRAG vs 传统 RAG vs Hybrid Search

  • 三者的适用场景对比矩阵
  • GraphRAG 的优势：总结性问题、多跳推理、实体关系查询
  • GraphRAG 的代价：构建成本高、Token 消耗大、延迟增加
  • 混合检索策略：Vector Search + Graph Search + Full-Text Search 的融合

Java 后端实践要点

  • Spring AI Neo4j Vector Store 的使用
  • LangChain4j + Neo4j 的集成方式
  • 知识图谱构建与增量更新的工程挑战
  • GraphRAG 在 Java 后端中的可行性：直接实现 vs 调用 Python GraphRAG 服务
  • 知识图谱构建与查询的性能优化（批量入库、索引优化、缓存热数据）
