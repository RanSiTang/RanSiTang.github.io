一、传统 RAG 的局限与 GraphRAG 的诞生

1. 问题的提出：当向量检索不够用

在第六章中我们介绍了 RAG 的核心思路：将文档向量化存入向量数据库，检索时找到语义最相似的 Chunk 注入 Prompt。这套方案在大多数”事实查询”场景下表现出色，但当面对以下三类问题时，向量检索的短板就会暴露：

第一类：总结性问题（Summarization Queries）

第二类：多跳推理问题（Multi-hop Reasoning）

第三类：实体关系查询

2. GraphRAG 的核心思想

GraphRAG 是 Microsoft 在 2024 年提出的一种新型 RAG 范式。它的核心思路可以用一句话概括：不要让 LLM 在碎片化的 Chunk 中”大海捞针”，而是先帮 LLM 把知识组织成有结构的图谱，让 LLM 在图谱中”按图索骥”。

GraphRAG 解决传统 RAG 局限的方式：

传统 RAG 局限 GraphRAG 如何解决
碎片化检索 知识图谱将碎片化的 Chunk 组织成实体-关系网络，检索时可以利用图结构导航
上下文断裂 图谱中的边（关系）显式表示了实体之间的逻辑关联，不会因 Chunk 切分而丢失
无全局感知 社区检测 + 社区摘要提供了从宏观到微观的多级知识视图
关系推理困难 图遍历可以在关系链上做多跳导航，找到”间接相关”的实体

二、知识图谱基础

1. 什么是知识图谱

知识图谱（Knowledge Graph）是一种用图结构来组织和表示知识的方式。它用节点（Node）表示实体，用边（Edge/Relationship）表示实体之间的关系，用属性（Property）描述实体和关系的特征。

3. GraphRAG 的代价与取舍

GraphRAG 不是在每个维度上都优于传统 RAG。它的核心代价是：

  • 构建成本高：对每个 Chunk 做 LLM 实体抽取，Token 消耗是文档本身的数倍。一份 1 万字的文档，构建 GraphRAG 索引可能需要消耗 5~10 万 Token（仅抽取阶段）
  • 更新复杂：新增文档需要增量抽取实体、做实体融合、更新社区结构——不是一个简单的”插入向量”操作
  • 细节损失：图是结构化的抽象，原始文本中的细微语义在经过”实体-关系”抽取后会丢失
  • 延迟层数多：一次完整的 GraphRAG 查询需要经过实体提取 → 图遍历 → 社区匹配 → Prompt 组装等步骤

3. GraphRAG 的 Java 实现 vs Python 桥接

GraphRAG 最重的部分（实体抽取、社区检测）在 Microsoft 的实现中是 Python 的。Java 后端有两种集成策略：

策略 做法 优势 劣势
Java 直接实现 用 Java 实现完整的 GraphRAG Pipeline（实体抽取、图构建、社区检测、检索） 统一技术栈、无跨语言开销 缺少成熟的图算法库（Leiden 需要用 Neo4j GDS 替代）、开发工作量大
Python Sidecar Java 做主流程编排，实体抽取和社区检测委托给 Python 服务 复用 Microsoft GraphRAG 的成熟实现、开发快 跨语言调用有延迟和运维复杂度
混合方案 离线索引阶段用 Python GraphRAG，在线查询阶段用 Java + Neo4j 各取所长、生产可用 两套系统需要数据对接
