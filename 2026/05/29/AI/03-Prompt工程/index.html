Prompt工程

发表于 2026-05-29 更新于 2026-06-07 分类于 AI

一、Prompt 的基本概念与结构

1. 什么是 Prompt

Prompt（提示词）是人与大语言模型之间的通信协议。如果 LLM 是一个能力强大但需要精确指令的执行者，那么 Prompt 就是你向它下达指令的语言。同样的模型，Prompt 写得好与不好，输出质量的差距可能大到像是两个完全不同的模型。

从工程角度来看，Prompt 的实质是：用自然语言描述业务需求，由 LLM 执行并返回结果。这与传统编程有本质区别——传统编程中，你用 Java 精确地告诉计算机每一步做什么；Prompt 编程中，你描述目标、设定约束、提供示例，模型自己决定如何达成目标。

Prompt 的组成从简单到复杂可以分为三个层次：

  • 基础层：一条简单的自然语言指令，如”将以下文本翻译成英文”
  • 结构层：包含角色设定、任务描述、输出格式约束的结构化 Prompt
  • 增强层：在结构层的基础上，加入 Few-shot 示例、思维链引导、外部知识上下文（RAG 检索结果）等增强要素

Prompt 工程（Prompt Engineering）就是系统化地设计、测试和优化 Prompt，以达到稳定、高质量输出的实践。它不是一次性的”写一段话”，而是需要版本管理、A/B 测试和持续迭代的工程活动。

二、Prompt 的组成要素

每一个有效的 Prompt 要素都有明确的职责。理解每个要素的作用，才能在设计 Prompt 时做出正确的取舍——加什么、不加什么，每个要素占多少 Token。

1. 角色设定（Role / Persona）

角色设定告诉模型”你是谁”，它决定了模型的回答视角和知识领域。例如：

  • “你是一名资深的 Java 后端工程师，精通 Spring Boot 和微服务架构”
  • “你是一位严谨的代码审查者，你的职责是发现代码中的潜在 Bug 和安全漏洞”

角色设定的效果机制：LLM 在训练过程中学习了大量不同角色的语料。指定角色相当于把模型的生成空间约束到该角色对应的语料分布区域，使其输出更符合该领域的表达习惯和知识深度。

角色设定的实践要点：

  • 越具体越好：”你是一个工程师”远不如”你是一个有 10 年经验的 Java 后端工程师，专注于高并发系统设计”有效
  • 与任务匹配：角色设定要和实际任务一致。让”安全审查者”去写代码，和让”后端开发”去做安全审查，效果都会打折扣
  • 不要过度依赖：角色设定有助于引导输出风格，但不能替代清晰的指令和约束。它只是辅助，不是核心

2. 任务描述（Task）

任务描述是 Prompt 的核心——明确告诉模型要做什么。一个好的任务描述有三个特征：

  • 明确（Explicit）：不说”分析这段代码”，而是”分析这段代码中可能导致死锁的并发问题”
  • 可执行（Actionable）：模型能基于给定的信息完成任务。如果任务需要模型没有的数据，要么提供数据，要么告诉它如何获取
  • 可验证（Verifiable）：任务的完成标准是清晰的。理想情况下，输出是否满足要求应该是可自动判断的

不好的任务描述 vs 好的任务描述：

不好 好
“优化这段代码” “优化以下 Java 方法的执行效率，重点关注循环中的数据库查询和对象创建”
“写个单元测试” “为以下 Service 方法编写 JUnit 5 单元测试，使用 Mockito Mock 所有外部依赖，覆盖正常路径、空参数和异常三种情况”
“分析日志” “从以下应用日志中找出所有 ERROR 级别日志，按发生频率排序，并给出每个错误的原因分析和修复建议”

3. 上下文信息（Context）

上下文是任务相关的背景知识和数据。模型无法访问外部信息，你必须显式地告诉它所有它需要知道的东西。

上下文的类型：

  • 领域知识：术语定义、业务规则、行业惯例——模型训练数据中可能有也可能没有的信息
  • 当前状态：用户的身份、系统的当前状态、历史对话摘要——这些模型绝不可能知道
  • 参考数据：文档片段（RAG 检索结果）、数据库查询结果、API 返回数据——模型的”外部记忆”

上下文的设计原则：

  • 先给定义，后给任务：模型对靠前的信息更关注。重要的定义和规则放在 Prompt 的靠前位置
  • 压缩而非堆砌：上下文不是越多越好。一段 500 Token 的精炼摘要，效果往往优于 5000 Token 的原始文档。LLM 在长文本的”中间部分”提取信息能力相对较弱（Lost in the Middle 现象）
  • 标注上下文边界：用 XML 标签或 Markdown 分隔符明确标记上下文的起止，如 <context>...</context>，帮助模型区分”这是参考资料”和”这是指令”

4. 约束条件（Constraints）

约束条件定义了输出的边界和规则——不是告诉模型做什么，而是告诉它不要做什么或在什么范围内做。约束的重要性不亚于任务描述本身，因为 LLM 在缺乏约束时可能产生意外但有破坏性的输出。

常见的约束类型：

  • 内容约束：”不要对你不确定的事实进行推断，如果信息不足请直接说明”
  • 格式约束：”输出必须是合法的 JSON，不要包含 Markdown 代码块标记”
  • 长度约束：”回答不超过 200 字，只需给出结论不需要解释过程”
  • 安全约束：”不要输出任何可执行的代码，不要提供可能被用于攻击的信息”
  • 语言约束：”始终使用中文回答，除非用户使用英文提问”

约束的设计要点：如果说任务描述是”开车去北京”，约束就是”限速 120、不要走国道、中途不要加油”。好的约束让模型在正确轨道上运行，过多的约束则会让模型束手束脚。

5. 输出格式（Output Format）

输出格式是约束条件中最重要的子类型。明确的格式要求能将非确定性的自然语言输出转化为可解析的结构化数据，这是 AI 模型与后端业务系统对接的关键。

6. Few-shot 示例（Examples）

Few-shot 示例是指在 Prompt 中提供一至多个输入-输出的范例，让模型理解期望的行为模式。

示例的三大作用：

  • 澄清格式：当输出格式较复杂时，一个例子比 500 字的格式描述更直观
  • 展示标准：示例展现了”什么是好的输出”的标准，比文字描述”输出质量要高”有效得多
  • 处理边界情况：通过精心挑选的边角示例，教会模型如何处理模糊输入和特殊情况

示例设计的最佳实践：

  • 从简单到复杂：如果给多个示例，难度应递增。第一个示例展示基础模式，后续示例逐步引入边界情况
  • 标注示例的意图：用注释说明”这个示例展示了 XX 情况”，帮助模型理解示例的用意
  • 示例要真实：用真实业务数据构造示例，而不是凭空编造。模型对虚构的示例适应性较差
  • 不要过量：2-3 个精心挑选的示例通常足够。5 个以上的示例收益递减，而且大量消耗 Token

三、Prompt 工程核心方法论

Prompt 工程不是玄学，有一套系统的方法可以让 Prompt 从”碰运气”变成”可预测”。以下是四种核心方法论，按复杂度从低到高排列。

1. Zero-shot：直接下指令，不给示例

Zero-shot（零样本）是最简单的 Prompt 方式——直接告诉模型做什么，不给任何示例。

Zero-shot 适用于：

  • 任务足够简单明确，模型已有充分的理解
  • 输出格式简单（如分类、二元判断）
  • Token 预算紧张的场景

Zero-shot 的局限：对于复杂任务（特定格式输出、多步骤推理、领域专有术语理解），纯 Zero-shot 的成功率较低。这时需要升级到 Few-shot。

2. Few-shot：给出范例，让模型举一反三

Few-shot（少样本）通过在 Prompt 中提供示例来引导模型的行为。LLM 具有强大的 In-Context Learning（上下文学习）能力——即不需要更新模型参数，仅通过在 Prompt 中展示几个例子，模型就能学会并复制这种模式。

Few-shot 的关键技巧：

  • 示例的多样性：示例应覆盖不同类型的输入，这样才能教会模型处理变化
  • 关键模式的重复：如果想让模型学会某个输出模式（如输出 JSON Schema 的某字段），在多个示例中反复展示该模式
  • 负面示例的价值：展示”什么是错的”有时比展示”什么是对的”更有效。例如：展示一个错误的 JSON 结构并标注”不要这样”

3. Chain of Thought（CoT）：让模型展示推理过程

Chain of Thought（思维链，简称 CoT）是一种引导模型逐步推理而不是直接给出答案的技术。它的核心思想来自对人类解决问题方式的观察——面对复杂问题时，大多数人不直接得出结论，而是通过一系列推理步骤逐步逼近答案。

CoT 最核心的技巧是在 Prompt 中加上一句：“Let’s think step by step”（让我们一步一步思考）。这看似简单的一句话，能将复杂推理任务的准确率提高 20%-50%。

CoT 的适用场景：

  • 数学计算和逻辑推理
  • 多条件判断（需综合多条规则作出决策）
  • 代码分析（Bug 诊断、安全审查）
  • 需要展示可解释性的场景（让用户看到 AI 为什么得出这个结论）

CoT 不适用的情况：

  • 简单的分类任务（直接判断更省 Token）
  • 创意生成（思维链会限制创造力）
  • 只需要结论不需要过程的场景

4. ReAct：推理与行动交替

ReAct（Reasoning + Acting）是 CoT 的升级版——模型不仅推理，还在推理过程中穿插调用外部工具。每次工具调用都是一次”行动-观察”循环：模型推理出需要什么信息，调用工具获取信息，观察结果后继续推理。

ReAct 是 Agent 智能体的核心推理范式（将在后续章节详述）。在 Prompt 工程层面，理解 ReAct 的关键是：对需要多步信息和多工具协作的复杂任务，设计 Prompt 时要预留”工具调用占位符”，让模型知道何时、如何请求外部帮助。

5. 方法论选型指南

实际项目中，这些方法通常是组合使用的：一个 Prompt 可能同时包含 Few-shot 示例、CoT 引导和 ReAct 工具调用。关键是根据任务复杂度选择恰当的层级，而非盲目追求高级方法。

四、Prompt 安全：注入攻击与防护

1. Prompt Injection 的原理

Prompt Injection（提示注入）是目前 AI 应用面临的最重要且独特的安全威胁。它的攻击原理是：通过在用户输入中嵌入恶意的指令文本，试图覆盖或绕过 System Prompt 中设定的行为约束。

一个经典的 Prompt Injection 示例：

如果模型没有足够的防护能力，它可能会直接输出 System Prompt 的内容，甚至按照攻击者的新指令行动。

Prompt Injection 的分类：

  • 直接注入（Direct Injection）：攻击者在用户输入中直接写入越狱指令，试图覆盖 System Prompt
  • 间接注入（Indirect Injection）：攻击者将恶意指令隐藏在 RAG 检索的文档中。当模型检索到该文档并作为上下文使用时，隐藏的指令就会被执行
  • 多模态注入：在图片、PDF 等非纯文本载体中嵌入隐藏指令

2. 防护策略

输入层防护：

  • 输入安全检查：在构造 Prompt 之前，对用户输入做安全扫描。检测常见的注入模式（如 “ignore previous instructions”、”you are now DAN” 等已知的越狱提示）
  • 输入长度限制：过长的用户输入更可能包含攻击内容，适当限制单次输入长度
  • 敏感词过滤：过滤掉用户输入中的 System Prompt 关键词（如 “system”、”instruction”、”guideline”）

Prompt 层防护：

  • Prompt 沙箱（Prompt Sandboxing）：将用户输入放在明确的标记区域内，清晰区分”系统指令”和”用户数据”。例如：
  • 重复强调约束：在 Prompt 的末尾再次强调核心安全规则，因为模型对 Prompt 末尾的信息关注度更高
  • 最小权限原则：Prompt 中不给模型超出任务需要的权限描述。不要写”你能访问所有系统功能”，只写”你能查询产品信息和订单状态”

输出层防护：

  • 输出内容审核：模型生成的回答在返回给用户之前，经过一道内容安全审核——检测是否泄露了内部信息、是否执行了注入指令
  • 结构化输出校验：如果期望 JSON 输出，校验其 Schema 是否合规。注入攻击常导致输出偏离预期格式，可被检测到

工程层防护：

  • 全链路日志：记录每次 LLM 调用的完整 Prompt 和输出，便于事后审计和分析攻击模式
  • 异常检测：监控输出内容的异常模式（如模型突然输出了 System Prompt 片段），触发告警

3. Prompt 模板的存储策略

Prompt 模板应该存在哪里？这取决于模板的变更频率和团队协作方式。

存储位置 适用场景 优点 缺点
classpath 文件（resources/prompts/） 模板稳定，随代码发布 版本管理（Git）、部署简单、修改需发版 变更需要重新部署
数据库 模板需频繁调整（如运营团队自行优化） 热更新、后台管理、A/B 测试灵活 缺少 Git 历史追溯、需要管理界面
配置中心（Apollo/Nacos） 模板需热更新 + 版本管理 结合了两者的优点：热更新 + 版本历史 + 灰度发布 依赖配置中心基础设施

推荐的混合策略：

  • 核心 Prompt 放在 classpath：如安全规则、角色设定等基础模板，通过 Git 管理，经过 Code Review
  • 业务 Prompt 放在数据库或配置中心：如营销话术、产品介绍等需频繁优化的模板，支持运营团队自助调整

4. Prompt 版本管理与 A/B 测试

Prompt 应该像代码一样有版本号、变更历史和回滚能力。

版本管理策略：

  • 文件命名规范：order-query-v1.st、order-query-v2.st，或在 Git 中通过 commit hash 追溯
  • 数据库版本表：记录每个 Prompt 模板的版本号、变更内容、变更人、变更时间和上线时间
  • 版本锁定：在生产环境中，明确指定 Prompt 模板的版本号（而非默认使用”最新版”），避免意外变更

A/B 测试架构：

A/B 测试的关键指标：

  • 任务成功率：输出是否能被正确解析和使用（如 JSON 解析成功率）
  • 用户满意度：点赞/踩、重试率、回复后追问率
  • Token 效率：平均每次调用的 Token 消耗
  • 延迟：P50 和 P99 响应时间

回滚策略：当 B 版本指标显著差于 A 版本时，将 B 版本的流量比例设为 0，全部流量切回 A 版本。数据库中的版本状态标记为 DEPRECATED，保留数据供事后分析。
