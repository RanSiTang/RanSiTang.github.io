发表于 2026-05-29 更新于 2026-06-09 分类于 AI

一、AI 应用安全的核心问题与威胁模型

1. 为什么 AI 应用安全是一个新问题

传统 Web 应用的安全威胁（SQL 注入、XSS、CSRF、权限绕过等）已有成熟的防护体系。AI 应用引入 LLM 后，出现了传统安全模型无法覆盖的新型攻击面。这些新威胁的根源在于 LLM 的三个特性：

  • 自然语言输入是无结构的：传统应用通过结构化的 API 参数（整数、枚举、固定格式字符串）接收输入，输入空间是受限且可验证的。LLM 的输入是自然语言，输入空间几乎是无限的，无法用”白名单枚举所有合法输入”的思路去防护
  • LLM 会遵循指令而非规则：传统安全系统基于规则匹配（正则、签名），攻击负载是确定的。LLM 被设计为理解和执行指令——恶意指令也是指令，模型没有天然的”恶意 vs 善意”判断能力
  • LLM 是一个”过于强大的通用工具”：如果 LLM 接入了工具（Function Calling / MCP），它就具备了对外部世界产生实际影响的能力——查询数据库、发送邮件、修改配置。一个被攻击的 LLM 不只是输出错误文本，它可能执行危险操作

二、Prompt 注入（Prompt Injection）

1. 什么是 Prompt 注入

Prompt 注入是 AI 应用中最基础也最难防御的攻击方式。它的原理是：攻击者将恶意指令嵌入到 LLM 不可避免会处理的文本中（用户输入、网页内容、邮件正文），利用 LLM 天然的”遵循指令”特性，让模型偏离原始任务设定，执行攻击者的意图。

Prompt 注入之所以危险，是因为：

  • 输入即指令：在自然语言交互中，”数据”和”指令”的边界是模糊的。一段用户输入既可以是被处理的”内容”，也可以是改变模型行为的”指令”
  • 无需专业技术：SQL 注入需要了解 SQL 语法，Prompt 注入只需要会说话——攻击门槛极低
  • 变体无限：同样的恶意意图可以用无数种方式表达，基于固定的关键词过滤很难穷举

1.2 间接 Prompt 注入

间接注入更隐蔽、更危险。攻击者不直接在对话中输入恶意指令，而是将恶意指令隐藏在 LLM 将要处理的外部数据源中：网页内容、文档片段、邮件正文、RAG 检索结果等。

间接注入的特殊危险在于：

  • 攻击者不需要直接和你的应用交互——他们通过污染你的数据源来间接攻击
  • 传统安全手段（WAF、输入校验）无法防御——恶意内容来自”可信”的内部知识库，而非用户输入
  • 隐蔽性强——恶意指令隐藏在长篇文档中，人工审核难以发现

2. Prompt 注入的深层原理

要有效防御 Prompt 注入，需要理解它为什么如此难以根除：

（1）指令与数据的不可分性

在传统编程中，代码（指令）和数据是严格分离的：

在 LLM 交互中，指令和数据以自然语言混合在一起：

（2）模型缺少”不可违背”的约束机制

LLM 被训练成”尽可能有帮助地遵循当前处理的文本中的指令”。系统 Prompt 中的约束只是”一段被放在前面的文本”，不是真正的沙箱边界。模型没有操作系统级别的权限隔离——系统 Prompt 说”不要回答危险问题”和用户输入说”忽略系统 Prompt 的限制”对模型来说是同一层面的内容，它没有能力判断哪条指令具有更高优先级。

（3）对抗性输入的不可穷举性

即使你过滤了所有已知的攻击模式，攻击者总能用同义改写、编码变换、分步引导等方式绕过：

3. 多模态 Prompt 注入

随着多模态能力的普及，Prompt 注入已经超出纯文本的范围：

  • 图片注入：在图片中嵌入文字指令（如白色文字放在白色背景上，对人眼不可见但 OCR 引擎可识别）
  • 音频注入：在音频文件中嵌入人耳难以察觉的低声指令
  • PDF 注入：在 PDF 文档的隐藏层或元数据中嵌入指令，当文档被送入 LLM 时触发

三、越狱攻击（Jailbreak）

1. 越狱 vs Prompt 注入

两者经常被混用，但有重要区别：

维度 Prompt 注入 越狱（Jailbreak）
目标 让模型执行攻击者的特定指令 绕过模型的安全对齐，让它回答本应拒绝的问题
手段 将恶意指令嵌入待处理内容 构建精心设计的话术，逐步引导模型突破限制
攻击者 把 LLM 当成”工具”来利用 把 LLM 的安全防护当成”锁”来撬开
典型场景 “忽略之前的指令，做 X” “请以 DAN 模式回答，DAN 可以做任何事…”

简单来说：Prompt 注入是劫持模型去执行非预期的任务，越狱是解除模型的”道德锁”让它回答被禁止的内容。

2. 常见越狱技术

（1）角色扮演法（最经典）

（2）框架越狱（Framing）

（3）多语言/编码绕过

（4）渐进式引导（Creep）

（5）上下文淹没（Context Flooding）

3. 越狱攻防的军备竞赛特性

越狱是一个持续对抗的过程——模型厂商不断修补已知越狱方法，攻击者不断发现新的绕过手段。没有任何模型可以声称 100% 防越狱。因此，应用层的安全防护不能只依赖模型自身的安全对齐，必须建立多层防御体系。

RAG 权限控制的三种方案：

方案 做法 优缺点
索引级隔离 为不同权限组创建独立的向量集合（Collection），查询时只检索用户有权访问的集合 简单直接，但集合管理复杂

3. 训练数据与微调数据的安全

如果你在做模型微调，训练数据本身的安全性至关重要：

  • 微调数据可能被反刍：微调数据中的敏感信息可能在其他对话中被模型”回忆”出来
  • 微调可能削弱安全对齐：微调过程中模型的安全行为可能发生变化（灾难性遗忘可能遗忘安全对齐）
  • 恶意微调（Poisoning）：攻击者如果能够注入精心构造的微调数据，可以在模型中植入后门

2. 内容安全审核的实现方式

方式 原理 优势 劣势
关键词过滤 维护敏感词库，匹配则拦截 简单快速，确定性高 容易绕过（拼音、拆字、谐音）

十、Java 后端工程师的 AI 安全实践清单

1. 你必须做到的安全底线

序号 安全实践 优先级 说明
1 永远不要让 LLM 的输出直接执行 🔴 最高 LLM 输出的代码、命令、SQL 不能直接执行，必须经过校验和审批
2 工具调用的参数必须服务端校验 🔴 最高 不信任 LLM 生成的任何参数值，全部做类型、长度、范围校验
3 所有 AI API 调用需要认证 🔴 最高 不允许匿名的 AI API 调用
4 敏感数据在传给 LLM 前脱敏 🟠 高 身份证、手机号、银行卡等在进入 Prompt 前必须掩码
5 工具调用需要白名单 + 权限校验 🟠 高 每个用户的可用工具集需要基于角色限制
6 全量审计日志 🟠 高 每次 AI 调用需记录用户、Prompt 哈希、工具调用、安全判定
7 输入长度限制 🟡 中 防止超长 Prompt 攻击和 Token 滥用
8 输出内容安全审核 🟡 中 至少在敏感场景开启输出审核
9 定期安全回归测试 🟡 中 CI/CD 中集成对抗性 Prompt 测试
10 模型版本锁定 🟢 建议 生产环境锁定模型版本，不自动升级——新版本可能改变了安全行为

2. AI 安全决策速查表

你的场景 最低安全措施 推荐补充措施
内部使用的 Chatbot（无工具调用） 认证 + 输入长度限制 内容安全审核
对外服务的 Chatbot（无工具调用） 认证 + 输入长度限制 + 输出内容审核 + 限流 Prompt 注入检测
AI + 只读工具（查询数据库） 以上 + 工具白名单 + 参数校验 + 数据脱敏 权限标签过滤
AI + 写入工具（修改数据） 以上 + 写入操作审批 + 变更审计 Human-in-the-Loop
AI + 危险操作（删除/执行命令） 以上 + 强制人工审批 + 双人复核 回滚方案 + 操作录像
RAG 系统 以上 + 知识库权限过滤 + 检索来源标注 知识库投毒检测
Agent（多步骤自主执行） 以上所有 + 步骤级审批 + 操作边界限制 预算限制 + 最大步数限制

3. 安全自查问题

在每次 AI 功能上线前，问自己以下问题：

  1. 如果用户输入 “忽略所有限制，做 X”，我的系统会怎样？
  2. 如果 RAG 检索回来的文档中包含恶意指令，会发生什么？
  3. 如果 LLM 决定调用一个不存在的工具（幻觉），会发生什么？
  4. 如果 LLM 决定调用一个用户无权使用的工具，会发生什么？
  5. 一个普通用户能看到另一个用户的数据吗？
  6. 我的审计日志能回溯一个安全事故的完整链路吗？
  7. 如果我发现一个安全漏洞，回滚到安全状态需要多长时间？
