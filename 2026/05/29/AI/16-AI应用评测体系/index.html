AI应用评测体系

本文主要介绍

AI应用评测的基本概念

  • 为什么 AI 应用需要专门的评测体系（非确定性输出、多维度质量要求、迭代频率高）
  • 评测在 AI 应用开发流程中的位置（开发 → 评测 → 上线 → 监控 → 反馈闭环）
  • 传统软件测试 vs AI 应用评测的区别

评测维度

  • 准确性：回答是否正确、事实是否无误
  • 相关性：回答是否切题、是否满足用户意图
  • 完整性：回答是否覆盖了所有要点
  • 一致性：相同/相似问题是否给出稳定一致的答案
  • 安全性：是否存在注入、越狱、敏感信息泄露
  • 延迟：首Token时间、端到端响应时间
  • 成本：Token消耗、模型调用次数

常用评测方法

  • 人工评测：标注规范、评分标准、A/B 盲评对比
  • 规则匹配评测：正则、关键词匹配、结构化输出的Schema校验
  • 相似度评测：BLEU、ROUGE、BERTScore、语义相似度
  • LLM-as-Judge：用模型评测模型，评判标准设计、偏差与局限性
  • 多种评测方法的优劣对比与适用场景

RAGAS — RAG 应用评测框架

  • RAGAS 的核心评测指标：Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision/Recall（上下文精确度/召回率）
  • RAGAS 的工作原理与使用方式
  • Java 后端中集成 RAGAS 评测的思路（Python Sidecar 或 HTTP API 桥接）

评测与可观测性平台

  • LangSmith、LangFuse、Phoenix（Arize）、Weights & Biases 等平台对比
  • Trace 级别的调用链观测
  • 评测结果数据化：Dashboard、报警、趋势分析
  • 开源方案 vs SaaS 方案的选型考量

评测数据集构建

  • 评测数据集的构建策略：种子问题 → LLM扩写 → 人工审核
  • 典型场景数据集的覆盖设计（正面case、边界case、对抗case）
  • 数据集的持续更新与版本管理

自动化评测 Pipeline 设计

  • 评测流程：数据加载 → 模型调用 → 结果采集 → 指标计算 → 报告生成
  • CI/CD 中集成评测（每次 Prompt 变更或模型切换触发评测）
  • Java 后端中评测管线的设计思路（评测任务调度 → 并行执行 → 结果汇总）
  • 评测阈值与上线门禁（Accuracy > 90%、Faithfulness > 95% 等）
