Embedding与向量化

发表于 2026-05-29 更新于 2026-06-09 分类于 AI

一、Embedding 的基本概念

1. 什么是 Embedding

Embedding（向量嵌入）是将文本、图像、音频等非结构化数据映射为高维空间中的稠密向量（Dense Vector）的技术。所谓”稠密”，是指向量的每一维都是有意义的浮点数，共同编码了原始数据的语义信息。

用形象的方式来理解：想象语义空间是一张巨大的地图，Embedding 就是给每个文本在这张地图上分配一个坐标。语义相近的文本——如”苹果很好吃”和”这个水果味道不错”——会被分配到地图上相邻的位置；而语义无关的文本——如”苹果很好吃”和”如何配置 Nginx”——则在地图上相距很远。

Embedding 之所以是 AI 应用开发的核心基础设施，原因在于：计算机无法直接”理解”自然语言的语义，但它可以高效地计算向量之间的距离。Embedding 将”语义理解”这个人类擅长但计算机不擅长的问题，转化为了”向量相似度计算”这个计算机极其擅长的问题。

从向量到距离再到相似度的转换，是 Embedding 工程实践的起点，后续所有关于检索、推荐和聚类的讨论都建立在这个基础之上。

2. 为什么需要 Embedding

在没有 Embedding 的时代，文本搜索主要依赖关键词匹配（如 MySQL 的 LIKE、Elasticsearch 的倒排索引）。关键词匹配存在几个难以克服的问题：

  • 同义词问题：用户搜”笔记本”，无法匹配文档中的”笔记本电脑”
  • 语义理解缺失：用户问”怎么退款”，无法匹配标题为”退货流程说明”的文档
  • 多语言断裂：中文搜索”苹果”，无法匹配英文文档中的”Apple”

Embedding 解决了这些问题：它不是匹配关键词，而是匹配语义。向量空间中距离相近的文本，语义就是相近的——不管用了什么词、什么语言。

Embedding 与 LLM 的关系：LLM 是”生成答案”的大脑，Embedding 是”找到相关信息”的眼睛。在 RAG 架构中，Embedding 负责从海量文档中检索出最相关的片段，LLM 负责基于这些片段生成答案。两者各司其职。

二、文本向量化的原理与实践

1. Embedding 模型的工作原理

Embedding 模型是一种专门训练的神经网络，输入一段文本，输出一个固定长度的浮点数数组（向量）。这个向量就是文本在高维语义空间中的坐标。

Embedding 模型的训练目标：让语义相近的文本产生距离相近的向量，语义无关的文本产生距离远的向量。这个目标通过在大量文本对数据（如搜索结果点击数据、问答对数据）上训练来实现。

主流 Embedding 模型：

模型 维度 特点
OpenAI text-embedding-3-small 512/1536 性价比高，支持动态维度缩减
OpenAI text-embedding-3-large 256/1024/3072 精度最高，支持维度缩减
Cohere Embed v3 1024 多语言效果好，支持分类和聚类场景
BGE-M3（BAAI） 1024 开源模型，中英文表现优秀，支持稀疏+稠密混合检索
Jina Embeddings v3 1024 支持任务特定 LoRA 适配器
GTE-Qwen2（阿里） 1536/4096 基于 Qwen 架构，中文表现最佳之一

Embedding 模型与 LLM 的区别：

Embedding 模型 LLM（Chat 模型）
输入 文本 文本（Messages）
输出 固定长度向量 可变长度文本
用途 语义搜索、聚类、分类 对话、生成、推理
调用方式 单次请求/响应 单次或流式
成本 极低（约为 LLM 调用的 1/100） 较高

Embedding 模型的核心价值是低成本地将语义转化为可计算的向量——它是 RAG 检索和语义搜索的基础引擎，而不是对话能力本身。

2. 向量维度

Embedding 向量的维度（Dimensions）是关键的工程决策参数。维度越高，语义编码越精细，但存储和计算成本也越高。

维度的直接影响：

因素 低维度（256-512） 高维度（1536-3072）
检索精度 略低，但足够应对大多数场景 最高
存储空间 每向量 1-2KB 每向量 6-12KB
检索速度 更快（索引小，计算量小） 更慢
适用场景 简单语义匹配、大规模语料 高精度检索、专业领域
  • 存储规模估算：以 1536 维为例，一个向量占用 1536 × 4 字节（float32）= 6KB。100 万条文档的向量索引约 6GB。如果切分后每个文档产生 3-5 个 Chunk，内存占用还会成倍增长
  • 动态维度缩减：OpenAI 的 text-embedding-3 系列支持在 API 参数中指定 dimensions，例如将 3072 维缩减为 256 维，精度损失约 5%-10%，但存储和计算成本降低 90% 以上
  • 精度损失的非线性特征：维度从 3072 降到 1536 时，精度损失通常不到 2%；但降到 256 以下时，损失开始明显加速。1024-1536 维是大多数场景的”甜点区”

选型建议：

  • 小型知识库（< 10 万条）：直接使用 1536 维，无需优化
  • 中型知识库（10 万-100 万条）：使用 1024 维，或使用 1536 维 + 向量索引压缩
  • 大型知识库（> 100 万条）：使用 256-768 维 + 多阶段检索（粗筛+精排）

3. Embedding 的批量调用

实际项目中，初始化知识库时需要向量化大量文档（可能上万条）。逐条调用 Embedding API 会导致请求数爆炸和延迟过高。正确的做法是使用批量 API：

OpenAI 批量 Embedding API：

单次请求可以传入最多 2048 条文本，大幅减少 HTTP 往返次数。

批量处理的分组策略：

  • 每次请求 100-500 条文本，既能利用批量 API 的并行能力，又能控制单次请求的延迟和失败重试成本
  • 分组之间使用并发线程池并行发起多个批量请求，进一步加速
  • 对于超大规模向量化任务（百万级以上），使用异步消息队列拆分任务，分批次并行消费，同时做好断点续传和失败重试

三、文本切分策略

在 RAG 应用中，文档通常很长（几页到几百页），而 Embedding 模型有输入长度限制（通常 512-8192 Token），且检索粒度越细结果越精准。因此需要将长文档切分为适当大小的文本块（Chunk）。

文本切分，或称 Chunking，是 RAG 系统设计和运维中最重要的预处理步骤之一。切分方式直接决定了检索召回的质量——切得太粗（Chunk 太大），检索精度下降且浪费 Token；切得太细（Chunk 太小），语义断裂、上下文丢失。

1. 文本切分的核心参数

Chunk Size（文本块大小）：

单个 Chunk 的 Token 数量上限。选择 Chunk Size 时需要在语义完整性和检索精度之间找到平衡：

  • 过大（> 2000 Token）：一个 Chunk 包含太多信息，检索时不够精准。而且大 Chunk 占用更多 Context Window 空间，减少了可注入的 Chunk 数量
  • 过小（< 100 Token）：语义不够完整。一个完整的段落可能被切成两半，导致下半部分失去上下文
  • 推荐范围：对于大多数中文文档，256-1024 Token 是合理的区间。问答类应用推荐 256-512 Token；长文档分析类推荐 512-1024 Token

Overlap（重叠量）：

相邻两个 Chunk 之间重叠的 Token 数量。Overlap 的目的是防止关键信息恰好出现在 Chunk 的边界处被切断。

  • 过小：边界信息丢失风险增加。一个句子被切在两段之间，两段都无法完整表达句意
  • 过大：存储冗余增加，且检索时可能返回内容高度重复的多个 Chunk
  • 推荐范围：通常设为 Chunk Size 的 10%-20%。如 Chunk Size = 512 Token，Overlap 设为 50-100 Token

Metadata（元数据）：

每个 Chunk 携带的结构化附加信息，用于检索时的过滤和排序。Metadata 本身不参与向量化，但存储在向量数据库中，用于检索后的过滤。常见的 Metadata 包括：

  • 来源文档 ID 和标题
  • Chunk 在原文中的页码或段落号
  • 文档类型（FAQ / 手册 / 合同 / 研报）
  • 创建时间和更新时间
  • 权限标签（哪些用户组可以检索到此 Chunk）

Metadata 的设计原则：在入库时多存，在检索时按需过滤。Metadata 不参与 Embedding 计算，存储成本极低，但能极大增强检索的灵活性和安全性。

2. 文本切分策略

文本切分有许多种策略，选择取决于文档类型、语义粒度和检索需求。以下是工程实践中最常用的几种策略及其适用条件。

2.1 基于字符的切分（Character Splitter）

最简单的切分方式，按字符数或 Token 数将文本切分为等长的块。

适用场景：格式规整的文档、初步验证时的快速切分、对检索精度要求不高的场景。

缺点：可能在句子中间切断，导致语义不完整。

2.2 基于分隔符的递归切分（Recursive Character Splitter）

这是目前最推荐、使用最广泛的切分策略。它会按照预设的分隔符优先级递归地切分文本：先尝试按段落（\n\n）切分，如果段落仍然太长，再按句子（。、！、？）切分，最后按从句或短语分隔符（，、；）切分。

适用场景：几乎所有文本类型，是 RAG 应用的默认首选策略。

优势：最大程度保留语义边界，尊重文档的自然结构。

2.3 基于语义的切分（Semantic Splitter）

使用 Embedding 模型计算相邻句子之间的语义相似度。当相似度显著下降时（即”话题切换”），在此处切分。语义切分的核心思路是让每个 Chunk 对应一个相对独立的话题或子主题。

适用场景：长文档、叙事性文本、会议记录等话题边界明显的文档。

代价：需要额外的 Embedding 计算，构建索引的时间是前两种方法的 2-3 倍。

2.4 基于文档结构的切分（Document Structure Splitter）

根据文档自身的结构标记（如 Markdown 的 # ## ### 标题层级、HTML 标签、PDF 的章节标记）进行切分。这种策略以文档的结构为切分边界，确保每个 Chunk 是文档中的一个逻辑单元。

适用场景：结构化良好的技术文档、API 文档、Markdown 文件。

优势：切分结果与文档的层级结构一致，检索结果更易于定位和追溯原文。

3. 多粒度索引策略

单一粒度的 Chunk 难以同时满足”精准检索”和”完整上下文”两个需求。多粒度索引是解决这个问题的工程方案：

  • 小 Chunk（128-256 Token）：用于向量检索，提供精准的匹配
  • 父文档（Parent Document）：小 Chunk 的来源文档或大段落，检索命中小 Chunk 后，将其父文档一起注入 LLM。也可以选择检索命中小 Chunk 后，向前后各扩展 N 个 Chunk（Context Window Expansion），作为上下文一起注入
  • 摘要 Chunk：对整个文档或章节生成摘要并向量化。全局性问题（”这份文档讲了什么？”）命中摘要 Chunk 而非碎片化的小 Chunk

这种策略在 LangChain 中称为 Parent Document Retriever，在 LlamaIndex 中称为 Hierarchical Node Parser。但策略本身与框架无关——核心思想是用不同粒度的索引处理不同粒度的查询。

四、向量相似度计算

向量化的目的不是为了”存储向量”，而是为了计算向量之间的相似度（距离）。相似度计算是语义搜索的数学基础。

1. 三种主要的相似度度量

余弦相似度（Cosine Similarity）：

衡量两个向量之间夹角的余弦值。这是 Embedding 场景中最常用的相似度度量。取值范围 [-1, 1]，1 表示方向完全一致（语义最接近），0 表示正交（无关），-1 表示方向完全相反。

余弦相似度之所以最常用于 Embedding 比较，是因为它忽略向量的长度（模），只关注方向。在语义空间中，向量的方向编码了语义内容，长度编码了置信度或信息量。只比较方向意味着不管文本长短，只要语义相近就能被检索到。

欧氏距离（Euclidean Distance）：

两个向量在多维空间中的直线距离。距离越小，语义越相近。欧氏距离受向量长度影响，推荐用于已做归一化处理的向量。

点积（Dot Product）：

两个向量对应维度乘积之和。当向量已归一化（模为 1）时，点积等价于余弦相似度。部分向量数据库（如 Pinecone、Qdrant）默认使用点积作为相似度度量以提升计算效率。

2. 相似度度量选型

度量方式 适用场景 说明
余弦相似度 通用语义搜索（默认首选） 不受文本长度影响，只关注语义方向
欧氏距离 向量已归一化、需要严格的距离度量 对向量长度敏感
点积 向量已归一化、追求极致性能 计算效率最高

实践建议：绝大多数场景下使用余弦相似度，它的语义区分能力最稳定。如果向量数据库只支持点积，确保 Embedding 向量已归一化即可等价替代。不要在这个环节过度优化——相似度度量之间的差异，远小于 Chunk 策略和 Embedding 模型选择带来的差异。

3. 批量 Embedding 的性能优化

初始化知识库时可能需要向量化数十万条文档，性能优化至关重要。以下方案按优化力度从小到大排列：

  • 使用批量 API：所有主流的 Embedding API 都支持一次请求传入多条文本（通常最多 2048 条）。批量 API 的吞吐远高于逐条调用，因为节省了大量 HTTP 往返开销
  • 并发请求：使用线程池并行发起多个批量请求，用 CompletableFuture 或 Reactor 管理并发和结果汇总，充分利用 API 提供商的并发容量
  • 本地缓存：对于重复出现的文本片段（如常见的文档模板、法律条文引用），缓存其向量结果，避免重复调用 API
  • 异步 + 断点续传：超大规模向量化任务（百万级）使用消息队列拆分任务，多消费者并行处理。每个任务记录处理进度，失败时从断点继续
