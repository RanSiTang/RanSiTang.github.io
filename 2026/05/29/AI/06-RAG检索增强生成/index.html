RAG检索增强生成

发表于 2026-05-29 更新于 2026-06-07 分类于 AI

一、RAG 的基本概念与定位

1. 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与文本生成结合的 AI 应用架构模式。它的核心思想是：在 LLM 生成答案之前，先从外部知识库中检索出与用户问题最相关的文档片段，然后将这些片段作为”参考资料”注入 Prompt，让 LLM 基于这些资料生成答案。

RAG 的本质是给 LLM 装上”搜索引擎”——LLM 不再仅依赖训练时记忆的知识，而是可以实时查询外部知识库，获取最新的、领域特定的、可信的信息。

RAG 的运行流程可以用一句话概括：用户提问 → 向量化查询 → 检索相关文档 → 组装增强 Prompt → LLM 基于资料生成答案。

RAG 不是某个特定的库或产品，而是一套架构范式。只要是”检索外部知识 → 注入 Prompt → 交给 LLM 生成”的模式，无论用什么工具实现，都属于 RAG 的范畴。

2. RAG 解决的核心问题：LLM 的三大局限

RAG 之所以成为 AI 应用开发中最基础的架构模式之一，是因为它直接解决了 LLM 的三个固有局限：

局限一：知识截止日期（Knowledge Cutoff）

LLM 的知识来自训练数据，训练完成后其知识停留在训练截止日期。对于截止日期之后的信息（如新产品发布、政策变更、最新技术动态），LLM 完全无知。RAG 通过检索外部实时更新的知识库，让 LLM 可以回答关于最新信息的问题。

局限二：幻觉（Hallucination）

当 LLM 被问到它不知道的信息时，它倾向于”编造”听起来合理但实际错误的答案——这就是幻觉。幻觉之所以危险，不是因为它明显错误（那样用户会察觉），而是因为它编造的答案往往听起来非常可信。RAG 通过将答案限定在检索到的真实文档范围内（in-context learning），大幅降低了幻觉的发生概率——LLM 被要求”基于资料回答”，而不是”凭记忆回答”。

局限三：领域知识缺失

通用 LLM 的训练语料以互联网公开内容为主，对于企业内部的私有知识（代码仓库、设计文档、运维手册、业务规范）完全不了解。RAG 通过检索企业内部知识库，让 LLM 能够回答需要私有领域知识的问题。

LLM 局限 表现 RAG 如何解决
知识截止日期 不知道最新信息 检索实时更新的外部知识库
幻觉 编造看似合理但错误的答案 基于检索到的真实文档回答，限定答案边界
领域知识缺失 不了解企业内部信息 检索企业内部私有知识库

RAG 的附加价值：

  • 可溯源（Attribution）：RAG 生成的答案可以附带引用来源（”这个答案来自《API 网关运维手册》第 3 章”），用户可以自行验证
  • 知识热更新：修改外部知识库后，RAG 的回答立即生效——不需要重新训练模型
  • 权限控制：不同用户检索不同权限的知识库，可以实现知识隔离

3. RAG vs 模型微调：两种知识注入范式

RAG 和模型微调（Fine-tuning）是给 LLM 注入外部知识的两种主要方式，它们不是互斥的，而是互补的。

对比维度 RAG 模型微调
原理 检索外部知识注入 Prompt 修改模型参数，将知识”写入”模型权重
知识更新 即时生效（更新知识库即可） 需要重新训练，周期长
知识容量 几乎无限（受限于向量数据库容量） 有限（受限于模型容量）
精确事实 强 — 直接返回原文片段 弱 — 知识被”模糊化”到参数中
风格与模式学习 弱 — 不改变模型行为模式 强 — 能教会模型特定风格和推理模式
可解释性 高 — 可溯源到具体文档 低 — 参数级修改难以溯源
成本 中（检索 + 推理成本） 高（训练 GPU 成本）
延迟 略增（增加检索步骤） 无额外延迟
适用场景 知识密集型 QA、企业知识库 特定语气风格、专业推理模式

决策框架：

在绝大多数企业 AI 应用中，RAG 是首选的知识注入方式，微调是进阶优化手段。原因很简单：企业在绝大多数情况下面临的是”怎么让模型知道我的最新文档”，而不是”怎么让模型学会一个全新的推理范式”。

二、RAG 的核心流程

RAG 系统包含两个阶段的处理管线：离线索引阶段（Offline Ingestion Pipeline）和在线检索生成阶段（Online Retrieval & Generation Pipeline）。

1. 离线阶段：文档索引管线

离线阶段的任务是将原始文档（PDF、Word、Markdown、网页等）转化为可供向量检索的索引数据。这个过程通常只在文档入库时执行一次，或在文档更新时增量执行。

1.1 文档加载（Document Loading）

文档加载是将各种格式的原始文件读入系统，转换为统一的文本表示。不同的文件格式需要不同的加载器（Reader）。

常见文档格式及其加载方式：

格式 加载工具/库 注意事项
PDF Apache PDFBox, Apache Tika 需要处理扫描件（OCR）、表格、多栏布局
Word (.docx) Apache POI 需要处理标题层级、表格、页眉页脚
Markdown 直接读取文本 注意 Frontmatter 元数据的提取
HTML Jsoup 去除广告、导航栏等噪声，只提取正文
数据库 JDBC 将结构化数据转为文本描述
代码仓库 Git + 自定义解析 保留文件路径和包结构作为元数据

文档加载的工程实践：

  • 统一文档模型：定义统一的 Document 对象，包含 content（文本内容）和 metadata（元数据，如来源文件路径、页码、作者、修改时间）
  • 格式白名单：只加载支持的格式，对不支持的格式给出明确错误提示
  • 文件大小限制：设置单文件大小上限（如 50MB），避免过大的文件（如扫描件、图片 PDF）阻塞处理管线

1.2 文档解析（Document Parsing）

文档解析是将加载后的文本进一步清洗和结构化——提取有意义的文本内容，去除噪声。

解析的核心挑战不是”读文件”，而是”提取出真正有用的文本”。

  • PDF 解析的难点：多栏布局导致文本顺序错乱；表格数据的结构化提取；扫描件需要 OCR；页眉页脚和页码的去除；嵌入图片中文字的提取
  • HTML 解析的难点：导航栏、广告、推荐列表等非正文内容的去除（Boilerplate Removal）；<table>、<code> 等语义标签的正确处理
  • 代码文件的解析：保留包路径、类名、方法名等结构化信息作为元数据；Javadoc / 注释的提取

1.3 文本切分（Chunking）

文本切分已在 04-Embedding与向量化.md 中详细讨论。对于 RAG 来说，文本切分是决定检索质量的关键预处理步骤。

RAG 场景下文本切分的特殊考量：

  • Chunk Size 与 LLM Context Window 的关系：Chunk Size 不能只看检索精度，还要考虑返回的 Chunk 在 LLM Context Window 中的占比。如果每个 Chunk 太大（如 4096 Token），返回 5 个就占用了 20000+ Token，留给 LLM 生成回答的空间很少
  • 语义完整性优先于大小均匀：宁可大小不均匀，也要保证每个 Chunk 的语义完整。一个好的切分器会在”自然断点”（段落边界、章节边界）处切分，而不是机械地按字符数切分
  • Overlap 在 RAG 中的实际价值：Overlap 不仅防止”边界截断”，更重要的是保证当一个答案需要的信息恰好分布在两个 Chunk 之间时，至少有一个 Chunk 能完整包含该信息。这在技术文档中特别重要——一个配置步骤可能跨越两个段落，Overlap 确保有一个 Chunk 同时包含两个段落的完整内容

Chunk Size 的经验数据：

文档类型 推荐 Chunk Size 推荐 Overlap
FAQ / 短问答 128-256 Token 20-50 Token
技术文档 / 手册 512-1024 Token 50-100 Token
长文章 / 研究报告 1024-2048 Token 100-200 Token
法律法规 / 合同 512-1024 Token 100-200 Token（需要较高 Overlap 保证条款完整性）

1.4 向量化（Embedding）

向量化是将切分后的文本 Chunk 转换为向量的过程。详见 04-Embedding与向量化.md。

在 RAG 的离线索引阶段，向量化的工程要点是：

  • 批量处理：逐条调用 Embedding API 会导致大量 HTTP 往返。使用批量 API，每次发送 100-500 条文本
  • 失败重试与幂等性：Embedding API 调用可能因网络抖动、限流等原因失败。每条文本使用唯一 ID，支持重试而不产生重复向量
  • 成本估算：每条 Chunk 的 Embedding 成本约为 LLM 调用的 1/100 - 1/200，但大规模索引时总量不容忽视。100 万条 Chunk × $0.02/1M tokens × 500 tokens/Chunk ≈ $10 USD
  • 模型一致性：索引阶段和检索阶段必须使用相同的 Embedding 模型。索引用 OpenAI text-embedding-3-small 而检索用 BGE-M3，会导致语义空间不一致，检索效果极差

1.5 向量存储（Vector Store）

向量化后的数据写入向量数据库。详见 05-向量数据库.md。

在 RAG 的索引阶段，向量存储的要点是：

  • 元数据保留：每个 Chunk 必须保留充分的元数据——来源文档名、页码/章节、文档分类、创建时间、Chunk 编号。这些元数据在检索时用于过滤，在生成答案时用于溯源引用
  • 索引类型选择：RAG 场景推荐 HNSW 索引，检索速度快、召回率高
  • Collection 隔离策略：不同知识域的数据放入不同 Collection（如 product_docs、internal_wiki、troubleshooting），检索时可以按需指定 Collection

2. 在线阶段：检索与生成

在线阶段是 RAG 系统在运行时的执行流程——用户发起提问，系统完成检索和生成。

2.1 查询向量化

用户的自然语言问题需要转换为向量，才能与向量数据库中的文档向量进行相似度匹配。

直接向量化 vs 查询预处理：

最简单的做法是将用户原始问题直接向量化。但在实际应用中，用户的问题往往不适合直接做检索查询——可能包含口语化表达、指代词、信息冗余或不足。因此，在向量化之前对查询进行预处理（Query Rewrite、查询扩展）是提升 RAG 检索质量的关键优化，详见第三章。

查询向量化的工程要点：

  • 延迟敏感：查询向量化是同步操作，直接影响用户体验。使用低延迟的 Embedding 模型（如 text-embedding-3-small 通常在 50-200ms 内完成）
  • 缓存：对于高频查询问题（如常见 FAQ），缓存查询向量和检索结果，避免重复计算和检索
  • 用户问题长度：过长的用户问题可能包含噪声信息，影响向量化质量。在向量化前做轻量清洗——截断过长问题、去除礼貌用语等

2.2 相似性检索

查询向量传入向量数据库，执行 ANN 检索，返回 TopK 个语义最相似的文档 Chunk。

检索参数设计：

参数 推荐值 理由
TopK 3-10 太少可能遗漏关键信息，太多引入噪声
相似度阈值 0.65-0.75 过滤掉语义不相关的结果
标量过滤 按需 按分类、时间、权限等维度缩小检索范围

检索结果的后处理：

  • 去重：高度相似的 Chunk 可能返回多条，去重后保留最相关的一条
  • 排序调整：按相似度降序排列，将最相关的结果放在 Prompt 最前面（LLM 对 Prompt 中靠前的内容关注度更高）
  • 上下文拼接：如果返回的 Chunk 在原文中是连续的（如 Chunk 3 和 Chunk 4 来自同一文档且相邻），将它们拼接为更大的上下文片段

2.3 Prompt 组装

检索结果需要被组织成一个结构化的 Prompt，才能注入 LLM。这是 RAG 中连接”检索”和”生成”的桥梁，Prompt 的组装质量直接影响最终答案的质量。

增强 Prompt 的典型结构：

Prompt 组装的关键原则：

  • 明确信息来源：每个 Chunk 附带来源标签（文档名、章节），让 LLM 在回答中能够引用
  • 边界约束：显式告诉 LLM “如果资料中没有相关信息，请说明而不是编造”。这降低了 LLM 在检索结果不相关时强行作答的概率
  • 相关性排序：将相似度最高的 Chunk 放在最前面——LLM 对 Prompt 中位置靠前的内容赋予更高的注意力权重
  • 不要过度注入：Chunk 数量不是越多越好，注入过多文档会稀释最相关信息的注意力。通常在 3-10 个 Chunk 之间

2.4 LLM 生成

LLM 基于增强后的 Prompt 生成最终答案。这个阶段的核心考量是引用溯源和拒答机制。

引用溯源（Citation）：要求 LLM 在答案中引用具体的信息来源。可以在 Prompt 中明确要求：”请在回答的每个关键事实后标注来源，格式为 [来源: 文档名, 章节]”。

拒答机制（Refusal）：当检索结果与问题相关性很低时，LLM 应该拒绝回答而不是”硬编”。实现方式是设置相似度阈值，当最佳匹配的相似度低于阈值（如 < 0.65）时，在 Prompt 中注入”参考资料：无相关信息”，并引导 LLM 生成拒答回复。

三、RAG 的检索优化策略

基础的 RAG（用户提问 → 直接向量化 → 检索 → 生成）在很多情况下表现已经不错，但在面对模糊查询、复杂查询和需要精确匹配的场景时，检索召回率会显著下降。RAG 检索优化是提升 RAG 系统质量的核心投入方向。

1. Query Rewrite（查询重写）

用户在使用 RAG 系统时，经常提出不完整、口语化或模糊的问题。这些问题直接向量化后，与知识库中正式的技术文档之间的语义距离很大，导致检索效果不佳。

问题示例：

用户原始问题 问题 重写后的检索查询
“那个限流怎么配啊？” 指代不明，口语化 “API 网关限流配置方法”
“上次说的那个数据库连接池的问题” 依赖对话上下文 “数据库连接池配置问题和解决方案”
“挂了” 极其模糊 “服务挂掉的原因和恢复方法”（需结合上下文推断）

Query Rewrite 的实现方式：

  • LLM 重写：使用 LLM 本身（通常是轻量模型以降低成本）将用户问题改写为更适合检索的表达。在 Prompt 中要求 LLM 消除指代、补全上下文、使用专业术语替代口语表达
  • 多轮对话上下文融合：在多轮对话场景中，用户的后续问题可能依赖对话历史（如”那第二个呢？”）。Query Rewrite 需要融合历史对话上下文，生成独立完整的检索查询

Query Rewrite 的关键原则：重写后的查询不是为了给用户看，而是为了最大化检索召回率。重写时应该使用知识库中可能出现的术语和表达方式，而不是用户的原话。

2. Multi-Query Retrieval（多查询检索）

对于复杂问题，一个检索查询可能无法覆盖所有相关方面。Multi-Query Retrieval 的策略是：从不同角度生成多个查询，分别检索后合并去重。

例如，用户问”如何提高 Spring Boot 应用的启动速度？”，可以生成：

  • 查询 1：”Spring Boot 启动优化方法”（直接角度）
  • 查询 2：”延迟初始化、懒加载配置”（技术角度）
  • 查询 3：”Spring Boot 启动慢的常见原因”（诊断角度）

这种策略特别适合需要综合多方面信息的复杂问题，检索的多样性显著优于单查询。

3. Hybrid Search（混合搜索）

混合搜索在 05-向量数据库.md 中有详细介绍。在 RAG 场景中，混合搜索的典型实践是向量检索（语义） + BM25 检索（关键词），通过 RRF（Reciprocal Rank Fusion，倒数排名融合）合并结果。

在 RAG 中启用混合搜索的信号：

  • 知识库包含大量专有名词、产品型号、错误码（这些是关键词检索的优势领域）
  • 用户经常搜索精确的代码片段、配置项名称
  • 知识库中的文档使用了多样化的术语表达同一概念

混合搜索的融合参数调优：

场景 α（语义权重） 说明
通用知识 QA 0.7 偏重语义理解
技术文档搜索（含代码/配置） 0.5 均衡语义和精确匹配
错误码/产品型号搜索 0.3 偏重关键词精确匹配

4. Rerank（重排序）

向量检索的 TopK 结果是基于向量相似度排序的，但向量相似度并不总是等于实际相关性——两个文本向量相似度高，可能是因为它们在主题上相近，但不一定意味着它们在内容上直接回答了用户的问题。

Rerank 的原理：使用一个更精确但更慢的相关性模型（通常是 Cross-Encoder），对向量检索返回的 TopK 结果进行精确的相关性打分，然后按新分数重新排序。Cross-Encoder 同时接收查询和文档作为输入，通过注意力机制计算精确的相关性分数，比向量相似度准确得多，但计算成本也高得多（不能用于全量检索）。

Rerank 模型选择：

模型 特点
Cohere Rerank v3 目前效果最佳的商业 Rerank 模型
BGE-Reranker-v2（BAAI） 开源首选，中英文表现好
Cross-Encoder（Sentence-Transformers） 通用方案，可自定义训练

Rerank 的成本权衡：Rerank 的单位成本远高于向量检索（Cross-Encoder 对每条查询-文档对都需要完整推理）。典型策略是先用向量检索召回 50-100 条候选，再用 Rerank 精选 5-10 条。

四、高级 RAG 模式

1. Self-Query Retrieval（自查询检索）

Self-Query Retrieval 是让 LLM 从用户的自然语言问题中自动抽取出结构化过滤条件，然后将过滤条件应用于向量检索。

例如，用户问”2024 年发布的关于微服务的架构文档有哪些？”，Self-Query 能将问题拆解为：

  • 语义查询：”微服务架构”
  • 结构化过滤：year = 2024 AND category = '架构文档'

Self-Query 的核心价值是让用户用自然语言表达过滤意图，而不需要在 UI 上设计复杂的筛选条件表单。在企业知识库搜索场景中，用户经常需要表达”某个时间段内的””某类文档中的””某个人写的”等条件，Self-Query 让这些条件能自然融入对话。

2. Parent-Child Chunk（父子文本块）

Parent-Child Chunk 是一种解决”检索粒度”与”上下文完整性”矛盾的策略。

问题：小 Chunk 检索精准，但缺乏上下文；大 Chunk 上下文完整，但检索精度下降。

解决方案：将文档切分为两种粒度的 Chunk：

  • Child Chunk（子块）：小块（如 256 Token），用于向量检索。小块语义聚焦，检索精度高
  • Parent Chunk（父块）：大块（如 1024-2048 Token），包含子块的完整上下文。在检索时，先通过子块匹配到最相关的小块，然后返回包含该子块的完整父块

Parent-Child Chunk 在技术文档、学术论文等结构化强的文档类型中效果特别好——检索精准定位到相关段落（Child），但让 LLM 看到该段落所属完整章节的上下文（Parent）。

3. 多跳检索（Multi-hop Retrieval）

多跳检索（Multi-hop Retrieval）处理的是需要综合多个文档的信息才能回答的问题。

例如，用户问：”张三参与的那个使用 Spring Cloud Gateway 的项目，部署在哪个 K8s 集群上？”这个问题需要：

  • 跳 1：检索”张三 参与 Spring Cloud Gateway 项目”→ 得到项目名称”电商中台”
  • 跳 2：检索”电商中台 K8s 集群 部署”→ 得到集群名称”prod-cluster-east-1”

多跳检索需要 RAG 系统具备”检索→获取信息→生成新查询→再检索”的迭代能力。实现方式通常是 Agent 模式（后续 09-Agent 章节会详细讨论）：Agent 在执行过程中自主判断是否需要进一步检索，并生成后续的检索查询。

4. 上下文压缩（Context Compression）

当检索返回的文档数量多、长度大时，直接全部注入 LLM 会占用大量 Context Window。上下文压缩（Context Compression）的策略是：对检索返回的文档内容进行摘要压缩，只保留与用户问题相关的部分。

实现方式是让 LLM 逐条处理检索结果：”以下文档片段中，哪些内容与用户问题’{用户问题}’相关？请提取出相关的句子，丢弃不相关的部分。”经过压缩的上下文更精简，可以注入更多来源的文档，或者节省 Context Window 留给 LLM 的回答。

上下文压缩适用于检索源文档很长（如整篇论文、长报告）的场景，但需要注意压缩本身会增加延迟和 LLM 调用成本。

七、RAG 的工程化问题

RAG 系统从 Demo 走到生产环境，需要解决一系列工程化问题。这些问题在原型阶段通常被忽略，但在真实业务场景中每一个都可能导致系统不可用。

1. 索引更新与增量同步

知识库不是静态的——文档会新增、修改、删除。RAG 系统需要感知这些变化并同步更新向量索引。

全量重建（Full Re-index）：定期（如每周）清空 Collection 并重新处理所有文档。这是最简单可靠的方案，但成本高（重新 Embedding 所有文档）、周期长（大规模知识库可能需要数小时）。

增量同步（Incremental Sync）：只处理变化的文档。通过监听文档变更事件（文件系统监听、数据库 binlog、消息队列），触发增量索引更新。

增量同步的工程挑战：

  • Chunk ID 的稳定性：文档内容变更后，切分边界可能移动，导致所有 Chunk 的 ID 都发生变化。需要设计稳定的 Chunk ID 生成策略（如基于内容哈希），使未变化的 Chunk 能复用已有的向量
  • 删除的可靠性：删除文档意味着需要删除与之关联的所有 Chunk。需要在元数据中记录”文档 ID → 所有 Chunk ID”的映射关系

2. 数据一致性

向量数据库和关系数据库之间的数据一致性是 RAG 工程化的核心挑战。

常见的不一致场景：

  • 向量已写入但事务回滚：关系数据库中的文档写入事务提交失败，但向量已经写入向量数据库（大多数向量数据库不支持事务回滚）
  • 向量未写入但文档已可见：用户在界面上看到了新上传的文档，但向量索引还未完成，导致该文档暂时搜索不到
  • 删除不彻底：关系数据库中的文档已标记为删除，但向量数据库中的对应向量未被清理

推荐的工程方案：

  • 消息队列解耦：文档写入 PostgreSQL 后，发送一条消息通知索引服务异步处理。核心数据库操作不受向量索引速度的影响
  • 状态标记：为每个文档维护一个 index_status 字段（pending / indexing / indexed / failed），搜索时只检索状态为 indexed 的文档
  • 定期对账：定时对比关系数据库中的文档列表和向量数据库中的向量列表，发现差异后补录或清理

4. RAG 质量评估

RAG 系统的质量不能仅靠”感觉还行”来评判，需要建立量化评估体系。

核心评估指标：

指标 定义 测量方法
召回率（Recall） 相关文档被检索到的比例 标注测试集中”应检索到的文档”与实际检索结果的交集
精确率（Precision） 检索结果中真正相关的比例 人工标注每个检索结果是否相关
上下文相关性 检索结果与问题的语义相关性 可以用 LLM 自动评分（如 Ragas 的 Context Relevancy）
答案忠实度（Faithfulness） 答案是否基于检索结果而非编造 逐句对照答案与检索结果，检测有无幻觉（Ragas Faithfulness）
答案相关性 答案是否直接回答了问题 LLM 评估答案与问题的相关性（Ragas Answer Relevancy）

评估工具：

  • Ragas：开源的 RAG 评估框架，提供 Context Precision、Context Recall、Faithfulness、Answer Relevancy 等指标
  • LangSmith：LangChain 的商业评估平台
  • 自建评估集：准备 50-200 个真实用户问题及其期望答案，每次索引或检索策略变更后运行评估集，量化变更效果

持续优化循环：

RAG 质量评估不是为了”证明系统有多好”，而是为了找到系统的瓶颈在哪里——是检索没找到（召回问题），还是找到了但没用好（Prompt 问题），还是知识库本身的信息不足。定位瓶颈才能做有针对性的优化。
