一、为什么 Java 后端需要 AI 集成框架

1. 直接调用 LLM API 的痛点

在没有 AI 集成框架之前，Java 后端接入 LLM 的方式是直接调用各模型厂商的 2HTTP API——自己封装 HttpClient、手动拼接 Prompt、自己解析 SSE 流式响应、自己管理对话历史。这种方式在原型阶段可行，但进入生产环境后会暴露出一系列工程化问题。

直接调用的十大痛点：

# 痛点 说明
1 多模型适配 每家厂商 API 格式不同（OpenAI vs Anthropic vs 国产模型），切换模型 = 重写调用代码
2 对话历史管理 多轮对话需要手动维护消息列表，Context Window 溢出需要手动截断和摘要
3 流式响应处理 SSE 解析、断线重连、背压控制——全是样板代码
4 Prompt 模板化 字符串拼接 Prompt 难以维护、难以复用、难以测试
5 Function Calling 集成 需要手动解析模型返回的工具调用、手动执行 Java 方法、手动构造 Tool Result 回传
6 RAG Pipeline 向量化 → 存储 → 检索 → 注入 Prompt 是一条长链路，每一步都需要集成和调优
7 可观测性 Token 用量统计、延迟监控、 Prompt 审计——裸调 API 时这些都需要自建
8 异常处理与重试 网络超时、Rate Limit、模型过载——每种错误的处理策略不同
9 安全与权限 API Key 管理、用户身份传递、敏感信息过滤——这些安全需求散落在各调用点
10 多模态支持 文本 + 图片 + 音频——不同模态的数据编码和传输方式不同

2. AI 集成框架的核心价值

AI 集成框架的定位类似于 JDBC 之于数据库——它提供了一套统一的抽象，让 Java 开发者用熟悉的编程模型（Spring Bean、依赖注入、声明式注解）来构建 AI 应用，而不需要关心底层模型厂商的 API 差异。

AI 集成框架给 Java 后端带来的五个核心价值：

  1. 统一抽象，屏蔽差异：一套 API 调用所有模型，切换模型只需改配置，不改代码
  2. Spring 原生体验：@Autowired 注入 ChatClient，@Tool 声明工具，application.yml 配置模型参数——和写 Spring Boot 业务代码完全一样的体验
  3. 工程化能力开箱即用：流式输出、重试、熔断、可观测性、对话历史管理——这些基础设施不需要自建
  4. 降低 AI 能力的集成门槛：RAG、Function Calling、Agent——这些看似高深的 AI 能力，在框架中被封装为几十行 Java 代码即可实现
  5. 生产级可靠性：连接池管理、Rate Limit 处理、模型降级和 Fallback——框架替你处理了与模型交互的各类异常情况

二、主流 Java AI 集成框架概述

1. Spring AI

Spring AI 是 Spring 官方推出的 AI 集成框架，定位是 “将 AI 能力融入 Spring 生态”。它的设计哲学和 Spring Data、Spring Security 一脉相承——提供统一的抽象接口，各模型厂商通过实现这些接口来接入。

核心设计理念：

Spring AI 不是要重新发明 AI 的概念（如 Chain、Agent、Memory 这些概念在 Python AI 生态中已有成熟定义），而是用 Spring 开发者熟悉的编程模型来封装这些概念——ChatModel 作为一个 @Service 被注入、@Tool 声明工具方法、VectorStore 作为 Spring Data 风格的 Repository、application.yml 集中管理所有 AI 配置。

2. LangChain4j

LangChain4j 是 LangChain（Python AI 生态中最流行的框架之一）的 Java 移植版，但不是简单的语法翻译——它在保持 LangChain 核心设计理念（Chain、Agent、Memory、RAG Pipeline）的同时，深度适配了 Java 的类型系统、异常处理机制和 Spring 生态。

核心设计理念：

LangChain4j 的关键设计是 AiServices——这是一个声明式 AI 服务层，让你通过定义 Java 接口的方式来声明 AI 能力，框架自动处理 LLM 调用、工具选择、对话管理和输出格式化。

3. Semantic Kernel for Java

Semantic Kernel（SK）是微软推出的 AI 编排框架，最初为 .NET 设计，后来推出了 Java 版本。它的核心定位是**”AI 编排操作系统”**——不只是调用 LLM，而是将 LLM 作为”内核”，周围可以连接各种”插件”（Planner、Memory、Connector），形成完整的 AI 应用编排能力。

核心设计理念：

Semantic Kernel 把 LLM 视为操作系统的 Kernel，Prompt 和 Tool 是”插件”，Planner 负责将用户目标规划为执行计划。这与 Spring AI 和 LangChain4j 的设计视角不同——SK 更强调”AI 编排”而非”API 封装”。

Semantic Kernel for Java 的现状：

Semantic Kernel for Java 目前处于早期阶段（0.x 版本），社区规模和文档完整度不及 Spring AI 和 LangChain4j。对于深度使用 .NET 或需要跨语言统一 AI 编排方案的组织，SK 的跨语言一致性是一个优势；但对于纯 Java 技术栈的团队，Spring AI 或 LangChain4j 是更成熟的选择。

三、Spring AI 核心 API

1. ChatClient —— 对话客户端

1.1 概念

ChatClient 是 Spring AI 中使用频率最高的入口类。它是对底层 ChatModel 的高层封装，提供了建造者模式（Builder Pattern）的流式 API，将模型调用、Prompt 构建、工具注册、Advisor 拦截等能力统一为一个调用链。

为什么需要 ChatClient？ 直接使用 ChatModel 接口需要手动构造 Prompt 对象（含 List<Message> + ChatOptions），样板代码多。ChatClient 把这一切封装为链式调用：.prompt().user("...").call().content()，一行链式调用完成全部操作。

关键设计：ChatClient 是不可变的线程安全对象。每次调用 .prompt() 都会创建一个新的 Prompt 构建器，不会修改 ChatClient 本身。这意味着同一个 ChatClient 实例可以被多个线程安全共享。

1.2 获取 ChatClient

ChatClient 通过 ChatClient.Builder 构建。Builder 在 Spring 容器中是 prototype scope（每次注入都是新实例），确保每个构建者之间互不干扰。

获取方式 适用场景 说明
注入 Builder，构造器中构建 全局配置一次设定，所有调用复用 推荐方式。默认 system prompt、temperature、tools、advisors 集中配置
每次调用时从 Builder 构建 需要动态切换参数的场景 每次 .build() 返回新的 ChatClient，更灵活但有额外开销

1.3 ChatClient.Builder 核心方法

Builder 的方法分为两类：设置默认值（对后续所有 .prompt() 调用生效）和构建 ChatClient。

方法签名 功能 说明
defaultSystem(String text) 设置默认 System Message 可被单次调用 .system() 覆盖
defaultSystem(Consumer<SystemMessageSpec> spec) 设置参数化的默认 System Message 支持 {placeholder} 占位符
defaultTemperature(double t) 设置默认采样温度（0~2） 值越低回答越确定，越高越随机
defaultTopP(double p) 设置默认核采样参数 与 temperature 配合使用
defaultTopK(int k) 设置默认 Top-K 采样 限制每步候选 token 数
defaultOptions(ChatOptions options) 设置默认模型选项 模型名、max_tokens 等
defaultTools(Object... tools) 注册默认工具（按对象） 框架自动扫描对象中带 @Tool 注解的方法
defaultFunctions(String... names) 注册默认工具（按 Bean 名） 从 Spring 容器中按名称查找
defaultFunctionCallbacks(FunctionCallback... cbs) 注册编程式 FunctionCallback 需手动实现接口
defaultAdvisors(RequestResponseAdvisor... advisors) 注册默认 Advisor 拦截器 如记忆管理、日志、RAG 检索
build() 构建不可变的 ChatClient 返回的 ChatClient 线程安全

易错点：

  • defaultTools() 和 defaultFunctions() 是对同一次注册的补充，不是覆盖。多次调用会累积工具。
  • defaultOptions() 设置的值会被单次 .options() 覆盖，不是合并。

1.4 发起一次调用：Prompt 构建 API

调用链：chatClient.prompt() → 中间构建方法 → .call() / .stream()

.prompt() 返回的是一个 Prompt 构建器，后续所有方法都是在构建这一次调用的参数。

1.4.1 Prompt 构建方法一览

方法签名 功能 覆盖关系 示例

5.1 概念

Function Calling（工具调用）让 LLM 能够在对话中自动决定何时调用外部工具，并生成结构化的调用参数。框架负责自动将 Java 方法暴露为 LLM 可识别的工具定义、解析模型返回的调用请求、执行对应的 Java 方法、将结果回传给模型——整个过程对开发者透明。

5.2 @Tool 注解 —— 声明式工具定义（推荐）

Spring AI 提供 @Tool 和 @ToolParam 两个注解来声明工具方法。这是推荐方式——只需在方法上添加注解，框架自动完成 JSON Schema 生成、参数反序列化、结果序列化。

5.2.1 注解定义

注解属性 类型 必填 说明
@Tool.description String ✅ 工具的功能描述。直接决定 LLM 何时调用此工具——描述越准确，调用时机越正确
@ToolParam.value String ❌ 参数的描述。帮助 LLM 正确填充参数值
@ToolParam.required boolean ❌（默认 true） 参数是否必填。设为 false 时 LLM 可以省略该参数

5.2.2 完整示例

易错点：

  • @Tool.description 是给 LLM 看的，不是给你看的。描述越准确，LLM 调用工具的时机越正确。写清楚”什么情况下应该调用这个工具“。
  • 方法参数的类型和名称直接影响 LLM 传递的参数。建议使用有意义的参数名（如 orderId 而非 id），并在 @ToolParam 中补充格式说明。
  • 工具方法中抛出的未捕获异常会被框架捕获并作为错误结果回传给 LLM。LLM 可能会基于错误信息再次尝试调用。

toolContext 用于向工具方法传递业务上下文（如当前用户ID、租户ID），避免工具方法与业务代码耦合：

关键设计原则：工具方法的参数应该只包含 LLM 需要理解和填充的信息（如订单状态、搜索关键词），业务身份信息（userId、tenantId）应该通过 toolContext 传递——LLM 不应该看到也不需要理解这些信息。

5.5.4 工具执行超时与异常处理

工具异常处理原则：

  • 抛出 ToolExecutionException 让框架将错误信息回传给 LLM——LLM 会基于错误消息生成用户友好的回复
  • 不要在工具方法中直接返回错误信息字符串（如 “查询失败”）——LLM 可能误以为这是查询结果
  • 工具方法应设置合理超时（防止模型等待过久）

2. AiServices —— 声明式 AI 服务

2.1 概念

AiServices 是 LangChain4j 最具特色的核心设计。它让你通过定义 Java 接口的方式来声明 AI 能力——框架自动处理 LLM 调用、工具选择、对话记忆管理和输出格式化。

对比直观理解：

  • Spring AI 方式：写 Controller → 注入 ChatClient → .prompt().user().call().content() ——命令式
  • LangChain4j AiServices 方式：定义接口 → 加注解 → assistant.chat(msg) ——声明式

核心思路：你只负责定义接口（”我需要什么”），框架负责实现（”怎么做”）。这和 Spring Data JPA（定义 findByName → 自动生成查询）是同一设计哲学。

3. ChatMemory —— 对话记忆管理

3.1 概念

ChatMemory 负责存储和管理对话历史。每次调用 LLM 时，框架自动将历史消息注入到 Prompt 中，实现多轮对话能力。LangChain4j 的 ChatMemory 与 AiServices 深度集成——通过 @MemoryId 注解自动按用户/会话隔离记忆。

3.2 ChatMemory 方法表

方法签名 功能 说明

void add(ChatMessage message) 追加一条消息 框架在每次调用后自动将 UserMessage 和 AiMessage 写入
void clear() 清空当前会话 用于”忘记上下文”功能

3.3 内置实现对比

实现类 淘汰策略 优点 缺点 适用场景
MessageWindowChatMemory.withMaxMessages(20) 保留最近 20 条消息 简单直观 长回复可能很快触发淘汰 短对话、快速轮次
TokenWindowChatMemory.withMaxTokens(4000, tokenizer) 保留最近 4000 Token 以内的消息 精确控制 Token 消耗 需要注入 Tokenizer 成本控制场景

易错点：

  • MessageWindowChatMemory 的 maxMessages(20) 是指保留最近 20 条消息，不是 20 轮对话。一条 UserMessage + 一条 AiMessage = 2 条消息 = 1 轮对话。
  • TokenWindowChatMemory 需要传入对应模型的 Tokenizer（如 OpenAiTokenizer），如果 Tokenizer 与模型不匹配，Token 计数会不准确。

八、框架使用中的通用最佳实践

1. 配置管理

配置原则：

  • API Key 用环境变量：${OPENAI_API_KEY}，绝不硬编码
  • 按 Profile 区分模型：开发用便宜/本地模型，生产用最强模型
  • 集中管理 Prompt：资源文件纳入版本控制和 Code Review
  • 连接池配置：为模型 API 调用配置合理的超时和重试

2. 异常处理与容错

容错策略：

策略 适用场景 实现方式
重试 网络波动、短暂 Rate Limit Spring Retry / Resilience4j
降级 主模型不可用 Fallback 到备用模型
熔断 模型持续不可用 Resilience4j CircuitBreaker
缓存 相同问题重复请求 Spring Cache + 语义 Hash
超时控制 防止长时间阻塞 为每次调用设置合理超时

3. 可观测性

AI 应用可观测性三个层次：

  1. 基础设施层：API 延迟、错误率、可用性 → Actuator + Prometheus + Grafana
  2. 模型调用层：Token 消耗、调用成本、模型响应时间 → 框架内置 Metrics
  3. 应用质量层：回答准确率、用户满意度、检索召回率 → 评测体系
