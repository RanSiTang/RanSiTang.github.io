AI工作流与编排

发表于 2026-05-29 更新于 2026-06-07 分类于 AI

一、AI Workflow 的基本概念与定位

1. 什么是 AI Workflow

AI Workflow（AI 工作流）是将多个 AI 操作（LLM 推理、工具调用、RAG 检索等）与非 AI 操作（条件判断、数据转换、人工审批等）按照预定义的有向图结构组合起来，形成一个可控、可观测、可复现的自动化处理流程。

理解 AI Workflow 最直观的方式是把它与传统业务流程自动化（如 Activiti、Camunda 工作流引擎）做类比：传统工作流编排的是”人工审批 → 系统处理 → 通知发送”等确定性步骤；AI Workflow 在此基础上引入了LLM 作为流程中的一个节点——这个节点的输入和输出是自然语言，行为具有一定的非确定性，但它被约束在流程图的固定位置上，输入输出受前后节点的控制。

2. Workflow 与 Agent 的核心区别

这是 AI 应用架构中最容易混淆的两个概念，它们的核心差异在于决策权的归属。

维度 AI Workflow（工作流） AI Agent（智能体）
决策方式 流程图预定义路径 LLM 自主决策下一步
控制权 流程引擎控制执行顺序 LLM 作为大脑规划执行顺序
可预测性 高——路径在设计中确定 低——模型可能走非预期的路径
灵活性 低——只能走预定义的路径 高——可应对未预见的情况
适用场景 标准化业务流程 开放式任务、探索性任务
工程复杂度 中（设计流程图） 高（需处理 Agent 失控风险）

用一句话概括：Workflow 是”你决定流程，LLM 执行节点”；Agent 是”LLM 决定流程，你提供工具”。

Workflow 的核心价值是将 LLM 的非确定性装进流程的笼子里——你知道用户提的问题会经过”查询改写 → 检索 → 相关性判断 → 回答生成”这个固定流程，每个阶段的输入输出都能被监控和干预。Agent 的核心价值是灵活性——面对开放式任务，不需要事先枚举所有可能路径。在实践中，两者经常组合使用：Agent 作为顶层的任务规划器，内部调用 Workflow 执行标准化的子任务。

3. AI Workflow 要解决的核心问题

问题一：单次 LLM 调用的能力边界

单次 LLM 调用在做复杂任务时存在明显局限——输出长度有限、推理深度不足、无法访问外部工具。Workflow 通过串联多个 LLM 调用和工具节点，将复杂任务拆解为多个专注的步骤，每个步骤职责单一、输入明确、输出可控。

问题二：LLM 输出的不可控性

LLM 的输出是概率性的——同样的输入可能产生不同质量的输出。在直接对话模式下，你只能通过 Prompt 来约束模型行为。Workflow 通过在 LLM 节点前后加入校验节点、条件路由节点、格式化节点，将模型的非确定性输出约束在可控范围内。

问题三：业务流程的标准化需求

企业级 AI 应用不只是”聊天”——它需要接入现有的业务流程（审批、通知、审计）。Workflow 将 AI 能力嵌入到标准化的业务流程中，使 AI 成为业务流程的一个步骤，而不是独立运行的”黑盒”。

二、AI Workflow 的核心节点类型

AI 工作流由不同类型的节点按照有向边连接而成。每种节点承担特定的职责，理解各节点的能力边界是设计好工作流的前提。

1. LLM 节点（LLM Node）

LLM 节点是 AI Workflow 的核心——它调用大语言模型完成特定的文本处理任务。

典型用途：

  • 意图分类：将用户的自然语言输入分类为预定义的意图标签（如”查询订单”、”提交工单”、”一般咨询”）
  • 信息抽取：从非结构化文本中抽取结构化字段（如从简历中抽取姓名、工作年限、技能列表）
  • 文本生成：基于上下文生成回复、摘要、报告
  • 语义判断：判断文本是否满足特定条件（如”检索结果是否与用户问题相关”）
  • Query Rewrite：将用户的模糊查询改写为更适合检索的精确查询

LLM 节点的配置要素：

配置项 说明 示例
System Prompt 节点的角色和行为规范 “你是一个意图分类器…”
User Prompt 模板 支持变量注入的模板 “用户输入：“
输出格式约束 JSON Schema、正则等 {"intent": "string", "confidence": "number"}
模型选择 不同节点可用不同模型 意图分类用轻量模型，生成用强模型

LLM 节点的设计原则：

  • 单一职责：一个 LLM 节点只做一件事。意图分类和回答生成应该分属两个节点，而不是放在同一个 Prompt 中
  • 输入明确化：LLM 节点的输入应该来自上游节点的结构化输出，而不是原始用户输入。例如，”回答生成”节点的输入是”检索到的文档 + 用户问题”，而不是”用户原话”
  • 输出结构化：尽可能约束 LLM 节点输出结构化数据（JSON），而不是自由文本。结构化输出可以被下游的条件节点和代码节点精确处理

2. 条件判断节点（Condition Node）

条件判断节点根据上游 LLM 节点或代码节点的输出结果，决定流程的下一步走向。它实现了工作流的分支逻辑。

条件判断有两种模式：

  • LLM 判断：由 LLM 节点输出一个分类标签或布尔值，条件节点根据标签路由。优势是灵活（能处理模糊语义），风险是 LLM 可能产生错误的分类
  • 规则判断：由纯代码逻辑（if-else、正则匹配、阈值比较）决定路由。优势是确定性和零成本，局限是只能处理可编码的规则

条件节点的设计建议：

  • 兜底分支必须有：为”无法判断”或”置信度不足”的情况设置兜底路由，通常是回退到通用回复或转人工
  • LLM 判断 + 规则兜底：先由 LLM 判断意图，当 LLM 输出的置信度低于阈值时，回退到规则判断。这结合了两者的优势
  • 分支路径不宜过多：超过 5-7 个分支时，LLM 的分类准确率开始下降。过多分支考虑分层分类——先粗分再细分

3. 工具调用节点（Tool Node）

工具调用节点执行具体的业务操作——调用后端 API、查询数据库、发送通知等。在 Function Calling 章节中详细讨论了工具定义和调用机制，这里聚焦于工作流中工具节点的编排特性。

在工作流中使用工具节点 vs 直接在 Agent 中使用工具：

工作流中的工具节点 Agent 中的工具调用
触发方式 流程图决定何时调用 LLM 自主决定何时调用
参数来源 上游节点的结构化输出 LLM 从用户话语中推理
出错处理 流程预设的异常路径 模型自行判断如何处理
适用场景 标准化业务操作 探索性、不确定性操作

工具节点的输入绑定：在工作流中，工具节点的参数通常不是由 LLM 直接从用户话语中抽取的，而是绑定自上游节点的结构化输出。例如，意图识别节点输出 {"intent": "query_order", "orderId": "ORD-12345"}，下游的订单查询工具节点直接使用 orderId 作为参数。这种方式消除了 LLM 参数推理的不确定性。

4. RAG 检索节点（Retrieval Node）

RAG 检索节点负责从外部知识库中检索与用户问题相关的文档片段。它通常嵌入在问答类工作流中，位于”查询处理”和”回答生成”之间。

RAG 检索节点的配置：

  • 检索源：向量数据库 Collection、全文搜索引擎、结构化数据库
  • 检索策略：纯向量检索、混合搜索、Multi-Query 检索
  • TopK 和阈值：在工作流层面控制检索参数，而非硬编码
  • 后处理：去重、排序、上下文拼接

5. 人工审批节点（Human-in-the-Loop Node）

人工审批节点在工作流中设置一个人工确认点——流程暂停，等待人工审核通过后再继续执行。

典型场景：

  • 敏感操作确认：LLM 生成的邮件、公告、工单需要人工审核后才能发送
  • 高风险决策：模型建议的操作可能造成业务影响（如批量取消订单），需要人工确认
  • 置信度不足时的兜底：当 LLM 判断置信度低于阈值时，转人工处理

人工审批节点的实现要点：

  • 暂停与恢复机制：工作流引擎需要支持暂停执行，保存上下文状态，在人工审批完成后恢复执行
  • 超时策略：设置审批超时时间，超时后按预设策略处理（自动驳回 / 自动通过 / 升级通知）
  • 审批上下文：审批界面需要展示充分的上下文——LLM 生成了什么、依据是什么、风险点在哪

6. 代码/脚本节点（Code Node）

代码节点执行确定性的数据处理逻辑——数据转换、计算、校验、格式化等。它是工作流中”确定性”的锚点，与 LLM 节点的”非确定性”形成互补。

典型用途：

  • 数据格式转换（JSON → XML → Markdown）
  • 参数校验与清洗
  • 数值计算（价格汇总、时间差计算）
  • 调用已有的 Java Service 方法

在工作流设计中，应该尽可能将确定性的逻辑放在代码节点中，只将需要语义理解的任务交给 LLM 节点。这样做不仅降低成本和延迟，也提高了流程的可预测性。

7. 数据转换节点（Transform Node）

数据转换节点负责在不同节点之间转换数据格式——上游节点的输出格式不一定匹配下游节点的输入格式。虽然可以归入代码节点，但由于其使用频率极高，值得单独讨论。

三、主流 AI 工作流编排方案

1. 低代码平台：Dify

Dify 是目前最流行的开源 AI 应用开发平台之一，提供可视化的 AI 工作流编排能力。

核心设计：

  • Chatflow 与 Workflow：Dify 区分两种应用类型——Chatflow（对话式，有记忆的交互式 AI）和 Workflow（批处理式，单次执行的自动化流程）
  • 可视化拖拽编排：在 Web UI 上拖拽节点、连线，配置参数，无需编写代码
  • 内置丰富的节点类型：LLM、知识检索（RAG）、Code（Python/JS）、HTTP 请求、IF/ELSE 条件、模板转换、变量聚合等
  • 知识库管理集成：平台内置文档上传、切分、向量化、检索的全流程管理，与 RAG 节点无缝衔接

Dify 适合的场景：

以非技术人员为主的团队、需要快速验证 AI 应用原型的场景、流程结构相对简单（节点数 < 30）的应用。

对 Java 后端的定位：Dify 可以作为 AI 逻辑层的前置服务，Java 后端通过 REST API 调用 Dify 应用，将 Dify 视为一个”AI 处理服务”。复杂的业务逻辑（事务管理、权限校验、数据聚合）仍在 Java 后端完成。

2. 低代码平台：Coze

Coze（扣子）是字节跳动推出的 AI Bot 开发平台，同样提供可视化工作流编排。

与 Dify 的差异：

  • 插件生态更丰富（飞书、抖音生态集成）
  • 更适合构建对话式 AI Bot，工作流能力相对 Dify 更偏对话场景
  • 国内部署和合规体验更好

3. 开发框架：LangGraph

LangGraph 是 LangChain 生态中的工作流编排框架，用**代码定义状态图（State Graph）**来编排 AI 工作流。与 Dify 的低代码路线不同，LangGraph 面向开发者，提供最大程度的控制力。

核心概念：

  • State（状态）：工作流中流动的数据对象，每个节点读取和更新 State
  • Node（节点）：工作流中的执行单元——可以是 LLM 调用、工具调用、任意 Python/Java 函数
  • Edge（边）：节点之间的连接——普通边（固定路由）和条件边（动态路由）
  • Graph（图）：由 Node 和 Edge 组成的有向图，定义了完整的执行流程

LangGraph 的核心优势：

  • 循环与条件分支：支持复杂的图结构（不只是 DAG），节点可以形成循环（如反复检索直到找到足够相关信息）
  • 状态持久化：内置 Checkpoint 机制，流程执行状态可以持久化，支持暂停、恢复、重放
  • 人机协同：原生支持 interrupt 机制，在指定节点暂停等待人工输入
  • 流式输出：支持节点级别的流式输出，用户可以看到流程的实时进展

4. 方案对比与选型

维度 Dify / Coze LangGraph Spring AI + 自研
开发方式 可视化拖拽 代码定义（Python/Java） Java 代码定义
学习曲线 低 中-高 中（Java 开发者友好）
灵活性 低（受限于平台提供的节点类型） 高（可自定义任意节点） 最高（完全控制）
部署方式 平台 SaaS 或 Docker 自部署 嵌入应用进程 嵌入 Spring Boot
监控与运维 平台内置仪表盘 需自建（LangSmith 商业方案） 集成 Spring Actuator / Micrometer

Java 后端团队的推荐路径：

原型阶段用 Dify 快速试错，确认 AI 流程可行后，用 Java 代码重新实现为生产级服务。这样做避免了低代码平台在规模化运维时的灵活性和集成度不足的问题。

五、工作流引擎与业务系统的集成

1. 工作流节点调用现有 Service 与 DAO

工作流中最大的工程价值之一是复用现有的 Java 业务服务。AI 工作流节点可以直接注入和调用 Spring Bean（Service、DAO、第三方 API 客户端），无需通过 HTTP 中转。

2. 工作流中的事务与补偿

AI 工作流跨多个节点执行，且 LLM 节点本身不支持事务回滚。需要设计补偿机制——当流程中的某一步失败时，执行补偿操作撤销前面已完成步骤的副作用。

补偿机制的核心原则：每一步”写操作”后立即注册对应的补偿函数，异常发生时逆序执行补偿。对于 LLM 节点的输出无法回滚的情况（LLM 的输出已经发送给用户），通过人工介入和审计日志来补救。

3. 工作流的监控与可观测性

AI 工作流的生产运维需要比传统工作流更丰富的监控维度——不仅要监控技术指标（延迟、错误率），还要监控 AI 质量指标（LLM 节点输出质量、检索召回率）。

AI 工作流监控的核心不是看”流程跑通了没有”，而是回答三个问题：哪些路径是常见的（优化主路径）、哪些节点是瓶颈（延迟和 Token 消耗最高）、哪些条件分支的 LLM 判断不稳定（准确率波动大）。这些问题的答案直接指导工作流的设计优化。
