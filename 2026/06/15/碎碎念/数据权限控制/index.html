数据权限控制——从裸奔到终极方案的五年演进

问题背景

假设我们正在开发一个农业信息化系统，核心数据有三张表：

表名 说明
farmer_info 农户信息
land_info 耕地信息
planting_info 种植信息

系统的组织架构分为 县 → 乡镇 → 村 三级，对应的角色与权限如下：

角色 数据可见范围
县级管理员 (COUNTY_ADMIN) 查看全县所有数据
乡镇管理员 (TOWN_ADMIN) 只能看本镇管辖的村数据
村级管理员 (VILLAGE_ADMIN) 只能看本村数据

这个需求本质上是一个行级数据权限问题：同一张表，不同角色看到的行不同。前端查询列表时，会传入 townCode 或 villageCode 进行筛选——但后端的权限控制绝不能依赖前端传参。

为了代码连贯性，文中统一使用以下基础类：

  • LoginUser：当前登录用户实体，包含 getRole() 和 getRegionCode() 方法。
  • SecurityUtils：获取当前登录用户的工具类。
  • RoleEnum：角色枚举，含 COUNTY_ADMIN、TOWN_ADMIN、VILLAGE_ADMIN。

面对这个极其通用的需求，我们的代码架构经历了五个阶段的痛苦演进。下面逐一展开。

致命问题

  • 水平越权（IDOR）：一个村级管理员（只能看 V01 村）在浏览器抓包后，把请求参数里的 villageCode 改成 V02，就能轻松拉取隔壁村的数据。这根本不叫权限控制，这叫裸奔。

第二阶段：简单粗暴——在 Service 层疯狂 if-else

安全审计给了差评。团队意识到问题后，第一反应是在业务代码里”补丁式”地加上权限校验。

实现方式

暴露的问题

  1. 代码冗余（违反 DRY 原则）：系统有 50+ 个查询接口（农户、耕地、农机、补贴……），这段 if-else 要复制粘贴 50 遍。
  2. 极易遗漏：新同事写了个新接口，忘了加这段判断，系统瞬间再次出现越权漏洞。
  3. 违反开闭原则：如果新增一个”片区管理员”角色，需要把所有 Service 层代码翻出来改一遍，维护成本极高。

暴露的新问题

虽然代码变优雅了，但架构上存在两个隐患：

  1. “静默修正”掩盖攻击行为：黑客故意传 V02，AOP 悄悄改成 V01，返回了 V01 的数据。对黑客而言是一种奇怪的体验；对系统而言，日志里没有任何越权尝试的记录，安全审计形同虚设。
  2. 强耦合业务字段：接口写死了 setTownCode / setVillageCode。如果新业务表叫”农机设备”且字段名是 town_id，或者”片区管理员”一次管 3 个镇（需用 IN 而非 =），这个设计就彻底瘫痪。

依然存在的弊端

这是很多传统开源框架（如 RuoYi 早期版本）的通用方案，但仍不完美：

  1. XML 侵入性太强：每个 Mapper.xml 的最后都必须手动加上 ${dataScopeSql}，少加一处权限就失效。
  2. 内部调用容易翻车：定时任务（如半夜统计全县数据）直接调用 Service，没经过 AOP 拦截，或没传 DTO，容易引发权限错乱或空指针。
  3. SQL 注入风险点：虽然 regionCode 来自服务端 session，但如果拼接方式不当或数据源不可信，${} 本身就是风险敞口。

第五阶段：终极方案——基于 JSQLParser AST 的 MyBatis 底层拦截器

设计理念

彻底抛掉在应用层拼接 SQL 的思路，直接在 MyBatis 底层进行拦截。

利用 JSQLParser 引擎，将即将执行的 SQL 解析成抽象语法树（AST），精准定位到 WHERE 节点，动态追加权限条件。好处是：

  • Service、Controller、DTO、XML 全部零侵入。
  • 无论上层代码怎么写，发往数据库的 SQL 都被死死卡住。
  • 连 LEFT JOIN 的关联表都能自动加上权限。

方案优点

  • 业务层零侵入：Service、Controller、DTO、XML 全部保持干净，开发人员完全感知不到权限代码的存在。
  • 绝对安全：底层拦截，连 LEFT JOIN 关联表都能自动追加权限条件。
  • 扩展性强：新增角色只需在 Handler 的 switch 中加一个分支，无需改动任何业务代码。

不可忽视的问题

  1. 微小性能损耗：每次查询都需经过 JSQLParser 解析 AST，带来微秒级消耗。缓解手段：在 Handler 中维护白名单，只解析特定业务表。

  2. 复杂 SQL 解析可能失败：如果历史 XML 中存在极其复杂的嵌套 SQL 或数据库特有函数，JSQLParser 可能抛出 JSQLParserException。建议：新项目从一开始就用，老项目迁移时逐表灰度。

  3. 权限绕过的管理成本：定时任务 / Feign 内部调用需要全量数据时，必须记得加 @InterceptorIgnore(dataPermission = "true")，忘了加就查不全。建议：在 Code Review 中重点检查定时任务 Mapper。

  4. 跨表权限的隐式陷阱（最隐蔽）：上面的代码默认每张业务表都有 town_code 和 village_code。但现实中 farmer_info 可能只存了 village_code（因为村属于镇，通过村即可推导镇），此时乡镇管理员执行 WHERE town_code = 'T01' 会直接报字段不存在。

    正确做法：

    • 方案 A：子查询反推 —— village_code IN (SELECT village_code FROM sys_village WHERE town_code = 'T01')
    • 方案 B：在 AST 中判断当前 SQL 是否已 JOIN 了村表，若未 JOIN 则动态追加。
    • 建议：在 Handler 中维护一份「表 → 权限字段映射」的配置，而不是硬编码字段名。

实战避坑指南

一、INSERT / UPDATE / DELETE 怎么办？

本文重点讨论了 SELECT 查询场景，但 MyBatis-Plus 的 DataPermissionInterceptor 同样会拦截 UPDATE 和 DELETE 的 WHERE 子句——一个村级管理员即使抓包改了 id 试图修改隔壁村数据，最终执行的 SQL 也会被自动追加 AND village_code = 'V01'，更新影响 0 行。

INSERT 则略有不同：

  • 前端提交时，requestBody 中的 villageCode 应在 Controller 层校验，确保它与当前用户的辖区一致。
  • 推荐做法：在 应用层层一处理，强制使用当前用户的所属机构覆盖参数或直接抛异常拒绝。

四、选型决策树

五、列表导出

列表导出（Excel 导出）往往复用同一个查询方法。如果导出接口绕过了 Controller 层的 AOP，数据就可能越权泄露。

排查清单：

  • 导出功能是否复用了带权限控制的查询方法？
  • 导出接口是否经过了 AOP / Interceptor 的拦截链？
  • 如果导出是异步的（消息队列），用户上下文是否正确传递？

数据权限控制本质上是一个横切关注点（Cross-cutting Concern），用 AOP 解决是天经地义的。区别只在于切在哪里：

  • 切在 DTO 参数（第三阶段）→ 侵入小，但静默覆盖，审计不友好。
  • 切在 SQL 片段（第四阶段）→ 审计友好，但 XML 侵入大。
  • 切在 SQL AST（第五阶段）→ 全面零侵入，是当前工程上的最优解。
