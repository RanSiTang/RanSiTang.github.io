视频处理与分发

发表于 2026-06-29 分类于 碎碎念 ， 视频处理与分发

一、视频技术基础

1. 视频的本质：静止画面的连续播放

视频在物理层面是一系列静止图像（帧）按固定时间间隔连续播放产生的视觉暂留效果。理解这一点是理解所有视频处理技术的起点。

这个量级的原始数据不可能直接存储和传输。因此视频技术的核心矛盾始终是：在可接受的视觉质量下，把数据量压缩到原始体积的 1/100 ~ 1/300。要解决这个矛盾，就需要引入视频编码技术。

2. 视频编码：从像素到压缩数据

编码器的目标是在尽可能保留视觉质量的前提下，大幅削减冗余数据。它主要从两个维度入手：空间冗余（单帧内相邻像素高度相似）和时间冗余（相邻帧之间的画面大量重复）。

2.1 编码器的核心手段

手段 原理 压缩贡献
帧内压缩 针对空间冗余——单帧内部的 JPEG 式压缩，用 DCT 变换剔除人眼不敏感的高频细节 单帧缩小 10~20 倍
帧间压缩 针对时间冗余——关键帧（I 帧）存完整画面，中间帧（P/B 帧）只存与前一个完整帧的差异部分 整段视频再缩小 5~10 倍
色度子采样 人眼对亮度敏感、对颜色不敏感——亮度信息全采样，颜色信息每 2~4 个像素才采一次（YUV 4:2:0） 直接砍掉一半颜色数据
熵编码 高频像素值用短码、低频用长码（哈夫曼 / CABAC） 再压缩 10%~20%

帧间压缩是视频区别于图片压缩的根本所在，也是以下现象的根本原因：

  • 视频文件比逐帧存 JPG 小得多
  • 视频编码远比图片编码耗 CPU
  • 视频播放时需要解码前序帧才能还原当前帧（这也是视频不能随意”跳帧”的原因）

I 帧 / P 帧 / B 帧简介：I 帧是完整画面，可以独立解码；P 帧依赖前一个 I 帧或 P 帧；B 帧同时依赖前后两帧。三种帧类型共同构成 GOP（Group of Pictures，图像组）。GOP 越大，压缩率越高，但 seek（跳转播放）越慢，因为需要找到最近的 I 帧才能开始解码。

2.2 主流的视频编码格式

有了编码原理，自然形成了不同的编码标准。各标准在压缩效率和兼容性之间做出了不同权衡：

编码 年份 压缩效率 兼容性 适用场景
H.264 (AVC) 2003 基准 (1×) 几乎 100% — 所有浏览器、所有设备 Web 首选，兼容性最好的格式
H.265 (HEVC) 2013 比 H.264 小 40%~50% ~95% — Safari 全支持，Chrome 桌面支持、移动端部分支持 4K/8K、带宽敏感场景
VP9 2013 接近 H.265 ~90% — Chrome/YouTube 主力，Safari 不支持 YouTube 场景
AV1 2018 比 H.265 再小 20%~30% ~85% — Chrome/Firefox 支持，Safari 17+ 下一代标准，编码极慢

现实选择：H.264 仍是目前 Web 视频的默认答案。除非带宽成本占比极高（如 YouTube 级别的视频平台），否则 H.265/AV1 编码多出来的 CPU 时间成本不一定划算。实际生产中最常见的做法是 H.264 主版本 + H.265 可选版本——后者按需转码，优先提供给 4K 场景或对带宽敏感的客户端。

2.3 封装格式（容器）

编码只解决了”如何压缩视频画面”的问题，但一个完整的视频文件还包含音频流、字幕、章节信息、时间戳等。这些内容需要被组织到一个统一的文件结构中，这就是**封装格式（容器）**的作用。

可以把它理解为：编码格式是压缩算法，封装格式是存放这些压缩数据的”盒子”。同一种编码（如 H.264）可以被装进不同的容器（MP4、MKV、FLV）。

封装格式 扩展名 支持的编码 流媒体支持 特点
MP4 .mp4 H.264/H.265 + AAC ❌ 需完整文件 兼容性最好，所有播放器都支持
WebM .webm VP8/VP9/AV1 + Opus/Vorbis ❌ Web 端专为 HTML5 设计
HLS (m3u8 + ts) .m3u8 H.264/H.265 + AAC/MP3 ✅ 原生流媒体 苹果主导，CDN 友好，切片加载
DASH .mpd 任意 ✅ 原生流媒体 国际标准，编码无关，自适应码率
FLV .flv H.264 + AAC ✅（依赖 RTMP） 直播历史格式，逐渐被 HLS 取代

MP4 的一个关键细节（faststart）：MP4 的元数据（moov atom）默认写在文件末尾。浏览器必须先下载完整文件或至少读到 moov 才能开始播放，这意味着用户需要等待整个文件传输完才能看到第一帧。

修复方式是编码时加 -movflags +faststart，强制将 moov 移到文件开头。代价是编码完成后需要额外做一次文件重写，但换来的是播放器可以边下载边播放。对于 Web 场景，这个参数几乎是必加项。

3. 视频元数据提取

理解了编码和封装之后，就能明白一个视频文件内部实际上包含了大量的描述性信息：时长、分辨率、编码格式、码率、帧率、旋转角度等。这些信息统称为元数据（Metadata）。

在后端处理视频时，几乎所有决策都依赖元数据。例如：

  • 视频编码是 WebM/VP9？Safari 用户会播放失败，需要转码为 H.264
  • 分辨率是 4K？考虑是否需要生成 1080P 的降级版本
  • 手机拍摄的视频带有 90° 旋转标记？直接播放会横屏显示
  • 码率异常高？可能是未经压缩的原始文件，需要先压缩再存储

因此，视频处理的第一步往往不是修改内容，而是读懂文件里有什么。

3.1 FFmpeg 命令行提取

ffprobe 是 FFmpeg 工具链中专门用于读取媒体信息的命令，不做任何转码操作，速度极快。在后端开发中，它既可以作为本地调试工具，也可以通过 ProcessBuilder 在 Java 中直接调用。

3.2 Java 提取：JAVE2

在 Java 后端中，有两种主流方式读取视频元数据，它们的底层都依赖 FFmpeg，但封装层次不同。

方案一：JAVE2（高层封装，适合快速集成）

JAVE2 对 FFmpeg 做了较高层的封装，API 简单，适合只需要读取常规元数据的场景，无需关心 FFmpeg 底层细节。

方案二：JavaCV（底层封装，功能更完整）

JavaCV 是对 FFmpeg 原生库的 JNI 绑定，功能比 JAVE2 强大得多，支持逐帧读取、实时流处理等高级操作。代价是包体积更大（包含本地 so/dll），API 更接近 FFmpeg 原始风格。推荐在需要做视频转码、帧截取、流处理时使用。

两种方案对比

对比维度 JAVE2 JavaCV
API 复杂度 低，开箱即用 高，接近 FFmpeg 原生风格
包体积 较小 较大（含本地库）
功能范围 元数据读取 + 基础转码 元数据 + 转码 + 逐帧 + 流处理
旋转角度 不支持 支持
推荐场景 只需读取元数据的轻量场景 需要做视频处理的完整功能场景

3.3 元数据的典型应用场景

元数据字段 典型应用
width / height 判断是否需要降分辨率转码、生成缩略图、选择转码档位（1080P/720P）
duration 预估转码耗时、按时长阶梯计费、限制最大上传时长
videoCodec 判断浏览器兼容性（如 Safari 不支持 VP9，需转为 H.264）
bitRate 判断原始质量等级，决定压缩力度和目标码率
rotation 修正手机竖屏视频在 Web 端横屏显示的问题
frameRate 监控摄像头视频可降至 15fps，常规视频保持 30fps 以节省存储
format 判断是否需要重新封装（如 FLV → MP4）以兼容 HTML5 播放器

二、视频转码

视频转码是视频处理链路中最核心也是最耗资源的环节。上一章我们了解了视频的编码原理、封装格式，以及如何从视频文件中提取元数据。本章在此基础上向前推进：拿到元数据之后，如何根据这些信息做出正确的转码决策，并在生产环境中稳定地执行它。

前置知识：视频文件的上传（分片上传、Presigned URL 直传）、存储（本地/OSS/MinIO）和安全校验请参见 [文件操作与传输] 笔记，本文不再赘述。元数据提取的具体实现参见上一章 §3。

1. 编码参数深入

在上一章中，我们知道了编码器通过帧内压缩、帧间压缩等手段削减冗余数据。但编码器并不是”全自动”的——它暴露了一系列参数，让开发者在画质、体积、编码速度三者之间进行权衡。理解这三个维度的参数，是写出合理转码配置的前提。

1.1 码率（Bitrate）

码率是每秒视频平均分配多少比特的数据量，直接决定了最终文件体积与画质上限。

理论上码率越高，画质越好，但两者并非线性关系：超过一定阈值后人眼无法分辨差异，体积却在线性增长。2 Mbps → 4 Mbps 体积翻倍，视觉提升可能只有 10%。因此码率的合理设置应参考分辨率和内容类型：

H.264 码率推荐值：

分辨率 推荐码率（静态内容：讲课 / 会议） 推荐码率（高动态内容：运动 / 游戏）
480p (SD) 500 ~ 1000 kbps 1000 ~ 1500 kbps
720p (HD) 1500 ~ 2500 kbps 2500 ~ 4000 kbps
1080p (FHD) 3000 ~ 5000 kbps 5000 ~ 8000 kbps
2K (QHD) 6000 ~ 10000 kbps 10000 ~ 15000 kbps
4K (UHD) 15000 ~ 25000 kbps 25000 ~ 40000 kbps

码率与分辨率要匹配：给 480p 的视频设置 8 Mbps 的码率，和给 1080p 设置 500 kbps 一样荒谬——前者浪费存储，后者让画面充满马赛克。合理做法是按上表结合内容类型取中间值，然后在实际业务中通过主观测试微调。

1.2 码率控制模式

确定了目标码率之后，还需要选择编码器以何种方式控制码率的分配。这是一个经常被忽视但影响巨大的参数：

模式 参数示例 原理 典型场景
CBR（固定码率） -b:v 2M 全程恒定码率，简单场景浪费比特，复杂场景画质下降 直播 — 需要恒定带宽以保稳定
VBR（可变码率） -b:v 2M -maxrate 4M -bufsize 8M 简单场景降码率，复杂场景提码率，整体平均贴近目标值 点播 — 追求体积与画质的平衡
CRF（恒定质量） -crf 23 不关心码率，只保证视觉质量一致（0~51，值越小质量越高，默认 23） 存档 / 母版转码 — 不在意体积

三种模式的核心区别在于：CBR 保带宽、VBR 保体积、CRF 保质量。Web 点播场景通常选 VBR；如果是内部存档或母版转码，则 CRF 是更合适的选择。

CRF 值与画质的经验对照：

CRF 值 视觉效果 推荐场景
18 视觉无损，几乎看不出任何压缩痕迹 存档母版
23 默认值，肉眼几乎不可见差异 Web 点播推荐
28 轻微压缩痕迹，可接受 移动端节省流量
35+ 明显马赛克和模糊 仅用于低质量预览

1.3 编码速度预设（preset）

码率控制模式解决了”分配多少比特”的问题，而 preset 参数则控制编码器”如何花这些比特”——更慢的预设会使用更复杂的算法（如更大范围的运动搜索、更精细的帧间预测），在相同视觉质量下挤出更小的体积。

preset 编码速度（相对实时） 同画质下的体积 推荐场景
ultrafast 极快（~10× 实时） 最大 直播转码
veryfast 很快（~6× 实时） 较大 实时处理
fast 快（~3× 实时） 中等 后端批量转码推荐
medium 中（~1× 实时） 较小 质量优先
slow 慢（~0.5× 实时） 小 存档
veryslow 极慢（~0.2× 实时） 最小 蓝光压制

生产建议：批量转码首选 fast。从 medium 到 veryslow，编码时间可能延长 5 ~ 10 倍，但体积通常只减少 10% ~ 15%，性价比极低。除非是对母版文件的一次性离线压制，否则不建议使用 slow 以下的预设。

1.5 完整转码参数模板（JAVE2）

为什么高度设 -2 而不是 -1：视频编码的色度子采样（YUV 4:2:0）要求每两个像素共享一个颜色采样点，因此宽高必须是偶数。-1 虽然也能等比缩放，但可能产出奇数高度，导致编码失败。-2 会在等比缩放后自动将高度向下取整到最近的偶数，是更安全的写法。

2. 多清晰度转码策略

掌握了单次转码的参数配置之后，下一个问题是：一个视频应该转出几个版本，每个版本的参数如何定义？

2.1 为什么需要多个清晰度？

单一清晰度无法同时满足所有场景：

  • WiFi 用户看 1080p 体验流畅，4G 用户看 1080p 可能持续缓冲
  • 手机小屏播放 1080p 与 720p 的视觉差异微乎其微，但流量消耗翻倍
  • 列表缩略图 / 内嵌播放窗口不需要加载完整高清文件
  • 不同设备屏幕分辨率差异巨大（手机 390px vs 显示器 1920px）

多清晰度转码配合后续的**自适应码率（ABR）**分发（如 HLS、DASH），可以让播放器根据当前网络带宽自动切换清晰度，做到”网好给高清，网差不卡顿”。

3. 异步转码架构

视频转码是极度 CPU 密集型的操作——一段 10 分钟的 1080p 视频，在普通服务器上完整转码三个档位可能需要 5 ~ 15 分钟。绝对不能在 HTTP 请求线程中同步执行：转码线程会长时间占用 Tomcat 工作线程，HTTP 连接极大概率超时，同时多个并发转码任务会把服务器 CPU 打满，导致整个应用无响应。

正确的做法是将转码任务彻底异步化：

3.1 上传接口：只负责接收，不负责处理

3.2 转码 Worker：消费队列任务

3.3 前端轮询状态

为什么用指数退避而不是固定间隔：视频越长，转码越慢。一个 2 小时的视频不可能在 2 秒内完成，每 2 秒发一次请求纯属无效 I/O。指数退避策略在前期提供快速反馈，后期自动降低轮询频率，是轮询场景的工程最佳实践。

3.4 三种异步方案的对比

方案 实现方式 优点 缺点 推荐场景
消息队列（首推） RabbitMQ / Kafka + 消费者 解耦、削峰、消息持久化、自动重试 引入额外中间件，运维成本稍高 生产环境，任务量大
Spring @Async @Async + 自定义线程池 零额外依赖，配置简单 服务重启任务丢失，无持久化，无重试 开发环境 / 小规模
数据库任务表轮询 定时扫描 status = PENDING 的记录 最简单，不依赖任何中间件 轮询间隔即延迟，任务量增大时 DB 压力高 极小规模 / 快速验证

为什么消息队列是生产首选：转码任务的特点是执行时间长（秒到分钟级）、失败成本高（几 GB 的原始视频重新下载很痛苦）、需要限流（无法让 100 个并发转码任务同时跑）。MQ 的消息持久化保证服务重启不丢任务；消费者并发数配置（concurrency = "2-4"）天然实现转码限流；死信队列提供失败任务的兜底处理——这三点是 @Async 和数据库轮询无法同时满足的。

三、视频截图与封面生成

视频封面是用户决定是否点击播放的第一印象，其重要性不亚于视频内容本身。前两章解决了视频的编码压缩与转码处理，本章聚焦一个相对独立但同样重要的环节：如何从视频中提取出一帧高质量的封面图。

这件事看似简单，实则有几个容易踩坑的细节：随机抽帧可能抽到全黑的片头、转场的模糊帧、或者人物眨眼的瞬间。本章从最基础的定点截帧出发，逐步介绍更智能的多帧择优策略。

1. 定点截帧：抽取指定时间点的画面

最直接的思路是：在视频的某个固定时间点截取一帧，作为封面。为了跳过片头黑场和片尾字幕，通常选取视频 10% ~ 20% 位置处的帧，而非从第 0 秒开始。

2. 智能封面选择策略

定点截帧虽然简单，但在以下场景下容易失效：

  • 片头黑场：直播录制、广告视频常在开头有几秒纯黑画面
  • 转场模糊帧：恰好抽到镜头切换的瞬间，画面虚化
  • 内容无代表性：字幕条、空镜头、纯色背景

更稳健的策略是：在视频中均匀采样若干帧，从中选出视觉质量最好的一帧。

四、流媒体与 HLS

前三章解决了视频从上传到处理的全流程：编码转码、元数据提取、封面生成。处理完成后，视频文件以 MP4 格式存储在 OSS 中，等待被用户访问。

然而，直接让用户下载 MP4 文件会带来一个根本性问题：用户必须等待整个文件下载完才能开始播放。一个 1 小时的 1080P 视频约 2 ~ 3 GB，在普通带宽下意味着等待数分钟，而用户可能只想看前 5 分钟。本章要解决的问题是：如何让视频”边加载边播放”，并且只加载用户实际观看的部分。

1. 流媒体的核心思想

流媒体的解决方案直截了当：不要让用户下载整个文件，而是把视频预先切成若干小段，用户看到哪里，就只加载那一段。

目前 Web 端最主流的流媒体协议是 HLS（HTTP Live Streaming），由苹果主导，也是本章的重点。

2. HLS（HTTP Live Streaming）

2.1 文件结构与工作原理

HLS 的核心是把一个完整的视频文件拆分为两类产物：一个播放列表文件（.m3u8）和若干视频分片（.ts）。

2.2 m3u8 文件结构

理解 m3u8 文件的格式，有助于排查播放问题和理解直播与点播的区别：

点播 vs 直播的本质区别：点播的 m3u8 包含 #EXT-X-ENDLIST，分片列表是固定的，播放器一次性解析完整列表后按需加载。直播的 m3u8 没有 #EXT-X-ENDLIST，播放器会以固定间隔（通常等于分片时长）持续轮询 m3u8，每次轮询都会发现新追加的分片。直播延迟（通常 10 ~ 30 秒）的本质就是：新画面需要经历”采集 → 编码 → 切片 → 上传 → 播放器轮询到”这个完整链路。

2.3 HLS 切片实现

HLS 切片推荐通过命令行调用 FFmpeg，原因与第三章封面截取相同——功能完整、依赖轻量、可充分利用服务器上已安装的 FFmpeg 版本。

切片关键参数说明

参数 作用 建议值及原因
hls_time 每个分片的目标时长 6 ~ 10 秒：太短导致 HTTP 请求过多；太长导致首帧等待时间长、拖拽精度差
hls_list_size m3u8 中保留的分片数 点播设 0（保留全部）；直播设 3 ~ 5（只保留最近几个分片，旧分片可删除）
independent_segments 强制每个分片以 I 帧开始 必须开启：否则分片间存在帧间依赖，拖拽进度条时播放器找不到参考帧，画面会出现花屏
hls_segment_filename 分片文件的命名模式 使用 %04d 四位补零，便于按序号排序和 CDN 缓存
hls_key_info_file AES-128 加密密钥配置文件路径 付费视频 / 版权内容需开启，防止 TS 分片被直接下载后本地播放

为什么 independent_segments 如此重要：回顾第一章的帧类型知识——P 帧和 B 帧依赖前序帧才能解码。如果一个分片的第一帧是 P 帧，播放器就必须先加载上一个分片才能解码当前分片，拖拽播放时链式依赖会导致严重卡顿和花屏。强制每个分片以 I 帧开头，才能让每个分片独立解码。

2.4 与 CDN 集成：上传切片产物

切片完成后，需要将 index.m3u8 和所有 .ts 文件上传至 OSS，并通过 CDN 分发。

m3u8 中分片路径的注意事项：FFmpeg 默认在 m3u8 中使用相对路径引用分片（如 segment_0000.ts），这意味着播放器会基于 m3u8 的 URL 拼接出分片的完整 URL。只要 m3u8 和 .ts 文件在 OSS / CDN 上处于同一目录，相对路径就能正确解析，无需修改 m3u8 文件内容。

2.5 前端播放

HLS 的浏览器兼容性并不统一——Safari（含 iOS）原生支持 HLS，但 Chrome 和 Firefox 不支持 <video> 标签直接播放 .m3u8 文件。解决方案是引入 HLS.js，它通过浏览器的 Media Source Extensions（MSE）API 在 JavaScript 层实现 HLS 解析，使得所有支持 MSE 的现代浏览器都能播放 HLS。

3. 多码率自适应：Master Playlist

单一清晰度的 HLS 解决了”不用一次性下载完整文件”的问题，但还没有解决网速波动时卡顿的问题。试想：用户在 WiFi 和 4G 之间切换，或者处于信号不稳定的环境，固定码率的视频流要么过高导致持续缓冲，要么过低在 WiFi 下浪费带宽。

自适应码率（ABR，Adaptive Bitrate）正是为此而设计的：在 HLS 的基础上再增加一层——一个汇总了所有清晰度入口的 Master Playlist，播放器根据实时网络状况在各清晰度之间动态切换，做到”网好自动切高清，网差自动降档不卡顿”。

3.1 文件结构

与单清晰度 HLS 相比，唯一的变化是在各子 m3u8 之上新增了一个 master.m3u8。各清晰度的分片结构完全不变，播放器只需将之前直接加载子 m3u8 的方式改为先加载 master.m3u8。

3.2 Master Playlist 格式解析

每个 #EXT-X-STREAM-INF 行描述一个清晰度档位，关键字段含义如下：

字段 含义
BANDWIDTH 该档位的峰值码率（bps）。播放器用此值与当前可用带宽比较，选择不超过带宽的最高清晰度
RESOLUTION 视频分辨率（宽×高）。播放器也可据此过滤掉超出屏幕物理分辨率的档位，避免无意义的高码率
CODECS 编解码器标识（见下方说明）。播放器据此判断硬件是否支持硬解该流

CODECS 字段详解：以 avc1.640028 为例，avc1 表示 H.264 编码，. 后的 6 位十六进制分别编码了 Profile（64 = High）、Constraint（00）和 Level（28 = 4.0，即支持最高 1080p@60fps）。mp4a.40.2 则表示 AAC-LC 音频。播放器在选择清晰度前会先检查自己的解码能力是否满足该 CODECS 描述，从而避免选出一个设备无法解码的流。实际开发中不需要手动计算这个值，使用下表中与 H.264 Profile 对应的固定值即可：

H.264 Profile CODECS 值 适用分辨率
Baseline avc1.42c01e 480p 及以下（兼容性最好）
Main avc1.4d401f 720p
High avc1.640028 1080p 及以上

3.4 一步到位：转码 + 切片 + 生成 Master Playlist

在实际生产中，与其先转出 MP4 再切片，不如让 FFmpeg 直接从源视频转码并输出 HLS 分片，省去中间的 MP4 文件，减少一次完整的磁盘 I/O。

为什么选择”转码 + 切片一步到位”而非先转 MP4 再切片：

  • 节省磁盘空间：省去了各清晰度的中间 MP4 文件，对于 1080p 视频这可能是几 GB 的差异
  • 减少总耗时：两步操作各需要完整读写一遍视频数据，合并为一步减少了一次磁盘 I/O
  • 代价：出错时只能从头重做该清晰度，没有中间产物可以复用。如果转码任务有重试需求，可考虑保留中间 MP4 文件

3.5 前端播放：只需将 m3u8 URL 指向 master.m3u8

HLS.js 自适应切换是完全自动的：开发者无需手动实现带宽检测和清晰度切换逻辑，HLS.js 内部维护了一套 ABR 算法（基于缓冲区水位和 EWMA 带宽估算），加载 master.m3u8 后即自动生效。如果产品需要提供”手动切换清晰度”的 UI，可通过 hls.currentLevel 和 hls.levels API 实现。

4. HLS vs DASH

DASH（Dynamic Adaptive Streaming over HTTP）是 MPEG 制定的国际标准，与 HLS 的核心思想相同——切片 + 播放列表——但在格式和生态上存在差异：

维度 HLS DASH
播放列表格式 .m3u8（纯文本，易读） .mpd（XML，结构复杂）
分片格式 .ts（MPEG-TS 容器） .m4s（fMP4，更高效）
编码支持 H.264 / H.265 + AAC 任意编码（编码无关）
Safari 支持 原生支持 不支持（需 dash.js 等库）
DRM 支持 FairPlay（苹果生态） Widevine + PlayReady（更通用）
典型使用方 国内各大视频平台、苹果生态 YouTube、Netflix、Disney+

实际选择建议：Web 端绝大多数场景用 HLS 即可。DASH 的核心优势是多 DRM 体系支持（同时覆盖 Chrome 的 Widevine 和微软的 PlayReady），以及分片格式（fMP4）的解析效率更高，这是 Netflix 这量级的流媒体平台选择 DASH 的原因。但 DASH 不支持 Safari 原生播放，意味着必须引入额外的 JavaScript 播放库——对于没有强版权保护需求的普通视频平台，这个额外复杂度没有必要。

五、CDN 分发与缓存

前四章完成了视频从上传、转码、截帧到 HLS 切片的全部处理工作。此时，所有文件已经存储在 OSS 中，等待被用户访问。然而，OSS 作为源站有一个根本性的局限：它是单点的，且通常部署在单一地域。北京的用户访问部署在上海的 OSS，物理距离带来的网络延迟无法通过任何软件优化消除。

CDN（内容分发网络）的作用正是在此：在全国乃至全球部署大量边缘节点，将用户频繁访问的内容缓存到离用户最近的节点，把”用户访问远在千里外的源站”变成”用户访问同城的边缘节点”。

1. 视频 CDN 面临的挑战

视频场景对 CDN 的挑战远比普通静态资源复杂。HLS 切片化之后，一个 1 小时的视频可能产生数百个 .ts 文件和若干 .m3u8 文件，带来以下几个独特问题：

挑战 具体表现
回源压力 热门视频的同一个 TS 分片被百万用户请求，若 CDN 未命中则每次都回源 OSS
首帧延迟 用户点击播放后，第一个 TS 分片必须足够快到达，否则出现”转圈等待”
冷热不均 新上传视频无缓存，老视频尾部分片可能数周才被请求一次
文件数量暴增 单文件 → 数百个分片，HTTP 请求数暴增，CDN 缓存索引压力增大

这些挑战对应了本章接下来要讨论的三个策略：缓存策略（解决回源压力）、预热（解决首帧延迟和冷启动）、多层架构（解决热点雪崩）。

2. CDN 缓存策略

缓存策略的核心问题是：哪些文件可以缓存多久。视频 CDN 中不同类型的文件有截然不同的更新频率，需要区别对待。

2.1 各文件类型的缓存策略

文件类型 Cache-Control 策略 原因
.ts 分片 max-age=31536000, immutable 分片内容一旦生成永不变更，同一 URL 始终对应同一内容，可永久缓存
点播 .m3u8 max-age=600（10 分钟） 点播内容一般不变，允许短期缓存以减少回源请求
直播 .m3u8 max-age=2（2 秒） 直播播放列表持续追加新分片，必须极短缓存，否则播放器收到过期列表
master.m3u8 max-age=600（10 分钟） 清晰度档位列表极少变更，短期缓存安全

immutable 指令的作用：告知 CDN 节点和用户浏览器”此 URL 对应的内容永远不会改变”。浏览器在缓存有效期内不会发送带 If-Modified-Since 或 If-None-Match 的条件请求——即使用户按 F5 强制刷新也不例外。这对 TS 分片来说是理想策略，因为 HLS 的分片 URL 包含序号（segment_0000.ts），一旦生成内容就固定不变，不存在”同 URL 内容更新”的情况。

2.2 缓存命中率与分片时长的关系

分片时长不仅影响播放体验（见第四章 §2.3），也直接影响 CDN 缓存的命中效率：

分片时长 CDN 缓存命中率 首帧等待 分析
2 秒 低 快 请求数量激增，大量长尾请求命中率低，CDN 缓存碎片化
6 秒 中 中 较为均衡
10 秒 高 可接受 推荐值——分片体积适中，缓存粒度合理，请求数量可控
20 秒 最高 慢 首帧需等待较长时间，用户体验下降

为什么分片越长命中率越高：每个 TS 文件对应一个缓存条目。分片越短，同样时长的视频产生的文件越多，视频尾部的冷门分片（大多数用户看完前 10 分钟就退出）几乎从不被请求，在 CDN 缓存中白占空间还会被 LRU 逐出。分片越长，热门前段的分片体积更大、命中价值更高，整体缓存利用率更好。

3. CDN 预热：解决冷启动问题

缓存策略保证了”已被访问过的内容”能被高效复用，但无法解决第一个访问者的体验问题——视频刚上传完成时，CDN 各节点上没有任何缓存，第一批用户（如运营刚发布的推荐视频）会直接击穿 CDN 回源 OSS，延迟可能比正常缓存命中高 10 倍以上。

预热（Prefetch / Warm-up） 的思路是：在用户访问之前，主动触发 CDN 节点向源站拉取文件并缓存。

3.1 按场景选择预热范围

全量预热一个视频的所有分片成本很高（数百个 HTTP 请求），需要根据视频的”热度预期”选择合适的预热范围：

场景 预热策略
普通用户上传 预热 前 3 个 TS 分片 + master.m3u8（确保首屏秒开，其余分片按需加载）
运营推荐 / 首页热门视频 预热所有分片（全量推送到边缘节点，彻底消除回源）
定时发布（预告片 / 新番） 定时任务在发布时间前 30 分钟执行全量预热

4. 多层 CDN 架构：防止回源雪崩

4.1 单层 CDN 的隐患

在热点事件下（如爆款视频突然上了热搜），数百个边缘节点可能在同一时刻对同一个冷门 TS 分片（刚被热门帖子引用的老视频）发生缓存 MISS，同时向 OSS 回源。OSS 在短时间内承受数百次并发回源请求，轻则请求排队延迟飙升，重则触发 OSS 的限流保护导致访问失败。

六、安全与防盗链

前五章构建了视频从上传、转码到 CDN 分发的完整链路。此时视频已经可以正常播放，但面临一个新的问题：任何人只要拿到 m3u8 的 URL，就可以免费播放甚至下载视频内容。对于付费视频、版权内容或企业内部资料，这显然是不可接受的。

本章介绍三种递进的安全手段，从最轻量的 Referer 防盗链，到生产可用的签名 URL，再到针对付费内容的 HLS 加密。三者可以叠加使用，安全级别依次提高。

1. Referer 防盗链（基础防护）

最简单的防盗链思路：检查 HTTP 请求的 Referer 头，只允许来自自己域名的请求加载视频资源。其他网站嵌入你的播放链接时，浏览器发出的请求 Referer 会指向那个外站，从而被拒绝。

valid_referers 各选项含义：

选项 含义
none 允许没有 Referer 头的请求（如直接在浏览器地址栏访问）
blocked 允许 Referer 存在但值被防火墙/代理抹去的请求
*.example.com 允许来自 example.com 及其所有子域名的请求

局限性：Referer 头由客户端提供，可以轻易伪造：

因此，Referer 防盗链只能防止普通用户的随意盗用，无法阻止有技术能力的人。它适合作为基础防护层，但不能单独作为付费内容的安全保障。

2. 签名 URL（生产推荐）

签名 URL 的原理与第一章[[文件操作与传输]]笔记中的 Presigned URL 一致：在 URL 中附加过期时间 + 签名，CDN 在返回内容前验证签名的合法性和时效性。即使 URL 被泄露，超过有效期后也自动失效，并且签名无法被伪造。

2.2 HLS 场景下签名 URL 的特殊问题

签名 URL 用在普通文件下载场景时没有任何问题，但 HLS 有一个特殊之处：播放器会自动解析 m3u8 文件，并按文件中的路径去加载 TS 分片。

这意味着：即使 master.m3u8 的 URL 带了签名，m3u8 文件内部引用的子路径也必须是带签名的，否则播放器在加载子 m3u8 和 TS 分片时会全部失败。

解决这个问题有两种方案：

方案一：服务端动态改写 m3u8 内容（推荐）

不将 m3u8 文件静态存储在 OSS，而是通过后端接口动态生成，在响应时将内部的分片路径全部替换为带签名的 URL：

方案二：CDN 回源鉴权（无需改写 m3u8）

由 CDN 在回源时统一处理鉴权，用户拿到的 URL 不带签名，CDN 边缘节点在向 OSS 回源时自动携带鉴权凭证。这种方案的缺点是：所有人都能访问 CDN 上的资源（只要知道 URL），无法做到用户级别的访问控制，适合公开视频的防盗链但不适合付费内容。

2.3 完整的播放鉴权流程

3. HLS 加密（付费内容保护）

签名 URL 解决了”没有 URL 就无法访问”的问题，但有一个残留漏洞：拿到签名 URL 的合法用户，可以在有效期内下载所有 TS 分片，在本地拼合成完整视频。对于付费视频或版权内容，这是不可接受的。

HLS AES-128 加密在分片层面解决这个问题：TS 分片以加密形式存储和传输，播放器需要向服务端请求解密密钥才能播放。下载了 TS 文件也无法在本地解密，因为密钥由服务端控制。

3.1 加密原理

关键点在于：密钥在服务端，TS 文件本身是密文。即使用户下载了所有 TS 文件，没有密钥也无法解密。而服务端的密钥接口有鉴权保护，只有付费用户才能获取密钥。

3.3 密钥分发接口（服务端鉴权）

密钥接口是整个加密体系的核心：只有通过鉴权的用户才能拿到密钥。

密钥安全存储：AES-128 密钥绝对不能明文存储在普通数据库字段中，建议使用以下任一方案：

  • KMS（密钥管理服务）：阿里云 KMS、AWS KMS，密钥由云平台硬件保护，应用只能调用加解密 API，拿不到明文密钥
  • 数据库 + 列加密：密钥字段使用应用层加密（AES 加密后再存储），主密钥通过环境变量注入，不写入代码

2. 关键设计决策总览

2.1 编码与格式

决策点 推荐选择 核心理由
视频编码 H.264 浏览器兼容性 100%；H.265/AV1 编码耗时高，除非带宽成本极高否则收益不显著
流媒体协议 HLS（m3u8 + ts） Safari 原生支持，Chrome/Firefox 通过 HLS.js 覆盖；CDN 缓存友好
封装格式（存档） MP4 + faststart 通用兼容；faststart 将 moov atom 前置，支持边下载边播放

2.2 转码参数

决策点 推荐选择 核心理由
码率控制模式 点播用 VBR，存档用 CRF 23 CBR 适合直播（恒定带宽），VBR 在体积与画质之间取平衡，CRF 保证质量一致
编码速度预设 fast medium → veryslow 体积只减少 1015%，但耗时增加 510 倍，性价比低
清晰度档位 三档（SD / HD / FHD） 覆盖 90% 场景；档位过多增加转码成本和运维复杂度
跳过超源分辨率 必须实现 将 480p 视频”转”为 1080p 只会得到模糊放大版，浪费存储和带宽

2.3 HLS 切片

决策点 推荐选择 核心理由
分片时长 8 ~ 10 秒 过短（2s）导致 HTTP 请求数激增；过长（20s+）首帧等待和拖拽精度下降
independent_segments 必须开启 强制每个分片以 I 帧开头，是自适应码率无缝切换和拖拽不花屏的共同前提
转码 + 切片方式 FFmpeg 一步到位 省去中间 MP4 文件，减少一次全量磁盘 I/O

2.4 异步架构

决策点 推荐选择 核心理由
转码执行方式 消息队列 + Worker 转码耗时分钟级，同步执行必然 HTTP 超时；MQ 持久化保证重启不丢任务
状态通知方式 前端轮询 + 指数退避 短视频 2s 轮询，长视频自动降频至 10s；WebSocket 实现稍复杂，收益有限
并发控制 消费者并发数限制 concurrency = "2-4" 限制同时运行的转码任务数，防止 CPU 打满影响其他服务

2.5 CDN 与缓存

决策点 推荐选择 核心理由
TS 分片缓存 immutable，max-age=31536000 分片内容永不改变，URL 唯一确定内容，可永久缓存
m3u8 缓存 点播 600s，直播 2s 直播列表持续更新，必须极短缓存；点播内容稳定，可适当缓存
CDN 预热范围 普通视频预热首屏，热门视频全量预热 预热所有分片成本较高，应按热度分级执行
多层架构 向 CDN 厂商确认 L2 中间层支持 热点事件时边缘节点同时 MISS 可导致 OSS 回源雪崩，中间层是必要防护

2.6 安全

决策点 推荐选择 核心理由
基础防盗链 CDN 签名 URL + OSS 私有 Bucket Referer 可伪造，签名 URL 是最低可行的安全方案
m3u8 签名 服务端动态改写内部路径 播放器不会主动为分片 URL 添加签名，必须在 m3u8 内容层面处理
付费内容保护 HLS AES-128 + 鉴权密钥接口 下载的 TS 分片是密文，没有密钥无法解密；密钥接口鉴权决定谁能播放
密钥存储 KMS 或数据库列加密 AES 密钥明文存数据库是严重的安全隐患
