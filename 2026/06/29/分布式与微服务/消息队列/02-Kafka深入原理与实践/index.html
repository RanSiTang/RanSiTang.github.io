Kafka深入原理与实践

发表于 2026-06-29 更新于 2026-07-18 分类于 分布式与微服务 ， 消息队列

一、Kafka 的定位

1. 不是”消息队列”，是”分布式提交日志”

理解 Kafka 有一个最关键的心智门槛：不要把 Kafka 当作传统消息队列来理解。

我们在前面的章节中介绍过传统 MQ（如 RabbitMQ）的核心设计——Broker 收到消息后推送给消费者，消费者确认后消息被删除。消息在这里是瞬态的，它的使命是”从 A 送到 B”，送达即完成。

Kafka 的设计哲学截然不同。它把自己定义为 分布式提交日志（Distributed Commit Log）——所有消息被持久化到磁盘，按写入顺序追加（Append-Only），消费者通过一个叫 offset 的游标来记录”我读到了哪里”，而不是”把这条消息消费掉并删除”。消息默认保留 N 天，期间可以被任意消费者反复回放。

flowchart LR
    subgraph RabbitMQ["传统 MQ（RabbitMQ）"]
        direction LR
        P1[Producer] -->|写入| Q1[Queue\nmsg1 msg2 msg3]
        Q1 -->|消费并删除| C1[Consumer]
        Q1 -.->|消费后| GONE["消息消失"]
    end

    subgraph Kafka["Kafka（分布式日志）"]
        direction LR
        P2[Producer] -->|追加写入| L1["Partition Log\n[msg1][msg2][msg3][msg4]...\n offset: 0  1  2  3"]
        L1 -->|Consumer A\noffset=2| CA["Consumer A\n读到 msg2"]
        L1 -->|Consumer B\noffset=0| CB["Consumer B\n从头重放"]
        L1 -.->|消费后| STAY["消息仍在磁盘\n可反复回放"]
    end

这一设计差异不是细节上的不同，而是根本理念的差异，它直接带来了传统 MQ 无法具备的几个关键能力：

能力 说明 典型使用场景
消息回溯 消费者可将 offset 重置到任意位置，重新消费历史消息 业务代码有 Bug，修复后重新处理历史数据
多消费者独立消费 不同消费者组各自维护独立的 offset，互不干扰 同一份数据同时被流处理引擎和数据仓库各自消费
持久化保证 消息写入即落盘 + 多副本，不因消费而删除 日志审计、合规存储
流式处理天然适配 消息以追加日志形式存储，天然适合 Flink / Spark Streaming 接入 实时数据管道、ETL

一句话记住 Kafka 的本质：Kafka 不是用来”发消息”的，是用来”记流水”的。

银行的转账流水是不可删除、可追溯的持久记录，而不是看完就扔的便签。Kafka 的每一条消息，都是这样的”流水账”。

2. Kafka 的核心应用场景

正是因为”分布式日志”这一定位，Kafka 在以下场景中几乎是不可替代的选择：

flowchart LR
    KAFKA[Kafka]

    KAFKA --> S1["日志采集\n应用日志 / 访问日志\n集中写入 Kafka\n再由 ELK 消费分析"]
    KAFKA --> S2["用户行为埋点\n点击 / 浏览 / 搜索事件\n实时流入 Kafka\n推荐系统 & BI 各自消费"]
    KAFKA --> S3["数据管道（ETL）\n数据库 binlog 写入 Kafka\n下游数仓 / 搜索引擎同步"]
    KAFKA --> S4["实时流处理\n配合 Flink / Spark Streaming\n做实时聚合、异常检测"]

Kafka 不擅长什么：Kafka 并不适合对消息延迟极敏感（微秒级）、需要复杂路由规则、或消息量很小但可靠投递要求极高的场景——这些是 RabbitMQ 的主场。选型时回到上一章的决策框架，按业务诉求来判断。

二、核心架构

1. Topic 与 Partition

在上一节中我们了解到，Kafka 本质上是一个分布式日志系统。那么这个”日志”在物理上是如何组织存储的？又是如何支撑百万级/秒吞吐量的？这就要从 Kafka 最核心的两个概念说起：Topic 和 Partition。

一个 Topic 是消息的逻辑分类，比如 order-events、user-logs。但如果一个 Topic 的所有消息都存储在单台机器上，吞吐量天然受限于单机磁盘 I/O，无法水平扩展。

Kafka 的解法是：将一个 Topic 切分为多个 Partition（分区），每个 Partition 是一个独立的、有序的、不可变的消息序列，分散存储在不同的 Broker 节点上。

flowchart TB
    T["Topic: order-events"]

    T --> P0["Partition 0\nmsg0 → msg1 → msg2"]
    T --> P1["Partition 1\nmsg3 → msg4 → msg5"]
    T --> P2["Partition 2\nmsg6 → msg7 → msg8"]

    P0 --> B1["Broker 1"]
    P1 --> B2["Broker 2"]
    P2 --> B3["Broker 3"]

分区带来三个核心能力：

能力 说明
水平扩展 单机撑不住时，增加 Partition 并分散到更多 Broker，线性提升吞吐
并行消费 消费者组内多个消费者各自负责不同分区，同时消费，整体消费速度成倍提升
局部有序 同一 Partition 内消息严格 FIFO；同一业务 Key 的消息路由到同一 Partition，保证该 Key 下消息有序

关键约束——Partition 数是消费者并行度的上限：

同一个消费者组内，一个 Partition 只能被一个消费者消费，但一个消费者可以消费多个 Partition。

实际推论：

  • Topic 有 6 个 Partition，启动 4 个消费者 → 部分消费者消费多个 Partition，都能被分配到任务
  • Topic 有 6 个 Partition，启动 8 个消费者 → 多出的 2 个消费者永远空闲，是资源浪费
  • Topic 有 3 个 Partition，即使启动 100 个消费者，也只有 3 个在工作

结论：消费者实例数超过 Partition 数量时，多余的消费者不会提升消费速度。规划 Partition 数量时要考虑好预期的最大消费者并发数。

2. Partition 的物理存储

了解了 Partition 的逻辑概念后，再深入一层：Partition 在磁盘上是如何存储的？为什么 Kafka 能在顺序写磁盘的前提下还能快速定位任意 offset 的消息？

2.1 目录与 Segment 文件结构

Partition 在磁盘上对应一个目录，目录内由多个 Segment（段） 文件组成。每个 Segment 由三个文件构成一组，文件名就是该 Segment 起始 offset（左填零至 20 位）：

三个文件各司其职：

文件 内容 作用
.log 消息的实际二进制数据 顺序追加写入，顺序读取
.index offset → 物理文件位置的稀疏映射 根据 offset 快速定位消息
.timeindex timestamp → offset 的稀疏映射 根据时间戳快速定位消息

2.2 Segment 的写入与切分机制

在任意时刻，每个 Partition 只有一个 Active Segment接受写入，所有新消息以 Append 方式追加到该 Segment 的 .log 文件末尾。当 Active Segment 达到切分条件时，Kafka 会关闭它并创建新的 Segment。

flowchart LR
    MSG[新消息到来] --> ACTIVE["写入 Active Segment\n（当前活跃段，末尾追加）"]

    ACTIVE --> CHECK{触发切分条件？}
    CHECK -->|"大小超过 log.segment.bytes\n默认 1GB"| ROLL[关闭当前 Segment\n创建新 Segment]
    CHECK -->|"时间超过 log.roll.ms\n默认 7 天"| ROLL
    CHECK -->|未触发| ACTIVE

    ROLL --> CLEAN{触发日志清理？}
    CLEAN -->|按时间保留\n超过 log.retention.hours| DELETE[删除旧 Segment]
    CLEAN -->|按大小保留\n超过 log.retention.bytes| DELETE
    CLEAN -->|Compaction 策略| COMPACT[同 Key 只保留最新值]

关键点：

  • 关闭的 Segment 不可变（Immutable）——不再接受写入，只服务于消费者的读取。不可变性带来两个好处：无需加锁即可并发读取；便于操作系统做 Page Cache 优化。
  • 写入永远是追加（Append-Only），读取大多是顺序扫描——这正是 Kafka 能将磁盘 I/O 全部转化为顺序读写的关键。

2.3 消息定位：两级查找

现在来回答开头的问题：如何快速定位任意 offset 的消息？Kafka 采用两级查找策略——先定位 Segment，再在 Segment 内定位消息。

第一级：定位 Segment

每个 Segment 文件名就是其起始 offset。查找 offset=500123 的消息时，对所有 Segment 文件名做二分搜索，找到起始 offset ≤ 500123 的最大 Segment（即 00000000000000500000）。

第二级：Segment 内定位——稀疏索引

Kafka 的 .index 文件不是对每条消息建立一条索引记录（稠密索引），而是每隔一定字节数（由 log.index.interval.bytes 控制，默认 4KB）记录一个 offset → 物理文件位置的映射。

        I100["offset=500100 → pos=48000"]
        I200["offset=500100 → pos=52800"]

2.4 为什么这套设计能实现高吞吐

将上面的机制综合来看，Kafka 的物理存储设计围绕一个核心原则：将随机 I/O 转化为顺序 I/O + 内存操作。

操作 实际 I/O 模式 为什么快
写入消息 顺序追加写 .log 文件 磁盘顺序写吞吐接近内存（机械盘约 600MB/s）
定位 Segment 内存中的二分搜索 文件名列表常驻内存
定位 offset 内存中的二分搜索 .index 文件极小，通过 mmap 映射到 Page Cache
读取消息 从定位点开始顺序扫描 顺序读 + Page Cache 命中率高
发送给消费者 sendfile 零拷贝 数据从 Page Cache 直接到网卡，不经过用户态

这是 Kafka 高吞吐的底层秘密之一：用稀疏索引、Segment 分段、顺序 I/O 三者配合，在廉价的机械硬盘上也能跑出极高的吞吐量。

关于顺序写、Page Cache 和零拷贝在操作系统层面的底层原理，第五章第 1 节 会做更深入的分析。

3. 副本机制

单个 Partition 只存在于一台 Broker 上，如果这台机器宕机，数据就丢了。Kafka 通过多副本（Replica）机制解决这个问题——每个 Partition 可以配置 N 个副本，分布在不同的 Broker 上，其中一个副本是 Leader，其余是 Follower。

3.1 读写模型

flowchart LR
    subgraph "Partition 0 副本分布"
        direction TB
        L["Leader\nBroker 1\n（承接所有读写）"]
        F1["Follower\nBroker 2\n（从 Leader 同步）"]
        F2["Follower\nBroker 3\n（从 Leader 同步）"]
    end

    PRODUCER[Producer] -->|写入| L
    CONSUMER[Consumer] -->|读取| L
    L -->|Pull 同步| F1
    L -->|Pull 同步| F2

    F1 -.->|同步落后超过阈值\n被踢出 ISR| ISR_OUT["移出 ISR"]

Follower 不承接任何客户端请求，只负责从 Leader 主动拉取（Pull） 数据并在本地持久化，以此保持与 Leader 的同步。

为什么不让 Follower 分摊读请求？

这是 Kafka 在吞吐量与实现复杂度之间的主动取舍：

  • 允许读 Follower 的问题：Follower 数据天然落后于 Leader，直接读取会破坏”读己所写”等一致性保证，需要额外的协调机制，实现复杂且引入延迟。
  • 只读 Leader 的好处：实现简单，语义清晰。集群层面的负载均衡通过将不同 Partition 的 Leader 分散到不同 Broker 来实现，而非在单个 Partition 内部分摊。

Kafka 2.4 后引入了 Follower Fetch，但其目标场景是跨数据中心（让同机房的 Consumer 读同机房的 Follower，降低跨机房流量），并非通用的读负载均衡。

3.2 ISR

并非所有 Follower 都能实时跟上 Leader 的写入速度。Kafka 用 ISR（In-Sync Replicas，同步副本集合） 来区分”跟得上”和”跟不上”的副本：

flowchart LR
    subgraph "Partition 0 副本分布"
        direction TB
        L["Leader\nBroker 1"]
        F1["Follower\nBroker 2\n"]
        F2["Follower\nBroker 3\n"]

    L -->|"及时拉取\n同步延迟 < replica.lag.time.max.ms"| F1
    L -->|"同步滞后超过阈值\n默认 30s"| F2

    F2 -.->|同步落后超过阈值\n被踢出 ISR| OSR["OSR（Out-of-Sync）\nBroker 3"]

ISR 的组成与变化规则：

  • ISR 包含 Leader 本身以及所有在 replica.lag.time.max.ms（默认 30s）内向 Leader 发送过拉取请求的 Follower
  • Follower 同步滞后超过阈值 → 踢出 ISR，降为 OSR（Out-of-Sync Replica）
  • Follower 恢复同步后 → 重新加入 ISR

3.3 ISR 为什么是 Leader 选举的安全边界

ISR 最关键的作用不是监控，而是约束 Leader 选举的范围：

Kafka 的 Leader 选举只在 ISR 内进行。

这条规则保证了：新 Leader 一定拥有所有已提交（committed）的消息，不会发生数据回退（即消费者已消费的消息在新 Leader 上消失）。

反过来想：如果允许 OSR 成为新 Leader，会发生什么？

只从 ISR 选举，就从根本上杜绝了这个问题。

ISR 为空怎么办？

极端情况下，所有 Follower 都滞后，ISR 只剩 Leader，此时 Leader 再宕机，ISR 就彻底为空。Kafka 提供了 unclean.leader.election.enable 参数来控制这种情况下的行为：

  • false（默认）：宁可 Partition 不可用，也不选 OSR 当 Leader → 优先保证一致性
  • true：允许 OSR 当选，Partition 恢复可用，但可能丢数据 → 优先保证可用性

这正是经典的 CAP 取舍在 Kafka 中的具体体现。

4. Kafka 集群的元数据管理

前几节解决了消息的存储与可靠性问题。但集群正常运转还需要另一类信息：集群中哪些 Broker 在线、每个 Partition 的 Leader 是哪个 Broker、Topic 有哪些配置……这些统称为元数据。元数据的管理方式，在 Kafka 的发展历史上经历了一次重要的架构演进。

4.1 旧方案：依赖外部 ZooKeeper（Kafka 2.x 及以前）

早期 Kafka 将元数据管理完全外包给 ZooKeeper——一个独立的分布式协调服务。所有 Broker 向 ZooKeeper 注册自身、上报心跳、参与 Leader 选举，Consumer 的早期版本甚至将消费位移（offset）也存在 ZooKeeper 中。

flowchart LR
    ZK["ZooKeeper 集群\n（独立维护）"]
    B1[Broker 1] <-->|注册 / 心跳 / 选举| ZK
    B2[Broker 2] <-->|注册 / 心跳 / 选举| ZK
    B3[Broker 3] <-->|注册 / 心跳 / 选举| ZK
    C[Consumer] -->|早期版本：offset 存 ZK| ZK

这个方案能用，但带来了明显的运维负担：部署一套 Kafka，就必须同时维护一套 ZooKeeper 集群。两个系统的版本兼容性、监控告警、故障排查都需要独立处理，学习成本和运维成本双双翻倍。

此外，ZooKeeper 本身也存在性能瓶颈——当集群规模扩大、元数据变更频繁时，ZooKeeper 会成为瓶颈，制约 Kafka 的水平扩展能力。

4.2 新方案：内置 KRaft 协议（Kafka 3.x+）

Kafka 3.x 引入了自研的 KRaft（Kafka Raft） 共识协议，将元数据管理彻底内化。核心思路是：用 Kafka 自己的日志来存储元数据。

具体来说，集群内部有一个特殊的内部 Topic __cluster_metadata，专门的 Controller 节点通过 Raft 协议在多个 Controller 之间保持该 Topic 的一致性，从而替代 ZooKeeper 承担所有元数据管理职责。

与 ZooKeeper 方案相比，KRaft 带来 了几个关键改进：

对比维度 ZooKeeper 方案 KRaft 方案
外部依赖 需要独立维护 ZooKeeper 集群 无外部依赖，Kafka 自闭环

元数据传播 Controller 需将元数据推送给所有 Broker Broker 主动从 Controller 拉取，效率更高

4.3 对开发者的实际影响

架构层面的变化，对日常开发的影响其实很小：

  • 代码层面：Producer / Consumer 的 API 完全不变，业务代码无需任何改动
  • 部署层面：新项目直接选用 KRaft 模式，省去搭建和维护 ZooKeeper 的成本
  • 迁移层面：存量项目可通过官方迁移工具从 ZooKeeper 模式平滑切换到 KRaft 模式，不需要重建集群

Kafka 4.0 已正式移除对 ZooKeeper 模式的支持，KRaft 成为唯一选项。如果你在新项目中使用 Kafka，直接用 KRaft 模式即可，无需再考虑 ZooKeeper。

三、Producer 生产者

前两章从集群视角理解了 Kafka 的整体架构——Topic 如何分区、Partition 如何存储、副本如何保证可靠性。现在视角切换到客户端：当我们在应用代码里调用 producer.send() 时，消息究竟=经历了什么？

1. 发送流程全景

为什么要有累加器（RecordAccumulator）？

如果每条消息都立即发起一次网络请求，在高吞吐场景下网络开销会成为瓶颈。累加器的作用是在内存中攒够一批消息后批量发送，用少量延迟换取大幅提升的吞吐量——这是 Kafka 高吞吐设计的核心手段之一，与第二章中 Broker 侧顺序写磁盘的设计思路一脉相承：尽可能把小 I/O 合并成大 I/O。

什么时候需要自定义分区器？

默认的哈希分区覆盖了绝大多数场景。只有存在特殊路由需求时，才考虑实现 Partitioner 接口，例如：

  • VIP 用户的订单路由到高性能 Broker 上的 Partition，普通用户走普通 Partition

自定义分区器需注意哈希倾斜问题：某些 Key 的消息量远多于其他 Key，会导致分区负载不均。

2.2 无 Key 消息的分区演进

上面说”无 Key 时轮询”，这是 Kafka 2.4 之前的行为。从 Kafka 2.4 开始，DefaultPartitioner 对无 Key 消息改为了 Sticky Partitioner（粘性分区），行为发生了本质变化：

纯轮询看起来更均匀，为什么反而要改掉它？原因出在批处理效率上。

回顾 1.1 节的发送流水线：消息写入 RecordAccumulator 后，要等 batch 填满或 linger.ms 超时才会发送。纯轮询将消息分散到所有分区，每个分区的 batch 都填得很慢，结果是大量 batch 靠 linger.ms 超时触发发送，batch 普遍偏小。

Sticky 策略的改进思路是：先把一个分区的 batch 填满，再切换到下一个分区。消息集中灌入同一分区，batch 填满更快，既减少了网络请求次数，也降低了实际发送延迟：

性能对比（同一 Producer 配置，100 万条无 Key 消息）：

指标 纯轮询（2.3-） Sticky（2.4+）
网络请求数 ~2000 次 ~500 次
平均 batch 大小 ~8KB ~32KB
平均延迟 ~12ms ~5ms
分区负载均衡 完全均匀 宏观均匀（batch 内集中）

升级或使用时需要注意的几个点：

  • 极致低延迟场景：若设置 linger.ms=0，Sticky 会持续粘在一个分区直到 batch 填满才切换，在消息量较低时反而可能引入额外等待。此时可显式指定 RoundRobinPartitioner 退回轮询行为。
  • 负载监控误判：Sticky 在单个 batch 发送前，微观上分区负载不均匀，不要误判为”分区倾斜”——拉长时间窗口观察，整体是均匀的。
  • 从 2.3 升级到 2.4+：无 Key 消息的分布模式会改变，但由于无 Key 消息本来就不保证顺序，业务层面无感知。

3. 可靠性配置

分区策略解决了消息”去哪里”的问题，接下来面对更关键的问题：消息发出去之后，怎么保证它不会丢？

Kafka Producer 的可靠性由三个配置共同决定：acks 控制写入确认的标准，retries 控制失败后的重试行为，enable.idempotence 防止重试带来的重复写入。三者缺一不可。

3.1 acks：写入确认标准

acks 是 Producer 最重要的配置，直接决定”消息到底会不会丢”。它控制的是：Broker 在什么条件下才向 Producer 返回成功确认。

acks 值 含义 丢消息风险 吞吐量 适用场景
0 Producer 不等 Broker 确认就认为成功 高 最高 日志采集、监控指标（丢几条无所谓）
1 Leader 写入即成功，不等 Follower 同步 中等（Leader 宕机且 Follower 未同步则丢失） 高 一般业务，可接受少量丢失
all / -1 所有 ISR 副本都写入才成功 几乎不丢 较低 订单、支付等关键数据

acks=all 并非万无一失：如果其他 Follower 因同步慢被踢出 ISR，ISR 中只剩 Leader 自身，acks=all 就退化为 acks=1，依然有丢消息风险。解决方法是在 Broker 端配置 min.insync.replicas=2，强制要求 ISR 中至少有 2 个副本才允许写入——这才是真正可靠的组合。

3.2 重试与幂等

acks=all 保证了 Broker 成功写入，但网络是不可靠的——Broker 明明写入成功，ACK 却在回传途中丢失，Producer 会误判为失败并触发重试，导致同一条消息被写入两次。

这两个问题需要两个配置分别解决：

  • retries：控制发送失败后的重试次数，防止网络抖动导致消息丢失
  • enable.idempotence：开启幂等写入，Kafka 为每条消息分配唯一序号，Broker 自动去重，防止重试导致重复写入

注意：enable.idempotence=true 开启后，Kafka 会自动将 acks 强制设为 all、retries 强制设为 MAX。因此实践中直接开启幂等即可，无需重复配置前两项。

我们会在后面精确一次语义的章节，详细讲解幂等生产者的实现原理，以及它的适用范围和限制。

5. 精确一次语义

前面我们得到了一个「不丢不重」的基础配置：开启幂等生产者。这个配置解决了90%场景下的可靠性问题，但很多人并不清楚它到底是怎么工作的，以及它的边界在哪里。

5.1 幂等生产者

幂等的实现原理并不复杂：每个 Producer 在初始化时会被分配一个全局唯一的 Producer ID（PID），并为发往每个 Partition 的消息维护一个从 0 开始单调递增的序列号（Sequence Number）。Broker 会持久化记录每个 PID + Partition 组合收到的最大序列号，收到重复序列号时直接忽略并正常返回 ACK。

sequenceDiagram
    participant P as Producer(PID=5)
    participant B as Broker

    P->>B: 发送 msg (PID=5, Seq=37)
    B->>B: 写入成功，记录 PID=5 的最新 Seq=37
    B-->>P: ACK 在网络中丢失

    P->>B: 超时重试，再次发送 (PID=5, Seq=37)
    B->>B: 检测到 PID=5 的 Seq=37 已存在
    B-->>P: 返回成功 ACK，但不重复写入

理解了原理，幂等的边界也就自然清晰了：

是否保证 原因
同一 Producer 重试不重复 保证 PID + Seq 唯一标识每条消息
Broker 重启后去重状态不丢 保证 序列号持久化到磁盘
消息顺序不乱 保证 Broker 强制校验序列号连续性
Producer 重启后仍然去重 无法保证 重启后 PID 重置，旧的去重状态失效
跨 Partition 的原子写入 无法保证 每个 Partition 的序列号相互独立

最后两行的限制，就是事务生产者要解决的问题。

5.2 事务生产者

事务生产者在幂等的基础上，提供了两个更强的保证：

  • 跨 Partition 原子写入：多个 Partition 的写入要么全部提交，要么全部回滚，不会出现部分成功的中间状态
  • 跨 Session 持久性：通过全局唯一的 transactional.id，Producer 重启后可以延续上一个 Session 的事务状态，而不是重置

协调这一切的是 Broker 内部的 Transaction Coordinator，它负责记录事务状态，并在提交或回滚时向各个 Partition 写入事务标记。

    participant TC as Transaction Coordinator
    participant T1 as order-topic
    participant T2 as inventory-topic

    P->>TC: initTransactions()（注册事务 ID，恢复上次未完成的事务）
    P->>TC: beginTransaction()
    P->>T1: send(orderId, orderJson)
    P->>T2: send(skuId, deductionJson)

    alt 全部成功
        P->>TC: commitTransaction()
        TC->>T1: 写入 Commit 标记（消费者可见）
        TC->>T2: 写入 Commit 标记（消费者可见）
    else 发生异常
        P->>TC: abortTransaction()
        TC->>T1: 写入 Abort 标记（消费者不可见）
        TC->>T2: 写入 Abort 标记（消费者不可见）
    end

事务的性能代价：事务涉及与 Transaction Coordinator 的多次网络往返，吞吐量会明显下降。只在强一致性场景（订单、支付）中使用，不要在高吞吐的日志链路中引入事务。

四、Consumer 消费者

1. Consumer Group

在学习消费者的具体配置之前，必须先理解 Kafka 消费模型中最核心、也最容易混淆的概念——Consumer Group（消费者组）。

理解它最好的切入点是一句话：

同一个消费者组内：竞争消费（点对点模式）。不同消费者组之间：广播消费（发布-订阅模式）。

回想第二章介绍的两种通信模式——点对点和发布-订阅。Kafka 通过消费者组这一设计，用一套机制同时实现了两种模式，这正是 Kafka 设计上的精妙之处。

    T["Topic: order-events\n（6 个 Partition）"]

    subgraph GroupA["消费者组 A（实时处理服务"]
        CA1["Consumer A1\n负责 P0、P1"]
        CA2["Consumer A2\n负责 P2、P3"]
        CA3["Consumer A3\n负责 P4、P5"]
    end

    subgraph GroupB["消费者组 B（数据归档服务"]
        CB1["Consumer B1\n负责 P0、P1、P2"]

  • 组内：6 个 Partition 被 A 组的 3 个消费者瓜分，每条消息只被其中一个消费者处理——点对点
  • 组间：A 组和 B 组各自独立消费全量消息，互不感知、互不影响——发布-订阅

关键推论：消费者组内的消费者实例数不应超过 Partition 数。多出来的消费者实例永远分不到 Partition，只能空闲等待，纯属资源浪费。在规划消费者组时，Partition 数量决定了消费并行度的上限。

2. Offset 管理

理解了消费者组的分工模式，接下来的问题是：每个消费者怎么知道”自己读到哪里了”？

Kafka 的答案是 offset（偏移量）。每个 Partition 中的消息都有一个单调递增的 offset，消费者通过记录”下一条要读的 offset”来追踪自己的消费进度。消息被消费后并不删除，offset 只是向前移动。

2.1 offset 存在哪里

存储方式 说明 特点
Kafka 内置 Topic（__consumer_offsets） 消费者将 offset 提交到这个内部 Topic，由 Kafka 统一管理 简单可靠，默认方式
外部存储（Redis / DB） 将 offset 与业务处理结果放在同一个存储中管理 可实现端到端原子性，但实现复杂

为什么会有”外部存储 offset”这种方案？

内置存储有一个天然的局限性：offset 提交和业务数据处理是两套独立的存储系统。假设消费者处理完消息、写入数据库成功，但提交 offset 到 Kafka 前崩溃了——重启后会重复消费同一条消息，再次写入数据库，产生脏数据。

外部存储方案的核心思路是把”写业务数据”和”记录 offset”放在同一个数据库事务里，利用关系型数据库的 ACID 保证两者原子完成。这是追求精确一次（Exactly-Once）消费的标准做法。代价也很明显：你需要自己管理 offset 的存储和清理，失去了 Kafka 原生消费者组的便利性。

正常情况下，消费者按部就班地消费、提交、前进。但生产环境中有三个边界场景必须提前理解。

2.2 自动起点策略

当一个消费者组首次启动（Kafka 没有这个组的 offset 记录），或者已提交的 offset 对应的消息已被 Retention 清理时——总之，Kafka 找不到一个有效的 offset——auto.offset.reset 决定了消费者的行为：

两种触发场景、三种取值，一张图说清楚：

flowchart TD
    CONSUMER[Consumer 请求消息]
    CONSUMER --> CHECK{已有有效的\n已提交 offset？}
    CHECK -->|有| NORMAL[从已提交 offset 继续消费]
    CHECK -->|无| TRIGGER{"原因？"}

    TRIGGER -->|"首次启动\n（新消费者组）"| RESET{auto.offset.reset}
    TRIGGER -->|"offset 越界\n（消息已被 Retention 清理）"| RESET

    RESET -->|earliest| E["从 Partition 最早可用的 offset 开始\n全量消费历史消息"]
    RESET -->|latest| L["从 Partition 最新 offset 开始\n只消费新到达的消息"]
    RESET -->|none| N["抛出异常\n消费进程启动失败，等待人工介入"]
值 行为 适用场景 风险提示
earliest 从 Partition 最早的消息开始消费 数据回溯、新服务首次部署需全量数据 消息堆积量大时可能消费很久才能追上最新
latest 从最新位置开始，只消费新到达的消息 只关心实时数据、历史数据无需处理 旧消息被跳过，无法恢复（除非手动 seek）
none 不自动选择，直接抛异常 严格要求必须从已有 offset 恢复 首次启动或 offset 丢失时直接报错

none 的实际用途：在金融、支付等强一致性场景中，如果你不确定 offset 从哪开始是安全的（earliest 可能导致大量重复，latest 可能导致数据丢失），最佳策略是让进程直接失败并触发告警，由人工确认后决策。出问题不可怕，可怕的是出了问题上层还不知道。

2.3 offset 越界

2.2 中提到了 offset 越界时 auto.offset.reset 的自动恢复行为。但在生产环境中，你不应该只依赖这个兜底机制——它只是在出事之后被动响应。更重要的课题是：如何预防 offset 越界，以及当真越界了如何最小化损失。

此时 Kafka 会自动回退到 auto.offset.reset，但后果取决于你的配置：

  • earliest：从 offset=5000 恢复，offset 1000~4999 的消息永久丢失（对你来说，不是对 Kafka 来说）
  • latest：从最新位置恢复，offset 1000 至今的消息全部跳过
  • none：进程终止，由人工评估后决定——是从 earliest 补数据，还是 seek 到指定时间点

预防 offset 越界的三条最佳实践：

  1. 监控消费 Lag：对 consumer_lag 设置分级告警——Lag 超过 1h 发通知，超过 Retention 的 50% 发紧急告警。越界不是瞬间发生的，Lag 持续增长就是前兆
  2. Retention 留足安全余量：如果业务允许消费者最长停机 3 天，Retention 至少配置 7 天。2x 原则：Retention ≥ 2 × 最大允许停机时间
  3. 关键业务使用 none + 告警：offset 越界时让进程失败，由人工决定是手动 seek 回退还是从 earliest 补数据——宁可停服等人工，不要静默丢数据

2.4 手动控制消费位置：consumer.seek()

auto.offset.reset 解决的是”找不到 offset 时自动怎么办”，但有时候你需要的不是自动策略，而是精确控制从哪个 offset 开始消费。常见场景：

场景 说明
消息回溯 昨天的业务逻辑有 Bug，修复后 seek 到昨天 00:00 的 offset，重新处理全天数据
跳过坏消息 某条消息格式损坏导致消费卡住，确认无法修复后 seek 跳过它
数据修复 上游系统修复了数据，seek 到指定时间范围重新消费以更新下游
消费进度迁移 将消费进度从一个消费者组迁移到另一个

offsetsForTimes() 的注意事项：

  • 返回的是大于等于指定时间戳的第一个 offset，不是精确匹配
  • 如果指定时间戳比 Partition 中最新的消息还新，返回值为 null，seek 前必须判空
  • 时间戳的精度取决于消息写入时的时间戳类型（CreateTime vs LogAppendTime），详见 2. Partition 的物理存储

3. 自动提交 vs 手动提交

offset 的提交时机直接决定了消息是否会被重复消费或丢失，是 Consumer 配置中最需要谨慎对待的部分。

3.1 自动提交

开启自动提交后，Kafka 客户端会在后台按固定时间间隔提交 offset，业务代码无需关心提交逻辑：

    participant B as Broker

4. Rebalance

消费者组在运行过程中并不是静止的——消费者实例会动态加入或退出，Topic 的分区数也可能变化。每当这些情况发生，Kafka 需要重新决定”哪个消费者负责哪些 Partition”，这个过程就是 Rebalance（再均衡）。

4.1 什么会触发 Rebalance

消费者组有一个 Group Coordinator（由某个 Broker 担任），负责监控组内所有消费者的存活状态。以下四种情况会触发 Rebalance：

    T1["新增消费者实例\n（扩容）"] --> R
    T2["消费者实例宕机\n（缩容）"] --> R
    T3["Topic 的 Partition 数变化"] --> R
    T4["消费者处理时间超过\nmax.poll.interval.ms\n被认定为假死"] --> R

    R["触发 Rebalance\n重新分配 Partition"]

在深入 Rebalance 流程之前，需要先弄清楚一个前置问题：Group Coordinator 是怎么确定的？

4.2 Group Coordinator 的定位

Kafka 不会为每个消费者组单独分配一个协调节点，而是复用了内部 Topic __consumer_offsets 的 Leader 机制：

flowchart LR
    G1["Group: order-processor\nhash(group.id) % 50 = 12"]
    G2["Group: log-collector\nhash(group.id) % 50 = 37"]

    G1 -->|"__consumer_offsets-12"| P12["Partition 12\nLeader: Broker 3"]
    G2 -->|"__consumer_offsets-37"| P37["Partition 37\nLeader: Broker 1"]

    P12 --> GC1["Group Coordinator\n（Broker 3）"]
    P37 --> GC2["Group Coordinator\n（Broker 1）"]

这个设计有两个推论值得记住：

  • Coordinator 自动容灾：Coordinator 所在 Broker 宕机时，__consumer_offsets 对应 Partition 的 Leader 会自动切换，新的 Leader Broker 自动接管 Coordinator 职责，无需人工干预。
  • Coordinator 天然分散：不同消费者组的 Coordinator 分散在不同 Broker 上，不存在单点压力。

4.3 Rebalance 的执行流程：JoinGroup → SyncGroup

确定了 Coordinator 之后，来看 Rebalance 内部的执行过程。整个协议分为两个阶段：

Phase 1 — JoinGroup：所有消费者向 Coordinator 申请加入，Coordinator 从中选出 Group Leader（通常是第一个发送 JoinGroup 请求的消费者），并将全部成员列表下发给它。

Phase 2 — SyncGroup：Group Leader 在本地计算分配方案，通过 SyncGroup 请求提交给 Coordinator；其他成员也发送 SyncGroup 请求（不含方案），等待 Coordinator 将 Leader 的分配结果下发给所有人。

sequenceDiagram
    participant C1 as Consumer 1（组内已有）
    participant C2 as Consumer 2（组内已有）
    participant C3 as Consumer 3（新加入）
    participant GC as Group Coordinator

    C3->>GC: ① JoinGroup 请求

    GC->>C1: ② 心跳响应携带 REBALANCE_IN_PROGRESS
    GC->>C2: ② 心跳响应携带 REBALANCE_IN_PROGRESS
    Note over C1,C2: 停止消费，准备重新加入

    C1->>GC: ③ JoinGroup 请求
    C2->>GC: ③ JoinGroup 请求

    GC->>GC: ④ 选出 Group Leader（C1）

    GC-->>C1: ⑤ JoinGroup 响应：你是 Leader，成员列表为 [C1,C2,C3]
    GC-->>C2: ⑤ JoinGroup 响应：Leader 是 C1，等待分配结果
    GC-->>C3: ⑤ JoinGroup 响应：Leader 是 C1，等待分配结果

    C1->>C1: ⑥ 执行分区分配算法

    C1->>GC: ⑦ SyncGroup 请求（含分配方案）
    C2->>GC: ⑦ SyncGroup 请求（不含方案，等待结果）
    C3->>GC: ⑦ SyncGroup 请求（不含方案，等待结果）

    GC-->>C1: ⑧ 下发分配结果：C1 → P0、P1
    GC-->>C2: ⑧ 下发分配结果：C2 → P2、P3
    GC-->>C3: ⑧ 下发分配结果：C3 → P4、P5

    Note over C1,C3: Rebalance 完成，各自开始消费分配到的 Partition

这个两阶段设计有两个关键点：

设计点 说明
Group Leader 负责分配 Coordinator 不执行分配算法，只负责收集和下发。分配逻辑完全由客户端控制（通过 partition.assignment.strategy），Broker 保持轻量
SyncGroup 的同步屏障 所有成员必须都发送 SyncGroup 请求后，Coordinator 才统一下发结果，保证全员在同一时刻切换到新的分配方案

4.4 Rebalance 的代价：Stop-The-World

传统的 Eager Rebalance 协议（默认行为）有一个严重问题：Rebalance 期间所有消费者全部暂停消费，无论其负责的 Partition 是否发生了变化。

sequenceDiagram
    participant C1 as Consumer 1（P0 P1）
    participant C2 as Consumer 2（P2 P3）
    participant C3 as Consumer 3（新加入）
    participant GC as Group Coordinator

    Note over C1,C2: 正常消费中...
    C3->>GC: 加入消费者组

    GC->>C1: 触发 Rebalance，停止消费
    GC->>C2: 触发 Rebalance，停止消费
    Note over C1,C3: 全员暂停，等待重新分配...

    GC->>C1: 分配 P0、P1
    GC->>C2: 分配 P2、P3
    GC->>C3: 分配 P4、P5

    Note over C1,C3: 恢复消费

C1 和 C2 负责的分区根本没有变化，却仍然被迫停止消费——这就是 Stop-The-World 效应。如果 Rebalance 频繁发生，系统会持续处于”暂停→恢复→暂停”的状态，消息积压随之而来。

4.5 减少 Rebalance 的策略

面对 Rebalance 的代价，有两个层面的应对手段：减少 Rebalance 的发生频率，以及降低单次 Rebalance 的影响范围。

策略一：避免不必要的 Rebalance

最常见的 Rebalance 诱因是消费者被误判为”假死”——处理逻辑太慢，超过了 max.poll.interval.ms，Coordinator 认为它已经挂掉。

策略二：使用 Cooperative Rebalance 减少停顿

Kafka 2.4+ 引入的 CooperativeStickyAssignor 将”全量停止重分配”改为”增量局部重分配”——只有分配发生变化的 Partition 才会短暂停止，其余 Partition 的消费完全不受影响：

sequenceDiagram
    participant C1 as Consumer 1（P0 P1）
    participant C2 as Consumer 2（P2 P3）
    participant C3 as Consumer 3（新加入）
    participant GC as Group Coordinator

    Note over C1,C2: 正常消费中...
    C3->>GC: 加入消费者组

    GC->>C1: 只撤回 P1
    Note over C1: 继续消费 P0，仅暂停 P1
    Note over C2: 完全不受影响，继续消费 P2 P3

    GC->>C3: 分配 P1
    Note over C1,C3: 消费恢复，全程只有 P1 短暂停止

对比维度 Eager（传统） Cooperative（2.4+）

五、Broker 端核心机制

前几章分别从 Broker 集群架构、Producer、Consumer 三个视角理解了 Kafka 的工作方式。本章回到 Broker 内部，深入两个支撑 Kafka 高吞吐与高可靠的底层机制：高性能存储的 OS 原理，以及多副本一致性的保障机制。

1. 高性能存储的底层原理

在 第二章 2.4 节 中，我们从物理存储设计的角度总结了 Kafka 高吞吐的原因——顺序追加写、稀疏索引、Page Cache、零拷贝。但这些结论背后的操作系统层面原理是什么？为什么顺序写能比随机写快几个数量级？Page Cache 相比 JVM 堆内存到底好在哪里？零拷贝节省的两次 CPU 拷贝发生在哪里？

本节从 OS 底层逐一展开这三个机制，理解它们之间的递进关系：顺序写解决写入性能，Page Cache 解决读取性能，零拷贝解决传输性能。

1.1 顺序写

随机写慢的根本原因是磁头寻道——每次写入前需要物理移动磁头到目标磁道，机械运动耗时在毫秒级。顺序写则是一次寻道后连续写入，磁头几乎不需要移动：

flowchart LR

        direction LR
        RW1["seek 到指定位置"] --> RW2["写入消息"] --> RW3["seek 到下一位置"]
    end

    subgraph RIGHT["顺序追加（Kafka）"]
        direction LR
        SW1["直接在文件末尾追加消息"] --> SW2["更新 offset 索引"]
    end

两种模式在性能上的差距是数量级级别的：

操作类型 速度
磁盘随机写 ~100 次/秒
磁盘顺序写 ~100MB/s+
内存随机写 ~100,000 次/秒

一个反直觉的事实：Kafka 在廉价机械硬盘上的顺序写速度，与千兆网卡的带宽量级相当。这意味着 Kafka 的性能瓶颈往往不在磁盘写入，而在网络传输——这也是 Kafka 引入零拷贝的根本动因。

1.2 Page Cache

顺序写解决了写入性能，但读取呢？如果每次消费都直接读磁盘，仍然是瓶颈。Kafka 的解法是完全依赖 OS Page Cache，而非在 JVM 堆内存中自建缓存：

flowchart TD
    subgraph WRITE["写入路径"]
        P[Producer] -->|写消息| PC["Page Cache（内存）"]
        PC -->|立即返回 ACK| P
        PC -.->|OS 异步刷盘| DISK[(磁盘)]
    end

    subgraph READ["读取路径"]
        C[Consumer] -->|读消息| PC2["Page Cache（内存）"]
        PC2 -->|大概率命中缓存，直接返回| C
        PC2 -.->|缓存未命中，从磁盘加载| DISK2[(磁盘)]
    end

在 Kafka 的典型部署场景中，Consumer 消费的消息往往是 Producer 刚写入的”热数据”，它们大概率还在 Page Cache 中，读取直接命中内存，根本不触发磁盘 I/O。

为什么不用 JVM 堆内存做缓存？

对比维度 JVM 堆内存缓存 OS Page Cache
GC 影响 大对象频繁分配，可能触发 Full GC，造成毫秒级停顿 由 OS 管理，完全不受 JVM GC 影响
进程重启 缓存随进程消失，重启后需要预热 缓存在 OS 层，Kafka 重启后仍然有效
数据拷贝 数据需从内核态拷贝到用户态（JVM 堆） 数据留在内核态，可直接配合零拷贝发送

最后一点是关键：Page Cache 中的数据可以通过 sendfile 系统调用直接传输到网卡，完全不经过用户态。这就是下一节零拷贝的基础。

1.3 零拷贝（Zero-Copy）

有了 Page Cache，数据已经在内存中了。但传统的发送流程仍然低效——数据需要在内核态和用户态之间来回拷贝：

flowchart 
    subgraph NORMAL["传统方式（4 次拷贝，2 次上下文切换）"]
        D1[(磁盘)] -->|"① DMA 拷贝"| KC1["内核缓冲区\n（Page Cache）"]
        KC1 -->|"② CPU 拷贝\n（内核→用户态）"| APP["用户态（Kafka JVM）"]
        APP -->|"③ CPU 拷贝\n（用户态→内核）"| KC2["Socket 缓冲区"]
        KC2 -->|"④ DMA 拷贝"| NET[网卡]
    end
flowchart 
    subgraph ZERO["零拷贝（sendfile，2 次拷贝，0 次上下文切换）"]
        D2[(磁盘)] -->|"① DMA 拷贝"| KC3["内核缓冲区\n（Page Cache）"]
        KC3 -->|"② DMA 拷贝\n（内核直接到网卡，跳过用户态）"| NET2[网卡]
    end

Kafka 通过 FileChannel.transferTo() 调用底层的 sendfile 系统调用，数据从 Page Cache 直接传到网卡，省去了 ② 和 ③ 两次 CPU 拷贝，以及内核态↔用户态的上下文切换。

1.4 三个机制的协同关系

回顾三个机制，它们并不是独立的优化点，而是一条完整的高性能数据链路：

三者缺一不可：顺序写让磁盘 I/O 不成为写入瓶颈，Page Cache 让大多数读操作留在内存，零拷贝让数据传输不消耗多余的 CPU。

2. ISR 与高水位

存储性能问题解决了，接下来是可靠性问题：在多副本架构下，如何保证消费者不会读到”尚未被足够多副本确认”的数据？这就是 High Watermark（高水位） 机制要解决的问题。

2.1 两个关键偏移量

理解高水位，先要理解两个概念：

  • LEO（Log End Offset）：某个副本下一条待写入消息的 offset，代表该副本”写到哪里了”
  • HW（High Watermark）：所有 ISR 副本中最小的 LEO，代表”所有 ISR 副本都已同步到哪里了”
flowchart LR
    subgraph P0["Partition 0（3 副本，均在 ISR 中）"]
        direction TB
        L["Leader（Broker 1）\n[m0][m1][m2][m3]\nLEO = 4"]
        F1["Follower（Broker 2）\n[m0][m1][m2]\nLEO = 3"]
        F2["Follower（Broker 3）\n[m0][m1]\nLEO = 2"]
        HW["HW = min(4, 3, 2) = 2\n\nm0、m1 对消费者可见\nm2、m3 对消费者不可见"]
    end

    L -.->|F1 从 Leader 拉取| F1
    L -.->|F2 从 Leader 拉取| F2

2.2 HW 的核心作用

HW 划定了一条可见性边界：HW 之前的消息才对消费者可见。这条规则是为了防止以下场景：

sequenceDiagram
    participant C as Consumer
    participant L as Leader（Broker 1）
    participant F as Follower（Broker 2）

    L->>L: 写入 m3（LEO=4）
    Note over F: m3 尚未同步到 Follower

    C->>L: 拉取消息
    L-->>C: 返回 m3（消费者读到了）

    Note over L: Leader 宕机
    Note over F: F 成为新 Leader，但没有 m3
    C->>F: 再次拉取，找不到 m3
    Note over C: 已消费的消息在新 Leader 上不存在，数据不一致

引入 HW 后，m3 在到达 HW 之前对消费者不可见，上述问题被彻底避免。

LEO vs HW vs acks=all 的关系澄清：

这三个概念容易混淆，但它们关注的是不同层面：

  • LEO：某个副本自己写到哪里了（副本视角）
  • HW：所有 ISR 副本都同步到哪里了，决定消费者可见性（Consumer 视角）
  • acks=all：Producer 等待所有 ISR 副本写入后才认为发送成功（Producer 视角）

acks=all 等待的是”各副本 LEO 推进”，HW 的推进是其自然结果——但两者是不同层面的机制，不要混为一谈。

2.3 HW 机制的缺陷

HW 解决了”消费者读到未同步数据”的问题，但它自身有一个经典缺陷：在 Leader 宕机切换时，Follower 仅依赖本地 HW 做日志截断，可能删除已被确认甚至已被消费的消息。

问题的根源在于 HW 的传播存在滞后——Leader 先推进自己的 HW，Follower 要等到下一轮 Fetch 响应才能得知新的 HW。在这个滞后窗口内发生 Leader 宕机，Follower 按自己落后的 HW 截断，就会出现问题：

sequenceDiagram
    participant L as Leader（Broker 1）
    participant F as Follower（Broker 2）
    participant C as Consumer

    Note over L: LEO=4，HW=2
    L->>F: 同步 m2（offset=2）、m3（offset=3）
    Note over F: 收到 m2、m3，LEO=4，但 HW 仍为 2（还未收到 HW 更新）

    F->>L: Fetch 确认收到 m2
    Note over L: 推进 HW=3（m2 已被所有 ISR 确认）
    L-->>C: HW=3，m2 对消费者可见

    Note over L: Leader 宕机，HW=3 的通知尚未发给 Follower

    Note over F: 被选为新 Leader
    Note over F: 按本地 HW=2 做日志截断
    Note over F: m2、m3 全部被删除
    Note over C: 已消费的 m2 在新 Leader 上消失，数据不一致

2.4 Leader Epoch

Kafka 0.11+ 引入了 Leader Epoch 来修复上述问题。核心思路是：用单调递增的纪元号替代 HW 作为日志截断的决策依据，将截断决策从”Follower 本地推测”升级为”向新 Leader 询问后精确执行”。

每次 Leader 切换，纪元号 +1，形成一条不可伪造的历史记录：

flowchart
    subgraph EPOCH["Leader Epoch 历史"]
        E0["Epoch 0\nLeader: Broker 1\noffset 0 ~ 3"]
        E1["Epoch 1\nLeader: Broker 2\noffset 4 ~ 7"]
        E2["Epoch 2\nLeader: Broker 1\noffset 8 ~ 11"]
        E0 --> E1 --> E2
    end

Follower 恢复时，不再自己猜截断点，而是主动询问新 Leader：

sequenceDiagram
    participant F as Follower（Broker 2）
    participant NewL as 新 Leader（Broker 3，Epoch 6）

    Note over F: 重启，本地记录：Epoch 5 最后 offset=95

    F->>NewL: 我的 Epoch 5 最后 offset=95，我应该截断到哪里？
    NewL->>NewL: 查询 Epoch 5 的合法范围：offset 0~92
    NewL-->>F: 截断到 offset=92

    Note over F: 精确截断到 92，不会误删 offset 92 之前已确认的消息

相比之下，纯 HW 截断会截到 offset=90（Follower 本地的 HW），把 offset 90~92 这段已被所有 ISR 确认的消息也一并删除。

对比维度 纯 HW 截断 Leader Epoch
截断依据 Follower 本地 HW（可能滞后） 向新 Leader 查询的精确 offset
决策方式 Follower 本地推测 全局共识
数据安全 Leader 切换时可能误删已确认消息 精确保证，不误删
引入版本 — Kafka 0.11+

一句话总结 Leader Epoch 的本质：HW 截断依赖 Follower 自己的视角，而 Follower 的视角天然滞后于 Leader。Leader Epoch 让 Follower 在截断前先问新 Leader，用全局视角替代本地视角，从根本上消除了滞后带来的不一致。

3. 日志清理

随着消息不断写入，磁盘终究会被占满。Kafka 提供了两种完全不同思路的清理策略，分别应对不同的业务诉求：

  • Retention（按时间/大小删除）：数据超过保留期限就删除，不关心 Key
  • Log Compaction（按 Key 去重）：只保留每个 Key 的最新值，不关心时间

3.1 Retention：按时间或大小删除

Retention 是最直观的清理方式，由后台线程定期检查每个 Segment，满足条件则删除：

flowchart TD
    CHECK{检查 Segment 是否过期}
    CHECK -->|"Segment 最后修改时间\n超过 log.retention.hours（默认 168h/7天）"| DELETE["删除整个 Segment 文件"]
    CHECK -->|"Partition 日志总大小\n超过 log.retention.bytes"| DELETE
    CHECK -->|"两者都未触发"| KEEP["保留，继续等待"]
    DELETE --> NOTE["删除单位是整个 Segment\n而非单条消息"]

为什么以 Segment 为粒度删除，而不是逐条删除？

逐条删除意味着在日志文件中间”挖洞”，需要随机写操作，破坏了 Kafka 顺序 I/O 的根基。删除整个 Segment 文件只是一次文件系统的 unlink 操作，极快且不影响其他 Segment 的读写——这与第二章介绍的”不可变 Segment”设计一脉相承。

3.2 Log Compaction：按 Key 去重保留最新值

Retention 策略的前提是”旧数据可以丢弃”。但有些场景截然不同：不需要完整的历史消息，只需要每个 Key 的最新状态。

以数据库变更事件为例：用户 user_1 的城市从”北京”改到了”深圳”。对于只关心当前状态的消费者，它不需要”北京”这条历史，只要最新的”深圳”就够了。Retention 会把所有历史消息都删掉，但 Log Compaction 会做得更精确——对同一个 Key 只保留最新的一条消息：

flowchart LR
    subgraph BEFORE["压缩前"]
        direction TB
        M1["offset=0\nKey: user_1\nvalue: 北京"]
        M2["offset=1\nKey: user_2\nvalue: 上海"]
        M3["offset=2\nKey: user_1\nvalue: 深圳（最新）"]
        M4["offset=3\nKey: user_3\nvalue: 广州"]
    end

    subgraph AFTER["压缩后"]
        direction TB
        N1["Key: user_1\nvalue: 深圳"]
        N2["Key: user_2\nvalue: 上海"]
        N3["Key: user_3\nvalue: 广州"]
    end

    BEFORE -->|"Compaction\n后台压缩线程"| AFTER

Log Compaction 有一个配套机制值得注意：Tombstone（墓碑消息）。如果想彻底删除某个 Key（而非保留最新值），可以发送一条 value 为 null 的消息。Compaction 线程会将其保留一段时间（由 delete.retention.ms 控制）后再彻底清除，确保所有消费者都有机会感知到这条”删除”事件，而不会在消费者读到之前就消失。

3.3 两种策略的选择

flowchart TD
    POLICY{业务诉求}
    POLICY -->|"只需保留最近一段时间的数据\n不关心 Key"| RET["delete（Retention）\n按时间或大小删除过期 Segment\n适合：日志采集、事件流"]
    POLICY -->|"只需每个 Key 的最新状态\n历史值可丢弃"| COMP["compact（Log Compaction）\n按 Key 去重，保留最新值\n适合：状态存储、CDC、配置分发"]
    POLICY -->|"时间窗口内保留全量\n超期后按 Key 去重"| BOTH["compact,delete（混合）\n适合：有合规要求的审计日志"]

典型应用场景对比：

场景 策略 说明
用户行为日志 delete 只保留最近 7 天，历史数据无需保留
数据库 CDC 变更 compact 每行只需最新状态，可从 offset=0 回放重建全量
服务配置分发 compact 消费者始终能读到最新配置，不需要历史版本
Kafka Streams 状态存储 compact KTable 只关心当前 Key 的最新 Value
审计日志（合规要求） compact,delete 保留 N 天全量，超期后按 Key 去重

3.2 发送方式

3.2.1 异步发送（推荐）

  • 原理：send() 立即返回 CompletableFuture，消息先放入生产者内存缓冲区，后台 I/O 线程批量发送，业务线程完全不阻塞。
  • 吞吐量：可达数万~数十万条/秒（配合 batch.size 和 linger.ms 调优）。
  • 回调线程：whenComplete 执行在 Kafka Producer 的 I/O 线程池中，切勿执行重发或阻塞操作，否则会影响整个 Producer 的网络线程，导致所有发送阻塞。
  • 失败处理最佳实践：依赖 Producer 内置重试（retries 配置），回调中仅记录日志、更新 Prometheus 计数器或写入一个非阻塞的队列，由独立线程重试。

3.2.2 同步发送

阻塞等待 Broker 确认后才返回，会严重拖低吞吐，只用于必须立即知道结果的 API 场景（如用户直接调用的 REST 接口）。

  • 风险：get() 会阻塞当前线程，若使用 Tomcat 等线程池会迅速耗尽请求线程，导致系统雪崩。
  • 超时控制：必须指定超时时间，否则可能永久阻塞。
  • 适用场景：仅当调用方必须等待发送结果（如同步的 RPC 响应接口）时使用，并需配合独立的线程池隔离或降级。
  • 性能：同步发送的单线程吞吐通常只有几百条/秒，远低于异步模式。

3.3 SendResult 与 RecordMetadata

3.3.1 概述

  • SendResult<K, V>：kafkaTemplate.send() 的返回结果（CompletableFuture 的完成值），封装了本次发送的完整信息。

3.4.4 Header 详解

  • 类型：org.apache.kafka.common.header.Header，实现类是 RecordHeader(key, value)。
  • Key：String，大小写敏感，建议使用 X- 前缀的习惯命名。
  • Value：byte[]，需自行编解码（常用 UTF-8 字符串、JSON 序列化或 Avro 等）。
  • 常见用途：
    • 分布式链路追踪（如 OpenTelemetry，传递 traceparent）
    • 事件类型、版本号、数据来源标识
    • 业务路由指令（消费者根据 Header 做出不同处理）
  • 数量与大小限制：
    • 无明确个数限制，但总 Headers 长度过大会降低性能，建议单个 Header value 不超过 1KB，总数不超过 10 个。
    • Kafka 消息的最大总大小受 Broker 配置 max.message.bytes 限制，Headers 也会计入总大小。

4.4 Acknowledgment——手动控制 Offset 提交

手动提交偏移是实现“至少一次”或“精确一次”语义的关键。

前提：必须设置 enable.auto.commit = false（Spring Boot 中默认 true，需显式关闭），否则手动 ack 无效。

4.4.1 ack-mode 取值

在 ContainerFactory 中设置 containerProperties.setAckMode(...)。

ack-mode 行为 适用场景
record 每条消息处理完后自动提交 不推荐，性能开销大
batch 每批 poll() 完成后自动提交 不推荐，可能重复消费
manual 调用 acknowledge() 后，异步提交（在下次 poll 前完成） 要求低延迟时可接受
manual_immediate 调用 acknowledge() 后，立即同步提交，阻塞直到 Broker 确认 生产环境强烈推荐
  • 为什么推荐 manual_immediate：同步提交能确保提交成功后再继续处理下一条，防止因提交失败导致的重复消费或丢失进度。
  • 手动 ack 调用时机：必须在消息处理成功后调用，且只能调用一次。

4.5 手动 ACK 的异常处理模式

处理失败时，正确区分可重试与不可重试异常，是避免死循环和消息丢失的核心。

最佳实践建议：

  • 定义自定义异常体系：NonRetryableException 和 RetryableException，避免直接依赖 JDK 标准异常。
  • 不可重试的消息必须调用 ack.acknowledge()，否则整个分区被阻塞，后续所有消息都无法被消费。
  • 可重试异常抛出后，交由 Spring 的 SeekToCurrentErrorHandler 或 DefaultErrorHandler 进行重试，通常配合 BackOff（固定间隔或指数退避）。
  • 配置死信主题（Dead Letter Topic），将最终无法处理的消息写入，方便人工介入。

6. 错误处理与重试

消息消费时如果抛出异常，Spring Kafka 提供了两种处理思路：

  • 阻塞重试：DefaultErrorHandler（在同一个消费者线程内重试，阻塞该分区）
  • 非阻塞重试：@RetryableTopic（将失败消息投递到独立的 retry Topic，主流程不受影响）

下面先介绍最常用的 DefaultErrorHandler。

6.1 DefaultErrorHandler 构造器

DefaultErrorHandler 负责消费异常的重试与死信处理。它会在消费者线程内阻塞重试，因此重试期间该分区无法消费后续消息。

注意：若使用 DeadLetterPublishingRecoverer，需要为其提供 KafkaTemplate，并确保其序列化器与主消息兼容（通常用 String 或字节数组发送 DLT 消息）。

6.1.1 BackOff 策略

陷阱提醒：如果 maxAttempts 设得过大，且消息处理时间较长，可能导致 max.poll.interval.ms 超时，从而引发不必要的消费者组 Rebalance。务必确保重试总时长 < max.poll.interval.ms。

6.1.2 死信恢复器

重试耗尽后将消息写入死信 Topic（DLT），以便人工排查。

最佳实践：在死信 Header 中保留完整异常信息和原始位置，便于后续重放工具自动恢复。

6.2 addNotRetryableExceptions()——跳过无意义重试

对于反序列化失败、格式错误、永久性业务异常，重试无意义，应直接跳过或直接发送到死信，避免阻塞分区。

说明：addNotRetryableExceptions 中的异常不会被重试，会直接交给 Recoverer 处理（如写入死信）。若未配置 Recoverer，则会直接抛出，通常导致容器停止（需配合重启策略）。

6.3 绑定 ContainerFactory

需要将配置好的 DefaultErrorHandler 注入到容器工厂，以便对所有 @KafkaListener 生效。也可以在某个 ContainerFactory 中单独设置，仅影响使用该工厂的监听器。

注意：如果同时定义了 @RetryableTopic 和 DefaultErrorHandler，@RetryableTopic 会覆盖容器的错误处理器，不需要重复设置。

6.4 @RetryableTopic——非阻塞重试

DefaultErrorHandler 重试时会阻塞当前分区，导致高吞吐场景下消息积压。@RetryableTopic 将失败消息发送到专门的 retry Topic，由另一个消费者独立处理重试，主流程不阻塞。

代价与注意事项：

  • 框架自动创建 order-events-retry-0、order-events-retry-1 等额外 Topic，需要 Broker 具有自动创建 Topic 的权限，或运维提前创建。
  • 重试延迟由 Topic 的时间轮控制，不阻塞主分区消费，但消息顺序会打乱（不同分区的重试消息交错消费）。
  • @RetryableTopic 本质上使用了 KafkaTemplate 发送失败消息到 retry Topic，因此需要配置好生产者（事务、幂等等）。
  • @DltHandler 必须写在同一类中，且方法签名与主监听方法兼容（可以省略 Acknowledgment）。
  • 若要在 DltHandler 中手动提交偏移，仍需注入 Acknowledgment，但通常死信后直接确认即可。

选型建议：

  • 如果业务逻辑要求分区内严格顺序且重试耗时极短（如瞬时网络抖动），选用 DefaultErrorHandler。
  • 如果追求高吞吐且允许短暂乱序，或重试可能耗时较长（如调用降级服务等待恢复），选用 @RetryableTopic。
  • 若同时需要非阻塞重试和顺序性，可以考虑在 retry Topic 中使用相同的 key 以保证顺序，但架构会更复杂，需谨慎评估。

注意：无论用哪种重试，不可重试的异常都必须明确排除，避免无限重试耗尽资源。

7. 进阶特性

7.1 幂等去重

Kafka 默认提供 “至少一次” 投递语义，网络重试、消费者 Rebalance 等都会导致重复消息。因此消费者必须自行实现幂等去重。

7.1.1 Redis SET NX

利用 Redis 的原子 SETNX 记录已处理消息 ID，适合低延迟场景。

  • TTL 设定：TTL 需大于业务最大重试窗口（通常 24~72 小时），避免因过期导致重复处理。
  • Key 设计：建议包含业务唯一键（如订单 ID），而非 offset，因为 offset 会跨重启失效。
  • 故障处理：若 Redis 不可用，可降级为记录失败日志并人工处理，不能直接放行消息（否则可能重复）。

7.1.2 数据库唯一索引

利用关系型数据库的唯一约束天然去重，适合强一致性要求。

处理时先插入一条记录，若抛出 DuplicateKeyException 则直接 ACK 跳过。

  • 优点：与业务数据同库，无需额外组件。
  • 缺点：写入压力大时可能成为瓶颈；需定期清理历史数据。

7.1.3 业务乐观锁（最优方案）

当消息处理本质上是状态更新时，利用数据库乐观锁可实现零额外存储的去重。

相同消息重复执行时，version 已变，affected rows = 0，直接 ACK 即可。

  • 适用条件：业务操作必须能表示为幂等 SQL（如 INSERT ... ON DUPLICATE KEY UPDATE、版本号比较）。
  • 注意：若业务涉及多个写操作，需将整个处理过程包装在数据库事务中，保证原子性。

选型总结：

方案 适用场景 复杂度 一致性
Redis SET NX 高频、低延迟，可接受最终一致 低 弱
DB 唯一索引 强一致性要求，同库方便 中 强
业务乐观锁 状态更新类操作，性能高，零额外存储 中 强

8.2.1 Producer 端

  • Spring Boot 可设置 spring.kafka.producer.flush-on-close=true 自动执行，无需手动。
  • 若配置了事务，关闭前必须确保事务已提交或回滚，否则可能残留悬挂事务，导致后续消费阻塞（需设置 transaction.timeout.ms）。

8.2.2 Consumer 端

Spring Kafka 容器在关闭时会自动：

  1. 停止 poll 新消息
  2. 等待当前批次处理完毕
  3. 提交偏移（若手动提交，需自己在监听器中调用 acknowledge()）
  4. 关闭连接

关键配置：

默认超时为 10 秒。若单条消息处理可能超过 10 秒，务必调大此值，否则容器会强制中断，可能导致偏移未提交或消息处理一半。

最佳实践：

  • 消费者 @PreDestroy 中一般不需要额外操作，除非有自定义资源需要清理。
  • 停机前最好先通过监控确认 Lag 接近 0，再执行关闭，避免积压过大导致停机超时。

七、Kafka Connect

在前一章中，我们讨论了 Spring Kafka——在应用代码中通过 KafkaTemplate 和 @KafkaListener 来收发消息。但 Kafka 生态中还有一个专门解决”数据搬移”的组件，它不属于业务代码，而是基础设施层的数据管道。

1. Kafka Connect 解决了什么问题

企业数据架构中有一个非常常见且”无聊”的需求：把 A 系统的数据导入 Kafka，或把 Kafka 的数据导出到 B 系统。

这些需求的特点是：不涉及复杂业务逻辑，就是纯数据搬移。如果每次都写一个 Spring Boot 应用来做这件事，你会反复实现：

  • 连接管理、重试、断点续传
  • offset 管理和容错
  • 并行度控制和水平扩展
  • 监控和运维接口

Kafka Connect 就是为这种”无聊但必须”的数据集成工作设计的标准化框架——它把这些通用能力封装好，你只需要写一个配置 JSON 就能启动一个数据管道。

八、Kafka Stream

在前面几章中，我们一直把 Kafka 当作”消息中转站”来使用——Producer 发消息，Consumer 收消息，Broker 负责存储和转发。但 Kafka 生态中还有一个常被低估的能力：Kafka Streams。

它回答了这样一个问题：消息从 Topic 中流出来之后，能不能不经过外部计算框架，在 Kafka 自己的生态里完成实时计算？

1. Kafka Streams 是什么

它的核心思想是：把 Kafka Topic 当作无限延伸的、持续追加的数据流，对这个数据流做 filter / map / join / aggregate 等操作，结果实时写回 Kafka。

  • 需要对接外部系统（MySQL / Elasticsearch / 数据湖）→ 选 Flink，Connector 生态更丰富
  • 需要复杂事件处理（CEP）、超大规模窗口计算 → 选 Flink，状态管理和容错机制更成熟
