RocketMQ深入原理与实践

发表于 2026-06-29 更新于 2026-07-19 分类于 分布式与微服务 ， 消息队列

一、RocketMQ 的定位

如果说 Kafka 是”日志系统出身、顺便做了消息队列”，那 RocketMQ 就是”电商交易出身、把分布式事务刻在基因里”。

RocketMQ 诞生于阿里巴巴的双十一场景。在淘宝的交易链路中，消息队列面临的核心问题与 Kafka 的日志采集场景截然不同：

维度 Kafka 的日志场景 RocketMQ 的交易场景
核心诉求 超高吞吐（百万级 QPS） 消息可靠 + 事务一致性
延迟要求 秒级可接受 毫秒级
消息可靠性 丢几条可以（日志采样） 一条都不能丢
特殊需求 流处理、历史回溯 事务消息、延时消息、消息过滤
消费模式 纯 Pull Pull + 长轮询（开发者感受像 Push）

一句话：Kafka 追求的是”快”，RocketMQ 追求的是”稳”。这不是优劣之分，而是场景匹配问题。

1.1 NameServer

NameServer 是 RocketMQ 最与众不同的设计——它是一个无状态、无 Leader 的路由注册中心。

核心特点：

  • 每个 NameServer 节点独立运行，互不通信，不选举、不投票
  • Broker 向所有 NameServer 注册自己的信息（长连接 + 心跳）
  • Producer / Consumer 向任意一个 NameServer 查询路由表

为什么不用 ZooKeeper？

阿里的目标是零外部依赖、极简部署。每个 NameServer 只有几万行代码，单进程启动仅需 512 MB 内存，不依赖任何第三方存储——这是”够用就好”的工程哲学。

与 Kafka 的对比：Kafka 老版本依赖 ZooKeeper 做元数据管理，ZooKeeper 本身是一套需要独立维护的复杂系统；RocketMQ 的 NameServer 开箱即用。后来 Kafka 推出 KRaft 模式去掉 ZooKeeper，某种意义上也是在追赶这种简洁性。

1.2 Broker

Broker 分为 Master 和 Slave 两种角色：

角色 职责 说明
Master（BrokerId = 0） 提供读写 一个 Master 可以挂多个 Slave
Slave（BrokerId > 0） 只读，分担消费压力 从 Master 同步数据

Broker 向 NameServer 注册的信息包括：自己管理了哪些 Topic、每个 Topic 的读写队列数、权限等。Producer / Consumer 查询 NameServer 时，拿到的就是这份路由表。

1.3 Topic 与 MessageQueue

与 Kafka 的 Topic → Partition 对应，RocketMQ 的模型是 Topic → MessageQueue：

  • 一个 Topic 可以有多个 MessageQueue，分散在不同 Broker 上，实现水平扩展
  • 读写队列分离：每个 MessageQueue 有独立的读队列（readQueueNums）和写队列（writeQueueNums），可以做灰度缩容——先把写队列数缩小，等已有消息被消费完再缩读队列

RocketMQ 的 MessageQueue 本质上就是 Kafka Partition 的对应物，都实现了消息的物理分片和并行消费。区别在于 RocketMQ 的队列数通常比 Kafka Partition 数多得多（默认一个 Broker 就有数十个 Queue），这使得 RocketMQ 在消息分散均匀性上更具优势。

2. 消息存储

Kafka 的模型是”每个 Partition 一个独立日志文件”。RocketMQ 采用了不同的思路：CommitLog 集中存储 + ConsumeQueue 轻量索引。

读写数据流

flowchart LR
    subgraph 写入路径
        P[Producer 消息] --> CL["CommitLog\n（顺序追加，单文件）"]
    end

    subgraph 异步构建索引
        CL -->|Dispatcher 异步写| CQ["ConsumeQueue\nTopicA/Q0 → offset1\nTopicA/Q1 → offset2\nTopicB/Q0 → offset3"]
    end

    subgraph 消费路径
        C[Consumer] -->|① 查索引，拿到物理偏移量| CQ
        C -->|② 按偏移量读数据| CL
    end

注意：ConsumeQueue 是由 Broker 后台的 ReputMessageService 异步构建的，每条索引固定 20 字节（CommitLog 物理偏移量 8 字节 + 消息大小 4 字节 + 消息 Tag 哈希 8 字节），极其轻量。

这种设计的好处：

  • 顺序写极致：所有 Topic 的消息共用同一个 CommitLog，始终是顺序追加，磁盘 IO 利用率最大化。无论有多少个 Topic，都是往一个文件末尾追加
  • 队列灵活：ConsumeQueue 只是索引，非常轻量（每条索引 20 字节），可以创建大量 Queue（Kafka Partition 多了会加重 ISR 同步负担，RocketMQ 的 Queue 多了影响小得多）
  • 消费者可以直接读 Slave：Slave 同步的是 Master 的 CommitLog，配置 slaveReadEnable=true 后，消费者可以从 Slave 读取消息，分担 Master 的读压力

与 Kafka Segment 模型的对比：

维度 Kafka RocketMQ
存储模型 每个 Partition 独立日志文件 全局 CommitLog + 索引
读写方式 每个 Partition 随机写（多文件并发） 全局顺序写（单文件）
读消息 直接从 Partition 日志读 先查 ConsumeQueue 找位置，再读 CommitLog
Queue/Partition 数量影响 ISR 同步维护压力大，不建议太多 ConsumeQueue 很轻量，多开没问题

CommitLog 集中存储的设计代价是读消息需要一次间接寻址（先查 ConsumeQueue 再读 CommitLog），但 RocketMQ 用 Page Cache + MMap 把这次寻址限制在内存操作里，对性能影响微乎其微。

3. DLedger

Kafka 通过 ISR 管理副本同步——Leader 维护一个”跟得上”的 Follower 列表，只有 ISR 内的节点参与写入确认。RocketMQ 早期也采用类似的 Master-Slave 同步（同步双写 / 异步复制），但在 4.5 版本后引入了 DLedger（Distributed Ledger）——一套基于 Raft 协议的提交日志。

3.1 传统 Master-Slave 的问题

核心矛盾：Master-Slave 模式下，可靠性和可用性很难同时保证。

3.2 DLedger 如何解决

DLedger 将 CommitLog 替换为一个基于 Raft 共识协议的分布式日志。每个 Broker 节点都是 Raft 组的一个成员，Leader 选举、日志复制、提交都由 Raft 协议保证：

3.3 DLedger vs Kafka ISR

维度 DLedger（RocketMQ） ISR（Kafka）
共识协议 Raft（多数派投票） ISR 集合维护（Leader 单方面管理）
提交条件 超过半数节点写入确认 所有 ISR 节点写入确认（acks=all）
Leader 选举 Raft 协议自动选举 Controller 指定新 Leader
一致性保证 强一致性（已提交消息不会被新 Leader 覆盖） 依赖 ISR 完整性
最小节点数 3（Raft 要求奇数节点） 1（单副本也跑）

一句话：DLedger 用标准 Raft 替代了 Master-Slave 的半手工同步，把 RocketMQ 的一致性提升了一个档次。但它需要至少 3 个节点（Raft 的硬要求），对小规模部署不够友好。

3.4 DLedger 模式下的消费

开启 DLedger 后，消费者只能从 Leader 读取消息（Follower 不提供读服务），因为只有 Leader 知道哪些消息已经”被提交”（多数派确认过）。这与 Kafka 的”消费者只能从 Leader 读”是一致的。

四、Consumer

1. 消费模型

RocketMQ 底层是 Pull 模型，但 SDK 做了 长轮询（Long Polling） 封装，对开发者表现为 Push 的效果——你定义一个 Listener，消息到达时自动回调：

长轮询的底层原理：

长轮询消除了纯 Pull 的空轮询问题，又保留了 Pull 的流控优势（消费者控制拉取速度）。这是 RocketMQ 在消费体验上优于 Kafka 的一个细节。

2. 两种消费模式

2.1 集群消费（Clustering）—— 默认

同消费者组内，一条消息只被一个消费者消费。 这是最常见的模式，对应 Kafka 的”同组竞争”。

2.2 广播消费（Broadcasting）

同消费者组内的每个消费者，各自消费全量消息。 对应 Kafka 中”每个消费者一个 Group”的用法。

与 Kafka 的对比：Kafka 用 Consumer Group 机制实现”组内竞争、组间广播”。RocketMQ 把这两种模式直接放在消费模式上——CLUSTERING 就是组内竞争，BROADCASTING 就是每个消费者都收全量。广播模式下，消息进度本地文件管理（不存 Broker），不支持故障转移。

3. 消费重试与死信队列

当消费者返回 RECONSUME_LATER，RocketMQ 不是立刻重投，而是有阶梯式延时重试：

RocketMQ 的死信队列不需要手动配置，它是 Topic 级别的自动内置机制。当消费重试达到上限（默认 16 次），消息自动转入一个名为 %DLQ%消费者组名 的 Topic，由人工或补偿程序处理，保证主消费链路不被污染。

与 Kafka 的对比：Kafka 没有内置的重试和死信机制，需要开发者在消费端自己实现重试逻辑和死信 Topic。RocketMQ 把消费重试做到 Broker 端，对开发者开销极小。这是 RocketMQ 在业务场景下优于 Kafka 的另一个关键差异。

五、事务消息

1. 问题场景

下单扣库存是一个典型需要分布式事务的场景：

在 Kafka 中，这个问题需要本地事务表 + 定时任务补偿（即所谓的”发件箱 + 反查”模式）。RocketMQ 把它做进了 Broker 里。

2. 事务消息的执行流程

sequenceDiagram
    participant PS as 订单服务(Producer)
    participant MQ as RocketMQ
    participant IS as 库存服务(Consumer)

    PS->>MQ: 1. 发送 half 消息（对消费者不可见）
    MQ-->>PS: 2. half 消息写入成功
    PS->>PS: 3. 执行本地事务（创建订单）
    PS->>MQ: 4a. 本地事务成功 → Commit
    PS-->>MQ: 4b. 本地事务失败 → Rollback
    PS-->>MQ: 超时未响应 → 回调 checkLocalTransaction()
    MQ->>IS: 5. Commit 后投递给消费者
    IS->>IS: 6. 扣库存

流程拆分说明：

  1. Half 消息：生产者发送一条”预备消息”，Broker 接收并持久化，但标记为”对消费者不可见”
  2. 执行本地事务：生产者收到 Broker 的 half 确认后，执行本地事务（比如创建订单）
  3. 提交/回滚：本地事务成功则通知 Broker Commit（消息对消费者可见），失败则 Rollback（消息删除）
  4. 事务回查：如果 Broker 长时间没收到 Commit/Rollback（比如生产者在执行本地事务时宕机），Broker 会定期回调生产者的 checkLocalTransaction() 接口，询问本地事务最终状态

4. 事务消息 vs 本地消息表（发件箱模式）

如果技术栈是 Kafka，通常用**本地消息表（发件箱模式）**实现类似的最终一致性：

对比：

维度 RocketMQ 事务消息 本地消息表 + Kafka

RocketMQ 的事务消息本质上就是把”本地消息表 + 定时轮询 + 补偿”这套方案做到了 Broker 里，开发者不需要重复造轮子。

六、延时消息

在《消息队列基础与核心模型》中我们讲了延时消息的概念（下单 30 分钟未支付自动取消）。RocketMQ 是三者中实现最优雅的——原生内置，无需任何间接方案。

延时消息的实现原理：

与 Kafka 的对比：Kafka 完全不原生支持延时消息，需要消费端自己判断时间并 Sleep 或再投递。RabbitMQ 可以用 TTL + DLX 间接实现，但机制绕、不直观。RocketMQ 的延时消息是最”开箱即用”的方案。唯一的限制是只有 18 个级别——如果需要 7 秒的精确延时，RocketMQ 做不到，需要自己扩展。

七、消息过滤

Kafka 在消息过滤上很弱——消费者只能订阅整个 Topic，无法按条件筛选消息。RocketMQ 在这方面提供了完善的过滤能力。

1. Tag 过滤（最常用）

一个 Topic 下的不同 Tag 相当于逻辑上的”子 Topic”，共享同一个 Topic 的 CommitLog 和 ConsumeQueue 资源，减少了 Topic 数量膨胀。

2. SQL 表达式过滤（基于属性）

原理：消息的属性在写入 CommitLog 时就一并存储了。消费过滤时，Broker 在投递前根据属性的值做简单的条件比较。SQL 过滤相比 Tag 过滤更灵活但更重，不适合超高吞吐.

八、Spring Boot RocketMQ

前面七章从架构原理、存储模型到事务消息、延时消息，完整覆盖了 RocketMQ 的核心理论。本章定位不同：不讲”为什么”，讲”怎么用”。所有理论概念（NameServer、CommitLog、事务回查等）的机制请回顾前七章。

版本选择：学习用 5.3.0（4.x 客户端兼容 + 5.0 新特性）。生产环境根据团队 RocketMQ 服务端版本来选对应版本。

2. 生产者：RocketMQTemplate

Spring Boot 自动配置了 RocketMQTemplate Bean，直接注入即可。它封装了 RocketMQ 原生 Producer 的全部发送操作，支持 destination 格式：topic:tag（用冒号分隔 Topic 和 Tag）。

2.1 syncSend() 方法签名

同步发送——阻塞等待 Broker 确认后才返回结果。适合必须立即拿到发送结果的场景。

参数 类型 说明
destination String 格式 topic:tag，如 "order-topic:OrderCreated"。tag 可省略，等同订阅所有 Tag
payload Object 消息体，自动序列化（默认 JSON）
message Message 使用 MessageBuilder 构建，可设置 Header、Key 等
timeout long 发送超时毫秒数。不指定则使用 YAML 配置的 send-message-timeout（默认 3000）
delayLevel int 延时级别（1~18），见 第五章延时消息

返回值 SendResult（参见 §2.6）包含 msgId、sendStatus、messageQueue 等信息，用于日志记录和问题排查。

2.1.1 发送方式对比

方式 方法 阻塞？ 吞吐量 适用场景
同步发送 syncSend() 是 中 重要业务，需立即确认
异步发送 asyncSend() 否 高 一般业务，可回调处理结果
单向发送 sendOneWay() 否 最高 日志/监控，可容忍丢失
顺序同步 syncSendOrderly() 是 低 同 Key 消息必须有序

2.2 asyncSend() 异步发送

不阻塞当前线程，结果通过 SendCallback 回调处理。发送吞吐量远高于同步模式。

SendCallback 接口：

回调方法 触发时机 注意事项
onSuccess() Broker 确认成功 可记录 msgId、queueId 用于追踪
onException() 发送失败/超时 回调运行在 Netty I/O 线程，禁止阻塞操作或重发（会导致线程耗尽）

回调线程陷阱：onException() 在 Netty I/O 线程中执行。绝对不能在回调中再次调用 syncSend()（死锁风险）或执行长时间阻塞操作（阻塞所有异步发送）。

2.3 sendOneWay() 单向发送

发送后不等待任何响应，不抛异常、不走回调。吞吐量最高，但消息可能静默丢失。

适用场景：日志收集、监控指标、不重要的心跳上报。

注意：单向发送无法获取 msgId 和发送结果。如果消息对业务有影响（哪怕间接影响），勿用此方法。

2.4 syncSendOrderly() 顺序发送

根据 hashKey 将消息路由到同一个 MessageQueue，保证该 Key 下的消息严格有序。

参数 说明
hashKey 哈希键，相同值路由到同一 Queue。通常用业务唯一 ID（订单号、用户 ID）

顺序发送的代价：如果目标 Queue 的某条消息消费失败 → 后续消息全部暂停等待重试。因此不要求顺序的场景优先用 syncSend()，吞吐更高。

2.5 convertAndSend() 对象转换发送

与 syncSend() 功能相同，但命名风格对齐 Spring Messaging 的 convertAndSend 约定。

区别：convertAndSend() 内部调用 syncSend()，区别仅在于方法名——让习惯 Spring JMS/RabbitMQ 的开发者更熟悉。推荐统一用 syncSend()，语义更明确。

2.6 SendResult 与消息元数据

syncSend() 和 asyncSend() 的返回结果都是 SendResult，封装了消息写入 Broker 后的元数据。

SendStatus 取值含义：

状态 含义
SEND_OK 发送成功，Broker 确认写入
FLUSH_DISK_TIMEOUT 写入成功但刷盘超时（消息可能未真正落盘）
FLUSH_SLAVE_TIMEOUT 主节点写入成功但 Slave 同步超时
SLAVE_NOT_AVAILABLE 主节点写入成功但无可用 Slave

生产环境：SendStatus != SEND_OK 时，消息可能已经写入 Broker 但未满足刷盘/同步策略。不能简单认为消息丢失——需要结合 Broker 端的刷盘机制（同步/异步）和 Slave 可用性来判断。最安全策略：记录日志 + 监控告警 + 必要时人工排查。

2.7 MessageBuilder 构造消息

当需要设置 Key、Header、延时级别 等更精细的控制时，用 MessageBuilder 构建 Message 对象。

RocketMQHeaders 常用常量：

常量 含义 示例值
KEYS 消息 Key "ORDER_001"
TAGS 消息 Tag "OrderCreated"
DELAY_TIME_LEVEL 延时级别（1~18） "5"（1 分钟）
TRANSACTION_ID 事务 ID 事务消息用

注意事项：

  • MessageBuilder.withPayload() 支持任何 Object 类型，由 RocketMQMessageConverter 自动序列化。
  • 在 destination 中指定 Tag（如 "topic:tag"）与 Header 中设置 TAGS 效果一致，但 destination 中的 Tag 优先级更高。
  • Key 不是必填，但强烈建议设置——Key 是消息查询、消息轨迹追踪的唯一标识。

3. 消费者：@RocketMQMessageListener

3.1 注解属性详解

属性说明：

  • topic / consumerGroup：必填。消费者组与实际业务隔离相关——不同消费者组独立维护消费进度，互不影响。
  • selectorType / selectorExpression：消息过滤，详解见 §3.7。
    • selectorType=TAG 时，selectorExpression 支持 "*"（全订阅）、"tag1"（单 Tag）、"tag1 || tag2"（多 Tag）。
    • selectorType=SQL92 时，表达式如 "amount > 1000 AND region = 'east'"，需 Broker 开启 enablePropertyFilter=true。
  • consumeMode：CONCURRENTLY（并发，默认，吞吐最高） vs ORDERLY（顺序，单线程，吞吐低）。详解见 §3.4。
  • messageModel：CLUSTERING（集群，组内一条消息只被一个消费者消费） vs BROADCASTING（广播，组内各消费者各自全量消费）。
  • consumeThreadMax：并发消费模式下的最大线程数。上线不超过 Queue 数——Queue=4 时，consumeThreadMax 设为 64 也不会有更多并行度。
  • maxReconsumeTimes：消费失败后的重试上限。超过此次数，消息自动转入 死信 Topic（%DLQ%消费者组名）。参见 §6.2。
  • consumeTimeout：单条消息处理超时（分钟）。在 ORDERLY 模式下，超时意味着当前 Queue 被阻塞等待重试，生产环境务必评估业务最长处理时间并留余量。

3.2 监听接口选择

@RocketMQMessageListener 注解的类必须实现 RocketMQListener<T> 接口。泛型 T 决定了如何接收消息体。

3.2.1 RocketMQListener<MessageExt>（完整消息）

最通用，可以访问消息的全部元信息。

3.2.2 RocketMQListener<String>（直接消息体）

框架自动提取 message.getBody() 并按 UTF-8 解码为字符串。

适用：消息体是纯文本/JSON 字符串，不需要访问 Header、Key 等元数据。

3.2.3 RocketMQListener<OrderDTO>（泛型自动反序列化）

指定业务对象类型，框架自动 JSON 反序列化。

限制：无法访问 msgId、keys、tags 等元数据。如需同时获取元数据 + 自动反序列化的对象 → 只能用 MessageExt 模式手动解码，或者注入 RocketMQMessageConverter 辅助。

3.3 MessageExt 的方法

MessageExt 对应 RocketMQ 的原生消息对象，承载了消息的全部元数据。

3.3.1 消息属性读取

生产者可以通过 MessageBuilder.setHeader() 传递自定义属性，消费者从 message.getProperties() 中读取。

自定义属性用于传递业务路由指令、数据来源、版本号等。所有属性以纯文本存储在 Broker 端，单个属性值建议不超过 1KB。

3.4 消费模式：CONCURRENTLY vs ORDERLY

维度 CONCURRENTLY（并发，默认） ORDERLY（顺序）
处理方式 多线程并发消费同一个 Queue 单线程串行处理每个 Queue
吞吐量 高 低（单线程瓶颈）
失败影响范围 只影响当前消息（其他消息不受影响） 阻塞整个 Queue（暂停后续消息消费）
失败重试 返回 RECONSUME_LATER，消息延时重试 返回 SUSPEND_CURRENT_QUEUE_A_MOMENT，暂停 Queue
适用场景 日志、通知、无关顺序的业务事件 订单状态流转、账户余额变更

顺序消费的代价：失败时暂停整个 Queue。若某个 Queue 积压大量失败消息，该 Queue 后续所有消息都会被阻塞。生产环境建议：将顺序性要求高的场景独立为 Topic，配置独立的消费者组，避免影响其他消息。

3.5 消费重试与死信队列

RocketMQ 消费失败后有内置的阶梯式延时重试，不需要手动编码。

在 Spring Boot 中，重试行为由注解控制：

触发重试的方式：

  • 方法抛出异常 → 自动触发重试
  • 在 CONCURRENTLY 模式下可返回 ConsumeConcurrentlyStatus.RECONSUME_LATER
  • 在 ORDERLY 模式下可返回 ConsumeOrderlyStatus.SUSPEND_CURRENT_QUEUE_A_MOMENT

死信 Topic 命名规则：%DLQ%消费者组名。例如消费者组 order-consumer-group → 死信 Topic %DLQ%order-consumer-group。

死信 Topic 自动创建，但需要有人订阅处理。建议为每个核心消费者配置一个死信处理监听器，对死信做人工排查 + 重放。

3.6 批量消费

Spring Boot Starter 不直接支持 @RocketMQMessageListener 级别的批量消费注解属性。需要批量消费时，有两种方式：

方式一：使用原生 DefaultMQPushConsumer + 设置 consumeMessageBatchMaxSize（配合 Spring Boot 手动管理生命周期）：

方式二：升级到 RocketMQ 5.0 + 使用 Proxy SDK（原生支持批量消费配置）。

大部分业务场景下，单条消费 + 数据库批量插入（攒够一批再写 DB）比真正的批量消费更简单且风险更可控。消息批量消费一旦有一条失败，整批都需要重试或手动拆分处理。

3.7 消息过滤：Tag 与 SQL92

原理已在第七章介绍，这里只讲 Spring Boot 中的配置。

过滤粒度对比：

过滤方式 粒度 性能 灵活度 额外配置
Tag 粗粒度 高 低 无
SQL92 属性级别 中 高 Broker 开启 enablePropertyFilter=true

同一个 Topic 下用 Tag 区分事件类型（OrderCreated、OrderPaid、OrderCanceled），用 SQL92 做更细的属性筛选（如按地区/金额）。Tag 是生产环境最常用的方式。

4. 事务消息（Spring Boot 版）

事务消息的原理已在第五章详细介绍（half 消息 → 本地事务 → Commit/Rollback → 回查），本节只讲 Spring Boot 中的 API。

4.1 RocketMQLocalTransactionListener 接口

实现这个接口并注册为 @RocketMQTransactionListener，框架自动将其绑定到事务生产者。

接口方法：

方法 触发时机 返回值 注意
executeLocalTransaction() 发送事务消息后立即回调 COMMIT/ROLLBACK/UNKNOWN 必须捕获所有异常，不能抛给框架
checkLocalTransaction() Broker 长时间未收到确认时回调 COMMIT/ROLLBACK/UNKNOWN 必须幂等（可能被多次回查）

4.2 sendMessageInTransaction() 方法

TransactionSendResult vs SendResult：

TransactionSendResult 继承 SendResult，额外包含 localTransactionState 字段——表示 executeLocalTransaction() 的返回值。但注意：这个状态是 Producer 端本地事务执行后的瞬时状态；如果是 UNKNOWN，最终结果要等回查 checkLocalTransaction() 后才能确定。

4.3 事务回查机制

什么时候会触发回查？

  • executeLocalTransaction() 返回 UNKNOWN
  • executeLocalTransaction() 抛出异常
  • 生产者宕机，Broker 超时未收到 Commit/Rollback

回查的频率由 Broker 端配置控制（默认 60s 首次回查，之后逐步增加），与 Spring Boot 应用代码无关。代码层面只需要保证 checkLocalTransaction() 幂等且快速响应。

与 Kafka 的区别：Kafka 没有原生事务消息，需要”本地消息表 + 定时任务”实现最终一致性。RocketMQ 用 @RocketMQTransactionListener 把事务回查做到框架里，代码量更少、实时性更高。详细对比见第五章 事务消息 vs 本地消息表。

5.2 延时消息最佳实践

  • 延时消息发送后是不可撤销的：发送到 Broker 后，即使还没到延时时间，也无法撤回。如果业务可能取消延时消息（如用户付款后取消超时取消），需要在消费端做业务判断——“收到取消消息时检查订单状态，已支付则忽略”。
  • 延时级别的精度是固定的——如果需要 7 秒的精确延时，RocketMQ 做不到。此时要么接受最近级别（5s 或 10s），要么用外部调度器（如 Redisson 的 RDelayedQueue）替代。
  • RocketMQ 5.0 支持任意精度定时消息（基于时间轮），但仍需 Broker 端升级并配置。

6. 错误处理

6.1 消费者异常处理模式

6.1.1 返回重试状态（CONCURRENTLY 模式）

核心规则：

  • RocketMQListener.onMessage() 中抛出任何未捕获异常 → 框架自动返回 RECONSUME_LATER → 触发延时重试
  • 方法正常结束（不抛异常）→ 框架返回 CONSUME_SUCCESS → 提交 offset
  • 不可重试的异常必须被 catch 住，不要抛出，否则会永远阻塞重试

6.1.2 区分可重试与不可重试

分类原则：

异常类型 示例 处理
可重试（临时） 网络超时、DB 死锁、下游 503 抛异常，重试
不可重试（永久） JSON 解析失败、业务规则冲突 catch 住，不抛出
不确定 未知 RuntimeException 保守方案：不捕获，触发重试（设置合理的 maxReconsumeTimes）

Spring Boot 应用关闭时，确保生产者缓冲区消息已全部发送、消费者已完成当前批次处理。

注意事项：

  • RocketMQTemplate 只是对 Producer 的薄封装，关闭时没有缓冲需要 flush。
  • 消费者 shutdown() 会等待当前批次处理完成后才真正关闭。
  • 若应用被强制 kill（kill -9），正在处理的消息会丢失 → RocketMQ 依赖重试 + DLQ 兜底。

11. 原生 API vs Spring Boot Starter

结尾给一个速查对比，帮你判断什么时候用哪种：

场景 推荐 API 理由
Spring Boot 项目，快速接入 RocketMQTemplate + 注解 自动配置、零 XML、参数集中管理
需要精细化控制（自定义拉取策略等） 原生 DefaultMQProducer / DefaultMQPushConsumer 全部参数可调，不受 Starter 封装限制
顺序消费 两种都可以 Spring Boot 通过 consumeMode=ORDERLY 支持
事务消息 两种都可以 Spring Boot 通过 @RocketMQTransactionListener 支持
批量消费 原生 API Spring Boot 注解不直接支持批量
5.0 Proxy + gRPC 原生 Proxy SDK rocketmq-spring-boot-starter 仍走 Remoting 协议

大多数场景：Spring Boot Starter 足够。只有在需要批量消费、自定义拉取策略、5.0 Proxy 模式时，才切换到原生 API。

九、RocketMQ 5.0 变革

2021 年发布的 RocketMQ 5.0 是自 4.0 以来最大的一次架构升级。它不仅仅是版本号的跳跃，更是从传统消息队列向云原生消息平台的范式转变。对于 Java 后端来说，理解 5.0 的变化意味着你能在面试和工作中说出”4.x 怎么做，5.0 怎么做得更好”。

1. 架构变化总览

核心变化：引入 Proxy 层。Proxy 是部署在 Broker 前端的无状态网关，负责协议转换、认证鉴权、限流熔断。

2. 五大关键变革

2.1 Proxy 模式：无客户端感知的升级

4.x 的问题：每次 RocketMQ 升级新功能（如新增消费模式、修改网络协议），都需要升级所有客户端（Producer/Consumer）——这在微服务场景下意味着数百个服务都要重新发版。

5.0 的方案：Proxy 层接管了与客户端的通信。升级 Broker 特性时，只要 Proxy 是兼容的，老客户端继续工作。客户端只需要支持 gRPC 协议即可——而 gRPC 是一个广泛支持的业界标准。

2.2 Controller 模式：从手工主备切换到自动选举

4.x 的 Broker 主备切换依赖外部协调（传统模式手工指定 Master-Slave，或者启用 DLedger 做 Raft 选举）。

5.0 引入了 Controller 组件（基于 jRaft），专职管理 Broker 的主备切换：

模式 适用版本 特点
传统 Master-Slave 4.x 手工切换，运维重
DLedger 4.5+ 基于 Raft，但和 commit log 耦合
Controller 5.0+ 推荐，专职选举，与存储解耦

2.3 Pop 消费模型：真正的 Serverless 友好

4.x 的消费模型：基于长连接 + 长轮询，Consumer 需要和 Broker 维持 TCP 长连接。这在传统微服务架构下没有问题，但在以下场景下很麻烦：

  • Serverless / FaaS：函数实例只在请求到来时启动，无法维持长连接
  • Kubernetes 弹性伸缩：Pod 频繁创建销毁，长连接的建立和断开开销大

5.0 的 Pop 模型：消费者通过 HTTP/gRPC 短连接发起 Pop 请求——“给我一条消息，我处理完再来拿”。处理完一条消息后不需要维持长连接，下次再发一个新的 Pop 请求即可。

Pop 模型的核心价值：让消息消费像 HTTP API 调用一样简单。Serverless 函数、边缘计算节点、短暂运行的批处理任务——这些场景终于可以原生使用 RocketMQ 了。

2.4 轻量级消息队列（Light Message Queue）

4.x 的 Topic 默认带读写队列、死信队列、重试 Topic——一组基础设施即使你只发几条测试消息，也有 8 个队列、若干内部 Topic 的初始化开销。

5.0 引入 Lite Topic（轻量级 Topic）：去掉了重试队列、死信队列、队列负载均衡等重型特性，极简设计。适合：

  • Serverless 函数的事件投递（函数自己处理重试逻辑）
  • IoT 场景下的高频小消息（设备心跳、传感器数据）
  • 简单的异步通知（不需要可靠性的）

2.5 Kafka 兼容性（Kafka Adapter）

5.0 内置了Kafka-on-RocketMQ 适配层，让标准 Kafka Producer/Consumer 可以直接连接到 RocketMQ Proxy，Proxy 在内部将 Kafka 协议请求转换为 RocketMQ 操作。这样原本跑在 Kafka 上的 Flink/Spark 任务，可以不改代码直接切换到 RocketMQ。

3. Java 开发者的迁移清单

重要：5.0 完全兼容 4.x 的 Remoting 协议。如果你不想改客户端代码，部署 5.0 的 Broker + Proxy，4.x 的客户端依然能照常工作。升级顺序：先升级服务端（Broker + Proxy），再逐步迁移客户端。

4. 5.0 新增的其他能力

特性 说明
消息 Schema 管理 内置 Schema Registry，支持 Protobuf/JSON Schema 校验
SQL92 过滤增强 从简单的属性过滤升级为标准 SQL92 表达式
定时消息增强 从 18 级固定延迟升级为任意精度定时（基于时间轮）
压缩支持 Broker 端支持消息体压缩存储（zstd/lz4）
消息队列多租户 支持 namespace 级别的逻辑隔离

总结 5.0 的核心精神：Proxy 解耦客户端和服务端；Controller 让主备切换自动化；Pop 消费适配 Serverless；Lite Topic 降低轻量场景的门槛；Kafka Adapter 打通两个生态。它不是 4.x 的修补，而是 RocketMQ 从”阿里内部的消息系统”走向”云原生标准消息平台”的关键一步。
