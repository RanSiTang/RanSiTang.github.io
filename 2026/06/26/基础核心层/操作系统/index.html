一、操作系统概述

1. 什么是操作系统

1.1 核心定义

核心定义： 操作系统（Operating System, OS）是管理计算机硬件与软件资源的系统软件，它运行在硬件之上，为上层应用程序提供可执行的程序环境，同时为底层硬件提供统一的抽象接口。

这个定义包含两个不可分割的职责，理解了这两个职责就理解了操作系统的全部设计逻辑：

  • 向下（管理者）：资源分配器。管理 CPU（调度）、内存（分配）、磁盘（文件系统）、网络等。
    • 核心矛盾：硬件资源有限，而申请请求无限。OS 负责“公平”与“高效”。
  • 向上（服务者）：虚拟机/抽象层。屏蔽硬件底层差异（如磁盘扇区、寄存器细节）。
    • 核心价值：提供统一的 API，实现“一次编写，处处运行”。

关键：操作系统本质上是 “硬件资源的拍卖官” + “硬件差异的屏蔽层”。拍卖官决定了谁先用 CPU、谁先用内存——这就是调度。屏蔽层则让同一个 Java 程序在 Intel 和 AMD CPU 上都能跑——这就是抽象。

  2. 忙等待（Busy waiting）：CPU 一直在空转检查状态寄存器，什么有用的事都不做
  3. 没有进程隔离：所有程序共享同一个物理地址空间，一个程序写歪了指针能把另一个程序干崩溃
  4. 没有权限控制：任何程序都能直接操作硬件，恶意程序可以轻易破坏系统

1.3 操作系统的三大设计目标

  1. 方便性（Convenience）：让程序员不用关心 DMA 控制器的寄存器地址
  2. 高效性（Efficiency）：在多个进程间公平且高效地分配 CPU、内存、I/O 带宽
  3. 可扩展性（Extensibility）：模块化设计，支持热插拔驱动和新文件系统

这三个目标之间天然存在矛盾。方便性要求更多的抽象层，但抽象层越多性能可能越差。整个操作系统的发展史，就是在这三者之间寻找平衡。

系统调用接口层的意义：用户态程序无法直接访问内核态，也不能直接操作硬件——所有操作都必须通过系统调用（syscall）请求内核”代办”。这一约束由 x86 CPU 的特权级别在硬件层面强制执行。

x86 CPU 的特权级别：x86 设计了四个特权环 Ring 0 ~ Ring 3。操作系统内核运行在 Ring 0（最高权限，可执行任何指令），用户程序运行在 Ring 3（最低权限，禁止执行 in/out/cli 等硬件操作指令）。如果用户态程序试图越权，CPU 会直接抛出 #GP 异常（General Protection Fault，通用保护故障）阻止执行——根本没有绕过空间。

Java 后端开发的每行代码都在用户态运行。当你写 new FileInputStream("data.txt") 时，JVM 最终会调用 open() 系统调用，从用户态陷入内核态，由内核完成实际的文件打开操作，然后返回文件描述符。这就是”用户态→内核态→硬件”的完整链路。

2.2 内核态与用户态的本质

维度 用户态（User Mode） 内核态（Kernel Mode）
CPU 特权级别 Ring 3（最低权限） Ring 0（最高权限）
可访问地址空间 仅进程自己的虚拟地址空间 全部物理内存 + 内核数据结构
可执行指令 常规指令（禁止执行 cli、hlt、in/out 等硬件控制指令） 所有 CPU 指令
映射 JVM 代码、C 标准库函数 printf 内核代码、驱动代码
崩溃后果 进程崩溃（core dump） 内核 panic（系统蓝屏/死机）

表里的 cli、hlt、in/out 是 x86 的硬件控制指令：cli 关中断，hlt 暂停 CPU，in/out 直接读写 I/O 端口。这些指令一旦被恶意或错误的用户程序执行，整个系统的稳定都会被破坏——所以 Ring 3 下 CPU 直接禁止执行它们。

上面区分了”两个态”各能干什么，一个自然的问题是：CPU 什么时候从用户态进入内核态？ 只有三种触发方式：

  1. 系统调用：用户进程主动请求内核服务（如 read()、write()）。这是主动陷入
  2. 中断（Interrupt）：外部硬件向 CPU 发信号（如网卡收到数据包）。这是被动打断
  3. 异常（Exception）：当前指令执行出错（如除零、缺页）。这也是被动打断

关键区别：系统调用是进程主动的、可预期的；中断和异常是外部的、不可预期的。但这三种情况的处理入口是同一个数据结构——中断向量表（Interrupt Vector Table，简称 IDT），它是一张由内核预先设置的表格，每个表项记录了某种中断/异常对应的处理函数地址。无论触发原因是什么，CPU 都通过查这张表跳转到内核的对应处理代码。

2.4 中断机制——OS 的”心跳”

2.2 节中提到，进入内核态有三种触发方式：系统调用、中断、异常。系统调用是进程主动请求，而后两者是被动触发的——它们都依赖中断机制。理解中断是理解后续「缺页中断」（第五章）、「时钟中断驱动调度」（第二章 CFS）、「epoll 事件通知」（第七章）的前置知识。

中断的本质：中断是硬件或软件向 CPU 发出的一个信号，通知 CPU “有事件需要立刻处理”。CPU 收到信号后暂停当前执行流，转而执行内核预设的中断处理程序，处理完毕后再恢复之前的执行流。

中断的三种类型：

类型 触发源 是否可屏蔽 典型例子
外中断（硬件中断） CPU 外部硬件 可屏蔽 网卡收到数据包、磁盘 I/O 完成、时钟滴答
内中断（异常） CPU 内部（指令执行出错） 不可屏蔽 除零错误（#DE）、缺页（#PF）、越权指令（#GP）
软中断 / trap 软件主动触发（系统调用） — int 0x80（32 位 syscall）、sysenter（64 位 syscall）

为什么要区分可屏蔽/不可屏蔽？ 可屏蔽中断（如网卡数据到达）优先级较低，CPU 在某些关键区域（如正在处理另一个中断）可以通过 cli 指令暂时关闭——这就是”关中断”保护临界区的原理。不可屏蔽中断（如内存 ECC 错误、电源故障）优先级最高，CPU 必须立刻响应，因为它们关乎数据正确性和硬件安全。

理解关键：中断是操作系统一切”被动响应”能力的根基。没有时钟中断，调度器无法抢占 CPU——这意味着一个死循环进程会永远霸占 CPU。没有缺页中断，虚拟内存无法按需加载——这意味着每个进程必须把全部地址空间一次性加载到物理内存，8GB 内存可能连一个大型 Java 应用都跑不了。

3. 操作系统的发展简史

3.1 从手工操作到批处理

阶段 时间 特征 问题
手工操作 1940s-1950s 程序员直接操作插拔线/开关 人机速度严重不匹配
批处理系统 1950s-1960s 作业提交到机房，按批次串行执行 CPU 等待慢速 I/O
多道批处理 1960s+ 内存同时驻留多道作业，CPU 不再空闲等 I/O 无交互能力

批处理时代的核心矛盾：CPU 速度是 I/O 的数十万倍。一道作业在等待 I/O 时，CPU 完全空闲。多道批处理的解法是：让多道作业同时驻留在内存中，A 等 I/O 时 CPU 切去执行 B——这是现代多任务操作系统的雏形。

3.2 分时系统与 UNIX 的诞生

1960s，贝尔实验室、MIT 和 GE 联合开发 Multics（多路信息计算系统），目标是构建一个支持多人同时交互的大型分时操作系统。Multics 因过于庞大而失败，但贝尔实验室的 Ken Thompson 把其中的精华抽出来写了一个更轻量的版本——UNIX。

UNIX 的设计哲学至今仍是 Linux 的基石：

  • 一切皆文件：硬件设备、管道、Socket 都用文件描述符统一操作。/dev/sda 对应的不是普通文件，而是一个块设备——但 open("/dev/sda") 和 open("/etc/hosts") 用的是同一个系统调用
  • 小工具组合：每个程序只做好一件事，通过管道 | 组合完成复杂任务：cat access.log | grep "ERROR" | wc -l
  • Shell 作为用户与内核之间的介质

3.3 Linux 的崛起与现代 OS 格局

4. 操作系统的核心组件总览

4.1 五大核心功能

4.1.1 进程管理（处理机管理）

进程管理也叫处理机管理。由于 CPU 是计算机最宝贵的资源，这个模块负责决定”谁在什么时候使用 CPU”。

主要功能：

  • 进程调度：决定哪个进程获得 CPU 执行权（如先来先服务、时间片轮转等算法）。
  • 进程同步与互斥：协调多个进程之间的先后顺序，防止它们在访问共享资源时发生冲突（例如使用锁机制）。
  • 进程通信（IPC）：负责不同进程之间的数据交换。
  • 进程控制：负责进程的创建、撤销、阻塞和唤醒。

简单理解：它就像一个指挥官，决定哪个任务先做，哪个后做，确保 CPU 不空闲且任务有序进行。

4.1.2 内存管理

内存管理负责管理计算机的内存（RAM）资源，确保每个程序都有足够的空间运行，且互不干扰。

主要功能：

  • 内存分配与回收：程序开始时分配内存，运行结束时回收内存。
  • 地址映射：将程序的逻辑地址（程序员看到的）转换为物理地址（硬件实际地址）。
  • 内存保护：确保进程 A 不会意外或恶意地访问进程 B 的内存空间。
  • 虚拟存储：当内存不足时，将硬盘的一部分充当内存使用（如页面交换），让用户感觉内存比实际大。

简单理解：它就像一个酒店前台，负责分配房间（内存空间），并确保客人们不会走错房间。

4.1.3 文件管理

文件管理模块负责处理计算机中的所有数据信息，将其组织成文件并存储在磁盘上。

主要功能：

  • 文件存储空间管理：记录磁盘上哪些地方存了文件，哪些地方是空的。
  • 目录管理：维护文件夹（目录）结构，方便用户查找和定位文件。
  • 文件读写管理：控制数据的读入和写出。
  • 文件保护与安全：设置权限（只读、读写、隐藏），防止非法访问。

简单理解：它就像一个图书管理员，负责把书（数据）分门别类放在书架上，并根据借书证决定谁能看什么书。

4.1.4 设备管理

设备管理负责管理所有外部设备（键盘、鼠标、显示器、硬盘、打印机、网卡等）。

主要功能：

  • 缓冲管理：解决 CPU 高速处理和外设低速传输之间的矛盾（如打印缓冲区）。
  • 设备分配与回收：根据请求分配设备，并在用完后收回。
  • 中断处理：处理硬件发出的信号（如鼠标点击）。
  • 驱动程序管理：提供统一的接口，让系统能识别和操作不同厂家生产的硬件。

简单理解：它就像是一个翻译官兼协调员，让 CPU 能听懂各种硬件的”方言”，并协调它们的工作。

4.1.5 用户接口/作业管理

这个模块是操作系统与用户（或程序员）之间的桥梁。

主要功能：

  • 命令接口：用户通过命令行（终端）或图形界面（GUI）直接操作电脑。
  • 程序接口（系统调用）：为开发者提供 API，让应用程序能请求操作系统完成任务（如”打开文件”这个动作其实是调用了系统接口）。
  • 作业控制：在批处理系统中，负责管理用户提交的一系列任务。

简单理解：它就是操作系统的服务窗口，用户通过这个窗口告诉电脑想做什么。

1. 进程

1.1 核心定义

为什么需要进程？

在回答”进程是什么”之前，先问一个更根本的问题：为什么不能让程序直接在 CPU 上跑？

答案藏在程序与进程的本质区别中：

  • 程序是静态的、被动的。它只是躺在磁盘上的一串指令序列，没有”生命”。程序不知道自己该用多少内存、该打开哪些文件、该在何时让出 CPU——它只是一段二进制数据。
  • 进程是动态的、主动的。当一个程序被加载到内存并开始执行，它就变成了一个进程——一个”活着的”执行实体。进程知道自己是谁（PID）、拥有哪些资源（地址空间、文件描述符）、执行到哪一步了（PC 寄存器）、以及处于什么状态。

没有进程抽象，世界会怎样？

  • 多个程序无法安全并发——代码段和数据段混在一起，A 程序可以随意读写 B 程序的内存
  • 程序执行到一半被打断→没有任何机制保存现场→只能重头再来
  • 资源无法隔离——一个程序把文件描述符写满，其他程序跟着遭殃

进程是 OS 为每个运行中的程序构建的”隔离房”——每个进程拥有自己独立的地址空间、独立的资源集合、独立的执行上下文。进程之间互不干扰（至少在设计意图上是这样）。

核心定义： 进程（Process）是程序在数据集上的一次执行过程，是操作系统进行资源分配和调度的基本单位。

这个定义有三个关键词：

  • 程序：静态的，存储在磁盘上的可执行文件（如 java、python、nginx 可执行文件本身）
  • 数据集：进程操作的对象（如 Java 进程操作的数据库连接、文件 IO 流、内存中的用户对象）
  • 执行过程：动态的，程序被加载到内存后 CPU 逐条执行指令的过程

程序和进程的关系（深入版）：

维度 程序（Program） 进程（Process）
存在形态 静态，磁盘上的文件 动态，内存中的执行实体
生命周期 持久（除非被删除） 临时（创建→执行→终止）
资源拥有 无（仅占用磁盘空间） 拥有 CPU 时间、内存、文件、I/O 设备
唯一性 一个程序可对应多个进程 每个进程通过 PCB 唯一标识
并发性 无 可与其他进程并发执行

一句话：程序 = 菜谱，进程 = 正在做菜的过程。 同一张菜谱可以被三个厨师分别在三个厨房同时做，用的是同一份指南，但各自有各自的锅（内存）、各自的进度（PC）、各自的状态（切菜中 / 焖煮中 / 出锅了）。

1.2 PCB：操作系统眼中的”进程”

操作系统并不关心你的 Java 代码在算什么——它关心的是如何描述和管理这个进程。这个描述结构就是 PCB（Process Control Block, 进程控制块）。

PCB 的正式定义与核心属性

PCB 全称进程控制块（Process Control Block），是操作系统专门为每个进程创建的、用于存放进程全部管理信息的数据结构，是进程存在的唯一标志。

这条定义包含一条最关键的因果链：进程创建 → 生成 PCB → 进程可被 OS 管理；进程终止 → 销毁 PCB → 进程在 OS 眼中消失。 PCB 是 OS 识别和管理进程的”身份证+档案袋”——没有 PCB，OS 就无法找到这个进程，更无法调度、挂起或终止它。

为什么说 PCB 和进程一一对应？

这是一个经典的面试题和内核设计问题，答案分三层：

  1. 定义层：PCB 是进程存在的唯一标志。 操作系统只能通过 PCB 来识别和控制一个进程——调度器通过 PCB 上的 vruntime 决定该不该运行它，内存管理通过 PCB 上的 mm_struct 知道它占用了哪些页。如果 OS 找不到一个进程的 PCB，这个进程对 OS 而言就是”不存在”的。

  2. 生命周期层：进程和 PCB 同生共死。 进程创建（fork()）时，OS 同步生成唯一的 PCB；进程终止（exit() + 父进程 wait()）后，OS 回收并销毁 PCB。两者生命周期完全绑定——不存在”有进程无 PCB”或”有 PCB 无进程”的过渡状态（僵尸态只是 PCB 还在、进程已死——详见 1.3 节）。

  3. 保障层：一一对应是并发执行和系统稳定的基础。 这种刚性绑定确保了每个进程拥有唯一的标识、独立的运行状态、独立的资源空间和完整的上下文切换能力。正是 PCB 的存在，使得 OS 能够在大规模并发场景下对每个进程进行精准的调度、管理和隔离——你可以在 top 里一眼看到 500 个线程各自的 PID、状态、CPU 占用，就是因为每个 LWP 都有自己独立的一份 task_struct。

1.3 经典三态模型——最简但够用

任何复杂的进程状态模型都从三态出发。三态模型是理解一切调度和并发控制的起点：

状态 含义 进程是否占有 CPU 能否立即运行
就绪态（Ready） 已获得 CPU 以外的所有必要资源，只等 CPU 时间片到位 否 ✅ 可以，就等调度器选中
运行态（Running） 正在 CPU 上执行指令 是 —
阻塞态（Blocked / Waiting） 正在等待某一事件发生（I/O 完成、信号、锁释放等），此时即使分配 CPU 也无法执行 否 ❌ 不行，必须等事件发生

1.3.3 从三态到五态、七态——更精细的生命周期

三态模型加上 新建态（进程刚创建、PCB 尚未完全初始化）和 终止态（进程已退出、PCB 未回收——即 Linux 的僵尸态 Z）就是教科书上的五态模型。在此基础上再加入两个挂起态（就绪挂起、阻塞挂起——进程被换出到外存）就是七态模型。

但不必死记——Linux 内核实际使用的是另一套更实用的状态标记（见下方 1.3.5 节）。五态和七态只是三态在「进程创建入口」和「进程退出出口」上的展开，核心状态机仍然是就绪 ↔ 运行 ↔ 阻塞。

挂起（Suspend）的本质：将进程从内存暂时调出到外存（swap），剥夺其占用的内存帧。为什么需要挂起？

原因 说明
内存严重不足 系统内存耗尽，必须把部分进程换出到磁盘让出物理页
调节系统负载 多核满载时主动挂起低优先级进程，让核心资源给关键任务
父进程请求 父进程要求暂停子进程（如调试器 SIGSTOP）
用户主动请求 用户按 Ctrl+Z 挂起前台进程
操作系统维护 内存碎片整理 / 检查点保存期间暂时挂起进程

后端视角：挂起≠终止。挂起的进程后被激活时从中断点继续执行，状态不变。Docker 容器的 pause 操作本质上就是向容器内所有进程发送 SIGSTOP，使它们进入挂起态——进程的 CPU 时间被清零、内存被保留但不参与调度。

1.3.4 单阻塞队列 vs 多阻塞队列——工程上的分水岭

维度 单阻塞队列 多阻塞队列
结构 系统只维护一个全局阻塞队列，所有阻塞原因不同的进程混在一起 每种阻塞原因单独一个队列
唤醒方式 事件发生时遍历整个队列查找匹配进程 —— O(n) 事件发生时直接定位对应队列 —— O(1)
内核开销 高（每次唤醒都要扫描全队列） 低（只操作目标队列）
并发支持 差，队列全局锁竞争 好，不同队列锁独立，多核可并行
优先级反转 容易触发——阻塞队列无序，低优先级进程可能排在队头 可避免——每个队列内部可独立排序
实际应用 仅适用于进程数极少的简单嵌入式系统（FreeRTOS 的某些配置） 所有现代操作系统（Linux、Windows、macOS）的标准实现

Linux 的实现：task_struct 中的等待队列（wait_queue_head_t）就是多阻塞队列的实现。不同的事件拥有独立的等待队列——磁盘 I/O 完成中断唤醒的是 wait_on_page_* 队列的进程，网卡中断唤醒的是 sk_buff 等待队列的进程，互不干扰。

多阻塞队列的四个核心优势：

  1. 大幅提升唤醒效率：事件发生 → 直接定位队列 → O(1) 唤醒，无需遍历
  2. 显著降低内核开销：总锁粒度从”一个全局锁”降到”一队列一个锁”
  3. 精细化阻塞管理：可为不同等待原因的进程设置不同的超时、优先级和唤醒策略
  4. 便于优先级动态调整：每个队列内部可独立实现优先级排序

1.3.7 操作系统会优先唤醒哪些进程？

唤醒策略直接影响系统吞吐量和响应时间。主流 OS 按以下优先级决策（从高到低）：

优先级 进程类型 原因
最高 I/O 刚完成的进程 I/O 是最慢的操作——用户等 I/O 的时间远长于等 CPU，优先唤醒 I/O 完成进程能最大程度缩短感知响应时间
次高 高优先级进程 实时任务（RT）和交互式任务（UI 线程）必须优先
中等 等待时间最长的进程 防止饥饿——不能让某些进程被长期”晾”在阻塞队列中
次低 短进程 快速完成一个短任务释放资源 > 让长任务多跑一小会

1.4 上下文切换的本质

当 CPU 从执行进程 A 切换到进程 B 时，OS 必须：

  1. 保存 A 的执行上下文：通用寄存器、程序计数器（PC）、栈指针（SP）、标志寄存器、浮点寄存器 → 存入 A 的 task_struct 中的 thread_struct
  2. 加载 B 的执行上下文：从 B 的 task_struct 中恢复上述所有寄存器
  3. 切换地址空间：将页表基址寄存器（x86 的 CR3）改为 B 的页表物理地址 → 这将导致 TLB（快表）全部失效，后续的内存访问都要走完整的页表遍历
切换类型 涉及的操作 典型耗时 触发频率
进程切换 全部上下文 + 地址空间 + TLB 刷新 1-5μs 调度器决定
线程切换 仅寄存器上下文（同一地址空间） 0.2-1μs 调度器决定
系统调用 少量寄存器 + 特权级切换 0.05-0.2μs 每次 syscall

后端启示：这就是为什么线程数不是越多越好。假设 CPU 只有 4 核，创建 400 个线程——线程切换本身消耗的 CPU 时间可能超过做正事的 CPU 时间。线程池的最佳大小不是一个固定数字，而是取决于 CPU 核数和任务的 I/O 占比。

2. 线程

理解了进程作为「资源容器」的角色后，一个自然的问题是：容器里的执行流可以不止一个吗？ 当然可以。一个进程内部可以有多个线程，它们共享进程的地址空间和文件描述符，但各自有独立的栈和寄存器——这就是线程。如果说进程回答的是「资源归谁」，那线程回答的是「谁来执行」。

2.1 核心定义

核心定义： 线程（Thread）是 CPU 调度的最小单位，是进程内部的一个独立执行流。同一进程内的所有线程共享该进程的地址空间（代码段、数据段、堆），但各自拥有独立的栈、程序计数器和寄存器集合。

进程 vs 线程对比：

维度 进程 线程
调度单位 资源分配的基本单位 CPU 调度的基本单位
资源拥有 独立的地址空间、文件描述符表 共享地址空间、共享文件描述符
隔离性 强：进程 A 崩溃不影响 B 弱：一个线程的栈溢出可能影响整进程
通信方式 IPC（管道/消息队列/共享内存） 直接读写共享变量（需同步）
创建开销 大（复制整个地址空间 + PCB） 小（分配栈 + 插入 task_struct）
切换开销 大（需要切换地址空间） 小（保留地址空间不变）

线程被称为轻量级进程（Lightweight Process, LWP），但”轻量级”不代表”越轻越好”。线程数过度时，上下文切换 + 缓存抖动会使整体吞吐量不升反降。

2.2 用户级线程 vs 内核级线程 vs 混合模型

模型 实现位置 优点 缺点
N:1（用户级线程） 用户态线程库 切换极快（0.01μs级别），不需要陷入内核 一个线程阻塞整个进程阻塞，无法利用多核
1:1（内核级线程） 内核 真正的并发（多核），一个线程阻塞不影响其他 创建/切换需陷入内核，开销大
N:M（混合/两级模型） 用户态调度器 + 内核线程池 兼顾轻量和多核 实现复杂（调度器的调度器），容易调优失控

2.2.1 Java 线程模型的演进

上面的 N:1 → 1:1 → N:M 理论框架，在 Java 生态中有精确的对应——JDK 二十多年的线程模型演进，恰好在三个阶段各站了一次：

阶段 JDK 版本 线程模型 说明
绿色线程 JDK 1.2 之前 N:1（用户级线程） 所有线程由 JVM 在用户态调度，单核时代够用，无法利用多核
平台线程 JDK 1.2 ~ JDK 20 1:1（内核级线程） 每个 Thread 对应一个 OS 内核线程，真正的多核并发
虚拟线程 JDK 21+ M:N（混合模型） JVM 在少量 OS 线程上调度大量虚拟线程，回归用户态调度

JDK 1.2 起采用 1:1 模型：每个 new Thread() 创建一个 OS 级的内核线程（在 Linux 上通过 pthread_create 创建一个轻量级进程）。Java 线程命名后可以通过 top -H -p <pid> 在宿主机上找到对应的内核线程。

JDK 21 引入 Virtual Thread（虚拟线程）：实质上是 M:N 模型——JVM 在少量 OS 线程（Carrier Thread）上调度大量虚拟线程。一个虚拟线程在阻塞 I/O 操作时 JVM 会自动将它从 carrier thread 上”卸载”，释放 carrier thread 去执行其他虚拟线程。这就是 Go 的 Goroutine 模式被引入 Java。

理解关键：JDK 21 虚拟线程解决了”线程池大小设计”这个困扰工程师数十年的问题——虚拟线程下你不必再纠结”I/O 密集型到底设多大线程池”，每个任务直接创建一个虚拟线程即可，JVM 帮你调度到有限的内核线程上。但注意：虚拟线程不适用于 CPU 密集型任务——那是传统平台线程（Platform Thread）的领地。

3. CPU 调度算法

现在系统中同时运行着几十上百个进程、上千个线程——但 CPU 核心只有那么几个。谁该先用，谁该排队？ 这就是 CPU 调度要回答的问题。调度算法决定了每个线程何时获得 CPU、能占用多久、又要何时让出——它本质上是在「吞吐量」和「响应时间」之间寻找平衡点。

3.1 调度的核心矛盾

CPU 调度的本质是在周转时间（完成一个任务要多久）和响应时间（任务多久才开始被处理）之间做取舍。没有一种算法能在所有维度上都最优。

调度目标 衡量指标 典型场景
吞吐量高 单位时间完成的任务数 批处理、离线计算
响应快 从请求到首次执行的时间 交互式应用、Web 服务
公平 每个任务都能获得 CPU 时间 多租户云环境
低延迟 实时任务的截止时间保障 工业控制、音视频流

操作系统层面的调度结果会直接传导到后端应用：

  • 一个线程在 TASK_UNINTERRUPTIBLE（D 状态）下不参与调度 → 该请求无响应
  • 一个线程的 nice 值被调高（低优先级）→ 你的业务线程可能被推迟执行
  • CFS 为每个任务保证公平的 CPU 时间 → 即使 JVM 里只有一个线程在忙，多个进程一起竞争时它并不占优势

3.2 经典调度算法速览

算法 策略 优点 缺点 适用场景
FCFS（First Come First Served） 先来先服务，无抢占 实现简单 护航效应（长任务挡短任务） 批处理
SJF（Shortest Job First） 预估运行时间最短的优先 平均周转时间最优 长任务可能饿死；无法预知运行时间 理论最优解
时间片轮转（RR） 每个进程轮流占用固定时间片 响应时间可控，公平 时间片太大≈FCFS，太小≈频繁切换 分时系统
优先级调度 高优先级任务先执行 兼顾响应性和吞吐量 低优先级任务可能饿死 实时 + 通用混合
多级反馈队列（MLFQ） 多个优先级队列 + 动态升降级 自适应，兼顾长短期任务 参数多，实现复杂 现代通用系统

核心概念 —— vruntime（虚拟运行时间）：

4. fork() 与 exec()：进程从何而来

到目前为止，我们讨论的都是「已经存在」的进程如何被调度。但进程本身是怎么诞生的？ 在 Linux 中，所有进程的起源都可以追溯到两个系统调用——fork() 创造进程的骨骼，exec() 注入进程的灵魂。理解这对组合，你就理解了每个 Shell 命令、每个 Java 进程的诞生过程。

4.1 fork() 的”分身术”

fork() 最反直觉的点：调用一次，返回两次——分别在父进程和子进程中返回不同的值。

写时复制（Copy-on-Write, CoW） 是现代 fork 的核心优化：

  • fork 后子进程并不真正拷贝父进程的整个地址空间，而是共享同一份物理页
  • 父子进程的页表项都被标记为只读
  • 当任一进程尝试写某页时 → 触发缺页异常 → 内核才真正拷贝该页 → 标记为可写
  • 效果：fork 一个 4GB 的 Java 进程只需几毫秒，内核只拷贝 task_struct 和页表，不拷贝实际数据

但是 COW 有内存压力上的风险：如果 fork 出来的子进程马上又要通过 exec() 加载新程序（Redis 的 BGSAVE 就类似这个模式），COW 的优化就很重要——因为 exec 之后旧地址空间直接丢弃，之前拷贝的页全白干了。

exec() 系列函数替换当前进程的地址空间为新的可执行程序影像。调用 exec 成功后不返回（原来的代码被替换了，没有”返回”的地方了）：

三、进程同步与互斥

第二章讲了进程和线程「如何存在、如何运行」。但一旦多个线程共享了数据，一个棘手的问题就出现了：你刚读到的值，在下一秒可能已经被另一个线程改掉了。并发执行的指令以不可预测的顺序交错——这是多线程编程一切复杂性的根源。第三章就来拆解这个问题：如何保护共享数据不被并发破坏？

1. 竞态条件

1.1 核心定义

核心定义： 竞态条件（Race Condition）是指多个执行单元并发访问共享数据时，最终结果取决于它们之间指令执行的相对顺序——而这个顺序是操作系统调度决定的，不可预测。

两个线程各做了一次 +1，期望最终值为 2，但实际可能是 1。指令交错执行是竞态条件的根源。

1.2 临界区

核心定义： 临界区（Critical Section）是访问共享资源的那一段代码，在这段代码执行期间，不允许其他线程进入同一个共享资源的临界区。

条件 含义 反例
互斥（Mutual Exclusion） 同一时刻最多一个线程在临界区内 两个线程同时写同一个文件
前进（Progress） 临界区空闲时，想进入的线程必须能有一个进入 永远没人进
有限等待（Bounded Waiting） 申请后必须在有限时间内进入 线程永远排不上号——饥饿
让权等待（Optional） 进不去的线程应释放 CPU 忙等待（spin lock）

忙等待不是无条件不好——在预期等待时间极短（纳秒级）的场景下，spin lock 省去了”阻塞→调度→唤醒”的开销，反而更高效。Java 的 synchronized 在轻量级锁阶段就使用了 CAS 自旋。

2. 硬件级同步原语

临界区的互斥访问靠什么来保证？软件方案（如 Peterson 算法、Dekker 算法）在理论上是正确的，但现代 CPU 的乱序执行和多核缓存一致性让纯软件方案在实际硬件上不再可靠。真正的互斥依赖 CPU 提供的原子指令——硬件保证一条指令执行期间，任何其他核心都不能插手。这就是接下来要讨论的硬件级同步原语。

2.1 Test-and-Set（TAS）

2.2 Compare-and-Swap（CAS）

CAS 是比 TAS 更灵活的原子指令，也是 Java 无锁编程的基石。

CAS 的 ABA 问题：线程 A 读到值为 0 → 线程 B 改成 1 又改回 0 → CAS 发现值还是 0，以为没有变过。解决：加版本号 → AtomicStampedReference。

CAS 开销的关键：CAS 是 LOCK 前缀的原子指令，在现代 CPU 上约 10-20 个时钟周期。1500 线程一起争用一个 AtomicInteger 时，每个成功的 CAS 背后有 1499 次失败的 CAS + 它们触发的缓存一致性协议开销。这就是为什么在高竞争场景下 CAS 自旋锁不如重量级锁——阻塞比烧 CPU 更划算。

2.3 CPU缓存与伪共享

前面的 CAS 开销分析引出了一个更深层的问题：为什么 1500 个线程 CAS 争用同一个变量会那么慢？答案藏在 CPU 缓存的设计中。

缓存行：CPU 读写的”最小单位”

核心定义： 缓存行（Cache Line）是 CPU 缓存与主内存之间数据传输的最小单位，典型大小为 64 字节。CPU 每次从内存读取数据时，不是按字节读，而是按缓存行整块读取——即使你只访问了一个 long（8 字节），CPU 也会把包含它的整个 64 字节缓存行加载到 L1/L2 缓存中。

缓存行存在的理由是空间局部性原理：你访问了数组的第 N 项，大概率接下来会访问第 N+1 项。一次把相邻数据全部拉上缓存，减少内存访问次数。

伪共享（False Sharing）：缓存行的黑暗面

核心定义： 伪共享（False Sharing）是指两个线程各自修改不同的变量，但这些变量恰好在同一个缓存行中，导致缓存一致性协议将整个缓存行在 CPU 核心之间反复失效和同步——尽管两个线程在逻辑上并无共享数据。

3. 软件级同步机制

硬件提供了 CAS 和 TAS 这样的原子指令，但直接在应用代码里写自旋循环太底层、太容易出错——你必须手工处理自旋次数、回退策略、ABA 问题等。操作系统和编程语言在硬件原语之上封装了更高层的同步机制——信号量、互斥锁、条件变量——让开发者用更安全的抽象来保护临界区。这就是接下来要讨论的内容。

3.1 信号量（Semaphore）

核心定义： 信号量是 Dijkstra 于 1965 年提出的同步原语，是一个受保护的整型变量，只能通过 P（Proberen，测试）和 V（Verhogen，增加）两个原子操作来修改。

信号量的核心思想：value 的正负含义不同：

  • value > 0：表示还有多少个空闲资源
  • value < 0：表示有多少个进程在等待（|value| = 等待队列长度）
  • value = 0：刚好用完，没有等待

PV 操作与进程状态转换的关系

PV 操作不仅修改信号量的值，更直接驱动进程状态的切换。理解了这一点，就能把上章的状态转换图和你每天都在用的并发原语连接起来：

操作 触发条件 进程状态转换 内部机制
P（acquire）——申请资源失败 信号量 value - 1 < 0（没有空闲资源） 运行 → 阻塞 进程被放入信号量的等待队列，主动放弃 CPU
P（acquire）——申请资源成功 信号量 value - 1 ≥ 0（有空闲资源） 保持运行态 拿资源走人，不阻塞
V（release）——有进程在等 信号量 value + 1 ≤ 0（有进程在等待） 阻塞 → 就绪 OS 从等待队列中唤醒一个进程，放入就绪队列
V（release）——无进程在等 信号量 value + 1 > 0 无状态转换 仅递增资源计数，不需要唤醒任何人

一句话：P 操作就是”拿不到就睡”（运行→阻塞），V 操作就是”叫一个等待者起床”（阻塞→就绪）。 你现在写的每一行 sem.acquire() / sem.release()，在 OS 层面都是在操作进程的状态机和等待队列——掌握这一层映射，你排查死锁、饥饿和性能问题时就不会只停留在 Java 线程 dump 上。

3.2 互斥锁（Mutex）

互斥锁是信号量 value=1 的特化情况——二值信号量。但与信号量有本质区别：

维度 信号量 互斥锁
用途 控制资源访问数量（计数） 保护临界区不被多个线程进入
获取与释放 可以不同线程（生产者释放，消费者获取） 必须同一个线程（谁 lock 谁 unlock）
递归性 不支持 可重入锁支持（同一线程多次 lock）
所有权 无 有（记录 holder）

3.3 条件变量（Condition Variable）

核心定义： 条件变量允许线程在某个条件不满足时原子地释放锁并进入等待，当条件可能变为满足时被唤醒重新获取锁。条件变量必须与互斥锁配合使用。

为什么必须用 while 而不是 if？

while 循环在这里构成了一个安全的守卫——即使虚假唤醒或条件被抢占了，循环会再次检查条件并可能重新等待。这一模式是并发编程中的基本法则。

四、死锁

第三章我们学习了如何用锁保护共享数据。但锁是一把双刃剑——用好了保证并发安全，用不好则轻则性能下降，重则整个系统卡死。上一章末尾的哲学家就餐问题已经暗示了这个风险：当多个线程互相等待对方手中的锁时，死锁就降临了。 第四章就来深入拆解死锁：它为什么发生、怎么预防、在不同层面（Java/数据库/分布式）各有什么应对策略。

1. 死锁的定义与必要条件

1.1 核心定义

核心定义： 死锁（Deadlock）是指两个或两个以上的进程（线程）在执行过程中，因争夺资源而造成的一种互相等待的现象——每个进程都在等待其他进程释放它们所需的资源，但没有任何一个进程会释放自己已经持有的资源，从而导致所有相关进程永久阻塞。

死锁 vs 活锁 vs 饥饿：

状态 特征 比喻
死锁 互相等待，谁都不放，永久阻塞 两辆车在单行道面对面停了，谁都不倒车
活锁 线程不断改变状态尝试获取锁但永远拿不到 两个人狭路相逢，同时往同一侧让——反复让到同一侧
饥饿 低优先级线程长期得不到调度/资源 VIP 通道永远有人插入，普通通道的人一直等

1.2 四个必要条件

死锁必须同时满足四个条件——破坏任一条件就可预防死锁：

条件 含义 关键词
互斥（Mutual Exclusion） 资源只能被一个进程/线程独占使用 不可共享
持有并等待（Hold and Wait） 进程持有一些资源的同时等待获取另一些资源 吃着碗里看着锅里
不可剥夺（No Preemption） 已分配的资源不能被强制收回，只能由持有者主动释放 只能主动放不能被动抢
循环等待（Circular Wait） 存在进程资源的环形等待链 首尾相接

这四个条件是 AND 关系，不是 OR 关系。 实际工程中，破坏”循环等待”最常见、破坏”持有并等待”次之。

前四章的核心主题是「计算」——进程如何运行、如何调度、线程之间如何协作。现在切换到操作系统另一个同等重要的职责：「存储」。内存是 CPU 能直接访问的唯一天然存储介质，但多道程序共享物理内存引发了一系列难题：进程 A 的指针写飞了会摧毁进程 B 吗？8GB 物理内存怎么让 20 个进程都以为独占 4GB 空间？JVM 的 -Xmx 参数和 OS 的内存管理究竟是什么关系？第五章就来回答这些问题。

1. 为什么需要内存管理

多道程序设计产生四个内存问题：

问题 描述 OS 的应对方案

MMU：内存管理的硬件基础

MMU（Memory Management Unit，内存管理单元） 是 CPU 内部的模块，负责将逻辑地址转换为物理地址。

  • CPU 发出的每条访存指令都经过 MMU
  • MMU 根据页表基址寄存器（x86 的 CR3）找到进程的页表
  • TLB（Translation Lookaside Buffer） 缓存最近的转换结果，TLB 命中时直接跳过页表遍历

1.5 早期方案：连续分配与分段

在分页成为主流之前，OS 用过两种更”原始”的内存管理方案。理解它们有助于理解为什么分页最终胜出——以及分页解决的到底是什么问题。

连续分配：每个进程占用物理内存中一整块连续的区域。分配算法有四种：

算法 策略 特点
首次适应 从低地址开始找，分配第一个够大的空闲块 简单，但低地址区碎片多
最佳适应 分配大小最接近需求的空闲块 产生大量微小外部碎片
最差适应 分配最大的空闲块 剩下的块仍比较大，但大块资源耗尽快
下次适应 从上一次分配位置继续向下找 避免反复扫描低地址区

连续分配的核心缺陷是外部碎片：虽然空闲总量可能够，但因不连续而无法分配。这就像停车场里虽然总空位够，但都是零散的单空位——一辆大巴（大段连续内存）仍然停不进去。

段式管理：按程序的逻辑段（代码段、数据段、栈段、堆）分别分配一块连续物理内存，每个段有自己的段表项 (基址, 段长)。优点是不同段可以分别保护和共享（如两个进程共享同一个代码段的物理拷贝），缺点仍然是外部碎片。

段页式管理：先分段，再在段内分页——段表项存的不再是基址而是页表指针。地址结构变为 段号 + 页号 + 页内偏移。这曾是 x86 的真实方案，但现代 OS 普遍回归纯分页——段页式多了一层查表，实际收益不大。

方案 逻辑划分 外部碎片 内部碎片 共享/保护
连续分配 无（整个进程一块） ✗ 严重 ✗ 有 困难
分段 按逻辑段（代码/数据/栈） ✗ 有 ✗ 有 天然支持（按段）
分页 按固定大小（4KB） ✓ 无 ✗ 有（最后一页半页） 页级（需额外机制）
段页式 先按逻辑段，再分页 ✓ 无 ✗ 有 天然支持（按段）

2. 分页与多级页表

前面列出了四个内存问题，其中「地址冲突」和「物理内存碎片」是内存分配的经典难题。操作系统的解决方案是分页——将物理内存切成固定大小的「页框」，用统一的页表维护虚拟地址到物理地址的映射关系。理解了分页，就理解了现代内存管理的第一步。

2.0 地址转换实例

在深入分页细节之前，先通过一个具体数值例子建立直觉——CPU 每发出一个虚拟地址，MMU 到底是怎么算出物理地址的：

计算题公式速记：逻辑地址 A，页大小 L → 页号 P = A / L，页内偏移 d = A % L。查表得物理块号 f → 物理地址 = f × L + d。如果考虑 TLB：有效访问时间 = TLB命中率 × (TLB时间 + 内存访问时间) + TLB未命中率 × (TLB时间 + 2 × 内存访问时间)——多级页表每次查表都要多一次访存。

2.1 分页——消灭外部碎片

核心定义： 分页将物理内存划分为固定大小的页框（Page Frame），将进程的逻辑地址空间划分为同样大小的页（Page）。页和页框等大（4KB），用一个页表维护两者的映射关系。

分页的关键优势：

  • 没有外部碎片：任何空闲的页框都能满足请求
  • 仅有内部碎片：进程最后一页平均浪费 2KB（半页），可接受
  • 共享简单：两个进程的页表可以指向同一个物理页框（如共享 libc.so）

2.2 多级页表——解决页表太大的问题

4GB 进程在 4KB 分页下需要 1,048,576 个页表项，每个 PTE 8 字节 → 页表占 8MB。多级页表将页表本身分页，只把用到的部分加载到内存。

多级页表的思想与 B+Tree 索引类似：虚拟地址里 80% 的区域是未分配的，多级页表中对应的中间项自动为”空”，一台页表内存从 8MB 降到 ~200KB。

3. 虚拟内存

3.1 核心定义

核心定义： 虚拟内存（Virtual Memory）是一种将逻辑地址空间与物理地址空间分离的内存管理技术。每个进程拥有独立的、连续的、足够大的逻辑地址空间，而实际常驻物理内存的可能只是其一小部分。当进程访问不在物理内存中的部分时，硬件触发缺页异常（Page Fault），由 OS 从磁盘换入所需页面。

六、文件系统

第五章讨论了内存——CPU 可以直接寻址的临时存储。但数据要持久保存，必须写到磁盘上。操作系统如何组织和管理磁盘上的持久数据？这就是文件系统的职责。有趣的是，内存管理用「页表」做映射，文件系统用「inode」做映射——凡是解决「物理不连续→逻辑连续」问题的子系统，核心都是一个映射表。 理解了这一个共同模式，第六章的学习会轻松很多。

1. 核心定义

核心定义： 文件系统是操作系统中负责组织、存储、检索和管理文件的子系统。文件的本质是一段有名字的字节序列。文件系统在它之上增加了三个关键属性：

属性 OS 中的实现 示例
元数据 inode（索引节点） 权限、所有者、大小、时间戳
组织结构 目录树（目录也是文件） /home/user/blog/post.md
访问控制 权限位 / ACL rwxr-xr--

凡是”物理不连续、逻辑上连续”的问题，一定有一个映射表在中间。 文件系统的答案就是 inode。

2. inode 与文件描述符

2.1 inode：文件的身份证

核心定义： inode（Index Node，索引节点）是文件系统中的核心元数据结构，每个文件（包括目录）在创建时被分配一个唯一的 inode 号（在该文件系统内唯一），inode 中存储了该文件除文件名外的一切信息。

关键区别：inode 不存储文件名。文件名存在目录的数据块中——目录本质上是一个「文件名 → inode 号」的映射表。

2.2 文件描述符：进程与文件的”连接”

核心定义： 文件描述符（File Descriptor, fd）是进程打开文件时内核返回的一个非负整数，它是进程文件描述符表中的索引。此后所有对该文件的读写关闭操作都通过这个数字进行。

3. VFS：虚拟文件系统

核心定义： VFS（Virtual File System，虚拟文件系统）是 Linux 内核中的一层抽象，它在系统调用接口与具体文件系统实现之间建立了一个统一的接口契约。

“一切皆文件”不是口号而是设计。VFS 将这个理念写死在内核代码结构中。

4. Page Cache：文件 I/O 性能的内核武器

核心定义： Page Cache 是 Linux 内核维护的一块内存区域，用于缓存从磁盘读取的文件内容。命中缓存直接返回（免去磁盘 I/O），未命中则从磁盘读取并写入 Page Cache 后再返回。

三种写策略对比：

策略 行为 write() 返回时机 数据安全性 性能
write-back 写入 Page Cache 即返回 立即返回 低（崩溃可能丢失数据） 最高
write-through 同时写入 Page Cache 和磁盘 磁盘写入完成后返回 高 低
ordered（日志型） 先写日志，后写实际位置 日志写入后返回 中高（崩溃可恢复） 中

Linux 默认使用 write-back 策略。这就是为什么突然断电可能导致最近几秒写入的数据丢失。数据库（MySQL InnoDB、PostgreSQL）为了解决这个问题，在应用层做了 redo log / WAL。

为什么 Kafka 这么快

Kafka 的高性能依赖三个 OS 机制：

  1. 顺序写（利用磁盘的顺序带宽 ~500MB/s，随机写 ~1MB/s）
  2. Page Cache（读写命中缓存 = 内存级速度）
  3. sendfile 零拷贝（consumer 消费消息时数据从 Page Cache 直接到网卡，不经用户态）

2. 五种 I/O 模型

2.1 阻塞 I/O（BIO）

两个阶段进程全程阻塞，编程模型最简单。

问题：一个线程只能处理一个连接。1000 个并发 = 1000 个线程。

2.2 非阻塞 I/O（NIO）

阶段 1 不阻塞（轮询模式），阶段 2 仍阻塞。缺点是忙等——反复轮询消耗大量 CPU。

非阻塞 I/O 的核心价值不在于”一直轮询”——而在于它让一个线程可以同时查询多个连接。这就是 I/O 多路复用的前置条件。

2.3 I/O 多路复用（核心）

核心定义： I/O 多路复用允许一个线程同时监控多个文件描述符（连接）的就绪状态。当其中任何一个 fd 变为可读/可写时，系统调用返回，线程再对就绪 fd 执行真正的 read/write。

与阻塞 I/O 的本质区别：

维度 BIO I/O 多路复用
阻塞在哪个调用上 recvfrom（等待一个 fd 的数据） select/epoll_wait（等待任意 fd 就绪）
一个线程能管理几个 fd 1 个 成千上万个
适用场景 连接数少（<100）、长连接 连接数多（>1000）、海量短连接

Redis 的线程模型就是 I/O 多路复用的经典案例：单线程 + epoll 同时监听多个客户端连接 + 事件驱动。每个请求快速完成（内存操作），不会阻塞其他请求。

2.4 信号驱动 I/O（SIGIO）

阶段 1 完全不阻塞（异步通知），阶段 2 仍需阻塞拷贝。在 Java 网络框架中没有直接使用。

2.5 异步 I/O（AIO）

Linux 内核的 AIO 实现一直不成熟。Java NIO 2.0 的 AIO 在 Linux 上底层其实是 epoll 模拟的——并非真正的异步 I/O。这也是为什么 Netty 选择了 epoll + Reactor 线程模型而非追求 AIO 的原因。

Selector 实现 Java NIO Selector（存在 CPU 100% 空轮询 bug） JNI 直接调用 epoll（完全绕过 Java Selector）

6. 总结：I/O 模型 → 后端框架的选择

理解关键：I/O 模型的选择取决于你程序在”等待数据”阶段做什么。内存操作（Redis）= 单线程多路复用足够。磁盘 I/O / 外部 RPC = 需要线程池处理阻塞操作。没有”永远正确”的 I/O 模型，只有”适合当前负载”的 I/O 模型。

八、进程间通信（IPC）

第七章一直在讨论单个进程如何与外部设备做 I/O——等数据、拷数据。但一个真实的后端系统由多个进程协作构成：Nginx worker 收到请求后要交给 PHP-FPM 处理，Redis 主进程 fork 出的子进程要写 RDB 文件。这些进程之间怎么交换数据？ 这就是进程间通信（Inter-Process Communication, IPC）要解决的问题。

1. IPC 概述

核心定义： 进程间通信（Inter-Process Communication, IPC）是指在不同进程之间传递数据、信号或协调执行的机制。由于每个进程拥有独立的地址空间，进程间通信必须通过操作系统内核作为”中介”来传递数据。

IPC 方式 速度 内核参与 主要使用场景
匿名管道 中 有 Shell 管道
命名管道 中 有 无亲缘进程通信
消息队列 中 有 结构化消息传递
共享内存 最快 无 高性能通信（RocketMQ）
信号量 — 有 进程/线程同步
信号 快 有 终止/挂起进程
Socket 中-慢 有 网络通信/本地通信

2. 管道

特点：单向、内核缓冲区（通常 64KB）、写端关闭时读端返回 EOF。

3. 共享内存——最快的 IPC

管道很好用——Shell 的 | 就是最直观的体现。但它有一个根本限制：数据必须经过内核缓冲区中转（从写端的内核缓冲区拷贝到读端的内核缓冲区）。能不能完全绕过内核的数据拷贝？可以的——让两个进程直接映射同一块物理内存。这就是共享内存，所有 IPC 中最快的方式。

核心定义：共享内存允许多个进程将同一段物理内存区域映射到各自的虚拟地址空间中。此后进程对此段内存的读写与普通内存访问完全相同——不需要经过内核中转。

所有 IPC 速度排名（大致数量级）：

方式 单次延迟（4KB 数据）
共享内存 ~0.1μs
Unix Domain Socket ~5μs
管道 / FIFO ~10μs
TCP Socket（本机） ~20μs
消息队列 ~20μs

共享内存快在哪里：共享内存的 read/write 是普通的 CPU load/store 内存指令——不经过系统调用，也不需要内核做用户缓冲区与内核缓冲区之间的数据拷贝。

RocketMQ 为什么用它：CommitLog 使用 mmap 映射为共享内存（写入速度与内存写入相同）+ 顺序刷盘——消息生产接近内存写入速度，消费通过 sendfile 零拷贝。

4. 消息队列

管道和共享内存解决的是「怎么传数据」的问题，但传数据这件事本身还有另一种需求：不光要传，还要排队、要按消息而不是字节流来组织。这就是消息队列（Message Queue）的价值。

核心定义：消息队列是内核维护的一个消息链表，发送方放一条消息，接收方取一条消息。与管道不同，消息队列是按”消息/报文”为单位的——接收方可以按类型、按优先级取消息，而不是只能按收到字节的顺序读。

Linux 提供了两种 OS 级消息队列：

  • System V 消息队列：老牌实现，支持消息类型（msgtype），可以通过类型号实现简单的优先级和多播
  • POSIX 消息队列（mq_open / mq_send / mq_receive）：更现代的接口，支持消息优先级、异步通知（mq_notify）

OS 级消息队列 vs 应用级消息中间件：

维度 OS 消息队列 Kafka / RocketMQ
范围 单机内核 分布式集群
持久化 内核缓冲区（重启丢失） 磁盘持久化（日志型存储）
吞吐量 低（受内核限制） 高（RocketMQ 单机数十万 TPS）
可靠性 无副本、无确认机制 多副本、ISR、ACK 确认
适用场景 单机进程间轻量消息传递 分布式异步解耦、削峰填谷

理解关键：OS 级消息队列是 IPC 的一个标准成员，但它的定位是”内核提供的基础消息传递机制”——不讲可靠性、不讲高可用、不讲跨机器路由。现代分布式消息中间件解决的是完全不同的工程问题，但这不代表 OS 消息队列没用：在单机、嵌入式、低延迟场景下它的轻量是优势。理解两者的区别，有助于避免”用了 Kafka 就以为搞懂了 OS 消息队列”的认知错位。

5. 信号

管道和共享内存解决的是「传数据」的问题。但在进程间还有一种更轻量的通信需求——只传递一个意图，不需要传递数据。比如”Docker 要关掉这个容器”，只需要一个信号量级的通知。这就是信号（Signal）的用途。信号不是用来传大段数据的，而是用来向目标进程传递事件通知——“请优雅退出”（SIGTERM）、”立刻终止”（SIGKILL）。

常见信号与后端场景：

信号 编号 后端常见场景
SIGQUIT 3 kill -3 <PID> 触发 JVM 打印线程栈
SIGKILL 9 docker kill、OOM Killer
SIGTERM 15 正常 shutdown 的信号，触发 ShutdownHook

6. Socket

前面讨论的 IPC 机制——管道、共享内存、信号——都仅限于同一台机器上的进程间通信。但如果进程分布在不同机器上呢？这就需要一个能跨越网络的通信机制。Socket 就是这样一种万能 IPC：它既能走网络协议栈实现跨机器的 TCP 通信，也能走 Unix Domain Socket 实现同机的高速通信。实际上，你写的所有 Java 网络代码，底层都是 Socket。

Unix Domain Socket：同机最快的 Socket

维度 TCP Loopback Unix Domain Socket
协议栈 完整 TCP/IP 无协议栈（纯内存拷贝）
三次握手 有 无
速度 ~20μs ~5μs

典型应用：Redis 本地连接、Nginx + PHP-FPM、Docker daemon API

各种 IPC 在 Java 中的映射

IPC 方式 Java 中的对应 典型框架
信号 Runtime.addShutdownHook() ShutdownHook
共享内存 MappedByteBuffer（基于 mmap） RocketMQ、Chronicle Queue
Unix Domain Socket java.net.UnixDomainSocketAddress（JDK 16+） Netty native transport
TCP Socket java.net.Socket / Netty 一切网络框架

九、Linux 内核关键机制

定位：容器化与线上运维必知的内核机制。 本章聚焦于 Linux 内核中直接关系后端工程实践的几个关键机制，也是理解 Docker 容器、Kubernetes 资源限制、线上 OOM 故障排查的必要基础。

前八章覆盖了操作系统内核的经典模块：进程、内存、文件、I/O、IPC——这些是任何一本 OS 教科书都会讲的内容。这一章我们聚焦几个教科书不会重点讲，但日常工作中随时会遇到的 Linux 内核特性：cgroups（容器资源限制的核心）、namespace（容器视图隔离的核心）、OOM Killer（线上进程神秘消失的元凶）、以及 TCP 内核参数（高并发场景必调）。

1. cgroups —— 资源隔离与限制

核心定义： cgroups（Control Groups，控制组）是 Linux 内核提供的资源限制、优先级分配、资源统计和进程分组隔离的机制。

cgroups 的两个核心能力：

能力 含义 对应场景
限制（Limit） 限制进程组能使用的最大资源量 容器内存 limit 防止单容器吃光宿主机
统计（Accounting） 记录每个进程组的资源用量 监控容器的 CPU/内存实际使用量

2. namespace —— 视图隔离

核心定义： namespace（命名空间）是 Linux 内核提供的进程视图隔离机制。cgroups 决定”能用多少”，namespace 决定”能看到什么”。

namespace 隔离的内容 Docker 中的效果

Network 网络接口、路由表 容器有独立的 eth0、IP

User 用户和组 ID 容器内 root 可映射为宿主机普通用户

3. OOM Killer

cgroups 给容器设了内存上限，namespace 给容器做了视图隔离，但一个问题随之而来：如果进程无视 cgroups 限制，申请了超过限额的内存，会发生什么？ 操作系统不能凭空变出物理内存，此时必须有一个机制来终结失控的进程——这就是 OOM Killer。

核心定义： OOM Killer（Out-Of-Memory Killer）是 Linux 内核在物理内存（+ swap）被完全耗尽时调用的进程终结者。它根据每个进程的 oom_score 选择得分最高的进程强制终止。

Java 进程为什么容易被杀

  1. 堆内存大 → RSS 在 oom_score 计算中权重高
  2. DirectBuffer 和 Native Memory 不被 GC 追踪 → 实际使用量可能远超 -Xmx
  3. 容器场景的内存限制低 → 更容易触发 OOM 阈值

4. TCP 内核参数调优
