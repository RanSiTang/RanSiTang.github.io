一、Spring 事件机制概述

1. 什么是 Spring 事件

Spring 事件（Spring Events）是 Spring 框架内置的事件驱动编程模型，本质上是观察者模式（Observer Pattern）在容器级别的实现。它允许不同的 Bean 之间通过事件进行通信，而发布者和监听者之间完全解耦——发布者不知道谁在监听，监听者也不关心事件从何而来。

核心价值：

  • 模块解耦：核心业务逻辑与旁路逻辑（通知、日志、审计）彻底分离
  • 可扩展性：新增监听器无需修改原有业务代码，符合开闭原则
  • 事务感知：Spring 提供了事务绑定事件，确保事件处理与事务提交原子化
  • 容器原生：无需引入消息队列等外部依赖，适合单体应用内的异步解耦

典型场景：

场景 说明
用户注册 注册成功 → 发邮件、送积分、创建初始订单
订单状态变更 订单完成 → 发送通知、更新统计、清理缓存
操作审计 敏感操作 → 记录审计日志，与业务逻辑解耦
缓存更新 数据变更 → 发布事件 → 清除相关缓存项
事务后处理 事务提交后 → 发送 MQ 消息、调用第三方接口

注意区分：Spring Event 是进程内的事件机制，不跨 JVM。跨服务通信应使用 RabbitMQ、Kafka 等消息中间件。

2. 核心组件

Spring 事件机制由三大组件构成：

组件 角色 对应观察者模式 说明
ApplicationEvent 事件 Event 封装事件数据，继承自 java.util.EventObject
ApplicationEventPublisher 发布者 Subject ApplicationContext 实现了该接口
ApplicationListener 监听者 Observer 对某一类事件做出响应的回调接口

3. 设计哲学：为什么需要进程内事件？

核心判断：当你发现某个操作触发后需要执行多个”附加动作”，且这些附加动作与核心逻辑没有数据依赖时——就该考虑事件驱动。

四、事件广播原理

1. 广播器：ApplicationEventMulticaster

Spring 在容器启动时初始化事件广播器，负责将事件分发给匹配的监听器：

关键设计：

  • 默认同步执行：发布事件的线程会阻塞，直到所有监听器处理完毕
  • 错误传播：默认情况下，某个监听器抛出异常会中断后续监听器
  • 可配置线程池：注入 Executor 即可切换为异步广播（影响全局）

不建议轻易改为全局异步：广播器改为异步后，所有事件处理都变为异步，包括框架内部事件，可能导致不可预期的启动问题。推荐方案是用 @Async 精准控制单个监听器的异步行为（见 六、异步事件）。

3. @TransactionalEventListener vs @EventListener

维度 @EventListener @TransactionalEventListener
触发时机 事件发布后即刻执行 事务完成（commit/rollback）后执行
是否保证数据可见 可能读到未提交数据 必然在事务结束后执行

默认行为 立即同步执行 事务完成后执行（无事务时抛异常）

fallbackExecution 属性：设为 true 时，如果当前没有活跃事务，监听器也会执行（等同于普通 @EventListener）。默认 false——必须配合事务操作使用。
