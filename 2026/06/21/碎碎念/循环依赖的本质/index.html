循环依赖的本质不是注入问题，而是职责耦合问题

发表于 2026-06-21 更新于 2026-06-22

一、这篇文章想解决什么问题

1. 你大概率见过这样的场景

两个 Service，互相需要对方：

然后 Spring 启动——

于是你开始自救：加 @Lazy 先跑起来再说；或者把构造器注入改成字段注入，假装没看见这个环；再不行就换成 Setter 注入，至少不报错了。

代码能跑了。但你心里清楚：这事没完。

2. 这篇文章不讲什么

  • 不讲 Spring 三级缓存是怎么”容忍”循环依赖的
  • 不讲 Bean 的生命周期和提前暴露半成品对象的原理
  • 不讲 @Lazy 在框架底层是怎么生成代理的

这些内容网上已经有大量优秀的文章。它们回答的问题是——“Spring 是怎么让你勉强跑起来的。”

3. 这篇文章讲什么

这篇文章回答的是另一个问题——“为什么你会走到需要 Spring 帮你擦屁股这一步。”

具体来说，会讲清楚四件事：

  1. 循环依赖出现的本质原因——不是注入方式的问题，而是职责边界的问题
  2. 遇到 A ↔ B 时，从设计上怎么拆——不是怎么绕过去，而是怎么解开
  3. 不同的解耦方式适合什么场景——不是只有”抽第三方”一种答案
  4. 怎么判断该选哪一种——一个能直接套用的决策框架

二、循环依赖不是技术问题，而是设计信号

1. 什么叫循环依赖

一句话：A 依赖 B，B 又依赖 A，两者形成闭环。

在代码层面，它有三种表现形式：

  • 构造器注入的互相引用：启动就炸，Spring 直接报 UnsatisfiedDependencyException
  • 字段注入/Setter 注入的互相引用：启动不报错，但逻辑上环仍然存在
  • 方法调用的互相调用：A 调 B 的某个方法，B 又回来调 A 的某个方法，不一定通过依赖注入体现

表现形式不同，本质都一样：两个模块互相知道对方，谁也离不开谁。

2. 为什么循环依赖通常不是好事

循环依赖不是”语法错误”，它完全可以通过框架机制运行起来。但它是一个设计坏味道。

  • 职责边界不清：
    当 A 需要调 B，B 也需要调 A，说明两个模块没有清晰的“谁管什么”的约定。它们像两个互相插手的部门——A 要做的事有一部分藏在 B 里，B 要做的有一部分又绕回 A。
  • 模块之间知道得太多：
    A 不仅知道自己该干什么，还知道 B 的存在、B 的接口、B 的调用时机。B 也一样。这违背了“最少知识原则”——每个模块本应该只和它真正需要的协作者打交道，而不是和“因为历史原因绑在一起的邻居”深度纠缠。
  • 单独测试几乎不可能：
    想测 PlantingService？你得先把 AlertService 准备好。想测 AlertService？PlantingService 又不能少。就算用 Mock，你也要同时 Mock 两个互相引用的对象——单元测试的门槛被大幅抬高。
  • 改一处，牵动另一处：
    循环依赖把两个模块焊在了一起。你改 A 的逻辑，B 的行为可能悄悄发生变化；你在 B 里加一个新功能，A 可能莫名其妙地受到影响。变更成本 = 两个模块的认知成本之和。

3. 核心观点

真正要解决的，不是”怎么让循环依赖跑起来”，而是”为什么这两个模块必须互相调用”。

循环依赖是代码在对你喊一句话：“我的职责划错了，你听到了吗？”

框架可以帮你容忍这个环，但它不能帮你消除这个环背后的设计问题。@Lazy 不是解药，是止痛药。

三、A ↔ B 到底是哪类问题

面对一个循环依赖，大多数人的第一反应是——“能不能抽一个公共类出来？”但这不是标准答案，甚至不一定是正确答案。

在动手解耦之前，先搞清楚它属于哪类问题。

1. 四类典型场景

从成因看，常见的循环依赖大致可以分成四类：

类型 典型特征 本质
共享能力型 A 和 B 都需要同一段逻辑，但这段逻辑历史上只放在了一方，另一方只能“借用” 公共能力没有抽离
流程协作型 A 触发 B 处理，B 处理完又需要回写或通知 A 缺少合理的协作机制
职责拆分过细型 原本是一件事，被强行拆成两个类，拆完后反而高度耦合 职责边界划错了
历史演化型 最初是单向依赖，随着需求迭代逐步反向调用，最终“长”成一个环 设计债未及时清理

1.1 共享能力型

典型场景：

本质：地址校验是一段公共能力，它不属于 OrderService 也不属于 UserService。只是历史上最先写在了 OrderService 里，后来者只能”借用”，借用多了就变成了环。

1.2 流程协作型

典型场景：就是引言里的种植—预警例子。

本质：

这里的问题不是“共用一段逻辑”，而是两个模块在协作完成一个业务流程。

也就是说，它们之间的关系是：

  • A 发生变化
  • B 基于这个变化进行处理
  • B 处理结束后，再通知 A 或影响 A 的后续状态

这类场景本身并不罕见，问题在于：

流程协作不等于互相直接调用。

如果把“流程上的先后关系”直接翻译成“代码上的双向依赖”，就很容易形成环。

1.3 职责拆分过细型

典型场景：

本质：

表面上看，这是把“校验”和“审批”拆开了，但实际上，这两个部分在业务上并不能真正独立：

  • 校验结果直接决定审批是否继续
  • 审批状态又反过来影响校验后的处理方式

这说明拆分出来的两个类并不是两个稳定职责，而是被人为切断的一件事。

1.4 历史演化型

典型场景：不是一次性设计出来的，而是”长”出来的。

本质：

这类循环依赖往往不是一次设计出来的，而是在多次需求迭代中逐步堆积出来的。

每一步看都合理：

  • 创建订单后通知物流，很合理
  • 发货后更新订单状态，也合理
  • 取消订单时同步物流，还是合理

但如果始终只关注局部需求，而不回头审视模块边界，最终就会形成一个结构性的环。

2. 三个诊断问题

拿到一个循环依赖，不要立刻开始“解耦”。

先判断一件事：

这个环到底是代码结构的问题，还是业务协作的问题？

下面用”种植模块 ↔ 预警模块”的真实场景来拆解。

  • 种植模块管理空间数据（如地块边界），是空间数据的唯一写入入口。
  • 预警模块负责空间校验（重叠检测、耕地边界比对），这是它的独占专业能力。
  • 业务流程：导入种植信息 → 触发预警校验 → 管理员处理 → 修正种植数据。

于是形成了这样一个环：

这个环要不要拆、怎么拆，不能凭感觉，要靠下面三个问题来判断。

问题一：A 调用的，是公共能力，还是 B 的独占专业能力？

这是第一步，也是最关键的一步。

  • 如果这段能力有多个消费者，说明它本质上是公共能力，应该独立出来。
  • 如果这段能力只有 A 在使用，且它本来就是 B 模块存在的核心理由，那它就是 B 的独占专业能力，不应该为了“去环”而硬抽出去。

套到当前场景：

如果强行抽走，AlertService 反而会被抽成一个空壳。

问题二：回调修改的数据，归谁所有？谁有写入权？

这是判断“能不能直接回调”的关键。

  • 如果数据归 A 所有，且只有 A 有写入授权，那 B 就不应该直接调用 A 的写方法。
    B 能做的是告诉 A：“我这边处理完了，结果是这样。”
    至于 A 要不要改、怎么改，由 A 自己决定。
  • 如果 B 对这份数据本来就有明确写入权，那直接调用可以接受，但这种授权关系必须是清晰、显式的，而不是靠“互相注入”默认成立。

套到当前场景：

这一步的核心不是“为了松耦合而用事件”，而是：

数据的拥有者，应该自己决定如何修改自己的数据。

问题三：这个环是代码写歪了，还是业务本来就需要这样协作？

前两个问题判断“能力归属”和“数据归属”，第三个问题判断“这个环该不该被消灭”。

  • 如果问题来自错误拆分、能力错放、边界混乱，那这是设计问题，应该修正结构。
  • 如果问题来自真实的业务流程协作，那这个流转本身不需要被消灭，应该改变的是协作方式，而不是业务事实。

套到当前场景：

这两个动作都符合业务事实，不是拍脑袋多出来的流程。

因此结论是：

所以解法不是消灭这个流程，而是把它改成更合适的协作方式，比如事件驱动。

三问串起来

3. 诊断结论与解法的对应关系

诊断结论 推荐解法
这是共享能力型，公共逻辑放错位置了 抽取第三方公共模块
这是流程协作型，且数据只能由一方写入 用事件驱动代替双向直接调用
这是流程协作型，且另一方本就拥有写权限 直接调用，但显式声明授权边界
这是职责拆分过细或边界错误 重新划分职责，必要时合并
这是历史演化导致的设计债，短期无法重构 先用延迟注入、框架补丁等方式止血，再逐步重构

以上是诊断框架。 下面第四节展开四种解法的具体代码与取舍，第五节用种植↔预警的完整案例走一遍从诊断到落地的全过程。

四、四种解耦方式及选择指南

1. 方式一：抽取第三方模块（共享能力的解法）

1.1 核心思想

把 A 和 B 互相需要的那部分能力，抽到一个独立的 C 中。让依赖关系从”环”变成”星”：

A 和 B 不再互相知道对方。它们都只依赖 C。

1.2 适用场景

  • 公共校验（地址校验、参数校验、业务规则校验）
  • 公共查询（按相同条件查同一张表、组装相同的 VO）
  • 公共规则计算（折扣计算、税率计算、积分计算）
  • 公共转换逻辑（DTO ↔ Entity ↔ VO 的映射）

1.3 代码示例

三个类的关系变成了：OrderService → AddressValidator ← UserService。环消失了。

1.4 优点

  • 职责最清晰——AddressValidator 只做一件事，所有人都知道去哪找它
  • 最容易理解和维护——新人看代码不会困惑”为什么 OrderService 要注 UserService”
  • 依赖关系从环变成星型，测试变得轻松

1.5 缺点

  • 不是所有场景都适合抽。如果你抽出来的 C 叫 PlantingAlertCoordinator，里面做的事是”先调种植、再调预警、再回调种植”——那你这不叫抽取，叫把环换了个地方放
  • 强行抽会造成”万能中间类”或”协调器垃圾桶”——一个类依赖所有人，所有人依赖一个类，只是把环换成了辐射状混乱

2. 方式二：事件驱动（流程通知的解法）

2.1 核心思想

不直接调用对方。而是发布一个”已经发生的事实”，由关心这个事实的模块自行处理。

A 和 B 之间不再有直接的代码依赖。它们只依赖事件对象本身。

2.2 适用场景

  • A 的变化需要触发 B 的处理，B 处理完又需要影响 A
  • 但双方在业务语义上不应该直接控制彼此
  • 未来可能还有其他模块也需要关心这些变化（扩展性需求）

2.3 代码示例

两个 Service 之间没有任何直接的代码依赖。PlantingService 的 import 列表里不会出现 AlertService，反之亦然。

2.4 优点

  • 彻底解耦：两个模块在代码层面不再互相知道对方
  • 扩展性极强：后续第三个模块需要关心种植更新，加一个 @EventListener 即可，无需改动现有代码
  • 符合业务语义：”你变了，我处理；我处理完，通知你”——这种关系天然就是事件驱动，而不是互相调用

2.5 缺点

  • 调用链不再直观：直接调用的代码，Ctrl+Click 可以一路跟下去。事件的调用链是”发布—订阅”，IDE 无法直接追踪”谁在监听这个事件”
  • 调试难度更高：出问题时，你需要在日志中关联事件发布和事件处理的时间线
  • 需要设计好事件语义：事件名字不好，解耦就变成换了一种形式的耦合

2.6 事件设计最重要的原则

事件表达的是”事实”，不是”命令”。

错误的事件命名 正确的事件命名 为什么
请修改种植信息事件 预警处理完成事件 前者是命令，后者是事实
触发预警事件 种植信息已更新事件 前者在指挥别人做事，后者在陈述发生了什么
回写状态事件 预警结果已产生事件 前者是”你去做 XXX”，后者是”XXX 已经发生了”

“请修改 XX 信息事件”本质还是隐式耦合——你只是把”直接调方法”换成了”发事件指挥别人”。真正的解耦是：我只说我经历了什么，谁关心谁自己来处理。

关于”命令事件”和”事实事件”的更深入对比，见第六节误区四。

2.7 一个实用的简化：部分事件化就够了

回到种植↔预警的例子：

要解环，不一定两个方向都改成事件。只要把其中一个方向改成事件，环就破了：

方案 B 更轻量：

PlantingService 仍然直接依赖 AlertService，但 AlertService 不再依赖 PlantingService。依赖方向从双向变成了单向，环消失了。改动最小，代价最低。

那什么时候用方案 A，什么时候用方案 B？

判断依据 用方案 B（单向事件化） 用方案 A（双向事件化）
调用者是否需要同步结果？ 不需要，调完就走 需要结果才能继续（保留直接调）
这个事件未来会有更多消费者吗？ 不会，暂时就这一个 会，其他模块也想关注这件事
两个模块之间还有”业务语义上的控制关系”吗？ 有——A 指挥 B 做事是合理的 没有——双方只是”我变了你跟进”的关系
代码改动的风险/成本 改一处就行，风险低 两边都改，成本更高

套到种植↔预警场景：PlantingService 调 AlertService.evaluateAlert() 是在说”请根据新数据做评估”，这是业务上合理的直接指挥（种植需要预警立即被评估）；反过来 AlertService 回调 PlantingService.updateAlertStatus() 只是”结果出来了”，没有指挥语义。所以方案 B 更自然：保留种植→预警的直接调用，把预警→种植的回调改为事件。

一个简单的经验法则：“你做这个”可以直接调，”我做完了”发事件。

3. 方式三：重新划分职责边界（职责混乱的解法）

3.1 核心思想

不是加一层，不是换一种注入方式，而是承认一个事实：A 和 B 现在的边界画错了。

有时候，循环依赖不是”缺少某个机制”，而是模块划分本身就是错的。

3.2 适用场景

  • A 和 B 的方法互相频繁调用，你中有我我中有你
  • 尝试抽第三方，但发现抽不干净——剩下的逻辑还是互相依赖
  • 在业务语义上，A 和 B 本来就是一个整体，只是被人为切开了
  • 两个模块之间的方法调用不像是协作，更像是”你帮我干半件事，我再帮你干半件事”

3.3 可选处理

3.3.1 合并模块

如果两个类拆开后反而耦合更重，那说明它们根本不该拆。

一个类的内部可以自由调用自己的私有方法，不存在循环依赖。单一职责原则说的是”一个类只负责一件事”——“审批订单”就是一件事。”校验”和”执行”是这件事的两个步骤，不一定是两个类的职责。

3.3.2 重新分层

如果 A 和 B 不是该不该拆的问题，而是谁管谁的问题：

PlantingAlertCoordinator 依赖 PlantingService 和 AlertService，但后两者彼此不互相依赖。环变成了有向无环图（DAG）。

注意区分：协调器是否有效，取决于你做了什么。如果 PlantingService 和 AlertService 的职责没有重新划分，只是把原来互相调用的代码搬到了协调器里——那就是换了个地方放环（见第五节”为什么不适合抽第三方”的详细分析）。有效的协调器要求：领域服务先瘦身到只做自己的核心职责，协调器再负责编排。

3.3.3 分离状态与规则

4. 方式四：延迟注入 / 非构造器注入（实在改不动的过渡方案）

4.1 核心思想

借助框架机制，让依赖”晚一点拿到手”。技术上让环能跑起来，设计上心里清楚这是妥协。

4.2 适用场景

  • 老系统改造，涉及 30 个模块、50 个调用链，重构成本过高
  • 临时止血——线上问题需要快速修复，不能花两周重构
  • 团队对业务理解还不够深，不确定怎么拆最合理——先跑着，想清楚了再动

4.3 常见做法

4.4 优点

  • 快速，改动小，先让系统跑起来
  • 对现有代码几乎零影响

4.5 缺点

  • 没有解决任何设计问题——环还在，只是框架帮你藏起来了
  • 容易隐藏风险——启动不报错不代表设计没问题，下次加功能可能就炸
  • Setter 注入和 ApplicationContext 获取让依赖关系不再显式——你无法通过构造器一眼看出这个类依赖谁

5. 一句话选择指南

共享能力，抽第三方；状态通知，用事件；职责混乱，重划边界；实在改不动，再用框架补丁。

五、实战案例：种植模块 ↔ 预警模块

1. 场景还原

回顾第三节中提到的农业信息化系统场景：

  • 种植模块（PlantingService）：管理种植信息的增删改查，是空间数据的唯一写入入口
  • 预警模块（AlertService）：负责空间校验（重叠检测、耕地边界比对等），是系统中唯一承担该职责的模块

业务流程：种植信息变更 → 触发预警重新评估 → 预警处理完 → 回写种植记录。

于是代码变成了：

构造器注入 → 启动报错 → 加 @Lazy → 跑起来了。但每一次加新功能，两个模块的边界就更模糊一点。

2. 用诊断框架分析

套用第三节的三个问题，逐问分析：

问题一：种植调预警做空间校验——这是公共能力，还是预警模块的独占专业能力？

空间重叠检测、耕地边界比对——整个系统中只有预警模块承担这项职责，没有第二个消费者。这是预警模块的独占专业能力，不应该为了”去环”而硬抽出去。如果强行抽走，AlertService 会被抽成一个空壳。

→ 不是共享能力型，不能抽第三方。进入问题二。

问题二：预警处理完后回写种植数据——这份数据归谁所有？谁有写入权？

空间数据（地块边界、种植面积）归种植模块管理，种植模块是唯一的写入入口。预警模块没有空间数据的写入授权。它不应该直接调用 plantingService.updateSpatialData()。

→ 数据只归一方所有，不应该直接回调。进入问题三。

问题三：这个环是代码写歪了，还是业务本来就需要这样协作？

“种植信息变更 → 触发空间校验”合理，”校验完成 → 修正种植数据”也合理。这两个动作都符合业务事实，不是拍脑袋多出来的流程。因此这个环不是”错误业务”，而是正确业务被错误地翻译成了双向直接调用。

→ 业务协作本身没毛病，问题出在协作方式。

三问结论：独占能力（不抽）→ 数据单方归属（不直接回调）→ 业务协作合理（不改流程）。典型解法：事件驱动——保持流程，改变协作方式。

3. 为什么不适合抽第三方

有人会说：”那抽一个 PlantingAlertService 出来，把互相调用的逻辑都放进去不就行了？”

看起来环消失了，但实际上：

  • PlantingAlertCoordinator 这个类本身没有业务含义——它只是一个”流程拼装器”
  • 种植和预警的流程不是一成不变的——需求增加（如”预警后还可能要生成工单”），这个类会越来越臃肿
  • 种植模块和预警模块依然通过 PlantingAlertCoordinator 间接耦合——只是把环从两个类之间移到了三个类之间

抽第三方的本质是把”共同的能力”提出来。 但这里种植和预警没有共同的能力——它们有共同的流程。流程不应该被”抽出来”，应该被编排。事件驱动就是一种编排方式。

5. 关键设计要点

第一：事件表达事实，不表达命令

  • PlantingInfoUpdatedEvent（种植信息已更新）——这是一个事实
  • TriggerAlertEvent（触发预警）——这是在命令别人做事

前者是”我发生了什么”；后者是”你给我做什么”。事件的精髓是前者。

第二：每个模块只关心自己的职责

  • 种植模块只关心”种植信息的增删改查”和”响应预警结果”——它不需要知道预警怎么算的
  • 预警模块只关心”根据信息评估预警”——它不需要知道种植的增删改查逻辑

第三：从”互相控制”变成”基于事实的协作”

旧的代码是”我调你、你调我”，双方牢牢绑在一起。新的代码是”我发布我的事实，你发布你的事实，各自决定要响应什么”。这是一种更松散、更有弹性的协作方式。

六、常见误区

1. 误区一：”能跑就算解决了”

用字段注入或 @Lazy 让代码跑起来，不代表设计合理。

Spring 的三级缓存机制可以让循环依赖不报错，这本质上是框架在帮你容忍技术层面的耦合，而不是帮你消除设计层面的问题。环还在，你每次改代码时依然要同时理解两个模块；你每次写测试依然要 Mock 两个互相依赖的对象。

“不报错” ≠ “没问题”。跑起来的循环依赖只是没有症状的病。

2. 误区二：”有循环依赖就一定抽第三方”

这是最常见的条件反射——看到环就想抽一个 C 出来。

但如果是流程协作型（A 触发 B，B 回写 A），抽第三方只会把一个”两两之间的环”变成一个”三人之间的环”——PlantingAlertCoordinator 看起来像是个新模块，实际上只是把环换了个地方放。

抽第三方的前提是”它们在共享能力”。如果它们是在协作流程，该用的不是抽第三方，是事件驱动。

3. 误区三：”用了事件就一定高级”

事件不是银弹。

如果你的场景是 OrderService 调 UserService.getUserName() 只是为了显示一个名字——这不需要事件。加事件只会让一个本来 Ctrl+Click 就能跟过去的方法调用，变得需要搜事件名才能找到处理者。

事件的代价是”调用链不再直观”。这份代价应该花在值得的地方——流程协作、一对多通知、跨模块解耦。如果只是一个简单的同步查询，直接调用就行。

4. 误区四：”把事件当命令”

这是事件驱动中最隐蔽的坑。

区别在于：前者在指挥别人做事，后者在告诉别人发生了什么。前者只是把 plantingService.updateStatus() 换成了 eventPublisher.publishEvent(new ModifyPlantingStatusEvent(...))——调用方式变了，耦合的本质没变。

如果你发事件时心里想的是”去让那个模块做 XXX”，那这个事件大概率是命令，不是事实。发事件时你应该想的是”我这边发生了什么，关心的人自己来处理”。

七、总结

1. 核心结论

循环依赖通常不是注入方式的问题，而是职责边界和协作方式的问题。

框架提供了很多方法让你”容忍”循环依赖——@Lazy、Setter 注入、三级缓存、提前暴露半成品对象。但它们都是在技术层面帮你的设计问题兜底。

真正的解法不是问**”Spring 怎么才能让我跑起来”，而是问“我为什么会写出 A 和 B 互相依赖的代码”**。

2. 解耦的真正目标

解耦不是”让他们不再互相引用”就完事了。解耦有三个层次：

  1. 代码层面的解耦：消除直接的 import 依赖，让模块可以独立编译、独立测试
  2. 认知层面的解耦：让开发者理解 A 时不需要同时理解 B，降低心智负担
  3. 演化层面的解耦：让 A 的变化不牵动 B，B 的增加不阻塞 A

事件驱动、抽第三方、重划边界——这些手段的意义不是为了”消除一个环”，而是为了达到这三个层次的目标。

解耦的本质，是把”互相控制”变成”基于边界的协作”。

这要求我们：想清楚每个模块”拥有什么数据”、”提供什么能力”、”不干涉什么边界”。当 A 调用 B 时，不是”我需要你帮我干活”，而是”我完成了我的部分，现在轮到你了”——两者之间有明确的契约和边界，谁也不能越界。

真正优秀的设计，不是让 A 和 B 勉强能互相注入，而是让 A 和 B 即使彼此不知道对方，也能把业务协作完成。
