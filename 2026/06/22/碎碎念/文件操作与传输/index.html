文件操作与传输

发表于 2026-06-22 更新于 2026-06-25 分类于 碎碎念 ， 文件操作与传输

一、HTTP 文件传输基础

1. multipart/form-data 协议

1.1 为什么需要一种新的编码格式？

HTTP 最初只有两种请求体编码方式：

编码方式 Content-Type 适用场景 局限性
URL 编码 application/x-www-form-urlencoded 简单键值对（如登录表单） 只能传文本，无法传二进制
纯文本 text/plain 纯文本数据 无结构，无法区分字段边界

核心矛盾：文件是二进制数据，无法进行 URL 编码。即使强行 Base64 编码后放入 URL 编码的表单，体积也会膨胀约 33%（Base64 膨胀） × 3（URL 编码对非 ASCII 字符的膨胀）= 近 4 倍的传输体积浪费。

multipart/form-data 的使命： 允许在一个 HTTP 请求中，同时传输文本字段和原始二进制文件，且不做任何编码转换。

1.2 报文骨架

核心要素：

要素 说明
boundary 请求头声明，用于分割各 Part；随机生成以防止与文件内容冲突
--boundary 每个 Part 的开始标记
--boundary-- 结尾标记——服务器以此判断请求体已全部接收，若缺失会一直等待直至超时
Content-Disposition 每个 Part 的元数据头，包含 name（字段名）和可选的 filename（存在时后端才识别为文件流）
Content-Type 仅文件 Part 携带，标识 MIME 类型；文本字段无此头

浏览器随机生成 boundary（如 ----WebKitFormBoundary + 16 位随机字符）以防止与文件内容冲突。详细的 Part API 解析流程见 [[JavaServlet]] 笔记。

1.3 流式解析与 fileSizeThreshold

Servlet 容器（Tomcat）不会先把整个请求体加载到内存再解析，而是边读 Socket 流边扫描 boundary 分隔符——一个 2GB 的文件上传不会导致 2GB 的内存分配。

fileSizeThreshold 决定了 Part 数据暂存内存还是写入临时文件：

  • 阈值太低（如 0）→ 所有文件都写磁盘，小文件 I/O 开销大
  • 阈值太高（如 10MB）→ 内存被大量上传文件占用，GC 压力大
  • 建议：设置为 1KB~4KB，大部分头像/小图标走内存，大附件走磁盘

请求结束后，临时文件会被容器自动删除。

2. 文件在 HTTP 中的编码方式

2.1 原生二进制（Binary）

  • 场景：multipart/form-data 文件上传
  • 原理：文件字节直接写入 HTTP 请求体，不做任何编码转换
  • 优点：零膨胀，传输效率最高
  • 缺点：只能在 POST/PUT 请求体中使用，不能嵌入 JSON/XML 等文本结构中

2.2 Base64 编码

  • 场景：需要在 JSON 结构中内嵌文件数据
  • 原理：将 3 字节的二进制数据编码为 4 个 ASCII 可打印字符（A-Z, a-z, 0-9, +, /）
  • 体积公式：Base64大小 = ceil(原始大小 / 3) × 4 ≈ 原始大小 × 1.33
  • 优缺点权衡：
优点 缺点
文本安全，可嵌入任何文本协议 体积膨胀约 33%
不依赖 multipart 请求格式 编解码消耗 CPU
前端可配合 DataURL 预览图片 不适合大文件（体积 + 解码开销）

Base64 有两种常见的应用模式：嵌入 JSON/XML 用于接口传输（见三-2），或封装为 DataURL 用于前端预览（见四-2）。

3. Content-Type（MIME）—— 文件传输的身份标识

Content-Type 是 HTTP 中描述数据格式的头，浏览器靠它决定如何处理收到的内容。

3.1 上传时：声明 vs 校验

上传时，浏览器在文件 Part 的 Content-Type 头中声明 MIME 类型：

但这个值来自浏览器的自动推断（基于文件扩展名），不可信任。恶意用户可以伪造 Content-Type: image/png 上传可执行文件。因此服务端必须做魔数校验（读取文件头几个字节验证真实类型），详见五-3.1 文件类型校验。

3.2 下载/访问时：决定浏览器行为

服务端返回文件时设置 Content-Type，浏览器据此决定渲染方式：

Content-Type 浏览器行为
image/png 直接渲染显示
application/pdf 内嵌预览或调用 PDF 插件
text/html 渲染为网页（注意 XSS 风险）
application/octet-stream 不尝试渲染，触发下载

application/octet-stream：二进制流，未知格式。浏览器不会尝试渲染，通常触发下载。当你不确定文件类型时，用它可避免浏览器意外执行不可信内容。

Content-Type 设错，浏览器行为就会异常——PDF 被当成文本渲染成乱码，HTML 文件被直接执行造成 XSS。

4. Content-Disposition —— 决定浏览器的处理方式

Content-Disposition 头告诉浏览器内容应该”展示”还是”下载”。三种角色对应对文件传输的三个场景：

值 场景 浏览器行为 本篇章节
form-data 上传 标记 Part 为表单数据，携带 name 和可选的 filename 本章 1.2
inline 预览 浏览器内直接展示（图片、PDF 等可渲染格式） 四、文件获取与预览
attachment 下载 强制弹出保存对话框，不尝试渲染 四、文件获取与预览

4.1 inline vs attachment

同一份 PDF，inline 让用户在浏览器里看，attachment 强制下载到本地。头像用 inline，报表导出用 attachment。

4.2 中文文件名

filename 参数只支持 ASCII，中文文件名需通过 RFC 5987 的 filename* 编码：

两者同时提供：filename 作为旧浏览器的降级方案，filename* 覆盖现代浏览器

二、文件存储方案

存储方案的选择是文件上传架构中影响最深远的决策。一旦上线并积累了 TB 级数据，迁移成本极高。

1. 本地磁盘存储

1.1 存储位置选择：避开“易失性”目录

在选择存储路径时，必须遵循**“外部化存储”**原则，严禁存储在项目运行目录下。

存储方案 路径获取方式 风险点 结论
项目运行目录 getRealPath("/uploads") 致命伤：重启或重新部署（War/Jar）时，该临时目录会被清空，导致用户文件丢失。 严禁使用
相对路径 new File("uploads/") 不确定性：相对于工作目录。工作目录会随启动脚本的位置改变而“偏移”，导致找不到文件。 不推荐
外部绝对路径 从配置文件读取，如 /opt/data/ 稳健：与项目代码完全解耦。不受重启、部署影响，易于备份。 最佳实践

1.2 文件访问的两类方案

存储到外部绝对路径后，浏览器无法直接通过物理路径（如 D:/...）访问文件，需通过以下方式暴露：

A. 预览接口（推荐：可控性强）

手动编写 Controller，读取文件流并回显。

  • 优势：可以轻松加入权限校验、动态水印、防盗链逻辑。
  • 适用：私密文档、用户头像、付费资源。

B. 虚拟路径映射（简单：性能好）

通过 Spring MVC 配置，将 URL 直接映射到磁盘物理路径。

  • 实现：registry.addResourceHandler("/files/**").addResourceLocations("file:" + uploadPath);
  • 适用：完全公开的静态资源，追求开发效率。

1.3 目录设计原则：应对系统性能瓶颈

不要将所有文件塞进同一个文件夹。当单目录文件数超过 10,000 个时，操作系统（NTFS/ext4）的文件检索性能会大幅下降。

推荐结构：按业务与日期分级

1.4 平台差异与跨平台处理

不同操作系统的路径处理逻辑存在差异，硬编码路径会导致代码无法在服务器（Linux）上运行。

路径分隔符 正斜杠 / 反斜杠 \
大小写 敏感（a.jpg != A.jpg） 不敏感
典型坑点 文件夹需手动 chmod 赋予权限 路径总长度受限（260字符）

2. 阿里云 OSS / AWS S3（对象存储）

2.1 为什么需要云存储？

维度 本地磁盘 OSS/S3
容量 受限于服务器硬盘（扩展需停机加盘） 近乎无限，自动弹性扩容
可靠性 磁盘损坏 = 数据永久丢失 多副本冗余（99.999999999% 数据持久性）
访问速度 依赖服务器带宽和地理位置 全球 CDN 加速，就近访问
安全 需自行防护（防火墙、WAF、DDoS） 内置防盗链、签名 URL、WAF 防护
成本模型 一次性硬件投入 + 运维人力 按量付费（存储费 + 流量费 + 请求费）
图片处理 需自行搭建图片服务器 内置图片处理（缩放、裁切、水印）— URL 参数即可

什么时候不适合上云？

  1. 内网隔离环境（如政务系统、军工系统）—— 数据不允许出机房
  2. 超高频读写（如每秒 10 万次文件读写）—— OSS 的请求费用会很高
  3. 延迟极度敏感（如微秒级文件访问）—— 网络 I/O 比本地磁盘 I/O 多一层延迟

2.2 两种上传架构

文件上传到 OSS 有两条路径，架构选型决定了安全边界和可控性：

2.2.1 后端中转上传（推荐）

文件先经过应用服务器，再做安全校验，最后转发到 OSS：

优点：所有校验逻辑集中在一处，可控性强；前端无需感知 OSS 的存在。
缺点：文件流经过应用服务器，占用带宽和内存；大文件并发上传时服务器压力大。

2.2.2 前端直传（需后端签发凭证）

浏览器直接从客户端上传到 OSS，后端只负责签发临时凭证：

优点：文件不经过应用服务器，节省带宽；上传进度可直接反馈给用户；天然支持分片+断点续传。

缺点——这正是为什么它不总是最佳选择：

  • 绕过服务端校验：文件直接从浏览器到 OSS，后端没有机会做类型校验、大小限制、病毒扫描。用户可能上传任意内容。
  • 依赖回调机制补校验：可以在 OSS 上配置上传回调（UploadCallback），让 OSS 在上传完成后通知后端做异步校验，但这已经晚了——恶意文件已经进了 Bucket。
  • 凭证泄露风险：虽然 STS Token 有有效期和最小权限，但前端环境不可控，Token 可能被 XSS 窃取。

2.2.3 如何选择？

场景 推荐方案 原因
用户头像、证件照等小文件 后端中转 文件小，服务器转发开销可忽略；需要严格校验（类型、尺寸、内容合规）
内部管理系统的文件上传 后端中转 需要记录操作日志、做权限校验，走后端是自然选择
大文件（>100MB）、视频上传 前端直传 + 回调 大文件走服务器中转带宽压力大；配合上传回调做异步校验
高并发上传（如直播截图） 前端直传 避免应用服务器成为瓶颈
公开 UGC 内容（社区发帖图片） 前端直传 + 回调 对实时校验要求不高，异步审核可接受

核心原则：只要能走后端中转，优先走后端。只有在大文件或高并发场景下，前端直传的带宽优势才值得你接受”绕过服务端校验”这个代价。

3. MinIO（私有化对象存储）

对于内网环境、数据合规要求高的场景，MinIO 是最流行的自建 S3 兼容存储。

3.1 MinIO 是什么？跟本地磁盘存储有什么区别？

本地磁盘存储和 MinIO 虽然都是”自己的机器”，但架构层面有本质区别：

本地磁盘存储本质上是文件系统操作——FileOutputStream 写到一个目录，读的时候用 FileInputStream 读回来。文件跟应用服务器绑定在一起。

MinIO 本质上是一个对象存储服务——它跑在独立的进程里，对外暴露 S3 兼容的 HTTP API。应用通过 PutObject / GetObject 这样的 API 调用来存取文件，而不是直接操作文件系统。

维度 本地磁盘 MinIO
架构 文件系统目录，跟应用同机 独立服务进程，通过网络 API 访问
文件访问 java.io.File / Path API S3 兼容 HTTP API（PutObject / GetObject）
多服务共享 需要 NFS/SMB 挂载，配置复杂 原生 HTTP 协议，任何服务都能访问
扩展性 垂直扩展（换更大的硬盘），有上限 水平扩展（加节点组成集群）
高可用 单点故障，磁盘损坏 = 数据丢失 多节点副本，单节点宕机不影响服务
访问控制 依赖 OS 文件权限（chmod） 内置 Bucket Policy + 签名 URL
运维成本 几乎为零 需要部署 + 监控 + 备份策略

3.1.1 什么时候适合本地磁盘存储？

MinIO 并不是本地磁盘的”升级替代品”——它们解决不同规模的问题。以下场景本地磁盘就够了：

  • 单机部署：只有一个应用实例，没有多服务共享文件的需求
  • 文件量小：几百到几千个文件，不需要分布式文件系统
  • 开发/测试环境：快速搭建，不想额外维护一个 MinIO 容器
  • 嵌入式场景：工控机、边缘设备等硬件资源有限的环境

3.1.2 什么时候需要 MinIO？

当”本地磁盘不够用”的信号出现时，就是 MinIO 登场的时候：

信号 说明
多服务共享 后端拆成了多个微服务，都要读写同一批文件——本地磁盘没法跨机器访问
不能上云但有对象存储需求 数据合规/内网隔离，但需要 S3 API（签名 URL、Bucket Policy、生命周期管理）
为将来上云做准备 现在跑在 MinIO 上，将来迁移到 OSS/S3 时改个 endpoint 即可——API 完全兼容
需要高可用 单机磁盘坏了文件就丢了，MinIO 集群模式有纠删码冗余
大文件、高并发 本地磁盘的 I/O 瓶颈是单机硬盘，MinIO 集群可以分散到多节点

3.5 Presigned URL：前端直传与临时下载

这是 MinIO 在工程中最常用的模式——后端签发临时凭证，前端直接与 MinIO 交互，无需经过后端中转文件流。既节约服务端带宽，又提升上传/下载速度。

3.4.1 上传签名 URL（PUT）

前端直传 MinIO 的完整链路：后端签发 → 前端 PUT → 成功回调。

与分片上传的结合：分片上传的每个分片可以各自申请一个签名 URL（对应同一个 object，但不同的 partNumber），前端并行 PUT 所有分片，最后用 completeMultipartUpload 合并。MinIO SDK 的 listParts + abortMultipartUpload 还能支持断点续传与废弃清理。

3.4.2 下载签名 URL（GET）

签名 URL 有效期的策略设计：

场景 建议有效期 理由
敏感文件下载（合同、报表） 60 秒 短有效期防泄露，点开即看，不保存链接
用户头像、缩略图 600 秒（10 分钟） 页面打开后一段时间内缓存有效，避免每次刷新都重新签名
视频流（m3u8 + ts 分片） 3600 秒（1 小时） 播放过程中可能跨越多个分片，需要足够长的有效期
大文件批量导出 1800 秒（30 分钟） 打包可能需要时间，保留一定缓冲

3.4.4 安全注意事项

  1. 限制上传 Content-Type：签名时携带 Content-Type 参数，MinIO 会在接收时校验，拒绝类型不匹配的上传。
  2. 限制文件大小：在 Backend 的 /upload/policy 接口中加入权限和大小校验，不要单靠 MinIO。
  3. objectName 防篡改：objectName 必须是后端生成的 UUID 或规则字符串，不能由前端直接传。否则用户可以通过修改文件名覆盖他人的文件。
  4. 设置 Bucket 级别的 Policy 兜底：即使签名 URL 泄露，也可以通过 Bucket Policy 限制只能 PUT、不能 DELETE（s3:PutObject 只开 PUT 权限，不开 s3:DeleteObject）。

3.6 纠删码概念与容量规划

纠删码（Erasure Coding）是 MinIO 区别于 FastDFS、本地磁盘等传统存储方案的核心技术壁垒，也是它能在通用硬件上实现企业级数据安全的底层原理。

3.5.1 纠删码 vs 多副本：原理对比

维度 多副本 (Replication) 纠删码 (Erasure Coding)
原理 数据完整复制 N 份（如 3 副本） 数据切分为 M 块数据 + N 块校验码（如 8+4）
冗余率 3 副本 = 200% 冗余 MinIO 默认 (N/2 + N/2) = 100% 冗余
容错能力 3 副本可容忍 2 块盘损坏 8+4 可容忍任意 4 块盘同时损坏
存储效率 33%（3 副本存 1TB 需 3TB 磁盘） 50%（MinIO 默认存的 1TB 需 2TB 磁盘）
恢复速度 慢（完整文件重新复制） 快（只需恢复丢失的数据块 + 校验块）
代表系统 HDFS、Ceph（副本模式） MinIO、Ceph（EC 模式）

MinIO 的默认纠删码策略：数据 + 校验各占一半，即 1 份数据 + 1 份校验 = 总容量的一半用于冗余。如果写入 1TB 原始文件，实际磁盘占用约 2TB。

类比理解：纠删码就像 RAID 6 的”软件升级版”。RAID 6 用 2 块校验盘保护整个阵列，MinIO 的 EC 模式相当于给每个对象独立做了数据保护，不需要硬件 RAID 卡，纯软件搞定。

3.5.2 MinIO 启动时的 EC 自动配置

MinIO 在启动时自动根据可用磁盘数选择纠删码模式，无需手动配置：

磁盘数 纠删码模式 冗余率 可容忍坏盘数
2 块 EC:1 (2 disks) 100% 0
4 块 EC:2 (2+2) 100% 1
8 块 EC:4 (4+4) 100% 4
16 块 EC:8 (8+8) 100% 8

约束规则：MinIO 要求最少 4 块磁盘才能启用纠删码模式。单机单盘部署（如 Docker 示例）是没有纠删码保护的，仅适用于开发测试。

3. 分片上传与断点续传

3.1 为什么需要分片？

问题 单次上传的致命缺陷
内存溢出 整个文件加载到内存 → MultipartFile.getBytes() 对 1GB 文件必然 OOM
超时中断 Nginx 默认 60s 超时、Spring 默认 30s 超时，上传 3GB 文件永远完不成
失败成本高 上传到 99% 后网络中断 → 全部重传，浪费 2.97GB 的已传流量
进度不可知 用户看着进度条不动，不知道是卡住了还是在传

分片上传的本质：将一个大文件切割成多个小块（chunk），独立上传，最后在服务端合并。这样一来，每个分片的传输符合 HTTP 协议的超时限制，某个分片失败只需重传该分片。

3.2 概念一：秒传（文件指纹去重）

秒传是分片上传的第一个”加速器”。它的核心思路非常简单：上传前先问服务端”这个文件你已经有吗？”，有就直接返回已存 URL，省去整个上传过程。

工作流程：客户端在上传前计算文件的 MD5 指纹，通过 /init 接口发送给服务端。服务端在数据库中查询这个 MD5——如果已存在，说明有其他人（或你自己）传过完全相同的文件，直接返回已有文件的访问 URL，上传流程到此结束。

适用场景：

  • 多人上传同一份文件（如团队共享的安装包、培训视频）
  • 修改文件名后重新上传（MD5 不变，文件内容没变）
  • 网盘场景：不同用户上传相同的热门资源

前提条件：

  • 文件指纹使用 MD5 + 文件大小 双重校验（MD5 理论上有碰撞概率，加上 size 可以排除 99.9999% 的情况）
  • 引用计数 > 1 的文件不可物理删除，只能逻辑删除（减引用计数）
  • 秒传判断在 /init 接口中完成，是初始化阶段的第一道检查

秒传的本质：以空间换时间——用一次数据库查询的代价，省去 GB 级别的网络传输。

3.3 概念二：断点续传（状态恢复）

断点续传是分片上传的第二个”保险丝”。核心同样朴素：上传中断后重新开始，只传还没传完的分片，已经传过的不用再传。

工作流程：服务端为每次上传任务分配一个唯一的 uploadId，已上传的分片以独立文件保存在临时目录（如 /tmp/chunks/{uploadId}/）中。当客户端断网重连后，带着 uploadId 重新调用 /init，服务端扫描临时目录，返回”你已经传了哪些分片”的列表。客户端跳过这些分片，只传剩余的。

与秒传的对比：

机制 粒度 判断依据 触发时机
秒传 整个文件 文件 MD5 /init 时检查
断点续传 单个分片 分片文件是否存在 /init 时返回已传分片列表

秒传解决的是”整个文件都不用传”的场景，断点续传解决的是”部分分片已经传了”的场景。两者都在 /init 阶段完成判断，但作用于不同粒度。

注意事项：临时分片目录需要定期清理。如果用户上传了一半就关掉了浏览器，分片数据会残留在磁盘上。建议用 @Scheduled 定时任务，清理超过 24 小时未修改的分片目录。

3.4 前后端交互协议

了解了秒传和断点续传这两个核心概念之后，我们来看分片上传在前后端之间具体如何协作。整体上定义了 3 个接口，各司其职：

接口 方法 职责 幂等性
/chunk/init POST 初始化上传任务：检查秒传 → 创建/复用 uploadId → 返回已传分片列表 ✅ 同一文件多次 init 行为一致
/chunk/upload POST 上传单个分片 ✅ 同一分片重复上传，覆盖写入
/chunk/merge POST 合并所有分片为最终文件，校验 MD5 ✅ 已合并的直接返回结果

完整上传流程：

sequenceDiagram
    participant F as 前端
    participant B as 后端

    Note over F: 1. 计算文件 MD5（增量哈希）

    F->>B: 2. POST /chunk/init {md5, filename, totalChunks, uploadId?}
    Note over B: 检查秒传（MD5 是否已存在）<br/>生成或复用 uploadId<br/>扫描临时目录已上传的分片
    B-->>F: 3. 返回 {uploadId, uploadedChunks}

    Note over F: 4. 遍历分片，跳过已上传的

    loop 逐个分片上传
        F->>B: POST /chunk/upload FormData {chunk, uploadId, chunkIndex}
        Note over B: 写入 /tmp/chunks/{uploadId}/{chunkIndex}
        B-->>F: 分片上传成功
    end

    F->>B: 5. POST /chunk/merge {uploadId, totalChunks, md5}
    Note over B: 按 chunkIndex 排序 → 流式合并<br/>校验最终文件 MD5<br/>清理临时分片目录
    B-->>F: 6. 返回文件 URL

为什么是 3 个接口而不是 2 个？ 有方案将合并逻辑合并到最后一个分片的上传请求中（”最后一个分片上传成功时自动触发合并”），但这存在并发问题：如果多个分片几乎同时到达，哪个才是”最后一个”？独立的 merge 接口让”最后一个”的判定权在前端——前端最清楚自己发了多少个分片——服务端只需老老实实合并，逻辑更清晰。

uploadId 由谁生成？ 推荐后端生成。后端可以保证唯一性、控制生命周期（过期清理）、关联用户身份做权限校验。前端生成 uploadId 虽然少一次 init 请求，但无法防止恶意构造、难以管理过期数据。

    User->>Backend: 1. POST /api/upload/policy
    Note over Backend: 2. 生成 objectName，验权
    Backend->>MinIO: 3. getPresignedObjectUrl (PUT)
    MinIO-->>Backend: 签名 URL
    Backend-->>User: 4. {presignedUrl}
    Note over User: 5. 前端直接 PUT 上传
    User->>MinIO: 6. PUT 签名 URL
    MinIO-->>User: 7. 200 OK
    User->>Backend: 8. POST /api/upload/callback
    Note over Backend: 9. 记录文件元数据
    Backend-->>User: 10. {url: "/files/xx.jpg"}

回调确认：MinIO 不会主动通知后端上传完成，必须由前端主动向后端报告。

4.4 安全注意事项

  1. 限制 Content-Type：签名时携带 Content-Type，存储服务校验匹配。
  2. 限制文件大小：后端签发接口中做大小校验。
  3. objectName 防篡改：对象名必须由后端生成 UUID，不由前端指定。
  4. Bucket Policy 兜底：只开 s3:PutObject，不开 DELETE。

2. DataURL 预览

DataURL 将图片等小文件以 Base64 编码内嵌在 HTML 中，适合上传前的本地预览，无需请求服务端。

  • 格式：data:[MIME类型][;base64],数据
  • 核心 API：FileReader.readAsDataURL()

DataURL 的本质：它是 Base64 编码的前端封装（见一-2.2），并非独立技术。因为编码过程需将整个文件读入内存并 Base64 转换，不适合大文件（>1MB 建议用下一节介绍的 Blob URL）。

优点 缺点
无需服务端支持 体积膨胀 33%
可直接赋值给 <img src> 不能被浏览器缓存
兼容性好 编码大文件阻塞主线程

3. Blob URL 预览

URL.createObjectURL() 是浏览器提供的高效预览方案——它不编码文件，而是在内存中创建一个指向 Blob 对象的临时引用。

为什么比 DataURL 快？

  • DataURL：读取整个文件 → CPU 密集的 Base64 编码 → 赋值给 src
  • Blob URL：创建一个内存引用 → 赋值给 src（零编码、零拷贝）

Blob URL 的局限性：只在当前页面生命周期内有效，刷新后自动失效。这是合理的——它专为”临时预览”而设计，用完就通过 revokeObjectURL 主动释放，不要试图保存到 localStorage。

三种预览/访问方式对比：

方式 原理 膨胀率 可缓存 适合大小 典型场景
直接 URL 服务端文件直出 0% ✅ 任意 线上正式访问
DataURL Base64 内嵌 ~33% ❌ < 1MB 极小图、CSS 内联
Blob URL 内存引用 0% ❌ 任意 本地预览

6. 多文件压缩包下载

用户选中多个文件，服务端实时打包成 ZIP 返回——不需要先打包完再返回，边打包边返回。

内存安全：ZipOutputStream 是流式的——它在写入一个 entry 的数据后，立即 flush 到 outputStream（即 HTTP 响应流）。即使打包 4GB 的视频合集，服务端的内存占用也只有缓冲区大小（默认 8KB）。这是它相对于”先打包到临时文件再返回”方案的核心优势。

五、格式处理与转换

1. 图片压缩与水印

压缩质量与体积的经验值（示例 12MP 相机照片）：

质量参数 输出大小 视觉差异 推荐场景

0.7 ~600KB 轻微噪点 列表缩略图
0.5 ~300KB 明显压缩痕迹 仅用于预览

2. 图片格式转换（WebP）

WebP 是 Google 推出的现代图片格式，相比 JPEG 体积减少 25%~35%，相比 PNG 减少 26%~34%，同时保持同等的视觉质量，是目前 Web 端最主流的图片压缩格式。

格式 同等质量下的典型体积 适用场景
JPEG 100% (基准) 照片、复杂色彩图片
PNG ~150% 图标、截图、需要透明通道
WebP 65%~75% Web 端首选，兼容性 > 97%
AVIF 50%~60% 下一代格式，兼容性 ~92%（Safari 16+ 才支持）

WebP 的局限：不支持 8K 以上的分辨率（JPEG 支持到 65535×65535）。超大图片建议继续用 JPEG 或切块处理。

3. 视频压缩

视频文件通常体积巨大，用户上传后直接存储原始文件成本高昂。视频压缩的核心是在体积、画质、编码时间三者之间找到平衡点。

Java 生态本身不内置视频编解码能力，常见方案基于 FFmpeg（C/C++ 音视频处理库）的 Java 封装。下面给出两种主流方案的对比和核心代码。

3.1 JAVE2（推荐）

JAVE2 是 FFmpeg 的轻量 Java 封装，内置各平台 FFmpeg 静态库，开箱即用。API 以”格式 + 编码器 + 码率 + 帧率”几个核心参数为主，足够覆盖常规压缩、转码、提取音频等场景。

jave-all-deps 包含各平台 FFmpeg 原生库，jar 体积约 150MB。项目体积敏感可用 jave-core + 服务器自行安装 FFmpeg。

参数 作用 建议值
bitRate 视频码率，直接决定画质和体积 1080p: 24Mbps；720p: 12Mbps
frameRate 帧率 常规视频 30fps；监控/动画可降到 15fps
VideoSize 分辨率上限 面向 Web 限制 1920 宽；-2 自动对齐偶数行
preset 编码速度 vs 压缩率 fast（推荐）：平衡；veryslow：最小体积，但耗时数倍

3.2 JavaCV（功能更全，逐帧可控）

如果你需要逐帧处理（加水印、滤镜、AI 识别），JavaCV 提供更底层的帧级 API：

3.3 方案对比与生产建议

维度 JAVE2 JavaCV 直接 ProcessBuilder
上手难度 简单 中等 需懂 FFmpeg CLI
功能覆盖 常规压缩/转码/提取音频 全部 FFmpeg 能力 + 逐帧处理 全部 FFmpeg CLI 能力
jar 体积 ~150MB (-all-deps) ~200MB+ 无额外依赖
适合场景 文件上传后的常规压缩 逐帧加水印、滤镜、AI 分析 服务器已有 FFmpeg，不想加依赖

生产环境关键提醒：视频转码非常吃 CPU，必须异步处理（消息队列 / @Async / 轮询任务表）。单次上传请求中同步转码会导致 HTTP 超时，用户体验极差。建议上传后立刻返回”处理中”，后台转码完成后通过回调或轮询通知前端。

2. 目录穿越攻击详解

攻击原理：文件名中包含 ../ 序列，试图将文件写到上传目录的上层目录。

防护方法（一条代码就够了）：

new File(name).getName() 的原理：File.getName() 返回路径中最后一个分隔符之后的部分。../ 和盘符都会在解析路径时被消耗掉，最终只保留纯文件名。

3. 恶意文件类型防护

攻击者上传一个 .jsp 木马，改扩展名为 .jpg 试图绕过前端校验。如果服务端不做类型防护，这个”图片”被保存到 web 目录后，攻击者访问它就能在服务器上执行任意代码。

3.1.1 扩展名白名单

最轻量的第一道过滤——只允许业务需要的类型：

3.1.2 魔数校验

读取文件头几个字节，比对已知的文件格式签名（详见三-1.2 魔数校验）。扩展名说它是 JPG，魔数说它是 JSP 源码——信魔数。

3.1.3 存储隔离 + 执行权限（兜底）

即使前两层都被绕过，只要文件不在 web 可执行目录且没有执行权限，恶意代码就无法运行：

防线 阻挡的攻击 被绕过的条件
扩展名白名单 脚本文件直接上传 (.jsp, .php) 改名 .jpg 即可绕过
魔数校验 换扩展名的恶意文件 将恶意代码嵌入合法文件头之后（图马）
存储隔离 访问上传文件触发服务端执行 文件已进入 web 可执行目录
执行权限 通过其他漏洞执行上传的恶意文件 攻击者已有 shell 权限

三层防线缺一不可——安全是纵深防御，不是单点校验。

4. 上传配额与 DoS 防护

4.1 文件大小限制

在流量到达应用之前就拦截超大请求：

限制层级 作用 超出时的表现
Nginx client_max_body_size 在请求体接收完之前就断开连接 413 Request Entity Too Large
Spring max-file-size 单个 Part 大小限制 MaxUploadSizeExceededException
Spring max-request-size 整个 multipart 请求总大小 MaxUploadSizeExceededException
