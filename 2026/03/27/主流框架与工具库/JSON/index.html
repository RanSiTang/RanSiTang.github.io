    一、JSON 基础概念

    1. JSON 的定义与历史背景

    1.1 JSON 全称与起源

    JSON 全称：JavaScript Object Notation（JavaScript 对象表示法）

    起源：JSON 由 Douglas Crockford（道格拉斯·克罗克福德）在 2001 年左右发现并推广。他注意到 JavaScript 的对象字面量语法天然适合作为数据交换格式，于是将其规范化。

    与 ECMAScript 的关系：JSON 语法是 ECMAScript 3 对象字面量语法的一个严格子集，这意味着任何合法的 JSON 都是合法的 JavaScript 表达式（但反过来不成立）。

    关键信息 说明
    提出者 Douglas Crockford
    时间 2001 年发现，2006 年 RFC 4627 首次标准化
    当前标准 RFC 8259（2017 年，取代之前版本）
    官方网站 json.org
    本质 与语言无关的纯文本数据格式

    1.2 JSON 与 XML 的对比及 JSON 胜出的原因

    在 JSON 流行之前，XML 是 Web 服务数据交换的主流格式。

    JSON 胜出的关键原因：

    • 轻量级，传输效率高
    • 与 JavaScript 天然契合，前端处理零成本
    • RESTful API 的兴起推动了 JSON 的普及

    1.3 JSON 的 RFC 规范（RFC 8259）

    RFC 演进历史：

    RFC 编号 年份 说明
    RFC 4627 2006 JSON 首个正式标准
    RFC 7159 2014 修订版，澄清细节
    RFC 8259 2017 当前标准，明确 UTF-8 为默认编码

    RFC 8259 的几个关键规定：

    1. 编码：JSON 文本必须使用 UTF-8 编码（之前版本允许 UTF-16/UTF-32）
    2. 顶层结构：JSON 文本可以是任意 JSON 值（不再限制必须是对象或数组）
    3. 数字：不限制精度和范围，但建议实现方注意互操作性

    2. JSON 的应用场景

    2.1 RESTful API 数据交换格式

    最核心的应用场景。现代 Web 开发中，前后端分离架构下，前端与后端通过 HTTP API 通信，JSON 是事实上的标准数据格式。

    sequenceDiagram

    participant Java后端

    前端/客户端->>Java后端: HTTP Request<br/>Content-Type: JSON
    Java后端-->>前端/客户端: HTTP Response<br/>Content-Type: JSON

    2.2 配置文件

    JSON 也常用于应用程序的配置文件，虽然在 Java 生态中 YAML 和 Properties 更常见，但 JSON 配置在某些场景仍有使用。

    常见的 JSON 配置文件：

    配置文件 说明
    package.json Node.js 项目配置（前端项目常见）
    tsconfig.json TypeScript 编译配置
    appsettings.json .NET 应用配置
    launch.json VS Code 调试配置
    自定义业务配置 如规则引擎配置、权限配置等

    2.3 NoSQL 数据库存储格式

    许多 NoSQL 数据库原生使用 JSON（或其二进制变体 BSON）作为数据存储格式，这使得 Java 后端与这些数据库交互时，JSON 处理能力尤为重要。

    数据库 JSON 相关特性
    MongoDB 使用 BSON（Binary JSON）存储，查询语法基于 JSON
    Elasticsearch 文档以 JSON 格式存储，查询 DSL 也是 JSON
    Redis 虽非 JSON 数据库，但常将 JSON 字符串作为 Value 存储
    MySQL 5.7+ 支持 JSON 字段类型，可存储和查询 JSON 数据
    PostgreSQL 支持 JSON 和 JSONB 类型，查询能力强大

    2.4 消息队列中的消息体格式

    在分布式系统中，服务间通过消息队列（如 Kafka、RabbitMQ、RocketMQ）进行异步通信，JSON 是最常用的消息体格式。

    三、Java 中的 JSON 库全景

    1. 主流 JSON 库对比与选型

    1.1 Jackson

    基本介绍：

    Jackson 是 Java 生态中最流行、功能最完善的 JSON 处理库，也是 Spring Boot 的默认 JSON 库。由 FasterXML 组织维护。

    核心特点：

    特点 说明
    Spring 生态默认 Spring Boot 开箱即用，无需额外配置
    功能完善 注解丰富、定制性强、支持各种高级特性
    性能优秀 在主流库中性能表现一流
    社区活跃 文档完善、问题响应快、更新频繁
    模块化设计 核心模块 + 扩展模块，按需引入

    适用场景：

    • Spring Boot 项目（首选）
    • 需要丰富注解和高度定制的场景
    • 企业级项目，追求稳定性和生态支持

    1.2 Gson

    基本介绍：

    Gson 是 Google 开发的 JSON 库，以简洁易用著称。API 设计简单，学习成本低，适合轻量级场景。

    核心特点：

    特点 说明
    简单易用 API 简洁，几乎零配置即可使用
    轻量级 单个 jar 包，无额外依赖
    容错性好 默认对格式不严格的 JSON 有较好容忍度
    Android 常用 在 Android 开发中应用广泛
    Google 维护 品质有保证

    适用场景：

    • Android 项目
    • 轻量级 Java 应用
    • 快速原型开发

    1.3 Fastjson2

    基本介绍：

    Fastjson2 是阿里巴巴开源的 JSON 库，是 Fastjson 的全新升级版本，重点解决了 Fastjson1 的安全问题。

    核心特点：

    特点 说明
    性能极高 官方宣称是最快的 JSON 库
    国内生态 中文文档完善，国内使用广泛
    安全增强 修复了 Fastjson1 的 AutoType 漏洞
    兼容性 提供 Fastjson1 兼容模式，便于迁移

    适用场景：

    • 对性能有极致要求的场景
    • 国内团队，熟悉阿里技术栈
    • 从 Fastjson1 迁移的项目

    1.4 JSON-B / JSON-P

    简要说明：JSON-B（JSON Binding）和 JSON-P（JSON Processing）是 Jakarta EE（原 Java EE）的官方 JSON 标准 API。

    规范 说明
    JSON-P (JSR 374) 底层处理 API，类似 Jackson 的流式 API
    JSON-B (JSR 367) 高层绑定 API，类似 Jackson 的数据绑定

    在标准 Jakarta EE 服务器（如 WildFly、Payara）上开发时可能会用到。但在 Spring Boot 生态中，Jackson 是事实标准，JSON-B 使用较少。

    8. 流式 API

    简要说明：流式 API 是 Jackson 的底层 API，提供最高性能和最低内存占用的 JSON 处理方式，但使用复杂度也最高。一般开发中很少直接使用，由上层的数据绑定和树模型内部调用。

    8.1 JsonParser 逐 token 读取

    JsonParser 以流的方式逐个读取 JSON token（如 {、字段名、值、}），不会将整个 JSON 加载到内存。

    8.2 JsonGenerator 逐 token 写入

    JsonGenerator 以流的方式逐个写入 JSON token，适合生成大型 JSON。

    8.3 适用场景

    场景 说明
    超大 JSON 文件 文件大到无法一次性加载到内存（如几百 MB）
    极致性能 每纳秒都要优化的高频交易场景
    内存受限环境 嵌入式设备或内存非常有限的服务

    为什么一般不用流式 API：

    • 代码复杂度高，容易出错
    • 数据绑定（ObjectMapper）已经足够高效
    • 可维护性差

    建议：除非有明确的性能瓶颈和内存限制，否则使用 ObjectMapper 即可。Jackson 的数据绑定底层已经使用流式 API，性能已经过优化。
