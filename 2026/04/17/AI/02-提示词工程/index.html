02-提示词工程

发表于 2026-04-17 分类于 AI

一、提示词工程基础认知

1. 什么是提示词工程 ？

简单来说，提示词工程就是一门“如何与 AI 高效对话以获得期望结果”的艺术和科学。它研究如何设计和优化输入给大语言模型（LLM）的文本（即“提示词”），从而精确地引导模型产生高质量、相关且符合特定要求的输出。

1.1 与一个“黑盒服务”的 API 交互

对于后端工程师而言，最贴切的理解方式是：

  • 大语言模型 (LLM)：一个部署在云端的、功能极其强大但行为不完全确定的“黑盒微服务”。
  • API 调用：我们通过 HTTP 请求调用这个服务。
  • 提示词 (Prompt)：就是这次 API 请求中，我们精心构造的 payload（请求体），通常是文本。
  • 模型输出 (Completion)：就是该 API 的 response（响应体）。

提示词工程，就如同在研究一个文档不完善、行为有随机性的 API 时，通过不断尝试和总结，摸索出最佳的请求参数（提示词）来获得稳定、可靠的响应。

1.2 核心目标

LLM的输出本质是概率性的、非结构化的文本。而我们的后端系统是建立在确定性的、结构化的数据之上的。提示词工程的核心目标，就是在两者之间建立一个可靠的管道，强制模型输出能被我们的程序稳定地消费。

这个目标可以分解为三个关键点：可预测、稳定、可控。

  • 稳定 (Stable)：
    • 问题：对于同一个提示词和任务，LLM 每次的输出可能都不完全一样，即使内容相似，措辞、格式也可能有差异。这对于需要确定性逻辑的后端系统是致命的。
    • 目标：通过优化提示词，让模型对于同类任务的输出在核心内容和结构上保持高度一致。例如，每次提取发票信息，都能稳定抽取出 金额、日期、发票号码 这几个关键字段，而不会有时多一个字段，有时少一个字段。这有点像编写幂等的服务接口，力求行为一致。
  • 可预测 (Predictable)：
    • 问题：模型默认返回的是自然语言文本，这对程序解析来说是一场灾难。
    • 目标：让输出的格式变得可预测。后端最希望得到的是结构化数据，比如一个严格的 JSON 对象。通过在提示词中明确要求输出格式（例如，提供 JSON Schema），我们可以让模型返回一个能被 Jackson 或 Gson 直接反序列化为 Java Pojo 的字符串，极大地降低了集成成本。可预测性意味着我们可以信赖模型返回的数据结构，从而放心地进行后续的业务处理（如存入数据库、传递给其他服务）。
  • 可控 (Controllable)：
    • 问题：模型可能会自由发挥，输出超出范围的内容、带有不必要的解释、或者产生“幻觉”（捏造事实）。
    • 目标：精确地控制模型行为的边界和范围。这包括：
      • 内容控制：限定模型只能基于你提供的上下文（Context）来回答问题，避免它使用外部知识胡编乱造。
      • 任务控制：明确指令是“翻译”就不要做“总结”，是“代码生成”就不要附加“使用说明”。
      • 安全控制：防止用户的恶意输入（提示词注入）歪曲了你的原始指令，导致模型执行非预期的操作，威胁后端系统安全。

2. 为什么后端工程师必须掌握提示词工程？

掌握提示词工程，并非要我们成为算法专家。而是从系统工程和软件质量的角度出发，将一个充满不确定性的AI能力，转化为可集成、可维护的可靠系统组件所必须具备的核心技能。

2.1 职责所在

我们每天都在与微服务和第三方API打交道。请将LLM看作一个全新的、行为有些“古怪”的微服务：

  • 熟悉的模式：我们通过API Key认证、HTTPS请求、JSON响应与它交互。
  • 全新的挑战（非确定性）：与传统的CRUD服务不同，LLM的响应具有内在的不确定性。
  • 后端的职责：我们的职责不是构建这个AI服务，而是成为它的“首席集成官”。如同我们会为不稳定的网络设计重试、熔断策略一样，我们也必须通过提示词工程来“驯服”这个不确定的AI服务。

核心认知：一个优秀的提示词，就是我们与AI服务之间签订的一份详尽的“服务水平协议 (SLA)”。它用来约束AI的行为，使其输出尽可能地接近确定性，从而能被我们的后端逻辑稳定地消化。

2.2 结构化输入与输出

后端系统的命脉是结构化数据。我们的代码、数据库、ORM都建立在清晰的POJO之上。而LLM天然的输出是自由的非结构化文本。

  • 痛点：直接解析自然语言字符串，意味着编写脆弱、复杂的正则表达式，这种代码难以维护，极易因模型输出的微小变化而崩溃。
  • 解决方案：提示词工程是解决此问题的最直接、最高效的手段。通过在提示词中下达精确的格式化指令，我们可以强制模型返回一个严格的JSON字符串。

3. 提示词的基本构成要素

一个设计精良的提示词，如同一个结构清晰、文档完备的API请求。将其拆解为标准化的构成要素，能极大提升输出的质量与稳定性，并为后续的维护和模板化管理提供便利。

我们可以将一个完整的提示词分解为以下五个核心要素，这里我将其调整为更符合后端开发构建请求的逻辑顺序：

指令 (Instruction) -> 上下文 (Context) -> 输入数据 (Data) -> 角色 (Role) -> 输出格式 (Output Format)

这个顺序并非一成不变，但它模拟了一个典型的后端处理流程：明确任务 -> 提供背景 -> 传入数据 -> 指定执行者 -> 定义返回格式。

3.1 指令 (Instruction) / 任务 (Task)

  • 定义：清晰、无歧义地告诉模型需要执行的具体动作。
  • 重要性：指令是提示词的核心驱动力。模糊的指令必然导致模糊的输出。
  • 最佳实践：
    • 使用强动词：用“生成”、“提取”、“翻译”、“总结”、“分类”、“重构”等词汇开头。
    • 避免模糊表述：杜绝“关于”、“处理一下”、“看看这个”等。
    • 分解复杂任务：如果任务包含多个步骤，请用1, 2, 3…明确列出。这不仅让模型理解更清晰，也便于你在调试时定位问题环节。

3.2 上下文 (Context)

  • 定义：为模型执行任务提供所有必要的背景知识、约束条件或外部数据。

  • 重要性：上下文为模型的“思考”提供了边界和依据。它是对抗“幻觉”最有效的武器。

  • 定义：需要被指令所处理的具体文本或数据。

  • 重要性：必须将输入数据与指令清晰地隔离开，这是防止“提示词注入”攻击的关键。

  • 最佳实践：使用清晰、强力的分隔符来包裹不受信任的用户输入。

3.4 角色 (Role)

  • 定义：为LLM设定一个具体的身份、专家人设或行为模式。
  • 重要性：这是最简单也最有效的技巧之一。通过赋予角色，可以“激活”模型内部与该角色相关的知识和行为模式，让输出更专业。
  • 示例：
    • 无角色：检查这段Java代码。
    • 有角色：你是一位对代码洁癖有极致追求的Java高级工程师，精通《Effective Java》。请审查以下代码...

3.5 输出格式 (Output Format)

  • 定义：对模型返回结果的结构、样式或格式的精确要求。
  • 重要性：对于后端，这是最具实战价值、必须强制执行的部分。它决定了AI的输出能否被程序自动、稳定地解析。
  • 最佳实践（按可靠性排序）：
    1. 提供Schema或模板 (最可靠)：给出期望输出的骨架，让模型填充。对于JSON，直接提供一个带类型的模板。
    2. 结合少样本示例 (Few-Shot)：给出一个完整的“输入 -> 期望格式输出”的例子，让模型模仿。
    3. 明确要求结构：直接下令“请以JSON格式返回”、“请返回一个Markdown表格”。

二、核心提示词设计原则与模式

1. 原则一：清晰与具体

这是提示词工程中最基础、也最重要的原则。大语言模型虽然强大，但它不是读心者。你给它的指令越模糊，它“自由发挥”的空间就越大，结果的不确定性也就越高。后端系统最害怕的就是不确定性。因此，我们必须像编写技术需求文档或API设计文档一样，力求每一个字都清晰、具体、无歧义。

1.1 使用明确的动词和指令，避免模糊表述

将与模型交互的过程，看作是一次精确的任务委托。每一个指令都应该是一个可执行的动作。

  • 反例（模糊表述）：
    • "关于这个bug报告..." (然后呢？要总结？要分析原因？要生成代码？)
    • "处理一下这段用户反馈。" (怎么处理？是情感分析？还是提取要点？)
    • "看看这段Java代码。" (看什么？找bug？做重构？还是加注释？)

这些模糊的指令会迫使模型去猜测你的意图，结果往往不符合预期，甚至南辕北辙。

  • 正例（明确动词和指令）：
    • "**总结**以下bug报告，**提取**复现步骤和期望结果。"
    • "**将**以下用户反馈**分类**为'功能建议'、'Bug报告'或'体验问题'。"
    • "**审查**以下Java代码的**安全漏洞**，特别是与SQL注入相关的风险点。"

1.2 提供足够的、相关的上下文信息，避免模型猜测

如果只给指令，不给上下文，模型就会基于它训练数据中的“通用知识”来作答，而这些知识很可能不适用于你具体的业务场景。

  • 反例（缺少上下文）：

    • 提示词："生成一封催缴邮件。"
    • 模型输出：可能生成一封催缴房租或信用卡账单的通用邮件，与你的业务（比如催缴SaaS服务费）完全无关。

1.3 为复杂任务提供详细的步骤拆解说明

当任务比较复杂时，指望模型一步到位地完美完成，是不现实的。这就像让一个程序员一次性写完一个包含前端、后端、数据库交互的复杂功能，而不做任何设计和拆解，结果必然是混乱的。更可靠的做法是，在提示词中将任务分解为一系列连续的、更小的步骤。

  • 反例（复杂任务一步到位）：
    • "分析这段代码，优化它，并用JSON格式解释你的修改。"

2. 原则二：给予模型“思考”的时间

大语言模型在处理复杂问题时，如果被要求立即给出最终答案，往往会因为“仓促行事”而犯错，这类似于人类在没有充分思考的情况下草率作答。这个原则的核心思想是，通过在提示词中设计特定的结构，引导模型先进行一步步的推理、分析或分解，然后再得出结论。这种“慢思考”的过程能显著提高复杂任务的准确性。

2.1 思维链 (Chain-of-Thought, CoT)

思维链（CoT）是提示词工程中一个里程碑式的技术。它不直接向模型索要答案，而是要求模型在给出最终答案之前，先显式地输出其一步步的推理过程。

2.1.1 CoT 的应用

CoT 在以下场景中对后端工程师尤其有用：

  • 复杂逻辑推理：

    • 场景：假设你需要根据一套复杂的业务规则（如优惠券、会员等级、活动叠加）来判断一个订单的最终价格。这些规则可能相互关联、互斥。

    • 不使用 CoT：根据规则A,B,C，计算订单X的最终价格。 -> 模型可能会直接给出一个错误的价格，你完全不知道它错在哪里。

    • 使用 CoT：请根据以下规则，计算订单X的最终价格。请先一步步列出你的计算过程，最后再给出 "最终价格：" 的结论。

    • 模型输出示例:

    • 价值：这个过程暴露了模型的推理路径。即使结果错误，我们也能轻易地发现是哪一步的规则理解错了，从而可以针对性地优化提示词中的规则描述。它把一个黑盒的判断过程，变成了一个可以审查（Debug）的白盒过程。

  • 代码生成与转换：

    • 场景：将一段包含特定业务逻辑的 Python 代码转换为等效的 Java 代码。
    • 不使用 CoT：把这段Python代码转成Java。 -> 模型可能直接给出一个语法有误或逻辑不完全等价的 Java 代码。
    • 使用 CoT：请将以下 Python 代码转换为等效的 Java 代码。请遵循以下步骤：1. 首先，用自然语言描述这段 Python 代码的核心逻辑是什么。2. 然后，识别出 Python 特有的库和语法，并找到对应的 Java 实现。3. 最后，生成完整的 Java 代码。
    • 价值：这个过程强制模型先“理解”再“翻译”，而不是机械地进行语法替换。第一步的逻辑描述可以帮助我们验证模型是否正确理解了需求。第二步则能体现模型的技术选型是否合理（比如 Python 的 requests 库对应 Java 的 HttpClient 还是 OkHttp）。这大大提高了最终生成代码的质量和可用性。

2.1.2 如何在提示词中引导模型进行 CoT

引导 CoT 的关键在于明确要求模型输出思考步骤。有几种常用方法：

  1. 直接指令法：这是最简单直接的方式。

    • "请一步步思考..."
    • "请列出你的推理过程，然后再给出最终答案。"
    • "在得出结论之前，请详细分析每一种可能性。"

  2. 结构化输出法：结合我们对输出格式的要求，可以定义一个包含思考过程的结构。

    • 提示词：... 请以JSON格式返回，包含 "reasoning_steps" (一个描述你思考步骤的字符串数组) 和 "final_answer" 两个键。
    • 这对于后端程序解析非常友好，我们可以选择只采纳 final_answer，而将 reasoning_steps 作为调试日志记录下来。

  3. 少样本示例法 (Few-Shot CoT)：提供一个或多个包含了完整思考过程的示例，让模型模仿。这是效果最好但成本也最高的方法。

    模型在看到前面的示例后，会自然地模仿这种“先思考，后作答”的模式，来解决你的“正式任务”。

2.2 任务分解

任务分解是“给予思考时间”原则的另一种体现，它与 CoT 密切相关但侧重点不同。CoT 强调的是“如何思考”，而任务分解强调的是“做什么（分步做）”。你可以要求模型自己先将一个复杂任务分解成一系列简单的子任务，然后再逐一执行。

  • 与 CoT 的区别：CoT 像是在解一道数学题时写下演算草稿，整个过程是连贯的。任务分解更像是为了完成“做一顿饭”这个复杂任务，先主动列出待办清单：“1. 买菜；2. 洗菜切菜；3. 烹饪；4. 洗碗”，然后再一项项去完成。

  • 应用场景：当你需要生成一个比较复杂的文档或代码项目时，任务分解非常有用。

    • 场景：为新的微服务 user-service 生成一个完整的 README.md 文件。

    • 使用任务分解的提示词：

    • 交互过程：

      1. 你：发送上述提示词。
      2. 模型：返回一个 Markdown 格式的大纲，如 1. 项目简介 2. 技术栈 3. 如何构建与运行 4. API文档 ...
      3. 你：（审查并可能修改后）发送新提示词：大纲很好。现在，请先生成“1. 项目简介”和“2. 技术栈”部分的内容。
      4. 模型：生成对应内容。
      5. … 后续交互继续 …

  • 价值：这种“分步走”的策略，让后端工程师对整个生成过程拥有了完全的控制权。你可以在每一步之间进行审查、修正和注入新的信息。这比一次性生成一个巨大的、可能不满意的结果，然后反复修改要高效得多。在构建 Agent 或复杂工作流时，这种思想尤为重要，它将一个大任务变成了由 LLM 调用和人工审核交织组成的 pipeline。

3. 核心模式一：少样本提示

**少样本提示（Few-Shot Prompting）**是一种强大的技术，它通过在提示词中向模型提供几个完整的示例（“样本”），来“教”会模型如何执行一个新任务。这利用了大型语言模型的“情境学习”（In-Context Learning）能力，即模型能从你给出的上下文中快速学习并模仿示例的模式，而无需重新训练模型本身。

3.1 零样本 (Zero-Shot)

定义：不向模型提供任何示例，直接下达指令，要求模型完成任务。我们之前讨论的大部分提示词技巧，如明确指令、设定角色等，都属于零样本的范畴。

示例：请将文本“Hello, world!”翻译成法语。

适用场景：适用于那些模型在训练数据中已经见过大量类似模式的、非常通用的任务，例如通用翻译、文本摘要、简单问答等。对于稍微复杂或具有特定领域格式的任务，零样本的效果往往不稳定。

3.2 单样本/少样本

定义：

  • 单样本 (One-Shot)：在提示词中提供一个完整的“输入 -> 输出”示例，然后再给出实际要处理的输入。
  • 少样本 (Few-Shot)：在提示词中提供多个（通常是2-5个）完整的“输入 -> 输出”示例。

重要性：少样本提示的核心价值在于**“示范”而非“解释”**。很多时候，向模型解释一个复杂的格式要求，远不如直接给它看一个例子来得清晰有效。模型非常擅长从具体例子中归纳模式。

4. 核心模式二：结构化输出

对于后端工程师而言，这可能是最具实用价值的提示词模式。无论模型内部的思考过程多么精彩，如果最终的输出是一段无法被程序稳定解析的自然语言，那么它在自动化流程中的价值将大打折扣。结构化输出模式的核心目标，就是强制模型返回格式规整、可被机器直接读取和处理的数据，其中最常见、最重要的就是 JSON 格式。

4.1. 强制要求 JSON 输出

最基础的方法，就是用明确的指令告诉模型你想要 JSON。

  • 基本指令：
    • “请以JSON格式返回结果。”
    • “请返回一个JSON对象。”
    • “你的回答必须是一个合法的JSON字符串。”
  • 存在的问题：仅仅使用这样的简单指令，有时并不完全可靠。模型可能会：
    1. 在 JSON 前后添加额外文本：比如返回 “好的，这是您要的JSON：\n{...}\n”。这会导致后端在用 Jackson 或 Gson 解析时，需要先用正则表达式或字符串查找来提取出 {...} 这部分，增加了处理的复杂性和脆弱性。
    2. 返回的 JSON 结构不符合预期：你可能想要 {"name": "...", "age": ...}，但模型可能返回 {"customerName": "...", "yearsOld": ...}，导致反序列化到你的 Java POJO 时失败。
    3. 返回非法 JSON：例如，在值的末尾添加一个多余的逗号（trailing comma），或者字符串没有被正确转义。
  • 改进的指令：为了解决上述问题，指令需要更具约束力。
    • “请严格返回一个JSON对象，不要在JSON代码块前后包含任何解释、注释或其他多余的文本。”
    • “你的输出必须且只能是一个可以被直接解析的JSON字符串。”

这些更强的指令提高了模型直接返回纯净 JSON 的概率，但要做到万无一失，还需要结合下面的方法。

4.2. 提供 JSON Schema 或示例

这是实现可靠结构化输出的最佳实践。与其让模型去猜测你想要的 JSON 结构，不如直接把这个结构告诉它。这相当于为 API 响应定义了清晰的契约（Contract）。

  • 方法一：提供 JSON 示例

    • 这与少样本提示的思想一致，通过一个具体的例子来展示你想要的结构。

    • 提示词示例：

    • 模型会参照这个示例，生成：{"name": "张伟", "email": "zhang.wei@email.com", "age": 32}。

  • 方法二：提供更详细的 JSON Schema 描述（效果更佳）

    • 你可以提供类似 JSON Schema 的描述，对每个字段的含义和类型进行说明，这能进一步提高准确性。

    • 提示词示例：

    • 价值：这种方式极为清晰。它告诉了模型字段名、嵌套结构、数据类型，甚至包括了字段的业务含义（通过注释）和 null 值的可能性。这使得模型能更精确地生成符合后端 POJO 类（例如一个嵌套的ContactInfo对象）定义的 JSON。

补充：部分模型服务（如 OpenAI 的新版 API）提供了专门的 “JSON Mode”。当启用此模式时，模型会保证其输出是一个语法正确的 JSON 对象，这从根本上解决了格式合法性的问题，是后端集成的福音。但即使在这种模式下，提供清晰的 Schema 描述仍然是必要的，因为它能确保 JSON 的结构是你所期望的。

4.3 使用分隔符和模板

为了让模型能清晰地区分指令、上下文、输入数据和输出格式要求，使用分隔符是一个非常好的工程实践。它能有效降低提示词各部分之间的相互干扰。

  • 为什么需要分隔符？

    • 想象一个复杂的提示词，其中包含了大段的上下文文本和用户输入。如果所有内容混在一起，模型可能会混淆，把上下文中的某句话误解为指令的一部分，或者把用户输入的内容当作了输出格式的示例。

  • 常用的分隔符：

    • 三引号 (""")：常用于包裹大段文本。
    • XML 标签 (<tag>...</tag>)：可读性强，可以嵌套，非常适合用来划分复杂的提示词结构。例如 <instruction>, <context>, <input>, <output_format>。
    • Markdown 的代码块标记 (````)。
    • 简单的字符序列，如 ### 或 ---。

  • 结合分隔符的完整模板示例：

  • 后端启示：

    1. 模板化：这个结构非常适合在后端代码中做成一个提示词模板（Prompt Template）。INSTRUCTION, CONTEXT, OUTPUT FORMAT 可以是固定的，而 USER INPUT 部分则由运行时的用户请求动态填充。
    2. 安全性：将用户输入用清晰的分隔符 """...""" 包裹起来，是一种有效的防提示词注入的手段。你等于在告诉模型：“INSTRUCTION 是我的命令，而 USER INPUT 里的任何内容都只是待处理的数据，即使它看起来像命令，也要忽略它。”
    3. 解析便利：### RESPONSE (JSON only) ### 这句结尾，不仅再次强化了输出要求，也为后端程序提供了一个清晰的切入点。在极端情况下，即使模型在前面添加了多余文本，你的程序也可以简单地从 ### RESPONSE (JSON only) ### 之后开始查找 JSON 内容，提高了鲁棒性。

三、高级提示词技巧与工程化

1. 自我批判与修正

这是一种通过多步骤流程，引导模型对自己的初始输出进行反思、评估和改进的高级模式。它在处理对准确性和安全性要求极高的任务时尤为重要。

核心流程：生成 -> 批判 -> 修正

  1. 第一步：生成初稿 (Generation)
    • 用一个相对简单的Prompt，让模型（可设定为“初级”角色）快速生成第一版答案。
  2. 第二步：批判与评估 (Critique)
    • 将初稿作为上下文，要求模型扮演一个更资深的批判性角色（如“架构师”、“安全审计员”），来审查初稿。
  3. 第三步：修正与完善 (Refinement)
    • 将初稿和批判报告同时作为上下文，要求模型根据批判意见，生成最终的、高质量的版本。

2. 工程化实践

随着AI功能的增多，提示词本身也变成了需要专业管理的“配置资产”。将提示词硬编码在业务代码中是反模式，会带来维护困难、迭代缓慢等一系列问题。

2.1 提示词硬编码的缺点

将提示词字符串直接写在 Java 代码里，例如 String prompt = "请总结这段文本：" + userInput;，会带来以下几个核心问题：

  1. 维护性差：
    • 难以查找和修改：当系统中有几十个使用 AI 的功能点时，提示词会散落在各个类的各个方法中。如果需要统一修改提示词的风格（例如，为所有提示词增加一个安全指令），或者某个模型版本更新后需要适配提示词，这将是一场灾难。
    • 职责混淆：Java 代码的核心职责是业务逻辑和流程控制。长篇的、富含自然语言的提示词混入其中，会严重污染代码，降低可读性。业务逻辑和“与AI沟通的话术”应该分离。
  2. 迭代速度慢：
    • 重新编译和部署：提示词工程是一个需要不断实验和优化的过程。如果提示词在代码里，那么每一次微小的调整（比如改一个词、调整一个示例）都需要修改代码、重新编译、打包、部署服务，这个流程非常笨重，极大地阻碍了快速迭代和 A/B 测试。
    • 非技术人员无法参与：提示词的优化有时也需要产品经理、运营人员的输入（尤其是在文案生成等场景）。如果提示词在代码里，他们将无法直接查看或参与改进。
  3. 可复用性低：
    • 系统中可能有很多地方需要执行类似的任务，比如“提取JSON”。如果每个地方都复制粘贴一大段提示词，不仅造成代码冗余，也使得后续的统一更新变得不可能。
  4. 难以调试和监控：
    • 当线上 AI 功能出现问题时，你需要快速知道当时传递给模型的完整提示词是什么。如果提示词是动态拼接的，并且分散在代码各处，追溯和复现问题会非常困难。

2.2 提示词模板

提示词模板是解决上述问题的核心方案。它的思想很简单：将提示词中固定不变的部分（如指令、角色、输出格式要求）和动态变化的部分（如用户输入、上下文数据）分离开来。

定义：一个包含占位符的文本模板。在运行时，程序会用具体的数据填充这些占位符，生成最终的完整提示词。

  1. 资源文件管理 (推荐)
    • 做法：将提示词模板存储在src/main/resources下的.properties或.yaml文件中。

3. 复杂流程的基石：ReAct思想简介

当任务需要与外部世界交互（查数据库、调API）时，单次“提问-回答”模式失效。ReAct框架为此提供了理论基础，它将“推理(Reason)”和“行动(Act)”交错结合，让LLM成为一个能主动思考、制定计划并利用工具的智能代理(Agent)。

核心循环：Thought -> Action -> Observation

  1. Thought (思考)：模型进行“内心独白”，分析目标，决定下一步做什么。
    • “用户想查天气，我需要调用天气工具。”
  2. Action (行动)：模型生成一个结构化的指令，指定要调用的工具和参数。
    • Action: search_weather(city="北京")
  3. Observation (观察)：后端系统执行这个Action（调用真实的天气API），并将返回结果作为“观察”再喂给模型。
    • Observation: {"temperature": "25°C", "condition": "晴"}

模型接收到Observation后，开始新一轮循环，直到最终完成任务。

四、 提示词的保障与提升

1. 提示词注入防护

1.2 防护策略一：指令与输入分离

核心思想：从结构上让模型分清“指令”和“数据”。

  • 最佳实践：使用XML标签或角色
    • XML标签：用<system_instructions>和<user_input>等标签明确包裹不同部分。XML的结构性最强，最能抵抗注入。
    • 角色role：在使用如OpenAI的Chat API时，善用system, user, assistant角色。将你的核心指令放在system角色中，这是从API层面提供的最强隔离。

后端实现要点：在构建Prompt模板时，强制使用这种结构化、带分隔符的格式。不要使用简单的字符串拼接。

1.3 防护策略二：输入/输出预处理与后处理

  1. 输入预处理 (门卫)
    • 关键词过滤 (脆弱)：建立黑名单（如“忽略”、“重置指令”），但容易被绕过。
    • 意图分析 (更优)：使用一个专门的“门卫AI”或简单分类器，预先判断用户输入是否包含恶意指令意图。如果检测到风险，直接拒绝服务。
  2. 输出后处理 (校验)
    • 内容审查：检查模型输出是否包含了不应泄露的、来自上下文的敏感信息。
    • 操作确认 (针对Agent)：对于调用高风险函数（如deleteUser）的Action，绝不直接执行。必须引入人工确认循环 (Human-in-the-loop)或二次确认机制。

后端实现要点：将这些处理逻辑封装成独立的Interceptor或Filter，应用在与LLM交互的核心流程中，形成一个可复用、可插拔的安全框架。

2. 控制“幻觉”

2.1 提示词级策略

这是最直接、也是最基础的控制幻觉的提示词技巧。默认情况下，LLM 被训练得“乐于助人”，它会尽力根据已有知识去猜测一个答案，即使它并不确定。我们需要通过明确的指令来改变它的这种默认行为。

  • 基本指令：

    • “如果根据你所掌握的信息无法回答这个问题，请直接回答‘我不知道’。”
    • “请只根据我提供的上下文信息来回答问题。如果上下文中没有相关信息，请明确说明这一点，不要自行猜测。”

  • 结合角色扮演：

    • “你是一个严谨的档案管理员。你的职责是根据档案库（即我提供的上下文）中的现有记录来回答问题。对于档案中没有记录的事情，你的标准回答是‘根据现有档案，无法确认此信息’。”
    • 通过赋予模型一个严谨、保守的角色，可以强化它“知之为知之，不知为不知”的行为模式。

  • 示例：

    • 提示词:

    • 期望的输出: “信息未提供”

    • 没有该指令时可能的幻觉输出: “王五在2022年的销售业绩是110万。” (模型可能会基于张三和李四的数据进行不当的推断和捏造)

2.2 架构级策略：RAG

这是控制幻觉最根本、最有效的架构级方案，它将提示词工程与外部知识库查询结合了起来。这个思想将在后续的 RAG 模块中详细展开，这里我们先做概要介绍。

  • 核心思想：不要依赖模型自身那庞大、不可控、可能已过时的内部知识。而是为每个问题，都动态地从一个我们自己可控的、可信的外部知识库（如公司的文档库、产品手册、数据库）中检索出最相关的几段信息，然后将这些信息作为实时的、精准的上下文注入到提示词中，并强制模型只基于这些上下文来回答问题。

  • 工作流程简介 (RAG)：

    1. 用户提问：例如，“我们的‘CodePilot Pro’产品支持Python的实时调试吗？”

    2. 检索 (Retrieve)：后端系统首先将用户的问题转换成一个查询向量，然后去“向量数据库”（后续模块）中检索出最相关的产品文档片段。例如，检索到了以下两段：

      • 文档片段1：“CodePilot Pro v2.5版本更新日志：新增对Python语言的实时调试支持...”
      • 文档片段2：“CodePilot Pro v2.4及以下版本支持Java和C#的调试...”

    3. 增强 (Augment)：后端将这些检索到的片段和用户问题组合成一个新的提示词。

    4. 生成 (Generate)：模型基于这个包含了确凿证据的提示词进行回答。

      • 可靠的输出：“是的，根据v2.5版本的更新日志，CodePilot Pro产品支持Python的实时调试。”

  • 为什么有效：RAG 模式将一个开放域的“创作”问题，转换成了一个闭卷的“阅读理解”问题。模型不再需要去记忆或猜测，只需要在给定的几段文本中找到答案并组织语言即可。这从根本上限制了模型“自由发挥”的空间，极大地降低了幻觉的产生。\

2.3 API参数级策略

temperature 是调用 LLM API 时一个非常重要的参数，它控制着模型输出的随机性。

  • 参数说明：
    • temperature 的值通常在 0 到 2 之间。
    • 较低的 temperature (如 0.0 - 0.3)：模型会倾向于选择概率最高的词语来生成下一个token。这使得输出更具确定性、重复性和保守性。
    • 较高的 temperature (如 0.8 - 1.2)：模型会增加选择低概率词语的可能性，使得输出更具多样性、创造性和随机性。
  • 如何利用它控制幻觉：
    • 对于需要事实准确性、逻辑严谨性的任务，应该使用较低的 temperature。
      • 场景：代码生成、SQL查询、从文本中提取结构化数据、基于上下文的问答 (RAG)。
      • 设置为 temperature=0 会让模型尽可能地给出最可信、最符合训练模式的答案。在这些场景下，我们不希望模型有任何“创造力”，我们追求的是稳定和可复现。
    • 对于需要创造性、多样性的任务，可以使用较高的 temperature。
      • 场景：市场营销文案生成、头脑风暴、写诗、聊天机器人。
      • 在这些场景下，一定的“幻觉”或“自由联想”有时反而是被期望的。
  • 后端实践：
    • 在封装 LLM API 调用时，应该将 temperature 作为一个可配置的参数。
    • 为不同类型的业务场景设置不同的默认 temperature 值。例如，一个 FactExtractionService 的默认 temperature 可能是 0.1，而一个 MarketingCopyService 的默认值可能是 0.9。
    • 这使得我们可以根据任务的性质，灵活地在“严谨”和“创新”之间进行权衡。

3. 质量提升

提示词工程不是一蹴而就的，它是一个需要持续评估和优化的科学过程。

3.1 建立“度量衡” - 黄金标准数据集

定义：一个精心挑选的“输入-期望输出”对的集合，是你评估提示词好坏的唯一客观标准。没有它，任何优化都是凭感觉的“盲人摸象”。

如何构建高质量的Golden Set：

  1. 覆盖典型场景 (Happy Path)：包含最常见、最核心的业务请求。
  2. 覆盖边界场景 (Edge Cases)：这体现了工程的专业性。
    • 模糊/无效输入：测试模型的鲁棒性。
    • 复杂输入：测试模型的理解能力上限。
    • 注入攻击尝试：测试模型的安全性。
  3. 保持小而精：初期20-50个高质量样本即可启动。
  4. 格式化管理：用JSONL、CSV或数据库表存储，便于程序读取和自动化测试。

3.2 实施“自动化测试” - 多维度评估方法

有了Golden Set，下一步就是自动化地比较“模型实际输出”与“期望输出”。

评估方法 核心原理 适用场景 后端实现
精确匹配 (Exact Match) 字符串完全相同。 分类标签、内容确定的JSON值。 Assert.assertEquals(expected, actual);
关键字/正则匹配 包含关键信息或符合特定模式。 摘要、SQL查询（检查表名、子句）。 assertThat(actual).contains("keyword");
assertThat(actual).matches(regex);
JSON结构/字段校验 验证JSON合法性、字段完整性、值类型。 所有要求JSON输出的场景（后端核心）。 ObjectMapper.readValue(), 反序列化到DTO，用Bean Validation或AssertJ进行断言。
LLM即裁判 (LLM-as-a-Judge) 用更强的模型（如GPT-4）来评估弱模型的输出。 复杂的自然语言输出，难以用规则判断。 封装一个evaluation_prompt，调用裁判模型API，解析其"pass"或"fail"的响应。

3.3 上线“灰度发布” - A/B测试

离线评估无法完全模拟真实世界的复杂性。一个提示词的最终效果，必须通过线上流量来验证。

核心思想：将线上用户流量分组，同时运行新旧两个版本的提示词，收集并比较关键业务指标。

后端工程师的实施关键点：

  1. 提示词版本管理：你的提示词管理系统（配置文件或数据库）必须支持版本化。
  2. 流量切分：利用**网关层 (Nginx/Gateway)**或在**业务代码**中，根据用户ID或随机数，动态决定本次请求使用哪个版本的提示词。
  3. 指标埋点与收集 (最关键)：定义并上报明确的业务指标。
    • 工程指标：
      • api_cost / token_consumption (成本)
      • api_latency (性能)
      • failure_rate / retry_count (稳定性)
    • 业务指标：
      • user_adoption_rate (用户是否采纳AI结果)
      • conversion_rate (AI建议是否提升了转化)
      • negative_feedback_rate (用户点“踩”的比例)
  4. 数据分析与决策：通过统计分析，判断新版提示词是否在关键指标上取得了显著的正面效果。

3.4 总结：建立持续迭代的闭环

一个专业的提示词工程流程，就是一个完整的、科学的闭环：

提出假设 -> 修改提示词 -> 离线评估 (Golden Set回归测试) -> 上线A/B测试 -> 数据分析 -> 固化成果/继续优化

最终目标：将对提示词的优化，从一种“艺术创作”，转变为一种数据驱动、可度量、可回滚的工程实践，这才是将AI能力可靠、高效地融入复杂业务系统的关键。
