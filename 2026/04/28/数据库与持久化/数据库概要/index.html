数据库概要

发表于 2026-04-28 更新于 2026-06-08 分类于 数据库与持久化

一、数据管理的演进：为什么需要数据库

1. 在没有数据库的年代

在数据库出现之前，应用程序的数据直接存储在操作系统的文件中。一个电商系统可能会这样组织数据：

这种方式在数据量和访问量都很小的时代勉强可用，但随着系统规模增长，文件存储的四大痛点逐渐暴露：

痛点 具体表现
数据冗余与不一致 同一个用户的地址信息在 users.txt 和 orders.txt 中各存一份，修改一处另一处忘记同步
缺乏标准访问方式 每个开发者都要自己写代码去解析文件、遍历查找。查询”2024年所有已支付订单”需要手写文件解析逻辑
无并发控制 两个用户同时下单，同时写 orders.txt，后写入的覆盖了先写入的——订单丢了
无安全保障 任何人只要能登录服务器，就能直接 cat 查看所有数据，没有用户权限体系

结论：文件系统能”存”数据，但不能”管理”数据。

2. 数据库的核心价值

数据库管理系统（DBMS，Database Management System）正是为了解决上述问题而诞生的。它提供了文件系统无法提供的四大核心能力：

2.1 数据持久化 (Data Persistence)

数据一旦写入，就不会因为程序关闭、服务器重启或断电而丢失。数据被安全地存储在非易失性存储介质（硬盘/SSD）上，配合 Redo Log、WAL（Write-Ahead Log） 等机制保证即使意外崩溃也不丢数据。

2.2 高效的数据检索与管理 (Data Retrieval & Management)

  • 索引 (Index)：通过 B+ 树、哈希表等数据结构，将”在大海里捞针”变成”按图索骥”——从 $O(N)$ 的全表扫描降至 $O(\log N)$ 的索引查找。
  • 声明式查询 (Declarative Query)：使用 SQL 描述”要什么”，数据库优化器自动决定”怎么找”，开发者无需关心底层执行路径。
  • 结构化组织：通过表（Table）、模式（Schema）将数据按业务实体有组织地存储，告别混乱的文件命名和格式约定。

2.3 并发控制 (Concurrency Control)

多个用户/应用程序同时读写数据库时，DBMS 通过以下机制保证数据正确性：

  • 事务 (Transaction)：将一系列操作打包为一个原子单元——要么全部成功，要么全部失败。
  • 锁机制 (Locking)：行级锁、表级锁，保证并发写操作不会互相覆盖。
  • 多版本并发控制 (MVCC)：读操作不加锁，通过数据快照实现”读写不冲突”。

2.4 数据一致性与完整性 (Data Consistency & Integrity)

数据库在引擎层面强制执行数据规则，而不是依赖应用程序代码：

  • 约束 (Constraints)：主键唯一、外键引用完整、NOT NULL、CHECK 等，从源头杜绝脏数据。
  • ACID 事务：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability），保证数据在任何情况下都处于逻辑正确的状态。
  • 权限控制：细粒度的用户权限管理（谁能看什么、谁能改什么），文件系统无法实现。

一句话总结：文件系统 = 数据的”纸箱子”，数据库 = 数据的”智能仓库”——不仅存，还负责检索、安全、并发、一致性的全套管理。

3. 基本概念辨析

在深入学习之前，先区分三个容易混淆的核心概念：

flowchart TB
    subgraph DBS["数据库系统 (DBS)"]
        App["应用程序\n(Java / Go / Python)"]
        DBMS["数据库管理系统 (DBMS)\nMySQL / PostgreSQL / Oracle"]
        DB["数据库 (Database)\n实际存储的数据文件"]
    end

    App -->|"SQL / API 调用"| DBMS
    DBMS -->|"管理、读写"| DB
概念 英文 是什么 类比
数据库 (DB) Database 持久化存储的数据本身，以及其组织结构（表、索引等） 图书馆里的书
数据库管理系统 (DBMS) Database Management System 管理数据库的软件系统，提供数据的增删改查、事务、安全等能力 图书馆的管理系统（借还书、编目、检索）
数据库系统 (DBS) Database System DB + DBMS + 应用程序的整体 整座图书馆（书 + 系统 + 管理员 + 读者）

日常口语中说的”数据库”，多数情况下指的是 DBMS（如”装一个 MySQL 数据库”）。

二、数据库分类全景图

1. 按数据模型分类

数据模型决定了”数据如何被组织和存储”，这是最根本的分类方式。

flowchart LR
    DB["数据库\n(按数据模型)"]
    DB --> SQL["关系型数据库\n(RDBMS)"]
    DB --> NoSQL["非关系型数据库\n(NoSQL)"]

    NoSQL --> KV["键值存储\nKey-Value"]
    NoSQL --> Doc["文档型\nDocument"]
    NoSQL --> WC["宽列存储\nWide-Column"]
    NoSQL --> Graph["图数据库\nGraph"]
    NoSQL --> TS["时序数据库\nTime-Series"]
    NoSQL --> Search["搜索引擎\nSearch Engine"]
    NoSQL --> Vector["向量数据库\nVector DB"]

1.1 关系型数据库 (RDBMS)

核心特征：数据以表（Table）的形式组织，表之间通过外键建立关联，使用 SQL 作为标准查询语言。

  • 数据模型：严格的二维表结构，行 = 记录，列 = 字段。Schema 必须预先定义。
  • 事务保证：ACID（原子性、一致性、隔离性、持久性）。
  • 代表产品：MySQL、PostgreSQL、Oracle、SQL Server、SQLite。

1.2 非关系型数据库 (NoSQL)

NoSQL 不是”No SQL”，而是 “Not Only SQL”——它们不使用传统的二维表结构，而采用更灵活的数据模型。

类型 数据模型 核心特点 代表产品 典型场景
键值存储 (KV) Key → Value 的简单映射 极速读写，支持过期、发布订阅 Redis、etcd 缓存、会话管理、分布式锁、排行榜
文档型 (Document) JSON/BSON 文档，嵌套结构 Schema 灵活，适合快速迭代 MongoDB、CouchDB 内容管理、用户画像、物联网数据
宽列存储 (Wide-Column) 行键 → 列族 → 列，稀疏矩阵 超大规模、高吞吐写入、按列存储 Cassandra、HBase 日志存储、时序指标、推荐系统
图数据库 (Graph) 节点 + 边（属性图模型） 擅长处理深度关联查询 Neo4j、JanusGraph 社交关系、知识图谱、风控关系网络
时序数据库 (TSDB) 时间戳 + 指标值 高压缩比、时间窗口聚合、自动过期 InfluxDB、TimescaleDB 监控指标、IoT 传感器、金融行情
搜索引擎 倒排索引 全文检索、相关性排序、聚合分析 Elasticsearch 商品搜索、日志分析、站内全文检索
向量数据库 高维向量嵌入 近似最近邻搜索 (ANN) Milvus、Pinecone RAG 知识库、图像检索、语义搜索

2. 按应用场景分类

2.1 OLTP：联机事务处理

OLTP（Online Transaction Processing） 是传统业务系统的主力场景。

维度 特征
业务特征 短事务、高并发、大量小读写
典型操作 下单、登录、转账——每次操作涉及的数据量小，但请求量大
数据特征 行式存储，当前数据，高度规范化（3NF）
核心需求 低延迟（毫秒级）、高可用、ACID 事务
代表产品 MySQL、PostgreSQL、Oracle

2.2 OLAP：联机分析处理

OLAP（Online Analytical Processing） 面向数据分析和商业智能场景。

维度 特征
业务特征 复杂聚合查询、大范围扫描、数据挖掘
典型操作 “上季度各区域销售额排行”、”用户留存率趋势”——每次查询扫描百万行
数据特征 列式存储，历史数据，反规范化（星型/雪花模型）
核心需求 高吞吐、复杂 SQL 优化、批量导入
代表产品 ClickHouse、Doris、Snowflake、Apache Hive

2.3 HTAP：混合事务/分析处理

HTAP（Hybrid Transactional/Analytical Processing） 是近年来兴起的新范式，试图在同一个系统中同时支撑 OLTP 和 OLAP 负载，消除数据从业务库到分析库的 ETL 延迟。

3. 按部署模式分类

三、关系型 vs 非关系型

这是面试和实际架构决策中最高频的话题。理解二者的本质差异，比记住一百个产品名更重要。

1. 五个维度的全景对比

对比维度 关系型数据库 (RDBMS) 非关系型数据库 (NoSQL)
数据模型 严格的二维表结构，Schema 必须预定义 灵活的数据模型（KV/文档/图/列族），Schema 可动态变化
数据完整性 强约束（主键、外键、CHECK），数据库层强制 弱约束或无约束，完整性由应用层负责
事务模型 ACID（强一致性）——银行转账必须保证 BASE（最终一致性）——允许短暂不一致，追求高可用
扩展能力 垂直扩展为主（升级硬件），水平扩展困难 水平扩展天然支持（加机器即可，数据自动分片）
查询能力 SQL 极其强大，支持复杂 JOIN、子查询、窗口函数 查询能力因产品而异，大多不支持 JOIN 或仅支持简单连接

关于 ACID vs BASE

ACID BASE
全称 Atomicity, Consistency, Isolation, Durability Basically Available, Soft state, Eventually consistent
核心承诺 事务结束后，数据立即一致 系统始终可用，数据最终会达到一致
类比 银行转账——一笔交易要么完整执行，要么完全撤销 社交媒体点赞数——短暂显示不一致可以接受，最终会同步
核心场景 金融、交易、订单、库存 社交动态、推荐系统、缓存、日志

2. 选型决策框架

flowchart TD
    Start["开始选型"] --> Q1{"数据结构是否固定？"}
    Q1 -- 是 --> Q2{"是否需要复杂事务\n和多表关联？"}
    Q2 -- 是 --> R["关系型数据库\nMySQL / PostgreSQL"]
    Q2 -- 否 --> Q3{"数据量级是否\n达到 TB/PB 级？"}
    Q3 -- 是 --> NoSQL_Big["Cassandra / HBase\n(宽列存储)"]
    Q3 -- 否 --> Mix["关系型 + Redis 缓存\n(最常用的组合)"]

    Q1 -- 否 --> Q4{"数据形态是什么？"}
    Q4 -- "JSON文档" --> DocDB["MongoDB"]
    Q4 -- "关联关系为主" --> GraphDB["Neo4j"]
    Q4 -- "时间序列指标" --> TSDB["InfluxDB / TimescaleDB"]
    Q4 -- "全文搜索" --> ES["Elasticsearch"]
    Q4 -- "高维向量" --> VectorDB["Milvus / Pinecone"]

现实中的大多数系统，采用的是”多模组合”架构——例如：MySQL 管核心业务数据 + Redis 做缓存和会话 + Elasticsearch 做商品搜索。各取所长，而不是一刀切。

四、应用程序如何访问数据库

了解数据库是什么之后，下一个问题是：Java 程序（或任何应用程序）如何与数据库通信？

1. 数据库访问的分层架构

flowchart TB
    subgraph 应用层
        Biz["业务逻辑层\n(Service / Controller)"]
    end

    subgraph 数据访问层
        ORM["ORM / SQL Mapper\n(MyBatis / JPA / Hibernate)"]
    end

    subgraph 连接管理层
        Pool["连接池\n(HikariCP / Druid)"]
    end

    subgraph 驱动层
        Driver["JDBC 驱动\n(MySQL Connector / pgJDBC)"]
    end

    subgraph 数据库层
        DBMS["数据库管理系统\n(MySQL / PostgreSQL / Redis)"]
    end

    Biz -->|"调用 Mapper / Repository"| ORM
    ORM -->|"SQL + 参数"| Pool
    Pool -->|"获取/归还连接"| Driver
    Driver -->|"原生网络协议 (TCP)"| DBMS

各层职责一句话

层级 职责 解决什么问题
ORM / SQL Mapper 对象与关系之间的自动映射 “我不想在代码里拼接 SQL 字符串，也不想手动把 ResultSet 转成 Java 对象”
连接池 复用数据库连接，管理连接生命周期 “每次请求都新建和销毁数据库连接太慢了，而且连接数有限”
JDBC 驱动 将标准的 JDBC API 调用翻译为数据库原生网络协议 “Java 代码写一套 API，底层能连 MySQL、PostgreSQL、Oracle 任何一种”
数据库 存储、检索、事务、安全 “数据要安全地持久化，同时支持高效查询和并发访问”
