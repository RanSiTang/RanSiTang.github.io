AI工作流与编排

发表于 2026-05-29 更新于 2026-06-07 分类于 AI

一、AI Workflow 的基本概念与定位

1. 什么是 AI Workflow

AI Workflow（AI 工作流）是将多个 AI 操作（LLM 推理、工具调用、RAG 检索等）与非 AI 操作（条件判断、数据转换、人工审批等）按照预定义的有向图结构组合起来，形成一个可控、可观测、可复现的自动化处理流程。

理解 AI Workflow 最直观的方式是把它与传统业务流程自动化（如 Activiti、Camunda 工作流引擎）做类比：传统工作流编排的是”人工审批 → 系统处理 → 通知发送”等确定性步骤；AI Workflow 在此基础上引入了LLM 作为流程中的一个节点——这个节点的输入和输出是自然语言，行为具有一定的非确定性，但它被约束在流程图的固定位置上，输入输出受前后节点的控制。

2. Workflow 与 Agent 的核心区别

这是 AI 应用架构中最容易混淆的两个概念，它们的核心差异在于决策权的归属。

维度 AI Workflow（工作流） AI Agent（智能体）
决策方式 流程图预定义路径 LLM 自主决策下一步
控制权 流程引擎控制执行顺序 LLM 作为大脑规划执行顺序
可预测性 高——路径在设计中确定 低——模型可能走非预期的路径
灵活性 低——只能走预定义的路径 高——可应对未预见的情况
适用场景 标准化业务流程 开放式任务、探索性任务
工程复杂度 中（设计流程图） 高（需处理 Agent 失控风险）

用一句话概括：Workflow 是”你决定流程，LLM 执行节点”；Agent 是”LLM 决定流程，你提供工具”。

Workflow 的核心价值是将 LLM 的非确定性装进流程的笼子里——你知道用户提的问题会经过”查询改写 → 检索 → 相关性判断 → 回答生成”这个固定流程，每个阶段的输入输出都能被监控和干预。Agent 的核心价值是灵活性——面对开放式任务，不需要事先枚举所有可能路径。在实践中，两者经常组合使用：Agent 作为顶层的任务规划器，内部调用 Workflow 执行标准化的子任务。

3. AI Workflow 要解决的核心问题

问题一：单次 LLM 调用的能力边界

单次 LLM 调用在做复杂任务时存在明显局限——输出长度有限、推理深度不足、无法访问外部工具。Workflow 通过串联多个 LLM 调用和工具节点，将复杂任务拆解为多个专注的步骤，每个步骤职责单一、输入明确、输出可控。

问题二：LLM 输出的不可控性

LLM 的输出是概率性的——同样的输入可能产生不同质量的输出。在直接对话模式下，你只能通过 Prompt 来约束模型行为。Workflow 通过在 LLM 节点前后加入校验节点、条件路由节点、格式化节点，将模型的非确定性输出约束在可控范围内。

问题三：业务流程的标准化需求

企业级 AI 应用不只是”聊天”——它需要接入现有的业务流程（审批、通知、审计）。Workflow 将 AI 能力嵌入到标准化的业务流程中，使 AI 成为业务流程的一个步骤，而不是独立运行的”黑盒”。

二、AI Workflow 的核心节点类型

AI 工作流由不同类型的节点按照有向边连接而成。每种节点承担特定的职责，理解各节点的能力边界是设计好工作流的前提。

1. LLM 节点（LLM Node）

LLM 节点是 AI Workflow 的核心——它调用大语言模型完成特定的文本处理任务。

典型用途：

  • 意图分类：将用户的自然语言输入分类为预定义的意图标签（如”查询订单”、”提交工单”、”一般咨询”）
  • 信息抽取：从非结构化文本中抽取结构化字段（如从简历中抽取姓名、工作年限、技能列表）
  • 文本生成：基于上下文生成回复、摘要、报告
  • 语义判断：判断文本是否满足特定条件（如”检索结果是否与用户问题相关”）
  • Query Rewrite：将用户的模糊查询改写为更适合检索的精确查询

LLM 节点的配置要素：

配置项 说明 示例
System Prompt 节点的角色和行为规范 “你是一个意图分类器…”
User Prompt 模板 支持变量注入的模板 “用户输入：“
输出格式约束 JSON Schema、正则等 {"intent": "string", "confidence": "number"}
模型选择 不同节点可用不同模型 意图分类用轻量模型，生成用强模型

LLM 节点的设计原则：

  • 单一职责：一个 LLM 节点只做一件事。意图分类和回答生成应该分属两个节点，而不是放在同一个 Prompt 中
  • 输入明确化：LLM 节点的输入应该来自上游节点的结构化输出，而不是原始用户输入。例如，”回答生成”节点的输入是”检索到的文档 + 用户问题”，而不是”用户原话”
  • 输出结构化：尽可能约束 LLM 节点输出结构化数据（JSON），而不是自由文本。结构化输出可以被下游的条件节点和代码节点精确处理

2. 条件判断节点（Condition Node）

条件判断节点根据上游 LLM 节点或代码节点的输出结果，决定流程的下一步走向。它实现了工作流的分支逻辑。

条件判断有两种模式：

  • LLM 判断：由 LLM 节点输出一个分类标签或布尔值，条件节点根据标签路由。优势是灵活（能处理模糊语义），风险是 LLM 可能产生错误的分类
  • 规则判断：由纯代码逻辑（if-else、正则匹配、阈值比较）决定路由。优势是确定性和零成本，局限是只能处理可编码的规则

条件节点的设计建议：

  • 兜底分支必须有：为”无法判断”或”置信度不足”的情况设置兜底路由，通常是回退到通用回复或转人工
  • LLM 判断 + 规则兜底：先由 LLM 判断意图，当 LLM 输出的置信度低于阈值时，回退到规则判断。这结合了两者的优势
  • 分支路径不宜过多：超过 5-7 个分支时，LLM 的分类准确率开始下降。过多分支考虑分层分类——先粗分再细分

3. 工具调用节点（Tool Node）

工具调用节点执行具体的业务操作——调用后端 API、查询数据库、发送通知等。在 Function Calling 章节中详细讨论了工具定义和调用机制，这里聚焦于工作流中工具节点的编排特性。

在工作流中使用工具节点 vs 直接在 Agent 中使用工具：

工作流中的工具节点 Agent 中的工具调用
触发方式 流程图决定何时调用 LLM 自主决定何时调用
参数来源 上游节点的结构化输出 LLM 从用户话语中推理
出错处理 流程预设的异常路径 模型自行判断如何处理
适用场景 标准化业务操作 探索性、不确定性操作

工具节点的输入绑定：在工作流中，工具节点的参数通常不是由 LLM 直接从用户话语中抽取的，而是绑定自上游节点的结构化输出。例如，意图识别节点输出 {"intent": "query_order", "orderId": "ORD-12345"}，下游的订单查询工具节点直接使用 orderId 作为参数。这种方式消除了 LLM 参数推理的不确定性。

4. RAG 检索节点（Retrieval Node）

RAG 检索节点负责从外部知识库中检索与用户问题相关的文档片段。它通常嵌入在问答类工作流中，位于”查询处理”和”回答生成”之间。

RAG 检索节点的配置：

  • 检索源：向量数据库 Collection、全文搜索引擎、结构化数据库
  • 检索策略：纯向量检索、混合搜索、Multi-Query 检索
  • TopK 和阈值：在工作流层面控制检索参数，而非硬编码
  • 后处理：去重、排序、上下文拼接

5. 人工审批节点（Human-in-the-Loop Node）

人工审批节点在工作流中设置一个人工确认点——流程暂停，等待人工审核通过后再继续执行。

典型场景：

  • 敏感操作确认：LLM 生成的邮件、公告、工单需要人工审核后才能发送
  • 高风险决策：模型建议的操作可能造成业务影响（如批量取消订单），需要人工确认
  • 置信度不足时的兜底：当 LLM 判断置信度低于阈值时，转人工处理

人工审批节点的实现要点：

  • 暂停与恢复机制：工作流引擎需要支持暂停执行，保存上下文状态，在人工审批完成后恢复执行
  • 超时策略：设置审批超时时间，超时后按预设策略处理（自动驳回 / 自动通过 / 升级通知）
  • 审批上下文：审批界面需要展示充分的上下文——LLM 生成了什么、依据是什么、风险点在哪

6. 代码/脚本节点（Code Node）

代码节点执行确定性的数据处理逻辑——数据转换、计算、校验、格式化等。它是工作流中”确定性”的锚点，与 LLM 节点的”非确定性”形成互补。

典型用途：

  • 数据格式转换（JSON → XML → Markdown）
  • 参数校验与清洗
  • 数值计算（价格汇总、时间差计算）
  • 调用已有的 Java Service 方法

在工作流设计中，应该尽可能将确定性的逻辑放在代码节点中，只将需要语义理解的任务交给 LLM 节点。这样做不仅降低成本和延迟，也提高了流程的可预测性。

7. 数据转换节点（Transform Node）

数据转换节点负责在不同节点之间转换数据格式——上游节点的输出格式不一定匹配下游节点的输入格式。虽然可以归入代码节点，但由于其使用频率极高，值得单独讨论。

三、主流 AI 工作流编排方案

1. 低代码平台：Dify

Dify 是目前最流行的开源 AI 应用开发平台之一，提供可视化的 AI 工作流编排能力。

核心设计：

  • Chatflow 与 Workflow：Dify 区分两种应用类型——Chatflow（对话式，有记忆的交互式 AI）和 Workflow（批处理式，单次执行的自动化流程）
  • 可视化拖拽编排：在 Web UI 上拖拽节点、连线，配置参数，无需编写代码
  • 内置丰富的节点类型：LLM、知识检索（RAG）、Code（Python/JS）、HTTP 请求、IF/ELSE 条件、模板转换、变量聚合等
  • 知识库管理集成：平台内置文档上传、切分、向量化、检索的全流程管理，与 RAG 节点无缝衔接

Dify 适合的场景：

以非技术人员为主的团队、需要快速验证 AI 应用原型的场景、流程结构相对简单（节点数 < 30）的应用。

对 Java 后端的定位：Dify 可以作为 AI 逻辑层的前置服务，Java 后端通过 REST API 调用 Dify 应用，将 Dify 视为一个”AI 处理服务”。复杂的业务逻辑（事务管理、权限校验、数据聚合）仍在 Java 后端完成。

2. 低代码平台：Coze

Coze（扣子）是字节跳动推出的 AI Bot 开发平台，同样提供可视化工作流编排。

与 Dify 的差异：

  • 插件生态更丰富（飞书、抖音生态集成）
  • 更适合构建对话式 AI Bot，工作流能力相对 Dify 更偏对话场景
  • 国内部署和合规体验更好

3. 开发框架：LangGraph

LangGraph 是 LangChain 生态中的工作流编排框架，用**代码定义状态图（State Graph）**来编排 AI 工作流。与 Dify 的低代码路线不同，LangGraph 面向开发者，提供最大程度的控制力。

核心概念：

  • State（状态）：工作流中流动的数据对象，每个节点读取和更新 State
  • Node（节点）：工作流中的执行单元——可以是 LLM 调用、工具调用、任意 Python/Java 函数
  • Edge（边）：节点之间的连接——普通边（固定路由）和条件边（动态路由）
  • Graph（图）：由 Node 和 Edge 组成的有向图，定义了完整的执行流程

LangGraph 的核心优势：

  • 循环与条件分支：支持复杂的图结构（不只是 DAG），节点可以形成循环（如反复检索直到找到足够相关信息）
  • 状态持久化：内置 Checkpoint 机制，流程执行状态可以持久化，支持暂停、恢复、重放
  • 人机协同：原生支持 interrupt 机制，在指定节点暂停等待人工输入
  • 流式输出：支持节点级别的流式输出，用户可以看到流程的实时进展

五、工作流引擎与业务系统的集成

1. 工作流节点调用现有 Service 与 DAO

工作流中最大的工程价值之一是复用现有的 Java 业务服务。AI 工作流节点可以直接注入和调用 Spring Bean（Service、DAO、第三方 API 客户端），无需通过 HTTP 中转。

2. 工作流中的事务与补偿

AI 工作流跨多个节点执行，且 LLM 节点本身不支持事务回滚。需要设计补偿机制——当流程中的某一步失败时，执行补偿操作撤销前面已完成步骤的副作用。

补偿机制的核心原则：每一步”写操作”后立即注册对应的补偿函数，异常发生时逆序执行补偿。对于 LLM 节点的输出无法回滚的情况（LLM 的输出已经发送给用户），通过人工介入和审计日志来补救。

3. 工作流的监控与可观测性

AI 工作流的生产运维需要比传统工作流更丰富的监控维度——不仅要监控技术指标（延迟、错误率），还要监控 AI 质量指标（LLM 节点输出质量、检索召回率）。

AI 工作流监控的核心不是看”流程跑通了没有”，而是回答三个问题：哪些路径是常见的（优化主路径）、哪些节点是瓶颈（延迟和 Token 消耗最高）、哪些条件分支的 LLM 判断不稳定（准确率波动大）。这些问题的答案直接指导工作流的设计优化。

一、Agent 的基本概念与定位

1. 什么是 Agent

Agent（智能体）是一种以 LLM 为推理核心、具备自主决策能力的 AI 应用形态。与被动等待用户指令的 Chatbot 不同，Agent 能够感知环境、制定计划、选择工具、执行动作、反思结果，在一个闭环中自主完成复杂任务。

用工程语言来定义：Agent = LLM（推理引擎）+ Planning（规划能力）+ Memory（记忆系统）+ Tools（工具集合）+ Reflection（反思机制）。这五个组件以 LLM 为中心协同工作，使 Agent 具备了超越单次问答的自主任务执行能力。

2. Agent 与 Chatbot、RAG、Workflow 的区别

这是 AI 应用架构中最基础也是最容易混淆的四个概念。它们的差异不在于”用了什么技术”，而在于谁掌握决策权。

维度 Chatbot RAG AI Workflow Agent
决策主体 用户 系统（固定流程） 流程引擎（预定义图） LLM（自主决策）
执行路径 单轮 Q&A 检索 → 生成（固定两阶段） 预定义的有向图 LLM 动态规划
灵活性 最低（一问一答） 低（固定模式） 中（分支预定义） 最高（自主选路）
可预测性 最高 高 高 低（模型可能走非预期路径）
适用任务 简单问答 知识库问答 标准化业务流程 开放式复杂任务

形态演进关系：

Agent 不是对前三种形态的替代，而是对它们的能力升级。一个 Agent 在执行任务时，内部可能调用 RAG 检索知识、执行预定义的 Workflow 子流程、进行多轮对话——但它自己决定何时调用什么、以什么顺序调用、什么时候任务算完成。

3. Agent 的自主层级

并非所有自称 Agent 的系统都有相同的自主程度。根据 LLM 掌握的控制权大小，可以划分出三个层级：

层级 控制权分布 典型表现 风险
L1：工具增强型 用户控制流程，Agent 只提供工具 用户说”查订单”，Agent 调用查询工具返回结果 低
L2：任务委派型 用户定义目标，Agent 规划并执行 用户说”帮我处理这个退款”，Agent 自主规划：查订单→判断可退款→创建退款单→通知用户 中
L3：完全自主型 Agent 持续运行，主动感知和响应 运维 Agent 持续监控系统指标，发现异常后自主排查、告警、甚至自动修复 高

目前企业级应用的最佳实践是L2 任务委派型——Agent 在明确的边界和约束下自主完成任务。L3 完全自主型在风险控制上仍有较大挑战，仅在少数高度受控的场景中可行。

二、Agent 的核心组成

1. Planning（规划能力）

Planning 是 Agent 区别于简单 Chatbot 的核心能力——面对一个复杂目标，Agent 不是一步到位，而是先制定执行计划，再逐步执行。

规划的两个层次：

  • 任务分解（Task Decomposition）：将复杂目标拆解为多个可执行的子任务。例如，”帮我准备 Q3 技术分享”被分解为”① 检索近半年的技术文档 ② 分析高频话题 ③ 选择 3 个候选主题 ④ 为每个主题生成大纲”
  • 执行排序（Task Ordering）：确定子任务的执行顺序和依赖关系。Agent 需要判断哪些任务可以并行（如同时检索多个数据源），哪些任务有依赖（必须先检索才能分析）

规划的两种生成方式：

方式 描述 优势 劣势
一次性规划 在开始执行前生成完整计划 全局视角，步骤之间逻辑一致 无法应对中途的意外发现
增量规划 执行一步，基于结果规划下一步 灵活，能基于新的信息调整 可能偏离全局最优路径

实践中，大多数 Agent 实现采用增量规划——执行每步后观察结果，再决定下一步。这更接近人类解决复杂问题的方式，也更好地利用了 LLM 的推理能力。

2. Memory（记忆系统）

Memory 是 Agent 实现上下文连续性的基础。没有记忆的 Agent 每一轮都像”失忆”一样重新开始。

三种记忆类型：

短期记忆的管理挑战：LLM Context Window 有限，对话历史增长到一定程度必须截断。简单的截断策略（如只保留最后 N 轮）会丢失早期的关键信息。更高级的做法是记忆摘要——用 LLM 将历史对话压缩为摘要，保留关键信息的同时控制 Token 消耗。

长期记忆的实现：长期记忆本质上是一个RAG 系统——将历史任务的经验、用户偏好存储为向量，当 Agent 处理新任务时检索相关记忆。例如，Agent 之前帮用户处理过”退款”任务，当下次用户再说”和上次一样退了吧”，Agent 能检索到上次的操作参数并复用。

3. Tool Use（工具使用）

Tool Use 是 Agent 与外部世界交互的接口。第 07 章 Function Calling 已详细讨论了工具定义和调用机制，在 Agent 语境下需要补充的是工具的使用策略。

Agent 对工具的使用比 Workflow 中预定的工具调用更灵活：

  • 自主选择：Agent 自己决定何时使用哪个工具——不需要流程图中预先指定
  • 组合使用：Agent 可以将多个工具组合使用——先用搜索工具找到相关文档，再用计算工具分析数据，最后用邮件工具发送报告
  • 错误恢复：工具调用失败时，Agent 可以分析失败原因并尝试替代方案——工具超时了就换一个查询策略，而不是简单报错

4. Reflection（反思与自我修正）

Reflection 是 Agent 在执行过程中评估自己的进展并调整策略的能力。它是 Agent 区别于”一次性输出”的关键特征——Agent 不是说一次就完，而是反复检查、修正、优化。

Reflection 的三个层次：

  • 结果评估：”工具返回的结果是否满足了我的需求？如果不满足，我该如何调整查询？”
  • 进展评估：”在完成总体目标的路径上，我进展到哪一步了？方向是否正确？”
  • 策略反思：”我目前的方法是最优的吗？有没有更好的方式？前几步的策略选择是否有改进空间？”

Reflection 的触发时机：

  • 每一步后：执行工具调用后，评估结果质量，决定是否需要调整
  • 阶段性检查：完成一组子任务后，检查进展是否符合预期
  • 遇到障碍时：工具报错、结果异常、信息不足时，分析原因并调整策略

5. Skills（技能体系）

Skills 是比 Tools 更高层次的抽象——Skill 是一组相关 Tool 的封装，加上使用这些 Tool 的领域知识和操作规范。

如果说 Tool 是”函数”（单一操作，输入→输出），Skill 就是”模块”（封装了领域知识、操作流程和多个 Tool 的协作方式）。例如：

  • Tool：queryDatabase(sql) ——执行一条 SQL 查询
  • Skill：DatabaseDiagnosis ——包含了查询慢查询日志、分析索引使用情况、检查连接池状态、生成优化建议等一系列操作，以及”先看整体指标再定位具体问题”的诊断方法论

Skill 的组成要素：

6. Skills 与 Tools 的区别与协作

维度 Tool（工具） Skill（技能）
粒度 原子操作 复合能力
封装内容 函数签名 + 描述 多个 Tool + 领域知识 + 操作规范
谁来选择 Agent / LLM 自主决定 Agent 选择 Skill，Skill 内部自行调度 Tool
知识包含 无 包含领域知识和最佳实践
类比 JDBC executeQuery() OrderService（封装了业务逻辑）
复用单元 单个 API 调用 可复用的业务能力模块

协作模式：

Agent 在规划阶段选择需要使用的 Skill，每个 Skill 在执行阶段自主调度内部的 Tool。这种两层架构降低了 Agent 的决策复杂度——Agent 不需要在数十个原子 Tool 中选择，而是从几个 Skill 中选择；具体的 Tool 调度交给 Skill 内部处理。

Skills 的设计直接影响了 Agent 的实际效能——精心设计的 Skills 封装了领域专家的判断逻辑，让 Agent 的决策更接近专家行为；设计粗糙的 Skills 只是一堆函数的堆砌，反而增加了 Agent 选择的困难。

三、Agent 的三种核心循环模式

Agent 的行为驱动核心是一个循环（Loop）：感知 → 思考 → 行动 → 观察 → 反思 → 再思考…。不同的循环模式决定了 Agent 的决策方式。

1. ReAct Loop（推理-行动循环）

ReAct（Reasoning + Acting）是最经典也是使用最广泛的 Agent 循环模式。其核心思想是推理和行动交替进行：每一步 Agent 先输出推理（分析当前状态、思考下一步做什么），然后执行一个动作（调用工具或输出结果），观察动作结果，再进入下一轮推理。

ReAct 的每一步都强制 Agent 输出 “思考→行动→观察” 的完整链条，这使得 Agent 的决策过程完全透明——你不仅看到它做了什么，还看到它为什么这样做。这种透明性在调试和审计时至关重要。

2. Plan-Execute（规划-执行模式）

Plan-Execute 模式将 Agent 的决策分为两个阶段：先制定完整计划，再逐步执行。它比 ReAct 更适合目标明确、步骤可预见的任务。

Plan-Execute vs ReAct 的选择：

ReAct Plan-Execute
适用任务 探索性强，路径不确定 目标明确，步骤可预见
效率 每步都推理，Token 消耗大 规划一次，执行多步，效率更高
灵活性 高，能随时调整 中，需要回到规划阶段调整
用户可见性 逐步看到进展 可以先审查计划再执行

实践中，Plan-Execute 和 ReAct 并不互斥。可以在宏观层面使用 Plan-Execute（先规划整体策略），在微观层面使用 ReAct（执行每个步骤时灵活调整）。

3. Reflection Loop（反思-改进循环）

Reflection Loop 在 ReAct 或 Plan-Execute 的基础上增加了一个后置的反思阶段——任务执行完成后，Agent 反思自己的表现，总结经验教训，将改进点存入长期记忆。

Reflection Loop 的价值在于让 Agent 从经验中学习，随着使用次数增长变得越来越”懂”如何高效完成任务。这是 Agent 从”能干活”进化到”会干活”的关键机制。

四、单 Agent、多 Agent 与工程化约束

1. 单 Agent 架构

单 Agent 架构是最基础的 Agent 形态——一个 LLM 实例负责所有的推理、规划和工具调用。它简单直接，适合中等复杂度的任务。

单 Agent 的适用边界：

  • 任务边界清晰，不涉及多个专业领域
  • 可用的工具数量在 10-20 个以内
  • 任务的决策链不会过长（< 15 轮 ReAct 循环）
  • 用户期望的完成时间在分钟级（非小时级）

当任务跨多个不相关的专业领域，或者工具数量膨胀到 LLM 难以有效选择时，单 Agent 的效率和准确性都会下降——这时需要考虑多 Agent 协作。

2. 多 Agent 协作架构

多 Agent 架构将任务分配给多个专业化的 Agent，每个 Agent 负责一个子领域，通过协作完成复杂任务。

常见协作模式：

模式 描述 适用场景
主从模式（Orchestrator-Worker） 一个主 Agent 负责任务分解和调度，多个 Worker Agent 负责执行具体子任务 有明确主流程的多步骤任务
对等协作（Peer-to-Peer） 多个专业 Agent 平等对话，各自贡献专业视角 需要多领域交叉分析的任务
辩论模式（Debate） 多个 Agent 分别生成答案，互相质疑和评审，最终融合 需要高质量、高可靠性输出的任务
分层模式（Hierarchical） 多层 Agent 结构，上级负责策略，下级负责执行，层层分解 超复杂任务（如大型系统重构）

多 Agent 的工程挑战：

  • 通信开销：Agent 之间的消息传递增加了 Token 消耗和延迟
  • 一致性：多个 Agent 可能给出矛盾的信息或建议，需要融合机制
  • 错误传播：一个 Agent 的错误决策可能影响下游 Agent 的判断
  • 调试困难：多 Agent 系统的决策链更长，定位问题根因更难

3. Agent 的工程化约束

Agent 的自主性是一把双刃剑——灵活性的代价是失控风险。工程化约束是让 Agent 在安全边界内发挥能力的必要手段。

约束一：行动边界（Action Boundary）

不是所有工具都应该暴露给 Agent。区分”安全工具”（只读查询、数据分析）和”危险工具”（删除数据、发送外部消息、修改配置）。危险工具需要额外的确认机制或完全禁止 Agent 自主调用。

约束二：步数限制（Step Limit）

Agent 可能在复杂任务中陷入无限循环——反复搜索、反复调整但无法收敛。设置最大 ReAct 步数（如 15 步），超过限制时强制终止并给出当前状态摘要。

约束三：Token 预算（Token Budget）

Agent 的每次推理、每次工具调用都消耗 Token。复杂任务可能消耗数万甚至数十万 Token。为每个 Agent 任务设置 Token 预算上限，超出预算时优雅降级。

约束四：用户确认（Human Confirmation）

对于高风险操作，Agent 执行前必须获得用户确认。不是停止 Agent，而是”Agent 规划好方案 → 展示给用户审核 → 用户确认后执行”——Agent 仍然自主规划，但执行权受控。

约束五：沙箱化（Sandboxing）

Agent 的工具调用应该在受控的环境中执行——数据库只读副本、API 调用的参数校验层、文件操作的限制目录。Agent 永远不能直接访问生产系统的关键资源。

一、AI 在后端开发中的角色定位

以 ChatGPT 在 2022 年底发布为起点，大语言模型（Large Language Model, LLM）在过去几年间以前所未有的速度渗透到软件工程的各个层面。LLM 是一种基于海量文本数据训练得到的深度学习模型，能够理解自然语言并生成类人文本。对于后端工程师而言，最重要的问题不是”AI 会不会取代我”，而是”我应该把 AI 放在系统的什么位置”。

1. AI 不是替代者，而是增强者

AI 并不替代后端。后端系统本质上解决的是数据的存储、流转、计算和权限控制问题，这些职责不会因为 AI 的出现而消失。真正发生变化的是后端能力的边界被大幅拓宽了——很多以前后端做不到或者做不好的事情，现在可以做到了。

举一个具体的例子：一个传统的工单系统，后端能做的是 CRUD（创建、查询、更新、删除工单），以及基于规则的状态流转。接入 AI 之后，后端可以额外提供的能力包括但不限于：

  • 智能分类：根据工单内容自动判断所属类别和优先级，背后是 LLM 的文本分类能力
  • 相似工单检索：从历史工单中找到与当前问题最相似的处理记录，背后是 Embedding 向量化与语义搜索
  • 自动摘要：将冗长的工单描述压缩为关键信息摘要，方便客服快速定位问题
  • 知识库问答：基于产品文档自动回答常见问题，背后是 RAG（Retrieval-Augmented Generation，检索增强生成）

这些能力不是替代后端逻辑，而是嵌入到后端业务流程中，成为新的业务能力层。

2. 从”业务实现者”到”AI 能力编排者”

传统 Java 后端工程师的核心工作是：理解需求 → 设计数据模型 → 编写 Service/DAO 层代码 → 暴露 REST API。这是一条确定的、可预测的工作链路。

AI 时代，后端工程师的角色发生了微妙但重要的迁移：你不再只是自己实现业务逻辑，而是要学会编排 AI 来完成逻辑。这意味着你需要同时理解：

  • 业务本身（和以前一样重要）
  • AI 能做什么、不能做什么（能力边界判断）
  • 如何把业务问题分解为”AI 适合做的部分”和”传统代码适合做的部分”（任务分解与编排）
  • 如何在不可靠的 AI 输出之上构建可靠的系统（工程化保障）

这种角色的变化，可以类比为从”手工艺人”到”工厂产线设计师”的转变——你不是不干活了，而是把精力从亲自完成每一道工序，转向设计工序之间的流转规则与质量检验标准。

3. 后端工程师的独特优势

在 AI 应用开发领域，后端工程师拥有一些其他人不具备的优势：

  • 系统设计能力：你知道如何设计高可用、可扩展的系统，而 AI 应用的工程化落地最终依赖的正是这些能力
  • 数据理解：你清楚数据从哪里来、流经哪些系统、最终去哪里，而 AI 应用的核心就是对数据的加工和增强
  • 工程化经验：你熟悉 CI/CD、监控告警、灰度发布、容错降级——这些是 AI 应用从 Demo 走向生产的关键
  • 业务边界意识：你知道哪些操作需要权限校验、哪些数据不能外泄、哪些行为需要审计——这是 AI 安全的基础

换句话说，训练模型可能是算法工程师的强项，但把模型变成可靠的生产服务，是后端工程师的主场。

二、后端系统中 AI 的三种集成深度

AI 在后端系统中的集成不是”全有或全无”的二元选择，而是一个渐进深入的连续谱。根据模型在决策链路中的自主程度，可以划分为三个层级。理解这三个层级的关系，有助于你在实际项目中做出合适的架构决策。

1. Level 1：API 调用层

这是最低门槛的集成方式。后端像调用任何第三方 API 一样调用 LLM 的接口，把模型的返回结果当作一个”高级函数”的输出来使用。

典型架构：客户端 → 后端 Service → LLM API → 后端处理结果 → 返回客户端

适用场景：文本分类、情感分析、内容摘要、翻译、代码注释生成等输入确定、输出可直接消费的单次调用任务。

关键特征：模型在这里只是一个被动的计算单元，它不做任何决策，只是根据输入生成输出。后端代码完全掌控调用时机、参数和结果处理。

这种模式的技术门槛最低，但它存在两个明显局限：一是模型无法访问企业内部数据（它只能基于训练数据回答），二是模型无法执行任何实际操作（它只能说，不能做）。

2. Level 2：能力编排层

在这个层级，模型不再孤立工作，而是与外部知识库和业务系统连接起来。这主要通过两种机制实现：

  • RAG（检索增强生成，Retrieval-Augmented Generation）：在调用模型之前，先从向量数据库或搜索引擎中检索出与用户问题相关的文档片段，将这些片段作为上下文注入 Prompt，让模型基于”外部知识”而非仅凭”训练记忆”来生成回答
  • Function Calling（函数调用，也称 Tool Calling）：让模型输出结构化的函数调用请求，后端据此执行真实的业务操作（查数据库、调接口、发消息等），将执行结果反馈给模型继续对话

典型架构：客户端 → 后端 Service → 知识检索（向量数据库/搜索引擎） → 构造增强 Prompt → LLM API → 结果处理（可能触发 Function Call → 执行业务逻辑 → 结果回传 → 模型继续推理）→ 返回客户端

适用场景：智能客服、企业知识库问答、业务查询助手（”帮我查一下这个订单的状态”）、文档审查等需要外部知识 + 业务操作的场景。

关键特征：模型开始有了”眼睛”（能看外部知识）和”手”（能调用业务接口），但它仍然按照预定义的流程执行——先检索还是先调工具、检索哪些数据源、工具调用的顺序，这些都由后端代码预先编排好。模型在这个层级是”被赋能”的，但不是”自主”的。

Level 2 是目前企业级 AI 应用的主流深度，它兼顾了能力增强和可控制性，覆盖了约 80% 的实际业务场景。

3. Level 3：Agent 自主决策层

Agent（智能体）是当前 AI 应用开发的最高集成深度。在这个层级，模型不再按照预先编排的固定流程执行，而是自主规划任务步骤、自主选择调用哪些工具、自主判断任务是否完成。

典型架构：用户给定目标 → Agent 自主分解为子任务 → 自主选择工具执行每个子任务 → 观察执行结果 → 根据结果调整计划 → 循环直到目标达成 → 返回最终结果

关键特征：模型从”执行者”变为”决策者”。后端工程师的角色从编排每一步做什么，变为定义模型可用的工具集（Tools）、设定行为约束（System Prompt）、设置安全边界（Guardrails），然后让模型在这些约束内自主行动。

Agent 的典型循环模式是 ReAct（Reasoning + Acting，推理与行动交替）：模型先思考下一步应该做什么（Reasoning），然后调用工具执行（Acting），观察工具返回结果，再基于结果思考下一步……如此往复直至任务完成。

适用场景：自动化运维（”检查所有服务健康状态，对有问题的服务查看日志并尝试重启”）、复杂数据分析（”分析这个季度各区域的销售情况，找出异常点并给出建议”）、多步骤业务流程自动化。

Agent 的自主性既是它的优势也是风险来源：模型可能做出错误的工具选择、陷入循环、或被恶意 Prompt 诱导执行危险操作。因此 Level 3 对安全设计、可观测性和人工干预机制的要求远高于前两个层级。

4. 三种深度的关系与选型原则

三种深度不是互相替代的关系，而是层层叠加的关系。Level 2 包含 Level 1 的所有能力，Level 3 又包含 Level 2 的所有能力。

选型原则很简单：不是越深越好，够用就好。如果 Level 1 能解决问题就不要上 Level 2，Level 2 能覆盖就不要上 Agent。每上升一个层级，系统的复杂度、不可控性和调试难度都会显著增加。实践中，大多数有价值的 AI 应用都落在 Level 2。

三、AI 给后端架构带来的五个核心变化

AI 的集成不是简单地”加一个 API 调用”0，而是从底层改变了后端系统的若干基本假设。理解这些变化，是做好 AI 后端工程化的前提。

1. 从确定性到概率性

传统后端的核心逻辑是确定性的：给定相同的输入和相同的数据库状态，输出一定是相同的。单元测试、集成测试、回归测试都建立在这个前提之上。

AI 改变了这个假设。LLM 的输出本质上是概率性的——同一个 Prompt 调用两次，模型可能给出措辞不同甚至结论不同的回答。Temperature（温度参数，控制输出的随机性，值越低输出越确定、越高越有创造性）可以调节随机程度，但不能消除它。

这对后端工程意味着：

  • 测试策略需要改变：无法再用”期望输出等于某字符串”的方式写测试，需要引入语义相似度评判、结构化输出 Schema 校验、LLM-as-Judge（用模型评判模型输出）等新的测试方法
  • 容错设计需要加强：必须假设模型可能输出不合法 JSON、遗漏关键字段、或给出不合逻辑的结论，在代码层做校验和兜底
  • 监控需要新维度：除了传统 QPS 和延迟，还需要监控幻觉率、输出格式合规率、内容安全违规率等

2. 从纯代码到 Prompt + Code

传统后端业务逻辑全部用代码（Java/Go/Python 等）编写。AI 引入后，业务逻辑扩展为三层结构：

  • Prompt 层：用自然语言描述任务、设定规则、提供示例，本质是在”编程”模型的行为
  • 工具函数层：传统 Java 代码，实现模型无法完成的确定性操作（数据库查询、权限校验、数值计算等）
  • 编排代码层：Java 代码负责串联 Prompt 和工具函数，控制调用顺序、处理异常、管理状态

Prompt 从”一段随手写的文本”升级为需要工程化管理的一等公民：版本管理、A/B 测试、效果评测、回滚机制——这些传统代码享有的工程实践，Prompt 同样需要。

3. 从 REST 到多协议通信

传统后端系统的对外通信几乎全是 REST（Representational State Transfer，表现层状态转移）风格的 HTTP 请求/响应。AI 集成引入了两种重要的新通信模式：

  • SSE（Server-Sent Events，服务端推送事件）：用于 LLM 的流式输出。模型边生成边推送 Token，用户可以看到逐字输出的效果，大幅改善体验。Spring Boot 中通过 SseEmitter 或 WebFlux 的 Flux<ServerSentEvent> 实现
  • MCP（Model Context Protocol，模型上下文协议）：Anthropic 提出的标准化协议，用于模型与外部工具/数据源之间的通信。MCP 定义了统一的 Server-Client 架构，使得不同的 AI 客户端可以复用同一套工具实现

理解 REST 的工程师需要适应 SSE 的长连接语义和背压处理（Backpressure，当下游消费速度慢于上游生产速度时进行的流量控制），以及 MCP 的松耦合工具注册模型。

4. 从固定成本到 Token 经济

传统后端系统的成本主要由服务器、带宽和数据库组成，这些成本随请求量线性增长，且边际成本相对固定。

AI 应用引入了一个全新的成本维度：Token 消耗。Token 是 LLM 处理文本的基本单位（英文中大致相当于 3/4 个单词，中文中大致相当于 1-2 个汉字）。每次 LLM 调用都按 Token 数量计费，输入和输出分别计价，且不同模型的价格差异可达数十倍。

这对架构的影响是深远的：

  • 缓存的价值被重估：传统后端中缓存主要用于提升性能，在 AI 应用中，缓存还直接节省 Token 成本——两个用户问相似问题，如果命中语义缓存（Semantic Cache），就完全不需要调用模型
  • 模型分层成为刚需：简单任务用便宜的小模型（如 DeepSeek-V3、Qwen-Turbo），复杂任务用贵的大模型（如 GPT-4、Claude Opus），是成本优化的基本策略
  • Prompt 设计要精打细算：Prompt 中的每个 Token 都花在刀刃上。过多的 System Prompt 和 Few-shot 示例虽然可能改善效果，但也显著抬高了每次调用的成本

5. 从权限控制到意图安全

传统后端安全主要围绕认证（Authentication，验证你是谁）和授权（Authorization，验证你能做什么）展开，即 RBAC（Role-Based Access Control，基于角色的访问控制）。

AI 引入后，攻击面扩大了：

  • Prompt Injection（提示注入）：攻击者通过在用户输入中嵌入恶意指令，试图覆盖 System Prompt 中的行为约束。例如在客服系统中输入”Ignore all previous instructions and…”来劫持模型行为
  • 工具越权调用：在 Agent 模式下，模型自主选择调用哪些工具。如果工具的权限边界设计不当，模型可能被诱导调用它不应该调用的接口
  • 数据泄露：模型在生成回答时可能不经意地复述训练数据中的敏感信息，或 RAG 检索出了用户无权访问的文档内容

这意味着后端安全体系需要增加一个新的维度：意图安全（Intent Security）——不只是控制”谁”能访问”什么”，还要防御”恶意输入意图”对模型行为的操控。

四、后端 AI 应用的技术要素全景

AI 后端开发涉及的技术要素可以归纳为七个层次。理解每一层的职责和它们之间的依赖关系，有助于形成系统化的知识框架。

1. 模型层

模型层是 AI 应用的基础设施，负责提供语言理解和生成能力。后端工程师需要掌握的核心概念包括：

  • LLM（大语言模型）：在海量文本数据上训练得到的深度学习模型。主流模型包括 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列、以及国产的 DeepSeek（深度求索的高性价比模型）、Qwen（阿里通义千问）、GLM（智谱清言）等
  • Token（词元）：模型处理文本的最小单位。一个中文字通常对应 1-2 个 Token，一个英文单词通常对应 1-3 个 Token。Token 数量决定了模型调用的成本和延迟
  • Context Window（上下文窗口）：模型单次能处理的最大 Token 总数（输入 + 输出之和）。主流模型的上下文窗口从 128K 到 2M Token 不等，决定了能”塞进去”多少信息
  • Temperature（温度）：控制输出随机性的参数，范围 0-2，值越低输出越确定和保守，越高输出越有创造性但也更容易偏离主题

2. Prompt 层

Prompt（提示词）是人与模型之间的通信协议。它的核心任务是将业务需求翻译为模型能理解并正确执行的自然语言指令。

一个结构良好的 Prompt 通常包含以下要素：

  • 角色设定（Role/Persona）：定义模型扮演的角色，如”你是一个资深的 Java 后端工程师”
  • 任务描述（Task）：明确需要模型完成的具体任务
  • 上下文信息（Context）：任务相关的背景知识和数据
  • 约束条件（Constraints）：输出格式要求、内容限制、行为边界
  • 示例（Few-shot Examples）：通过一至多个输入-输出示例，让模型理解期望的行为模式

Prompt 的核心方法论包括 Zero-shot（不提供示例，直接给指令）、Few-shot（提供少量示例）、Chain of Thought（思维链，要求模型展示逐步推理过程）和 ReAct（推理与行动交替）。Prompt 工程的目标不是写出最复杂的 Prompt，而是找到用最小的 Token 开销达到稳定效果的 Prompt。

3. 知识层

LLM 的知识截止于其训练数据的时间点，且无法访问企业内部的私有数据。知识层通过引入外部知识来解决这两个问题：

  • Embedding（向量嵌入）：将文本映射为高维空间中的稠密向量。语义相近的文本，其向量在空间中的距离也相近。Embedding 是实现语义搜索的基础
  • 向量数据库（Vector Database）：专门存储和检索向量的数据库。核心能力是基于向量相似度（如余弦相似度、欧氏距离）快速找到 Top-K 个最相似的向量。常见方案包括 Milvus、Qdrant、Weaviate、Pinecone，以及传统数据库的向量扩展如 pgvector
  • RAG（检索增强生成）：在调用 LLM 之前，先检索出相关的外部知识片段，注入到 Prompt 的上下文中，让模型基于这些知识生成回答。RAG 的核心流程是：文档加载 → 文本切分（Chunking）→ 向量化 → 存储 → 检索 → 增强生成

4. 行动层

模型能”说”但不能”做”。行动层通过 Function Calling 机制让模型能够触发真实的业务操作：

  • Function Calling（函数调用）：在调用 LLM 时，不仅传 Prompt，还传一组可用函数的定义（名称、描述、参数 Schema）。模型在生成回答时，如果判断需要调用某个函数来获取信息或执行操作，它不直接执行函数，而是返回一个结构化的函数调用请求（包含函数名和参数）。后端代码解析这个请求、执行实际的业务函数、将结果回传给模型，模型再基于结果继续推理
  • Tool Calling（工具调用）：Function Calling 的广义延伸，不限于后端函数，也包括调用外部 API、操作数据库、读写文件等

Function Calling 是连接”AI 推理”和”业务执行”的桥梁，也是 Level 2 和 Level 3 集成的关键技术。

5. 编排层

当单个 LLM 调用不够用时（多步推理、多数据源、多工具协作），编排层介入来管理执行流程：

  • Workflow（工作流）：预先定义好的、确定性的执行流程。每一步做什么（调 LLM、查数据库、调工具、等待人工审批）由代码决定。Dify、Coze、LangGraph 等平台提供了可视化的流程编排能力
  • Agent（智能体）：模型自主决定执行流程。Agent 根据目标自行分解任务、选择工具、判断完成条件

两者的本质区别在于决策权在谁手上：Workflow 的决策权在工程师（确定性），Agent 的决策权在模型（自主性）。

6. 框架层

框架层提供统一的抽象来减少重复代码。Java 生态中两个主要的 AI 集成框架：

  • Spring AI：Spring 生态的官方 AI 扩展。提供 ChatClient（统一模型调用）、EmbeddingClient（统一向量化）、VectorStore（统一向量数据库访问）、@Tool 注解（声明工具函数）等开箱即用的能力，与 Spring Boot 深度整合
  • LangChain4j：LangChain 的 Java 实现，设计上更接近 LangChain/Python 的编程模型，提供 ChatLanguageModel、EmbeddingModel、AiServices、@Tool 等 API

7. 工程层

工程层解决 AI 应用从”能跑”到”能上线”的问题，包括安全防护、缓存策略、成本追踪、可观测性（Observability，通过 Log/Trace/Metrics 了解系统内部运行状态的能力）和评测体系。这一层是目前大多数 AI 应用从 Demo 阶段迈向生产环境的主要瓶颈。

五、典型的 AI 后端架构模式

将上述技术要素组合起来，实践中形成了几种可复用的架构模式。

1. 简单代理模式（Proxy Pattern）

这是最简单的集成模式。后端作为 LLM API 的代理层，封装 API Key、格式化请求、处理响应。后端在这里的核心价值是安全和控制：不暴露 API Key 给前端、记录调用日志、实施访问控制、对输入输出做内容审核。

适用场景：文本润色、翻译、摘要、代码注释生成等无需外部知识的单次任务。

2. RAG 知识库模式（RAG Pattern）

核心流程是一个完整的 ETL（Extract-Transform-Load，数据抽取-转换-加载）Pipeline：

  1. 离线阶段（索引构建）：文档上传 → 解析（PDF/Word/Markdown → 纯文本）→ 文本切分（按语义边界切为适当大小的 Chunk）→ Embedding 向量化 → 存入向量数据库
  2. 在线阶段（检索生成）：用户提问 → 问题向量化 → 向量数据库检索 Top-K 相关 Chunk → 将检索结果与用户问题一起构造 Prompt → LLM 生成回答

RAG 模式的关键设计决策包括：Chunk Size（文本块大小，决定检索粒度）、Overlap（相邻块之间的重叠量，防止语义断裂）、检索策略（纯向量 vs 混合检索 vs 多路召回重排序）。

适用场景：企业知识库问答、产品文档助手、合同/法规检索回答、智能客服。

3. 工具增强模式（Tool-Augmented Pattern）

与 RAG 模式提供”知识”不同，工具增强模式提供的是”行动能力”。模型不只是能回答问题，还能帮用户查订单、看库存、创建工单、发送通知。

关键设计要点：

  • 工具定义：每个工具需要清晰的名称、自然语言描述、参数 JSON Schema。描述的清晰度直接影响模型选择工具的准确率
  • 幂等与安全：查询类工具应幂等（多次调用结果一致）；写入类工具需要额外的权限校验和安全确认
  • 超时与降级：工具调用可能超时或失败，需要合理的超时设置和降级策略

适用场景：业务数据查询助手、智能运维助手、企业内部效率工具。

4. Agent 编排模式（Agent Orchestration Pattern）

Agent 模式将流程控制权从代码转移给模型。Agent 的典型执行循环如下：

  1. 思考（Thought）：分析当前状态，决定下一步应该做什么
  2. 行动（Action）：选择并调用合适的工具
  3. 观察（Observation）：获取工具执行结果
  4. 反思（Reflection）：评估是否达成目标，决定继续或结束

关键设计要点：

  • 工具集设计：Agent 可用的工具需要职责清晰、边界明确、描述准确。工具数量不宜过多（通常建议 5-15 个），过多会降低模型选择的准确率
  • 终止条件：Agent 可能陷入无效循环，需要设置最大步数、超时时间和成本上限
  • 人工介入点：对于高风险操作（如删除数据、发送通知），设置人工确认环节（Human-in-the-Loop）

适用场景：复杂数据分析、自动化运维、多步骤业务流程。

5. 混合模式（Hybrid Pattern）

实际的企业级 AI 应用中，很少只使用单一模式。混合模式将上述模式组合起来：

  • 简单问题走 Proxy 模式直接回答
  • 知识类问题走 RAG 模式检索文档
  • 操作类问题走 Tool-Augmented 模式调用后端服务
  • 复杂的多步骤任务走 Agent 模式自主编排

混合模式需要一个**路由层（Router）**来判断请求应该走哪条路径——这本身可以是一个轻量级 LLM 调用，也可以是基于规则的分类器。混合模式是构建企业 AI 中台（AI Middle Platform）的典型架构。

六、Java 在 AI 生态中的定位

1. 正视 Java 在 AI 领域的真实位置

AI 模型的训练、微调和研究的首选语言是 Python，这一点短期内不会改变。PyTorch、Transformers、vLLM 等核心库都是 Python 生态的产物。

但 AI 应用不等于 AI 模型训练。AI 工程化落地与企业级服务化的主力语言，Java 仍然占据重要位置。绝大多数企业的后端技术栈是 Java/Spring Boot，它们需要的不是用 Java 重新训练模型，而是把模型能力可靠地集成到现有系统中。

因此 Java 在 AI 生态的定位是清晰的：Python 负责造发动机（模型训练与推理），Java 负责造整车（将发动机封装为可驾驶的汽车，配上方向盘、刹车、仪表盘）。

2. Java 生态的核心优势

在 AI 应用开发场景下，Java 生态有几项关键优势：

  • Spring Boot 的成熟度：依赖注入、自动配置、Starter 机制、Actuator 监控等能力，使得 AI 集成框架（如 Spring AI）可以以与 Spring Data、Spring Security 一致的方式融入现有体系
  • 微服务治理：服务发现、负载均衡、熔断降级、分布式追踪等能力，对于需要调用多个模型和外部服务的复杂 AI 应用至关重要
  • 数据基础设施集成：Java 在关系型数据库（MySQL/PostgreSQL → MyBatis/JPA）、缓存（Redis → Spring Cache）、消息队列（Kafka/RabbitMQ → Spring Messaging）等方面的集成生态极为成熟——而 AI 应用恰好需要频繁地与这些基础设施交互
  • 运维与监控：Micrometer + Prometheus + Grafana 的技术栈可以平滑延伸覆盖 AI 应用的 Token 消耗、模型延迟等新指标

3. Java 调用 AI 的典型模式

Java 后端调用 AI 能力的典型模式不是”用 Java 跑模型”，而是 Java 作为 HTTP 客户端 + 编排层：

  • 通过 HTTP 调用 OpenAI 兼容 API（几乎所有国产模型都兼容 OpenAI 的 API 格式）
  • 通过 SSE 流式消费模型的输出
  • 通过 MCP 协议注册和管理工具
  • 与 Python 服务协作：Python 运行模型推理或 GraphRAG 这类特定服务，Java 通过 HTTP/gRPC 调用

Spring AI 和 LangChain4j 的目标正是为上述模式提供统一的、符合 Java 习惯的抽象层，让 Java 工程师不需要关心底层 HTTP 细节，用熟悉的 @Service、@Tool、ChatClient 等 API 即可集成 AI 能力。

4. 务实的技术选型态度

对于 Java 后端团队，务实的态度是：

  • 模型层面：不造轮子，直接使用成熟的模型 API 服务和托管平台
  • 框架层面：优先评估 Spring AI 或 LangChain4j，避免裸写 HTTP + JSON 解析
  • 特殊需求：需要 GraphRAG、需要模型微调等场景，部署 Python Sidecar 服务，不要用 Java 硬扛
  • 核心竞争力：把精力放在业务理解、架构设计、工程化保障上——这些才是后端工程师不可替代的价值

七、后端工程师需要建立的 AI 思维模型

人适应技术，而不是技术适应人。以下五个思维转变，是 Java 后端工程师在 AI 时代保持竞争力的关键。

1. Prompt 即代码

传统认知中，Prompt 是”写给 AI 看的一段话”，是临时的、一次性的。但在工程实践中，Prompt 应当被视为一种新的编程形式——用自然语言描述逻辑，由 LLM 执行。

这意味着 Prompt 需要享有与代码同等的工程待遇：

  • 版本管理：Prompt 修改应该走 Git，有 commit message 记录变更原因
  • A/B 测试：新旧 Prompt 并行运行，对比效果数据后决定是否切换
  • 效果评测：每次 Prompt 变更应该有对应的评测 Pipeline 自动跑分
  • 回滚机制：效果变差时应能快速回退到上一个 Prompt 版本
  • 代码审查：Prompt 变更也应该经过 Code Review，检查逻辑完整性、边界情况覆盖、Token 开销

2. 模型是第三方依赖

把模型看作一种特殊的第三方依赖，会帮助你建立正确的工程习惯：

  • 模型有版本号（如 gpt-4-0613、claude-sonnet-4-6），版本之间行为可能不同
  • 模型会 deprecate（废弃） 和 sunset（下线），需要关注模型提供商的版本生命周期通知
  • 模型有性能差异，两个同样号称”支持 128K 上下文”的模型，在长文本检索精度上可能差距显著
  • 模型需要 Fallback（故障转移）策略：主模型不可用时自动切换到备用模型，或降级为缓存响应

用管理 Maven 依赖的方式来管理模型：锁定版本、测试升级、监控变化、准备回退方案。

3. 非确定性是常态

在传统后端开发中，非确定性行为（Non-deterministic Behavior）是需要追查和修复的 Bug。在 AI 应用中，非确定性是系统设计的基线假设，不是异常。

如何在一个非确定性的组件之上构建可靠的系统？核心策略是在合适的层面引入确定性：

  • 结构化输出（Structured Output/JSON Mode）：约束模型输出合法 JSON，用 JSON Schema 校验字段完整性，将”非确定性的自然语言”转化为”确定性的结构化数据”
  • 校验与兜底：对模型输出进行规则校验（字段必填、格式合法、值域正确），校验失败时走降级逻辑（返回默认值 / 提示用户重试 / 切换备用模型）
  • 多层防御：不依赖单一 Prompt 调用来做关键决策。重要场景加入二次确认、人工审核、或并行调用多个模型交叉验证

4. Token 即成本

传统后端中的成本优化通常聚焦于服务器资源（CPU、内存、带宽）和数据库查询。AI 应用增加了一个必须持续关注的成本维度：Token 消耗。

在实践中，有效的 Token 成本控制策略包括：

  • 模型分层路由：简单任务用小模型（便宜），复杂任务用大模型（贵但能力强）。路由决策可以由规则驱动（关键词匹配）或由一个低成本的小模型来做
  • 语义缓存（Semantic Cache）：将历史问答记录 Embedding 后缓存。新问题先计算与缓存中问题的相似度，相似度超过阈值则直接返回缓存答案，完全不调用模型
  • 精确匹配缓存：对于固定的 System Prompt + 相同/高度相似的 User Input，直接返回缓存的模型响应
  • Prompt 瘦身：定期审查 Prompt 模板，删除无效的冗余描述、过长的 Few-shot 示例
  • 可观测性驱动优化：必须有 Token 消耗的实时看板，知道钱花在了哪里，才能做有针对性的优化

5. 安全边界外移

传统后端安全模型中，WAF（Web Application Firewall，Web应用防火墙）和 API Gateway 是第一道防线，负责拦截 SQL 注入、XSS 等攻击。但在 AI 应用中，用户输入直接进入 Prompt 上下文并影响模型行为，传统的攻击签名匹配无法识别 Prompt Injection。

安全防线需要从网关层外移到模型交互层：

  • 输入过滤：在构造 Prompt 之前，对用户输入做安全扫描和敏感词过滤
  • Prompt 沙箱：将用户输入放在 Prompt 的特定标记区域内（如 <user_input>...</user_input>），明确告诉模型”这部分是用户输入，不包含系统指令”
  • 工具白名单与权限最小化：Agent 可调用的工具应遵循最小权限原则（Least Privilege），工具实现中应包含独立的权限校验，不依赖模型判断
  • 输出审核：模型输出在返回给用户之前，经过内容安全审核和敏感信息脱敏
  • 全链路审计：记录每次 LLM 调用的完整输入输出，便于事后追溯和分析

一、SQL 基础认知

1. SQL 是什么

  • 核心定义：SQL（Structured Query Language）是用于管理和操作关系型数据库的标准声明式语言。
  • 声明式特性 (Declarative)：
    • 逻辑与实现分离：开发者仅需描述“要什么”（数据目标），无需关心“怎么找”（执行路径）。
    • 自动优化：数据库查询优化器（Optimizer）会根据统计信息自动选择最优的物理执行方案。
  • 标准与方言：
    • 通用性：遵循 ANSI/ISO SQL 标准，核心语法（如 SELECT, JOIN）在主流数据库间高度一致。
    • 差异性：各数据库（MySQL、PostgreSQL、Oracle）在实现标准的基础上，存在独特的**“方言”**（Dialects），差异体现在高级函数、底层调优机制及扩展语法。

BIGINT 大范围整数，精度由实现定义，但标准要求其精度 ≥ INTEGER 超大表主键、分布式 ID、毫秒时间戳

注意：SQL 标准仅规定了类型之间的精度关系：SMALLINT ≤ INTEGER ≤ BIGINT，具体位数由实现决定（大多数实现中分别为 2、4、8 字节）。

1.1.2 定点数

类型 说明 典型用途
NUMERIC(p, s) 精确定点数。p 为精度（总有效位数），s 为标度（小数位数）。实现必须精确提供 p 位精度 货币金额、财务数据、费率
DECIMAL(p, s) / DEC(p, s) 与 NUMERIC 类似，但实现可以提供大于 p 的实际精度 货币金额、财务数据、费率

关键点

  • p（精度 / Precision）：数字中有效位的总数。
  • s（标度 / Scale）：小数点后的位数。
  • NUMERIC vs DECIMAL 的标准差异：NUMERIC(5, 2) 严格存储 5 位有效数字；DECIMAL(5, 2) 允许实现在内部提供更多位数（即 p 是最低保证）。在实际使用中，大多数数据库对两者的实现完全相同。

1.1.3 近似数值类型

类型 说明 典型用途

关键点

  • FLOAT(p) 中的 p 是二进制精度，不是十进制位数。标准规定：p ≤ 24 时效果等同于 REAL；p 在 25–53 之间时效果等同于 DOUBLE PRECISION。

2. 数据完整性约束

数据完整性约束是定义在表或列上的规则，用于防止无效数据进入数据库。它们是确保数据质量的第一道防线，将业务规则强制施加于数据库层面。

2.1 主键约束 (PRIMARY KEY)

主键是表中用于唯一标识每一行数据的列或列的组合。

  • 核心特性：
    • 唯一性 (Unique)：主键列中的每个值都必须是唯一的，不允许重复。
    • 非空性 (Not Null)：主键列中的值不允许为 NULL。
  • 规则：
    • 一个表只能有一个主键。
    • 主键可以由单个列组成（单列主键），也可以由多个列共同组成（复合主键）。
  • 最佳实践： 推荐使用与业务无关的、自增的整数（如 INT UNSIGNED AUTO_INCREMENT）作为主键，这种键被称为“代理键”(Surrogate Key)。

三、SQL 语言核心

1. DDL（数据定义语言）

1.1 核心特征

核心特性：隐式事务提交（Implicit Commit）

这是 DDL 区别于 DML 最重要的特性，也是最容易踩坑的地方：

  • 大多数数据库中，执行一条 DDL 语句会触发隐式提交，将当前会话中所有未提交的事务自动提交。
  • DDL 语句本身在大多数数据库中无法通过 ROLLBACK 撤销。
  • 这意味着：DDL 是破坏性的、不可逆的，执行前必须充分确认。

例外：PostgreSQL 是少数几个将 DDL 纳入事务管理的数据库，支持在事务块中执行 CREATE TABLE、DROP TABLE 等操作并 ROLLBACK。

1.3.2 修改列

修改列在各数据库之间的语法差异较大，是方言最密集的区域之一。

标准 SQL 写法：

``sql
– 修改列的数据类型
ALTER TABLE table_name ALTER [COLUMN] column_name SET DATA TYPE new_type;

– 重命名列
ALTER TABLE table_name RENAME [COLUMN] old_name TO new_name;
``

MySQL 写法：

``sql
– 只修改数据类型/约束（不改名）
ALTER TABLE users MODIFY COLUMN phone VARCHAR(30) NOT NULL;

– 重命名 + 修改类型（必须同时指定类型）
ALTER TABLE users CHANGE COLUMN phone mobile VARCHAR(20) NOT NULL;
– 原名⬆ 新名⬆
``

MySQL 特别注意：CHANGE 即使不改名，也必须把列名写两遍；MODIFY 不能改名但可用于修改类型和约束。

PostgreSQL 写法：

``sql
– 修改数据类型
ALTER TABLE users ALTER COLUMN phone TYPE VARCHAR(30);

– 重命名列
ALTER TABLE users RENAME COLUMN phone TO mobile;
``

原则：INSERT 语句始终显式写出字段列表，这是最基本的工程规范。

3.1.3 DELETE 与逻辑删除的认知

实际业务开发中，真正执行物理 DELETE 的场景其实很少。

原因：

  • 数据一旦物理删除，无法恢复
  • 业务上很多”删除”操作实际上需要保留记录（如订单取消、账号注销）
  • 数据审计、问题排查都依赖历史数据

因此，业务系统中大量使用逻辑删除来替代物理删除。

物理 DELETE 的真实使用场景：

  • 日志类、临时数据的定期清理
  • 开发/测试环境的数据重置
  • 明确不需要保留的过期数据清理

不适合使用逻辑删除的场景：

  • 操作日志定期归档清理：数据量大，物理删除更合适
  • 临时中间表数据清理：无需保留
  • 测试数据清理：物理删除即可

1.1 什么是事务

1.1.1 一组操作要么都成功要么都失败

事务（Transaction）是数据库中一组不可分割的操作序列，这组操作要么全部成功提交，要么全部失败回滚，不存在”部分成功”的中间状态。

用最经典的转账场景来理解：

如果操作一成功、操作二失败（如系统崩溃），就会出现”A 扣了钱，B 没收到”的数据不一致问题。

事务保证了这两个操作是一个整体：

flowchart TD
    A["BEGIN 开启事务"] --> B["执行操作一\nA账户 -100"]
    B --> C{操作一成功?}
    C -- 否 --> R["ROLLBACK\n所有操作撤销\n数据恢复原状"]
    C -- 是 --> D["执行操作二\nB账户 +100"]
    D --> E{操作二成功?}
    E -- 否 --> R
    E -- 是 --> F["COMMIT\n提交事务\n数据永久生效"]

1.1.2 事务在业务一致性中的价值

事务保护的是业务数据的一致性，即多个相关操作要么都完成，要么都不发生。

常见的需要事务保护的业务场景：

  • 转账/支付：扣款 + 入账，必须同时成功
  • 下单：创建订单 + 扣减库存，必须同时成功
  • 积分兑换：扣减积分 + 创建兑换记录，必须同时成功
  • 批量操作：批量更新多条记录，要么全部更新要么全部不更新
  • 退款：更新订单状态 + 退还库存 + 退还积分，必须同时成功

2. ACID 基础理解

2.1 原子性、一致性、隔离性、持久性

ACID 是事务的四个核心特性，是数据库事务的基础理论。

2.1.1 每个特性的直观理解

A - 原子性（Atomicity）

  • 定义：事务中的所有操作，要么全部成功，要么全部失败。

  • 直观理解：事务是不可分割的”原子”，没有中间状态。

  • 保证手段：数据库通过 Undo Log 实现回滚。

  • 转账场景：扣款和入账要么都发生，要么都不发生。

C - 一致性（Consistency）

  • 定义：事务执行前后，数据库从一个合法状态转换到另一个合法状态。

  • 直观理解：业务规则始终满足，如转账前后两个账户的总金额不变。

  • 保证手段：原子性、隔离性、持久性共同保证，加上应用层的业务逻辑。

  • 转账场景：A+B 的总余额在转账前后保持不变。

I - 隔离性（Isolation）

  • 定义：并发执行的多个事务之间互相隔离，一个事务的中间状态对其他事务不可见。

  • 直观理解：多个事务并发执行时，每个事务感觉像是在”独占”数据库。

  • 保证手段：数据库通过锁机制和 MVCC 实现不同隔离级别。

  • 转账场景：A 向 B 转账的过程中，C 查询 A 的余额，应该看到转账前的余额，而不是中间状态。

D - 持久性（Durability）

  • 定义：事务一旦提交，对数据库的修改就是永久的，即使系统崩溃也不会丢失。

  • 直观理解：COMMIT 之后，数据一定在，断电重启也不会消失。

  • 保证手段：数据库通过 Redo Log（WAL）实现崩溃恢复。

  • 转账场景：COMMIT 后，即使服务器立刻断电，转账结果也不会丢失。

2.1.2 对业务开发的实际意义

ACID 特性 对 Java 开发者的实际意义
原子性 @Transactional 保证方法内的操作要么全成功要么全回滚
一致性 业务逻辑正确 + 事务保护 = 数据始终合法
隔离性 并发场景下，需要理解隔离级别来避免数据异常
持久性 COMMIT 后不用担心数据丢失，数据库自己保证

2.2 隔离性带来的并发现象认知

当多个事务并发执行时，如果隔离性不够强，会出现以下几种数据异常现象。这是理解事务隔离级别的基础。

2.2.1 脏读（Dirty Read）

  • 定义：一个事务读到了另一个事务”尚未提交”的数据。
  • 问题：另一个事务随后回滚，读到的数据就是”脏数据”（不存在的数据）。

2.2.2 不可重复读（Non-Repeatable Read）

  • 定义：同一个事务内，两次读取同一行数据，得到了不同的结果。
  • 原因：两次读取之间，另一个事务修改并提交了该行数据。

2.2.3 幻读（Phantom Read）

  • 定义：同一个事务内，两次执行相同的查询，第二次查询返回了第一次没有的”幻影行”。
  • 原因：两次查询之间，另一个事务插入了符合条件的新行并提交。
  • 与不可重复读的区别：
    • 不可重复读：同一行数据，前后两次读到的值不同（UPDATE 导致）
    • 幻读：前后两次查询，行数不同（INSERT 导致）

三种并发现象的对比总结：

现象 读到了什么异常数据 原因
脏读 另一事务未提交的数据 隔离性最弱
不可重复读 同一行数据前后值不同 另一事务 UPDATE 后提交
幻读 同一查询前后行数不同 另一事务 INSERT 后提交

解决这些问题需要设置合适的事务隔离级别（READ UNCOMMITTED / READ COMMITTED / REPEATABLE READ / SERIALIZABLE），不同隔离级别解决不同的并发问题。隔离级别的配置和具体实现属于 MySQL / PostgreSQL 专题内容。

九、SQL 性能意识入门

1. SQL 性能意识

1.1 SQL 的双重标准

在生产环境中，一条合格的 SQL 必须同时满足：

  • 正确性：结果符合业务逻辑。
  • 性能：在预定时间内返回（通常为毫秒级），不导致系统资源枯竭。

1.2 慢 SQL 的线上危害

  1. 用户侧：接口响应超时（P99 飙升），用户流失。
  2. 系统侧：连接池耗尽（阻塞其他请求）、CPU/IO 100%。
  3. 连锁反应：触发数据库熔断、全系统宕机。

2. 常见慢 SQL 典型成因与优化

2.1 资源浪费型：查得太多

错误写法 (Anti-Pattern) 优化方案 (Best Practice) 说明
SELECT * FROM table SELECT id, name FROM table 避免包含 TEXT/BLOB 等大字段，减少网络 IO。
SELECT ... FROM orders SELECT ... LIMIT 20 必须强制分页，防止百万级数据撑爆 JVM 内存。
SELECT COUNT(*) FROM table SELECT COUNT(*) FROM table WHERE status=1 杜绝无条件的统计全表。

2.2 索引失效型：条件不当

核心原则：WHERE 条件中，索引字段本身禁止进行任何运算。

  • 函数运算：
    • 错误：WHERE YEAR(created_at) = 2024
    • 正确：WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
  • 算术运算：
    • 错误：WHERE amount + 100 > 500
    • 正确：WHERE amount > 400
  • 隐式类型转换：
    • 错误：WHERE phone = 138... (若字段为 VARCHAR，数字会导致失效)
    • 正确：WHERE phone = '138...'
  • 模糊查询：
    • 错误：LIKE '%关键词%' (全模糊全表扫描)
    • 正确：LIKE '关键词%' (前缀匹配可走索引)

2.3 逻辑冗余型：结构不合理

  1. 存在性检查：
    • 错误：SELECT COUNT(*) FROM orders WHERE user_id = 1
    • 正确：SELECT 1 FROM orders WHERE user_id = 1 LIMIT 1 (找到即返回)
  2. JOIN 滥用：
    • 原则：单条 SQL 关联表建议不超过 3-4 张。
    • 方案：超长 SQL 应拆分为多次简单查询，在 Java 业务层进行结果组装（DTO mapping）。
  3. 强制排序：若只需数据，不关心顺序，应去掉无意义的 ORDER BY 减少 CPU 排序开销。

3. SQL 优化原则

3.1 预判数据量级

优化必要性取决于生产环境的数据规模：

数据量 全表扫描的感受 是否需要优化
1 万行 几毫秒，感觉不到 暂不需要
100 万行 几百毫秒到几秒 需要关注
1000 万行 几秒到几十秒 必须优化
1 亿行 分钟级 紧急优化

建议：开发阶段用小数据量，不要因为”测试环境很快”就认为没问题。要预估生产环境的数据量，提前评估 SQL 的性能风险。

3.2 优化流程

  1. 监控：通过慢查询日志（Slow Log）发现耗时 SQL。
  2. 诊断：使用 EXPLAIN 分析执行步骤。
  3. 修复：加索引、改写 SQL 或引入 Elasticsearch 等外部索引。
  4. 复核：对比优化前后的执行时间与 Rows 变化。

十、SQL 安全、规范与工程实践

1. SQL 安全防护

1.1 SQL 注入（SQL Injection）

  • 核心本质：恶意数据与 SQL 指令混合，通过破坏原始 SQL 的语义结构来执行非预期操作。
  • 经典案例：
    • 预期：WHERE username = '张三'
    • 注入：用户输入 ' OR '1'='1 → WHERE username = '' OR '1'='1'（逻辑恒真，绕过登录或泄露全表）。

1.2 防御纵深体系

  1. 参数化查询（预编译）：绝对核心防御。(详情关注JDBC、ORM章节)
  2. 输入清洗与校验：在应用层拦截非法字符（如单引号、分号、特殊符号）。
  3. 最小权限原则：数据库账号禁止跨库操作，生产环境禁止 DDL 权限。
  4. 安全屏障：部署 Web 应用防火墙 (WAF) 过滤异常流量。

2. 数据库书写规范

2.1 可读性与维护性准则

  • 字段明确化：严禁 SELECT *。字段列表应垂直对齐，每行一个字段。
  • JOIN 层次化：关联表独立成行，ON 条件与 LEFT JOIN 对齐。
  • 条件分组化：复杂 AND/OR 必须使用括号明确优先级，避免逻辑歧义。
  • 语义化别名：统计函数或复杂字段必须使用 AS 定义具有业务含义的别名。

2.2 复杂 SQL 构建策略（递进式构建）

  1. 原子化：先编写并验证基础 SELECT（含核心 WHERE 过滤）。
  2. 关联化：逐个加入 JOIN 语句，验证关联结果集正确性。
  3. 聚合化：添加 GROUP BY 及聚合函数。
  4. 精细化：最后加入 HAVING、ORDER BY 及 LIMIT 分页。

3. 生产环境变更规范（高危操作指南）

3.1 变更三要素：备份、验证、事务

  • WHERE 铁律：任何 UPDATE / DELETE 必须先通过 SELECT COUNT(*) 验证受影响行数。
  • 备份策略：
    • 逻辑备份：INSERT INTO backup_table SELECT * FROM table WHERE ...。
    • 导出备份：重要数据操作前导出 CSV/SQL 文件。
  • 原子性操作：生产手动变更必须显式开启事务（BEGIN），验证无误后执行 COMMIT，否则立即 ROLLBACK。

3.2 生产变更标准化流程

  1. 离线验证：在测试/预发环境执行，确保 SQL 语法与性能达标。
  2. 影响评估：确认影响行数，评估是否会触发大表锁定。
  3. 低峰执行：选择业务低流量时间段执行。
  4. 闭环验证：执行后对比核心指标，确保业务逻辑无损。
