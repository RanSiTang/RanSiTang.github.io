数据库迁移

发表于 2026-06-08 分类于 数据库与持久化 ， Java数据访问与持久化

一、为什么需要数据库迁移

1. 核心问题：Schema 的演化与管理

1.1 问题的起源

在软件开发的早期，数据库变更通常是最”原始”的操作方式——开发人员直接在数据库服务器上执行 SQL 脚本来修改表结构。这种方式在单人开发或极小团队中勉强可行，但随着团队规模和系统复杂度的增长，它暴露出一系列致命问题。

想象一个典型的场景：

开发人员 A 在本地数据库添加了一个 phone 列，开发人员 B 删除了 legacy_status 列并重命名了 user_type 表。两周后，当代码部署到测试环境时，没人记得究竟执行过哪些 SQL，更没人知道应该如何把生产环境的数据库升级到与代码匹配的状态。

这种混乱就是”数据库变更地狱 (Database Change Hell)”。

1.2 没有版本控制的数据库变更会引发什么？

问题 具体表现
状态漂移 (Drift) 开发、测试、生产三个环境的数据库 Schema 各不相同，Bug 只在特定环境复现
变更不可追溯 “谁在什么时候改了什么？为什么？” — 无法回答这些问题
部署噩梦 每次发布都是 DBA 手工执行一堆 SQL 脚本，顺序、依赖全凭记忆
回滚困难 出问题时无法准确知道应该执行哪些反向 SQL 来恢复
多分支冲突 Git 可以自动合并代码，但数据库变更的合并没有任何自动化支持
新人上手成本 新开发者拿到代码后，不知道如何搭建一个与团队一致的本地数据库

1.3 数据库迁移的核心哲学

数据库迁移 (Database Migration) 的核心思想是：像管理源代码一样，用版本控制的方式管理数据库 Schema 的变更。

这个类比可以帮助理解：

概念 源代码 数据库 Schema
版本控制工具 Git Flyway / Liquibase
变更单元 Commit Migration / Changeset
变更历史 git log flyway_schema_history 表 或 DATABASECHANGELOG 表
当前状态 HEAD 指向的快照 数据库所有已应用迁移的集合
协作方式 拉取 → 合并 → 推送 编写迁移脚本 → 提交到代码库 → 部署时自动执行
回滚 git revert 编写反向迁移 (Undo Migration) 或手动回滚

核心原则：数据库的每一次结构变更（DDL）和数据变更（DML）都应该被记录为一个有序的、不可变的迁移脚本，这些脚本与应用程序源代码一同存储在版本控制系统中。应用启动时，迁移工具会自动检测并执行尚未被应用过的迁移脚本，从而保证数据库 Schema 始终与代码期望的状态一致。

2. 问题域界定：ORM 能替代迁移工具吗？

2.1 Hibernate/JPA 的 ddl-auto 策略

许多开发者第一次接触到 Schema 自动化管理是通过 Hibernate/JPA 的 ddl-auto 配置：

策略 行为
none 什么都不做
validate 仅校验实体与表结构是否匹配，不匹配则启动失败
update 根据实体定义自动修改表结构（仅新增，不会删除列或表）
create 每次启动时删除旧表并重建
create-drop 启动时创建，关闭时删除（常用于集成测试）

2.2 为什么 ddl-auto 不够？

维度 ddl-auto=update 专用迁移工具 (Flyway/Liquibase)
变更粒度的控制 全自动，无法精确控制每一步变更 精确到每一步，开发者定义每次变更
生产环境安全性 极不安全，可能误操作 安全的，变更经过审核和测试
数据迁移 不支持（只改结构，不改数据） 支持，迁移脚本可以同时包含 DDL 和 DML
回滚能力 不支持 支持（Liquibase 原生支持，Flyway 企业版支持）
变更历史追溯 无记录 完整的变更历史表，记录每次迁移的详细信息
多环境一致性 依赖”魔法的”自动推断 确定的，所有环境执行相同的迁移脚本
复杂变更 无法处理（如：列重命名、数据清洗后改类型） 完全可控
团队协作 无法审查变更 SQL 脚本随代码提交，走 Code Review 流程

2.3 最佳实践的组合：ORM + 迁移工具

结论：ORM 负责”表达”对象与表之间的映射关系，迁移工具负责”执行” Schema 的变更。二者不是替代关系，而是互补关系。生产环境中，ddl-auto 应始终设置为 validate 或 none，Schema 的变更完全交由迁移工具管理。

二、Flyway

1. Flyway 概述

1.1 定义与定位

Flyway 是 Boxfuse 公司开发的开源数据库迁移工具，是 Java 生态中最流行的 Schema 版本管理方案之一。它的设计哲学极其简单：

“版本化的 SQL 脚本，按顺序执行。”

Flyway 不做任何复杂的事情——它只是追踪哪些 SQL 脚本已经执行过，然后按版本号顺序执行尚未被应用的脚本。这种极简主义是其最大的优势。

1.2 核心设计原则

  1. 约定优于配置 (Convention over Configuration)：遵循文件命名规范即可，几乎不需要额外配置。
  2. SQL 优先 (SQL-First)：迁移脚本就是纯 SQL 文件，无学习成本，DBA 可以直接审查和修改。
  3. 不可变性 (Immutability)：一旦某个版本的迁移被应用到数据库，它就永远不能被修改。
  4. 幂等性追求：虽然 Flyway 不强制幂等，但通过追踪表保证同一个迁移只被执行一次。

1.3 Flyway 的版本管理模式

Flyway 在目标数据库中维护一张元数据表（默认表名为 flyway_schema_history），用于记录所有已应用的迁移：

这张表是 Flyway 的”大脑”——每次应用启动时，Flyway 会扫描 classpath 中的迁移文件，与这张表中已记录的迁移进行比对，只执行那些未曾被应用过的迁移。

2.4 版本化迁移 (V) vs 可重复迁移 (R)

特性 版本化迁移 (V) 可重复迁移 (R)

版本号 必须有，且唯一 无版本号

2.5 命名避坑指南

  1. 版本号一旦发布就不可修改：V1.0.0__xxx.sql 已提交到 Git 并被应用到了数据库后，就不能再改它的内容或版本号。
  2. 分隔符是双下划线 __：单下划线会破坏版本号与描述的解析。
  3. 描述部分使用下划线分隔单词：V1__Create_user_table.sql 而不是 V1__Create user table.sql（空格在命令行中可能导致问题）。
  4. 不要在一个版本号上”追加”变更：如果要修改 V1 中创建的表，应该写一个新的 V2 迁移，而不是修改 V1 本身。

3.2 状态详解

状态 含义 数据库表现
Pending 迁移文件存在，但尚未被应用到数据库 flyway_schema_history 表中没有该版本记录

Superseded 可重复迁移的最新执行记录与新文件比对 仅用于信息展示
Ignored 被配置为忽略 不执行也不报错

3.3 关键状态问题的处理

Failed 状态：这是最需要处理的状态。当迁移执行失败时（例如 SQL 语法错误），Flyway 会在数据库中记录一条 success = false 的记录。在旧版 Flyway 中，修复后续的迁移脚本后，你需要手动删除这条失败记录才能重新运行。从 Flyway 9.x 开始，可以使用 flyway repair 命令自动清理失败的迁移记录。

Missing 状态：这意味着有人删除了一个已被应用的迁移文件。Flyway 会拒绝启动。你需要恢复该文件，或者使用 flyway repair 将其从 history 表中移除（谨慎！这样做可能会丢失关键的 DDL 变更历史）。

4. 核心命令详解

4.1 migrate (最核心的命令)

作用：扫描并执行所有处于 Pending 状态的迁移。

执行流程：

  1. 扫描 classpath 中 db/migration 目录下的所有迁移文件
  2. 查询 flyway_schema_history 表获取已应用的迁移列表
  3. 计算增量：Pending = 文件列表 - 已应用列表
  4. 按版本号升序执行所有 Pending 的版本化迁移
  5. 检查可重复迁移是否有变更，如有则重新执行
  6. 将成功/失败记录写入 flyway_schema_history 表

在 Spring Boot 中：只要引入了 flyway-core 依赖，migrate 会在应用启动时自动执行，无需任何手动调用。

作用：在已有数据库上首次引入 Flyway 时，为当前数据库状态设置一个”起点”。

使用场景：你的数据库已经运行了几年，有一百多张表，现在想引入 Flyway 来管理后续的变更。你不能从 V1__Create_all_tables.sql 开始，因为那些表已经存在了。此时你使用 baseline 命令，设置基线版本为 1.0。之后 Flyway 只会执行版本号大于 1.0 的迁移。

4.6 repair

作用：修复 Flyway 的 history 表。

具体会做什么：

  • 删除所有 success = false 的记录（即失败迁移的记录）
  • 重新计算已应用迁移的 checksum（适用场景：迁移文件意外被微小修改但不影响实际 DDL 结果）
  • 删除版本号在当前迁移文件中找不到的 history 记录

使用场景：迁移执行失败后，你修复了 SQL 错误，在执行 repair 之后重新 migrate。

4.7 undo (企业版特性)

作用：回滚最近执行的版本化迁移。

需要编写对应的 U 前缀迁移文件（如 U1.0.1__Drop_email_column.sql）。Flyway 会按版本号降序执行撤销迁移。

三、Liquibase

1. Liquibase 概述

1.1 定义与定位

Liquibase 是另一款重量级的开源数据库迁移工具，由 Datical 公司维护。与 Flyway 的极简 SQL 哲学不同，Liquibase 提供了一种基于抽象描述（Changelog）的、多格式支持的数据库变更管理方案。

如果用一个比喻：

  • Flyway 像是 Git，简单直接：一个版本对应一个文件，按顺序执行。
  • Liquibase 更像是声明式的 Kubernetes YAML：你描述”数据库应该长什么样”的变更步骤，Liquibase 负责翻译和执行。

1.2 核心设计理念

  1. 抽象 DSL：不直接写 SQL，而是通过 XML / YAML / JSON 描述变更意图，Liquibase 自动翻译为目标数据库的 SQL 方言。
  2. 跨数据库兼容：同一套 ChangeLog 可以（理论上）在 MySQL、PostgreSQL、Oracle 等多种数据库上执行。
  3. 回滚内建：每个 Changeset 都可以声明对应的 rollback 语句，Liquibase 原生支持回滚操作。
  4. 上下文 (Context) 与标签 (Labels)：精细控制哪些迁移应在哪些环境执行。

2. 核心概念

2.1 ChangeLog（变更日志）

ChangeLog 是 Liquibase 的根配置文件，可以理解为迁移的”目录”。它引用一个或多个 Changeset 文件。

2.2 Changeset（变更集）

Changeset 是 Liquibase 的最小迁移单元。每个 Changeset 由 id + author + 文件路径 三者联合唯一标识。

2.3 Changeset 的标识与不可变性

每个 Changeset 的唯一标识由以下三者组合：

  1. id — Changeset 的 ID
  2. author — 作者
  3. filePath — Changeset 所在的文件路径（自动确定）

三元组确定后，一旦 Changeset 被执行，它的 id、author 和文件路径都不应该再改变，内容也不能被修改。Liquibase 通过 checksum（MD5）来检测 Changeset 是否被篡改。

如果你需要修改一个已应用的迁移，正确做法是：

  • 编写一个新的 Changeset 来补救
  • 或者使用 clearCheckSums 命令重置校验和（谨慎！）

2.4 DATABASECHANGELOG 表

与 Flyway 的 flyway_schema_history 表类似，Liquibase 使用 DATABASECHANGELOG 表追踪迁移历史：

列名 说明
ID Changeset 的 id
AUTHOR Changeset 的 author
FILENAME Changeset 所在文件的路径
DATEEXECUTED 执行时间
ORDEREXECUTED 执行顺序
EXECTYPE 执行类型：EXECUTED、MARK_RAN、RERAN 等
MD5SUM Changeset 内容的校验和
DESCRIPTION 变更摘要

TAG 标签（用于回滚目标定位）
DEPLOYMENT_ID 部署 ID

3. 四大格式对比

Liquibase 支持四种格式来编写 Changeset定义：

3.1 XML 格式（最成熟、最常用）

优点：IDE 支持最佳（有 Schema 约束），功能最完整，文档最丰富。

缺点：啰嗦，写起来比较繁琐。

3.2 YAML 格式（简洁，可读性好）

优点：比 XML 简洁很多，可读性好。

缺点：IDE 支持不如 XML。

3.3 JSON 格式

优点：适合 JSON 工具链（如前端项目、Node.js 项目）。

缺点：冗长，很少在 Java 生态中使用。

3.4 格式化 SQL（最接近 Flyway 风格）

优点：SQL 开发者最熟悉，简单场景最直接。

缺点：SQL 本质上一旦复杂起来就不跨数据库，失去了 Liquibase 的一大优势；且 -- 注释形式上不如 XML 结构化。

3.5 格式选择指南

场景 推荐格式
企业级 Java 项目多数据库支持 XML
中小型项目追求简洁 YAML
DBA 主导的团队 格式化 SQL
纯 MySQL/PostgreSQL 单数据库 格式无所谓，甚至直接 Flyway

4. Preconditions（前置条件）

4.1 作用

Preconditions 是 Liquibase 提供的安全检查机制。在 Changeset 执行之前，Liquibase 会先评估 Preconditions 是否满足，如果不满足，根据 onFail / onError 配置采取相应行动。

4.2 常见前置条件

4.3 onFail 策略

值 行为
HALT 停止执行，抛出异常（默认）
CONTINUE 跳过当前 Changeset，继续执行后续，但不记录到 DATABASECHANGELOG（下次还会尝试）
MARK_RAN 跳过当前 Changeset，并将其标记为”已执行”（不会再尝试）
WARN 输出警告日志，继续执行（谨慎使用！）

4.4 Flyway 中有对应机制吗？

Flyway 没有内置的 Preconditions 机制。在 Flyway 中实现类似功能，你需要：

  • 在 SQL 迁移脚本中使用 CREATE TABLE IF NOT EXISTS、条件语句（在支持的数据库中）等 SQL 原生手段
  • 或者使用 Java-based Migration 来执行预检查逻辑

这是 Liquibase 相比 Flyway 的一个重要差异化优势——它让你可以在迁移工具层面声明和执行安全检查。

3. 实际场景选型建议

3.1 选 Flyway 的情况

  • 团队高度以 SQL 为中心，DBA 需要直接审查和编写迁移
  • 追求极简，不想引入复杂的概念和配置
  • 大多数时候不需要回滚迁移（或者通过”正向迁移修复错误”的方式来补救）
  • 使用 Spring Boot 生态，享受开箱即用的集成体验
  • 数据库类型单一（只使用 MySQL 或 PostgreSQL），不需要跨数据库兼容

3.2 选 Liquibase 的情况

  • 需要跨多种数据库类型部署（一套 ChangeLog 适配 MySQL + Oracle + PostgreSQL）
  • 需要社区版就能用的回滚功能
  • 需要 Preconditions 来做执行前的安全检查
  • 需要在不同环境执行不同的迁移（Context / Labels 精细控制）
  • 已经有一定规模的团队，愿意承受学习成本换取灵活性
  • 需要 Diff 功能来比较两个数据库的差异

3.3 一句话总结

Flyway 用 SQL 的简单换取了很多高级功能（需要企业版）；Liquibase 用学习的复杂度换取了开源版的功能完整性。

4. 市场趋势

根据近年的开发者调查和开源项目趋势：

  • Flyway 在 Spring Boot 生态中占据主导地位（Spring Boot 默认集成方案）
  • Liquibase 在大型企业和需要多数据库兼容的场景中仍有稳固的市场份额
  • 如果从零开始选型且没有特殊需求，Flyway 是更安全的选择

1.1 回滚的三种哲学

哲学 A：”正向修复”（Flyway 社区版的默认立场）

不做回滚。如果迁移有问题，编写一个新的迁移来修复：

优点：最简单，不需要额外的回滚工具。所有迁移都是”正向的”，变更历史线性清晰。

缺点：如果 V1.0.0 还未广泛部署，在其他环境可能增加不必要的”修正”步骤。

哲学 B：”声明式回滚”（Liquibase 的方式）

每个迁移先编写正向变更，同时声明回滚逻辑。出问题时一键回滚。

优点：操作简便，自动化程度高。

缺点：回滚脚本本身也需要测试，增加了开发负担。有些变更在物理上无法安全回滚（如删除了某列中的数据）。

哲学 C：”混合策略”（最常见的企业实践）

大多数错误通过正向修复解决。只有少数可预测的高风险变更（如大表重建、核心业务逻辑相关的数据变更）才准备回滚方案。

一、AI 在后端开发中的角色定位

以 ChatGPT 在 2022 年底发布为起点，大语言模型（Large Language Model, LLM）在过去几年间以前所未有的速度渗透到软件工程的各个层面。LLM 是一种基于海量文本数据训练得到的深度学习模型，能够理解自然语言并生成类人文本。对于后端工程师而言，最重要的问题不是”AI 会不会取代我”，而是”我应该把 AI 放在系统的什么位置”。

1. AI 不是替代者，而是增强者

AI 并不替代后端。后端系统本质上解决的是数据的存储、流转、计算和权限控制问题，这些职责不会因为 AI 的出现而消失。真正发生变化的是后端能力的边界被大幅拓宽了——很多以前后端做不到或者做不好的事情，现在可以做到了。

举一个具体的例子：一个传统的工单系统，后端能做的是 CRUD（创建、查询、更新、删除工单），以及基于规则的状态流转。接入 AI 之后，后端可以额外提供的能力包括但不限于：

  • 智能分类：根据工单内容自动判断所属类别和优先级，背后是 LLM 的文本分类能力
  • 相似工单检索：从历史工单中找到与当前问题最相似的处理记录，背后是 Embedding 向量化与语义搜索
  • 自动摘要：将冗长的工单描述压缩为关键信息摘要，方便客服快速定位问题
  • 知识库问答：基于产品文档自动回答常见问题，背后是 RAG（Retrieval-Augmented Generation，检索增强生成）

这些能力不是替代后端逻辑，而是嵌入到后端业务流程中，成为新的业务能力层。

2. 从”业务实现者”到”AI 能力编排者”

传统 Java 后端工程师的核心工作是：理解需求 → 设计数据模型 → 编写 Service/DAO 层代码 → 暴露 REST API。这是一条确定的、可预测的工作链路。

AI 时代，后端工程师的角色发生了微妙但重要的迁移：你不再只是自己实现业务逻辑，而是要学会编排 AI 来完成逻辑。这意味着你需要同时理解：

  • 业务本身（和以前一样重要）
  • AI 能做什么、不能做什么（能力边界判断）
  • 如何把业务问题分解为”AI 适合做的部分”和”传统代码适合做的部分”（任务分解与编排）
  • 如何在不可靠的 AI 输出之上构建可靠的系统（工程化保障）

这种角色的变化，可以类比为从”手工艺人”到”工厂产线设计师”的转变——你不是不干活了，而是把精力从亲自完成每一道工序，转向设计工序之间的流转规则与质量检验标准。

3. 后端工程师的独特优势

在 AI 应用开发领域，后端工程师拥有一些其他人不具备的优势：

  • 系统设计能力：你知道如何设计高可用、可扩展的系统，而 AI 应用的工程化落地最终依赖的正是这些能力
  • 数据理解：你清楚数据从哪里来、流经哪些系统、最终去哪里，而 AI 应用的核心就是对数据的加工和增强
  • 工程化经验：你熟悉 CI/CD、监控告警、灰度发布、容错降级——这些是 AI 应用从 Demo 走向生产的关键
  • 业务边界意识：你知道哪些操作需要权限校验、哪些数据不能外泄、哪些行为需要审计——这是 AI 安全的基础

换句话说，训练模型可能是算法工程师的强项，但把模型变成可靠的生产服务，是后端工程师的主场。

二、后端系统中 AI 的三种集成深度

AI 在后端系统中的集成不是”全有或全无”的二元选择，而是一个渐进深入的连续谱。根据模型在决策链路中的自主程度，可以划分为三个层级。理解这三个层级的关系，有助于你在实际项目中做出合适的架构决策。

1. Level 1：API 调用层

这是最低门槛的集成方式。后端像调用任何第三方 API 一样调用 LLM 的接口，把模型的返回结果当作一个”高级函数”的输出来使用。

典型架构：客户端 → 后端 Service → LLM API → 后端处理结果 → 返回客户端

适用场景：文本分类、情感分析、内容摘要、翻译、代码注释生成等输入确定、输出可直接消费的单次调用任务。

关键特征：模型在这里只是一个被动的计算单元，它不做任何决策，只是根据输入生成输出。后端代码完全掌控调用时机、参数和结果处理。

这种模式的技术门槛最低，但它存在两个明显局限：一是模型无法访问企业内部数据（它只能基于训练数据回答），二是模型无法执行任何实际操作（它只能说，不能做）。

2. Level 2：能力编排层

在这个层级，模型不再孤立工作，而是与外部知识库和业务系统连接起来。这主要通过两种机制实现：

  • RAG（检索增强生成，Retrieval-Augmented Generation）：在调用模型之前，先从向量数据库或搜索引擎中检索出与用户问题相关的文档片段，将这些片段作为上下文注入 Prompt，让模型基于”外部知识”而非仅凭”训练记忆”来生成回答
  • Function Calling（函数调用，也称 Tool Calling）：让模型输出结构化的函数调用请求，后端据此执行真实的业务操作（查数据库、调接口、发消息等），将执行结果反馈给模型继续对话

典型架构：客户端 → 后端 Service → 知识检索（向量数据库/搜索引擎） → 构造增强 Prompt → LLM API → 结果处理（可能触发 Function Call → 执行业务逻辑 → 结果回传 → 模型继续推理）→ 返回客户端

适用场景：智能客服、企业知识库问答、业务查询助手（”帮我查一下这个订单的状态”）、文档审查等需要外部知识 + 业务操作的场景。

关键特征：模型开始有了”眼睛”（能看外部知识）和”手”（能调用业务接口），但它仍然按照预定义的流程执行——先检索还是先调工具、检索哪些数据源、工具调用的顺序，这些都由后端代码预先编排好。模型在这个层级是”被赋能”的，但不是”自主”的。

Level 2 是目前企业级 AI 应用的主流深度，它兼顾了能力增强和可控制性，覆盖了约 80% 的实际业务场景。

3. Level 3：Agent 自主决策层

Agent（智能体）是当前 AI 应用开发的最高集成深度。在这个层级，模型不再按照预先编排的固定流程执行，而是自主规划任务步骤、自主选择调用哪些工具、自主判断任务是否完成。

典型架构：用户给定目标 → Agent 自主分解为子任务 → 自主选择工具执行每个子任务 → 观察执行结果 → 根据结果调整计划 → 循环直到目标达成 → 返回最终结果

关键特征：模型从”执行者”变为”决策者”。后端工程师的角色从编排每一步做什么，变为定义模型可用的工具集（Tools）、设定行为约束（System Prompt）、设置安全边界（Guardrails），然后让模型在这些约束内自主行动。

Agent 的典型循环模式是 ReAct（Reasoning + Acting，推理与行动交替）：模型先思考下一步应该做什么（Reasoning），然后调用工具执行（Acting），观察工具返回结果，再基于结果思考下一步……如此往复直至任务完成。

适用场景：自动化运维（”检查所有服务健康状态，对有问题的服务查看日志并尝试重启”）、复杂数据分析（”分析这个季度各区域的销售情况，找出异常点并给出建议”）、多步骤业务流程自动化。

Agent 的自主性既是它的优势也是风险来源：模型可能做出错误的工具选择、陷入循环、或被恶意 Prompt 诱导执行危险操作。因此 Level 3 对安全设计、可观测性和人工干预机制的要求远高于前两个层级。

4. 三种深度的关系与选型原则

三种深度不是互相替代的关系，而是层层叠加的关系。Level 2 包含 Level 1 的所有能力，Level 3 又包含 Level 2 的所有能力。

选型原则很简单：不是越深越好，够用就好。如果 Level 1 能解决问题就不要上 Level 2，Level 2 能覆盖就不要上 Agent。每上升一个层级，系统的复杂度、不可控性和调试难度都会显著增加。实践中，大多数有价值的 AI 应用都落在 Level 2。

三、AI 给后端架构带来的五个核心变化

AI 的集成不是简单地”加一个 API 调用”0，而是从底层改变了后端系统的若干基本假设。理解这些变化，是做好 AI 后端工程化的前提。

1. 从确定性到概率性

传统后端的核心逻辑是确定性的：给定相同的输入和相同的数据库状态，输出一定是相同的。单元测试、集成测试、回归测试都建立在这个前提之上。

AI 改变了这个假设。LLM 的输出本质上是概率性的——同一个 Prompt 调用两次，模型可能给出措辞不同甚至结论不同的回答。Temperature（温度参数，控制输出的随机性，值越低输出越确定、越高越有创造性）可以调节随机程度，但不能消除它。

这对后端工程意味着：

  • 测试策略需要改变：无法再用”期望输出等于某字符串”的方式写测试，需要引入语义相似度评判、结构化输出 Schema 校验、LLM-as-Judge（用模型评判模型输出）等新的测试方法
  • 容错设计需要加强：必须假设模型可能输出不合法 JSON、遗漏关键字段、或给出不合逻辑的结论，在代码层做校验和兜底
  • 监控需要新维度：除了传统 QPS 和延迟，还需要监控幻觉率、输出格式合规率、内容安全违规率等

2. 从纯代码到 Prompt + Code

传统后端业务逻辑全部用代码（Java/Go/Python 等）编写。AI 引入后，业务逻辑扩展为三层结构：

  • Prompt 层：用自然语言描述任务、设定规则、提供示例，本质是在”编程”模型的行为
  • 工具函数层：传统 Java 代码，实现模型无法完成的确定性操作（数据库查询、权限校验、数值计算等）
  • 编排代码层：Java 代码负责串联 Prompt 和工具函数，控制调用顺序、处理异常、管理状态

Prompt 从”一段随手写的文本”升级为需要工程化管理的一等公民：版本管理、A/B 测试、效果评测、回滚机制——这些传统代码享有的工程实践，Prompt 同样需要。

3. 从 REST 到多协议通信

传统后端系统的对外通信几乎全是 REST（Representational State Transfer，表现层状态转移）风格的 HTTP 请求/响应。AI 集成引入了两种重要的新通信模式：

  • SSE（Server-Sent Events，服务端推送事件）：用于 LLM 的流式输出。模型边生成边推送 Token，用户可以看到逐字输出的效果，大幅改善体验。Spring Boot 中通过 SseEmitter 或 WebFlux 的 Flux<ServerSentEvent> 实现
  • MCP（Model Context Protocol，模型上下文协议）：Anthropic 提出的标准化协议，用于模型与外部工具/数据源之间的通信。MCP 定义了统一的 Server-Client 架构，使得不同的 AI 客户端可以复用同一套工具实现

理解 REST 的工程师需要适应 SSE 的长连接语义和背压处理（Backpressure，当下游消费速度慢于上游生产速度时进行的流量控制），以及 MCP 的松耦合工具注册模型。

4. 从固定成本到 Token 经济

传统后端系统的成本主要由服务器、带宽和数据库组成，这些成本随请求量线性增长，且边际成本相对固定。

AI 应用引入了一个全新的成本维度：Token 消耗。Token 是 LLM 处理文本的基本单位（英文中大致相当于 3/4 个单词，中文中大致相当于 1-2 个汉字）。每次 LLM 调用都按 Token 数量计费，输入和输出分别计价，且不同模型的价格差异可达数十倍。

这对架构的影响是深远的：

  • 缓存的价值被重估：传统后端中缓存主要用于提升性能，在 AI 应用中，缓存还直接节省 Token 成本——两个用户问相似问题，如果命中语义缓存（Semantic Cache），就完全不需要调用模型
  • 模型分层成为刚需：简单任务用便宜的小模型（如 DeepSeek-V3、Qwen-Turbo），复杂任务用贵的大模型（如 GPT-4、Claude Opus），是成本优化的基本策略
  • Prompt 设计要精打细算：Prompt 中的每个 Token 都花在刀刃上。过多的 System Prompt 和 Few-shot 示例虽然可能改善效果，但也显著抬高了每次调用的成本

5. 从权限控制到意图安全

传统后端安全主要围绕认证（Authentication，验证你是谁）和授权（Authorization，验证你能做什么）展开，即 RBAC（Role-Based Access Control，基于角色的访问控制）。

AI 引入后，攻击面扩大了：

  • Prompt Injection（提示注入）：攻击者通过在用户输入中嵌入恶意指令，试图覆盖 System Prompt 中的行为约束。例如在客服系统中输入”Ignore all previous instructions and…”来劫持模型行为
  • 工具越权调用：在 Agent 模式下，模型自主选择调用哪些工具。如果工具的权限边界设计不当，模型可能被诱导调用它不应该调用的接口
  • 数据泄露：模型在生成回答时可能不经意地复述训练数据中的敏感信息，或 RAG 检索出了用户无权访问的文档内容

这意味着后端安全体系需要增加一个新的维度：意图安全（Intent Security）——不只是控制”谁”能访问”什么”，还要防御”恶意输入意图”对模型行为的操控。

四、后端 AI 应用的技术要素全景

AI 后端开发涉及的技术要素可以归纳为七个层次。理解每一层的职责和它们之间的依赖关系，有助于形成系统化的知识框架。

1. 模型层

模型层是 AI 应用的基础设施，负责提供语言理解和生成能力。后端工程师需要掌握的核心概念包括：

  • LLM（大语言模型）：在海量文本数据上训练得到的深度学习模型。主流模型包括 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列、以及国产的 DeepSeek（深度求索的高性价比模型）、Qwen（阿里通义千问）、GLM（智谱清言）等
  • Token（词元）：模型处理文本的最小单位。一个中文字通常对应 1-2 个 Token，一个英文单词通常对应 1-3 个 Token。Token 数量决定了模型调用的成本和延迟
  • Context Window（上下文窗口）：模型单次能处理的最大 Token 总数（输入 + 输出之和）。主流模型的上下文窗口从 128K 到 2M Token 不等，决定了能”塞进去”多少信息
  • Temperature（温度）：控制输出随机性的参数，范围 0-2，值越低输出越确定和保守，越高输出越有创造性但也更容易偏离主题

2. Prompt 层

Prompt（提示词）是人与模型之间的通信协议。它的核心任务是将业务需求翻译为模型能理解并正确执行的自然语言指令。

一个结构良好的 Prompt 通常包含以下要素：

  • 角色设定（Role/Persona）：定义模型扮演的角色，如”你是一个资深的 Java 后端工程师”
  • 任务描述（Task）：明确需要模型完成的具体任务
  • 上下文信息（Context）：任务相关的背景知识和数据
  • 约束条件（Constraints）：输出格式要求、内容限制、行为边界
  • 示例（Few-shot Examples）：通过一至多个输入-输出示例，让模型理解期望的行为模式

Prompt 的核心方法论包括 Zero-shot（不提供示例，直接给指令）、Few-shot（提供少量示例）、Chain of Thought（思维链，要求模型展示逐步推理过程）和 ReAct（推理与行动交替）。Prompt 工程的目标不是写出最复杂的 Prompt，而是找到用最小的 Token 开销达到稳定效果的 Prompt。

3. 知识层

LLM 的知识截止于其训练数据的时间点，且无法访问企业内部的私有数据。知识层通过引入外部知识来解决这两个问题：

  • Embedding（向量嵌入）：将文本映射为高维空间中的稠密向量。语义相近的文本，其向量在空间中的距离也相近。Embedding 是实现语义搜索的基础
  • 向量数据库（Vector Database）：专门存储和检索向量的数据库。核心能力是基于向量相似度（如余弦相似度、欧氏距离）快速找到 Top-K 个最相似的向量。常见方案包括 Milvus、Qdrant、Weaviate、Pinecone，以及传统数据库的向量扩展如 pgvector
  • RAG（检索增强生成）：在调用 LLM 之前，先检索出相关的外部知识片段，注入到 Prompt 的上下文中，让模型基于这些知识生成回答。RAG 的核心流程是：文档加载 → 文本切分（Chunking）→ 向量化 → 存储 → 检索 → 增强生成

4. 行动层

模型能”说”但不能”做”。行动层通过 Function Calling 机制让模型能够触发真实的业务操作：

  • Function Calling（函数调用）：在调用 LLM 时，不仅传 Prompt，还传一组可用函数的定义（名称、描述、参数 Schema）。模型在生成回答时，如果判断需要调用某个函数来获取信息或执行操作，它不直接执行函数，而是返回一个结构化的函数调用请求（包含函数名和参数）。后端代码解析这个请求、执行实际的业务函数、将结果回传给模型，模型再基于结果继续推理
  • Tool Calling（工具调用）：Function Calling 的广义延伸，不限于后端函数，也包括调用外部 API、操作数据库、读写文件等

Function Calling 是连接”AI 推理”和”业务执行”的桥梁，也是 Level 2 和 Level 3 集成的关键技术。

5. 编排层

当单个 LLM 调用不够用时（多步推理、多数据源、多工具协作），编排层介入来管理执行流程：

  • Workflow（工作流）：预先定义好的、确定性的执行流程。每一步做什么（调 LLM、查数据库、调工具、等待人工审批）由代码决定。Dify、Coze、LangGraph 等平台提供了可视化的流程编排能力
  • Agent（智能体）：模型自主决定执行流程。Agent 根据目标自行分解任务、选择工具、判断完成条件

两者的本质区别在于决策权在谁手上：Workflow 的决策权在工程师（确定性），Agent 的决策权在模型（自主性）。

6. 框架层

框架层提供统一的抽象来减少重复代码。Java 生态中两个主要的 AI 集成框架：

  • Spring AI：Spring 生态的官方 AI 扩展。提供 ChatClient（统一模型调用）、EmbeddingClient（统一向量化）、VectorStore（统一向量数据库访问）、@Tool 注解（声明工具函数）等开箱即用的能力，与 Spring Boot 深度整合
  • LangChain4j：LangChain 的 Java 实现，设计上更接近 LangChain/Python 的编程模型，提供 ChatLanguageModel、EmbeddingModel、AiServices、@Tool 等 API

7. 工程层

工程层解决 AI 应用从”能跑”到”能上线”的问题，包括安全防护、缓存策略、成本追踪、可观测性（Observability，通过 Log/Trace/Metrics 了解系统内部运行状态的能力）和评测体系。这一层是目前大多数 AI 应用从 Demo 阶段迈向生产环境的主要瓶颈。

五、典型的 AI 后端架构模式

将上述技术要素组合起来，实践中形成了几种可复用的架构模式。

1. 简单代理模式（Proxy Pattern）

这是最简单的集成模式。后端作为 LLM API 的代理层，封装 API Key、格式化请求、处理响应。后端在这里的核心价值是安全和控制：不暴露 API Key 给前端、记录调用日志、实施访问控制、对输入输出做内容审核。

适用场景：文本润色、翻译、摘要、代码注释生成等无需外部知识的单次任务。

2. RAG 知识库模式（RAG Pattern）

核心流程是一个完整的 ETL（Extract-Transform-Load，数据抽取-转换-加载）Pipeline：

  1. 离线阶段（索引构建）：文档上传 → 解析（PDF/Word/Markdown → 纯文本）→ 文本切分（按语义边界切为适当大小的 Chunk）→ Embedding 向量化 → 存入向量数据库
  2. 在线阶段（检索生成）：用户提问 → 问题向量化 → 向量数据库检索 Top-K 相关 Chunk → 将检索结果与用户问题一起构造 Prompt → LLM 生成回答

RAG 模式的关键设计决策包括：Chunk Size（文本块大小，决定检索粒度）、Overlap（相邻块之间的重叠量，防止语义断裂）、检索策略（纯向量 vs 混合检索 vs 多路召回重排序）。

适用场景：企业知识库问答、产品文档助手、合同/法规检索回答、智能客服。

3. 工具增强模式（Tool-Augmented Pattern）

与 RAG 模式提供”知识”不同，工具增强模式提供的是”行动能力”。模型不只是能回答问题，还能帮用户查订单、看库存、创建工单、发送通知。

关键设计要点：

  • 工具定义：每个工具需要清晰的名称、自然语言描述、参数 JSON Schema。描述的清晰度直接影响模型选择工具的准确率
  • 幂等与安全：查询类工具应幂等（多次调用结果一致）；写入类工具需要额外的权限校验和安全确认
  • 超时与降级：工具调用可能超时或失败，需要合理的超时设置和降级策略

适用场景：业务数据查询助手、智能运维助手、企业内部效率工具。

4. Agent 编排模式（Agent Orchestration Pattern）

Agent 模式将流程控制权从代码转移给模型。Agent 的典型执行循环如下：

  1. 思考（Thought）：分析当前状态，决定下一步应该做什么
  2. 行动（Action）：选择并调用合适的工具
  3. 观察（Observation）：获取工具执行结果
  4. 反思（Reflection）：评估是否达成目标，决定继续或结束

关键设计要点：

  • 工具集设计：Agent 可用的工具需要职责清晰、边界明确、描述准确。工具数量不宜过多（通常建议 5-15 个），过多会降低模型选择的准确率
  • 终止条件：Agent 可能陷入无效循环，需要设置最大步数、超时时间和成本上限
  • 人工介入点：对于高风险操作（如删除数据、发送通知），设置人工确认环节（Human-in-the-Loop）

适用场景：复杂数据分析、自动化运维、多步骤业务流程。

5. 混合模式（Hybrid Pattern）

实际的企业级 AI 应用中，很少只使用单一模式。混合模式将上述模式组合起来：

  • 简单问题走 Proxy 模式直接回答
  • 知识类问题走 RAG 模式检索文档
  • 操作类问题走 Tool-Augmented 模式调用后端服务
  • 复杂的多步骤任务走 Agent 模式自主编排

混合模式需要一个**路由层（Router）**来判断请求应该走哪条路径——这本身可以是一个轻量级 LLM 调用，也可以是基于规则的分类器。混合模式是构建企业 AI 中台（AI Middle Platform）的典型架构。

六、Java 在 AI 生态中的定位

1. 正视 Java 在 AI 领域的真实位置

AI 模型的训练、微调和研究的首选语言是 Python，这一点短期内不会改变。PyTorch、Transformers、vLLM 等核心库都是 Python 生态的产物。

但 AI 应用不等于 AI 模型训练。AI 工程化落地与企业级服务化的主力语言，Java 仍然占据重要位置。绝大多数企业的后端技术栈是 Java/Spring Boot，它们需要的不是用 Java 重新训练模型，而是把模型能力可靠地集成到现有系统中。

因此 Java 在 AI 生态的定位是清晰的：Python 负责造发动机（模型训练与推理），Java 负责造整车（将发动机封装为可驾驶的汽车，配上方向盘、刹车、仪表盘）。

2. Java 生态的核心优势

在 AI 应用开发场景下，Java 生态有几项关键优势：

  • Spring Boot 的成熟度：依赖注入、自动配置、Starter 机制、Actuator 监控等能力，使得 AI 集成框架（如 Spring AI）可以以与 Spring Data、Spring Security 一致的方式融入现有体系
  • 微服务治理：服务发现、负载均衡、熔断降级、分布式追踪等能力，对于需要调用多个模型和外部服务的复杂 AI 应用至关重要
  • 数据基础设施集成：Java 在关系型数据库（MySQL/PostgreSQL → MyBatis/JPA）、缓存（Redis → Spring Cache）、消息队列（Kafka/RabbitMQ → Spring Messaging）等方面的集成生态极为成熟——而 AI 应用恰好需要频繁地与这些基础设施交互
  • 运维与监控：Micrometer + Prometheus + Grafana 的技术栈可以平滑延伸覆盖 AI 应用的 Token 消耗、模型延迟等新指标

3. Java 调用 AI 的典型模式

Java 后端调用 AI 能力的典型模式不是”用 Java 跑模型”，而是 Java 作为 HTTP 客户端 + 编排层：

  • 通过 HTTP 调用 OpenAI 兼容 API（几乎所有国产模型都兼容 OpenAI 的 API 格式）
  • 通过 SSE 流式消费模型的输出
  • 通过 MCP 协议注册和管理工具
  • 与 Python 服务协作：Python 运行模型推理或 GraphRAG 这类特定服务，Java 通过 HTTP/gRPC 调用

Spring AI 和 LangChain4j 的目标正是为上述模式提供统一的、符合 Java 习惯的抽象层，让 Java 工程师不需要关心底层 HTTP 细节，用熟悉的 @Service、@Tool、ChatClient 等 API 即可集成 AI 能力。

4. 务实的技术选型态度

对于 Java 后端团队，务实的态度是：

  • 模型层面：不造轮子，直接使用成熟的模型 API 服务和托管平台
  • 框架层面：优先评估 Spring AI 或 LangChain4j，避免裸写 HTTP + JSON 解析
  • 特殊需求：需要 GraphRAG、需要模型微调等场景，部署 Python Sidecar 服务，不要用 Java 硬扛
  • 核心竞争力：把精力放在业务理解、架构设计、工程化保障上——这些才是后端工程师不可替代的价值

七、后端工程师需要建立的 AI 思维模型

人适应技术，而不是技术适应人。以下五个思维转变，是 Java 后端工程师在 AI 时代保持竞争力的关键。

1. Prompt 即代码

传统认知中，Prompt 是”写给 AI 看的一段话”，是临时的、一次性的。但在工程实践中，Prompt 应当被视为一种新的编程形式——用自然语言描述逻辑，由 LLM 执行。

这意味着 Prompt 需要享有与代码同等的工程待遇：

  • 版本管理：Prompt 修改应该走 Git，有 commit message 记录变更原因
  • A/B 测试：新旧 Prompt 并行运行，对比效果数据后决定是否切换
  • 效果评测：每次 Prompt 变更应该有对应的评测 Pipeline 自动跑分
  • 回滚机制：效果变差时应能快速回退到上一个 Prompt 版本
  • 代码审查：Prompt 变更也应该经过 Code Review，检查逻辑完整性、边界情况覆盖、Token 开销

2. 模型是第三方依赖

把模型看作一种特殊的第三方依赖，会帮助你建立正确的工程习惯：

  • 模型有版本号（如 gpt-4-0613、claude-sonnet-4-6），版本之间行为可能不同
  • 模型会 deprecate（废弃） 和 sunset（下线），需要关注模型提供商的版本生命周期通知
  • 模型有性能差异，两个同样号称”支持 128K 上下文”的模型，在长文本检索精度上可能差距显著
  • 模型需要 Fallback（故障转移）策略：主模型不可用时自动切换到备用模型，或降级为缓存响应

用管理 Maven 依赖的方式来管理模型：锁定版本、测试升级、监控变化、准备回退方案。

3. 非确定性是常态

在传统后端开发中，非确定性行为（Non-deterministic Behavior）是需要追查和修复的 Bug。在 AI 应用中，非确定性是系统设计的基线假设，不是异常。

如何在一个非确定性的组件之上构建可靠的系统？核心策略是在合适的层面引入确定性：

  • 结构化输出（Structured Output/JSON Mode）：约束模型输出合法 JSON，用 JSON Schema 校验字段完整性，将”非确定性的自然语言”转化为”确定性的结构化数据”
  • 校验与兜底：对模型输出进行规则校验（字段必填、格式合法、值域正确），校验失败时走降级逻辑（返回默认值 / 提示用户重试 / 切换备用模型）
  • 多层防御：不依赖单一 Prompt 调用来做关键决策。重要场景加入二次确认、人工审核、或并行调用多个模型交叉验证

4. Token 即成本

传统后端中的成本优化通常聚焦于服务器资源（CPU、内存、带宽）和数据库查询。AI 应用增加了一个必须持续关注的成本维度：Token 消耗。

在实践中，有效的 Token 成本控制策略包括：

  • 模型分层路由：简单任务用小模型（便宜），复杂任务用大模型（贵但能力强）。路由决策可以由规则驱动（关键词匹配）或由一个低成本的小模型来做
  • 语义缓存（Semantic Cache）：将历史问答记录 Embedding 后缓存。新问题先计算与缓存中问题的相似度，相似度超过阈值则直接返回缓存答案，完全不调用模型
  • 精确匹配缓存：对于固定的 System Prompt + 相同/高度相似的 User Input，直接返回缓存的模型响应
  • Prompt 瘦身：定期审查 Prompt 模板，删除无效的冗余描述、过长的 Few-shot 示例
  • 可观测性驱动优化：必须有 Token 消耗的实时看板，知道钱花在了哪里，才能做有针对性的优化

5. 安全边界外移

传统后端安全模型中，WAF（Web Application Firewall，Web应用防火墙）和 API Gateway 是第一道防线，负责拦截 SQL 注入、XSS 等攻击。但在 AI 应用中，用户输入直接进入 Prompt 上下文并影响模型行为，传统的攻击签名匹配无法识别 Prompt Injection。

安全防线需要从网关层外移到模型交互层：

  • 输入过滤：在构造 Prompt 之前，对用户输入做安全扫描和敏感词过滤
  • Prompt 沙箱：将用户输入放在 Prompt 的特定标记区域内（如 <user_input>...</user_input>），明确告诉模型”这部分是用户输入，不包含系统指令”
  • 工具白名单与权限最小化：Agent 可调用的工具应遵循最小权限原则（Least Privilege），工具实现中应包含独立的权限校验，不依赖模型判断
  • 输出审核：模型输出在返回给用户之前，经过内容安全审核和敏感信息脱敏
  • 全链路审计：记录每次 LLM 调用的完整输入输出，便于事后追溯和分析

一、大语言模型（LLM）基本概念

1. 什么是 LLM

大语言模型（Large Language Model，简称 LLM）是一种基于 Transformer 架构（一种利用自注意力机制处理序列数据的深度学习架构），在海量文本数据上训练得到的神经网络模型。它的核心能力是根据给定的上文，预测下一个最可能出现的 Token，这个看似简单的”下一个词预测”任务，在大规模数据和参数的加持下，涌现出了理解、推理、生成等复杂能力。

LLM 的发展经历了几个关键阶段：

  • 2017 年：Google 发布论文《Attention Is All You Need》，提出 Transformer 架构，奠定了后续所有大模型的技术基础
  • 2018-2019 年：BERT（Google）、GPT-1/GPT-2（OpenAI）相继发布，验证了预训练 + 微调（Pre-training + Fine-tuning）范式的有效性
  • 2020-2022 年：GPT-3 展示了惊人的 Few-shot 学习能力，模型参数规模突破千亿
  • 2022 年底：ChatGPT 发布，将 LLM 从实验室带入了大众视野，也开启了 AI 应用开发的浪潮
  • 2023-2025 年：开源模型（Llama、Qwen、DeepSeek 等）快速追赶闭源模型，多模态能力、Agent 能力、长上下文成为竞争焦点

对于后端工程师而言，不需要深入理解 Transformer 的数学细节，但需要理解 LLM 的几个核心概念，因为它们直接影响 API 调用方式、成本控制和系统设计。

2. 主流模型简介

当前市场上的主流 LLM 可以分为闭源和开源两大阵营：

闭源模型（通过 API 调用）：

模型 提供商 特点
GPT-4o / GPT-4.1 OpenAI 综合能力最强，多模态原生支持，生态最完善
Claude Opus 4 / Sonnet 4 Anthropic 长上下文能力突出（200K），安全性设计领先，代码与推理能力强
Gemini 2.5 Pro Google 原生多模态，上下文窗口可达 2M Token，与 Google 生态深度整合

开源模型（可自部署或通过 API 调用）：

模型 提供商 特点

选型建议：生产环境中通常不会只用一个模型。一个务实的策略是——主力模型选一个能力强且稳定的（如 GPT-4o 或 Claude Sonnet 4），同时接入一个高性价比模型（如 DeepSeek-V3）处理简单任务，再备一个开源模型（如 Qwen）用于敏感数据场景的私有化部署。

3. Token

Token（词元）是 LLM 处理文本的最小语义单位。它不是字符也不是单词，而是介于两者之间的”子词”（Subword）：

  • 英文中，一个常见单词如 “hello” 通常是 1 个 Token，而 “indescribable” 可能是 3-4 个 Token（in + describ + able）
  • 中文中，一个汉字通常是 1-2 个 Token。例如”你好”约 2 个 Token，”人工智能”约 3-4 个 Token
  • 标点符号、空格、特殊字符也各占 Token

Token 之所以重要，是因为大模型的计费、性能限制和上下文容量全部以 Token 为单位：

  • 计费：API 按 Token 数量收费，分为 Input Token（输入）和 Output Token（输出），通常 Output Token 价格是 Input Token 的 2-5 倍
  • 上下文限制：每个模型有最大 Context Window，即单次能处理的 Token 总数上限
  • 延迟：Token 生成速度（Tokens Per Second, TPS）决定了用户等待时间

实际项目中，预估 Token 消耗的简单公式：英文文本字符数 ÷ 4 ≈ Token 数，中文文本字符数 ÷ 1.5 ≈ Token 数。更准确的计数可以使用模型提供商的 Tokenizer 工具（如 OpenAI 的 tiktoken 库）或在线工具。

4. Context Window

Context Window（上下文窗口）是指模型单次推理能处理的 Token 总数上限——包括输入 Prompt 和模型生成的 Output 之和。

主流模型的 Context Window 大致范围：

  • 128K Token：GPT-4o、Claude Sonnet 4、DeepSeek-V3、Qwen 3 的标准窗口。足够容纳一本约 300 页的英文小说
  • 200K Token：Claude Opus 4 等高端模型
  • 1M-2M Token：Gemini 2.5 Pro（2M）、Qwen 3 某些配置（1M）。理论上可以一次处理整部《三体》三部曲

Context Window 对应用设计的直接影响：

  • 超窗口内容会被截断：输入超过上下文窗口时，模型提供商通常会截断最早的部分（或拒绝请求）。你需要在前端截断，而不是等模型报错
  • 长对话会累积成本：多轮对话中，每次请求都要将历史 Messages 全部带上。对话越长，每次调用的 Input Token 越多，成本越高。这就需要对话摘要（Summarization）或滑动窗口等策略来管理上下文
  • “大海捞针”是真实挑战：窗口大不代表模型能有效利用全部内容。大窗口模型在长文本中间位置的信息提取准确率（Needle in a Haystack 测试）各有差异，选型时需关注

5. 采样参数

LLM 在生成每个 Token 时，实际上是计算词汇表中每个可能的 Token 的概率分布，然后从中采样。以下三个参数控制了这个采样过程：

Temperature（温度，范围 0-2）：

控制输出的随机性和创造性。

  • Temperature = 0 或接近 0：模型几乎总是选择概率最高的 Token，输出高度确定、一致。适用于代码生成、数据提取、分类等需要稳定性的任务
  • Temperature ≈ 0.7-1.0：适度的随机性，适用于通用对话和内容生成
  • Temperature > 1.0：输出更多样化但也更不可控，容易出现离题和幻觉。适用于创意写作、头脑风暴

需要注意的是，Temperature = 0 并不意味着输出完全确定——模型提供商的后端差异、浮点运算精度等因素仍可能导致微小变化。

Top-P（核采样，Nucleus Sampling，范围 0-1）：

从累积概率超过 Top-P 的最小 Token 集合中采样。例如 Top-P = 0.9 表示模型只从概率最高的、累积概率达 90% 的 Token 中采样，忽略长尾的低概率 Token。

实践中的建议：同时调整 Temperature 和 Top-P 通常没有必要且容易混乱，选择其一即可。Temperature 更直观常用；Top-P 在需要动态调整输出多样性时更灵活（如根据输入复杂度自动适配）。

Top-K：

只从概率最高的 K 个 Token 中采样。Top-K 在早期模型中常用，现在的模型多使用 Top-P 替代，因为它能根据概率分布自动调整候选集大小。

场景 Temperature 说明

JSON/结构化输出 0 - 0.2 结合 JSON Mode 使用
事实问答 0 - 0.3 减少幻觉
通用对话 0.7 - 1.0 自然流畅

6. 多轮对话的消息结构

LLM 的 Chat API 采用结构化的消息列表（Messages）来组织对话。每条消息包含两个字段：Role（角色）和 Content（内容）。

三种核心角色：

System Message（系统消息）：

User Message（用户消息）：

用户输入的实际问题和上下文。在多轮对话中，通过交替排列 User 和 Assistant 消息来构建对话历史。

Assistant Message（助手消息）：

模型的历史回复。在构造 API 请求时，你需要把之前模型生成的回复以 Assistant Message 的形式放入 Messages 列表，这样模型才能”记住”自己说过什么。

消息构造的关键规则：

  1. Messages 必须是 Role 交替排列，以 User 消息结束（不能以 Assistant 消息结尾）
  2. 历史消息越长，每次调用的成本越高——所有历史 Messages 都计入 Input Token
  3. Function Call 结果以特殊的 role: "tool" 消息形式插入（详见后续 Function Calling 章节）
  4. 不同模型对 System Message 的重视程度不同——开源模型（尤其微调版）对 System Message 的遵循度不如 GPT-4/Claude 等闭源模型高

4. 封装统一的 LLM Gateway 层

当系统需要同时接入多个模型（如 GPT-4o 做复杂推理、DeepSeek 做简单问答、私有化 Qwen 处理敏感数据）时，直接在业务代码中调用不同模型会导致配置分散、切换困难。这时候需要封装一个统一的 LLM Gateway（网关）层。

LLM Gateway 的核心设计：

Gateway 层的核心能力：

  • 模型路由（Model Routing）：根据请求的模型名或请求属性（复杂度、延迟要求、成本预算）将请求路由到不同的模型提供商
  • 负载均衡（Load Balancing）：当一个模型有多个实例（或多 API Key）时，自动分配请求
  • Fallback 与故障转移：主模型不可用时自动切换到备用模型
  • 成本追踪（Cost Tracking）：统一记录所有模型调用的 Token 消耗，提供按模型、按业务维度的成本报表

适用场景：企业级多模型部署、AI 中台建设、需要统一管控 Token 成本和调用安全的场景。

5. 四种方式的选型建议

一个务实的选择策略：小型项目从 Spring AI 起步（如果使用 Spring Boot）或 LangChain4j 起步（如果需要框架无关性）；企业级场景在框架之上自建 Gateway 层进行统一管控。

四、流式调用（SSE）深度解析

1. SSE 协议原理

SSE（Server-Sent Events，服务端推送事件）是一种基于 HTTP 的服务端向客户端推送数据的技术。与 WebSocket 的全双工不同，SSE 是单向的——服务端推送，客户端只能接收。

SSE 之所以成为 LLM 流式输出的标准协议，原因有二：

  • 实现简单：服务端只需设置 Content-Type: text/event-stream，然后逐行输出 data: <json>\n\n 即可。不需要 WebSocket 的握手和帧协议
  • 浏览器原生支持：前端可以直接使用 EventSource API 消费 SSE，不需要额外库

SSE 数据格式：

每条消息以 data: 开头，以双换行 \n\n 结束。[DONE] 是 OpenAI 定义的特殊结束标记。

3. 流式调用对用户体验的影响

流式调用的价值需要从两个指标来理解：

  • TTFT（Time To First Token，首 Token 延迟）：从用户发送请求到收到第一个 Token 的时间。流式方式下这个值通常明显低于非流式的总耗时，用户感觉系统”立刻开始响应”
  • 总耗时（Total Latency）：所有 Token 生成完毕的时间。流式和非流式的总耗时基本相同

流式输出的心理效应：用户感知的延迟主要是 TTFT，而非总耗时。一个 200ms 首 Token + 10 秒总耗时的流式体验，远好于 10 秒后一次性返回的非流式体验。

非流式调用的适用场景：并非所有场景都需要流式。以下场景适合非流式：

  • 后端异步任务（如离线生成报告、批量处理）
  • 需要完整结果后再处理（如 JSON 解析、数据提取）
  • 输出很短（几十 Token），流式的边际收益不大

4. 流式场景下的错误处理

流式调用的错误处理比非流式更复杂，因为错误可能发生在流的中间（模型已经生成了部分内容）：

  • 连接中断：使用 retry() 操作符实现自动重连。注意：流式调用通常在生成内容后不支持断点续传，重连意味着重新发起完整请求
  • 超时处理：流式调用中，两次 Token 推送之间的间隔如果超过阈值（Idle Timeout），应触发超时。可以设置读超时（Read Timeout）和空闲超时（Idle Timeout）
  • 部分内容处理：流中断时，前端应展示已生成的部分内容，并提示用户”生成中断，请重试”
  • 背压（Backpressure）：当下游消费速度慢于模型生成速度时，框架应能缓存或丢弃中间结果。WebFlux 默认支持背压

五、多模型管理与容错策略

1. 多模型切换的典型场景

生产环境中使用多个模型不是过度设计，而是应对以下现实需求的必然选择：

成本优化：

简单任务（文本分类、关键词提取、格式转换）用小模型，复杂任务（多步推理、代码生成、长文档分析）用大模型。例如：

  • DeepSeek-V3 处理 80% 的日常调用（便宜，每百万 Token 约 2 元 RMB）
  • GPT-4o 处理 15% 的复杂推理（贵，每百万 Token 约 15 元 RMB）
  • Claude Opus 4 处理 5% 的高难度长文档分析

这种分层策略可以将总成本降低 60% 以上。

能力互补：

不同模型有各自的强项。例如：

  • GPT-4o 综合能力强，JSON 结构化输出最稳定
  • Claude 长文本理解和代码生成极其出色
  • DeepSeek-R1 推理链详细，适合需要展示推导过程的场景
  • Qwen 中文能力扎实，适合中文知识库问答

地域合规与数据安全：

敏感数据不能出境时，使用国内模型的 API（如通过阿里云百炼调用 Qwen）或私有化部署开源模型。非敏感数据可以调用海外模型利用其综合能力优势。

  • 模型差异透明化：备用模型的输出质量可能不同，必要时在响应中标注（如 [由备用模型生成]）
